# 軽量SDDワークフロー

大規模コードベースへの機能追加・改修で、既存機能調査・仕様策定・設計・実装・クリーンアップを段階化する。
調査と実装はサブエージェントへ分業し、メインセッションのコンテキスト消費を抑える。

初めに開発中バージョンディレクトリ（既定では`docs/v{next}/`配下）に作業版ドキュメントを作成し、実装と設計判断をそこへ集約する。
リリース済み機能の最新仕様は恒常配置（既定では`docs/features/`・`docs/topics/`）を正とし、開発中の変更は恒常配置を直接編集しない。
リリース後、開発中バージョンディレクトリの内容を`spec-driven-promote`で恒常配置へマージ統合する。

## 適用条件と非適用条件

適用対象:

- 複数の機能追加・改修をまとめてリリースする場合

非適用:

- バグ修正・イシュー対応、単発の機能追加・改修（`plan-mode`→`plan-implementation`へ進む）
- 既存プロジェクトへの初回導入（`spec-driven-init`を使う）
- リリース完了後の昇格作業（`spec-driven-promote`を使う）

プロジェクトがspec-drivenを採用しない旨を明記している場合は、本スキル群を起動せずユーザーに伝える。

## 用語

具体パス表記はすべて既定値で、プロジェクトのドキュメントで別配置が規定されていればそちらを優先する。

正式語として対称形「作業版ドキュメント」「恒常配置ドキュメント」を使う。
ファイルパスを直接指す文脈に限り短縮形「作業版`.md`」「恒常配置`.md`」を使ってよい。
「恒久ドキュメント」など指す対象が揺れる総称表現は用いない。

- 軽量SDD: 本スキルが提供するワークフローの通称
- 恒常配置: リリース済み機能の最新仕様を置く場所（既定: `docs/features/`・`docs/topics/`配下）。プロジェクトの正
- 恒常配置ドキュメント: `docs/features/{機能名}.md`（機能ドキュメント）と`docs/topics/{トピック名}.md`（横断ドキュメント）
- 開発中配置: 次期リリース作業中のみ存在する一時的な作業場所（既定: `docs/v{next}/`配下）。
  リリース完了後に`spec-driven-promote`で恒常配置へマージ統合され、物理ディレクトリは消滅する
- 開発中バージョンディレクトリ: 開発中配置の物理ディレクトリ（既定: `docs/v{next}/`）
- 次期バージョン（`{next}`）: 次期リリースのバージョン番号。ステップ1でユーザーと合意の上で確定する。
  末尾が`.0`で終わる成分は末尾から順に省略する
 （例: `3.0.0`→`v3`、`1.2.0`→`v1.2`、`1.2.3`はそのまま`v1.2.3`）。
  バージョン番号が定まらないプロジェクトは日付ベース（`v2026-04-18`）や単純連番（`v1`）も使ってよい
- 作業テーマ: 今回のリリースで加える変更の単位（例: `SSO追加`）。恒常配置の機能単位と1対多・多対1・1対1のいずれにもなりうる
- 作業版ドキュメント: 開発中配置に置く作業テーマ単位の`.md`（例: `docs/v{next}/SSO追加.md`）。
  恒常配置の対応先・背景・スコープ・受け入れ基準・主要設計判断・却下した代替案を集約する。
  既存改修時は「改修前スナップショット」節も持つ
- 骨子: `spec-writer`がステップ2後半で一括生成する作業版ドキュメントの初版
- 計画ファイル: plan mode内で作成する大枠方針と実装詳細の統合ドキュメント。
  パスは`~/.claude/plans/{自動生成ファイル名}.md`。
  `## 却下した代替案`・`## 変更履歴`節が判断経緯の一次記録となる

## ディレクトリ規約

### 恒常配置（常時存在）

- 機能ドキュメント: `docs/features/{機能名}.md`。背景・スコープ・受け入れ基準・主要設計判断・却下した代替案
- 横断ドキュメント: `docs/topics/{トピック名}.md`。機能ドキュメントと同じテンプレートを流用し、該当しない節は省略する

機能名・トピック名は日本語の自然な表現を使う（例: `docs/features/ユーザー認証.md`）。

### 開発中配置（次期リリース作業中のみ）

- 作業版ドキュメント: `docs/v{next}/{作業テーマ名}.md`・`docs/v{next}/{トピック名}.md`。
  冒頭に「恒常配置の対応先」節を設け、昇格時のマージ対象を明示する。
  既存改修時のみ「改修前スナップショット」節を設け、恒常配置の該当節を改修前にコピーする
 （Before/After検証専用、昇格時に破棄）
- バージョン目次: `docs/v{next}/README.md`。バージョン目的・作業テーマ一覧・横断ドキュメント一覧の3節
- サブエージェント出力（一時ファイル、Cleanupステップで削除）。`.cache/`配下に置きgit管理から外す。
  `.gitignore`への`docs/v*/.cache/`追記は`spec-driven-init`が担当する
  - `docs/v{next}/.cache/{作業テーマ名}.research-{nn}.md` — `spec-researcher`出力。
    観点ごとに1ファイル、`{nn}`はゼロパディング2桁の連番

### ドキュメント粒度

恒常配置は機能単位（例: `認証.md`）で全体像を網羅する。開発中配置は作業テーマ単位（例: `SSO追加.md`）で今回のリリースの変更を集約する。
例えば既存`認証.md`へSSOを追加する改修では、開発中配置に`SSO追加.md`を置き、昇格時に`認証.md`へマージ統合する。
新機能追加で単純な1対1対応になる場合もある。

### 前段階・後段階の別スキル

- `spec-driven-init`: 既存プロジェクトへの初回導入（既存コード・ドキュメント・既存`docs/v*/`群から恒常配置の初版整備）
- `spec-driven-promote`: リリース完了後の昇格（作業版ドキュメントを恒常配置へマージ統合し、開発中バージョンディレクトリを削除）

どちらも手動起動または本スキルからの明示呼び出しでのみ使用する（自動起動対象ではない）。

## 横断ドキュメントの判定

横断ドキュメントと機能ドキュメントは命名規則とテンプレートが同じで、用途と配置ディレクトリのみ異なる。
判断基準は「複数機能で参照される設計判断・ガイドライン・共通用語」かどうか。
迷う場合は機能ドキュメントへ書き、後から必要に応じて`docs/topics/`へ昇格する段階的運用を推奨する。
最終判断はステップ2後半で`spec-writer`が行い、判定が曖昧な場合はメインへエスカレーションする。

## 参照ファイル

- `references/workflow.md`: 3ステップの詳細手順・既存機能改修フローとの差分・サブエージェント呼び出しテンプレート
- `references/templates.md`: 作業版ドキュメント・恒常配置ドキュメント・`README.md`・参照コメント・レビュアー呼び出しの書式テンプレート
- `references/operations.md`: 中断・再開の判定基準

## 3ステップワークフロー

ステップ終端にユーザー確認ゲートは置かない（ステップ間の進行許可は求めない）。
ただし作業テーマ名・バージョン番号・横断ドキュメント化の要否など内容的な判断は都度ユーザーに確認する。
進行管理はメインのTaskCreateで行い、残タスクと開発中バージョンディレクトリ配下のファイル存在から再開ポイントを判断する。

| ステップ | 目的 | 主担当 |
| --- | --- | --- |
| 1. Setup&Explore | 要件確定・既存機能調査 | メイン＋`spec-researcher` |
| 2. Plan&Implement | 計画・ドキュメント・実装・検証・コミット | メイン、`plan-mode`、`plan-implementation`、`spec-writer` |
| 3. Cleanup | 一時ファイル削除・追加コミット | メイン |

詳細手順は`references/workflow.md`を参照する。

### 自動トリガー時の確認

本スキルは条件付きで自動起動もありうる（`description`参照）。
自動起動した場合は必ず最初にユーザーへ「本スキルで進めてよいか」を確認する（副作用の大きい多段作業のため）。

## plan-mode・plan-implementationとの連携

ステップ2では`plan-mode`スキルを内部呼び出しして計画ファイル作成・codexレビューまでを進め、
`ExitPlanMode`後は`plan-implementation`スキルへ切り替えて実装・検証・レビュー・コミットの全工程を委譲する。

`ExitPlanMode`の直後にメインは`spec-writer`を起動し、`plan-implementation`の実装フェーズと並行で
作業版ドキュメントの骨子・`README.md`エントリ・必要なら横断ドキュメントを生成させる。
`spec-writer`と`plan-implementer`はいずれも計画ファイルをインプットとして独立に動く。
両者の完了を待ってレビューフェーズへ進み、合格・コミット完了後にステップ3（Cleanup）へ移る。

計画ファイルはcodex向けの照合材料として、恒常配置ドキュメント（および作業版`.md`）は将来の人間向け参照資料として機能する。
ドキュメント側には実装詳細を含めず、実装とドキュメントの整合性は`plan-reviewer`の`spec`種別が確認する。

## サブエージェント分業

本スキル配下で起動するサブエージェントは`spec-researcher`・`spec-writer`の2つで、いずれも既定で`model: sonnet`。
役割・手順・出力フォーマット・制約はそれぞれの`agents/`定義を参照する。
実装・検証・レビューを担うエージェントは`plan-implementation`の所掌（`plan-implementer`・`plan-reviewer`）。

メインの責務はステップ進行管理・ユーザー確認・`plan-mode`／`plan-implementation`呼び出しに集中させ、
調査と骨子立ち上げのコンテキストはサブエージェント側に閉じ込める。

## 参照コメント方針

コードから恒常配置ドキュメントへの参照コメントは最小限に絞る（過剰な参照はコード可読性を損ねるため）。

- 対象: 新規モジュールの冒頭、および大きめの新規クラス・関数の先頭
- 形式: 1行で恒常配置`.md`のみを指す（例: `# 仕様: docs/features/ユーザー認証.md`）
- 開発中は開発中配置のパス（`docs/v{next}/{作業テーマ名}.md`）を指してよい。
  `spec-driven-promote`によるマージ統合時に恒常配置のパスへ一括書き換える
- `.research-*.md`は参照しない（実装完了時に削除されリンク切れを招くため）

既存ファイルへの小さな追加や、恒常配置ドキュメントで直接触れられていない細部には参照コメントを書かない。
言語別の書式例は`references/templates.md`を参照する。

## Gotchas

- 一時ファイルの削除漏れ: `.research-*.md`はCleanupステップで削除する。
  `.research-{nn}.md`はゼロパディング2桁の連番で統一する
- 新規バージョンの最初の作業テーマで`docs/v{next}/.cache/`が未作成だと`spec-researcher`の出力が失敗する。
  メインが空ディレクトリのみ先行作成する（骨子・`README.md`の生成は`spec-writer`の責務）
- `.gitignore`への`docs/v*/.cache/`追記漏れ: 通常は`spec-driven-init`が追記する。
  本スキルを直接使い始めた場合は、初回起動時に確認し未登録なら追記する
 （中間成果物がgit管理下に混入し品質ツールに拾われるため）
- `README.md`の更新漏れ: 作業テーマ・横断ドキュメント追加時は`docs/v{next}/README.md`の各一覧へ反映する。
  `spec-writer`が追加し、`plan-reviewer`の`spec`種別が最終確認する
- 計画ファイル→作業版`.md`の転記漏れ: `spec-writer`が判断経緯を「主要設計判断」「却下した代替案」節へ転記する。
  `plan-reviewer`の`spec`種別が転記漏れを検査する
- 作業版`.md`の命名: 恒常配置の機能名と同じ名前を付けると単純上書き運用を誘発し粒度差が隠れる。
  作業テーマを端的に表す名前（例: `SSO追加.md`）を付け、昇格時のマージ対象機能を「恒常配置の対応先」節に明示する
- 既存改修時の恒常配置の直接編集禁止: 改修は作業版`.md`を新規作成して行い、恒常配置を直接編集しない
 （昇格前に恒常配置と実装が乖離するのを避けるため）
- 改修前スナップショットは昇格時に恒常配置へマージせず破棄する（Before/After検証専用、`spec-driven-promote`が除外する）
- 次期バージョン番号はステップ1でユーザー確認のうえ確定する（プロジェクトの既存命名規則と整合させる）
- 自動トリガー時は必ず最初に起動可否をユーザーへ確認する（descriptionの条件絞りでも誤検知は起こりうる）
