# 軽量SDDワークフロー

大規模コードベースへの機能追加・改修で、仕様策定・既存機能調査・設計・実装・クリーンアップを段階化する。
各ステップで生成した情報を恒久ドキュメント（既定では`docs/features/`・`docs/topics/`配下）と一時メモ（`.working.md`）に分け、将来の類似改修時の参照材料として蓄積する。
リリース済み機能の最新仕様は恒常配置（既定では`docs/features/`・`docs/topics/`）に常時集約し、次期リリース作業中は開発中バージョンディレクトリ（既定では`docs/v{next}/`配下）へ作業版を置く。
調査と実装はサブエージェントへ分業し、メインセッションのコンテキスト消費を抑える。

## 適用条件と非適用条件

適用対象:

- 既存の大規模コードベースに新機能を追加する
- 既存機能を改修し、恒久ドキュメントの更新と既存機能との整合確認が必要
- 仕様と既存機能の整合確認が必要で、検討漏れによるデグレードのリスクが大きい
- 設計判断を恒久ドキュメントに残し、将来の類似改修で参照したい

非適用:

- バグ修正・イシュー対応（`bugfix`スキルを使う）
- 単発の小規模変更（1〜2ファイル程度で影響範囲が自明）
- 仕様が未確定で、仕様化コストより動作例の方が意思決定を早める場合（`plan-mode`で軽量プロトタイプ反復を検討）
- リネーム・リファクタリングのみ（`plan-mode`の計画ファイルで十分）
- 既存プロジェクトにspec-drivenを導入するための恒常配置の初回整備（`spec-driven-init`スキルを使う）
- リリース完了後の開発中バージョンディレクトリから恒常配置への昇格（`spec-driven-promote`スキルを使う）

## 配置の既定と上書き

本スキル群で使う恒常配置・開発中配置のパスは既定値である。
既定値は恒常配置が`docs/features/`・`docs/topics/`、開発中配置が`docs/v{next}/`配下。
本文および`references/`配下の具体パス表記はすべてこの既定を前提とした例示であり、プロジェクトのドキュメントで別の配置が規定されている場合はそちらを優先する。
プロジェクトがspec-drivenを採用しない旨を明記している場合は、本スキル群（`spec-driven`・`spec-driven-init`・`spec-driven-promote`）を起動せず、その旨ユーザーに伝える。

## ディレクトリ規約

恒久ドキュメントは「恒常配置」と「開発中配置」の2系統に分かれる。
恒常配置は常時存在し、プロジェクトの既存機能仕様すべての正となる。
開発中配置は次期リリース作業中のみ一時的に存在し、新規追加・改修される仕様の作業版を置く。
リリース完了後、作業版は`spec-driven-promote`スキルで恒常配置へマージ統合され、開発中ディレクトリは消滅する。

### 粒度差の前提

恒常配置と開発中配置はドキュメントの粒度が異なりうる。

- 恒常配置: 機能単位（例: `認証.md`）。機能の全体像を網羅する
- 開発中配置: 作業テーマ単位（例: `SSO追加.md`）。今回のリリースで加える変更を集約する

例えば既存の`認証.md`に対してSSO対応を追加する改修では、開発中配置には`SSO追加.md`を置き、昇格時に`認証.md`へマージ統合する。
単純な1対1対応（開発中`.md`と恒常配置`.md`がほぼ同粒度）も従来通りサポートする。

### 恒常配置（常時存在）

- 機能ドキュメント
  - パス: `docs/features/{機能名}.md`
  - 内容: 背景・スコープ・受け入れ基準・主要設計判断・却下した代替案
  - 命名: 機能名は日本語の自然な表現を使う（例: `docs/features/ユーザー認証.md`）
- 横断ドキュメント（複数機能で参照される設計判断・ガイドライン・共通用語）
  - パス: `docs/topics/{トピック名}.md`
  - 内容: 機能ドキュメントと同じテンプレートを流用し、受け入れ基準など該当しない節は省略する
  - 命名: 機能ドキュメントと同様に日本語の自然な表現を使う

### 開発中配置（次期リリース作業中のみ）

- 機能・横断ドキュメントの作業版
  - パス: `docs/v{next}/{作業テーマ名}.md`・`docs/v{next}/{トピック名}.md`
  - 用途: 今回のリリースで加える変更を集約する。作業テーマは恒常配置の機能単位と1対多・多対1・1対1のいずれにもなりうる
  - 各作業版の冒頭に「恒常配置の対応先」節を設け、昇格時のマージ対象（例: `docs/features/認証.md`）を明示する
- `README.md`（次期バージョン全体の概要と目次）
  - パス: `docs/v{next}/README.md`
  - 内容: バージョン目的・作業テーマ一覧・横断ドキュメント一覧の3節（実装ステータスは扱わない）
  - 役割: 次期リリース作業期間中の入口。昇格時に役目を終える
- 一時ドキュメント（実装完了時に削除）
  - パス: `docs/v{next}/{作業テーマ名}.working.md`
  - 内容: 現在のステップ・既存調査メモ・判断経緯（計画ファイル転記元）・タスク分解・レビュー履歴・未決事項
  - git管理下に置く（作業中コミットで中間成果物を保存するため）
  - 横断ドキュメントと`README.md`には一時版を設けない。作業版`.md`を直接編集する

### 計画ファイル

- パス: `~/.claude/plans/{自動生成ファイル名}.md`（Claude Code標準のplan fileパスを変えずに使用）
- ステップ2で1本だけ作成し、大枠方針と実装詳細を同一ファイルへ集約する。codexレビューを通す
- 計画ファイル確定後、判断経緯の要点を開発中`.md`・`.working.md`へ転記し、計画ファイルは`~/.claude/plans/`配下に残置する

### 次期バージョン番号

`{next}`は次期リリースのバージョン番号を指し、ステップ1でユーザーと合意の上で確定する。
末尾が`.0`で終わる成分は省略する（例: `3.0.0`→`v3`、`1.2.0`→`v1.2`、`1.2.3`はそのまま`v1.2.3`）。パス長を抑え、MINOR以下を使わないプロジェクトでも自然な表記になる。
バージョン番号が定まらないプロジェクトでは、日付ベース（`v2026-04-18`）や単純連番（`v1`）も使ってよい。

### 前段階・後段階の別スキル

- 既存プロジェクトへのspec-driven導入（既存コード・ドキュメント・既存`docs/v*/`群から恒常配置の初版を整備する作業）は`spec-driven-init`スキルで扱う
- リリース完了後の恒常配置への昇格（開発中バージョンディレクトリの作業版を恒常配置へマージ統合し、開発中ディレクトリを削除する作業）は`spec-driven-promote`スキルで扱う

どちらも手動起動または本スキルからの明示呼び出しでのみ使用する（ユーザー相談への自動起動対象ではない）。

## 横断ドキュメントの判定

横断ドキュメントと機能ドキュメントは命名規則とテンプレートが同じで、用途と配置ディレクトリのみ異なる。
判断基準は「複数機能で参照される設計判断・ガイドライン・共通用語」にあたるかどうか。
該当するなら恒常配置では`docs/topics/`、開発中配置では`docs/v{next}/`（作業テーマと同居）へ置く。
迷う場合は機能ドキュメントへ書き、後から必要に応じて`docs/topics/`へ昇格する段階的運用を推奨する（早期の抽象化で維持コストが増えるのを避けるため）。

## 3ステップワークフロー

詳細手順・既存機能改修フロー（新規追加との差分）・サブエージェント呼び出しテンプレートは`references/workflow.md`を読む。
恒久`.md`と一時`.working.md`のテンプレートは`references/templates.md`を読む。
中断・再開と導入チェックリスト（通常フロー外）は`references/operations.md`を読む。

| ステップ | 目的 | 主な成果物 | 主担当 |
| -- | -- | -- | -- |
| 1. Setup&Explore（plan mode前） | 要件・作業テーマ・次期バージョンの確定、恒常配置と開発中配置の確認、既存機能調査 | 開発中`.md`骨子・`.working.md`初版・`README.md`エントリ・調査メモ | メイン＋spec-researcher |
| 2. Plan&Implement（plan mode 1回出入り） | 計画ファイル1本（大枠＋実装詳細）作成・codexレビュー合格、ExitPlanMode後の判断経緯転記と実装 | 計画ファイル（`~/.claude/plans/`）、開発中`.md`の判断経緯節、実装差分 | メイン＋plan-mode＋spec-implementer |
| 3. Cleanup（plan mode後） | 検証・ドキュメント整合性レビュー・コード品質レビュー・一時ファイル削除・コミット | format/lint/test合格、両レビュー合格、`.working.md`削除、コミット | メイン＋spec-reviewer＋code-quality-reviewer |

ステップ終端にユーザー確認ゲートは置かない。判断が必要な場面は都度ユーザーに確認する。
恒常配置と開発中配置の両方が常時参照対象となる。関連する既存機能は恒常配置を起点に特定し、進行中の関連仕様変更がある場合は開発中配置から取得する。

### 自動トリガー時の確認

本スキルは条件付きで自動起動もありうる（`description`参照）。
自動起動した場合は、必ず最初にユーザーへ「本スキルで進めてよいか」を確認する（副作用の大きい多段作業のため）。
ユーザーが別方式を希望すればスキルを抜ける。

## plan-modeとの連携

ステップ2では`plan-mode`スキルを内部呼び出しし、計画ファイル作成とcodexレビューを委譲する。
codexレビュー手順・MCP優先／CLIフォールバックの切り替え・指摘への対応フローは`plan-mode`に従う。
本スキル側ではcodexレビュー手順を再記述しない（SSOTを`plan-mode`に置き、保守性を確保するため）。

ステップ2ではplan modeへ1回だけ出入りし、計画ファイルを1本作成する。
大枠方針・主要設計判断・実装詳細（file:line・変更内容）を同一計画ファイルにまとめ、codexレビュー合格で`ExitPlanMode`する。
計画ファイルの`## 却下した代替案`・`## 変更履歴`節が判断経緯の一次記録となる。

計画ファイル確定後、`ExitPlanMode`の直後にメインが以下を行う。

1. 計画ファイル内の判断経緯（`## 却下した代替案`・`## 変更履歴`・ユーザー指示で確定した方針）から、将来の類似改修でも参照される要点を開発中`.md`の該当節（主要設計判断・却下した代替案）および`.working.md`の「判断経緯」節へ転記する
2. 計画ファイルの「変更内容」節を起点にTaskCreateへタスク分解して登録する
3. spec-implementerへ順次委譲する

計画ファイル（`~/.claude/plans/{自動生成ファイル名}.md`）はcodex向けの照合材料として機能し、恒久`.md`は将来の人間向け参照資料として機能する。
両者の関心の違いを意識し、計画ファイルと恒久`.md`の両方に同じ情報を重複させない（重複による陳腐化を避けるため）。

## サブエージェント分業ガイド

- `spec-researcher`（`model: sonnet`）: 既存機能調査・影響範囲分析を担う。観点ごとに並列起動可
- `spec-implementer`（`model: sonnet`）: タスク単位の実装を担う。依存関係の都合から原則順次起動
- `spec-reviewer`（`model: sonnet`）: 仕様適合性・ドキュメント整合性レビューを担う読み取り専用エージェント。機能単位（全タスク完了・format/lint/test合格後）に1回起動
- `code-quality-reviewer`（`model: sonnet`）: コード品質レビューを担う読み取り専用エージェント。spec-reviewer合格後に機能単位で起動
- 一過性の単純作業（単純な検索・機械的な置換など）は都度Agentツールで`haiku`を呼ぶ

メインセッションはステップ進行管理・ユーザー確認・`plan-mode`呼び出しに集中する。
調査・実装・レビューのコンテキストはサブエージェントに閉じ込め、メイン側のコンテキスト汚染を避ける。

レビュアーは直列で起動し、`format/lint/test` → `spec-reviewer` → `code-quality-reviewer`の順序を固定する。
品質レビューは仕様適合性を前提に成立するため、順序を逆転させない。
指摘があれば`spec-implementer`へ差し戻して修正後に再レビューし、両レビュアーが合格を返すまでループする。

## 参照コメント方針

コードから恒久ドキュメントへの参照コメントは最小限に絞る。
過剰な参照はコード可読性を損ねるため。

- 対象: 新規モジュールの冒頭、および大きめの新規クラス・関数の先頭
- 形式: 1行で恒久`.md`のみを指す（例: `# 仕様: docs/features/ユーザー認証.md`）
- 最終形は恒常配置のパス（`docs/features/`・`docs/topics/`）を指す。参照先の安定性を保つため
- 開発中は開発中配置のパス（`docs/v{next}/{作業テーマ名}.md`）を指してよい。`spec-driven-promote`スキルによるマージ統合時に、恒常配置のパス（該当機能ファイル）へ一括書き換える
- 一時`.working.md`は参照しない（実装完了時に削除されるためリンク切れを招く）

既存ファイルへの小さな追加や、恒久`.md`で直接触れられていない細部には参照コメントを書かない。

言語別の具体的な書式例は`references/templates.md`を参照する。

## Gotchas

- `.working.md`の削除漏れ: Cleanupステップで`references/operations.md`の導入チェックリストを使い必ず確認する
- `README.md`の更新漏れ: 作業テーマや横断ドキュメントを追加したのに`docs/v{next}/README.md`の一覧へ反映し忘れると入口が陳腐化する。Cleanupステップで必ず確認する
- 計画ファイル→開発中`.md`の転記漏れ: `ExitPlanMode`直後に転記せず実装を進めると、判断経緯が計画ファイル内のみに残り将来参照しにくくなる。`spec-reviewer`の観点で転記漏れを検査するが、メイン側でも転記完了を実装着手前に済ませる
- 恒久ドキュメント間の矛盾検出: 作業テーマ間・横断間・`README.md`との相互矛盾は`spec-reviewer`の責務で検査する。メイン側で見落としてもレビュアーで捕捉する前提だが、Cleanupステップの順序を守り最終反映後にレビュアーへ渡す
- 昇格時の参照コメント書き換え漏れ: 開発中配置のパスを指す参照コメントは`spec-driven-promote`スキルの実行時に恒常配置のパスへ一括書き換える。書き換え漏れは昇格後のリンク切れを招くため、昇格スキル内の検証手順を必ず踏む
- 既存改修時の恒常配置の直接編集: 改修作業は開発中ディレクトリに作業テーマ単位の`.md`を新規作成して行い、恒常配置を直接編集しない（昇格前に恒常配置と実装が乖離するのを避けるため）。恒常配置の既存記述は作業版の「恒常配置の対応先」節で参照する
- 開発中`.md`の命名: 恒常配置の機能名と同じ名前で作業版を置くと単純上書き運用を誘発し、粒度差が隠れる。作業テーマを端的に表す名前（例: `SSO追加.md`）を付け、昇格時のマージ対象機能を「恒常配置の対応先」節に明示する
- 計画ファイルと恒久`.md`の情報重複: 計画ファイルはcodex向け照合材料、恒久`.md`は将来の人間向け参照資料、という関心の違いを意識する
- `plan-mode`のcodexレビュー指示文の再記述: 本スキル側で書き直さず`plan-mode`へ委譲する（保守性確保のため）
- 次期バージョン番号の推測: ステップ1でユーザーに確認してから確定する。プロジェクトの既存命名規則と整合させる
- 自動トリガー時のユーザー確認省略: 自動起動した場合は必ず最初にユーザーへ起動可否を確認する。descriptionで条件を絞っても誤検知は起こりうるため
