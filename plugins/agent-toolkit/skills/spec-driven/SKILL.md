# 軽量SDDワークフロー

大規模コードベースへの機能追加・改修で、仕様策定・既存機能調査・設計・実装・クリーンアップを段階化する。
各フェーズで生成した情報を恒久ドキュメント（`docs/features/`・`docs/topics/`配下）と一時メモ（`.working.md`）に分け、将来の類似改修時の参照材料として蓄積する。
リリース済み機能の最新仕様は恒常配置（`docs/features/`・`docs/topics/`）に常時集約し、次期リリース作業中は開発中バージョンディレクトリ（`docs/v{next}/`配下）へ作業版を置く。
調査と実装はサブエージェントへ分業し、メインセッションのコンテキスト消費を抑える。

## 適用条件と非適用条件

適用対象:

- 既存の大規模コードベースに新機能を追加する
- 既存機能を改修し、恒久ドキュメントの更新と既存機能との整合確認が必要
- 仕様と既存機能の整合確認が必要で、検討漏れによるデグレードのリスクが大きい
- 設計判断を恒久ドキュメントに残し、将来の類似改修で参照したい

非適用:

- バグ修正・イシュー対応（`bugfix`スキルを使う）
- 単発の小規模変更（1〜2ファイル程度で影響範囲が自明）
- 仕様が未確定で、仕様化コストより動作例の方が意思決定を早める場合（`plan-mode`で軽量プロトタイプ反復を検討）
- リネーム・リファクタリングのみ（`plan-mode`の計画ファイルで十分）
- 既存プロジェクトにspec-drivenを導入するための恒常配置の初回整備（`spec-driven-init`スキルを使う）
- リリース完了後の開発中バージョンディレクトリから恒常配置への昇格（`spec-driven-promote`スキルを使う）

## ディレクトリ規約

恒久ドキュメントは「恒常配置」と「開発中配置」の2系統に分かれる。
恒常配置は常時存在し、プロジェクトの既存機能仕様すべての正となる。
開発中配置は次期リリース作業中のみ一時的に存在し、新規追加・改修される恒久ドキュメントの作業版を置く。
リリース完了後、作業版は`spec-driven-promote`スキルで恒常配置へ昇格され、開発中ディレクトリは消滅する。

### 恒常配置（常時存在）

- 機能ドキュメント
  - パス: `docs/features/{機能名}.md`
  - 内容: 背景・スコープ・受け入れ基準・主要設計判断・却下した代替案
  - 命名: 機能名は日本語の自然な表現を使う（例: `docs/features/ユーザー認証.md`）
- 横断ドキュメント（複数機能で参照される設計判断・ガイドライン・共通用語）
  - パス: `docs/topics/{トピック名}.md`
  - 内容: 機能ドキュメントと同じテンプレートを流用し、受け入れ基準など該当しない節は省略する
  - 命名: 機能ドキュメントと同様に日本語の自然な表現を使う

### 開発中配置（次期リリース作業中のみ）

- 機能・横断ドキュメントの作業版
  - パス: `docs/v{next}/{機能名}.md`・`docs/v{next}/{トピック名}.md`
  - 用途: 新規追加は初版を、既存改修は恒常配置からコピーした作業版を置き、作業中の差分を集約する
- `README.md`（次期バージョン全体の概要と目次）
  - パス: `docs/v{next}/README.md`
  - 内容: バージョン目的・機能一覧・横断ドキュメント一覧の3節（実装ステータスは扱わない）
  - 役割: 次期リリース作業期間中の入口。昇格時に役目を終える
- 一時ドキュメント（実装完了時に削除）
  - パス: `docs/v{next}/{機能名}.working.md`
  - 内容: 現在のフェーズ・既存調査メモ・タスク分解・レビュー履歴・未決事項
  - git管理下に置く（作業中コミットで中間成果物を保存するため）
  - 横断ドキュメントと`README.md`には一時版を設けない。作業版`.md`を直接編集する

### 計画ファイル

- パス: `~/.claude/plans/{自動生成ファイル名}.md`（Claude Code標準のplan fileパスを変えずに使用）
- Designフェーズで大枠planと実装planの2本を作成し、いずれもcodexレビューを通す。確定後に要点のみを恒久`.md`へ転記する

### 次期バージョン番号

`{next}`は次期リリースのバージョン番号を指し、Intakeフェーズでユーザーと合意の上で確定する。
末尾が`.0`で終わる成分は省略する（例: `3.0.0`→`v3`、`1.2.0`→`v1.2`、`1.2.3`はそのまま`v1.2.3`）。パス長を抑え、MINOR以下を使わないプロジェクトでも自然な表記になる。
バージョン番号が定まらないプロジェクトでは、日付ベース（`v2026-04-18`）や単純連番（`v1`）も使ってよい。

### 前段階・後段階の別スキル

- 既存プロジェクトへのspec-driven導入（既存コード・ドキュメント・既存`docs/v*/`群から恒常配置の初版を整備する作業）は`spec-driven-init`スキルで扱う
- リリース完了後の恒常配置への昇格（開発中バージョンディレクトリの作業版を恒常配置へ移し、開発中ディレクトリを削除する作業）は`spec-driven-promote`スキルで扱う

どちらも手動トリガー専用で、ユーザーが明示的に起動する。

## 横断ドキュメントの判定

横断ドキュメントと機能ドキュメントは命名規則とテンプレートが同じで、用途と配置ディレクトリのみ異なる。
判断基準は「複数機能で参照される設計判断・ガイドライン・共通用語」にあたるかどうか。
該当するなら恒常配置では`docs/topics/`、開発中配置では`docs/v{next}/`（機能ドキュメントと同居）へ置く。
迷う場合は機能ドキュメントへ書き、後から必要に応じて`docs/topics/`へ昇格する段階的運用を推奨する（早期の抽象化で維持コストが増えるのを避けるため）。

## 5フェーズワークフロー

詳細手順・既存機能改修フロー（新規追加との差分）・ユーザー確認ゲート文例・サブエージェント呼び出しテンプレートは`references/workflow.md`を読む。
恒久`.md`と一時`.working.md`のテンプレートは`references/templates.md`を読む。
中断・再開と導入チェックリスト（通常フロー外）は`references/operations.md`を読む。

| フェーズ | 目的 | 主な成果物 | 主担当 |
| -- | -- | -- | -- |
| 1. Intake | 要件・機能名・次期バージョンを確定 | 開発中`.md`骨子（新規）または恒常配置からのコピー（改修）、`.working.md`初版、`README.md`のエントリ | メイン |
| 2. Explore | 既存機能調査・影響範囲の把握 | `.working.md`の調査メモ | spec-researcher |
| 3a. Design（大枠plan） | 大枠方針・主要設計判断の骨格を合意 | 大枠計画ファイル（`~/.claude/plans/{自動生成名}.md`）、codex合格と`ExitPlanMode` | メイン＋plan-mode |
| 3b. Design（具体化） | 開発中`.md`の詳細化・必要なら横断ドキュメント切り出し | 開発中`.md`の設計節、開発中横断`.md` | メイン |
| 3c. Design（実装plan） | 実装詳細（file:line・変更内容）を合意 | 実装計画ファイル（`~/.claude/plans/{自動生成名}.md`）、codex合格と`ExitPlanMode` | メイン＋plan-mode |
| 4. Tasks | 実装タスクの分解・登録 | `.working.md`のタスク分解、TaskCreate登録 | メイン |
| 5. Implement＆Cleanup | タスク実装・検証・最終反映・レビュー・一時ファイル削除・コミット | 実装差分、開発中`.md`のみ残置（恒常配置への移動は`spec-driven-promote`で行う） | spec-implementer |

各フェーズ終端でユーザー確認ゲートを通過してから次フェーズへ進む。
ゲートの具体文例と成果物チェックは`references/workflow.md`に委譲する。
恒常配置と開発中配置の両方が常時参照対象となる。関連する既存機能は恒常配置を起点に特定し、進行中の関連仕様変更がある場合は開発中配置から取得する。

## plan-modeとの連携

Designフェーズでは`plan-mode`スキルを内部呼び出しし、計画ファイル作成とcodexレビューを委譲する。
codexレビュー手順・MCP優先／CLIフォールバックの切り替え・指摘への対応フローは`plan-mode`に従う。
本スキル側ではcodexレビュー手順を再記述しない（SSOTを`plan-mode`に置き、保守性を確保するため）。

Designは3サブフェーズで構成し、plan modeを2つの合意ゲート（大枠合意と実装合意）として使う。

- 3a大枠plan（plan mode内）: 大枠方針・主要設計判断の骨格を計画ファイルに書く。codexレビュー合格と大枠合意を経て`ExitPlanMode`
- 3b具体化（plan mode外）: 開発中`.md`の詳細を書き上げ、必要に応じて横断ドキュメントを切り出す。plan modeの編集制約下では開発中`.md`を触れないため、いったん外に出てから作業する。末尾でユーザー確認ゲートを通過する
- 3c実装plan（plan mode内）: 3b末尾の合意を受けてメインが`EnterPlanMode`ツールで再突入し、新規計画ファイルへ実装詳細（file:line・変更内容）を書く。codexレビュー合格と実装合意を経て`ExitPlanMode`

計画ファイル（`~/.claude/plans/{自動生成ファイル名}.md`）はcodex向けの照合材料として機能する。
3a・3cで計画ファイルは2本生成するが、いずれも`~/.claude/plans/`配下に残置する（Claude Code標準の挙動に揃える）。
確定した設計判断のうち将来の類似改修でも参照される要点のみを開発中`.md`（昇格後は恒常配置`.md`）へ転記する。
計画ファイルと恒久`.md`の両方に同じ情報を書かない（重複による陳腐化を避けるため）。

## サブエージェント分業ガイド

- `spec-researcher`（`model: sonnet`）: 既存機能調査・影響範囲分析を担う。観点ごとに並列起動可
- `spec-implementer`（`model: sonnet`）: タスク単位の実装を担う。依存関係の都合から原則順次起動
- `spec-reviewer`（`model: sonnet`）: 仕様適合性レビューを担う読み取り専用エージェント。機能単位（全タスク完了・format/lint/test合格後）に1回起動
- `code-quality-reviewer`（`model: sonnet`）: コード品質レビューを担う読み取り専用エージェント。spec-reviewer合格後に機能単位で起動
- 一過性の単純作業（単純な検索・機械的な置換など）は都度Agentツールで`haiku`を呼ぶ

メインセッションはフェーズ進行管理・ユーザー確認・`plan-mode`呼び出しに集中する。
調査・実装・レビューのコンテキストはサブエージェントに閉じ込め、メイン側のコンテキスト汚染を避ける。

レビュアーは直列で起動し、`format/lint/test` → `spec-reviewer` → `code-quality-reviewer`の順序を固定する。
品質レビューは仕様適合性を前提に成立するため、順序を逆転させない。
指摘があれば`spec-implementer`へ差し戻して修正後に再レビューし、両レビュアーが合格を返すまでループする。

## 参照コメント方針

コードから恒久ドキュメントへの参照コメントは最小限に絞る。
過剰な参照はコード可読性を損ねるため。

- 対象: 新規モジュールの冒頭、および大きめの新規クラス・関数の先頭
- 形式: 1行で恒久`.md`のみを指す（例: `# 仕様: docs/features/ユーザー認証.md`）
- 最終形は恒常配置のパス（`docs/features/`・`docs/topics/`）を指す。参照先の安定性を保つため
- 開発中は開発中配置のパス（`docs/v{next}/{機能名}.md`）を指してよい。`spec-driven-promote`スキルによる昇格時に、恒常配置のパスへ一括書き換える
- 一時`.working.md`は参照しない（実装完了時に削除されるためリンク切れを招く）

既存ファイルへの小さな追加や、恒久`.md`で直接触れられていない細部には参照コメントを書かない。

言語別の具体的な書式例は`references/templates.md`を参照する。

## Gotchas

- `.working.md`の削除漏れ: Cleanupフェーズで`references/operations.md`の導入チェックリストを使い必ず確認する
- `README.md`の更新漏れ: 機能や横断ドキュメントを追加したのに`docs/v{next}/README.md`の一覧へ反映し忘れると入口が陳腐化する。Cleanupフェーズで必ず確認する
- 恒久ドキュメント間の矛盾検出: 機能間・横断間・`README.md`との相互矛盾は`spec-reviewer`の責務拡張で検査する。メイン側で見落としてもレビュアーで捕捉する前提だが、Cleanup手順の順序を守り最終反映後にレビュアーへ渡す
- 昇格時の参照コメント書き換え漏れ: 開発中配置のパスを指す参照コメントは`spec-driven-promote`スキルの実行時に恒常配置のパスへ一括書き換える。書き換え漏れは昇格後のリンク切れを招くため、昇格スキル内の検証手順を必ず踏む
- 既存改修時の恒常配置の直接編集: 改修作業は恒常配置の該当`.md`を開発中ディレクトリへコピーしてから行い、恒常配置を直接編集しない（昇格前に恒常配置と実装が乖離するのを避けるため）
- 計画ファイルと恒久`.md`の情報重複: 計画ファイルはcodex向け照合材料、恒久`.md`は将来の人間向け参照資料、という関心の違いを意識する
- `plan-mode`のcodexレビュー指示文の再記述: 本スキル側で書き直さず`plan-mode`へ委譲する（保守性確保のため）
- 次期バージョン番号の推測: Intakeフェーズでユーザーに確認してから確定する。プロジェクトの既存命名規則と整合させる
