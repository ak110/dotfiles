# テンプレート集

spec-drivenスキルで使う恒久ドキュメント（機能・横断・`README.md`）・一時ドキュメント・参照コメントの書式テンプレート。
参照コメントの配置方針などはSKILL.md本体を参照する（本ファイルは書式のみ提示する）。

恒常配置（`docs/features/`・`docs/topics/`）の最終形と開発中配置（`docs/v{next}/`配下）の一時形は同一テンプレートを流用する。
昇格（`spec-driven-promote`スキル）時にファイルを移動するだけで運用が接続するため、両者の構成を揃えておく。
本ファイル中の具体パス表記は既定値であり、プロジェクトのドキュメントで別の配置が規定されている場合はそちらに読み替える（`spec-driven`スキルSKILL.mdの「配置の既定と上書き」節を参照）。

## 機能ドキュメント（`{機能名}.md`）

実装後も残し、将来の類似改修時に参照される。
コードを見ただけでは分からない情報（なぜその設計を採ったか・却下した代替案）を中心に書く。

````markdown
# {機能名}

## 背景

（なぜこの機能を追加するか。ビジネス要求・ユーザー要望・既存の課題など）

## スコープ

（本機能で扱う範囲と扱わない範囲）

### 含むもの

-
-

### 含まないもの

-
-

## 受け入れ基準

（機能が完成したと判断できる客観的な基準）

-
-

## 主要設計判断

（設計で確定した主要な方針と理由。将来の類似改修時に参照される核心情報）

### {判断項目1}

- 決定内容:
- 理由:

### {判断項目2}

- 決定内容:
- 理由:

## 却下した代替案

（検討したが採用しなかった代替案と却下理由。同じ議論の再発を防ぐため）

- 代替案A — 却下理由:
- 代替案B — 却下理由:
````

## 横断ドキュメント（`{トピック名}.md`）

複数機能で参照される設計判断・ガイドライン・共通用語をまとめる。
機能ドキュメントのテンプレートを流用し、以下の差分だけ意識する。

- 受け入れ基準は機能単位でないため省略してよい
- 「スコープ」は対象範囲（どの機能から参照されるか・どういう場面で参照するか）に読み替える
- 「背景」には横断化した経緯（どの機能間で共通と判断したか）を書くと再利用しやすい

## バージョン概要（`README.md`）

`docs/v{next}/README.md`に置く次期バージョン全体の入口。
次期リリース作業期間中のみ存在し、昇格時に役目を終える（開発中バージョンディレクトリごと削除される）。
ステータス列は含めない（進行状況の管理は`.working.md`・TaskCreateに委ね、恒久ドキュメントに陳腐化する情報を載せないため）。

````markdown
# v{next}

## バージョン目的

（このバージョンで実現したい目的・テーマ）

## 機能一覧

（本バージョンで追加・変更する機能。各機能ドキュメントへのリンクと一行説明）

- [docs/v{next}/{機能名1}.md]({機能名1}.md) — 一行説明
- [docs/v{next}/{機能名2}.md]({機能名2}.md) — 一行説明

## 横断ドキュメント一覧

（複数機能で参照される設計判断・ガイドライン・共通用語）

- [docs/v{next}/{トピック名1}.md]({トピック名1}.md) — 一行説明
- [docs/v{next}/{トピック名2}.md]({トピック名2}.md) — 一行説明
````

## レビュアー呼び出し（spec-reviewer・code-quality-reviewer）

Phase 5で機能単位に直列起動するレビュアーの呼び出しテンプレート。
両レビュアーに共通する引数は「機能名・開発中`.md`のパス・`BASE_SHA`・レビュー対象外の一時ファイル一覧」の4点。
`spec-reviewer`には加えてドキュメント整合性検査のため差分に含まれる他の恒久ドキュメントと、既存改修時は改修前の該当節スナップショットを渡す。

`BASE_SHA`はPhase 5開始時点の`git rev-parse HEAD`の結果を使う。
spec-implementerはタスク単位でコミットしない設計のため、レビュー時点の差分は未コミットの作業ツリー差分として扱う。
差分取得は`git diff {BASE_SHA}`と未追跡ファイルを含む作業ツリー参照（`git status`・`Read`）で行う。
`docs/v{next}/{機能名}.working.md`などの一時ファイルは実装完了時に削除される想定のため、評価対象外として除外する。

### spec-reviewer

```text
以下の機能について仕様適合性・ドキュメント整合性レビューを実施してください。

機能名: {機能名}
開発中ドキュメント: `docs/v{next}/{機能名}.md`
差分内の他の恒久ドキュメント: {同一開発中ディレクトリ配下の他機能・横断・README.mdのうち差分に含まれるもの}
恒常配置側の該当ドキュメント: `docs/features/{機能名}.md` または `docs/topics/{トピック名}.md`（既存改修時のみ）
改修前の該当節スナップショット: `docs/v{next}/{機能名}.working.md`内の該当節（既存改修時のみ）
BASE_SHA: {Phase 5開始時点のHEAD}
対象外ファイル: docs/v{next}/{機能名}.working.md

差分取得: `git diff {BASE_SHA}`と、未追跡ファイルを含む作業ツリー参照で行うこと
制約: 読み取り専用。実装者レポートは鵜呑みにせず、コードと差分で独立検証すること
出力: 仕様適合（✅）または指摘（❌、file:line形式）
```

### code-quality-reviewer

```text
以下の機能についてコード品質レビューを実施してください。

機能名: {機能名}
開発中ドキュメント: `docs/v{next}/{機能名}.md`
BASE_SHA: {Phase 5開始時点のHEAD}
対象外ファイル: docs/v{next}/{機能名}.working.md

差分取得: `git diff {BASE_SHA}`と、未追跡ファイルを含む作業ツリー参照で行うこと
制約: 読み取り専用。coding-standardsスキルを事前に呼び出し、品質基準に従うこと
出力: Strengths / Issues (Critical/Important/Minor) / Assessment (approve/reject)
```
