# 3ステップワークフロー詳細

軽量SDDの各ステップで踏む手順とサブエージェント呼び出しテンプレートをまとめる。
新規追加フローを基準に記述し、末尾に「既存機能改修フロー」として差分のみをまとめる。
中断・再開や導入チェックリストなど通常フロー外の手順は`operations.md`を参照する。
リリース完了後の昇格作業は`spec-driven-promote`スキル、既存プロジェクトへの初回導入は`spec-driven-init`スキルを参照する。
本ファイル中の具体パス表記は既定値であり、プロジェクトのドキュメントで別の配置が規定されている場合はそちらに読み替える（`spec-driven`スキルSKILL.mdの「配置の既定と上書き」節を参照）。

## 全体の流れ

1. Setup&Explore（plan mode前）— 要件確定・ドキュメント骨格作成・恒常配置および開発中配置の確認・既存機能調査
2. Plan&Implement（plan mode 1回出入り）— 計画ファイル作成（大枠＋実装詳細）・codexレビュー合格・ExitPlanMode後の判断経緯転記と実装
3. Cleanup（plan mode後）— 検証・レビュー・一時ファイル削除・コミット

ステップ終端にユーザー確認ゲートは置かない。判断が必要な場面は都度ユーザーに確認する。
ステップ進行中の状態は`.working.md`の「現在のステップ」節で管理する。

## ステップ1: Setup&Explore

目的: 要件・作業テーマ・次期バージョンを確定し、ドキュメント骨格を作成し、恒常配置・開発中配置および既存コードベースを十分に確認して、ステップ2のインプットを整える。

plan modeに入る前の準備工程である。本ステップを十分に行うことで、plan mode内での既存ドキュメント参照漏れを防ぐ。

手順:

1. 以下をユーザーに確認する
   - 作業テーマ名（日本語の自然な表現。今回の変更を端的に表す。例: `SSO追加`）
   - 新規追加か既存機能改修かの区分
   - 機能の目的・想定ユーザー・成功条件
   - 次期リリースのバージョン番号（プロジェクトの命名規則に合わせる）
   - 既存改修の場合、対応する恒常配置ファイル（`docs/features/{機能名}.md`または`docs/topics/{トピック名}.md`）を特定する
2. 恒常配置と開発中配置を確認する
   - 恒常配置: `docs/features/`・`docs/topics/`配下の関連ファイルを読む
   - 開発中配置: `docs/v{next}/`配下に並行作業中の作業版があれば読む
   - 確認結果（関連する既存記述・想定される矛盾点など）を記憶してステップ2のインプットにする
3. 開発中バージョンディレクトリに作業版`.md`を作成する
   - パス: `docs/v{next}/{作業テーマ名}.md`
   - `templates.md`の作業テーマテンプレートで骨子を作成し、背景・スコープ・受け入れ基準の初版を埋める
   - 冒頭の「恒常配置の対応先」節に、昇格時のマージ対象ファイル（例: `docs/features/認証.md`）を明示する。新規機能で該当先がない場合はその旨記載する
4. 一時ドキュメント初版を作成する
   - パス: `docs/v{next}/{作業テーマ名}.working.md`
   - `templates.md`の一時テンプレートを流用し、空の調査メモ欄・判断経緯欄を用意する
   - 既存改修の場合は「改修前の該当節スナップショット」を併記する領域を用意し、Before/Afterの整合を後続のレビューで検証できるようにする
5. `README.md`を整備する
   - パス: `docs/v{next}/README.md`
   - 未作成なら`templates.md`の`README.md`テンプレートで新規作成する
   - 作業テーマ一覧節に本作業テーマのエントリ（`{作業テーマ名}.md`へのリンクと一行説明）を追加する
6. 既存機能・依存関係・影響範囲を調査する
   - 調査観点を列挙する（関連する既存モジュール・テスト・ドキュメント・外部API・設定など）
   - 独立した観点ごとに`spec-researcher`エージェントを並列起動する
   - 結果を`.working.md`の「既存調査メモ」節に集約する
   - 想定外の制約や方針変更が必要な点を洗い出し、必要なら開発中`.md`の「スコープ」を更新する

spec-researcher呼び出しテンプレート:

```text
以下の観点で既存コードベースを調査してください。

対象: {リポジトリパスまたはモジュール名}
観点: {具体的な調査観点}
恒常配置: `docs/features/`・`docs/topics/`配下の関連ファイルも確認対象に含める
開発中配置: `docs/v{next}/`配下に並行作業中の作業版があれば確認対象に含める
出力形式: 構造化したMarkdown。ファイルパスとfile:line参照を必ず含めること
制約: 読み取り専用。コード変更は行わないこと
```

## ステップ2: Plan&Implement

目的: plan modeへ1回出入りして計画ファイル1本（大枠方針と実装詳細を集約）を作成し、codexレビューを通して合意を得る。`ExitPlanMode`後に判断経緯を開発中ドキュメントへ転記し、タスク分解して実装する。

plan modeの編集制約下でも、計画ファイル作成に必要なcodexレビューは`plan-mode`スキルの指示に従って実行する（同スキル内で制約対象外として扱う）。

### 2a plan mode内（計画ファイル作成）

1. `EnterPlanMode`でplan modeに入る
2. `plan-mode`スキルを呼び出す（内部呼び出し）
3. 計画ファイル（`~/.claude/plans/{自動生成ファイル名}.md`）に大枠方針・主要設計判断・実装詳細（file:line・変更内容）を1本にまとめる
4. `plan-mode`の手順に従ってcodexレビューを受ける
5. codexの指摘がなくなるまで計画ファイルを更新する
6. 判断経緯（codex指摘への対応・却下した代替案・ユーザー指示で確定した方針）が計画ファイルの`## 却下した代替案`・`## 変更履歴`節に記録されていることを確認する
7. `ExitPlanMode`で合意ゲートを通過する

### 2b plan mode外（判断経緯の転記と実装）

`ExitPlanMode`直後に、実装着手前に必ず以下を行う。

1. 計画ファイルの判断経緯を開発中ドキュメントへ転記する
   - 将来の類似改修でも参照される要点を、開発中`.md`の「主要設計判断」「却下した代替案」節へ反映する
   - 計画固有の一過性情報（codex指摘の個別対応履歴など）は`.working.md`の「判断経緯」節へ蓄積する
   - 両者の重複を避け、恒久情報は開発中`.md`、経緯情報は`.working.md`に分ける
2. 複数機能で参照される設計判断・ガイドライン・共通用語が現れた場合は、横断ドキュメント（`docs/v{next}/{トピック名}.md`）へ切り出す（判定基準は`spec-driven`SKILL.mdの「横断ドキュメントの判定」節を参照）
3. 横断ドキュメントを追加した場合は`docs/v{next}/README.md`の横断ドキュメント一覧にエントリを追加する
4. タスク分解とTaskCreate登録
   - 計画ファイルの「変更内容」節を起点にタスクを分解する
   - `.working.md`の「タスク分解」節にタスク一覧を記載する（spec-implementerへの指示がそのまま書けるレベル）
   - TaskCreateで各タスクを登録する（検証・コミットタスクも含める）
5. 実装着手直前のHEADを`BASE_SHA`として記録する（`git rev-parse HEAD`）。ステップ3のレビュアー呼び出しで差分起点として参照する
6. 実装
   - TaskCreateの未完了タスクを先頭から1つずつ取り出す
   - 各タスクを`spec-implementer`エージェントに委譲する
   - 実装結果を確認し、TaskUpdateで完了にする

タスク粒度の目安:

- 1タスクあたり1〜3ファイル程度の変更
- 1タスク内で検証が完結するよう、ファイル横断の依存を避ける
- テストコードの追加は対応する実装タスクと同一タスクにまとめる

spec-implementer呼び出しテンプレート:

```text
以下のタスクを実装してください。

タスク: {タスクの具体的な記述}
参照: `docs/v{next}/{作業テーマ名}.md` の該当節、ステップ2で作成した計画ファイル（`~/.claude/plans/{自動生成ファイル名}.md`）
制約: coding-standardsスキルを事前に呼び出し、品質基準に従うこと
完了条件: 変更後にプロジェクトのformat/lint/testが全て通ること
```

## ステップ3: Cleanup

目的: 全タスク完了後に検証・最終反映・レビュー・一時ファイル削除・コミットを行う。

手順:

1. プロジェクトのformat/lint/test（`coding-standards`で示される手順）を実行する
2. 開発中`.md`へ最終状態を反映する
   - 受け入れ基準の確認結果・実装を経て確定した事項を反映する
   - 既存改修の場合はBefore/Afterの意図的差分がドキュメントに正しく表現されているか確認する
   - 計画ファイルからの判断経緯転記が漏れていないか再確認する
3. `docs/v{next}/README.md`の作業テーマ一覧・横断ドキュメント一覧が最新化されているか確認する（ステップ2で追加した横断ドキュメントの反映漏れに注意）
4. 新規追加した参照コメントの指し先（開発中配置のパス）が実在し、内容が関連していることを確認する
5. `spec-reviewer`エージェントに仕様適合性・ドキュメント整合性レビューを委譲する
   - 引数は以下の通り
     - 作業テーマ名
     - 開発中`.md`のパス
     - 差分に含まれる他の恒久ドキュメント
     - 恒常配置側の該当ドキュメント（既存改修時のみ）
     - 改修前の該当節スナップショット（既存改修時のみ、`.working.md`内）
     - 計画ファイルのパス
     - `BASE_SHA`
     - レビュー対象外の一時ファイル一覧
   - 「差分に含まれる他の恒久ドキュメント」には、同一開発中ディレクトリ配下の他作業テーマ・横断・`README.md`のうち差分に含まれるものを列挙する
   - 指摘があれば該当タスクを`spec-implementer`へ差し戻して再実装し、最終反映（手順2・3）も含めて再実施した上で、再度format/lint/test・spec-reviewerのループを回す
   - `✅ 仕様適合`が返るまで次ステップへ進まない
6. `code-quality-reviewer`エージェントにコード品質レビューを委譲する
   - 引数: 作業テーマ名・開発中`.md`のパス・`BASE_SHA`・対象外一時ファイル
   - 指摘があれば`spec-implementer`へ差し戻して再実装し、最終反映も含めて再実施した上で、再度format/lint/test・spec-reviewer・code-quality-reviewerのループを回す
   - `Assessment: approve`が返るまで次ステップへ進まない
   - 指摘が繰り返される場合（同種の指摘が2回以上続くなど）はループを中断してユーザーへエスカレーションする
7. 一時`.working.md`を削除する
8. 全差分を1コミットまたは意味単位で複数コミットにまとめる

手順2・3を`spec-reviewer`実行前に置くのは、最終版のドキュメントと実装を一括で整合性検査するため。
恒常配置への反映は本ステップでは行わない。リリース完了後に`spec-driven-promote`スキルで実施する。

spec-reviewer・code-quality-reviewer呼び出しテンプレートは`templates.md`を参照する。

## 既存機能改修フロー

既存機能の改修（仕様変更・既存設計判断の見直し・スコープ縮小など）は、新規追加フローと共通の3ステップを踏む。
以下は新規追加フローとの差分のみ記述する。明記のない手順は新規追加フローと同じ。

### ステップ1差分

- 対応する恒常配置ファイル（`docs/features/{機能名}.md`または`docs/topics/{トピック名}.md`）をユーザーと合意で特定する
- 開発中`.md`は作業テーマ単位で新規作成する（恒常配置からのコピーはしない）。冒頭の「恒常配置の対応先」節に対応先を明示する
- `.working.md`に「改修前の該当節スナップショット」を併記し、Before/Afterの整合を後続のレビューで検証できるようにする
- 当該機能が他機能・横断ドキュメントから参照されているかを`docs/features/`・`docs/topics/`配下の全文検索で確認し、波及する記述の更新要否を洗い出す

### ステップ2差分

- 計画ファイル作成時、恒常配置と開発中作業テーマの粒度差を前提に変更内容を記述する
- 判断経緯転記では、「却下した代替案」は既存エントリを維持し追記のみ行う（過去の判断を消さず、改修判断の履歴を残す）
- 「主要設計判断」は現行判断を上書きしてよいが、変更理由と旧判断との差分を同節内に明記する（昇格後に読む人向けの情報）

### ステップ3差分

- `spec-reviewer`へ「改修前の該当節スナップショット」も渡し、Before/Afterの意図的差分と意図しない齟齬を区別させる
- 既存実装のリネームや削除が伴う場合、関連するテスト・参照コメント・ドキュメント間リンクの追随を確認する
