# 5フェーズワークフロー詳細

軽量SDDの各フェーズで踏む手順・ユーザー確認ゲート文例・サブエージェント呼び出しテンプレートをまとめる。
新規追加フローを基準に記述し、末尾に「既存機能改修フロー」として差分のみをまとめる。
中断・再開や導入チェックリストなど通常フロー外の手順は`operations.md`を参照する。
リリース完了後の昇格作業は`spec-driven-promote`スキル、既存プロジェクトへの初回導入は`spec-driven-init`スキルを参照する。

## 全体の流れ

1. Intake — 要件聞き取り
2. Explore — 既存調査（spec-researcher）
3. Design — 3サブフェーズで設計確定
   - 3a大枠plan（plan mode内）— 骨格の計画ファイル作成・codexレビュー・大枠合意
   - 3b具体化（plan mode外）— 恒久`.md`の詳細化・必要なら横断ドキュメント切り出し
   - 3c実装plan（plan mode内）— 実装計画ファイル作成・codexレビュー・実装合意
4. Tasks — タスク分解・登録
5. Implement＆Cleanup — 実装（spec-implementer）・検証・削除・コミット

各フェーズ終端でユーザー確認ゲートを通過してから次フェーズへ進む。
ゲート前に一時`.working.md`の「現在のフェーズ」を更新し、中断に備える。Designでは`3a`・`3b`・`3c`まで記録する。

## Phase 1: Intake

目的: 要件・機能名・次期バージョンを確定し、ドキュメント骨格を作成する。

手順:

1. 以下をユーザーに確認する
   - 機能名（日本語の自然な表現）
   - 新規追加か既存機能改修かの区分（既存改修なら恒常配置の該当`.md`を特定する）
   - 機能の目的・想定ユーザー・成功条件
   - 次期リリースのバージョン番号（プロジェクトの命名規則に合わせる）
2. 開発中バージョンディレクトリに作業版`.md`を用意する
   - 新規追加: `docs/v{next}/{機能名}.md`に`templates.md`の機能ドキュメントテンプレートで骨子を作成し、背景・スコープ・受け入れ基準の初版を埋める
   - 既存改修: 恒常配置の`docs/features/{機能名}.md`または`docs/topics/{トピック名}.md`を`docs/v{next}/{機能名}.md`へコピーしてから編集を開始する
3. 一時ドキュメント初版を作成する
   - パス: `docs/v{next}/{機能名}.working.md`
   - `templates.md`の一時テンプレートを流用し、空の調査メモ欄を用意する
   - 既存改修の場合は「改修前の該当節スナップショット」を併記する領域を用意し、Before/Afterの整合を後続のレビューで検証できるようにする
4. `README.md`を整備する
   - パス: `docs/v{next}/README.md`
   - 未作成なら`templates.md`の`README.md`テンプレートで新規作成する
   - 機能一覧節に本機能のエントリ（`{機能名}.md`へのリンクと一行説明）を追加する
5. ユーザー確認ゲートを通過する

ゲート文例:

> 機能名・バージョン・背景・スコープを以下で確定し`README.md`へエントリを追加しました。Exploreフェーズ（既存機能調査）に進んでよいですか。修正があれば指摘してください。

## Phase 2: Explore

目的: 既存機能・依存関係・影響範囲を調査し、設計判断の材料を集める。

手順:

1. 調査観点を列挙する（関連する既存モジュール・テスト・ドキュメント・外部API・設定など）
2. 常時参照対象として以下をすべて走査する
   - 恒常配置: `docs/features/`・`docs/topics/`配下の全ファイル。関連する既存機能の特定はここを起点にする
   - 開発中配置: `docs/v{next}/`配下の作業版`.md`（存在する場合）。進行中の関連仕様変更はここから取得する
3. 独立した観点ごとに`spec-researcher`エージェントを並列起動する
4. 結果を`.working.md`の「既存調査メモ」節に集約する
5. 想定外の制約や方針変更が必要な点を洗い出し、必要なら開発中`.md`の「スコープ」を更新する
6. ユーザー確認ゲートを通過する

spec-researcher呼び出しテンプレート:

```text
以下の観点で既存コードベースを調査してください。

対象: {リポジトリパスまたはモジュール名}
観点: {具体的な調査観点}
出力形式: 構造化したMarkdown。ファイルパスとfile:line参照を必ず含めること
制約: 読み取り専用。コード変更は行わないこと
```

ゲート文例:

> 既存機能の調査が完了し`.working.md`に集約しました。Designフェーズ（設計・計画ファイル作成）に進んでよいですか。

## Phase 3: Design

目的: 設計を3サブフェーズで確定する。plan modeを大枠合意（3a）と実装合意（3c）の2つのゲートとして使い、その間の3bで恒久`.md`の具体化を行う。

plan modeでは計画ファイル以外を編集できない制約があるため、開発中`.md`の更新は3b（plan mode外）で行う。
3aと3cで計画ファイルは2本生成するが、いずれも`~/.claude/plans/`配下に残置する（Claude Code標準の挙動に揃える）。

### 3a 大枠plan（plan mode内）

1. `plan-mode`スキルを呼び出す（内部呼び出し）
2. 計画ファイル（`~/.claude/plans/{自動生成ファイル名}.md`）に大枠方針と主要設計判断の骨格を書く
3. `plan-mode`の手順に従ってcodexレビューを受ける
4. codexの指摘がなくなるまで計画ファイルを更新する
5. `ExitPlanMode`で大枠合意ゲートを通過する

ゲート文例（3a末尾）:

> 大枠方針と主要設計判断の骨格を計画ファイルに書き、codexレビューが通りました。この方針で具体化（3b）に進んでよいですか。

### 3b 具体化（plan mode外）

1. 3aで合意した大枠を元に、開発中`.md`の「主要設計判断」「却下した代替案」節を具体化する
2. 複数機能で参照される設計判断・ガイドライン・共通用語が現れた場合は、横断ドキュメント（`docs/v{next}/{トピック名}.md`）へ切り出す（判定基準はSKILL.mdの「横断ドキュメントの判定」節を参照）
3. 横断ドキュメントを追加した場合は`docs/v{next}/README.md`の横断ドキュメント一覧にエントリを追加する
4. 計画ファイルと開発中`.md`の両方に同じ情報を書かない
5. ユーザー確認ゲートを通過する

ゲート文例（3b末尾）:

> 開発中`.md`の具体化と横断ドキュメントの整理が終わりました。実装計画（3c）に進んでよいですか。

### 3c 実装plan（plan mode内）

1. メインが`EnterPlanMode`ツールで再突入する
2. 新規計画ファイル（`~/.claude/plans/{自動生成ファイル名}.md`）に実装詳細（file:line・変更内容）を書く
3. `plan-mode`の手順に従ってcodexレビューを受ける
4. codexの指摘がなくなるまで計画ファイルを更新する
5. `ExitPlanMode`で実装合意ゲートを通過する

ゲート文例（3c末尾）:

> 実装計画ファイルのcodexレビューが通りました。Tasksフェーズ（タスク分解）に進んでよいですか。

## Phase 4: Tasks

目的: 実装可能な粒度のタスクに分解し、TaskCreateへ登録する。

手順:

1. 計画ファイルの「変更内容」節を起点にタスクを分解する
2. `.working.md`の「タスク分解」節にタスク一覧を記載する（spec-implementerへの指示がそのまま書けるレベル）
3. TaskCreateで各タスクを登録する（検証・コミットタスクも含める）
4. ユーザー確認ゲートを通過する

タスク粒度の目安:

- 1タスクあたり1〜3ファイル程度の変更
- 1タスク内で検証が完結するよう、ファイル横断の依存を避ける
- テストコードの追加は対応する実装タスクと同一タスクにまとめる

ゲート文例:

> タスク分解を完了しTaskCreateへ登録しました。Implementフェーズ（実装とクリーンアップ）に進んでよいですか。

## Phase 5: Implement＆Cleanup

目的: タスクを順次実装し、最終反映・検証・レビュー・一時ファイル削除・コミットを完了する。

手順:

1. フェーズ開始時点のHEADを`BASE_SHA`として記録する（`git rev-parse HEAD`）。以降のレビュアー呼び出しで差分起点として参照する
2. TaskCreateの未完了タスクを先頭から1つずつ取り出す
3. 各タスクを`spec-implementer`エージェントに委譲する
4. 実装結果を確認し、TaskUpdateで完了にする
5. 全タスク完了後、プロジェクトのformat/lint/test（`coding-standards`で示される手順）を実行する
6. 開発中`.md`へ最終状態を反映する（受け入れ基準の確認結果・実装を経て確定した事項など）。既存改修の場合はBefore/Afterの意図的差分がドキュメントに正しく表現されているか確認する
7. `docs/v{next}/README.md`の機能一覧・横断ドキュメント一覧が最新化されているか確認する（3bで追加した横断ドキュメントの反映漏れに注意）
8. 新規追加した参照コメントの指し先（開発中配置のパス）が実在し、内容が関連していることを確認する
9. `spec-reviewer`エージェントに仕様適合性・ドキュメント整合性レビューを委譲する
   - 引数: 機能名・開発中`.md`のパス・差分に含まれる他の恒久ドキュメント・恒常配置側の該当ドキュメント（既存改修時）・改修前の該当節スナップショット（既存改修時、`.working.md`内）・`BASE_SHA`・レビュー対象外の一時ファイル一覧
   - 「差分に含まれる他の恒久ドキュメント」には、同一開発中ディレクトリ配下の他機能・横断・`README.md`のうち差分に含まれるものを列挙する
   - 指摘があれば該当タスクを`spec-implementer`へ差し戻して再実装し、最終反映（手順6・7）も含めて再実施した上で、再度format/lint/test・spec-reviewerのループを回す
   - `✅ 仕様適合`が返るまで次ステップへ進まない
10. `code-quality-reviewer`エージェントにコード品質レビューを委譲する
    - 引数: 機能名・開発中`.md`のパス・`BASE_SHA`・対象外一時ファイル
    - 指摘があれば`spec-implementer`へ差し戻して再実装し、最終反映も含めて再実施した上で、再度format/lint/test・spec-reviewer・code-quality-reviewerのループを回す
    - `Assessment: approve`が返るまで次ステップへ進まない
    - 指摘が繰り返される場合（同種の指摘が2回以上続くなど）はループを中断してユーザーへエスカレーションする
11. 一時`.working.md`を削除する
12. 全差分を1コミットまたは意味単位で複数コミットにまとめる
13. ユーザー確認ゲートを通過する

手順6・7を`spec-reviewer`実行前に置くのは、最終版のドキュメントと実装を一括で整合性検査するため。
恒常配置への反映は本フェーズでは行わない。リリース完了後に`spec-driven-promote`スキルで実施する。

spec-implementer呼び出しテンプレート:

```text
以下のタスクを実装してください。

タスク: {タスクの具体的な記述}
参照: `docs/v{next}/{機能名}.md` の該当節、3cで作成した実装計画ファイル（`~/.claude/plans/{自動生成ファイル名}.md`）
制約: coding-standardsスキルを事前に呼び出し、品質基準に従うこと
完了条件: 変更後にプロジェクトのformat/lint/testが全て通ること
```

spec-reviewer・code-quality-reviewer呼び出しテンプレートは`templates.md`を参照する。

ゲート文例:

> 実装・最終反映・両レビュー・検証が完了し、`.working.md`を削除しました。コミットしてよいですか。

## 既存機能改修フロー

既存機能の改修（仕様変更・既存設計判断の見直し・スコープ縮小など）は、新規追加フローと共通の5フェーズを踏む。
以下は新規追加フローとの差分のみ記述する。明記のない手順は新規追加フローと同じ。

### Intake差分

- 恒常配置の該当`.md`（`docs/features/{機能名}.md`または`docs/topics/{トピック名}.md`）を`docs/v{next}/{機能名}.md`へコピーしてから編集を開始する
- `.working.md`には「改修前の該当節スナップショット」を併記し、Before/Afterの整合を後続のレビューで検証できるようにする
- 昇格時に恒常配置を上書きするため、ファイル名と構成は恒常配置と揃える

### Explore差分

- 恒常配置と開発中配置（本機能以外で並行作業中のものがあればそれも）の両方を調査対象とする
- 当該機能が他機能・横断ドキュメントから参照されているかを`docs/features/`・`docs/topics/`配下の全文検索で確認し、波及する記述の更新要否を洗い出す

### Design 3b差分

- 「却下した代替案」は既存エントリを維持し追記のみ行う（過去の判断を消さず、改修判断の履歴を残す）
- 「主要設計判断」は現行判断を上書きしてよいが、変更理由と旧判断との差分を同節内に明記する（昇格後に読む人向けの情報）

### Implement差分

- `spec-reviewer`へ「改修前の該当節スナップショット」も渡し、Before/Afterの意図的差分と意図しない齟齬を区別させる
- 既存実装のリネームや削除が伴う場合、関連するテスト・参照コメント・ドキュメント間リンクの追随を確認する
