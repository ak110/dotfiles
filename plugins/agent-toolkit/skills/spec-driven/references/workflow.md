# 5フェーズワークフロー詳細

軽量SDDの各フェーズで踏む手順・ユーザー確認ゲート文例・サブエージェント呼び出しテンプレートをまとめる。
中断・再開や導入チェックリストなど通常フロー外の手順は`operations.md`を参照する。

## 全体の流れ

1. Intake — 要件聞き取り
2. Explore — 既存調査（spec-researcher）
3. Design — 3サブフェーズで設計確定
   - 3a大枠plan（plan mode内）— 骨格の計画ファイル作成・codexレビュー・大枠合意
   - 3b具体化（plan mode外）— 恒久`.md`の詳細化・必要なら横断ドキュメント切り出し
   - 3c実装plan（plan mode内）— 実装計画ファイル作成・codexレビュー・実装合意
4. Tasks — タスク分解・登録
5. Implement＆Cleanup — 実装（spec-implementer）・検証・削除・コミット

各フェーズ終端でユーザー確認ゲートを通過してから次フェーズへ進む。
ゲート前に一時`.working.md`の「現在のフェーズ」を更新し、中断に備える。Designでは`3a`・`3b`・`3c`まで記録する。

## Phase 1: Intake

目的: 要件・機能名・次期バージョンを確定し、ドキュメント骨格を作成する。

手順:

1. 以下をユーザーに確認する
   - 機能名（日本語の自然な表現）
   - 機能の目的・想定ユーザー・成功条件
   - 次期リリースのバージョン番号（プロジェクトの命名規則に合わせる）
2. 恒久ドキュメント骨子を作成する
   - パス: `docs/v{version}/{機能名}.md`
   - `templates.md`の恒久テンプレートを流用し、背景・スコープ・受け入れ基準の初版を埋める
3. 一時ドキュメント初版を作成する
   - パス: `docs/v{version}/{機能名}.working.md`
   - `templates.md`の一時テンプレートを流用し、空の調査メモ欄を用意する
4. `README.md`を整備する
   - パス: `docs/v{version}/README.md`
   - 未作成なら`templates.md`の`README.md`テンプレートで新規作成する
   - 機能一覧節に本機能のエントリ（`{機能名}.md`へのリンクと一行説明）を追加する
5. ユーザー確認ゲートを通過する

ゲート文例:

> 機能名・バージョン・背景・スコープを以下で確定し`README.md`へエントリを追加しました。Exploreフェーズ（既存機能調査）に進んでよいですか。修正があれば指摘してください。

## Phase 2: Explore

目的: 既存機能・依存関係・影響範囲を調査し、設計判断の材料を集める。

手順:

1. 調査観点を列挙する（関連する既存モジュール・テスト・ドキュメント・外部API・設定など）
2. 独立した観点ごとに`spec-researcher`エージェントを並列起動する
3. 結果を`.working.md`の「既存調査メモ」節に集約する
4. 想定外の制約や方針変更が必要な点を洗い出し、必要なら恒久`.md`の「スコープ」を更新する
5. ユーザー確認ゲートを通過する

spec-researcher呼び出しテンプレート:

```text
以下の観点で既存コードベースを調査してください。

対象: {リポジトリパスまたはモジュール名}
観点: {具体的な調査観点}
出力形式: 構造化したMarkdown。ファイルパスとfile:line参照を必ず含めること
制約: 読み取り専用。コード変更は行わないこと
```

ゲート文例:

> 既存機能の調査が完了し`.working.md`に集約しました。Designフェーズ（設計・計画ファイル作成）に進んでよいですか。

## Phase 3: Design

目的: 設計を3サブフェーズで確定する。plan modeを大枠合意（3a）と実装合意（3c）の2つのゲートとして使い、その間の3bで恒久`.md`の具体化を行う。

plan modeでは計画ファイル以外を編集できない制約があるため、恒久`.md`の更新は3b（plan mode外）で行う。
3aと3cで計画ファイルは2本生成するが、いずれも`~/.claude/plans/`配下に残置する（Claude Code標準の挙動に揃える）。

### 3a 大枠plan（plan mode内）

1. `plan-mode`スキルを呼び出す（内部呼び出し）
2. 計画ファイル（`~/.claude/plans/{自動生成ファイル名}.md`）に大枠方針と主要設計判断の骨格を書く
3. `plan-mode`の手順に従ってcodexレビューを受ける
4. codexの指摘がなくなるまで計画ファイルを更新する
5. `ExitPlanMode`で大枠合意ゲートを通過する

ゲート文例（3a末尾）:

> 大枠方針と主要設計判断の骨格を計画ファイルに書き、codexレビューが通りました。この方針で具体化（3b）に進んでよいですか。

### 3b 具体化（plan mode外）

1. 3aで合意した大枠を元に、恒久`.md`の「主要設計判断」「却下した代替案」節を具体化する
2. 複数機能で参照される設計判断・ガイドライン・共通用語が現れた場合は、横断ドキュメント（`docs/v{version}/{トピック名}.md`）へ切り出す（判定基準はSKILL.mdの「横断ドキュメントの判定」節を参照）
3. 横断ドキュメントを追加した場合は`docs/v{version}/README.md`の横断ドキュメント一覧にエントリを追加する
4. 計画ファイルと恒久`.md`の両方に同じ情報を書かない
5. ユーザー確認ゲートを通過する

ゲート文例（3b末尾）:

> 恒久`.md`の具体化と横断ドキュメントの整理が終わりました。実装計画（3c）に進んでよいですか。

### 3c 実装plan（plan mode内）

1. メインが`EnterPlanMode`ツールで再突入する
2. 新規計画ファイル（`~/.claude/plans/{自動生成ファイル名}.md`）に実装詳細（file:line・変更内容）を書く
3. `plan-mode`の手順に従ってcodexレビューを受ける
4. codexの指摘がなくなるまで計画ファイルを更新する
5. `ExitPlanMode`で実装合意ゲートを通過する

ゲート文例（3c末尾）:

> 実装計画ファイルのcodexレビューが通りました。Tasksフェーズ（タスク分解）に進んでよいですか。

## Phase 4: Tasks

目的: 実装可能な粒度のタスクに分解し、TaskCreateへ登録する。

手順:

1. 計画ファイルの「変更内容」節を起点にタスクを分解する
2. `.working.md`の「タスク分解」節にタスク一覧を記載する（spec-implementerへの指示がそのまま書けるレベル）
3. TaskCreateで各タスクを登録する（検証・コミットタスクも含める）
4. ユーザー確認ゲートを通過する

タスク粒度の目安:

- 1タスクあたり1〜3ファイル程度の変更
- 1タスク内で検証が完結するよう、ファイル横断の依存を避ける
- テストコードの追加は対応する実装タスクと同一タスクにまとめる

ゲート文例:

> タスク分解を完了しTaskCreateへ登録しました。Implementフェーズ（実装とクリーンアップ）に進んでよいですか。

## Phase 5: Implement＆Cleanup

目的: タスクを順次実装し、検証・レビュー・一時ファイル削除・コミットを完了する。

手順:

1. フェーズ開始時点のHEADを`BASE_SHA`として記録する（`git rev-parse HEAD`）。以降のレビュアー呼び出しで差分起点として参照する
2. TaskCreateの未完了タスクを先頭から1つずつ取り出す
3. 各タスクを`spec-implementer`エージェントに委譲する
4. 実装結果を確認し、TaskUpdateで完了にする
5. 全タスク完了後、プロジェクトのformat/lint/test（`coding-standards`で示される手順）を実行する
6. `spec-reviewer`エージェントに仕様適合性レビューを委譲する
   - 引数: 機能名・恒久ドキュメントのパス・`BASE_SHA`・レビュー対象外の一時ファイル一覧（`.working.md`など）
   - 指摘があれば該当タスクを`spec-implementer`へ差し戻して再実装し、再度format/lint/test・spec-reviewerのループを回す
   - `✅ 仕様適合`が返るまで次ステップへ進まない
7. `code-quality-reviewer`エージェントにコード品質レビューを委譲する
   - 引数: spec-reviewerと同じ（機能名・恒久ドキュメントのパス・`BASE_SHA`・対象外一時ファイル）
   - 指摘があれば`spec-implementer`へ差し戻して再実装し、再度format/lint/test・spec-reviewer・code-quality-reviewerのループを回す
   - `Assessment: approve`が返るまで次ステップへ進まない
   - 指摘が繰り返される場合（同種の指摘が2回以上続くなど）はループを中断してユーザーへエスカレーションする
8. 一時`.working.md`を削除する
9. 恒久`.md`へ最終状態を反映する（受け入れ基準の確認結果など）
10. `docs/v{version}/README.md`の機能一覧・横断ドキュメント一覧が最新化されているか確認する（3bで追加した横断ドキュメントの反映漏れに注意）
11. 全差分を1コミットまたは意味単位で複数コミットにまとめる
12. ユーザー確認ゲートを通過する

spec-implementer呼び出しテンプレート:

```text
以下のタスクを実装してください。

タスク: {タスクの具体的な記述}
参照: `docs/v{version}/{機能名}.md` の該当節、3cで作成した実装計画ファイル（`~/.claude/plans/{自動生成ファイル名}.md`）
制約: coding-standardsスキルを事前に呼び出し、品質基準に従うこと
完了条件: 変更後にプロジェクトのformat/lint/testが全て通ること
```

spec-reviewer・code-quality-reviewer呼び出しテンプレートは`templates.md`を参照する。

ゲート文例:

> 実装・両レビュー・検証が完了し、`.working.md`を削除しました。コミットしてよいですか。
