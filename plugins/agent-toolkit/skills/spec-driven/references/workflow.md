# 3ステップワークフロー詳細

軽量SDDの各ステップで踏む手順とサブエージェント呼び出しテンプレートをまとめる。
新規追加フローを基準に記述し、末尾に「既存機能改修フロー」として差分のみをまとめる。
中断・再開は`operations.md`を参照する。
リリース完了後の昇格作業は`spec-driven-promote`スキル、既存プロジェクトへの初回導入は`spec-driven-init`スキルを参照する。

## 全体の流れ

1. Setup&Explore（plan mode前）— 要件確定・配置場所特定・既存機能の並列調査（リサーチ結果ファイル出力まで）
2. Plan&Implement（plan mode 1回出入り）— 計画ファイル作成（大枠＋実装詳細）・codexレビュー合格・ExitPlanMode後の設計ドキュメント立ち上げと実装
3. Cleanup（plan mode後）— 検証・レビュー・一時ファイル削除・コミット

ステップ終端にユーザー確認ゲートは置かない。判断が必要な場面は都度ユーザーに確認する。
ステップ進行中の状態はメインのTaskCreateで管理し、中断時はTaskCreate残タスクと開発中バージョンディレクトリ配下のファイル存在から再開ポイントを特定する。

## ステップ1: Setup&Explore

目的: 要件・作業テーマ・次期バージョンを確定し、恒常配置と開発中配置の場所を特定し、既存機能の並列調査結果をファイル出力として揃えて、ステップ2のインプットを整える。

plan modeに入る前の準備工程である。本ステップでは骨子・`README.md`エントリの生成は行わない（それらはステップ2後半で`spec-designer`が一括生成する）。
本ステップを十分に行うことで、plan mode内での既存ドキュメント参照漏れを防ぐ。

手順:

1. 以下をユーザーに確認する
   - 作業テーマ名（日本語の自然な表現。今回の変更を端的に表す。例: `SSO追加`）
   - 新規追加か既存機能改修かの区分
   - 機能の目的・想定ユーザー・成功条件
   - 次期リリースのバージョン番号（プロジェクトの命名規則に合わせる）
   - 既存改修の場合、対応する恒常配置ファイル（`docs/features/{機能名}.md`または`docs/topics/{トピック名}.md`）を特定する
2. 恒常配置と開発中配置の場所を特定する
   - 恒常配置: `docs/features/`・`docs/topics/`配下の関連ファイルのパスを押さえる（読み込みは`spec-researcher`へ委譲してよい）
   - 開発中配置: `docs/v{next}/`配下に並行作業中の作業版があればパスを押さえる
   - 開発中バージョンディレクトリ（`docs/v{next}/`）と一時ファイル置き場（`docs/v{next}/.cache/`）が未作成であれば、空ディレクトリのみ先行作成する。目的は`spec-researcher`のリサーチ結果ファイル書き込み先を用意するためで、骨子・`README.md`の生成はこの時点では行わない
3. 既存機能・依存関係・影響範囲を並列調査する
   - 調査観点を列挙する（関連する既存モジュール・テスト・ドキュメント・外部API・設定など）
   - 独立した観点ごとに`spec-researcher`エージェントを並列起動する。各起動に対して出力ファイルパス`docs/v{next}/.cache/{作業テーマ名}.research-{nn}.md`（`{nn}`はゼロパディング2桁の連番）を1つずつ割り当て、同一連番の重複割り当てを避ける
   - 各サブエージェントは構造化Markdownを出力ファイルへ書き出し、呼び元にはファイルパスと1〜2行サマリーのみを返す
   - メインは戻り値のファイルパスとサマリーを手元に保持するだけで、内容の集約・転記は行わない（集約はステップ2後半で`spec-designer`が担う）

spec-researcher呼び出しテンプレート:

```text
以下の観点で既存コードベースを調査してください。

対象: {リポジトリパスまたはモジュール名}
観点: {具体的な調査観点}
恒常配置: `docs/features/`・`docs/topics/`配下の関連ファイルも確認対象に含める
開発中配置: `docs/v{next}/`配下に並行作業中の作業版があれば確認対象に含める
出力ファイル: `docs/v{next}/.cache/{作業テーマ名}.research-{nn}.md`（このファイルへ構造化Markdownを書き出す）
出力形式: 観点ごとに節分けし、ファイルパスとfile:line参照を必ず含める
戻り値: 出力ファイルパスと1〜2行サマリー
制約: 書き込みは上記出力ファイル1つのみ。コード・他ドキュメントの変更は行わないこと
```

## ステップ2: Plan&Implement

目的: plan modeへ1回出入りして計画ファイル1本（大枠方針と実装詳細を集約）を作成し、codexレビューを通して合意を得る。`ExitPlanMode`後に判断経緯を作業版ドキュメントへ転記し、タスク分解して実装する。

plan modeの編集制約下でも、計画ファイル作成に必要なcodexレビューは`plan-mode`スキルの指示に従って実行する（同スキル内で制約対象外として扱う）。

### 2a plan mode内（計画ファイル作成）

1. `EnterPlanMode`ツールを呼び出し、plan modeへ入る
2. `plan-mode`スキルを呼び出す（内部呼び出し）
3. 計画ファイル（`~/.claude/plans/{自動生成ファイル名}.md`）に大枠方針・主要設計判断・実装詳細（file:line・変更内容）を1本にまとめる
4. `plan-mode`の手順に従ってcodexレビューを受ける
5. codexの指摘がなくなるまで計画ファイルを更新する
6. 判断経緯（codex指摘への対応・却下した代替案・ユーザー指示で確定した方針）が計画ファイルの`## 却下した代替案`・`## 変更履歴`節に記録されていることを確認する
7. `ExitPlanMode`で合意ゲートを通過する

### 2b plan mode外（設計ドキュメント立ち上げと実装）

`ExitPlanMode`直後に、実装着手前に必ず以下を行う。

1. `spec-designer`を呼び出し、骨子を一括生成させる
   - 入力は計画ファイルとステップ1のリサーチ結果ファイル群
   - 生成対象は作業版ドキュメントの骨子・`docs/v{next}/README.md`エントリ・必要に応じて横断ドキュメント
   - 計画ファイルの判断経緯のうち将来参照される要点は作業版`.md`の「主要設計判断」「却下した代替案」節へ`spec-designer`が転記する。一過性の経緯情報は計画ファイルの`## 却下した代替案`・`## 変更履歴`節に残ったままとし、作業版ドキュメントへは転記しない
   - 既存改修時は作業版`.md`の「改修前スナップショット」節へ恒常配置の該当節を`spec-designer`が貼り付ける（昇格時は恒常配置にマージせず破棄する）
   - 横断ドキュメント化の要否も`spec-designer`が判定する（判定基準は`spec-driven`SKILL.mdの「横断ドキュメントの判定」節）。判定が曖昧な場合はエスカレーションされるため、メインはユーザーへ確認する
2. `spec-designer`の戻り値で生成／更新ファイルを確認する
   - 作業版`.md`・`README.md`更新エントリ・横断ドキュメント（あれば）の存在とパスを確かめる
3. 実装着手直前のHEADを`BASE_SHA`として記録する（`git rev-parse HEAD`）。ステップ3のレビュアー呼び出しで差分起点として参照する
4. タスク分解とTaskCreate登録
   - 計画ファイルの「変更内容」節を起点にタスクを確定する
   - TaskCreateで各タスクを登録する（検証・コミットタスクも含める）。進行管理権はメインにあり、登録自体は`spec-designer`でなくメインが行う
5. 実装
   - TaskCreateの未完了タスクを先頭から1つずつ取り出す
   - 各タスクを`spec-implementer`エージェントに委譲する
   - 実装結果を確認し、TaskUpdateで完了にする

タスク粒度の目安:

- 1タスクあたり1〜3ファイル程度の変更
- 1タスク内で検証が完結するよう、ファイル横断の依存を避ける
- テストコードの追加は対応する実装タスクと同一タスクにまとめる

spec-designer呼び出しテンプレート:

```text
以下の作業テーマの設計ドキュメントを立ち上げてください。

作業テーマ名: {作業テーマ名}
次期バージョン: {next}
区分: 新規追加 / 既存機能改修
計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
リサーチ結果ファイル: {`docs/v{next}/.cache/{作業テーマ名}.research-{nn}.md` の一覧}
恒常配置の対応先候補: `docs/features/{機能名}.md` または `docs/topics/{トピック名}.md`（既存改修時、新規は「該当なし」）

出力:
- `docs/v{next}/{作業テーマ名}.md` 骨子（templates.mdの作業テーマテンプレートに準拠。既存改修時は「改修前スナップショット」節へ恒常配置の該当節を貼り付ける）
- `docs/v{next}/README.md` の作業テーマ一覧にエントリ追加
- 必要なら横断ドキュメント `docs/v{next}/{トピック名}.md` を新規作成し、README.mdの横断ドキュメント一覧にも追加

戻り値: 生成／更新したファイルパス一覧と1行サマリー、判定が曖昧だった事項のエスカレーション
制約: 実装コードは編集しない。書き込み対象は `docs/v{next}/` 配下に限定する
```

spec-implementer呼び出しテンプレート:

```text
以下のタスクを実装してください。

タスク: {タスクの具体的な記述}
参照: `docs/v{next}/{作業テーマ名}.md` の該当節、ステップ2で作成した計画ファイル（`~/.claude/plans/{自動生成ファイル名}.md`）
差し戻し指摘（該当時のみ）: `docs/v{next}/.cache/{作業テーマ名}.review-spec.md` または `docs/v{next}/.cache/{作業テーマ名}.review-quality.md`
制約: coding-standardsスキルを事前に呼び出し、品質基準に従うこと
完了条件: 変更後にプロジェクトのformat/lint/testが全て通ること
```

## ステップ3: Cleanup

目的: 全タスク完了後に検証・最終反映・レビュー・一時ファイル削除・コミットを行う。

手順:

1. プロジェクトのformat/lint/test（`coding-standards`で示される手順）を実行する
2. 作業版`.md`へ最終状態を反映する
   - 受け入れ基準の確認結果・実装を経て確定した事項を反映する
   - 既存改修の場合はBefore/Afterの意図的差分がドキュメントに正しく表現されているか確認する（「改修前スナップショット」節と本文の差分が意図通りか）
   - 計画ファイルからの判断経緯転記が漏れていないか再確認する
3. `docs/v{next}/README.md`の作業テーマ一覧・横断ドキュメント一覧が最新化されているか確認する（ステップ2で追加した横断ドキュメントの反映漏れに注意）
4. 新規追加した参照コメントの指し先（開発中配置のパス）が実在し、内容が関連していることを確認する
5. `spec-reviewer`エージェントに仕様適合性・ドキュメント整合性レビューを委譲する
   - 引数は以下の通り
     - 作業テーマ名
     - 作業版`.md`のパス（`docs/v{next}/{作業テーマ名}.md`）
     - 差分に含まれる他の作業版ドキュメント・`README.md`・横断ドキュメント
     - 恒常配置側の該当ドキュメント（既存改修時のみ）
     - 改修前の該当節スナップショット（既存改修時のみ、作業版`.md`内の「改修前スナップショット」節）
     - 計画ファイルのパス
     - `BASE_SHA`
     - レビュー対象外の一時ファイル一覧（`docs/v{next}/.cache/`配下の`.research-*.md`・`.review-spec.md`・`.review-quality.md`）
     - 出力ファイルパス（`docs/v{next}/.cache/{作業テーマ名}.review-spec.md`）
   - 「差分に含まれる他の作業版ドキュメント・`README.md`・横断ドキュメント」には、同一開発中バージョンディレクトリ配下の他作業テーマ・横断・`README.md`のうち差分に含まれるものを列挙する
   - 戻り値は判定（✅/❌）と指摘件数のみ。詳細はメインが`docs/v{next}/.cache/{作業テーマ名}.review-spec.md`を読んで把握する
   - 指摘があれば該当タスクを`spec-implementer`へ差し戻す（`review-spec.md`のパスを入力として渡す）。再実装後は最終反映（手順2・3）も含めて再実施した上で、再度format/lint/test・spec-reviewerのループを回す（同一ファイルを上書き）
   - `✅ 仕様適合`が返るまで次ステップへ進まない
6. `code-quality-reviewer`エージェントにコード品質レビューを委譲する
   - 引数: 作業テーマ名・作業版`.md`のパス・`BASE_SHA`・対象外一時ファイル一覧・出力ファイルパス（`docs/v{next}/.cache/{作業テーマ名}.review-quality.md`）
   - 戻り値はAssessmentと指摘件数のみ。詳細はメインが出力ファイルを読んで把握する
   - 指摘があれば`spec-implementer`へ差し戻して再実装する（`review-quality.md`のパスを入力として渡す）。再実装後は最終反映も含めて再実施し、再度format/lint/test・spec-reviewer・code-quality-reviewerのループを回す
   - `Assessment: approve`が返るまで次ステップへ進まない
   - 指摘が繰り返される場合（同種の指摘が2回以上続くなど）はループを中断してユーザーへエスカレーションする
7. 一時ファイル群を削除する
   - `docs/v{next}/.cache/{作業テーマ名}.research-*.md`（ステップ1で出力した全ファイル）
   - `docs/v{next}/.cache/{作業テーマ名}.review-spec.md`
   - `docs/v{next}/.cache/{作業テーマ名}.review-quality.md`
8. 全差分を1コミットまたは意味単位で複数コミットにまとめる

手順2・3を`spec-reviewer`実行前に置くのは、最終版のドキュメントと実装を一括で整合性検査するため。
恒常配置への反映は本ステップでは行わない。リリース完了後に`spec-driven-promote`スキルで実施する。

spec-reviewer・code-quality-reviewer呼び出しテンプレートは`templates.md`を参照する。

## 既存機能改修フロー

既存機能の改修（仕様変更・既存設計判断の見直し・スコープ縮小など）は、新規追加フローと共通の3ステップを踏む。
以下は新規追加フローとの差分のみ記述する。明記のない手順は新規追加フローと同じ。

### ステップ1差分

- 対応する恒常配置ファイル（`docs/features/{機能名}.md`または`docs/topics/{トピック名}.md`）をユーザーと合意で特定する
- `spec-researcher`の観点に「改修前の該当節スナップショット取得」を含め、対応先ファイルの該当節を`docs/v{next}/.cache/{作業テーマ名}.research-{nn}.md`のいずれかで記録する。ステップ2で`spec-designer`が作業版`.md`の「改修前スナップショット」節へ貼り付ける
- 当該機能が他機能・横断ドキュメントから参照されているかを`docs/features/`・`docs/topics/`配下の全文検索観点を`spec-researcher`へ委譲し、波及する記述の更新要否を洗い出す

### ステップ2差分

- 計画ファイル作成時、恒常配置と開発中作業テーマの粒度差を前提に変更内容を記述する
- `spec-designer`へは恒常配置の対応先候補を入力として渡す。既存改修時は作業版`.md`の「改修前スナップショット」節を`spec-designer`がリサーチ結果から貼り付ける
- 判断経緯転記では、「却下した代替案」は既存エントリを維持し追記のみ行う扱いとする（過去の判断を消さず、改修判断の履歴を残す）。`spec-designer`がこの方針で転記する
- 「主要設計判断」は現行判断を上書きしてよいが、変更理由と旧判断との差分を同節内に明記する（昇格後に読む人向けの情報）

### ステップ3差分

- `spec-reviewer`へ作業版`.md`内の「改修前スナップショット」節も参照させ、Before/Afterの意図的差分と意図しない齟齬を区別させる
- 既存実装のリネームや削除が伴う場合、関連するテスト・参照コメント・ドキュメント間リンクの追随を確認する

## サブエージェント分業

役割・起動条件・運用原則は`spec-driven`スキルSKILL.mdの「サブエージェント分業」節を参照する。
サブエージェントの起動はすべてAgentツールの`model`パラメーターに`"sonnet"`を指定する。各テンプレートのテキストは`prompt`パラメーターとして渡す。
