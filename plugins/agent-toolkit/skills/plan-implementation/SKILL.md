# 計画後の実装・検証・レビュー工程

`ExitPlanMode`で合意を得たあとに進める実装・検証・レビュー・コミットの詳細手順、
サブエージェント呼び出しテンプレート、レビュアー出力ファイルパス規約をまとめる。
計画ファイルの`## 実装・検証・レビュー`節からは本スキルを参照する前提で、
SSOTは本スキル側に置く。

## スコープ

本スキルのスコープは「実装 → 検証 → レビュー → コミット完了」まで。
ユーザーから「コミットはしない」「後でまとめてコミットする」などの明示指示がある場合のみコミット工程を省略する。
コミットメッセージ方針は配布ルール`~/.claude/rules/agent-toolkit/agent.md`の「検証とコミット」節に従う。

## 全体の流れ

1. `ExitPlanMode`直後にメインが実装着手直前のHEADを`BASE_SHA`として記録する
2. 計画ファイルの`## 変更内容`節を起点にタスク分解し、`TaskCreate`へ登録する。
   タスク分解時に依存関係を洗い出し、末尾に検証タスクを1件加える
3. 実装タスクを`plan-implementer`へ委譲する。依存のないタスクは並列起動し、依存があるタスクは解決順に順次委譲する。
   タスクの難易度に応じてモデル（`haiku`/`sonnet`/`opus`）を選択する
4. 全実装タスク完了後、検証タスクを専用の`plan-implementer`へ1件委譲する。
   計画ファイル`## 実装・検証・レビュー`節の「検証手順」を実行し、警告ゼロまで内部ループで修正する
5. 検証合格後に`spec-reviewer`を直列で先行起動する。合格後、`code-quality-reviewer`と
   `document-quality-reviewer`を並列起動する
6. 指摘があれば該当の実装タスクを`plan-implementer`へ差し戻し、再実装 → 検証タスク再起動 →
   `spec-reviewer`再起動 → 2レビュアー並列再起動のループを回す
7. 全レビュー合格後、メインがコミットする（コミット禁止指示がある場合のみ省略）

軽微な実装（例: 数行の修正・既存パターンを踏襲した追加・ドキュメントのみの修正など）は、
メイン判断で検証タスク分離やレビューを省略してよい。判断基準は本ファイル末尾の「レビュー省略の判断基準」節を参照する。

`spec-driven`文脈から呼ばれた場合も同じ流れで進める（呼び元による分岐は設けない）。
`spec-driven`文脈では`ExitPlanMode`直後にメイン側で`spec-writer`を並行起動するが、
本スキルはそれを意識せず実装フェーズを進めてよい（両者とも計画ファイルをインプットとして独立に動く）。

## 実装フェーズ

### BASE_SHAの記録

`ExitPlanMode`直後、実装着手前のHEADを`BASE_SHA`として記録する。

```bash
git rev-parse HEAD
```

レビュー時の差分起点として`BASE_SHA`を参照する。
`plan-implementer`はタスク単位でコミットしない設計のため、レビュー時点の差分は未コミットの作業ツリー差分として扱う。

### タスク分解とTaskCreate登録

計画ファイルの`## 変更内容`節を起点にタスクを確定し、`TaskCreate`へ登録する。
実装タスクに加えて、末尾に検証タスクを1件登録する。コミットは工程の最後にメインが行うためタスク化不要
（ただし「検証」「コミット」の2件はセッション冒頭で`TaskCreate`に登録すること。詳細は`agent.md`の運用方針に従う）。

タスク粒度の目安:

- 1タスクあたり1〜3ファイル程度の変更
- 1タスク内で検証が完結するよう、ファイル横断の依存を避ける
- テストコードの追加は対応する実装タスクと同一タスクにまとめる
- 依存のない実装タスクは並列起動可能なように切り出す

### 実装タスクの委譲

依存関係を分析し、依存のない実装タスクは`plan-implementer`を並列起動する
（同一メッセージ内で複数のAgentツールを呼び出す）。
依存があるタスクは解決順に順次委譲する。
実装タスクではformat/lint/testの自己検証は行わない
（linterが単一ファイル指定に対応しない場合があり、個別実行はコスト高で漏れも生じやすいため、検証タスクへ集約する）。

タスク難易度に応じてAgentツールの`model`パラメーターを指定してモデルを選択する。
選択指針は`agent.md`の「サブエージェントのモデル選択」節に従う。
`plan-implementer`のfrontmatter既定は`sonnet`だが、`model`パラメーターで上書きする運用とする。

plan-implementer（実装タスク）呼び出しテンプレート:

```text
以下のタスクを実装してください。

タスク種別: 実装
タスク: {タスクの具体的な記述}
計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
{spec-driven文脈のみ} 作業版ドキュメント: `docs/v{next}/{作業テーマ名}.md`
差し戻し指摘（該当時のみ）: {レビュアー出力ファイルパス}
制約: coding-standardsスキルを事前に呼び出し、品質基準に従うこと。format/lint/testの自己検証は行わないこと。gitコミット・pushは行わないこと
```

### 検証タスクの委譲

全実装タスク完了後、専用の`plan-implementer`を1件ディスパッチして検証タスクを実施する。
呼び出し元（メイン）は計画ファイル`## 実装・検証・レビュー`節の「検証手順」をタスクプロンプトに含めて渡す。
検証タスクは指示された手順を全て警告ゼロで通るまで内部ループで実施する
（auto-fix可能な指摘は修正し、手作業の修正が必要な箇所は原因解析と修正をする）。
メインのコンテキストを温存するため、ループは専用タスク内で完結させる。

モデル選択は手順の複雑さに応じて行う。auto-fix中心で単純な場合は`haiku`、
原因解析が必要そうな場合は`sonnet`を既定とする。深い設計判断が絡む場合のみ`opus`を検討する。

plan-implementer（検証タスク）呼び出しテンプレート:

```text
以下の検証タスクを実施してください。

タスク種別: 検証
計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
{spec-driven文脈のみ} 作業版ドキュメント: `docs/v{next}/{作業テーマ名}.md`
検証手順（警告ゼロまで内部ループで実行・修正すること）:
{計画ファイル「検証手順」の全項目を転記}

制約: coding-standardsスキルを事前に呼び出すこと。gitコミット・pushは行わないこと。
全手順の警告がゼロになるまで、auto-fix・原因解析・再実装を繰り返すこと。
手順外の追加修正を勝手に行わないこと（範囲外と判断した事項はエスカレーションする）
```

## レビューフェーズ

### 前提

レビューフェーズに入る前に以下を満たしていることを確認する。

- 全実装タスクが完了している
- 検証タスクが合格している（計画ファイル「検証手順」が全て警告ゼロ）
- `plan-implementer`以外の最終調整（`spec-driven`文脈では`spec-writer`による作業版ドキュメントの立ち上げ含む）が完了している

### spec-reviewer起動（直列先行）

仕様適合性・ドキュメント間整合性レビューを`spec-reviewer`に委譲する。
`spec-reviewer`は後続2レビュアーの前提となる仕様合格を担うため、並列ではなく直列で先行起動する。

spec-reviewer呼び出しテンプレート:

```text
以下の実装について仕様適合性レビューを実施してください。

計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
BASE_SHA: {ExitPlanMode直後に記録したHEAD}
対象外ファイル: {一時ファイル一覧}
出力ファイル: {出力ファイルパス}（差し戻しループでは上書き）

{spec-driven文脈のみ}
作業テーマ名: {作業テーマ名}
作業版ドキュメント: `docs/v{next}/{作業テーマ名}.md`
差分内の他の作業版ドキュメント・`README.md`・横断ドキュメント: {同一開発中バージョンディレクトリ配下の他作業テーマ・横断・README.mdのうち差分に含まれるもの}
恒常配置側の該当ドキュメント: `docs/features/{機能名}.md` または `docs/topics/{トピック名}.md`（既存改修時のみ）
改修前の該当節スナップショット: `docs/v{next}/{作業テーマ名}.md`内の「改修前スナップショット」節（既存改修時のみ）

差分取得: `git diff {BASE_SHA}`と、未追跡ファイルを含む作業ツリー参照で行うこと
制約: 書き込みは出力ファイル1つのみ。実装者レポートは鵜呑みにせず、コードと差分で独立検証すること。gitコミット・pushは行わないこと
戻り値: 判定（✅/❌）と指摘件数
```

指摘があれば該当タスクを`plan-implementer`へ差し戻す（レビュアー出力ファイルのパスを差し戻し指摘として渡す）。
再実装後は検証タスクを再起動し、合格を確認してから`spec-reviewer`を同一出力ファイルへ上書きで再起動する。
`✅ 仕様適合`が返るまで次ステップへ進まない。

### code-quality-reviewerとdocument-quality-reviewerの並列起動

`spec-reviewer`合格後、`code-quality-reviewer`（コード品質）と`document-quality-reviewer`（ドキュメント単体品質）を
同一メッセージ内でAgentツールを2つ並列呼び出しして起動する。
両者はレビュー対象が異なるため独立に進められる。

ドキュメント変更が無いまたは軽微な変更のみの場合は、本ファイル末尾の「レビュー省略の判断基準」を流用して
メイン判断で`document-quality-reviewer`を省略してよい。
ただし`spec-driven`文脈での作業版ドキュメント追加や、
恒常化先（`CLAUDE.md`・`.claude/rules/`・`.claude/skills/`・プロジェクトドキュメント）への反映を伴う場合は省略しない。

code-quality-reviewer呼び出しテンプレート:

```text
以下の実装についてコード品質レビューを実施してください。

計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
BASE_SHA: {ExitPlanMode直後に記録したHEAD}
対象外ファイル: {一時ファイル一覧}
出力ファイル: {出力ファイルパス}（差し戻しループでは上書き）

{spec-driven文脈のみ}
作業テーマ名: {作業テーマ名}
作業版ドキュメント: `docs/v{next}/{作業テーマ名}.md`

差分取得: `git diff {BASE_SHA}`と、未追跡ファイルを含む作業ツリー参照で行うこと
制約: 書き込みは出力ファイル1つのみ。coding-standardsスキルを事前に呼び出し、品質基準に従うこと。gitコミット・pushは行わないこと
戻り値: Assessment（approve/reject）と指摘件数（Critical/Important/Minor別）
```

document-quality-reviewer呼び出しテンプレート:

```text
以下の実装に付随するドキュメント変更についてドキュメント単体の品質レビューを実施してください。

計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
BASE_SHA: {ExitPlanMode直後に記録したHEAD}
対象外ファイル: {一時ファイル一覧}
出力ファイル: {出力ファイルパス}（差し戻しループでは上書き）

{spec-driven文脈のみ}
作業テーマ名: {作業テーマ名}
作業版ドキュメント: `docs/v{next}/{作業テーマ名}.md`

差分取得: `git diff {BASE_SHA}`と、未追跡ファイルを含む作業ツリー参照で行うこと
制約: 書き込みは出力ファイル1つのみ。writing-standardsスキルを事前に呼び出し、品質基準に従うこと。gitコミット・pushは行わないこと
戻り値: Assessment（approve/reject）と指摘件数（Critical/Important/Minor別）
```

2レビュアーのいずれかで指摘が出た場合は両者の指摘を統合して該当の実装タスクを`plan-implementer`へ差し戻し、
再実装 → 検証タスク再起動 → `spec-reviewer`再起動 → 2レビュアー並列再起動のループを回す。
両レビュアーがともに`Assessment: approve`を返すまで次ステップへ進まない。
指摘が繰り返される場合（同種の指摘が2回以上続くなど）はループを中断してユーザーへエスカレーションする。

### レビュー合格後の変更時の再実行規則

レビュー合格後に成果物（コード・ドキュメント・テストなど）を変更した場合は、
変更量に応じて検証タスクと各レビュアーを再実行する。

- コード変更あり: 検証タスク → `spec-reviewer` → `code-quality-reviewer`と`document-quality-reviewer`並列
 （ドキュメント変更を伴わなければ`document-quality-reviewer`は省略可）
- ドキュメントのみの変更あり: 検証タスク → `spec-reviewer` → `document-quality-reviewer`。
  対象は`spec-driven`文脈の作業版ドキュメント・`README.md`や、
  恒常化先の`CLAUDE.md`・`.claude/rules/`・`.claude/skills/`・プロジェクトドキュメントなど。
  コード挙動が変わらないため`code-quality-reviewer`は省略してよい
- 軽微な誤字修正・コメント調整など: メイン判断で再実行省略可

## コミットフェーズ

全レビュー合格後、メインがコミットする。
コミットメッセージ方針・`amend`/`fixup`使い分けは配布ルール`~/.claude/rules/agent-toolkit/agent.md`の
「検証とコミット」節に従う（本スキルでは再掲しない）。

ユーザーから以下のような明示指示がある場合のみコミット工程を省略する。

- 「コミットはしない」「後でまとめてコミットする」
- 「pushまで不要」（pushは既定で実施しないため通常は該当しない）

## レビュー省略の判断基準

以下の条件を総合的に勘案し、レビュー省略が妥当かメインが判断する。

- 変更規模が小さい（数行〜数十行程度、1〜2ファイル）
- 影響範囲が限定的（公開インターフェースを変更しない、他モジュールへ波及しない）
- 既存パターンを踏襲している（類似実装が既に存在する、設計判断を伴わない）
- 判断の余地が少ない（単純な誤字修正・既知バグの明確な修正など）
- テスト済み・検証済み（自動テストで挙動が担保されている）

レビュー省略時は計画ファイルの`## 変更履歴`節にその判断理由を記録する。
判断に迷ったら省略せず実行する方針とする。

## レビュアー出力ファイルパス規約

計画ファイル名（`~/.claude/plans/{自動生成ファイル名}.md`）から派生する出力ファイル名は、
拡張子直前に`.review-spec`・`.review-quality`・`.review-docs`を挿入する規則で生成する。

- `spec-driven`文脈: `docs/v{next}/.cache/{作業テーマ名}.review-{kind}.md`
- 単独`plan-mode`文脈: `~/.claude/plans/{plan名}.review-{kind}.md`
  - 例: `~/.claude/plans/foo-bar-plan-m-xxx.md` →
    `~/.claude/plans/foo-bar-plan-m-xxx.review-spec.md`・
    `~/.claude/plans/foo-bar-plan-m-xxx.review-quality.md`・
    `~/.claude/plans/foo-bar-plan-m-xxx.review-docs.md`

`{kind}`は以下のいずれか。

- `spec`: 仕様適合性・ドキュメント間整合性レビュー（`spec-reviewer`）
- `quality`: コード品質レビュー（`code-quality-reviewer`）
- `docs`: ドキュメント単体品質レビュー（`document-quality-reviewer`）

差し戻しループでは同一ファイルを上書きする（前回指摘と混在させない）。
レビュー完了後、`spec-driven`文脈のファイルはCleanupで削除する。
単独`plan-mode`文脈のファイルは計画ファイル本体と同じ`~/.claude/plans/`配下に残るが、
作業完了後の扱いはユーザー判断に委ねる（通常は削除してよい）。

単独`plan-mode`文脈の出力先`~/.claude/plans/`はユーザー全体で共有される（プロジェクトローカルでない）。
そのため複数プロジェクトを並行して作業する場合に、他セッションのレビュー結果と取り違える事故が起こりうる。
以下の運用で衝突を避ける。

- 出力ファイル名は計画ファイル名（Claude Code標準の一意なハッシュ入りファイル名）をそのまま使い、
  計画名の一部を短縮・省略しない
- 本スキル起動時（レビュアー起動前）に`~/.claude/plans/{plan名}.review-{kind}.md`が既存なら、
  前回セッションの残骸の可能性があるため上書き前にユーザーへ存在を報告し、必要に応じて削除または退避する

## サブエージェントへのコミット禁止指示

上記の各呼び出しテンプレートには「gitコミット・pushは行わないこと」を制約として必ず含める。
コミット事故防止のため、コミット・pushはメインの責務として一元管理する。
詳細は配布ルール`~/.claude/rules/agent-toolkit/agent.md`の「検証とコミット」節を参照。
