# 計画後の実装・検証・レビュー工程

本スキルは、`agent-toolkit:plan-mode`での計画ファイル合意後の、
実装・検証・レビュー・コミットの手順を示す。

1. タスク分解
2. 実装
3. 検証
4. レビュー
5. コミット

これらの工程では、計画ファイルを唯一の入力として参照する。

計画ファイルがフェーズ分けされていた場合は、上記の工程を各フェーズ分繰り返す。

## 1. タスク分解

計画ファイルを起点にタスク分解し、`TaskCreate`へ登録する。
タスク分解時に依存関係を洗い出し、検証・レビュー・コミットタスクまで登録する。

タスク分解時は、サブエージェントの活用を前提とし、以下の点を考慮した適切な粒度に分解する。

- 時間効率（依存関係のない部分は並列化）
- コンテキスト効率（1エージェントあたりの作業量を増やし過ぎない）
- コスト効率（簡単なものは極力sonnet/haikuモデルを活用）

ただし、分割のし過ぎは全ての効率を悪化させるため、細かく分割し過ぎないことに注意。
（それぞれのサブエージェントが計画ファイルや必要スキルなどを読むため）
小規模なタスクであればメイン側で直接実行した方がよい場合もある。
（文脈が限られたり精度の低いモデルで作業すると、成果物の品質が低下し、後の工程で手間が増えるため）

検証タスクは上記の効率を考慮して実装タスクとまとめてもよい。

ただし、レビュータスクは必ず独立させ、観点毎に別のサブエージェントを起動する。
（実装の経緯を持ち込まず、クリーンな状態でレビューさせるため）
レビュータスクも、上記の効率を考慮し、1観点に対して複数のサブエージェントを起動してもよい。
また、最終判断はメイン側（opus/sonnet）が行う前提でより安価なモデルを使用してもよい。

コミットはメインが直接行う。（通常小規模なタスクのため）

## 2. 実装

分解されたタスクに従い、実装タスクを`plan-implementer`へ委譲する。

呼び出しテンプレート:

```text
以下のタスクを実装してください。

- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`

## タスク

{タスクの具体的な記述}

## 差し戻し指摘（該当時のみ）

{メインが統合した指摘テキストをここに直接貼り付ける}

## 制約

- 作業開始前に`agent-toolkit:coding-standards`スキルを呼び出し、品質基準に従うこと
- gitコミット・pushは行わないこと
```

## 2. 検証

全実装タスク完了後、専用の`plan-implementer`で検証タスクを実施する。

呼び出しテンプレートは実装タスクと同じものを使用し、実装→検証などに適宜置き換える。

## 3. レビュー

検証タスクの完了後、レビュータスクを開始する。

### 初回レビュー（3種別並列）

計画ファイルの`## 実装・検証・レビュー`節「レビュー実施方針」項目を確認し、起動対象の種別を特定する。
種別`spec`（仕様適合性）・`code`（コード品質）・`docs`（ドキュメント単体品質）を並列呼び出しする。

起動要否は計画時に決定済みであるが、実装での追加変更（ドキュメントの新規追加・削除など）で
判断が変わる場合は、メイン判断で追加起動・省略を調整してよい。

### plan-reviewer呼び出しテンプレート（初回）

種別`spec`・`code`・`docs`で共通の骨子を使い、種別固有の欄を差し替える。
初回レビューでは対象範囲の指摘を出し切ること。再レビュー（`followup`）では新規指摘を追加しない前提で、
この初回で見つかった問題をすべて列挙する。

```text
以下の実装について{レビュー種別名}レビューを実施してください。

- 担当観点: {担当観点}
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
- 対象フェーズ: {単一フェーズ計画では「全体」、複数フェーズ計画では「フェーズN: 題目」}
- 対象外ファイル: {一時ファイル一覧}
- 差分取得: `git diff HEAD`と、未追跡ファイルを含む作業ツリー参照で行う

## 制約

- 作業開始前に{事前呼び出しスキル}スキルを呼び出し、品質基準に従うこと
- Bashは読み取り系操作に限定すること。ファイルへの書き込みは行わないこと
- 一度のレビューですべての問題点を洗い出すこと（指摘を小出しにせず初回で徹底的に指摘すること）
- gitコミット・pushは行わないこと
```

レビュー種別ごとの差分:

| 種別 | 事前呼び出しスキル | 担当観点 |
| --- | --- | --- |
| `spec` | なし | 仕様適合性・複数ドキュメント間整合性 |
| `code` | `agent-toolkit:coding-standards` | コード品質 |
| `docs` | `agent-toolkit:writing-standards` | ドキュメント単体品質 |

### 指摘の統合と差し戻し

各レビュアーの戻り値本文を受け取り、メインが指摘を統合する。
必要に応じてユーザーへ確認して対応方針を決定する。
（計画に照らし合わせて妥当な修正であれば確認無しで修正してよい）

最終的に、対応が必要な指摘が1件以上あれば、
どのような修正をするかをユーザーに簡潔に報告したうえで、
当該タスクを適切な`plan-implementer`へ依頼する。

### 再レビュー（followup）

差し戻し後の再実装が完了したら、検証タスクを再起動して合格を確認する。
その後`plan-reviewer`（種別`followup`・モデル`haiku`）を1件起動し、前回統合指摘への対応状況を確認する。

ユーザーが「対応不要」と判断した指摘は、判断理由とともに統合済みテキスト内の当該指摘の行末尾に
`（ユーザー判断: 不対応 — <理由>）`の書式で注記する。
差し戻しプロンプトの「差し戻し指摘」欄には対応する指摘のみを埋め込み、
不対応と判断された指摘は除外する。

再レビューでは新規指摘を原則追加しない（明らかな回帰が発見された場合のみ末尾に短く併記する）。

### plan-reviewer呼び出しテンプレート（followup）

「前回統合指摘:」欄には、前節「指摘の統合と差し戻し」で管理している統合済み指摘テキストを転記する。
各指摘行末尾に`（ユーザー判断: 不対応 — <理由>）`注記が付いている指摘は、
`followup`レビュアーの対応確認対象から除外される前提で、注記付きのまま転記する。

転記例:

```text
```

```text
以下の実装についてfollowupレビューを実施してください。

- レビュー種別: followup
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
- 対象フェーズ: {単一フェーズ計画では「全体」、複数フェーズ計画では「フェーズN: 題目」}
- 差分取得: `git diff HEAD`と、未追跡ファイルを含む作業ツリー参照で行うこと

## 前回指摘

- `path/to/file.py:L10` — 戻り値の型アノテーションが欠けている
- `path/to/file.py:L20` — 変数名が不明瞭 （ユーザー判断: 不対応 — 現状仕様で問題なし）
}

## 制約

- Bashは読み取り系操作に限定すること。ファイルへの書き込みは行わないこと。
- gitコミット・pushは行わないこと。
```

### 差し戻しループのエスカレーション

`followup`で妥当な指摘が2回以上残る場合はループを中断してユーザーへエスカレーションする。

## 5. コミット

全レビューに合格したら、計画ファイルの`## 実装・検証・レビュー`節「コミット方針」項目に従ってコミットする。
単一フェーズ計画ではこのコミットで作業完了。
複数フェーズ計画ではフェーズ単位の中間コミットになり、コミット後に次フェーズのタスク分解へ戻る。
