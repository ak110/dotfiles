# 計画後の実装・検証・レビュー工程

`ExitPlanMode`で合意を得たあとに進める実装・検証・レビュー・コミットは必ず本スキルの内容に従い実施する。

この工程では計画ファイルを唯一の入力として参照する。

## スコープ

本スキルのスコープは「実装 → 検証 → レビュー → コミット完了」まで。
ユーザーから「コミットはしない」「後でまとめてコミットする」などの明示指示がある場合のみコミット工程を省略する。
コミットメッセージ方針は配布ルール`~/.claude/rules/agent-toolkit/agent.md`の「検証とコミット」節に従う。

## 全体の流れ

1. 計画ファイルの`## 変更内容`節を起点にタスク分解し、`TaskCreate`へ登録する。
   タスク分解時に依存関係を洗い出し、末尾に検証タスクを1件加える
2. 実装タスクを`plan-implementer`へ委譲する。依存のないタスクは並列起動し、依存があるタスクは解決順に順次委譲する。
   タスクの難易度に応じてモデル（`haiku`/`sonnet`/`opus`）を選択する
  （メイン側で実装は一切行わないこと）
3. 全実装タスク完了後、検証タスクを専用の`plan-implementer`へ1件委譲する。
   計画ファイル`## 実装・検証・レビュー`節の「検証手順」を実行し、警告ゼロまで内部ループで修正する
4. 検証合格後に`spec-reviewer`を直列で先行起動する。合格後は計画ファイルの「レビュー実施方針」に従い、
   `code-quality-reviewer`・`document-quality-reviewer`のうち起動対象のレビュアーのみ実行する。
   両方起動する場合は並列起動する
5. 指摘があれば該当の実装タスクを`plan-implementer`へ差し戻し、再実装 → 検証タスク再起動 →
   `spec-reviewer`再起動 → 計画ファイルの「レビュー実施方針」に従う後続レビュアー再実行のループを回す
6. 全レビュー合格後、メインがコミットする（コミット禁止指示がある場合のみ省略）

レビュー起動要否は計画ファイルの`## 実装・検証・レビュー`節「レビュー実施方針」に従う。
計画時の判断基準は`plan-mode`スキルの「レビュー実施方針の判断基準」節を参照する。
軽微な実装（例: 数行の修正・既存パターンを踏襲した追加・ドキュメントのみの修正など）で
計画ファイルが全レビューの省略を指示している場合は、検証タスク分離とレビューの両方を省略してよい。

`spec-driven`文脈から呼ばれた場合も同じ流れで進める（呼び元による分岐は設けない）。
`spec-driven`文脈では`ExitPlanMode`直後にメイン側で`spec-writer`を並行起動するが、
本スキルはそれを意識せず実装フェーズを進めてよい（両者とも計画ファイルをインプットとして独立に動く）。

## 複数フェーズの運用

計画ファイルが複数フェーズに分割されている場合（`## 変更内容`・`## 実装・検証・レビュー`節の内側に
`### フェーズN: 題目`サブ節がある場合）、フェーズ単位で「実装→検証→レビュー→コミット→次フェーズ」を順次反復する。
フェーズ分割の有無は計画ファイル側の構造で決まる。詳細は`plan-mode`スキルの「複数フェーズ計画の記述方法」節を参照する。

各フェーズ開始時にメインが行う準備:

- 当該フェーズの`## 変更内容`サブ節を起点にタスク分解する（前フェーズで完了したタスクは完了確定済みとする）
- 検証手順とレビュー実施方針も当該フェーズのサブ節に記載があればそれを優先し、無ければ全体節を参照する
- 各レビュアー呼び出しテンプレートの「対象フェーズ」欄に当該フェーズ名（`フェーズN: 題目`）を渡す
- フェーズ完了時のコミット後、`git status`が空であることと`TaskCreate`の当該フェーズタスクがすべて完了であることを
  確認してから次フェーズへ進む

差分起点はフェーズ間コミット後の`HEAD`が常に成立する
（`plan-implementer`はタスク単位でコミットしないため、前フェーズのコミットを起点に当該フェーズの差分が成立する）。
単一フェーズ計画ではレビュアーへ「対象フェーズ」として「全体」を渡し、本スキルの全体フローを1回実行して完了する。

## 実装フェーズ

### タスク分解とTaskCreate登録

計画ファイルの`## 変更内容`節（複数フェーズ計画では当該フェーズのサブ節）を起点にタスクを確定し、
`TaskCreate`へ登録する。
実装タスクに加えて、末尾に検証タスクを1件登録する。コミットは工程の最後にメインが行うためタスク化不要
（ただし「検証」「コミット」の2件はセッション冒頭で`TaskCreate`に登録すること。詳細は`agent.md`の運用方針に従う）。

タスク粒度の目安:

- 1タスクあたり1〜3ファイル程度の変更
- 1タスク内で検証が完結するよう、ファイル横断の依存を避ける
- テストコードの追加は対応する実装タスクと同一タスクにまとめる
- 依存のない実装タスクは並列起動可能なように切り出す

### 実装タスクの委譲

依存関係を分析し、依存のない実装タスクは`plan-implementer`を並列起動する
（同一メッセージ内で複数のAgentツールを呼び出す）。
依存があるタスクは解決順に順次委譲する。
実装タスクではテスト一式の自己実行は行わない
（フルテスト個別実行はコスト高で漏れも生じやすいため、検証タスクへ集約する）。
ただしファイル個別実行をサポートするlinter／formatterのみ、変更対象ファイルへ事前実行する運用は許容する
（自動修正可能な指摘の早期回収が目的）。

タスク難易度に応じてAgentツールの`model`パラメーターを指定してモデルを選択する。
選択指針は`agent.md`の「サブエージェントのモデル選択」節に従う。
`plan-implementer`のfrontmatter既定は`sonnet`だが、`model`パラメーターで上書きする運用とする。

plan-implementer（実装タスク）呼び出しテンプレート:

```text
以下のタスクを実装してください。

タスク種別: 実装
タスク: {タスクの具体的な記述}
計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
{spec-driven文脈のみ} 作業版ドキュメント: `docs/v{next}/{作業テーマ名}.md`
差し戻し指摘（該当時のみ）: {レビュアー出力ファイルパス}
制約: coding-standardsスキルを事前に呼び出し、品質基準に従うこと。
ファイル個別実行をサポートしているlinter／formatterのみ変更対象ファイルへ事前実行してよいが、
テスト一式の自己実行は行わないこと。gitコミット・pushは行わないこと
```

### 検証タスクの委譲

全実装タスク完了後、専用の`plan-implementer`を1件ディスパッチして検証タスクを実施する。
呼び出し元（メイン）は計画ファイル`## 実装・検証・レビュー`節の「検証手順」をタスクプロンプトに含めて渡す。
検証タスクは指示された手順を全て警告ゼロで通るまで内部ループで実施する
（auto-fix可能な指摘は修正し、手作業の修正が必要な箇所は原因解析と修正をする）。
メインのコンテキストを温存するため、ループは専用タスク内で完結させる。

モデル選択は手順の複雑さに応じて行う。auto-fix中心で単純な場合は`haiku`、
原因解析が必要そうな場合は`sonnet`を既定とする。深い設計判断が絡む場合のみ`opus`を検討する。

plan-implementer（検証タスク）呼び出しテンプレート:

```text
以下の検証タスクを実施してください。

タスク種別: 検証
計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
{spec-driven文脈のみ} 作業版ドキュメント: `docs/v{next}/{作業テーマ名}.md`
検証手順（警告ゼロまで内部ループで実行・修正すること）:
{計画ファイル「検証手順」の全項目を転記}

制約: coding-standardsスキルを事前に呼び出すこと。gitコミット・pushは行わないこと。
全手順の警告がゼロになるまで、auto-fix・原因解析・再実装を繰り返すこと。
手順外の追加修正を勝手に行わないこと（範囲外と判断した事項はエスカレーションする）
```

## レビューフェーズ

### 前提

レビューフェーズに入る前に以下を満たしていることを確認する。

- 全実装タスクが完了している
- 検証タスクが合格している（計画ファイル「検証手順」が全て警告ゼロ）
- `plan-implementer`以外の最終調整（`spec-driven`文脈では`spec-writer`による作業版ドキュメントの立ち上げ含む）が完了している

### spec-reviewer起動（直列先行）

仕様適合性・ドキュメント間整合性レビューを`spec-reviewer`に委譲する。
`spec-reviewer`は後続レビュアーの前提となる仕様合格を担うため、並列ではなく直列で先行起動する。

spec-reviewer呼び出しテンプレート:

```text
以下の実装について仕様適合性レビューを実施してください。

計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
対象フェーズ: {単一フェーズ計画では「全体」、複数フェーズ計画では「フェーズN: 題目」}
対象外ファイル: {一時ファイル一覧}
出力ファイル: {出力ファイルパス}（差し戻しループでは上書き）

{spec-driven文脈のみ}
作業テーマ名: {作業テーマ名}
作業版ドキュメント: `docs/v{next}/{作業テーマ名}.md`
差分内の他の作業版ドキュメント・`README.md`・横断ドキュメント: {同一開発中バージョンディレクトリ配下の他作業テーマ・横断・README.mdのうち差分に含まれるもの}
恒常配置側の該当ドキュメント: `docs/features/{機能名}.md` または `docs/topics/{トピック名}.md`（既存改修時のみ）
改修前の該当節スナップショット: `docs/v{next}/{作業テーマ名}.md`内の「改修前スナップショット」節（既存改修時のみ）

差分取得: `git diff HEAD`と、未追跡ファイルを含む作業ツリー参照で行うこと
制約: 書き込みは出力ファイル1つのみ。実装者レポートは鵜呑みにせず、コードと差分で独立検証すること。gitコミット・pushは行わないこと
戻り値: 出力ファイルパスと指摘件数
```

指摘件数が1件以上なら該当タスクを`plan-implementer`へ差し戻す（レビュアー出力ファイルのパスを差し戻し指摘として渡す）。
再実装後は検証タスクを再起動し、合格を確認してから`spec-reviewer`を同一出力ファイルへ上書きで再起動する。
指摘件数が0件になるまで次ステップへ進まない。

### 後続レビュアー起動（code-quality-reviewer・document-quality-reviewer）

`spec-reviewer`合格後、計画ファイルの`## 実装・検証・レビュー`節「レビュー実施方針」項目を確認する。
そこで起動対象に指定された`code-quality-reviewer`（コード品質）・
`document-quality-reviewer`（ドキュメント単体品質）のみを実行する。
両方起動する場合は同一メッセージ内でAgentツールを2つ並列呼び出しする
（両者はレビュー対象が異なるため独立に進められる）。
片方のみ起動する場合は通常どおり単独呼び出しでよい。

起動要否は計画時に決定済みであるが、実装フェーズでの追加変更（ドキュメントの新規追加・削除など）で
判断が変わる場合は、メイン判断で追加起動・省略を調整してよい。
その場合でも以下に該当する変更では`document-quality-reviewer`を省略しない。

- `spec-driven`文脈での作業版ドキュメント追加
- 恒常化先（`CLAUDE.md`・`.claude/rules/`・`.claude/skills/`・プロジェクトドキュメント）への反映を伴う変更

code-quality-reviewer呼び出しテンプレート:

```text
以下の実装についてコード品質レビューを実施してください。

計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
対象フェーズ: {単一フェーズ計画では「全体」、複数フェーズ計画では「フェーズN: 題目」}
対象外ファイル: {一時ファイル一覧}
出力ファイル: {出力ファイルパス}（差し戻しループでは上書き）

{spec-driven文脈のみ}
作業テーマ名: {作業テーマ名}
作業版ドキュメント: `docs/v{next}/{作業テーマ名}.md`

差分取得: `git diff HEAD`と、未追跡ファイルを含む作業ツリー参照で行うこと
制約: 書き込みは出力ファイル1つのみ。coding-standardsスキルを事前に呼び出し、品質基準に従うこと。gitコミット・pushは行わないこと
戻り値: 出力ファイルパスと指摘件数
```

document-quality-reviewer呼び出しテンプレート:

```text
以下の実装に付随するドキュメント変更についてドキュメント単体の品質レビューを実施してください。

計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
対象フェーズ: {単一フェーズ計画では「全体」、複数フェーズ計画では「フェーズN: 題目」}
対象外ファイル: {一時ファイル一覧}
出力ファイル: {出力ファイルパス}（差し戻しループでは上書き）

{spec-driven文脈のみ}
作業テーマ名: {作業テーマ名}
作業版ドキュメント: `docs/v{next}/{作業テーマ名}.md`

差分取得: `git diff HEAD`と、未追跡ファイルを含む作業ツリー参照で行うこと
制約: 書き込みは出力ファイル1つのみ。writing-standardsスキルを事前に呼び出し、品質基準に従うこと。gitコミット・pushは行わないこと
戻り値: 出力ファイルパスと指摘件数
```

起動したレビュアーのいずれかで指摘件数が1件以上の場合は、指摘を統合して該当の実装タスクを`plan-implementer`へ差し戻す。
そこから以下のループを回す。

- 再実装 → 検証タスク再起動 → `spec-reviewer`再起動 → 起動対象レビュアーの再実行（両方なら並列）

軽微な指摘修正の場合は再レビューはスキップしてよい。
指摘が繰り返される場合（同種の指摘が2回以上続くなど）はループを中断してユーザーへエスカレーションする。

### レビュー合格後の変更時の再実行規則

レビュー合格後に成果物（コード・ドキュメント・テストなど）を変更した場合は、
変更量に応じて検証タスクと各レビュアーを再実行する。

判断の原則として、既合格状態での変更が軽微（変更規模が小さく影響範囲が限定的で設計判断を伴わない）と判断できる場合は、
メイン判断で検証・レビューの再実行をスキップしてよい。
判断材料は`plan-mode`スキルの「レビュー実施方針の判断基準」と同じ観点を参照する。
計画ファイルの「レビュー実施方針」で起動対象外となっているレビュアーは、
変更後も省略理由が引き続き成立するなら再実行時も省略してよい。

以下は変更規模別の既定パターン。計画ファイルの「レビュー実施方針」と齟齬がある場合は計画ファイル側を優先する。

- コード変更あり: 検証タスク → `spec-reviewer` →
  起動対象の後続レビュアー（`code-quality-reviewer`・`document-quality-reviewer`の該当分）
 （ドキュメント変更を伴わなければ`document-quality-reviewer`は省略可）
- ドキュメントのみの変更あり: 検証タスク → `spec-reviewer` → `document-quality-reviewer`。
  対象は`spec-driven`文脈の作業版ドキュメント・`README.md`や、
  恒常化先の`CLAUDE.md`・`.claude/rules/`・`.claude/skills/`・プロジェクトドキュメントなど。
  コード挙動が変わらないため`code-quality-reviewer`は省略してよい
- 軽微な誤字修正・コメント調整など: メイン判断で再実行省略可

## フェーズごとのコミット

全レビュー合格後、メインがコミットする。
単一フェーズ計画ではこのコミットで作業完了。
複数フェーズ計画ではフェーズ単位の中間コミットになり、コミット後に次フェーズの実装フェーズへ戻る。

コミットメッセージ方針・`amend`/`fixup`使い分けは配布ルール`~/.claude/rules/agent-toolkit/agent.md`の
「検証とコミット」節に従う（本スキルでは再掲しない）。

ユーザーから以下のような明示指示がある場合のみコミット工程を省略する。

- 「コミットはしない」「後でまとめてコミットする」
- 「pushまで不要」（pushは既定で実施しないため通常は該当しない）

## レビュアー出力ファイル運用

レビュアー出力ファイルのパス構築規則は計画ファイルの`## 実装・検証・レビュー`節「レビュアー出力ファイル規約」項目に従う。
パス構築規則の詳細は`plan-mode`スキルの計画ファイル構成節を参照する。

- メインは戻り値の指摘件数のみで合否を判定する。指摘0件は合格として次工程へ進み、1件以上は差し戻しに回す
- 差し戻し時は出力ファイルパスを`plan-implementer`プロンプトに含めて渡し、サブエージェント側で本文を直接読み取らせる
 （メイン側で本文を読み込まず、コンテキスト消費を抑える）
- 差し戻しループでは同一ファイルを上書きする（前回指摘と混在させない）
- レビュー完了後、`spec-driven`文脈のファイルはCleanupで削除する。
  単独`plan-mode`文脈のファイルは計画ファイル本体と同じ`~/.claude/plans/`配下に残るが、
  作業完了後の扱いはユーザー判断に委ねる（通常は削除してよい）
- 単独`plan-mode`文脈の出力先`~/.claude/plans/`はユーザー全体で共有される（プロジェクトローカルでない）。
  複数プロジェクトを並行して作業する場合に他セッションのレビュー結果と取り違える事故が起こりうるため、
  以下の運用で衝突を回避する
  - 出力ファイル名は計画ファイル名（Claude Code標準の一意なハッシュ入りファイル名）をそのまま使い、
    計画名の一部を短縮・省略しない
  - 本スキル起動時（レビュアー起動前）に計画ファイルの「レビュアー出力ファイル規約」が示すパスへ既存ファイルがあれば、
    前回セッションの残骸の可能性があるため上書き前にユーザーへ存在を報告し、必要に応じて削除または退避する
