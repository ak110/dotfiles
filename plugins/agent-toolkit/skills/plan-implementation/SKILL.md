# 計画後の実装・検証・レビュー工程

`ExitPlanMode`で合意を得たあとに進める実装・検証・レビュー・コミットは必ず本スキルの内容に従い実施する。
この工程では計画ファイルを唯一の入力として参照する。
スコープは「実装 → 検証 → レビュー → コミット完了」まで。
ユーザーから明示指示がある場合のみコミット工程を省略する。

## 全体の流れ

1. 計画ファイルの`## 変更内容`節を起点にタスク分解し、`TaskCreate`へ登録する。
   タスク分解時に依存関係を洗い出し、末尾に検証タスクを1件加える
2. 実装タスクを`plan-implementer`へ委譲する。
   依存のないタスクは並列起動し、依存があるタスクは解決順に順次委譲する。
   タスクの難易度に応じてモデル（`haiku`/`sonnet`/`opus`）を選択する。
   メイン側での実装は行わない
3. 全実装タスク完了後、検証タスクを専用の`plan-implementer`へ1件委譲する
4. 検証合格後に`plan-reviewer`（種別`spec`・`code`・`docs`のうち起動対象のみ）を
   同一メッセージ内で並列起動する。
   起動対象は計画ファイルの「レビュー実施方針」に従う
5. 各レビュアーの戻り値本文からメインが指摘を統合し、指摘があれば該当の実装タスクを
   `plan-implementer`へ差し戻す。差し戻し指摘は統合済みテキストをプロンプトに直接埋め込む。
   再実装後は検証タスク再起動 →
   `plan-reviewer`（種別`followup`・モデル`haiku`）再起動のループを回す
6. 全指摘が解消したら計画ファイルの「コミット方針」項目に従ってメインがコミットする
  （コミット禁止指示がある場合のみ省略）

レビュー起動要否は計画ファイルの`## 実装・検証・レビュー`節「レビュー実施方針」に従う。
計画時の判断基準は`plan-mode`スキルの「レビュー実施方針の判断基準」節を参照する。
軽微な実装（例: 数行の修正・既存パターンを踏襲した追加・ドキュメントのみの修正など）で
計画ファイルが全レビューの省略を指示している場合は、検証タスク分離とレビューの両方を省略してよい。

`spec-driven`文脈から呼ばれた場合も同じ流れで進める（呼び元による分岐は設けない）。
`spec-driven`文脈では`ExitPlanMode`直後にメイン側で`spec-writer`を並行起動するが、
本スキルはそれを意識せず実装フェーズを進めてよい（両者とも計画ファイルをインプットとして独立に動く）。

## 複数フェーズの運用

計画ファイルが複数フェーズに分割されている場合（`## 変更内容`・`## 実装・検証・レビュー`節の内側に
`### フェーズN: 題目`サブ節がある場合）、フェーズ単位で「実装→検証→レビュー→コミット→次フェーズ」を順次反復する。
フェーズ分割の有無は計画ファイル側の構造で決まる。
詳細は`plan-mode`スキルの「複数フェーズ計画の記述方法」節を参照する。

各フェーズ開始時にメインが行う準備:

- 当該フェーズの`## 変更内容`サブ節を起点にタスク分解する（前フェーズで完了したタスクは完了確定済みとする）
- 検証手順とレビュー実施方針も当該フェーズのサブ節に記載があればそれを優先し、無ければ全体節を参照する
- 各レビュアー呼び出しテンプレートの「対象フェーズ」欄に当該フェーズ名（`フェーズN: 題目`）を渡す
- フェーズ完了時のコミット後、`git status`が空であることと`TaskCreate`の当該フェーズタスクがすべて完了であることを
  確認してから次フェーズへ進む

差分起点はフェーズ間コミット後の`HEAD`が常に成立する
（`plan-implementer`はタスク単位でコミットしないため、前フェーズのコミットを起点に当該フェーズの差分が成立する）。
単一フェーズ計画ではレビュアーへ「対象フェーズ」として「全体」を渡し、本スキルの全体フローを1回実行して完了する。

## 実装フェーズ

### タスク分解とTaskCreate登録

計画ファイルの`## 変更内容`節（複数フェーズ計画では当該フェーズのサブ節）を起点にタスクを確定し、
`TaskCreate`へ登録する。
実装タスクに加えて、末尾に検証タスクを1件登録する。
コミットは工程の最後にメインが行うためタスク化不要
（ただし「検証」「コミット」の2件はセッション冒頭で`TaskCreate`に登録すること。
詳細は`agent.md`の運用方針に従う）。

タスク粒度の目安:

- 1タスクあたり1〜3ファイル程度の変更
- 1タスク内で検証が完結するよう、ファイル横断の依存を避ける
- テストコードの追加は対応する実装タスクと同一タスクにまとめる
- 依存のない実装タスクは並列起動可能なように切り出す

### plan-implementer呼び出しテンプレートの共通事項

呼び出し時に共通制約・戻り値をプロンプトに含めるため、テンプレートには各タスク固有の欄のみ記載する。

共通制約:

- gitコミット・pushは行わないこと

共通戻り値:

- `status`（`completed`・`blocked`・`needs_escalation`のいずれか）
- `summary`
- `blockers`（`blocked`時）
- `escalation_reason`（`needs_escalation`時）

各`status`の判定基準は`plan-implementer`エージェント定義の「エスカレーション判断基準」節を参照する。
メインは受領した`status`に応じて次のとおり対応する。

- `completed`: 次工程へ進む
- `needs_escalation`: `escalation_reason`に基づき次段のモデルで再dispatchする
 （`haiku`→`sonnet`、`sonnet`→`opus`）
- `blocked`: `blockers`を確認してユーザー判断を仰ぐ

### 実装タスクの委譲

依存関係を分析し、依存のない実装タスクは`plan-implementer`を並列起動する
（同一メッセージ内で複数のAgentツールを呼び出す）。
依存があるタスクは解決順に順次委譲する。
実装タスクではテスト一式の自己実行は行わない
（フルテスト個別実行はコスト高で漏れも生じやすいため、検証タスクへ集約する）。
ただしファイル個別実行をサポートするlinter／formatterのみ、変更対象ファイルへ事前実行する運用は許容する
（自動修正可能な指摘の早期回収が目的）。

タスク難易度に応じてAgentツールの`model`パラメーターを指定してモデルを選択する。
選択指針は`agent.md`の「サブエージェントのモデル選択」節に従う。
`plan-implementer`のfrontmatter既定は`sonnet`だが、`model`パラメーターで上書きする運用とする。

plan-implementer（実装タスク）呼び出しテンプレート:

```text
以下のタスクを実装してください。

タスク種別: 実装
タスク: {タスクの具体的な記述}
計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
{spec-driven文脈のみ} 作業版ドキュメント: `docs/v{next}/{作業テーマ名}.md`
差し戻し指摘（該当時のみ）:
{メインが統合した指摘テキストをここに直接貼り付ける}
制約: coding-standardsスキルを事前に呼び出し、品質基準に従うこと。
ファイル個別実行をサポートしているlinter／formatterのみ変更対象ファイルへ事前実行してよいが、
テスト一式の自己実行は行わないこと。
```

### 検証タスクの委譲

全実装タスク完了後、専用の`plan-implementer`を1件ディスパッチして検証タスクを実施する。
呼び出し元（メイン）は計画ファイル`## 実装・検証・レビュー`節の「検証手順」をタスクプロンプトに含めて渡す。
検証タスクは指示された手順を全て警告ゼロで通るまで内部ループで実施する
（auto-fix可能な指摘は修正し、手作業の修正が必要な箇所は原因解析と修正をする）。
メインのコンテキストを温存するため、ループは専用タスク内で完結させる。

モデル選択は手順の複雑さに応じて行う。
auto-fix中心で単純な場合は`haiku`、原因解析が必要そうな場合は`sonnet`を既定とする。
深い設計判断が絡む場合のみ`opus`を検討する。

plan-implementer（検証タスク）呼び出しテンプレート:

```text
以下の検証タスクを実施してください。

タスク種別: 検証
計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
{spec-driven文脈のみ} 作業版ドキュメント: `docs/v{next}/{作業テーマ名}.md`
検証手順（警告ゼロまで内部ループで実行・修正すること）:
{計画ファイル「検証手順」の全項目を転記}

制約: coding-standardsスキルを事前に呼び出すこと。
全手順の警告がゼロになるまで、auto-fix・原因解析・再実装を繰り返すこと。
手順外の追加修正を勝手に行わないこと（範囲外と判断した事項はエスカレーションする）。
```

`status`の判定基準とメインの対応は実装タスクと同じ。

## レビューフェーズ

レビューフェーズに入る前に以下を満たしていることを確認する。

- 全実装タスクが完了していること
- 検証タスクが合格していること（全手順の警告ゼロ）
- `plan-implementer`以外の最終調整が完了していること
 （`spec-driven`文脈では`spec-writer`による作業版ドキュメントの立ち上げを含む）

### 初回レビュー（3種別並列）

計画ファイルの`## 実装・検証・レビュー`節「レビュー実施方針」項目を確認し、起動対象の種別を特定する。
種別`spec`（仕様適合性）・`code`（コード品質）・`docs`（ドキュメント単体品質）を同一メッセージ内で並列呼び出しする。
各種別はレビュー対象が異なるため独立に進められる。

起動要否は計画時に決定済みであるが、実装フェーズでの追加変更（ドキュメントの新規追加・削除など）で
判断が変わる場合は、メイン判断で追加起動・省略を調整してよい。
ただし以下の変更では種別`docs`を省略しない。

- `spec-driven`文脈での作業版ドキュメント追加
- 恒常化先（`CLAUDE.md`・`.claude/rules/`・`.claude/skills/`・プロジェクトドキュメント）への反映を伴う変更

### plan-reviewer呼び出しテンプレート（初回）

種別`spec`・`code`・`docs`で共通の骨子を使い、種別固有の欄を差し替える。
初回レビューでは対象範囲の指摘を出し切ること。再レビュー（`followup`）では新規指摘を追加しない前提で、
この初回で見つかった問題をすべて列挙する。

```text
以下の実装について{レビュー種別名}レビューを実施してください。

レビュー種別: {spec | code | docs}
計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
対象フェーズ: {単一フェーズ計画では「全体」、複数フェーズ計画では「フェーズN: 題目」}
対象外ファイル: {一時ファイル一覧}

{spec-driven文脈のみ・全種別共通}
作業テーマ名: {作業テーマ名}
作業版ドキュメント: `docs/v{next}/{作業テーマ名}.md`

{spec種別かつspec-driven文脈のみ}
差分内の他の作業版ドキュメント・`README.md`・横断ドキュメント:
{同一開発中バージョンディレクトリ配下の他作業テーマ・横断・README.mdのうち差分に含まれるもの}
恒常配置側の該当ドキュメント: `docs/features/{機能名}.md` または `docs/topics/{トピック名}.md`（既存改修時のみ）
改修前の該当節スナップショット: `docs/v{next}/{作業テーマ名}.md`内の「改修前スナップショット」節（既存改修時のみ）

{種別固有・注記}
事前呼び出しスキルは下記差分テーブルの該当種別に従う。

差分取得: `git diff HEAD`と、未追跡ファイルを含む作業ツリー参照で行うこと
制約: Bashは読み取り系操作に限定すること。ファイルへの書き込みは行わないこと。
実装者レポートは鵜呑みにせず、コードと差分で独立検証すること。gitコミット・pushは行わないこと。
戻り値: 指摘一覧のMarkdown（指摘なし時は「指摘なし」と記載すること）
```

レビュー種別ごとの差分:

| 種別 | 事前呼び出しスキル | 担当観点 |
| --- | --- | --- |
| `spec` | なし | 仕様適合性・複数ドキュメント間整合性 |
| `code` | `coding-standards` | コード品質 |
| `docs` | `writing-standards` | ドキュメント単体品質 |

### 指摘の統合と差し戻し

各レビュアーの戻り値本文を受け取り、メインが指摘を統合する。
統合時は重複する指摘をまとめ、種別間で矛盾する指摘があれば`spec`を優先して調停する。

指摘が1件以上あれば該当タスクを`plan-implementer`へ差し戻す。
差し戻しプロンプトの「差し戻し指摘」欄に統合済み指摘テキストをそのまま埋め込む
（ファイル経由ではなくプロンプトへの直接埋め込みとする）。

統合後、対応可否が方針レベルで割れる指摘やスコープ外として扱うべき指摘がある場合は、
差し戻し前にユーザーへ提示して判断を仰ぐ。
ユーザーが「対応不要」と判断した指摘は、判断理由とともに統合済みテキスト内の当該指摘の行末尾に
`（ユーザー判断: 不対応 — <理由>）`の書式で注記する。
差し戻しプロンプトの「差し戻し指摘」欄には対応する指摘のみを埋め込み、
不対応と判断された指摘は除外する。
計画ファイル`## 変更履歴`への記録は従来通り行う。

なお、統合済みテキスト（注記付きで保持）と差し戻しプロンプトへ埋め込むテキスト（対応指摘のみ）は別管理となる。

### 再レビュー（followup）

差し戻し後の再実装が完了したら、検証タスクを再起動して合格を確認する。
その後`plan-reviewer`（種別`followup`・モデル`haiku`）を1件起動し、前回統合指摘への対応状況のみを確認する。

再レビューでは新規指摘を原則追加しない（明らかな回帰が発見された場合のみ末尾に短く併記する）。

### plan-reviewer呼び出しテンプレート（followup）

「前回統合指摘:」欄には、前節「指摘の統合と差し戻し」で管理している統合済み指摘テキストを転記する。
各指摘行末尾に`（ユーザー判断: 不対応 — <理由>）`注記が付いている指摘は、
`followup`レビュアーの対応確認対象から除外される前提で、注記付きのまま転記する。

転記例:

```text
- `path/to/file.py:L10` — 戻り値の型アノテーションが欠けている
- `path/to/file.py:L20` — 変数名が不明瞭 （ユーザー判断: 不対応 — 現状仕様で問題なし）
```

```text
以下の実装についてfollowupレビューを実施してください。

レビュー種別: followup
計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
対象フェーズ: {単一フェーズ計画では「全体」、複数フェーズ計画では「フェーズN: 題目」}
前回統合指摘:
{メインが統合した前回指摘テキストをここに直接貼り付ける。
各指摘行末尾に「（ユーザー判断: 不対応 — <理由>）」注記が付いている指摘は対応確認の対象外。
}

差分取得: `git diff HEAD`と、未追跡ファイルを含む作業ツリー参照で行うこと
制約: Bashは読み取り系操作に限定すること。ファイルへの書き込みは行わないこと。
gitコミット・pushは行わないこと。
戻り値: 対応済み/部分対応/未対応の一覧と未対応件数
```

### 差し戻しループのエスカレーション

`followup`で未対応指摘が2回以上残る場合はループを中断してユーザーへエスカレーションする。

再レビュー・再検証の要否判定基準は`plan-mode`スキルの「レビュー実施方針の判断基準」節を参照する。
計画ファイルの「レビュー実施方針」で省略が指示されているレビュアーは、再実行時も省略してよい。

## フェーズごとのコミット

全指摘が解消したら、計画ファイルの`## 実装・検証・レビュー`節「コミット方針」項目に従ってメインがコミットする。
単一フェーズ計画ではこのコミットで作業完了。
複数フェーズ計画ではフェーズ単位の中間コミットになり、コミット後に次フェーズの実装フェーズへ戻る。

「コミット方針」が計画ファイルに記載されていない場合は、配布ルール`~/.claude/rules/agent-toolkit/agent.md`の
「検証とコミット」節の原則に従う。

ユーザーから「コミットはしない」「後でまとめてコミットする」などの明示指示がある場合のみコミット工程を省略する。
