# spec-driven導入ワークフロー

現行版ドキュメント（`docs/features/`・`docs/topics/`）の初版を整備する。

## 手順

1. `spec-driven`スキルを呼び出し、用語・配置規約・テンプレートを確認する
2. 既存コードと既存ドキュメントを確認する
3. 機能ドキュメントと横断ドキュメントの候補を整理する
4. ユーザーとドキュメント単位を合意する
5. `docs/features/{機能名}.md`または`docs/topics/{トピック名}.md`を作成する
6. コード中の参照コメントを現行版ドキュメントのパスへ書き換える
7. リンクと参照コメントの指し先を検証する

## 方針

- ドキュメント単位は利用者視点の機能を基準に決める
- 複数機能で共有する設計判断は`docs/topics/`へ置く
- 既存資料から背景、スコープ、受け入れ基準、主要設計判断、却下した代替案を抽出する
