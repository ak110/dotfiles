# spec-driven導入ワークフロー

現行版ドキュメント（`docs/features/`・`docs/topics/`）の初版を整備する。

## 手順

1. `agent-toolkit:spec-driven`スキルを呼び出し、用語・配置規約・テンプレートを確認する
2. 既存コードと既存ドキュメントを確認する。
   背景・スコープ・設計判断などに関する情報が不足する場合は、推測で進めず先にユーザーへ確認する
3. 機能ドキュメントと横断ドキュメントの候補を整理する（暫定で可）
4. ユーザーとドキュメント単位を合意する。合意が得られるまで手順5以降に進まない
5. 合意した単位で`docs/features/{機能名}.md`または`docs/topics/{トピック名}.md`を作成する
6. コード中の参照コメントを現行版ドキュメントのパスへ書き換える
7. リンクと参照コメントの指し先を検証する

## 方針

- ドキュメント単位は利用者視点の機能を基準に決める
- 複数機能で共有する設計判断（例: 複数モジュールから参照されるリトライ戦略、
  全機能が依存するデータモデルなど）は`docs/topics/`へ置く。
  単一機能内で閉じる設計判断はその機能ドキュメント内に書く
- 既存資料から背景、スコープ、受け入れ基準、主要設計判断、却下した代替案を抽出する
- 本スキルは現行版ドキュメントの初版整備のみを担当する。`docs/v{next}/`配下は`agent-toolkit:spec-driven`スキルの担当
