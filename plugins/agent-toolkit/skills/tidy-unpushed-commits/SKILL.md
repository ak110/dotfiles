# 未プッシュコミットの整理

## このスキルの目的

未プッシュコミットの整理を慎重で再現性のある手順で行い、修正単位が壊れる事故を構造的に防ぐ。
`git reset --hard` の後に一括ステージしてコミットしなおすような乱暴な手順は禁止する。
cherry-pick連鎖を中核とし、退避タグとコンテンツハッシュ検証で安全性を担保する。

## 呼び出しモードと既定動作

本スキルはユーザーがコミット整理を明示的に依頼したときのみ使う。
積極的な自動トリガーは行わない。
`/tidy-unpushed-commits` スラッシュコマンドでの明示呼び出し、またはユーザーが「コミットをまとめたい」などと明確に指示した場面が対象である。
軽微なコミットが溜まっているように見えるだけで勝手に起動してはならない。

呼び出された後は、後述の自動判定で対象範囲と整理計画を組み立てる。
組み立てた計画はユーザー承認を経てから実行する。

明示的な細部指示がある場合はそれを優先する。
たとえば特定範囲のみ整理する、reorderせずsquashだけする、特定コミットを除外するなどの指示があればそちらに従う。

自動判定の結果を実行に移す前に、必ず整理計画をユーザーに提示して承認を得る。

## 適用範囲と前提条件

- 未プッシュコミットのみを対象とする
- プッシュ済みコミットには絶対に触れない
- merge commitを含まないリニアな履歴のみ対象とする
- upstreamが設定されており、`@{u}` が解決できること
- Conventional Commits接頭辞を慣用的な解釈で運用していること
- 無害種別の相対位置入れ替えで意味が壊れないリポジトリであること

## 使わない判断基準

以下のいずれかに該当する場合は本スキルを使わず、ユーザーに手動対応を促す。

- 整理対象範囲にmerge commitが1つでも含まれる
- Conventional Commitsの慣用的解釈で接頭辞を運用していない
- 整理対象が20コミットを超える
- 整理対象範囲にpush済みコミットが含まれる

## 前提確認

整理を開始する前に必ず以下を順番に実行する。

```bash
git status
git -c fetch.prune=false -c fetch.pruneTags=false fetch --quiet
git log --oneline --decorate @{u}..HEAD
BASE="$(git merge-base HEAD @{u})"
git rev-list --merges "$BASE..HEAD"
```

- `git status` でworktreeとindexの状態を確認する
- `git fetch` でupstreamを最新化する。`fetch.prune`と`fetch.pruneTags`はローカル限定の参照（前回の整理で残った退避refなど）を巻き込みで消しうるため、このfetchに限り明示的に無効化する
- `git log --oneline --decorate @{u}..HEAD` で整理対象範囲のコミット一覧を確認する
- 各コミットに `origin/<branch名>` の参照が付いていればプッシュ済みである。プッシュ済みコミットには触れない
- `BASE` は整理対象範囲の起点である。これより先は絶対に触らない
- `git rev-list --merges` の出力が空でなければ、範囲にmerge commitが含まれる。即中断してユーザーに報告する

## uncommitted changes の取り込み

worktreeまたはindexに変更がある場合、`git diff` などで内容を提示してユーザーに判断を仰ぐ。
自動分類してはならない。

このとき内容を踏まえた推奨動作を必ず添える。
たとえば変更内容が軽微で無害種別に見えるなら「無害として取り込むことを推奨する」と示す。

ユーザーの回答に応じて以下のいずれかに分岐する。

- 無害として取り込むと判断した場合、WIPコミットとして暫定的にコミットし整理対象範囲に組み込む

    ```bash
    git add -A
    git commit -m "wip: tidy-unpushed-commits intake"
    ```

    WIPコミットは退避タグより前に作成する。
    退避タグは整理開始時点の真の起点を指す必要があるためである。
    取り込んだWIPコミットは整理サイクル内で他のコミットとまとめられる
- 機能変更が混じっていると判断した場合、スキルを中断する。先に意味のある単位でコミットしてから再度呼び出してほしい、とユーザーに伝える
- 整理サイクルから除外すると判断した場合、`git stash --include-untracked` で退避し、整理完了後に `git stash pop` で戻す

## 対象コミットの選定

整理対象範囲の各コミットをコミットメッセージから以下の3カテゴリに分類する。
判断基準はシンプルに保ち、迷ったらユーザーに聞く方針を徹底する。

### 無害コミット

接頭辞が `docs:` / `chore:` / `ci:` / `build:` / `style:` のいずれかであるコミット。
スコープ付きの `docs(readme):` なども同様に扱う。

整理上の扱い: squash対象。機能コミットを跨いで自由にreorderしてよい。

### 機能コミット

接頭辞が `feat:` / `fix:` / `perf:` のいずれかであるコミット。

整理上の扱い: 相対順序を変えない。基本はsquashもしない。
ただし後述のペア例外に該当する場合はsquashを提案できる。

### 怪しいコミット

上記のどちらにも該当しないコミット。
具体的には次のようなものが該当する。

- 接頭辞がないコミット
- 接頭辞が `refactor:` や `test:` のコミット
- メッセージが意味を持たない雑コミット。たとえば句点のみや `wip` 相当、空白のみなどのものを指す

これらは一律に自動分類せず、整理計画の提示時にユーザーへ個別確認する。

#### 怪しいコミットのユーザー確認

怪しいコミットは `git show` で内容を提示し、変更内容を踏まえた推奨動作を添えてユーザーに選んでもらう。

推奨動作の例は以下のとおりである。

- 内容が明らかにドキュメント修正やCI設定なら「無害コミット側に取り込む」を推奨する
- 直前または直後の機能コミットと同じ範囲のリファクタリングやテスト追加なら「その機能コミットにsquashする」を推奨する。この例外は後述のペア例外として扱う
- それ以外で意味ある独立変更なら「機能コミットとして位置固定で扱う」を推奨する
- 整理の必要がない変更なら「この整理サイクルから除外する」を推奨する

最終判断はユーザーが行う。回答があるまで自動処理してはならない。

### 機能コミットとのペア例外

機能コミットに続く `refactor:` や `fix:`, `test:` など補助的なコミットが同一論理単位である場合、冗長な分割になっている。
この場合、機能コミットにまとめる方が履歴が簡潔になる。

検出方法は以下のとおりである。

- 機能コミットの直前または直後に隣接する怪しいコミットがあり、メッセージや変更内容から同じ論理単位と見受けられる場合、このペアを検出する
- 判定が曖昧なので自動squashせず、整理計画の提示時に「この機能コミットに続くリファクタリングをまとめる推奨です」とユーザーへ確認する
- ユーザーが承認した場合のみ、機能コミット側にsquashする。メッセージは元の機能コミットのものを引き継ぐか、ユーザーが選んだ草案を使う

このペア例外は怪しいコミットのユーザー確認と一体で運用する。

## squashグルーピングの機械的提案

以下の2種類のグループ候補を提案する。

1. 無害コミット同士のグループ。接頭辞の種別または主要な対象ディレクトリが同じものをまとめる
2. 機能コミットとのペア例外。機能コミットに続く怪しいコミットが同一論理単位と見受けられる場合、その機能コミットにまとめる候補を提案する

機能コミットどうしのsquashは提案しない。

## ユーザーによるグループ単位の個別承認

種別やディレクトリが同じでも、別々の論理単位を誤ってまとめれば履歴の品質は下がる。
機械的提案だけではこのリスクを排除できない。
そのため、各squashグループはユーザーの明示承認なしに統合してはならない。

整理計画の提示時は以下を行う。

1. 各候補グループについて、グループ内の元コミットそれぞれの `git show --stat` の概要を並べて表示する。必要ならフルのdiffも提示する
2. グループごとに想定される最終コミットメッセージの草案を提示する
3. ユーザーに以下の選択肢から選んでもらう
    - 承認: そのグループを提案通りに統合する
    - 分割: グループ内のコミットを部分集合に分けて別々のグループとして扱う。どのコミットを1つのグループとするかはユーザーが指示する
    - 除外: そのグループの統合を取りやめる。元のコミットはそのまま位置固定で残す
4. すべての候補グループに対する承認、分割、除外が確定するまで実行フェーズに進まない

実行フェーズで統合されるのはユーザーが明示的に承認したグループだけである。
分割または除外されたコミットは元の位置のまま保たれる。

## 整理計画の提示テンプレート

ユーザー承認を求めるときは、以下の形式で整理計画全体を1枚にまとめて提示する。

### 提示時の原則

- コミットへの参照は必ず「連番 + subject」の形式で行う。ハッシュ単独での参照は禁止（ユーザーが識別できないため）
- gitコマンドをそのまま列挙してユーザーに見せない。実行詳細は隠し、計画の意味のみを示す
- 各squashグループは「元コミット列」「推奨理由」「統合後メッセージ草案」「判断選択肢」の4要素を必ず含める
- ユーザーへの質問は箇条書きで番号を振り、「yes/no」または選択式で答えられる形にする

### テンプレート

```text
整理計画:

【元順序】
1. feat: ユーザー認証を追加
2. docs: READMEにセットアップ手順を追記
3. refactor: 認証モジュールの関数を整理
4. fix: NPE対策
5. docs: CHANGELOG更新
6. ci: workflowのタイムアウト延長

【新順序（提案、<N>コミットに集約）】
1. feat: ユーザー認証を追加           ← 元1「feat: ユーザー認証を追加」
                                        + 元3「refactor: 認証モジュールの関数を整理」ペアsquash
2. fix: NPE対策                      ← 元4「fix: NPE対策」位置固定
3. docs: READMEとCHANGELOGを更新     ← 元2「docs: READMEにセットアップ手順を追記」
                                        + 元5「docs: CHANGELOG更新」無害squash
4. ci: workflowのタイムアウト延長     ← 元6「ci: workflowのタイムアウト延長」そのまま

【squashグループ】
Group A（ペア例外）
  元コミット:
    - 元1: feat: ユーザー認証を追加
    - 元3: refactor: 認証モジュールの関数を整理
  推奨: 機能コミット側に統合(feat接頭辞を維持)
  統合後メッセージ草案:
    feat: ユーザー認証を追加
  判断: 承認 / 分割 / 除外 ?

Group B（無害docs）
  元コミット:
    - 元2: docs: READMEにセットアップ手順を追記
    - 元5: docs: CHANGELOG更新
  推奨: 同種接頭辞かつ関連ドキュメントの無害squash
  統合後メッセージ草案:
    docs: READMEとCHANGELOGを更新
  判断: 承認 / 分割 / 除外 ?

【確認事項】
1. Group A を承認してよいか?
2. Group B を承認してよいか?
3. 新順序の並び（feat → fix → docs → ci）でよいか?
```

ユーザーが全グループに対して判断を返した後、実行フェーズに入る。

## 実行手順

### ステップ1: 退避refの作成

```bash
TS="$(date +%Y%m%d-%H%M%S)"
BACKUP_REF="refs/tidy-backup/${TS}"
git update-ref "${BACKUP_REF}" HEAD
```

退避refはこの後の全操作の起点である。
`refs/tidy-backup/` は `refs/tags/` でも `refs/remotes/` でもない独自名前空間のため、`fetch.prune` / `fetch.pruneTags` のどちらにも影響されない。
バックグラウンドで `git fetch` が走っても消えず、復旧時の安全な基準点になる。

タグやブランチを使わない理由は以下のとおりである。

- タグ（`refs/tags/`）: `fetch.pruneTags=true`が設定された環境下ではバックグラウンドfetchで消える
- ブランチ（`refs/heads/`）: 通常の`git branch`出力に紛れ、ユーザーが気付かずに削除・上書きする恐れがある

### ステップ2: 元コミットのコンテンツハッシュ収集

```bash
git log --reverse --format=%H "$BASE..${BACKUP_REF}" | while read sha; do
  hash=$(
    git diff --binary --no-color --no-renames --full-index \
      --src-prefix=a/ --dst-prefix=b/ "${sha}^..${sha}" \
      | sha256sum | cut -d ' ' -f 1
  )
  printf "%s %s\n" "$hash" "$sha"
done > /tmp/tidy-orig.txt
```

`git patch-id` は空白差分を無視するため使わない。
本スキルでは空白のみの変更も別物として識別する必要がある。
シェル関数ではなくインラインで書いているのは、スキル本文がLLMのコンテキストへ読み込まれる際に `$1` のような位置パラメーターが意図せず展開されてしまう副作用を避けるためである。

### ステップ3: 現在のブランチ名を記憶

```bash
BRANCH="$(git symbolic-ref --short HEAD)"
```

### ステップ4: detached HEADで再構成を開始

```bash
git checkout --detach "$BASE"
```

以降、作業はdetached HEAD上で進める。
元のブランチ参照は付け替えの最後まで動かさない。

### ステップ5: 承認された順序でcherry-pick

承認済みの新順序に従って元コミットを1つずつcherry-pickする。

```bash
git cherry-pick <元コミットSHA>
```

squashグループを合流させる場合は、グループの最初のコミットをcherry-pickした後、残りを `--no-commit` でcherry-pickして `git commit --amend` で合流する。

```bash
git cherry-pick <先頭コミットSHA>
git cherry-pick --no-commit <2番目コミットSHA>
git commit --amend -m "docs: READMEとCHANGELOGを更新"
```

### conflict発生時の挙動

- reorder起因のconflictが発生した場合。該当箇所のreorderのみ諦める。`git cherry-pick --abort` で巻き戻し、当該コミットを元の相対位置に戻して進行する。reorder全体は中断しない
- squash起因のconflictが発生した場合。自動解決を試みず、即座にユーザーに報告する。`git cherry-pick --abort` で中断し、退避タグからロールバックして整理計画を再検討する

`git checkout --theirs` や `--ours` で安易に解決してはならない。修正単位が壊れる。

### ステップ6: ブランチ参照の付け替え

すべてのcherry-pickが完了したら、元ブランチの参照を新HEADに付け替える。

```bash
git branch -f "$BRANCH" HEAD
git checkout "$BRANCH"
```

## 検証手順

### ステップ7: 整理後コミットのコンテンツハッシュ収集

```bash
git log --reverse --format=%H "$BASE..HEAD" | while read sha; do
  hash=$(
    git diff --binary --no-color --no-renames --full-index \
      --src-prefix=a/ --dst-prefix=b/ "${sha}^..${sha}" \
      | sha256sum | cut -d ' ' -f 1
  )
  printf "%s %s\n" "$hash" "$sha"
done > /tmp/tidy-new.txt
```

### ステップ8: 境界保存の確認

承認済みの整理計画と突き合わせて以下を1つずつ確認する。

- squashされない元コミット。そのコンテンツハッシュが新側のいずれか1つのコミットのハッシュと完全一致しなければならない。reorderされていてもよい
- squashされた新コミット。親基準を揃えた再現比較を行う

squashされた新コミットの再現比較は以下の手順で行う。

1. 新コミットの実際の親を `git rev-parse` で取得する
2. detached HEADでその親をチェックアウトする
3. 計画上の対応する元コミット群を `git cherry-pick --no-commit --allow-empty` で順次stageする。すべてstageし終えたら `git commit --allow-empty -F <統合後メッセージファイル>` で1コミットに統合する。新コミットと同じ親を持つ単一コミットとして再現する必要があるため、途中で `git commit` してはならない
4. 統合後のコミットをステップ2と同じ手順でコンテンツハッシュ化し、新コミットのハッシュと比較する
5. 完全一致しなければ検証失敗

検証用の一時操作はすべてdetached HEAD上で完結する。
退避タグや作業ブランチには一切触れない。

### ステップ9: 最終ツリー比較

```bash
git diff "${BACKUP_REF}..HEAD"
```

出力が空でなければ事故とみなし、即ロールバックする。
コンテンツハッシュ比較と独立した裏取りとして機能する。

この比較が空である限り、整理後の最終ツリーは整理前と完全一致している。
したがって `make test` などの追加のテスト実行は不要である。
中間コミットの状態は変わるため厳密にはbisect可能性に影響しうるが、最終状態の等価性が機械的に証明されている以上、本スキルの責務はここで完結する。

### ステップ10: 退避refの削除

上記の検証がすべて成功したら退避refを削除する。

```bash
git update-ref -d "${BACKUP_REF}"
```

pushは別作業として手動で行うケースが多いため、退避refを残し続けるとリポジトリにゴミが溜まる。
検証が機械的に成立している以上、追加の安全マージンとしてrefを残す必要はない。

## 失敗時の復旧

検証失敗やcherry-pick中の致命的なエラー発生時は以下の順序で復旧する。

```bash
git cherry-pick --abort 2>/dev/null || true
git status
git stash --include-untracked 2>/dev/null || true
git checkout "$BRANCH"
git reset --hard "${BACKUP_REF}"
git stash pop 2>/dev/null || true
```

退避refは成功するまで削除しない。
復旧完了後、ユーザーに失敗原因を報告して整理計画を再検討する。

## 絶対に避ける操作

本スキルでは以下の操作を禁止する。修正単位を壊す直接の原因になるためである。

- NG: `git reset --hard` でworktreeごと巻き戻す。進行中の変更が消える。退避タグからの復旧の場合を除く
- NG: `git reset --soft` でupstreamまで戻したあとの一括ステージと一括コミット。元の修正単位が失われる
- NG: `git add -A` でまとめ直す。分離していた変更が混ざる。WIPコミット作成時のみ例外
- NG: `git push --force` や `--force-with-lease` の使用。プッシュ済みコミットへ波及する恐れがある
- NG: `git rebase --root` の使用。範囲が広すぎて事故が拡大する
- NG: 退避タグ作成前の操作。復旧不能な状態で進行することになる
- NG: conflict時の `git checkout --theirs` や `--ours` による安易な解決。修正単位が壊れる
- NG: `git commit --amend` でプッシュ済みコミットを書き換える。force pushが必須になる
