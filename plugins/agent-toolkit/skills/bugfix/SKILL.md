# バグ・障害対応

ユーザーからのバグ報告やIssueへの対応時は、plan modeを使用して以下の4ステップをこの順序で必ず扱う。
全て必須であり、省略してはならない。
着手前に必ずTaskCreateでタスク登録すること。各ステップ完了時には、実施内容・確認結果・残存リスクの有無をユーザーへ報告すること。

## 1. 根本原因の特定

- 事象の現象面を固定する。少なくとも、期待値、実際値、発生条件、再現手順、発生頻度、初出時期を明確にする
- 時系列、ログ、メトリクス、変更履歴、設定差分、関連Issue/PRを確認し、事実関係を固定する
- 影響範囲を把握する。波及する機能、データ、ユーザー操作、外部連携、権限、環境差異を洗い出す
- 発生原因と検知失敗原因を両方特定する
  - 発生原因: なぜ誤りを作り込んだか
  - 検知失敗原因: なぜ既存のテスト、レビュー、監視、運用で検知できなかったか
- トリガー、根本原因、寄与要因を区別して整理する。複合要因であれば全て列挙する
- 5 Whysを、発生、検知、影響拡大、復旧遅延の各観点で適用し、表面的な症状で止めず上流要因まで追跡する
- 原因仮説ごとに、支持事実と棄却事実を整理し、最終的に採用した原因を明示する

## 2. 対策方法の決定

- 対症療法だけで済ませず、必ず根本療法を含める
- 多層防御の観点から、以下を必要十分な範囲で検討する
  - 発生防止
  - 早期検知
  - 影響緩和
  - 復旧容易化
- 対策案ごとに、期待効果、副作用、適用範囲、未対応となるリスクを整理する
- 修正方針には、必要に応じてコード修正、設定修正、データ補正、移行、監視追加、アラート条件見直し、運用手順修正、ドキュメント更新を含める
- 過剰な対策は避けるが、同種不具合の再発確率と影響度に照らして妥当な防御水準を確保する
- 正常系だけでなく、異常系、境界値、権限差異、環境差異、並行実行、互換性影響も必要に応じて確認対象へ含める

## 3. 類似箇所の見直し

- 同じ原因や実装パターンによる不具合が、コード、設定、テスト、ドキュメント、監視設定、データ処理の他の箇所に残っていないか確認する
- 共通処理、派生実装、コピー実装、類似条件分岐、同一インターフェース利用箇所を重点的に確認する
- 根本原因の修正で自動的に解消される見込みがある場合も、影響範囲を確認し、その結果を報告する
- 類似箇所を修正しない判断をした場合は、非該当理由を明記する

## 4. 再発防止策の検討

多層防御の考え方に基づき、少なくとも以下3軸で対策を講じる。

### 発生を防ぐ対策

- 設計、インターフェース、型、バリデーション、責務分離、デフォルト値、設定構造、レビュー観点を見直し、同種のミスが起きにくい構造へ改善する
- 特定技術やライブラリの誤用が原因であれば、参照すべき一次資料へのリンク、注意点、禁止事項をコメントまたはドキュメントへ明記する
- 人の注意に依存する対策のみで終わらせず、可能ならコードや仕組みで防ぐ

### 検知を早める対策

- 再現手順をテストコード化し、回帰テストとして恒久化する方針を定める
- 追加テストは、修正前に失敗し、修正後に成功することを確認すべきものとして定義する
- 検知失敗原因に対応するテストレイヤーを選定する。既存レイヤーで不足した場合は、ユニット、結合、E2E、契約テスト、監視、静的解析、Lint、型検査の追加または強化を検討する
- 必要に応じて、ログ、メトリクス、トレース、アラートの追加または見直しを検討する

### 記録・共有

- 原因、対策、判断理由、非対象とした範囲、確認結果を、CLAUDE.md、運用手順書、設計書、コメント等の再利用される場所へ記録または記録方針として明示する
- 以下の場合は追記対象として必ず扱う
  - 同種のユーザー指摘が繰り返されている場合
  - 暗黙知だった判断基準を今回明文化した場合
  - 特定技術や運用上の落とし穴が原因だった場合
  - 今後の保守で再度誤りやすい箇所だと判断される場合

## 共通原則

- 推測で修正してはならない。事実、再現結果、ログ、コード、テスト結果に基づいて判断すること
  - plan mode時も調査のためのコマンド実行などは行ってよい
- 修正方針は、修正前に失敗を確認すべきテストまたは再現手順と対で定義すること
- トリガー、現象、影響、根本原因、寄与要因、検知失敗原因を混同しないこと
- 恒久対策と暫定対策は区別して扱い、恒久対策を必ず含めること
- 影響範囲の確認対象には、コード、設定、データ、テスト、ドキュメント、監視、運用手順を含めること
- 修正対象外とした範囲がある場合は、その理由を明記すること

## 完了条件

- 根本原因、寄与要因、検知失敗原因の仮説と根拠が整理されている
- 恒久対策を含む修正方針が定義されている
- 確認すべき影響範囲と類似箇所が列挙されている
- 追加または更新すべきテスト、監視、ドキュメントの方針が明示されている
- 実装時に行う確認項目と残存リスクが報告されている
