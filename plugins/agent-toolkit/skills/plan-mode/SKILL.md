# 計画モード・計画ファイル作成

plan modeで計画ファイルを作成する際の構成・codexレビュー手順・反復運用をまとめる。

ユーザーの指示がバグ・障害・イシューの調査・修正・改善に該当する場合、
他のセクションより先に`## バグ対応`節と`references/bugfix.md`を必ず参照する。

## バグ対応（最優先で確認）

ユーザーの指示がバグ・障害・エラー・デグレードなどに関する調査・修正・改善に該当する場合、
他セクションを読む前に`references/bugfix.md`を読み込むこと。
そのうえで「原因特定→修正と再発防止→類似箇所→知見記録」の4ステップをすべて確実に実施する。
再現の有無や発覚経路（CI失敗・テスト失敗・例外発生・挙動変化の報告など）は問わない。

該当する場合、本スキル呼び出し直後の最初のツール呼び出しは`Read`で`references/bugfix.md`を開くことに固定する。
それ以前に計画ファイルの作成や他セクションの解釈へ進んではならない。
この順序を守らないと、構成テンプレートを先に読んだ勢いで対症療法的な計画に陥りやすく、再発防止の検討が欠落する。

## plan mode未進入時の進め方

本スキルのトリガー条件には通常モードから呼ばれる経路がある。
例えば「複雑な指示・多段階の作業を受けたとき」「バグ・障害・エラー等の調査相談を受けたとき」である。
現在plan modeに入っていなければ、ユーザーに方針を伝えた上で`EnterPlanMode`ツールを呼び出してから本手順に入る。
計画ファイル作成は`EnterPlanMode`下で行うため。

## 計画着手前の要件対話

ユーザー指示が曖昧・自信なさげ・複数の解釈が成り立つ場合は、即座に計画ファイル作成へ入らない。
まず選択肢提示などで要件を対話的に明確化してから計画ファイル作成に進む。
曖昧さを残したまま骨子を書き始めると、計画ファイルへの追記とユーザー確認が多ターンに分散する。
その結果、トークン効率と最終的な計画品質の両方が低下するため。

- 対話で確定させる観点は以下に絞る（実装詳細には立ち入らない）
  - 目的・ゴール
  - 制約・前提
  - 成功条件
  - スコープ境界
  - 方針の選択肢
- 推奨手段は`AskUserQuestion`
  - 複数の論点はバッチでまとめて提示し、多ターンの段階的プロンプトを避ける（やり取りが分散するとユーザー側の負担も増えるため）
- 対話で得た結論は、計画ファイル作成時に以下へ転記する。エージェントの推測判断と混ぜない
  - `## 背景`（経緯と目的）
  - `## 対応方針 > ユーザー合意済み事項`（合意した方針）
- plan-mode自体のトリガー条件に該当しない場合（単純な単一ファイル修正・会話だけの質問など）はこの対話フェーズも省略してよい

## 計画ファイル作成の基本

- 計画ファイルは実装フェーズで単独参照される前提で書く。
  実装フェーズはコンテキスト圧縮・セッション境界などで計画時の会話履歴を失った状態から開始されることがあり、
  計画ファイルが唯一の入力源になりうるため、計画時点の判断はすべて計画ファイル内に書き切る
- 対応方針決定までに得た判断は、計画ファイルに書き残すだけでは将来のセッションに届かない
  - 対象: ユーザー合意・選択肢提示の結果・エージェントの既定判断
  - 計画ファイル自体は一時的な成果物であり、次回以降のセッションでは参照されない
  - そのため、同じ計画内に「恒久化先への反映作業」を必ず組み込む
  - 判定手順: `### ユーザー合意済み事項`と`### エージェント判断`の各項目について以下を毎回問う
    - 別のセッション・別の作業でも同じ判断が必要か
    - 放置すると同じユーザー確認が再発するか
    - 共通化しない設計判断を別の編集者に上書きされうるか
  - 該当するものは恒久化先を決めて、同じ計画の`## 変更内容`に更新作業として明記する
  - 反映先の候補（いずれも将来のセッションで自動的に読み込まれる場所を選ぶ）
    - プロジェクト方針なら`CLAUDE.md`
    - ファイル種別ごとの規約なら`.claude/rules/`
    - 特定タスク手順なら`.claude/skills/`
    - 利用者向け情報ならプロジェクトドキュメント（`docs/`配下など）
  - Noと判断した項目も、後で振り返れるよう判断理由を計画ファイル内に簡潔に残す
  - この工程は必須扱いとする。検討のみで終わらせず、実際の反映作業まで計画に含めて初めて「恒久化」と見なす
- plan modeの制約で必要な調査が実行できない場合、ユーザーに確認の上で先に実行する
- `## 調査結果`・`## 変更内容`の記述に入る前に、変更対象ファイルと周辺ファイルを通読し、
  既存の章構成・記述粒度・抽象度・命名規則を把握する
  - 例えばドキュメント追記では、対象ファイルを通読せずに書き始めると既存記述との粒度不整合を招きやすい
- ソースコードだけでなく、ドキュメントへの影響も必ず確認する。設計意図などの追記も検討する
- 検証は自動実行できる手順を最優先する（手動検証は再現性が低く、回帰テストとしても機能しないため）
  - 既存の自動チェック基盤で不足する場合は新たな仕組みの導入も検討する（大規模な変更になる場合はユーザー確認要）
  - 手動検証は自動化が技術的・コスト的に困難な場合に限定し、理由を計画ファイル内に明記する
  - 「修正前は失敗し、修正後は成功する」ことを観測可能にする手段を用意する
- 要件が曖昧で仕様を確定しにくい場合は、詳細な仕様化よりも軽量なプロトタイプ反復を提案する選択肢を検討する
  - 仕様化コストより動作例の方が意思決定を早める場合があるため
- 方針転換はユーザーへ可否を確認してから行う
- 計画ファイルもMarkdownのため、`writing-standards`スキルに従う
  - 1行の表示幅制限などMarkdown記述ルールを含む

## 計画ファイルの構成

以下の構成を厳守する。
人間は判断に関わる重点セクションを冒頭で確認でき、codexは実装レベルの照合材料を後半でまとめて参照できる。
両者の関心が混ざらない構造にするため。

1. `# タイトル`
2. `## 背景` — なぜこの変更をするか。経緯と期待する成果（人間向け）
3. `## 対応方針` — 変更内容・影響範囲・段取り・リスク扱い（人間向け）。必要に応じて表なども使い分かりやすく書く
   1. `### ユーザー合意済み事項` — ユーザーが発話または確認で合意した事項（エージェントの推測を混ぜず厳密に分離）
   2. `### エージェント判断` — ユーザー指示外でエージェントが既定した事項。差し戻し・見直しの候補
   3. `### 却下した代替案` — 別案と却下理由（codexの「なぜ別の方法を取らないのか」指摘を予防）
4. `## 調査結果` — 既存コードの挙動・依存・制約（codex向け照合材料）
5. `## 変更内容` — 変更対象ファイルの絶対パス、変更方針、参照箇所のfile:line（codex向け）。
   コード修正は抽象度を保ち、ドキュメント修正は完全な文面を含めてよい
  （詳細は後述の「構成を守るための注意」節を参照）
6. `## 実装・検証・レビュー` — `ExitPlanMode`後の実装・検証・レビュー工程に必要な計画固有の情報を記述する。
   詳細手順は`agent-toolkit:plan-implementation`スキルを参照する。
   本節には以下の項目を順に記述する
   - 「実装時に事前呼び出しが必要なスキル」リスト。
     `agent-toolkit:plan-implementation`は必ず含める。
     変更対象に応じて`agent-toolkit:coding-standards`・`agent-toolkit:writing-standards`・
     `agent-toolkit:spec-driven`などを追加列挙する（対象変更に含まれなければ列挙しない）
   - 「検証手順」リスト。その計画を完了してコミットする前に必要な検証を具体的に記述する。
     プロジェクト方針として定められた手順（format/lint/test一式など）と、計画固有の観測手順の両方を含める。
     自動化不可な手動確認が必要な場合は理由付きで明示する
   - 「レビュー実施方針」。既定では`plan-reviewer`の3種別（`spec`・`code`・`docs`）をすべて起動する扱いとする。
     省略する種別があれば種別名と省略理由を列挙する。
     全種別を省略する軽微実装の場合もこの項目に判断根拠を記述する。
     判断基準は後述の「レビュー実施方針の判断基準」を参照する
   - 「コミット方針」1行記述。コミット有無とコミットメッセージのsubject行を記述する。
     本文（subject行以降）は任意で2行目以降に添える
7. `## 変更履歴` — 前回更新からの差分を記録する（初回は`- 初版`のみ記載する）
   - codex指摘への対応と不対応見解、ユーザー指示による修正、エージェントの自発修正をまとめて追記する
   - 目的はcodexの再指摘回避と人間の差分レビュー支援
   - 構成が変わるなどして過去の指摘が意味をなしていない状態になったものは適宜削除する

### 構成を守るための注意

- セクション・サブセクションの追加や順序の変更は禁止
  - ユーザーへの確認事項は計画ファイル提示前に解消する運用のため、未決事項などは記述不可
  - 例外: `## 変更内容`・`## 実装・検証・レビュー`節内のフェーズ分割サブ節（`### フェーズN: 題目`）は許容する
   （詳細は後述の「複数フェーズ計画の記述方法」節）
- 冗長な説明や詳細は避け、必要な情報だけを簡潔に記述
- 情報を複数セクションに重複させない
- 連番や個数の表記は極力避ける（修正時の影響箇所が増えるため）
  - ただし構造定義など順序が本質的に意味を持つ列挙は連番を使ってよい
- 背景や方針で抽象的な宣言文（「機能的挙動は変えず」「最小差分で」など）を書く場合、変更内容と矛盾しないかを必ず確認する
  - 先回りで縮退的制約を課すと本質的な改善を阻害するため、ユーザー合意のある記述に限定する
- codexレビューに備えて対応方針・調査結果を十分に記述しておく
- コードの記述は最小限とし、codexがリポジトリ実体と照合できる粒度に留める
  - 実装時にエージェントが判断可能な細部（変数名・エラーメッセージ文言・ループ構造・条件分岐の内訳・
    小規模な制御フローなど）は計画ファイルに書かない
  - 計画ファイルには方針・境界・挙動・外部インターフェース・確定済みの制約のみを書く
  - 粒度の目安:「この記述を読んだ別の実装者が方針を誤解せず進められる」最低限まで
- `## ユーザー合意済み事項`にはユーザーの発話・確認済み合意事項のみを書く
  - エージェントの推測や既定判断は `## エージェント判断` に分離する
- `# タイトル`から`## 却下した代替案`までのセクションには実装レベルの記述を書かない
  - 実装レベルの記述の例: ファイル名・シンボル名・file:line・コード片・外部APIの具体値・特定ミドルウェアや設定キーなど
  - 判断材料としての抽象度を保つため、仕様・挙動・方針の粒度に留める
  - これらの詳細は`## 調査結果`と`## 変更内容`に書く
  - 例外: `## 変更内容`では、ドキュメント修正に限り変更後の完全な文面をコードブロック引用で含めてよい
   （評価可能性が必要な範囲で適宜選択する。コード修正は従来通り抽象度を保つ）
- `## 変更履歴` は更新単位ごとに項目を分け、その下に差分要約と関連する指摘・判断を並べる
  - 指摘に対応しない場合は該当エントリ内に理由を併記する（不要な再指摘を避けるため）
- 上記のセクション構成は`posttooluse` hookでも検査される（`scripts/posttooluse.py`の`_PLAN_REQUIRED_H2`）。
  検査対象は大見出し（`##`）のみで、フェーズ分割サブ節（`###`）は検査対象外

### 複数フェーズ計画の記述方法

大規模変更でフェーズ間にコミットを挟みたい場合、計画ファイル1本のなかでフェーズ分割を許容する。
分割しないシンプル計画と、フェーズ分割計画のいずれも本スキルで扱える。

- フェーズ分割の記述場所: `## 変更内容`と`## 実装・検証・レビュー`の2節の内側に
  `### フェーズ1: 題目`・`### フェーズ2: 題目` …のサブ節を並べる
  - フェーズ題目は端的に表す名詞句（例: `フェーズ1: 既存テスト整備`）
  - フェーズ番号は連番で振る
- 他の大見出し（`## 背景`・`## 対応方針`・`## 調査結果`・`## 変更履歴`）はフェーズ分割せず全体で記述する
- フェーズごとに、`## 変更内容`サブ節には当該フェーズで触る変更対象ファイルと変更方針を、
  `## 実装・検証・レビュー`サブ節には当該フェーズの検証手順とレビュー実施方針を書く。
  全フェーズ共通の手順や方針は親節（フェーズサブ節の外側）にまとめてよい
- 詳細は`plan-implementation`スキルの「複数フェーズの運用」節を参照する
- レビュアーへ渡す「対象フェーズ」入力にはサブ節タイトル（`フェーズN: 題目`）をそのまま使う

複数フェーズ計画の骨子例（`## 変更内容`・`## 実装・検証・レビュー`のみ抜粋）:

```markdown
## 変更内容

### フェーズ1: 既存テスト整備

- 修正: `tests/upload_test.py` のフィクスチャを共通化

### フェーズ2: 上限値の引き上げ

- 修正: `server/config.py` の上限定義を50MBへ
- 修正: `infra/nginx.conf` のbody size上限を50MBへ

## 実装・検証・レビュー

- 実装時に事前呼び出しが必要なスキル
  - `agent-toolkit:plan-implementation`
  - `agent-toolkit:coding-standards`

### フェーズ1: 既存テスト整備

- 検証手順
  - `uv run pyfltr run-for-agent`
- レビュー実施方針: `plan-reviewer`の`code`種別のみ起動。仕様変更を伴わないため`spec`種別は省略
- コミット方針: コミットする。subject: `test(upload): フィクスチャを共通化する`

### フェーズ2: 上限値の引き上げ

- 検証手順
  - `uv run pyfltr run-for-agent`
  - 50MB境界のテストが追加されており修正前は失敗・修正後は成功することを確認する
- レビュー実施方針: 既定通り`plan-reviewer`の3種別すべてを起動する
- コミット方針: コミットする。subject: `feat(upload): ファイルサイズ上限を50MBへ引き上げる`
```

### レビュー実施方針の判断基準

`## 実装・検証・レビュー`節の「レビュー実施方針」項目でレビュアーの起動要否を判断する際の基準。
判断に迷ったら省略せず起動する方針とする（見落としの損失が起動コストより大きいため）。
判断結果と理由は「レビュー実施方針」項目に書き下し、別節（`## 変更履歴`など）へ分散させない。

以下の条件を総合的に勘案し、省略が妥当かメインが判断する。

- 変更規模が小さい（数行〜数十行程度、1〜2ファイル）
- 影響範囲が限定的（公開インターフェースを変更しない、他モジュールへ波及しない）
- 既存パターンを踏襲している（類似実装が既に存在する、設計判断を伴わない）
- 判断の余地が少ない（単純な誤字修正・既知バグの明確な修正など）
- テスト済み・検証済み（自動テストで挙動が担保されている）

レビュアー別の観点例:

- `spec`種別: 仕様適合性・ドキュメント間整合性。実装が仕様と一対一で対応し判断の余地がない場合に省略候補
- `code`種別: コード品質。コード変更がmanifestやバージョン更新のみで実ロジック変更がない場合に省略候補
- `docs`種別: ドキュメント単体品質。ドキュメント変更が無いか、軽微な誤字修正のみの場合に省略候補

### 記述例

以下は全セクションを通した記述例。

```markdown
# ファイルアップロード上限を10MBから50MBへ引き上げ

## 背景

高解像度画像をアップロードできないという要望が継続して寄せられている。
ストレージと転送帯域の増強が完了し運用コストの懸念が解消されたため、上限値を引き上げて要望に応える。

## 対応方針

### ユーザー合意済み事項

- 上限値は50MBに変更
- 超過時のエラーメッセージ文言は既存のものを流用
- 経路上の各層に個別設定されているサイズ制限やタイムアウトを事前に確認し、整合確認を検証範囲に含める

### エージェント判断

- 上限値の定義を一元化してSSOTを確立
- 既存の超過判定とエラー通知の挙動は維持し、参照値のみを差し替え

### 却下した代替案

- ファイル種別ごとに上限を変える案 — 運用が複雑化する割に現時点で必要性が明確でないため
- 上限を撤廃する案 — 想定外のリソース消費や不正利用のリスクが残るため

## 調査結果

- サーバー側の上限定義: `server/config.py:34`
```

## codexレビュー

ユーザーに計画ファイルを提示する前に、codexでレビューする。
plan modeの制約下でもcodexレビューは制約対象外として扱える（計画ファイル自体のレビューのため）。

### MCPツール版（優先）

`mcp__codex__codex` / `mcp__codex__codex-reply` が利用可能なら優先して使う。

- 初回: `mcp__codex__codex` を以下のパラメーターで呼び出す
  - `cwd`: `"{project_directory}"`
  - `sandbox`: `"danger-full-access"`
  - `prompt`: 以下の文言をそのまま使う（CLIフォールバック時も同一文言を使用する）

    ```text
    {plan_full_path} この計画ファイルをレビューして。
    以下の手順で行うこと。
    1. 考えられる指摘候補を内部的に網羅列挙する
    2. そこから確実に問題である致命的かつ本質的な欠陥のみを残し最終応答とする
    3. 今回の応答で全て出し切り、再レビューで追加の新規指摘を出さない

    些末な指摘、不要な提案、推測による指摘は禁止。
    計画ファイルの記述スタイル（構成・表現・書き方）に対する指摘も禁止。（記述間の矛盾は除く）
    実装時にエージェントが判断可能な細部（変数名・エラーメッセージ文言・ループ構造・小規模な制御フローなど）への指摘は禁止。
    指摘する場合は修正方針も簡潔に併記すること。
    指摘が無い場合は指摘無しと応答すること。
    ```

- 2回目以降: `mcp__codex__codex-reply` を以下のパラメーターで呼び出す
  - `threadId`: 前回の戻り値から取得したthreadId
  - `prompt`: 以下の文言をそのまま使う（CLIフォールバック時も同一文言を使用する）

    ```text
    {plan_full_path} 計画ファイルを更新したので再レビューして。
    更新後の計画ファイルに対しても初回と同じ手順（候補の網羅列挙→致命的かつ本質的な欠陥のみ残す→今回の応答で全て出し切る）で検査すること。
    実装時にエージェントが判断可能な細部（変数名・エラーメッセージ文言・ループ構造・小規模な制御フローなど）への指摘は禁止。
    ```

レビュー結果は戻り値の `content` フィールドに含まれる。

### CLIフォールバック版

MCPツールが利用できない場合はcodex CLIを試す。
CLIの詳細コマンドは `references/codex-cli.md` を読み込んでから実行する。
対象は`codex exec` / `codex exec resume`のオプション・session id抽出手順などである。

### どちらも使えない場合

MCPもCLIも使えない場合は、計画ファイル提示前にユーザーに状況を報告し、以下の選択を仰ぐ。

- codex CLIを別途セットアップし、codex MCPをインストールしてやり直す（推奨）
  - インストールコマンド: `claude mcp add --scope=user codex codex mcp-server`
- codex CLIを別途セットアップしてやり直す
- 当該環境ではcodexレビューを永続的にスキップする旨をプロジェクトの `CLAUDE.local.md` に記載してスキップする
  - 当該ファイルはgitignore推奨

## codexレビューの進め方

- ユーザーへの確認事項がある場合はcodexレビュー前にユーザーに確認し結論を確定させる
  - 方針レベルの合意を実装詳細レベルの合意と誤解しないよう注意
- codexレビューでユーザー判断が必要とされた場合、その場ですぐにユーザーに確認して結論を確定させる
- codexの指摘がなくなるまで更新とレビューを繰り返す
  - 合否判断は呼び元が行ってよい（指摘が的外ればかりの場合なども合格扱いでよい）
- 一度指摘ゼロになった後に限り、その後の軽微な修正では再レビューを省略してよい
  - 方針レベルの変更時は再レビューする
- codexの指摘に基づいて計画ファイルを修正する際は、どのような修正をするかユーザーに伝える

## 実装・検証・レビューへの接続

`ExitPlanMode`後の実装・検証・レビュー工程は`agent-toolkit:plan-implementation`スキルに分離している。
本スキルは計画ファイル作成とcodexレビューに専念し、合意後はメインから`plan-implementation`を呼び出して
実装・検証・レビューを進める。
計画ファイルの`## 実装・検証・レビュー`節には、その計画に固有の情報
（事前呼び出しスキル・検証手順・レビュー実施方針・コミット方針）を集約する。
工程詳細は`plan-implementation`スキルが持つ。

## Gotchas

- 要件が固まる前に計画ファイルの骨子を書き始めがち
  - 曖昧さが残る段階では`AskUserQuestion`などで選択肢を確定させてから計画ファイル作成に進む
- 方針レベルの合意を実装詳細レベルの合意と取り違えやすい。ユーザーに確認するときは粒度（方針か実装詳細か）を明示する
- エージェントの推測判断を `## ユーザー合意済み事項` に混入させがち
  - 発話・確認済みの合意のみをそこに書き、推測や既定判断は `## エージェント判断` に分離する
- 抽象的な縮退宣言（「機能的挙動は変えず」「最小差分で」など）を先回りで書きがち
  - ユーザー合意のある範囲に限定する（本質的な改善を阻害するため）
- 連番や個数の表記を安易に使いがち。修正時に影響箇所が連鎖するため、順序が本質的に意味を持つ場合に限定する
- SKILL.md本文の構成テンプレートから書き始めてバグ対応4ステップを省略しがち
  - トリガーに該当する依頼では最初のツール呼び出しを`references/bugfix.md`の`Read`に固定する
  - 本スキル本文の他セクションはその後で解釈する
- 計画で決めた判断（`### ユーザー合意済み事項`・`### エージェント判断`）を計画ファイルに書いただけで終わらせがち
  - 再利用される判断は、将来のセッションで自動読み込みされる場所への反映作業を、同じ計画の`## 変更内容`に必ず組み込む
  - 反映先の例: CLAUDE.md・ルール・スキル・プロジェクトドキュメント
- 計画ファイルの`## 変更内容`に実装細部（変数名・エラーメッセージ文言・小規模な制御フローなど）まで書き込みがち
  - 方針・境界・挙動・外部インターフェース・確定済みの制約のみを書く。細部は実装エージェントの裁量に任せる
