# 計画モード・計画ファイル作成

plan modeで計画ファイルを作成する際の構成・codexレビュー手順・反復運用をまとめる。

ユーザーの指示がバグ・障害・イシューの調査・修正・改善に該当する場合、
他のセクションより先に`## バグ対応`節と`references/bugfix.md`を必ず参照する。

## バグ対応（最優先で確認）

ユーザーの指示がバグ対応などの場合、他セクションを読む前に`references/bugfix.md`を読み込み、
「原因特定→修正と再発防止→類似箇所→知見記録」の4ステップをすべて確実に実施する。

この手順を省略すると対症療法に留まりやすく、再発防止の検討が欠落する。
`references/bugfix.md`を参照しないまま計画ファイル作成に進まないこと。

## plan mode未進入時の進め方

本スキルのトリガー条件には「複雑な指示・多段階の作業を受けたとき」「バグ・障害・エラー等の調査相談を受けたとき」のように通常モードから呼ばれる経路がある。
現在plan modeに入っていなければ、ユーザーに方針を伝えた上で`EnterPlanMode`ツールを呼び出してから本手順に入る（計画ファイル作成は`EnterPlanMode`下で行うため）。

## 計画着手前の要件対話

ユーザー指示が曖昧・自信なさげ・複数の解釈が成り立つ場合は、即座に計画ファイル作成へ入らず、選択肢提示などで要件を対話的に明確化してから計画ファイル作成に進む。
曖昧さを残したまま骨子を書き始めると、計画ファイルへの追記とユーザー確認が多ターンに分散し、トークン効率と最終的な計画品質の両方が低下するため。

- 対話で確定させる観点は以下に絞る（実装詳細には踏み込まない）
  - 目的・ゴール
  - 制約・前提
  - 成功条件
  - スコープ境界
  - 方針の選択肢
- 推奨手段は`AskUserQuestion`。複数の論点はバッチでまとめて提示し、多ターンの段階的プロンプトを避ける（やり取りが分散するとユーザー側の負担も増えるため）
- 対話で得た結論は、計画ファイル作成時に`## 背景`（経緯と目的）と`## 対応方針 > ユーザー合意済み事項`（合意した方針）に転記する。エージェントの推測判断と混ぜない
- plan-mode自体のトリガー条件に該当しない場合（単純な単一ファイル修正・会話だけの質問など）はこの対話フェーズも省略してよい

## 計画ファイル作成の基本

- 対応方針決定までにユーザーとのやり取りなどがあった場合、経緯をドキュメントに残す作業の追加を検討する
  - ユーザー確認や指摘で得た判断基準が計画固有の一過性の事情にとどまらない場合、CLAUDE.md・プロジェクトドキュメント・既存ルールなど再利用される場所へ反映する作業を同じ計画内に含める（再発防止）
  - 反映先は `## 変更内容` に記載し、今後のセッションで同じ確認が発生しない状態まで恒久化する
- plan modeの制約で必要な調査が実行できない場合、ユーザーに確認の上で先に実行する
- ソースコードだけでなく、ドキュメントへの影響も必ず確認する。設計意図などの追記も検討する
- 検証は自動実行できる手順を最優先する（手動検証は再現性が低く、回帰テストとしても機能しないため）
  - 既存の自動チェック基盤で不足する場合は新たな仕組みの導入も検討する（大規模な変更になる場合はユーザー確認要）
  - 手動検証は自動化が技術的・コスト的に困難な場合に限定し、理由を計画ファイル内に明記する
  - 「修正前は失敗し、修正後は成功する」ことを観測可能にする手段を用意する
- 要件が曖昧で仕様を確定しにくい場合は、詳細な仕様化よりも軽量なプロトタイプ反復を提案する選択肢を検討する（仕様化コストより動作例の方が意思決定を早める場合があるため）
- 方針転換はユーザーへ可否を確認してから行う

## 計画ファイルの構成

以下の構成を厳守する。人間は判断に関わる重点セクションを冒頭で確認でき、codexは実装レベルの照合材料を後半でまとめて参照でき、両者の関心が混ざらない構造にするため。

1. `# タイトル`
2. `## 背景` — なぜこの変更をするか。経緯と期待する成果（人間向け）
3. `## 対応方針` — 変更内容・影響範囲・段取り・リスク扱い（人間向け）。必要に応じて表なども使い分かりやすく書く
   1. `### ユーザー合意済み事項` — ユーザーが発話または確認で合意した事項（エージェントの推測を混ぜず厳密に分離）
   2. `### エージェント判断` — ユーザー指示外でエージェントが既定した事項。差し戻し・見直しの候補
   3. `### 却下した代替案` — 別案と却下理由（codexの「なぜ別の方法を取らないのか」指摘を予防）
4. `## 調査結果` — 既存コードの挙動・依存・制約（codex向け照合材料）
5. `## 変更内容` — 変更対象ファイルの絶対パス、変更方針、参照箇所のfile:line（codex向け）
6. `## 検証` — 修正前は失敗し修正後は成功することを観測可能な自動化手順
7. `## 変更履歴`（2回目以降のみ）— 前回更新からの差分を記録する。codex指摘への対応と不対応見解、ユーザー指示による修正、エージェントの自発修正をまとめて追記し、人間の差分レビューを支援する

### 構成を守るための注意

- セクション・サブセクションの追加や順序の変更は禁止
  - ユーザーへの確認事項は計画ファイル提示前に解消する運用のため、未決事項などは記述不可
- 冗長な説明や詳細は避け、必要な情報だけを簡潔に記述
- 情報を複数セクションに重複させない
- 連番や個数の表記は極力避ける（修正時の影響箇所が増えるため）
  - ただし構造定義など順序が本質的に意味を持つ列挙は連番を使ってよい
- 背景や方針で抽象的な宣言文（「機能的挙動は変えず」「最小差分で」など）を書く場合、変更内容と矛盾しないかを必ず確認する
  - 先回りで縮退的制約を課すと本質的な改善を阻害するため、ユーザー合意のある記述に限定する
- codexレビューに備えて対応方針・調査結果を十分に記述しておく
- コードの記述は最小限とし、codexがリポジトリ実体と照合できる粒度に留める
- `## ユーザー合意済み事項`にはユーザーの発話・確認済み合意事項のみを書く
  - エージェントの推測や既定判断は `## エージェント判断` に分離する
- `# タイトル`から`## 却下した代替案`までのセクションには実装レベルの記述（ファイル名・シンボル名・file:line・コード片・外部APIの具体値・特定ミドルウェアや設定キーなど）を書かない
  - 判断材料としての抽象度を保つため、仕様・挙動・方針の粒度に留める
  - これらの詳細は`## 調査結果`と`## 変更内容`に書く
- `## 変更履歴` は更新単位ごとに項目を分け、その下に差分要約と関連する指摘・判断を並べる
  - 指摘に対応しない場合は該当エントリ内に理由を併記する（不要な再指摘を避けるため）
- 上記のセクション構成は`posttooluse` hookでも検査される（`scripts/posttooluse.py`の`_PLAN_REQUIRED_H2`）

### 記述例

以下は全セクションを通した記述例。
`## 変更履歴` は2回目以降の更新でのみ記載するが、構造を示すためここでは含めている。

```markdown
# ファイルアップロード上限を10MBから50MBへ引き上げ

## 背景

高解像度画像をアップロードできないという要望が継続して寄せられている。
ストレージと転送帯域の増強が完了し運用コストの懸念が解消されたため、上限値を引き上げて要望に応える。

## 対応方針

### ユーザー合意済み事項

- 上限値は50MBに変更
- 超過時のエラーメッセージ文言は既存のものを流用
- 経路上の各層に個別設定されているサイズ制限やタイムアウトを事前に確認し、整合確認を検証範囲に含める

### エージェント判断

- 上限値の定義を一元化してSSOTを確立
- 既存の超過判定とエラー通知の挙動は維持し、参照値のみを差し替え

### 却下した代替案

- ファイル種別ごとに上限を変える案 — 運用が複雑化する割に現時点で必要性が明確でないため
- 上限を撤廃する案 — 想定外のリソース消費や不正利用のリスクが残るため

## 調査結果

- サーバー側の上限定義: `server/config.py:34`
- クライアント側の上限定義: `client/limits.ts:12`（サーバーと独立して保持されており不整合が発生しやすい）
- リバースプロキシのbody size上限: `infra/nginx.conf:58` で20MB
- 関連テスト: `tests/upload_test.py:40-78`

## 変更内容

- 修正: `server/config.py:34` の上限値を50MBへ変更
- 削除: `client/limits.ts:12` の重複定義を撤去し、設定API経由で取得する形へ変更
- 修正: `infra/nginx.conf:58` のbody size上限を50MBへ引き上げ
- 修正: `tests/upload_test.py:40-78` の境界値ケースを更新

## 検証

- `uv run pyfltr run-for-agent tests/upload_test.py`

## 変更履歴

1. codexレビュー反映
   - 指摘: リバースプロキシのbody size上限が未考慮 → 対応: 調査結果と変更内容に反映
   - 指摘: 新旧の上限値を切り替えるfeature flagを用意すべき → 不対応: 切り戻しは設定値の差し替えで十分対応でき、feature flagを追加する運用複雑度の増加に見合う利点がないため
2. ユーザー指示反映
   - 検証手順を `uv run pyfltr run-for-agent tests/upload_test.py` の実行に変更
```

## codexレビュー

ユーザーに計画ファイルを提示する前に、codexでレビューする。
plan modeの制約下でもcodexレビューは制約対象外として扱える（計画ファイル自体のレビューのため）。

### MCPツール版（優先）

`mcp__codex__codex` / `mcp__codex__codex-reply` が利用可能なら優先して使う。

- 初回: `mcp__codex__codex` を以下のパラメーターで呼び出す
  - `cwd`: `"{project_directory}"`
  - `sandbox`: `"danger-full-access"`
  - `prompt`: 以下の文言をそのまま使う（CLIフォールバック時も同一文言を使用する）

    ```text
    {plan_full_path} この計画ファイルをレビューして。
    以下の手順で行うこと。
    1. 考えられる指摘候補を内部的に網羅列挙する
    2. そこから確実に問題である致命的かつ本質的な欠陥のみを残し最終応答とする
    3. 今回の応答で全て出し切り、再レビューで追加の新規指摘を出さない

    些末な指摘、不要な提案、推測による指摘は禁止。
    計画ファイルの記述スタイル（構成・表現・書き方）に対する指摘も禁止。（記述間の矛盾は除く）
    指摘する場合は修正方針も簡潔に併記すること。
    指摘が無い場合は指摘無しと応答すること。
    ```

- 2回目以降: `mcp__codex__codex-reply` を以下のパラメーターで呼び出す
  - `threadId`: 前回の戻り値から取得したthreadId
  - `prompt`: 以下の文言をそのまま使う（CLIフォールバック時も同一文言を使用する）

    ```text
    {plan_full_path} 計画ファイルを更新したので再レビューして。
    更新後の計画ファイルに対しても初回と同じ手順（候補の網羅列挙→致命的かつ本質的な欠陥のみ残す→今回の応答で全て出し切る）で検査すること。
    ```

レビュー結果は戻り値の `content` フィールドに含まれる。

### CLIフォールバック版

MCPツールが利用できない場合はcodex CLIを試す。
CLIの詳細コマンド（`codex exec` / `codex exec resume` のオプション・session id抽出手順など）は `references/codex-cli.md` を読み込んでから実行する。

### どちらも使えない場合

MCPもCLIも使えない場合は、計画ファイル提示前にユーザーに状況を報告し、以下の選択を仰ぐ。

- codex CLIを別途セットアップし、`claude mcp add --scope=user codex codex mcp-server` でcodex MCPをインストールしてやり直す（推奨）
- codex CLIを別途セットアップしてやり直す
- 当該環境ではcodexレビューを永続的にスキップする旨をプロジェクトの `CLAUDE.local.md` に記載してスキップする（当該ファイルはgitignore推奨）

## codexレビューの進め方

- ユーザーへの確認事項がある場合はcodexレビュー前にユーザーに確認し結論を確定させる
  - 方針レベルの合意を実装詳細レベルの合意と誤解しないよう注意
- codexレビューでユーザー判断が必要とされた場合、その場ですぐにユーザーに確認して結論を確定させる
- codexの指摘がなくなるまで更新とレビューを繰り返す。合否判断は呼び元が行ってよい（指摘が的外ればかりの場合なども合格扱いでよい）
- 一度指摘ゼロになった後に限り、その後の軽微な修正では再レビューを省略してよい
  - 方針レベルの変更時は再レビューする
- codexの指摘に基づいて計画ファイルを修正する際は、どのような修正をするかユーザーに伝える

## Gotchas

- 要件が固まる前に計画ファイルの骨子を書き始めがち。曖昧さが残る段階では`AskUserQuestion`などで選択肢を確定させてから計画ファイル作成に進む
- 方針レベルの合意を実装詳細レベルの合意と取り違えやすい。ユーザーに確認するときは粒度（方針か実装詳細か）を明示する
- エージェントの推測判断を `## ユーザー合意済み事項` に混入させがち。発話・確認済みの合意のみをそこに書き、推測や既定判断は `## エージェント判断` に分離する
- 抽象的な縮退宣言（「機能的挙動は変えず」「最小差分で」など）を先回りで書きがち。ユーザー合意のある範囲に限定する（本質的な改善を阻害するため）
- 連番や個数の表記を安易に使いがち。修正時に影響箇所が連鎖するため、順序が本質的に意味を持つ場合に限定する
