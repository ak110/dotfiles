# 計画モード・計画ファイル作成

本スキルは、Claude Code標準のplan modeを強化し、より効率的かつ高品質な成果を得るための手順とガイドラインを提供する。

本スキルの利用時、エージェントは以下の手順に従い作業すること。

1. バグ対応手順
2. 計画ファイル作成前の要件対話
3. ユーザー要求の認識合わせ
4. 恒久化検討
5. リファクタリング検討
6. 計画ファイルの作成
7. 計画ファイルのcodexレビュー
8. 計画の実行

## 工程の進め方

全工程に共通する進行方法を定義する。

### 起動直後の動作

「1. バグ対応手順」の判定結果に応じて分岐する。

- バグ対応に該当する場合: 本スキル呼び出し直後の最初のツール呼び出しを`references/bugfix-process.md`の`Read`に固定する。
  その読了後、TaskCreateで対象工程（1〜8）を一括登録する
- バグ対応に該当しない場合: 直ちにTaskCreateで対象工程（2〜8）を一括登録する
 （「1. バグ対応手順」の工程は登録対象から除外する）

### TaskCreate/TaskUpdate運用

- 各工程開始時にTaskUpdateで`in_progress`へ遷移する
- 各工程完了時にTaskUpdateで`completed`へ遷移する

### 次工程への移行

- 各工程の完了時に決定事項を箇条書きでユーザーへ報告し、ユーザー応答を待たずに次工程へ進む
- 例外:「2. 計画ファイル作成前の要件対話」内のAskUserQuestion応答は通常の対話として待つ

## 1. バグ対応手順（最優先で確認）

ユーザーの指示がバグ・障害・エラー・デグレードなどの対応に該当する場合、
他セクションを読む前に`references/bugfix-process.md`を読み込み、手順に従い確実に対応する。

再現の有無や発覚経路（CI失敗・テスト失敗・例外発生・挙動変化の報告など）は問わない。
該当する場合、本スキル呼び出し直後の最初のツール呼び出しは`Read`で`references/bugfix-process.md`を開くことに固定する。
それ以前に計画ファイルの作成や他セクションの解釈へ進んではならない。
この順序を守らないと、構成テンプレートを先に読んだ勢いで対症療法的な計画に陥りやすく、再発防止の検討が欠落するため。

## 2. 計画ファイル作成前の要件対話

ユーザー指示が曖昧・複数の解釈が成り立つ場合は、そのまま計画ファイル作成へ進んではならない。
先に選択肢を提示など、要件を対話的に明確化してから計画ファイル作成に進む。

曖昧さを残したまま骨子を書き始めると、計画ファイルへの追記とユーザー確認が多ターンに分散する。
その結果、トークン効率と最終的な計画品質の両方が低下するため。

- 対話で確定させる観点は以下に絞る（実装詳細には立ち入らない）
  - 目的・ゴール
  - 制約・前提
  - 成功条件
  - スコープ境界
  - 方針の選択肢
- 推奨手段は`AskUserQuestion`
  - 複数の論点はバッチでまとめて提示し、多ターンの段階的プロンプトを避ける（やり取りが分散するとユーザー側の負担も増えるため）
- 対話で得た結論は、計画ファイル作成時に以下のセクションへ転記する。エージェントの推測判断と混ぜない
  - `## 背景`（経緯と目的）
  - `## 対応方針 > ユーザー合意済み事項`（合意した方針）

### 「レビュー」「検討」のような指示の解釈

ユーザーから「レビューして」「検討して」「見直して」のような指示を受けた場合、単純な調査報告で完結させない。
以下の流れまでを含む指示として解釈する。

1. 対象を調査し改善案・選択肢を提示する
2. ユーザー選択を取得する（`AskUserQuestion`等）
3. 選択された案を計画ファイルに反映する
4. `ExitPlanMode`で承認を得たうえで実装する

調査報告だけで停止すると、ユーザーは追加で「実装して」と指示する必要があり手戻りが増える。
明示的に「調査だけ」と指示された場合のみ報告で完結させる。

## 3. ユーザー要求の認識合わせ

「2. 計画ファイル作成前の要件対話」で曖昧さを除去した直後、計画ファイルの骨子作成へ進む前に、
ユーザー指示の解釈を箇条書きでテキスト提示し、認識齟齬の有無をユーザーへ確認する。

「2.」がAskUserQuestionを使った曖昧さの除去であるのに対し、本ステップは
自分の理解を網羅的に明示してユーザーの合意を得る位置づけ。
推測のまま骨子作成へ進むと計画ファイル提示後の手戻りが発生しやすいため、必ず実施する。

- 提示形式: 論点ごとに見出しを立て、各論点配下にbullet項目で解釈・対応方針を列挙する
- 提示内容: ユーザー指示の各要求項目、エージェントが補完する判断材料、対応する／見送る項目の区別
- 確認手段: テキスト本文として提示し、ユーザーからの自由記述による補足・修正を受け付ける
 （AskUserQuestionは選択肢を絞る道具のため本ステップでは使わない）
- 確認後の処理: ユーザーから明確な反対が無ければ「4. 恒久化検討」へ進む。
  反対や追加要望があれば本ステップを再実施する

## 4. 恒久化検討

本工程では、計画作成過程でユーザーとの対話や調査を通じて得られたdurableな知見を整理し、
永続化先と文面を確定する。

列挙対象:

- ユーザーが発話または確認で合意した事項のうち、当該計画固有でない知見
- エージェントが判断した事項のうち、他の計画でも横展開すべき選定基準や指針
- 却下した代替案のうち、将来の意思決定で再利用できる判断根拠
- 調査で得た制約・前提・仕様の理解
- `AskUserQuestion`等の対話でユーザーが付した選択理由・補足
 （選択そのものより選定理由の方が将来の意思決定で再利用される性質を持つため）
- プロジェクト方針的な情報を含むユーザー指示全般

列挙の判定はpermissiveに倒す。些細な判断基準でも可能な限り残し、迷った場合は列挙する側を選ぶ。
真に判断材料が無い計画では「無し」と明記する。

永続化先候補:

- プロジェクト方針なら`CLAUDE.md`
- ファイル種別ごとの規約なら`.claude/rules/`
- 特定の作業手順なら`.claude/skills/`
- 利用者向け情報ならプロジェクトドキュメント（`docs/`配下など）

検討結果の記録:

確定した知見は計画ファイルの`### エージェント判断`配下にドキュメント文面例外として記述する。
永続化先パスを示し、永続化先に書く文面をコードブロックで提示する。
実装エージェントが解釈ぶれなく転記できるよう、自然文の指示ではなく確定文面を載せる。

## 5. リファクタリング検討

リファクタリングは開発と同時に行うものであり、別途時間を取って実施するものではない。
本来の対応によって理想的な状態が崩れるなら、それを補うリファクタリングまで完了して初めて修正完了とみなす。

本工程では、計画対象の本来の対応に内包されない「副次的な既存不備」を洗い出す。

列挙対象:

- コード変更を契機に発見した既存不備
  - 追加処理を受け取る対象モジュールの責務が変更前から既に広くなりすぎている
  - 今回追加する処理と類似する既存処理が分散しており共通化できていない
  - ファイル行数・関数行数が変更前から既に肥大化している（分割・抽出を検討）
  - 計画対象の変更で陳腐化する既存実装・既存テストがある
- ドキュメント変更を契機に発見した既存不備
  - 追加記述の粒度に対して既存節の粒度が不整合
  - 既存節構成が変更後の内容に合わず再編が必要
  - 同一トピックが複数箇所に分散しており、本計画とは別にSSOT集約が必要
- その他、計画作成中に発見した既存記述の不備（陳腐化・誤り・不整合・テスト不足・観測不能な挙動）

真に該当が無い場合は「無し」と明記する。

検討結果の記録:

確定した副次対応は計画ファイルの`## 変更内容`に本来の変更と並べて項目として列挙する。
本来の変更と副次対応を同一セクションに並べることで、実装時に両者を合わせて対応しやすくする。

## 6. 計画ファイルの作成

### 実装方式判定

ExitPlanMode後の計画の実装方式は、以下のいずれかを選ぶことができる

- `careful-impl`: `agent-toolkit:careful-impl`スキルを呼び出して実装工程を実施
 （サブエージェントでコンテキストを分離した実装・セルフレビューを実施する品質の高い方式）
- `careful-impl不使用`: メインが直接実装工程を実施（軽微な変更に適した方式）

既定では`careful-impl不使用`とする。
以下のいずれかに該当する場合のみ`careful-impl`を採用する。

- ユーザー指示内に`セルフレビュー`、または`careful-impl使用`の指示がある場合
 （「しっかりレビュー」「品質重視で」等の自然言語表現は採用合図として扱わない。誤起動を避けるため）
- 大規模計画と判断される場合は`AskUserQuestion`で採用可否を確認する
  - 大規模判定の目安: 計画ファイルが複数フェーズ分割を要する規模、
    または公開インターフェース変更を伴う複数ファイル横断の設計変更
  - 軽微な修正（誤字・単一関数修正・既存パターン踏襲）では確認もスキップし`careful-impl不使用`とする
  - 質問文は`careful-impl`の概念を知らないユーザーでも判断できるよう、外部仕様レベルの補足を必ず添える。
    例を以下に示す

    ```text
    計画の実行に careful-impl スキルを使用しますか？
    （サブエージェントを活用し注意深くレビューするため、時間・コストはかかるが高品質な作業を行えるスキル）
    ```

### 参照テンプレート読み込み

計画ファイルの骨子を書き始める直前に、`references/plan-file-guidelines.md`を必ずReadで読み込む。
テンプレートには全セクションの記述例に加え、計画ファイルの構成・セクション別の注意点・
複数フェーズ計画の骨子例を収録している。
`plan-file-guidelines.md`は内容量が多いため、参照時点で読み込むことで要件対話中の不要な負荷を避ける。

## 7. 計画ファイルのcodexレビュー

ユーザーに計画ファイルを提示する前に、必ずcodexレビューを実施する。
実施直前に`references/codex-review.md`を読み込み、そこに記述された手順
（MCPツール版優先・CLIフォールバック・未導入時の選択・レビューの進め方）に従う。

## 8. 計画の実行

`実装方式: careful-impl`の場合は、メインから`agent-toolkit:careful-impl`スキルを呼び出してその後の工程を実施する。

`実装方式: careful-impl不使用`の場合は、メインが`## 実行方法`に従って実装・検証・コミットを行う。
全コミット完了後、メイン側で恒久化チェックを実施する。
本チェックは`careful-impl`スキルの`## 5. 恒久化チェック`と同等の効果を持つガードであり、
恒久化漏れを検知する。

- 計画ファイルの`### エージェント判断`配下のドキュメント文面コードブロックを再読し、
  各永続化先ファイルへ反映済みであることを確認する
- 計画ファイルの`## 変更内容`の`副次:`プレフィクス付きの項目を再読し、反映済みであることを確認する
- 実装中に新規発見したdurableな知見（設計判断・調査で得た制約・横展開すべき類似箇所への指針など）があれば、
  計画ファイルの`### エージェント判断`配下にコードブロック付きで追記し、
  対応する永続化先の編集も併せて行う
- 反映漏れや追加が必要な編集を発見した場合は、追加コミットで永続化先を更新する
