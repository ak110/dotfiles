# 計画ファイル作成ガイドライン

計画ファイル作成時に参照する記述例と、構成・注意点・レビュー省略可否の説明を収録する。

## 全セクションを通した記述例

以下は単一フェーズ計画の記述例。

```markdown
# ファイルアップロード上限を10MBから50MBへ引き上げ

## 背景

高解像度画像をアップロードできないという要望が継続して寄せられている。
ストレージと転送帯域の増強が完了し運用コストの懸念が解消されたため、上限値を引き上げて要望に応える。

## 対応方針

### ユーザー合意済み事項

- 上限値は50MBに変更
- 超過時のエラーメッセージ文言は既存のものを流用
- 経路上の各層に個別設定されているサイズ制限やタイムアウトを事前に確認し、整合確認を検証範囲に含める

### エージェント判断

- 上限値の定義を一元化してSSOTを確立
- 既存の超過判定とエラー通知の挙動は維持し、参照値のみを差し替え

### 却下した代替案

- ファイル種別ごとに上限を変える案 — 運用が複雑化する割に現時点で必要性が明確でないため
- 上限を撤廃する案 — 想定外のリソース消費や不正利用のリスクが残るため

## 調査結果

- サーバー側の上限定義: `server/config.py:34`
- クライアント側の上限定義: `client/limits.ts:12`（サーバーと独立して保持されており不整合が発生しやすい）
- リバースプロキシのbody size上限: `infra/nginx.conf:58` で20MB
- 関連テスト: `tests/upload_test.py:40-78`

## 変更内容

- 修正: `server/config.py:34` の上限値を50MBへ変更
- 削除: `client/limits.ts:12` の重複定義を撤去し、設定API経由で取得する形へ変更
- 修正: `infra/nginx.conf:58` のbody size上限を50MBへ引き上げ
- 修正: `tests/upload_test.py:40-78` の境界値ケースを更新

## 実行方法

- 実装方式: careful-impl
- 実装時に事前呼び出しが必要なスキル
  - `agent-toolkit:coding-standards`
- 検証手順
  - `uv run pyfltr run-for-agent tests/upload_test.py`
  - 追加したテストが修正前は失敗し修正後は成功することを確認する
- セルフレビュー方針: `careful-code-reviewer`のみ起動。仕様変更を伴わないため`careful-spec-reviewer`は省略
- コミット方針: コミットする。subject: `feat(upload): ファイルサイズ上限を50MBへ引き上げる`

## 変更履歴

- 初版
- codexレビュー指摘反映（1回目）
  - 却下: 新旧の上限値を切り替えるfeature flagを用意すべき — 切り戻しは設定値の差し替えで十分対応でき、feature flagを追加する運用複雑度の増加に見合う利点がないため

## 計画ファイル

`~/.claude/plans/upload-limit-increase-concurrent-hickey.md`
```

## セクション構成と記述要件

計画ファイルは以下のセクション構成を厳守する。
人間は判断に関わる重点セクションを冒頭で確認でき、codexは実装レベルの照合材料を後半でまとめて参照できる。
両者の関心が混ざらない構造にするため。

セクション・サブセクションの追加や順序の変更は禁止。
例外: `## 変更内容`・`## 実行方法`節内のフェーズ分割サブ節（`### フェーズN: 題目`）は許容する。

以下の各サブ節で「目的」「記述要件」「禁止事項」「例外」を示す。
まず全体に共通する禁止事項を記述し、その後に各セクション固有の要件を上から順に整理する。

### 判断材料セクション全体に共通する禁止事項

`# タイトル`・`## 背景`・`## 対応方針`・`### ユーザー合意済み事項`・`### エージェント判断`・`### 却下した代替案`の
判断材料セクション群には以下の禁止事項が共通して適用される。

- プログラムレベルの実装詳細を書いてはならない
  - 禁止対象: 関数名・メソッド名・クラス名・変数名・ファイル名・file:line・コード片・
    外部APIの具体値・特定ミドルウェアや設定キーなど、識別子レベルの記述全般
  - 理由: これらの節は「なぜ」「どの方向へ」の判断材料を提示する場であり、実装詳細が混入すると
    方針の抽象度が崩れてユーザーが妥当性を評価しにくくなる。
    また実装エージェントが計画記述を過剰な拘束として受け取り、より適切な実装選択肢を排除する
  - 違反例:「`parse_plan()`で解析処理を一本化する」「`src/foo.py`の`Handler`クラスを修正する」
  - 正例:「計画ファイル解析処理のエントリを単一化する」「リクエスト処理層の入力検証を強化する」
  - 実装詳細は`## 調査結果`と`## 変更内容`にのみ記述する
- 例外: ドキュメント修正の変更後文面は、後述の`### 対応方針`のルールに従う限り禁止対象外とする

### タイトル（`# タイトル`）

- 目的: 計画の主題を1行で示す

### 背景（`## 背景`）

- 目的: なぜこの変更をするか。経緯と期待する成果（人間向け）

#### バグ対応サマリーの記録

バグ対応時は`## 背景`節末尾に以下の表を必ず含める。経緯・目的の本文を先に書き、その下に表を置く。
表形式で1か所に集約することで、別セッションへの引き継ぎと振り返りが容易になる。

| 項目 | 内容 |
| --- | --- |
| 根本原因 | 起点まで追跡した一次原因を1〜2文で記述 |
| 修正方針 | 直接の修正内容を1〜2文で記述（再発防止策と対で検討する） |
| 再発防止策 | 自動で機能する仕組み（テスト・CI・型制約など）を優先して記述 |
| 類似見直し結果 | 根本原因に基づき類似パターンの他箇所を確認した範囲と結果 |
| 知見の記録 | 恒久化先（CLAUDE.md・ルール・スキル等）と転記内容 |

本表は対症療法ではなく根本対応に至ったことを示すチェックリストでもある。
`references/bugfix-process.md`の手順で得た結論を本表に集約する。
「類似見直し結果」欄は計画ファイル作成前に類似箇所の調査を完了させ、確定した結果を記入する。
未調査のまま「調査予定」などの保留表現で残したまま計画ファイル提示に進んではならない。

### 対応方針（`## 対応方針`）

- 目的: 変更内容・影響範囲・段取り・リスク扱い（人間向け）
- 記述要件: 必要に応じて表なども使い分かりやすく書く
- 禁止事項: 抽象的な宣言文（「機能的挙動は変えず」「最小差分で」など）を書く場合、
  変更内容と矛盾しないかを必ず確認する
  - 先回りで縮退的制約を課すと本質的な改善を阻害するため、ユーザー合意のある記述に限定する
- 例外: ドキュメント修正の変更後文面は`## 対応方針`配下にコードブロックで示してよい
  - ドキュメントは方針そのものが文面として現れるため、文面を示すことが方針提示と等価になる
  - ドキュメント修正計画で抽象的な方針だけに留めると評価可能性が下がるため、
    変更後の文面そのものをコードブロックで示す
  - 合意由来の文面は`### ユーザー合意済み事項`配下、
    エージェントの既定判断による文面は`### エージェント判断`配下に置く
  - 1コードブロックが概ね50行を超える大量修正に限り`## 変更内容`側へ退避してよい。
    それ未満は必ず`## 対応方針`側に置く

### ユーザー合意済み事項（`### ユーザー合意済み事項`）

- 目的: ユーザーが発話または確認で合意した事項（エージェントの推測を混ぜず厳密に分離）
- 記述要件: ユーザーの発話・確認済み合意事項のみを書く
- 禁止事項: エージェントの推測や既定判断を混ぜない。`### エージェント判断`に分離する
  - `CLAUDE.md`や`~/.claude/`配下などにある指示も単独ではユーザー発話とは見なさず、エージェント判断に置く

### エージェント判断（`### エージェント判断`）

- 目的: ユーザー指示外でエージェントが既定した事項。修正・見直しの候補

### 却下した代替案（`### 却下した代替案`）

- 目的: 検討の結果として選択肢になり得たが採らなかった案を残し、
  codexの「なぜ別の方法を取らないのか」指摘を予防する
- 記述要件: 選択肢になり得た案のみを書く。明らかに非現実的な案や、
  そもそも検討対象として浮上しなかった案を網羅的に列挙する必要はない
- 該当する案が無い場合は「無し」と明記する

### 調査結果（`## 調査結果`）

- 目的: 既存コードの挙動・依存・制約（codex向け照合材料）

### 変更内容（`## 変更内容`）

- 目的: 変更対象ファイルの絶対パス、変更方針、参照箇所のfile:line（codex向け）
- 禁止事項: コード修正内容をコードブロックで転記しない
  - 方針・境界・挙動・外部インターフェース・確定済み制約のみを書き、
    変数名・エラーメッセージ文言・ループ構造・条件分岐の内訳・小規模な制御フローは書かない
   （計画ファイル由来の具体記述が誤読されて実装されると、本来不要な制約として固定化されるため）
  - 粒度の目安:「この記述を読んだ別の実装者が方針を誤解せず進められる」最低限まで

### 実行方法（`## 実行方法`）

- 目的: `ExitPlanMode`後の実装・検証・セルフレビュー・コミットに必要な計画固有の情報を記述する
- 記述要件: 以下の項目を上から順に記述する
  - `- 実装方式`（1行記述）
    - 値は`careful-impl`または`careful-impl不使用`のいずれか
    - `careful-impl`の場合、メインから`agent-toolkit:careful-impl`スキルを呼び出して実装・検証・レビュー・コミットを行う
    - `careful-impl不使用`の場合、メインが`## 実行方法`に従って実装・検証・コミットを直接行う
  - `- 実装時に事前呼び出しが必要なスキル` リスト
    - 変更対象に応じて`agent-toolkit:coding-standards`・`agent-toolkit:writing-standards`・
      `agent-toolkit:spec-driven`などを列挙する（対象変更に含まれなければ列挙しない）
    - `agent-toolkit:careful-impl`は`実装方式`項目から自動派生するため本リストには含めない
  - `- 検証手順` リスト
    - その計画を完了してコミットする前に必要な検証を具体的に記述する
    - プロジェクト方針として定められた手順（format/lint/test一式など）と、計画固有の観測手順の両方を含める
    - 自動化不可な手動確認が必要な場合は理由付きで明示する
  - `- セルフレビュー方針`（1行記述、`実装方式: careful-impl`の場合のみ記述する）
    - 既定では`careful-spec-reviewer`・`careful-code-reviewer`・`careful-docs-reviewer`の
      3エージェントをすべて起動する扱いとする
    - 省略するエージェントがあれば名称と省略理由を列挙する
  - `- コミット方針`（1行記述）
    - コミット有無とコミットメッセージのsubject行を記述する
    - 本文（subject行以降）は任意で2行目以降に添える

### 変更履歴（`## 変更履歴`）

- 目的: 却下した指摘の記録に限定する（初回は`- 初版`のみ記載する）
  - 対応済みの指摘・自発修正・ユーザー指示反映は本文側（`## 背景`〜`## 実行方法`）へ統合し、
    本節へは重複させない。本文と履歴の双方に同じ内容を書くと整合確認が手作業になるため
  - 却下した指摘のみ本節に残すことで、codexの再指摘回避と却下理由の参照容易性を維持する
- 記述要件: 更新単位ごとに見出し（`- 初版`、`- codexレビュー指摘反映（1回目）`等）を立て、
  見出し配下のbullet項目は却下した指摘とその理由のみとする
  - 却下が無い更新では見出しのみ残し、配下項目は省略してよい（更新があった事実を残すため見出しは保持する）
  - 対応した指摘は本文側に反映済みのため本節には書かない

### 計画ファイル（`## 計画ファイル`）

- 目的: 末尾に計画ファイルの絶対パスを記述する
  - コンテキストのクリア時にClaude Codeがパスを渡さず内容だけ渡す問題に対するワークアラウンド

## 計画ファイル全体の注意点

複数セクションに横断する記述方針。

- 計画ファイルは実装工程で単独参照される前提で書く。
  実装工程はコンテキスト圧縮・セッション境界などで計画時の会話履歴を失った状態から開始されることがあり、
  計画ファイルが唯一の入力源になりうるため、計画時点の判断はすべて計画ファイル内に書き切る
- 対応方針決定までに得た判断は、計画ファイルに書き残すだけでなく、「恒久化先への反映作業」を必ず組み込む
  - 計画ファイル自体は一時的な成果物であり、次回以降のセッションでは参照されないため
  - 対象: ユーザー合意・選択肢提示の結果・エージェントの既定判断
  - 判定手順: `### ユーザー合意済み事項`と`### エージェント判断`の各項目について以下を毎回問う
    - 別のセッション・別の作業でも同じ判断が必要か
    - 放置すると同じユーザー確認が再発するか
    - 共通化しない設計判断を別の編集者に上書きされうるか
  - 該当するものは恒久化先を決めて、同じ計画の`## 変更内容`に更新作業として明記する
  - 反映先の候補（いずれも将来のセッションで自動的に読み込まれる場所を選ぶ）
    - プロジェクト方針なら`CLAUDE.md`
    - ファイル種別ごとの規約なら`.claude/rules/`
    - 特定の作業手順なら`.claude/skills/`
    - 利用者向け情報ならプロジェクトドキュメント（`docs/`配下など）
  - Noと判断した項目も、後で振り返れるよう判断理由を計画ファイル内に簡潔に残す
  - この工程は必須扱いとする。検討のみで終わらせず、実際の反映作業まで計画に含めて初めて「恒久化」と見なす
- plan modeの制約で必要な調査が実行できない場合、ユーザーに確認の上で先に実行する
- `## 調査結果`・`## 変更内容`の記述に入る前に、変更対象ファイルと周辺ファイルを通読し、
  既存の章構成・記述粒度・抽象度・命名規則を把握する
  - 例えばドキュメント追記では、対象ファイルを通読せずに書き始めると既存記述との粒度不整合を招きやすい
- ソースコードだけでなく、ドキュメントへの影響も必ず確認する。設計意図などの追記も検討する
- 検証は自動実行できる手順を最優先する（手動検証は再現性が低く、回帰テストとしても機能しないため）
  - 既存の自動チェック基盤で不足する場合は新たな仕組みの導入も検討する（大規模な変更になる場合はユーザー確認要）
  - 手動検証は自動化が技術的・コスト的に困難な場合に限定し、理由を計画ファイル内に明記する
  - 「修正前は失敗し、修正後は成功する」ことを観測可能にする手段を用意する
- 要件が曖昧で仕様を確定しにくい場合は、詳細な仕様化よりも軽量なプロトタイプ反復を提案する選択肢を検討する
  - 仕様化コストより動作例の方が意思決定を早める場合があるため
- 方針転換はユーザーへ可否を確認してから行う
- 計画ファイルもMarkdownのため、`agent-toolkit:writing-standards`スキルを呼び出す
- 実装方法を検討するため、`agent-toolkit:coding-standards`スキルも呼び出す
- 冗長な説明や詳細は避け、必要な情報だけを簡潔に記述
- 情報を複数セクションに重複させない
- 連番や個数の表記は極力避ける（修正時の影響箇所が増えるため）
  - ただし構造定義など順序が本質的に意味を持つ列挙は連番を使ってよい
- 後続のcodexレビューに備えて対応方針・調査結果を十分に記述しておく

## careful-impl採用時のreviewer選択基準

`実装方式: careful-impl`の計画では「セルフレビュー方針」項目に各reviewerエージェントの起動要否を記述する。
以下の条件を総合的に勘案し、起動要否を判断する。

- 変更規模が小さい（数行〜数十行程度、1〜2ファイル）
- 影響範囲が限定的（公開インターフェースを変更しない、他モジュールへ波及しない）
- 既存パターンを踏襲している（類似実装が既に存在する、設計判断を伴わない）
- 判断の余地が少ない（単純な誤字修正・既知バグの明確な修正など）
- テスト済み・検証済み（自動テストで挙動が担保されている）

エージェント別の観点例:

- `careful-spec-reviewer`: 仕様適合性・ドキュメント間整合性。実装が仕様と一対一で対応し判断の余地がない場合に省略候補
- `careful-code-reviewer`: コード品質。コード変更がmanifestやバージョン更新のみで実ロジック変更がない場合に省略候補
- `careful-docs-reviewer`: ドキュメント単体品質。ドキュメント変更が無いか、軽微な誤字修正のみの場合に省略候補

判断に迷ったら省略せず起動する方針とする（見落としの損失が起動コストより大きいため）。
判断結果と理由は「セルフレビュー方針」項目のみに書き下し、別節（`## 変更履歴`など）へ分散させない。

## 複数フェーズ計画の記述方法

大規模な変更でフェーズ間にコミットを挟みたい場合、計画ファイル1本のなかでフェーズ分割を許容する。
分割しないシンプル計画と、フェーズ分割計画のいずれも本スキルで扱う。

ただし、過度なフェーズ分割はコスト増大・実装時間の長期化・全体最適の阻害につながるため、
分割は計画に「独立した複数の大規模な機能変更」が含まれる場合に限る。
例えば「実装と、その実装のためのリファクタリングと、それらに関するドキュメント更新」などは
原則1つのフェーズにまとめて実施するものとする。（下記の例は説明のための例外）

- フェーズ分割の記述場所: `## 変更内容`と`## 実行方法`の2節の内側に
  `### フェーズ1: 題目`・`### フェーズ2: 題目` …のサブ節を並べる
  - フェーズ題目は端的に表す名詞句（例: `フェーズ1: 既存テスト整備`）
  - フェーズ番号は連番で振る
- 他の大見出し（`## 背景`・`## 対応方針`・`## 調査結果`・`## 変更履歴`）はフェーズ分割せず全体で記述する
- フェーズごとに、`## 変更内容`サブ節には当該フェーズで触る変更対象ファイルと変更方針を、
  `## 実行方法`サブ節には当該フェーズの検証手順・セルフレビュー方針・コミット方針を書く。
  全フェーズ共通の手順や方針（実装方式・事前呼び出しスキルなど）は親節（フェーズサブ節の外側）にまとめる
  - `実装方式`は計画ファイル全体で単一の値とし、フェーズごとに切り替えない。
    一度`careful-impl`に入ると同一セッションで全フェーズが`careful-impl`配下で進行するため、
    途中フェーズだけ`careful-impl不使用`に切り替える運用は成立しない
  - `セルフレビュー方針`はフェーズごとに変わりうる（フェーズの変更内容に応じて起動reviewerが異なるため）ので
    各フェーズのサブ節に記述する
- レビュアーへ渡す「対象フェーズ」入力にはサブ節タイトル（`フェーズN: 題目`）をそのまま使う

`## 実行方法`節では全フェーズ共通の「実装方式」「実装時に事前呼び出しが必要なスキル」を親節に置き、
フェーズ別サブ節に「検証手順」「セルフレビュー方針」（`実装方式: careful-impl`の場合のみ）・「コミット方針」を記述する。
`## 変更内容`と`## 実行方法`のみ抜粋する。

```markdown
## 変更内容

### フェーズ1: 既存テスト整備

- 修正: `tests/upload_test.py` のフィクスチャを共通化

### フェーズ2: 上限値の引き上げ

- 修正: `server/config.py` の上限定義を50MBへ
- 修正: `infra/nginx.conf` のbody size上限を50MBへ

## 実行方法

- 実装方式: careful-impl
- 実装時に事前呼び出しが必要なスキル
  - `agent-toolkit:coding-standards`

### フェーズ1: 既存テスト整備

- 検証手順
  - `uv run pyfltr run-for-agent`
- セルフレビュー方針: `careful-code-reviewer`のみ起動。仕様変更を伴わないため`careful-spec-reviewer`は省略
- コミット方針: コミットする。subject: `test(upload): フィクスチャを共通化する`

### フェーズ2: 上限値の引き上げ

- 検証手順
  - `uv run pyfltr run-for-agent`
  - 50MB境界のテストが追加されており修正前は失敗・修正後は成功することを確認する
- セルフレビュー方針: 全reviewerを起動
- コミット方針: コミットする。subject: `feat(upload): ファイルサイズ上限を50MBへ引き上げる`
```
