# 体系的なバグ対応手順

対症療法（表面的な症状の解消）と根本対応（原因の除去と再発防止）の違いを常に意識する。

| 事象 | 対症療法 | 根本対応 |
| --- | --- | --- |
| compose.yamlのバージョン不一致 | バージョン番号を揃える | 依存更新が起点と特定し、更新時の自動検証を追加 |
| 本番DBのクエリータイムアウト | タイムアウト値を延長 | データ量増加に伴うインデックス不足を特定し追加 |
| API連携の突然の失敗 | リトライ処理を追加 | 外部APIの仕様変更を特定し、スキーマ検証を導入 |

ユーザーからの報告への対応時は、plan modeを使用して以下の手順で扱う。

## 1. 根本原因の特定

- 事象を再現する。期待値、実際値、発生条件を明確にする
- 表面的な症状で止めず、その状態が生まれた起点まで追跡する
  - コード変更、データの蓄積・変質、外部サービスの仕様変更、設定の不整合、環境差異など、原因の形態を先入観で限定しない
  - 変更履歴（gitログ、デプロイ履歴）、ログ、メトリクス、設定差分を確認し、事実に基づいて原因を絞り込む
- 複雑な制御フロー・状態遷移・依存関係は、ASCII図や箇条書きのフローで自分の理解を可視化してから原因絞り込みに進む
  - 頭の中だけで追うと見落としが生じるため
- 原因仮説が複数ある場合は、支持する事実と棄却する事実を整理し、採用した原因を明示する
- フロントエンドやWeb UIの挙動確認が必要な場合は、Playwright系やChrome DevTools系のMCPツールが利用可能であれば優先する
  - コンソールログとネットワークログを直接取得でき、観測精度が上がるため

## 2. 根本原因への対処

根本原因が判明したら、「直接の修正」「再発防止策」「類似見直し」の3点を同時に検討する。
これらは独立したタスクではなく、同一原因に対する多角的な対処である。
1つでも欠けた状態で計画ファイル提示に進むと、修正漏れや再発を招きやすい。

### 直接の修正

- 修正方針は、修正前に失敗し修正後に成功するテストや再現手順と対で定義する
- 修正方針はユーザーの本来の要求（求める方向性・スコープ・成果像）と常に整合させる。
  副次論点や外部指摘の解消を優先するあまり、当初の要求から逸脱した対処へ変更しないよう、
  方針候補を比較するときは「本来の要求からの距離」を判定軸の1つとして必ず置く

### 再発防止策

- 直接の修正のみで再発防止策がなければ、同じ問題が再び起きうる
- 人の注意に依存するものより、自動で機能する仕組み（テスト、CI、バリデーション、型制約など）を優先する
- 過剰な対策は避け、再発確率と影響度に見合った水準にする

### 類似見直し

- 同じ根本原因やパターンによる問題が他の箇所に残っていないか確認する
- 共通処理、コピー実装、類似条件分岐を重点的に調べる
- 根本原因の修正で自動解消される箇所も、影響範囲を確認して報告する

## 3. 知見の記録

- 根本原因・再発防止策・判断理由を再利用される場所（CLAUDE.md、運用手順書、コメントなど）に記録する
- 特に、暗黙知だった判断基準の明文化や、今後も誤りやすい箇所の注意喚起を重視する

## 計画ファイルへの集約

各ステップの結論は計画ファイルの`## 背景`直下の表に集約する
（表の項目は`plan-file-guidelines.md`の「バグ対応サマリーの記録」節で定義する）。

## Gotchas

- 症状の消失と根本原因特定の完了を区別する
  - 表面的に解消した後も「なぜその状態になったか」を起点まで追い、対症療法と根本対応を分けて扱う
- 修正方針の検討は再現条件の明文化を前提とする。期待値・実際値・発生条件が曖昧なまま進めるとテストの再現性が確保できない
- 再発防止策は自動で機能する仕組み（テスト・CI・型制約など）を第一選択にする。人の注意に依存する運用は次善策として扱う
- 「直接の修正」「再発防止策」「類似見直し」の3点は一体で検討する。どれか1つでも欠けた状態で計画ファイル提示に進まない
- codexレビュー等の外部レビュー指摘に応じて根本対応の方針自体を差し替える場合は、
  計画ファイルへ反映する前に「ユーザーの本来の要求と整合するか」を再評価する。
  指摘解消の最短経路ではなく、本来の要求に沿った経路を優先する。
  整合性に迷う場合はエージェント単独で決定せず、方針案をユーザーへ提示してから反映する
