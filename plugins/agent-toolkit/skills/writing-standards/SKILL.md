# ドキュメント品質

プロダクションレベルのドキュメント品質を保つための共通ベース指示。
`agent.md`・`styles.md`が共通基盤（敬体/常体の使い分け・粒度・日本語表記ルール・現在の状態の直接記述など）を担い、本スキルはドキュメント固有の構成・Markdown・README・文書種別の品質基準を追加する位置づけ。
重複する基本方針は`styles.md`を正とし、本スキルでは繰り返さない。
プロジェクト固有の規約（`CLAUDE.md`・`.claude/rules/`配下）が優先で、本スキルは不足分を埋める。

## ドキュメント品質（全ドキュメント共通）

### 構成・構造

- 対象読者を明確にする。1つのドキュメントが複数の読者層を対象にする場合はセクションを分ける
- 冒頭にドキュメントの目的・概要を置き、読者が「読むべきか」を即座に判断できるようにする
- 前提条件（必要な知識・ツール・環境）は本題の前に配置する
- 目次はページの長さと構造に応じて判断する。見出しが5つ以上ある場合や、スクロールが必要な長さでは目次を付ける
- 見出しの階層は3段（`##` / `###` / `####`）までを目安とする。それ以上は構造の見直しを検討する
- 関連する内容は近くにまとめ、読者が文書内を行き来しなくて済むようにする

### ディレクトリ構成

OSSプロジェクトの典型的なドキュメント配置パターンを示す。
プロジェクト方針がある場合はそちらに従う。

- ルート直下の定番ファイル: `README.md`、`CONTRIBUTING.md`、`CHANGELOG.md`、`LICENSE`
- `docs/`配下のレイアウト: ガイド・リファレンス・チュートリアルなど種別ごとにサブディレクトリを切る
  - 例: `docs/guide/`（利用者向け手順）、`docs/development/`（開発者向け情報）、`docs/api/`（API仕様）
- ファイル粒度: 1ファイル1トピックを基本とする。1ファイルが300行を超えたら分割を検討する
- ファイル命名: ケバブケース（`setup-guide.md`）を基本とする。定番ファイル（`README.md`・`CHANGELOG.md`等）は大文字の慣例に従う

### 内容

基本方針（コードから自明なことを書かない・現在の状態を直接記述する・具体例を添える）は`styles.md`に集約している。
本節ではドキュメント固有の追加基準のみ扱う。

- コマンド例やコードスニペットはコピー&ペーストでそのまま動くことを前提に書く。省略やプレースホルダーを使う場合は明示する
- 外部リソースへのリンクは、リンク切れのリスクを考慮し、要点を本文にも記載する

### 対象読者による書き分け

ユーザー向けと開発者向けでは、想定知識・文体・情報の粒度が異なる。

- ユーザー向け（利用ガイド・README・エラーメッセージなど）:
  - 専門用語は初出時に説明する。内部実装の詳細には触れない
  - 「何ができるか」「どう使うか」を中心に書く
  - 敬体（「〜します」「〜してください」）を基本とする
- 開発者向け（アーキテクチャ文書・CONTRIBUTING・コメントなど）:
  - 前提知識を共有する開発者を想定し、簡潔に書く
  - 「なぜそう設計したか」「どこに注意が必要か」を中心に書く
  - 常体（「〜する」「〜である」）を基本とする

### 改訂・保守

- 追記する内容の粒度が既存の記述と整合しているか確認する。必要に応じて既存部分や章構成の変更も行い、全体の一貫性を保つ
- 陳腐化した記述は積極的に削除する。「古い情報が残っている」ことは「情報がない」より有害である
- SSOTを維持する。同じ情報を複数箇所に書かず、正規の場所からリンクする。やむを得ず重複する場合はどちらが正とするかを明記する
- ドキュメントの変更がコードの変更と連動する場合は、同じコミットまたはPRに含める

## Markdown記述スタイル

- `**`は強調したい箇所のみとし、箇条書きの見出しなどでの使用は禁止する
- できるだけmarkdownlintが通るように書く
  - 特に注意するルール:
    - MD022 - Headings should be surrounded by blank lines
    - MD031 - Fenced code blocks should be surrounded by blank lines
    - MD040 - Fenced code blocks should have a language specified
- 1文ごとに改行する（一行が長くなって読みづらく、差分レビューも難しくなるため）
  - 機械的な改行で文の区切りが不自然になる場合は、短文を一文にまとめるなど書き換えでの対応も検討する
- 図はMermaid記法で書く
- 別のMarkdownファイルへのリンクは用途によって書き分ける
  - Markdownソースのまま読まれる想定のファイル（`README.md`・`CLAUDE.md`・GitHub閲覧前提のdocsなど）では、`[プロジェクトルートからのパス](記述個所からの相対パス)`で書く。閲覧者がリンク先のパスを一目で把握できる利点がある
    - 例: `docs/api.md`から`docs/guide/setup.md`へリンク → `[docs/guide/setup.md](guide/setup.md)`
  - ドキュメント生成ツール（mkdocs・Sphinx・Docusaurusなど）でHTMLほかの形式に変換される想定のファイルでは、リンクテキストをページタイトルや自然な表現にする。変換後の利用者には内部パスが無意味なため
    - 例: `[セットアップ手順](guide/setup.md)`
- テーブルは列数が多い場合や内容が長い場合は箇条書きへの変換を検討する

## README規約

READMEはプロジェクトの顔であり、初見の読者が最も先に読むファイル。

- 標準構成パターン（プロジェクト方針がない場合）:
  1. プロジェクト名と一文の概要
  2. 主な機能・特徴（箇条書き）
  3. インストール・セットアップ手順
  4. 基本的な使い方（コード例やコマンド例）
  5. ドキュメントへのリンク（`docs/`がある場合）
- 全セクションが必須ではない。プロジェクトの規模と性質に応じて取捨選択する
- `CLAUDE.md`との役割分担: READMEは人間の読者向け、`CLAUDE.md`はClaude Code向け。READMEにClaude固有の指示を書かない。両者で重複する情報（ビルド手順など）はREADMEを正とし、`CLAUDE.md`からはREADMEを参照するか要点のみ再掲する

## 技術文書の書き方

文書の種別ごとに求められる構成と注意点が異なる。

- 手順書（セットアップ・デプロイ・移行など）:
  - 番号付きリストで手順を示す。各ステップは1つの動作に限定する
  - 前提条件・事前準備を冒頭に明記する
  - 成功時の期待結果を各ステップまたは末尾に記載する
  - 失敗時の対処やロールバック手順も含める
- トラブルシューティング:
  - 症状 → 原因 → 対処の3要素で構成する
  - よくある問題から順に並べる
  - エラーメッセージは検索しやすいように正確に引用する
- API文書:
  - エンドポイント・パラメーター・レスポンスを網羅する
  - リクエスト/レスポンスの具体例を必ず付ける
  - エラーケースとそのレスポンスも記載する
- CHANGELOG:
  - 読者はエンドユーザーである。内部リファクタリングの詳細は不要
  - 破壊的変更は目立つように記載し、移行手順を添える
  - 日付とバージョンを明記する
