# スキル（`.claude/skills/`、Agent Skills）編集ガイドライン

## スキルの役割

スキルはオンデマンドで読み込まれる拡張指示。
ユーザーが `/skill-name` で呼び出すか、Claudeが関連性を判断して自動で読み込む。

常に適用すべき規約・方針にはルール（`.claude/rules/`）を使う。

## ディレクトリ構造

```text
.claude/skills/<skill-name>/
├── SKILL.md           # メイン指示（必須、500行以下を目安）
├── references/        # 詳細なリファレンス（必要時に読み込み）
├── scripts/           # 実行可能スクリプト
└── examples/          # 出力例
```

サポートファイルはSKILL.mdから参照し、読み込みタイミングを明示する。

### Progressive Disclosure（段階的読み込み）

スキルは3層で読み込まれる。

1. メタデータ:（name + description）— 常にコンテキストに存在（～100語）
2. SKILL.md本体: — スキル呼び出し時に読み込み（500行以下を目安）
3. バンドルリソース: — 必要時にオンデマンドで読み込み（サイズ制限なし）

SKILL.mdが肥大化したら、詳細をリファレンスファイルに分離し、
SKILL.mdからは「いつ・何を読むか」のポインタだけ残す。
300行超のリファレンスには目次を付ける。

ドメインが複数ある場合はリファレンスを分離する。

```text
cloud-deploy/
├── SKILL.md          # ワークフローと選択ロジック
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```

## 配置場所

| 場所 | パス | 適用範囲 |
| --- | --- | --- |
| ユーザー | `~/.claude/skills/<name>/SKILL.md` | 全プロジェクト |
| プロジェクト | `.claude/skills/<name>/SKILL.md` | そのプロジェクトのみ |

## YAML frontmatter

```yaml
---
name: my-skill # /my-skill で呼び出し（省略時はディレクトリ名）
description: 何をするか、いつ使うか # トリガー判定に使われる（重要）
---
```

主要フィールドは以下の通り。

- `name`: スキル名。小文字・数字・ハイフンのみ、最大64文字
- `description`: スキルの説明。Claudeが自動トリガーするかの判定に直結する。
  Claudeはスキルを「使わない」方向に偏る傾向があるため、記述は積極的にする。
  ユーザーが自然に使うキーワードや、トリガーすべき具体的な状況を含める
  - 例:「社内データを表示するシンプルなダッシュボードを作成する。
    ユーザーがダッシュボード、データ可視化、社内メトリクスに言及したら使用する」
- `user-invocable`: ユーザーが `/name` で呼び出せるかどうか。既定値は `true` のため明示指定は冗長。
  ユーザーに呼び出させたくないスキル（Claudeのみが参照するバックグラウンド知識など）に限り `false` を指定する
- `disable-model-invocation: true`: Claudeによる自動起動を無効化し、`/name` の手動起動のみに限定する。
  副作用の大きいワークフロー（デプロイ・引き継ぎファイル出力など）で意図しない起動を防ぐ用途に使う
- `user-invocable: false` と `disable-model-invocation: true` の違い:
  前者はユーザーの手動起動を禁止しClaudeのみが呼び出し可能。
  後者はClaudeの自動起動を禁止しユーザーのみが手動起動可能。両者は逆方向の制限である
- `context: fork`: サブエージェントで実行。会話履歴にアクセスしない
- `agent`: `context: fork` 時のサブエージェントタイプ（例: `Explore`、`Plan`）
- `allowed-tools`: スキル実行時に許可なしで使えるツールを制限

## 変数

- `$ARGUMENTS`: 呼び出し時の全引数
- `$ARGUMENTS[N]` / `$N`: N番目の引数（0始まり）
- `${CLAUDE_SKILL_DIR}`: SKILL.mdのあるディレクトリパス
- `${CLAUDE_SESSION_ID}`: セッションID
- `` !`command` ``: 読み込み前にシェルコマンドを実行し、出力で置換

## 記述のコツ

- 指示は命令形で書く
- 「なぜそうするか」を説明し、MUSTやNEVERの多用を避ける。
  モデルは理由を理解すると想定外のケースにも適切に対応できる
- 出力フォーマットはテンプレートで明示する:

  ```markdown
  ## レポート構造

  # [タイトル]

  ## 概要

  ## 主な発見

  ## 推奨事項
  ```

- 具体例を含める（入力→出力の対で示すと効果的）
- テスト実行で繰り返し生成されるスクリプトはscripts/ にバンドルする
- 特定の例にだけ効く狭い指示は避け、汎用的に書く
- スキルの運用で繰り返し発生した失敗事例は `## Gotchas` セクションに蓄積する。
  再発防止に有効で、スキル本文の抽象的な指示を具体例で補う
- Gotchasの記述は禁止事項ではなく、「陥りやすい判断パターンと回避策」の形で書く（禁止事項の列挙はコンテキスト汚染を招くため）
- frontmatterで設定した仕様（`disable-model-invocation: true`による自動起動の無効化など）を
  `description`や本文で再記述しない。
  システム側で保証される挙動を繰り返すとコンテキストが増え、
  frontmatterと本文で表現に齟齬が生じた時の矛盾リスクも生じる
- `description`では起動条件の重複表現を避ける
 （例:「明示指示でのみ起動する。`/name`での手動トリガー専用」のように同じ意味の文を並べない）。
  重複はモデルがどちらを優先するかの揺れと、後の改訂時の片側更新漏れを招く
