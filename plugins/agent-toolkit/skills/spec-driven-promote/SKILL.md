# spec-driven昇格ワークフロー

次版ドキュメントを現行版ドキュメントへ統合し、`docs/v{next}/`を削除する。

## 手順

1. `agent-toolkit:spec-driven`スキルをSkillツールで呼び出し、用語・配置規約・テンプレートを把握する
2. 昇格対象の`docs/v{next}/`を確認する。
   `OVERVIEW.md`を除く`.md`ファイルを作業テーマドキュメントとして扱う
3. 各作業テーマドキュメントの「現行版ドキュメントの対応先」節から統合先を特定する。
   対応先が`docs/topics/`を指せば横断、`docs/features/`を指せば機能として統合する
4. 次版ドキュメントの内容を`docs/features/`または`docs/topics/`へ統合する（後述の統合方針に従う）
5. コード中の参照コメントを現行版ドキュメントのパスへ書き換える
6. `docs/v{next}/`を削除する。
   先に手順5を済ませておくことで、参照元と新パスの突き合わせが容易になる
7. `grep -r "docs/v{next}"`などで`docs/v{next}/`への参照が残っていないことを確認する
8. リンクと参照コメントの指し先が実在のファイルを指していることを検証する

## 統合方針

- 新規機能は`docs/features/`に新規ファイルとして作成する
- 既存機能の改修は対応先の現行版ドキュメントへ統合する
  - 現行版ドキュメントのタイトル・ファイル名は保持し、新規追加分は節として吸収する
  - 既存節がある場合は追記または上書き、無い場合は節を新設する（「却下した代替案」節も対象）
  - 事実が並立する場合（背景・スコープ・受け入れ基準など）は追記する
  - 判断が衝突する場合は新判断で上書きし、旧決定は「却下した代替案」へ廃止理由を一行添えて移す
- 横断ドキュメントは`docs/topics/`へ統合する。
  横断/機能の判定は作業テーマドキュメントの「現行版ドキュメントの対応先」節を根拠とする
- 「主対応」「関連」のうち「関連」として参照されているドキュメントは、
  新たな共通決定が含まれる場合のみ追記し、それ以外は現状維持とする
- 主要設計判断は現行の判断として整理する
- 却下した代替案は判断履歴として残す
- 次版ドキュメントの「現行版ドキュメントの対応先」節は統合先を案内する次版固有の記述なので、
  現行版へ持ち込まない
- `docs/v{next}/OVERVIEW.md`は次版作業の進行管理用ファイルなので統合せず、
  手順6での`docs/v{next}/`削除時に併せて破棄する
