# spec-driven昇格ワークフロー

`spec-driven`の運用では次期リリース作業期間中のみ開発中バージョンディレクトリ（`docs/v{next}/`配下）が存在する。
本スキルはその作業版ドキュメントを恒常配置（`docs/features/`・`docs/topics/`）へマージ統合する。
昇格により開発中バージョンディレクトリは消滅し、恒常配置のみが残る。

起動時にまず`Skill`ツールで`spec-driven`を呼び出し、用語・配置規約・粒度差の前提・テンプレートを読み込んでから本スキルの手順を実行する。
本スキル本文は昇格作業に固有の差分のみを記述する。

## 適用条件と非適用条件

`spec-driven`の「適用条件と非適用条件」に加え、本スキル固有の条件。

適用対象:

- spec-drivenのCleanupが完了し、リリースまで済んだ開発中バージョンディレクトリがある
- ユーザーが「リリース完了後のひと段落した時点」と判断したタイミング

非適用:

- spec-driven導入前のプロジェクトで既存`docs/v*/`群を恒常配置に起こしたい（`spec-driven-init`を使う）
- 開発中バージョンディレクトリが存在しない、または作業が未完了
- 昇格範囲を一部の作業テーマだけに限定したいがユーザー合意がとれていない

## 起動タイミング

本スキルは起動タイミングを規定しない。ユーザーがリリース後のひと段落した時点で判断して明示起動する。
早期昇格はリリース直後の追加修正の反映先判断を難しくするため、軽微な手直しが収束してから起動する。

## ディレクトリ対応

- `docs/v{next}/{作業テーマ名}.md` → `docs/features/{機能名}.md`または`docs/topics/{トピック名}.md`（マージ統合手順で確定）
- `docs/v{next}/{トピック名}.md` → `docs/topics/{トピック名}.md`（既存統合または新規作成）
- `docs/v{next}/README.md` → 昇格対象外（ディレクトリごと削除で消滅）
- 作業版`.md`内の「改修前スナップショット」節 → Before/After検証専用のため恒常配置へマージせず破棄

作業テーマと恒常配置の対応関係は1対1・1対多・多対1のいずれもありうる。
各作業版の冒頭「恒常配置の対応先」節と`README.md`の「作業テーマ一覧」を基に確定する。曖昧ならユーザーに確認する。

## 手順

1. 対象の開発中バージョンディレクトリをユーザーと合意する
2. 各作業版の対応関係（1対1・1対多・多対1・新規作成）を確定する
3. マージ統合する（次節「マージ統合の節別方針」参照）
4. 実装コード中の参照コメントを一括書き換える
   - 旧: `docs/v{next}/{作業テーマ名}.md`を指す参照コメント
   - 新: 手順2で確定した恒常配置パス
   - リポジトリ全体を対象とし、書き換え漏れを残さない
5. 開発中バージョンディレクトリを削除する（`README.md`を含めディレクトリごと削除し、gitに反映する）
6. 昇格後の整合性を検証する
   - `docs/v{next}/`への参照が残っていないか全文検索（コード・ドキュメント・`CHANGELOG.md`等）
   - 恒常配置の`.md`内リンクが壊れていないか（相対パスの破損）
   - 参照コメントの指し先が実在するか
7. 完了と検証結果をユーザーに報告する。変更履歴の置き場（`CHANGELOG.md`・`docs/releases/`等）はプロジェクト方針に従う

## マージ統合の節別方針

対応先ごとにdiffを生成し、節単位で以下を適用してユーザーに提示する。合意を得たら統合を適用する。
1対1かつ同粒度で完全置換が妥当なケースは節別統合を省き上書き更新でよい。
迷う場合は節別方針を採用する（情報欠落のリスクを避けるため）。

- 背景: 既存の末尾に追記して重複は統合する。再整理する内容なら置換も可（ユーザー合意必須）。新規機能は初版として書き出す
- スコープ・受け入れ基準: 節別に統合し、重複は統合・矛盾は置換（ユーザー合意必須）。
  受け入れ基準の旧基準破棄が意図的であることを確認する
- 主要設計判断: 矛盾しない項目は追記、見直した項目は置換（変更理由を節内に明記）
- 却下した代替案: 原則追記のみ（過去の判断を消さない）。再検討が発生した場合は経緯と結論を既存エントリに追記する

既存改修の上書き更新時は、作業版`.md`の「改修前スナップショット」節と既存の恒常配置を照合し、
意図的差分が反映されているか確認する（スナップショット節自体は恒常配置へ転記しない）。

## 中断・リトライ

検証手順6で失敗した場合は、該当箇所のみ手順4以降をやり直す。
分類ミスに気づいた場合は該当ファイルを別ディレクトリ（`features`↔`topics`）へ移動し、参照コメントも追随する。
重大な差し戻しが必要な場合は`git restore`または`git reset`でコミット前の状態へ戻してから再実行する
（本スキルの手順はローカル作業に閉じておりリモートへの影響はない）。
部分昇格は原則非推奨だが、ユーザー合意があれば一部ファイルのみ先行昇格してもよい
（その場合は残ファイルと`README.md`を保持し、ディレクトリは削除しない）。

## 制約

- 引数無しで起動する。対象ディレクトリの選定と対応関係の確定はユーザーとの対話で行う
- 恒常配置の上書きまたは統合前にdiffを必ずユーザーに提示して合意を得る
- 参照コメントの書き換え漏れは昇格後のリンク切れに直結するため、手順4と検証手順6は必ず実施する
