# spec-driven昇格ワークフロー

spec-drivenスキルの運用では、次期リリース作業期間中のみ開発中バージョンディレクトリ（`docs/v{next}/`配下）が存在する。
本スキルはこの作業版ドキュメントを恒常配置（`docs/features/`・`docs/topics/`）へマージ統合する。
昇格により開発中バージョンディレクトリは消滅し、恒常配置のみが残る。

## 本スキル起動時の前提

本スキルは`spec-driven`スキルに依存する。
配置規約・テンプレート・ディレクトリ構造などの共通仕様は`spec-driven`側に集約しており、本スキルは昇格作業に特化した差分のみ記述する。

起動時に最初に以下を行う。

1. `Skill`ツールで`spec-driven`を呼び出し、配置規約・粒度差の前提・テンプレートを読み込む
2. 読み込んだ内容を前提に本スキルの手順を実行する

`spec-driven`を読み込まずに進めると配置パスやテンプレートが揺らぎ、マージ統合の判断も不正確になるため、本ステップは必ず行う。

## 適用条件と非適用条件

`spec-driven`の「適用条件と非適用条件」に加え、本スキル固有の条件。

適用対象:

- spec-drivenのCleanupが完了し、リリースまで済んだ開発中バージョンディレクトリがある
- ユーザーが「リリース完了後のひと段落した時点」と判断したタイミング
- 恒常配置へマージ統合することで整合性維持の負担を軽減したい

非適用:

- spec-driven導入前のプロジェクトで既存`docs/v*/`群を恒常配置に起こしたい（`spec-driven-init`スキルを使う）
- 開発中バージョンディレクトリが存在しない、または作業が未完了
- 昇格範囲を一部の作業テーマだけに限定したいがユーザー合意がとれていない

## 起動タイミングの判断

本スキルは起動タイミングを規定しない。
ユーザーがリリース完了後のひと段落した時点で判断し、明示的に起動する。
早期に昇格するとリリース直後の追加修正の反映先判断が難しくなるため、リリース後の軽微な手直しが収束したのを確認してから起動することを推奨する。

## ディレクトリ構造の対応

| 昇格前（開発中配置） | 昇格後（恒常配置） |
| -- | -- |
| `docs/v{next}/{作業テーマ名}.md` | `docs/features/{機能名}.md`または`docs/topics/{トピック名}.md`（対応先はマージ統合手順で確定） |
| `docs/v{next}/{トピック名}.md` | `docs/topics/{トピック名}.md`（既存を統合更新、なければ新規作成） |
| `docs/v{next}/README.md` | 昇格対象外。ディレクトリごと削除で消滅する |
| `docs/v{next}/{作業テーマ名}.working.md` | 存在しない前提。残っていれば事前にCleanup漏れとして報告し、対応を確認する |

作業テーマと機能／横断の対応関係は複数パターンがある。

- 1対1: 作業テーマが恒常配置の1つの機能・横断と対応（単純統合）
- 1対多: 1つの作業テーマが恒常配置の複数機能を更新（例: 横断的な方針変更）
- 多対1: 複数の作業テーマが恒常配置の同一機能を更新（例: 機能の複数側面への追加）

各作業版の冒頭の「恒常配置の対応先」節と、`.working.md`（昇格前に削除済みのはず）、および`README.md`の「作業テーマ一覧」の併記情報を基に、対応関係を確定する。
区分が曖昧ならユーザーに確認する。

## 手順

1. 対象の開発中バージョンディレクトリをユーザーと合意する
   - `docs/v*/`配下のディレクトリを列挙し、候補を提示する
   - 昇格対象と除外対象（作業途中・リリース未完了など）を明示的に確認する
   - 対象ディレクトリ内の`.working.md`残存有無を確認する。残っていれば昇格を保留しユーザーに対応を仰ぐ
2. 各作業版の対応関係を確定する
   - 各`docs/v{next}/{作業テーマ名}.md`の「恒常配置の対応先」節と`README.md`の「作業テーマ一覧」を起点に分類する
   - 1対1・1対多・多対1のいずれかに分類し、曖昧なファイルはユーザーに確認してから確定する
   - 新規機能で恒常配置に対応先がない場合は「新規作成」として扱う
3. マージ統合する（詳細は「マージ統合の節別方針」参照）
   - 1対1かつ同粒度で完全置換が妥当なケースは上書き更新
   - 1対多・多対1・節単位の統合が必要なケースはdiff提示→ユーザー合意→統合適用の半自動フロー
   - 新規作成ケースは恒常配置に初版として書き出す
4. 実装コード中の参照コメントを一括書き換えする
   - 対象: `docs/v{next}/{作業テーマ名}.md`を指す参照コメント
   - 書き換え先: 手順2で確定した恒常配置のパス（`docs/features/{機能名}.md`または`docs/topics/{トピック名}.md`）
   - 検索はリポジトリ全体を対象とし、書き換え漏れを残さない
5. 開発中バージョンディレクトリを削除する
   - `docs/v{next}/`配下の`README.md`を含め、ディレクトリごと削除する
   - gitで追跡されている場合は削除をステージに載せる
6. 昇格後の整合性を検証する
   - `docs/v{next}/`への参照が残っていないか全文検索で確認する（`docs/`配下・ソースコード・`CHANGELOG.md`など）
   - 恒常配置の`.md`ファイル内リンクが壊れていないか確認する（相対パスがディレクトリ変更で破損していないか）
   - 参照コメントの指し先が実在するか確認する
7. ユーザーに昇格完了と検証結果を報告する
   - 変更履歴の置き場（`CHANGELOG.md`・`docs/releases/`など）はプロジェクト方針に委ねる（本スキルでは規定しない）
   - リリース済みバージョンの記録が必要な場合はユーザーが別途対応する

## マージ統合の節別方針

恒常配置ドキュメントの各節ごとに、統合方針（追記・置換・併記）を使い分ける。
対応先ごとにdiffを生成し、節単位で以下を適用してユーザーに提示する。合意を得たら統合を適用する。

### 背景

- 既存機能の改修: 原則追記（背景は履歴的情報が蓄積する節のため）。作業テーマの背景が既存背景を再整理する内容なら置換も可。ユーザー合意を得る
- 新規機能: 初版として書き出す

### スコープ

- 「含むもの」「含まないもの」を節別に統合する
- 既存の項目と作業テーマの項目を併記し、重複は統合、矛盾は置換（改修が新方針を採用した場合）。矛盾解決はユーザー合意必須

### 受け入れ基準

- 改修により基準が変わる場合は置換、追加される場合は追記
- 旧基準の破棄が意図的であることをユーザーに確認する

### 主要設計判断

- 現行判断と矛盾しない項目は追記
- 現行判断を見直した項目は置換（ただし変更理由を節内に明記）
- 作業テーマ側で「変更理由と旧判断との差分」が記述されていればそれを活かす

### 却下した代替案

- 原則追記のみ（過去の判断を消さないため）
- 既存エントリと同一の代替案が再検討された場合は、再検討の経緯と結論を既存エントリに追記する

## 1対1対応の場合の簡略手順

作業テーマと恒常配置が1対1で同粒度の場合、節別統合を省略して上書き更新してもよい。

- 開発中`docs/v{next}/{作業テーマ名}.md`を恒常配置（`docs/features/{機能名}.md`または`docs/topics/{トピック名}.md`）へ上書き
- 冒頭の「恒常配置の対応先」節は恒常配置側には不要なので削除する
- 既存改修の場合は恒常配置の既存ファイルを上書き更新する（`.working.md`内の「改修前の該当節スナップショット」と照合し、意図的差分が反映されているか確認する）

ただし、恒常配置と作業テーマの粒度が異なる場合（1対多・多対1、または節の一部のみ更新）は、上書きではなく「マージ統合の節別方針」に従う。
迷った場合は節別方針を採用する（情報欠落のリスクを避けるため）。

## 昇格中断・リトライ

検証手順6で失敗した場合の扱い。

- 参照コメントの書き換え漏れ: 手順4へ戻り、該当箇所を追加で書き換える
- 恒常配置のリンク切れ: 新規作成された恒常配置`.md`のリンク（相対パス）を修正する
- 分類ミスに気づいた場合: 該当ファイルを恒常配置の別ディレクトリ（`features`↔`topics`）へ移動し、参照コメントも追随する
- マージ統合の誤り（節の重複や欠落、矛盾解決の失敗）: 該当節の統合をやり直す。`git diff`で差分を確認し、ユーザーと統合方針を再合意する
- 部分昇格: 原則として一括昇格を推奨するが、ユーザー合意があれば一部ファイルのみ先行昇格してもよい。その場合は残ファイルと`README.md`を保持し、ディレクトリは削除しない

重大な差し戻しが必要な場合は、`git restore`または`git reset`でコミット前の状態へ戻してから再実行する。
本スキルの手順はローカル作業に閉じており、リモートへの影響はない。

## spec-drivenスキルとの関係

- 本スキルは`spec-driven`スキルの後段階として機能する。通常の機能開発サイクルでは`spec-driven`で開発中ディレクトリを使い、リリース完了後に本スキルで昇格する
- 既存プロジェクトへの初回導入（恒常配置の初版整備）は`spec-driven-init`スキルが担う
- 本スキル実行後、通常のspec-drivenワークフローは恒常配置のみを参照して進められる状態となる

## 制約

- 起動は手動（`/agent-toolkit:spec-driven-promote`）または`spec-driven`からの明示呼び出しのみ。ユーザー相談への自動起動はしない
- 引数無しで起動する。対象ディレクトリの選定と対応関係の確定はユーザーとの対話で行う
- 恒常配置の既存ファイルを上書きまたは統合するため、上書き・統合前の差分を必ずユーザーに提示して合意を得る
- 参照コメントの書き換え漏れは昇格後のリンク切れに直結するため、手順4と検証手順6は必ず実施する
