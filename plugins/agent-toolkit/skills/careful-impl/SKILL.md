# 計画後のセルフレビュー付き計画実行

本スキルは、`agent-toolkit:plan-mode`での計画ファイル合意後の、計画実行手順を示す。

1. タスク分解
2. 実装
3. 検証
4. コミット
5. 恒久化計画チェック
6. レビュー

これらの工程では、計画ファイルを唯一の入力として参照する。
計画ファイルがフェーズ分けされていた場合は、「2. 実装」「3. 検証」「4. コミット」を各フェーズ分繰り返す。
全フェーズ・全コミットが完了したら「5. 恒久化計画チェック」「6. レビュー」を計画全体に対して一度だけ実施する。

## 共通制約

全工程のサブエージェント呼び出しに共通する制約。

- gitコミット・pushは行わないこと（コミットはメインが直接行う）

## 1. タスク分解

計画ファイルを起点にタスク分解し、`TodoWrite`へ登録する。
タスク分解時に依存関係を洗い出し、検証・コミット・レビュータスクまで登録する。

タスク分解時は、サブエージェントの活用を前提とし、以下の点を考慮した適切な粒度に分解する。

- 時間効率（依存関係のない部分は並列化）
- コンテキスト効率（1エージェントあたりの作業量を増やし過ぎない）
- コスト効率（簡単なものは極力sonnet/haikuモデルを活用）

ただし、分割のし過ぎは全ての効率を悪化させるため、細かく分割し過ぎないことに注意。
（それぞれのサブエージェントが計画ファイルや必要スキルなどを読むため）
小規模なタスクであればメイン側で直接実行した方がよい場合もある。
（文脈が限られたり精度の低いモデルで作業すると、成果物の品質が低下し、後の工程で手間が増えるため）

検証タスクは上記の効率を考慮して実装タスクとまとめてもよい。

ただし、レビュータスクは必ず独立させ、観点毎に別のサブエージェントを起動する。
（実装の経緯を持ち込まず、クリーンな状態でレビューさせるため）
レビュータスクも、上記の効率を考慮し、1観点に対して複数のサブエージェントを起動してもよい。
また、最終判断はメイン側（opus/sonnet）が行う前提でより安価なモデルを使用してもよい。

コミットはメインが直接行う。（通常小規模なタスクのため）

サブエージェント起動時のモデルは、`Agent`ツールの`model`パラメーターで指定する
（`"haiku"`・`"sonnet"`・`"opus"`のいずれか）。
プロンプト本文にモデル名を書いてもサブエージェントの動作モデルは変わらない。
モデルの使い分け基準は次の通り。

- `haiku`: 単純で確定的な作業（明確な条件での検索・抽出、単純なファイル修正・置換など）
- `sonnet`: 軽度の推論が必要な作業（方向性の固まっている小規模なコーディング・文書記述・レビューなど）。既定
- `opus`: 創造的で難度の高い作業（バグの原因調査や方針が未確定のコーディングなど）。コスト最高のため最終手段

## 2. 実装

分解されたタスクに従い、実装タスクを`careful-implementer`へ委譲する。
`careful-implementer`のfrontmatterでは`agent-toolkit:coding-standards`と
`agent-toolkit:writing-standards`を自動ロードする。
そのため呼び出しテンプレートにスキル呼び出し指示は含めない。

呼び出しテンプレート:

```text
以下のタスクを実装してください。

- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`

## タスク

{タスクの具体的な記述}

## 修正指摘（該当時のみ）

{メインが統合した指摘テキストをここに直接貼り付ける}
```

## 3. 検証

全実装タスク完了後、検証を実施する。
検証は「1. タスク分解」節の効率判断に従い、実装タスクに統合しても独立タスクとして分離してもよい。
独立タスクとして`careful-implementer`へ委譲する場合は、下記テンプレートを使用する。

```text
以下のタスクを実施してください。

- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`

## タスク

{検証内容の具体的な記述。例: `uv run pyfltr run-for-agent` を実行し、警告ゼロ・テストグリーンを確認する等}
```

## 4. コミット

各実装・検証タスクが合格したら、計画ファイルの`## 実行方法`節「コミット方針」項目に従ってコミットする。
複数フェーズ計画ではフェーズ単位の中間コミット、複数コミット計画ではコミット単位の中間コミットになる。
中間コミットでは恒久化計画チェックを行わない（次の「## 5. 恒久化計画チェック」で一括実施するため）。

「全コミット完了」の判定はメイン側が行う。
判定基準は「計画ファイルの`## 実行方法`節『コミット方針』項目に列挙された全コミットが`HEAD`に反映された状態」である。
単一コミット計画の場合は当該1コミットを終えた時点、複数コミット計画では最終コミット完了時点が該当する。
判定後、次の「## 5. 恒久化計画チェック」へ進む。

## 5. 恒久化計画チェック

全コミット完了後、メイン側で以下のチェックを実施する。
本チェックは、`## 変更内容`への重複列挙を廃した運用において恒久化漏れを検知する唯一のガードである。
本チェックが省略されると、durableな知見が永続化先に反映されないまま計画ファイルとともに失われる。
レビュー前の位置で実施することで、反映漏れに対する追加コミットも次の「## 6. レビュー」の対象差分に含める。

- 計画ファイルの`## 恒久化計画`を再読し、列挙された各項目が永続化先へ反映済みであることを確認する
- 計画ファイルの`## リファクタリング計画`を再読し、列挙された各項目が反映済みであることを確認する
- 実装中に新規発見したdurableな知見（設計判断・調査で得た制約・横展開すべき類似箇所への指針など）があれば、
  `## 恒久化計画`に追加し、対応する永続化先の編集も併せて行う
  - 発掘経路として、各実装サブエージェント報告の`changed`行・`escalation_reason`・「計画外編集や予定外知見」報告欄を読み返し、
    計画には記載されていなかった編集や追加判断を拾い出す
  - 拾い出した項目のうち、計画固有でない判断基準・横展開すべき指針・プロジェクト方針に繋がるものを`## 恒久化計画`へ追加する
- 反映漏れや追加が必要な編集を発見した場合は、追加コミットで永続化先を更新する

## 6. レビュー

全コミット・恒久化計画チェック完了後、計画全体の差分を対象として一度だけレビューを実施する。
レビュー工程ではBashは読み取り系操作（`git diff`・`git status`・`git log`・`ls`・`rg`相当）に限定し、ファイルへの書き込みは行わない。

### 初回レビュー（2種別並列）

`careful-spec-reviewer`と`careful-impl-reviewer`を常に並列で起動する。
計画ファイル内の起動要否指定は参照せず、両種別を必ず起動する。

### 初回レビュー呼び出しテンプレート

共通骨子を使い、種別固有の欄を差し替えて2エージェントを並列起動する。
初回レビューでは対象範囲の指摘をすべて洗い出すこと。
再レビュー（followup）では新規指摘を追加しない前提で、この初回で見つかった問題をすべて列挙する。

プレースホルダー`{レビュー種別名}`には`spec`・`impl`のいずれかを記入する。

```text
以下の実装について{レビュー種別名}レビューを実施してください。

- 担当観点: {担当観点}
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
- 対象範囲: 計画全体（全コミット完了後の差分）
- 対象外ファイル: {一時ファイル一覧}
- 差分取得: `git diff {基準点}..HEAD`と、未追跡ファイルを含む作業ツリー参照で行う
  - `{基準点}`は計画着手前のコミットを指す不変の参照点を選ぶ。例: `origin/master`相当の上流ブランチHEAD、
    または計画着手前に控えておいた特定コミットSHA。複数コミット計画でも基準点は変えない

## 制約

- 指摘を小出しにせず、一度のレビューですべての問題点を洗い出すこと
```

レビュー種別ごとの担当観点:

| 種別 | 起動エージェント | 担当観点 |
| --- | --- | --- |
| `spec` | `careful-spec-reviewer` | 計画ファイルと成果物の仕様適合性 |
| `impl` | `careful-impl-reviewer` | コード単体品質・ドキュメント単体品質・成果物間の整合性 |

### 指摘の統合と修正依頼

各レビュアーの戻り値本文を受け取り、メインが指摘を統合する。
必要に応じてユーザーへ確認して対応方針を決定する（計画に照らし合わせて妥当な修正であれば確認無しで修正してよい）。

最終的に対応が必要な指摘が1件以上あれば、どのような修正をするかをユーザーに簡潔に報告したうえで、当該タスクを`careful-implementer`へ依頼する。
修正の再実装が完了したら検証タスクを再起動して合格を確認し、修正反映の追加コミットを行う。

### 再レビュー（followup）

修正反映コミット後、`careful-followup-reviewer`（モデル`haiku`）を1件起動し、前回統合指摘への対応状況を確認する。

ユーザーが「対応不要」と判断した指摘は、判断理由とともに統合済みテキスト内の当該指摘の行末尾に
`（ユーザー判断: 不対応 — <理由>）`の書式で注記する。
修正プロンプトの「修正指摘」欄には対応する指摘のみを埋め込み、不対応と判断された指摘は除外する。

followupでは通常の新規指摘を追加しない（初回レビューで網羅されている前提）。
例外として、前回指摘対応の副作用で生じた明らかな回帰のみ`regression`欄に末尾併記される。

### followup呼び出しテンプレート

「## 前回指摘」欄には、前節「指摘の統合と修正依頼」で管理している統合済み指摘テキストを転記する。
各指摘行末尾に`（ユーザー判断: 不対応 — <理由>）`注記が付いている指摘は、`careful-followup-reviewer`の対応確認対象から除外される前提で注記付きのまま転記する。

```text
以下の実装についてfollowupレビューを実施してください。

- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
- 対象範囲: 計画全体（全コミット完了後の差分）
- 差分取得: `git diff {基準点}..HEAD`と、未追跡ファイルを含む作業ツリー参照で行うこと（`{基準点}`は初回レビューと同一の参照点を使う）

## 前回指摘

- `path/to/file.py:L10` — 戻り値の型アノテーションが欠けている
- `path/to/file.py:L20` — 変数名が不明瞭 （ユーザー判断: 不対応 — 現状仕様で問題なし）
```

### 修正ループのエスカレーション

`careful-followup-reviewer`が`missing`または`partial`を1件以上返した場合、修正対象として扱う。
`regression`は前回指摘対応の副作用で生じた明らかな回帰のみを記録する欄であり、検出された場合は例外的に修正対象へ加える。
通常の新規指摘はfollowupでは追加しない（初回レビューで網羅されている前提）。
これら「妥当な未対応指摘が残った状態」が連続2回発生した場合、修正ループを中断してユーザーへエスカレーションする。
具体的には1回目のfollowupで`missing`/`partial`/`regression`のいずれかが残り、修正再実装後の2回目のfollowupでも残る状態を指す。

全レビュー合格で作業終了。
