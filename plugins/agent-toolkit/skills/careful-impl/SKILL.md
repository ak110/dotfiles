# 計画後のセルフレビュー付き計画実行

本スキルは、`agent-toolkit:plan-mode`での計画ファイル合意後の、計画実行手順を示す。

1. タスク分解
2. 実装
3. 検証
4. レビュー
5. コミット

これらの工程では、計画ファイルを唯一の入力として参照する。

計画ファイルがフェーズ分けされていた場合は、上記の工程を各フェーズ分繰り返す。

## 1. タスク分解

計画ファイルを起点にタスク分解し、`TaskCreate`へ登録する。
タスク分解時に依存関係を洗い出し、検証・レビュー・コミットタスクまで登録する。

タスク分解時は、サブエージェントの活用を前提とし、以下の点を考慮した適切な粒度に分解する。

- 時間効率（依存関係のない部分は並列化）
- コンテキスト効率（1エージェントあたりの作業量を増やし過ぎない）
- コスト効率（簡単なものは極力sonnet/haikuモデルを活用）

ただし、分割のし過ぎは全ての効率を悪化させるため、細かく分割し過ぎないことに注意。
（それぞれのサブエージェントが計画ファイルや必要スキルなどを読むため）
小規模なタスクであればメイン側で直接実行した方がよい場合もある。
（文脈が限られたり精度の低いモデルで作業すると、成果物の品質が低下し、後の工程で手間が増えるため）

検証タスクは上記の効率を考慮して実装タスクとまとめてもよい。

ただし、レビュータスクは必ず独立させ、観点毎に別のサブエージェントを起動する。
（実装の経緯を持ち込まず、クリーンな状態でレビューさせるため）
レビュータスクも、上記の効率を考慮し、1観点に対して複数のサブエージェントを起動してもよい。
また、最終判断はメイン側（opus/sonnet）が行う前提でより安価なモデルを使用してもよい。

コミットはメインが直接行う。（通常小規模なタスクのため）

サブエージェント起動時のモデルは、`Agent`ツールの`model`パラメーターで指定する
（`"haiku"`・`"sonnet"`・`"opus"`のいずれか）。
プロンプト本文にモデル名を書いてもサブエージェントの動作モデルは変わらない。
既定は`sonnet`とし、単純で確定的な作業は`haiku`、創造的で難度の高い作業のみ`opus`を検討する。

## 2. 実装

分解されたタスクに従い、実装タスクを`careful-implementer`へ委譲する。
`careful-implementer`のfrontmatterでは`agent-toolkit:coding-standards`と
`agent-toolkit:writing-standards`を自動ロードする。
そのため呼び出しテンプレートにスキル呼び出し指示は含めない。

呼び出しテンプレート:

```text
以下のタスクを実装してください。

- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`

## タスク

{タスクの具体的な記述}

## 修正指摘（該当時のみ）

{メインが統合した指摘テキストをここに直接貼り付ける}

## 制約

- gitコミット・pushは行わないこと
```

## 3. 検証

全実装タスク完了後、検証を実施する。
検証は「1. タスク分解」節の効率判断に従い、実装タスクに統合しても独立タスクとして分離してもよい。
独立タスクとして`careful-implementer`へ委譲する場合は、下記テンプレートを使用する。

```text
以下のタスクを実施してください。

- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`

## タスク

{検証内容の具体的な記述。例: `uv run pyfltr run-for-agent` を実行し、警告ゼロ・テストグリーンを確認する等}

## 制約

- gitコミット・pushは行わないこと
```

## 4. レビュー

検証タスクの完了後、レビュータスクを開始する。

### 初回レビュー（2種別並列）

`careful-spec-reviewer`と`careful-impl-reviewer`を常に並列で起動する。
計画ファイル内の起動要否指定は参照せず、両種別を必ず起動する。

### 初回レビュー呼び出しテンプレート

種別ごとに対応するエージェント（`careful-spec-reviewer`・`careful-impl-reviewer`）を
`Agent`ツールで並列起動する。共通骨子を使い、種別固有の欄を差し替える。
初回レビューでは対象範囲の指摘をすべて洗い出すこと。再レビュー（followup）では新規指摘を追加しない前提で、
この初回で見つかった問題をすべて列挙する。

`careful-spec-reviewer`はスキル事前ロード無しで起動する。
`careful-impl-reviewer`のfrontmatterでは`agent-toolkit:coding-standards`と
`agent-toolkit:writing-standards`を自動ロードする。
そのためいずれもプロンプト側にスキル呼び出し指示は含めない。

プレースホルダー`{レビュー種別名}`には`spec`・`impl`のいずれかを記入する。

```text
以下の実装について{レビュー種別名}レビューを実施してください。

- 担当観点: {担当観点}
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
- 対象フェーズ: {単一フェーズ計画では「全体」、複数フェーズ計画では「フェーズN: 題目」}
- 対象外ファイル: {一時ファイル一覧}
- 差分取得: `git diff HEAD`と、未追跡ファイルを含む作業ツリー参照で行う

## 制約

- Bashは読み取り系操作に限定すること。ファイルへの書き込みは行わないこと
- 一度のレビューですべての問題点を洗い出すこと（指摘を小出しにせず初回で徹底的に指摘すること）
- gitコミット・pushは行わないこと
```

レビュー種別ごとの担当観点:

| 種別 | 起動エージェント | 担当観点 |
| --- | --- | --- |
| `spec` | `careful-spec-reviewer` | 計画ファイルと成果物の仕様適合性 |
| `impl` | `careful-impl-reviewer` | コード単体品質・ドキュメント単体品質・成果物間の整合性 |

### 指摘の統合と修正依頼

各レビュアーの戻り値本文を受け取り、メインが指摘を統合する。
必要に応じてユーザーへ確認して対応方針を決定する。
（計画に照らし合わせて妥当な修正であれば確認無しで修正してよい）

最終的に、対応が必要な指摘が1件以上あれば、
どのような修正をするかをユーザーに簡潔に報告したうえで、
当該タスクを`careful-implementer`へ依頼する。

### 再レビュー（followup）

修正の再実装が完了したら、検証タスクを再起動して合格を確認する。
その後`careful-followup-reviewer`（モデル`haiku`）を1件起動し、前回統合指摘への対応状況を確認する。

ユーザーが「対応不要」と判断した指摘は、判断理由とともに統合済みテキスト内の当該指摘の行末尾に
`（ユーザー判断: 不対応 — <理由>）`の書式で注記する。
修正プロンプトの「修正指摘」欄には対応する指摘のみを埋め込み、
不対応と判断された指摘は除外する。

再レビューでは新規指摘を原則追加しない（明らかな回帰が発見された場合のみ末尾に短く併記する）。

### followup呼び出しテンプレート

「## 前回指摘」欄には、前節「指摘の統合と修正依頼」で管理している統合済み指摘テキストを転記する。
各指摘行末尾に`（ユーザー判断: 不対応 — <理由>）`注記が付いている指摘は、
`careful-followup-reviewer`の対応確認対象から除外される前提で、注記付きのまま転記する。

```text
以下の実装についてfollowupレビューを実施してください。

- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
- 対象フェーズ: {単一フェーズ計画では「全体」、複数フェーズ計画では「フェーズN: 題目」}
- 差分取得: `git diff HEAD`と、未追跡ファイルを含む作業ツリー参照で行うこと

## 前回指摘

- `path/to/file.py:L10` — 戻り値の型アノテーションが欠けている
- `path/to/file.py:L20` — 変数名が不明瞭 （ユーザー判断: 不対応 — 現状仕様で問題なし）

## 制約

- Bashは読み取り系操作に限定すること。ファイルへの書き込みは行わないこと
- gitコミット・pushは行わないこと
```

### 修正ループのエスカレーション

followupレビューで妥当な未対応指摘が残る結果が連続2回発生した場合、
修正ループを中断してユーザーへエスカレーションする。
（1回目のfollowupで未対応指摘が残り、修正再実装後の2回目のfollowupでも未対応指摘が残る状態を指す）

## 5. コミット

全レビューに合格したら、計画ファイルの`## 実行方法`節「コミット方針」項目に従ってコミットする。
単一フェーズ計画ではこのコミットで作業完了。
複数フェーズ計画ではフェーズ単位の中間コミットになり、コミット後に次フェーズのタスク分解へ戻る。
