# 実現可能性の事前検証の細則

本ファイルは`investigation-rules.md`「実現可能性の事前検証の細則」節から分離した実機検証手順の詳細である。

外部依存物（自身の管理外コードに相当するOSS・ライブラリ・SDK・プラグイン・外部CLI・外部サービス等）に
依存する計画では、表面的なAPI仕様（公開関数・コマンドフラグ・設定キーの型と既定値）だけで前提を確定しない。
依存物のソース・公式ドキュメントの実装節・既知issue・テストコードのいずれかを読み、
次の観点を実機観測に先行して確認し`## 調査結果`へ転記する。

- 入力値ごとに分岐する内部処理経路
- 関連する他設定・グローバル状態・環境変数への副次書き込み
- 読み込み順・評価順・初期化順への依存
- エラー経路・リトライ・タイムアウトの暗黙仕様

自プロジェクト内CLI・モジュールの内部挙動を計画文の根拠とする場合も対象関数本体のみで前提を確定せず、
エントリポイント・前段フック処理まで読み込んで分岐順序と副作用を確認したうえで`## 調査結果`へ反映する。

実機検証が環境制約（Bash実行不可・対象ツール未インストール等）で実行不能な場合は
当該前提を採用した計画を確定させない。工程6へ進む前に環境制約の解消を待つか、
当該前提に依存しない代替設計へ切替える。代替設計への切替えが困難な場合は
`AskUserQuestion`で環境制約の解消可否をユーザーへ確認する分岐を許容する。

設定ファイル変更・CLI挙動変更を伴う計画では、対象CLIに検証用サブコマンド
（`critique`・`lint`・`validate`・`check`・`config`・`defaults`等）があれば計画段階で実機実行する。
出力に応じて`## 変更内容`の文面案・`## 実行方法`の検証ステップを`## 調査結果`へ転記のうえ確定する。

入力文字列の形式判別を伴う設計（URL対ローカルパス、正規化済み識別子対リテラル等）では、
判別ロジックの候補を3〜5の代表入力に対し机上または実機で適用し、結果を`## 調査結果`へ転記する。

正規表現パターン・glob記法（`**`・`*`・`?`等）を計画文・テストコードへ転記する場合の検証手順を定める。
実機検証（`python -c "import re; print(bool(re.search(<pattern>, <input>)))"`）または公式仕様照合を実施する。
代表入力（任意階層・単一階層など）に対する結果を`## 調査結果`へ転記する（机上推定で確定しない）。

新規正規表現パターンを追加または既存パターンを拡張する計画では候補文字列を既存の機械検出辞書と照合する。
照合対象は`_match_scope_escalation`関数・`pyfltr.colloquial.check`（`load_patterns`で辞書ロード後`first_hit`で判定）等とし、検出時は代替表現へ差し替え非検出まで再検査する。

規範文追記に検出コマンド（`grep`・`find`・`rg`等）を含める場合、対象ファイル群に対し計画段階で実測し
想定ヒットを`### 事実確認済み事項`へ記録してから反映文言を確定する。行折り返し・複数行に跨る対象は
行単位検索で捕捉できないため、Read通読または複数行対応オプション（`grep -P -z`等）と組み合わせる案を並記する。

計画本文に外部CLI（`gh`・`codex`・`atk`・`git`等）の具体的な引数構成を記述する場合、
対象CLIの`--help`または公式ドキュメント参照で実在確認し、実測ソースを`### 事実確認済み事項`小見出しへ記録する。

計画本文へ新設の条件分岐（既存コードに対する`[置換後]`ブロック内の`if`/`else`等）を含める場合、当該分岐が依拠する2つ以上の集合・範囲・スコープの間に包含関係の成立可否を確認する。包含関係とは部分集合・部分文字列・上位スコープと下位スコープの関係等を指す。成立する場合は対象分岐が到達可能であることを最低1つの具体的な入力例（反例トレース）で計画段階に確認したうえで`### 事実確認済み事項`へ記録する。

pylint採用プロジェクトでPythonファイルを追加・移動・追記する計画では現行行数を`wc -l`で確認する。`max-module-lines`（既定1000行）超過見込み時は責務分離を優先する（抑制コメントの新規追加は`agent-toolkit:coding-standards`規定で不採用とする）。

規範文書への追記案を計画する場合は追記先候補ファイルの現行行数を`wc -l`で実測する（実測の対象範囲は`plan-file-guidelines.md`「6-1. 参照テンプレート読み込み」節を参照する）。`agent-toolkit:agent-standards`「文書サイズ上限」節の対象文書に該当するときは、現行行数と追記量の概算から改訂後200行超過が見込まれるか判定する（厳密な事前計算は不要）。
超過が見込まれる場合は追記先を代替候補へ切り替えるか、同一計画スコープへ既存記述の縮減対応を組み込む。

非同期リソース（Blob URL・EventSource・setInterval・AbortController等）の生成・キャッシュ・破棄を扱う計画を対象とする。自プロジェクト内フレームワーク共通ガード（認可・CSRF・ロケール制約・レートリミット等）を扱う計画も対象とする。
非同期リソースは3経路（取得中の対象削除・取得中のコンポーネント破棄・キャッシュ内の非アクティブ検出）を計画本文で明示する。
フレームワーク共通ガードは対象エンドポイントの現行ガード適用状況を事前に実機観測して`## 調査結果`へ転記し、既存機能の権限モデルへリンク・委譲する設計を採用する場合は認可条件を事前に実機確認する。

- `agent-toolkit/rules/01-agent.md`「品質最優先」節の外部ライブラリ・SDKのAPI仕様・動作前提を
  公式リファレンスと実機再現で裏付ける規定を、計画段階のAPI名断定へ適用する細則は次のとおり。
  - 計画本文へ外部ライブラリ・SDKのAPIの実在・非実在を断定して書く前に、裏付けを取る。
    対象APIは関数名・メソッド名・オプション名・クラス名・`import`パス等を含む
  - 裏付け手段は対象バージョンの公式リファレンス参照および実機確認のいずれかとする。
    実機確認手段は当該ライブラリソースツリーへの`grep`・`import`後の`dir`列挙等とする
  - 裏付けが取れない場合は`### 却下した代替案`側へ推測扱いとして分離し、確定命題として書かない。
    結果は`### 事実確認済み事項`へ記録する
  - 捕捉対象は次の相違を含む。
    サブモジュール分割で`pkg.__init__`が空で`pkg.sub`に実装がある構成、
    JS/TS等の非import型ライブラリのメソッド名、他ライブラリと同名APIの誤帰属
- 規範文書に外部URLを追加または改訂する場合は計画確定前に`uvx pyfltr run-for-agent --commands=lychee`または`lychee <path>`で対象ファイルの外部リンクを実機検証する。404や持続的リダイレクト検知時はtext記述への降格・節名参照への変更などで扱う
- 反映結果が利用者に届く観測経路（表示・ログ・出力フォーマット等）の実装を反映側と別に確認する。
  診断メッセージ生成側の変更なら表示経路（CLI render・logger・JSONL出力フォーマット等）の該当分岐実装を`Read`で確認し、追加した診断が表示されうるかを事前検証する
- 新規テストfixture導入を含む計画では、fixtureが構築する隔離環境（対象ディレクトリ・設定ファイル・サンプルファイル）を実際に構築して当該テストの期待結果（rcや呼び出し回数など）を1回実測する。
  fixtureが対象ツール群を実際に呼び出すかを対象ツールの有効化ゲート・対象パターン・除外設定と照合して確認する。実測なしでfixture構成を計画本文へ確定しない
