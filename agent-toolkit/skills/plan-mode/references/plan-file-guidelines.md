# 計画ファイル作成ガイドライン

## 全セクションを通した記述例

````markdown
# ファイルアップロード上限を10MBから50MBへ引き上げ

## 変更履歴

- 初版
- codexレビュー指摘反映（1回目）
  - 却下: 新旧の上限値を切り替えるfeature flagを用意すべき — ロールバックは設定値の差し替えで十分対応でき、feature flagを追加する運用複雑度の増加に見合う利点がないため

## 背景

高解像度画像をアップロードできないという要望が継続して提示されている。
ストレージと転送帯域の増強が完了し運用コストの懸念が解消されたため、上限値を引き上げて要望に応える。

## 対応方針

### ユーザー合意済み事項

- 上限値は50MBに変更
- 超過時のエラーメッセージ文言は既存のものを流用
- 経路上の各層に個別設定されているサイズ制限やタイムアウトの整合確認を検証範囲に含める

### エージェント判断

- 上限値の定義を一元化してSSOTを確立
- 既存の超過判定とエラー通知の挙動は維持し、参照値のみを差し替え
- `docs/architecture/limits.md` の「設定値管理」節に3層整合ルールを追記する

### 却下した代替案

- ファイル種別ごとに上限を変える案 — 運用が複雑化する割に現時点で必要性が明確でないため
- 上限を撤廃する案 — 想定外のリソース消費や不正利用のリスクが残るため

### 恒久化・リファクタリング内容

- 恒久化: 3層を整合させる運用方針を`docs/architecture/limits.md`へ追記する
  - 追記文面は`### エージェント判断`に記載
- リファクタリング: クライアント側の独立保持を解消し設定API経由のSSOTへ統一する
  - 実装は`## 変更内容`に記載

## 調査結果

- `server/config.py:34`: サーバー側で上限値を10MBとして定義する
- `client/limits.ts:12`: クライアント側で同じ値を独立保持し、サーバーと整合させる仕組みは無い
- `infra/nginx.conf:58`: リバースプロキシの`client_max_body_size`が20MB
- `tests/upload_test.py:40-78`: 上限境界のテストがあり、いずれも10MB前提

## 変更内容

### `server/config.py`・`infra/nginx.conf`

上限値定義を50MBへ更新する。

### `client/limits.ts`

独立保持を解消し、設定API経由でサーバーから取得する形へ統一する。

```diff
-export const UPLOAD_SIZE_LIMIT_MB = 10;
+import { fetchUploadLimit } from "./api/config";
+export const UPLOAD_SIZE_LIMIT_MB = await fetchUploadLimit();
```

### `tests/upload_test.py`

新上限基準で改訂する。
同値分割と境界値分析でケースを抽出する（49.9MB成功・50MB成功・50.1MB失敗・0MB失敗）。

## 実行方法

- 以下のスキルを呼び出す
  - `agent-toolkit:plan-impl`
  - `agent-toolkit:coding-standards`
- 計画に従い実装する
- 以下の検証を実施する
  - `uvx pyfltr run-for-agent tests/upload_test.py`
  - 追加したテストが修正前は失敗し修正後は成功することを確認する
- `agent-toolkit:commit`スキルを呼び出す
- コミットする
  - 件名案: `feat(upload): ファイルサイズ上限を50MBへ引き上げる`
- `agent-toolkit:careful-review`スキルを呼び出す

## 進捗ログ

## 計画ファイル（本ファイル）のパス

`~/.claude/plans/upload-limit-increase-concurrent-hickey.md`
````

## セクション構成と記述要件

### 判断材料セクション全体に共通する禁止事項（`# タイトル`・`## 変更履歴`・`## 背景`・`## 対応方針`配下のすべて）

- プログラムレベルの実装詳細（関数名・変数名・行番号・コード片・設定キー等の識別子レベル）を書かず`## 調査結果`と`## 変更内容`にのみ記述する（ファイル名・パス単独は禁止対象外）
- 実装方法・対象ファイル列挙・ファイル別作業項目は`## 変更内容`または`## 実行方法`へ集約する
- 他セクションと記述を重複させない。例外: ドキュメント修正の変更後文面は実装詳細禁止の対象外

### タイトル（`# タイトル`）: 計画の主題を1行で示す

### 変更履歴（`## 変更履歴`）: 更新単位ごとに見出しを設け、箇条書きの各項目に指摘と修正方針を記述する

- 1項目1〜2行、`<指摘要約> — <修正方針>`の形式で書く。却下指摘は`却下: <指摘要約> — <却下理由>`の形式で書く
- エントリは時系列順（古い→新しい）に並べ、新しい更新は末尾へ追記する
- 見出し表記: `- 初版` / `- codexレビュー指摘反映（N回目）` / `- ユーザー判断反映（N回目）`
- 全面改訂（計画の根幹が覆る規模の方針転換）時は変更履歴を「初版」へ戻し旧履歴を残さない。codexレビューも初回扱いで再実施する

### 背景（`## 背景`）: なぜこの変更をするか（人間向け）

情報源はユーザーの発話・対話で確認した動機・経緯・期待する成果、またはユーザーが提示した文面の原文
（レビュー指摘・エラー出力・スタックトレース・要件抜粋・ログ等）に限定する。提示素材は原文を転記し、長文素材は`text`コードブロックで包む。
背景を埋めるためだけの追加質問はしない。バグ対応時の調査結果表は本制約の対象外。

バグ対応時は`## 背景`節末尾に以下の表を必ず含める（複数バグ対応時はバグごとに表を作成し並べて配置する）。

| 項目 | 内容 |
| --- | --- |
| 根本原因 | 起点まで追跡した一次原因を1〜2文で記述 |
| 修正方針 | 直接の修正内容を1〜2文で記述（再発防止策と対で検討する） |
| 再発防止策 | 自動で機能する仕組み（テスト・CI・型制約など）を優先して記述 |
| 類似見直し結果 | 根本原因に基づき類似パターンの他箇所を確認した範囲と結果 |
| 知見の記録 | 恒久化先（CLAUDE.md・ルール・スキル等）と転記内容 |

「類似見直し結果」欄は計画提出前に調査を完了し確定した結果を記入する。表内はプロジェクト内の概念名で書き、ファイルパス・関数名は`## 調査結果`・`## 変更内容`へ委ねる。

### 対応方針（`## 対応方針`）: 変更内容・影響範囲・段取り・リスク扱い（人間向け）

- 抽象的な宣言文（「機能的挙動は変えず」「最小差分で」など）を書く場合、変更内容と矛盾しないか必ず確認する

### ユーザー合意済み事項（`### ユーザー合意済み事項`）: ユーザーが発話または確認で合意した事項のみを記述する

エージェントの推測・既定判断・CLAUDE.md等の指示内容は`### エージェント判断`に分離する。実行方法に属する情報は含めない。

### エージェント判断（`### エージェント判断`）: ユーザー指示外でエージェントが既定した事項。修正・見直しの候補

撤廃・削除を伴う変更では対応範囲（依存先含むか）を必ず明示する。
規範記述ファイル群の改訂計画では改訂後の試算物理行数を`agent-toolkit:agent-standards`「文書サイズ上限」節に従いファイルごとに記載する。

### 却下した代替案（`### 却下した代替案`）: 選択肢になり得たが採らなかった案を残す

該当案が無い場合は「なし」と明記する。

### 恒久化・リファクタリング内容（`### 恒久化・リファクタリング内容`）: 工程4・5で実施すると決めた内容を記載する

対象が無い場合も「なし」と明記する。バグ対応時に`## 背景`の調査結果表で記載済みの場合は「背景表に記載」と参照する。

### 調査結果（`## 調査結果`）: 既存コードの挙動・依存・制約（codex向け照合材料）。後続工程で参照できる粒度で記述する

### 変更内容（`## 変更内容`）: 実装するファイルと変更方針を実装者が再調査せず着手できる粒度で記述する。コード片を含めてよい

- 公開関数の追加・変更を含む計画では、対応する単体テストファイルへの追加・修正項目を含める
  - テスト設計は計画段階で同値分割・境界値分析（JSTQB準拠）により具体化し、独立観点のみ新規クラスへ分離する（既存クラス守備範囲は当該クラス改修で吸収）
- `### エージェント判断`の方針を対象ファイルを伴わずに言い換えて再列挙しない

### 実行方法（`## 実行方法`）: 実装からレビューまでを実行順の手順として箇条書きで記述する

- 冒頭に呼び出すスキルを列挙し、先頭の`agent-toolkit:plan-impl`を必須として作業種別に応じたスキルを続ける
  - 対象リポジトリの`AGENTS.md`・`CLAUDE.md`・`.claude/rules/`配下から検証・レビュー・コミット・ドキュメント確認の順序および全体／個別実行の指定を`grep -rn`で抽出し`## 実行方法`の手順順序へ反映する
- 実装の実施を1ステップとして置く
- 検証の実施を1ステップとして置き、計画固有の観点と自動チェック手順の両方をサブバレットへ含める
  - 記述対象は実装後の成果物に対する検証コマンドのみとし、計画ファイル本文への機械チェックは含めない
- `agent-toolkit:commit`スキルの呼び出しを1ステップとして置く
- コミットの実施を1ステップとして置き、コミット件名案をサブバレットへ記す
  - 件名案は`agent-toolkit:commit`スキルに従い決定する。
    固定文言体系の有無は対象リポジトリの`CLAUDE.md`・`AGENTS.md`・`.claude/rules/`配下と直近コミット履歴で確認する
- レビューの実施を1ステップとして置く。既定は`agent-toolkit:careful-review`の呼び出し
  - プロジェクト固有のレビュースキル・エージェントを併用する場合は併記する
  - ユーザーが明示的にレビュー不要と指示した場合のみ「レビューは実施しない」と記述する

### 進捗ログ（`## 進捗ログ`）: 区切りの節目ごとに`### <ラベル>`見出しと箇条書きで追記する。初版作成時は空でよい

- 記述粒度は判断・成果・残課題・失敗試行を1行ずつ程度にとどめ、コード差分や長文の調査結果は転記しない
- 確定形が計画内容と乖離した場合（レビュー指摘対応によるdiffコードブロック内文面案の意味的変更を含む）は計画ファイル本文も最終確定形へ更新する
- `### <ラベル>`見出しは時系列ラベルに限定する（`### 着手`・`### 検証完了`・`### コミット`・`### レビュー反映`等）
  - 段階分割見出し（`### 第N段階`・`### 後続範囲`・`### 未着手範囲`・`### 計画ファイル更新メモ`・`### 次回対応`・`### 修正候補`・`### TODO`等）は規範違反として作成しない。これらは未着手範囲の繰越宣言と本文修正の実装段階以降への持ち越し経路を成立させるため。

### 計画ファイル（本ファイル）のパス（`## 計画ファイル（本ファイル）のパス`）: 末尾に絶対パスを記述する

## 計画ファイル全体の遵守事項

- 計画ファイル自立性を満たし、計画時の判断・調査・ユーザー確認はすべて計画ファイル内で確定する（実装段階や計画提出後への先送り表現は本体に書かない。例外は`## 実行方法`の検証ステップ）
- `## 調査結果`・`## 変更内容`の記述に入る前に、変更対象ファイルを原則全体通読する
  - 横断的な置換・参照除去・キーワード削除では対象キーワードを`grep -rn`等で全件列挙し、副作用マッチ候補（境界条件・複数行パターン等）も含め`## 調査結果`に列挙してから置換パターンを確定する
- 同一ファイル内へ新規テストクラス・新規節を追加する計画では、追加先ファイル全体の既存単位の見出し（クラス定義・節見出し等）を一覧化し、各単位の名称と守備範囲を計画ファイルへ明記する
- 計画ファイル記載の既存パスは計画提出前に実在確認する。新規作成予定のパスは`## 変更内容`で明示し親ディレクトリの実在を確認する
- `## 対応方針`配下にコードブロックで提示した文面案は`## 変更内容`より優先し、実装時の転記で独立完成文として成立するか確認する
- ユーザー判断で計画方針を転換した場合、各セクションを連動更新し、散在する旧前提値は`grep`等で全件列挙して漏れなく更新する
- 検証結果が想定と異なる場合は「環境差異」等と解釈せず進捗ログに乖離を記録して計画の前提を見直しユーザーへ報告する
- 検証は自動実行を最優先し、手動検証は自動化困難な場合に限定して理由を明記する
- コミット粒度は`agent-toolkit:commit`スキルの規定に従う
- 計画範囲外ファイルでの違反検知は自己判断で書き換えず修正方針をユーザーへ提示する（計画ファイル内の不整合は計画提出前に解消）
- 計画ファイル本文で対象位置・対象量を示す際、改訂で変動する絶対数値（行番号・行範囲・行数・件数）の直書きを避け、節名・小見出し・冒頭文引用などの相対参照で特定する（`## 調査結果`の確定値は対象外）
  - 新節の配置先を相対参照（「〜近傍」「〜の後」等）で示す場合、参照先節と新節が同じ親節（H2）配下に位置するかを`grep -n '^## ' <対象ファイル>`等で確認する
- 計画ファイル本文のdiffコードブロック内側に3バッククォートを含む場合は外側を4バッククォートで包む
- 計画ファイル本文は作文段階で`agent-toolkit:writing-standards`「Markdown記述スタイル」節の規定を事前適用する
- 初版Write前に本文と`## 変更内容`配下のdiff +側文面を対象リポジトリ直下の一時ファイルへ出力し機械チェックする
  - `uvx pyfltr run-for-agent --no-fix <一時ファイル>`と`agent-toolkit:writing-standards`配下`scripts/check_line_width.py <一時ファイル>`を実行する。対象リポジトリ配下に置かないと`textlint`・`markdownlint`設定が継承されず`skipped`扱いとなる
