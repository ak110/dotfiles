# 計画ファイル作成ガイドライン

## 全セクションを通した記述例

サンプル例は[sample.md](sample.md)を参照する。

## セクション構成と記述要件

### 判断材料セクションの共通禁止事項（`# タイトル`・`## 変更履歴`・`## 背景`・`## 対応方針`配下）

- 実装詳細（関数名・変数名・行番号・コード片・設定キー等の識別子）は`## 調査結果`・`## 変更内容`のみに記述する。
  実装方法は`## 変更内容`・`## 実行方法`にのみ記述し、他セクションと重複させない
  （ファイル名・パス単独とドキュメント修正の変更後文面は禁止対象外）

### タイトル（`# タイトル`）

計画の主題を1行で示す。

### 変更履歴（`## 変更履歴`）

更新単位ごとに見出し（`- 初版`・`- codexレビュー指摘反映（N回目）`・`- サブエージェントレビュー指摘反映（N回目）`・`- ユーザー判断反映（N回目）`・`- 方針転換（N回目）`）を時系列順で並べる。
各項目は`<指摘要約>。<修正方針>`形式で1〜2行記述する。
却下は`却下: <指摘要約>。<却下理由>`、方針転換（自発的転換含む）は`<転換前→転換後>。<契機>。<理由>`の形式とする。
全面改訂（計画の目的・対応方針・成功条件の転換）時は変更履歴を「初版」へ戻し旧履歴を残さない
（工程7の各種レビューも初回扱い）。変更履歴節への記述義務は方針転換・全面改訂・却下の履歴保持に限定する。
通常の指摘反映見出しは`## 変更内容`本文をSSOTとして直接更新する。

### 背景（`## 背景`）

なぜこの変更をするか。`## 背景`配下は`### 計画メタ情報`（必須）・`### 経緯`・`### 現状`・`### 提示素材`・`### バグ調査結果`（いずれも該当なしなら省略可）のH3で分類する。背景を埋めるためだけの追加質問はしない。

### 計画メタ情報（`### 計画メタ情報`）

起動経路（`process-feedbacks経由`・`plan-and-add-feedback経由`・`その他（直接起動等）`のいずれか）と
対象リポジトリ（`~/dotfiles`等のルートパス）を箇条書きで記載する（必須）。

### 経緯（`### 経緯`）

動機・依頼内容・期待する成果の要約（人間向け）。情報源はユーザー発話・対話確認内容に限定する。

### 現状（`### 現状`）

現行の挙動・構成・問題点を記述する（エージェント調査由来の要約可。実装識別子の禁止は共通禁止事項に従う）。

### 提示素材（`### 提示素材`）

原文を転記する（書式統一以外の改変不可）。長文素材は`text`コードブロックで包む。
フィードバック処理由来の計画ではフィードバック本文を`atk fb show`出力のまま`text`ブロックへ転記する（書式統一もしない）。

### バグ調査結果（`### バグ調査結果`）

バグ対応表の配置先に用いる。記入項目は根本原因、直接原因の候補、修正方針、再発防止策、類似見直し結果、知見の記録の各項とする。
複数バグ対応時はバグごとに表を作成する。裏付けが取れない因果チェーンの段は`直接原因の候補`欄へ列挙し、断定形の`根本原因`欄には書かない。

### 対応方針（`## 対応方針`）

変更内容・影響範囲・段取り・リスク扱い（人間向け）。

### ユーザー合意済み事項（`### ユーザー合意済み事項`）

ユーザーが発話または確認で合意した事項のみを記述する。
エージェントの推測・既定判断・CLAUDE.md等の指示内容は`### エージェント判断`へ移動する。
フィードバック原文・レビュー指摘等の提示素材に含まれる反映先案・反映内容案・具体的な文面提案は、
ユーザー発話・確認で合意した事項に含めない。素材の提示は`## 背景`へ原文転記し、
採否・具体位置・具体文面はエージェント判断として`### エージェント判断`側へ記述する。
自律実行でユーザー発話が無い場合は「なし（自律実行のためユーザー合意なし）」と明記し、
起動プロンプト由来の制約・工程列挙は合意に含めず出所を明示して`### エージェント判断`へ記述する。

### エージェント判断（`### エージェント判断`）

ユーザー指示外でエージェントが既定した事項。撤廃・削除を伴う変更では対応範囲（依存先を含むか）を必ず明示する。
`plan-implementer`委譲は担当範囲を実装のみに限定する（検証・コミット・レビュー・pushは呼び出し元。
`agent-toolkit/rules/03-claude-code.md`「サブエージェントの活用」節の例外規定に従う）。
並列サブエージェントへ実装委譲する計画では、担当ファイル集合と参照テストファイル集合の対応関係を
マトリクス形式で列挙し、同一ファイルの複数エージェント割当が無いか点検して単一エージェントまたはメインへ再設計する。
検証・git commit・git push等の担当主体（既定はメイン側完結、個別サブエージェント担当時は
`git worktree`分離の根拠）を明記し、各委譲プロンプトへ上記対応関係を前提条件として埋め込む。
フィードバック採用計画で反映内容が原フィードバック主旨に限定される場合、次の3点を本節へ記述する。
原主旨と反映内容の対応関係。`agent-toolkit:process-feedbacks`「採用時の反映内容の縮小禁止」規定に
基づく原文どおり採用の根拠。範囲外に残る根本原因の扱い方針（却下代替案が併存する時は`### 却下した代替案`節にも併記）。

### 却下した代替案（`### 却下した代替案`）

選択肢になり得たが採らなかった案を残す（該当案が無い場合は「なし」と明記する）。

### 恒久化・リファクタリング内容（`### 恒久化・リファクタリング内容`）

工程4・5で決定した内容を記載する（対象が無い場合は「なし」と明記する）。
バグ対応時に`### バグ調査結果`表で記載済みの場合は「バグ調査結果表に記載」と参照する。
通常のplan mode工程4「恒久化検討」では、対応が必要な事項をすべて当該計画ファイルへ統合する
（残課題別フィードバック切り出しは禁止。`01-agent.md`のセッション分割禁止バレット参照）。
「別フィードバックとして投入」は`plan-and-add-feedback`スキル経由の運用に限る。
同スキルが計画完了時点で固定書式本文（`session-review`定義形式、末尾`text`コードブロック）を
`add-feedback`経由で投入する。`## 実行方法`へ`atk fb add`のシングルクォート形式で投入コマンドを明記する。

### 調査結果（`## 調査結果`）

既存コードの挙動・依存・制約（codex向け照合材料）を、後続工程で参照できる粒度で記述する。
`wc -l`実測値は`### 対象ファイルの現状`H3へ列挙する（記録手順は「6-1. 参照テンプレート読み込み」節）。

### 事実確認済み事項（`### 事実確認済み事項`）

調査結果の裏付けを1行ずつ列挙する（対象ファイル実在確認・現行記述箇所・grep/Read結果を含む）。適合済み判定時は根拠を転記する。

### 変更内容（`## 変更内容`）

実装するファイルと変更方針を実装者が再調査せず着手できる粒度で記述する（コード片可）。
冒頭に「対象ファイル一覧」H3を必須配置し、変更対象ファイル全件をチェックボックス形式（`- [ ]`）で列挙し、
続けて各ファイル変更方針をH3見出しで記述する。
各ファイル変更方針H3配下では、変更後の最終文面または差分（unified diff形式）を`text`コードブロックで埋め込み、
計画ファイル本文のみで変更を再現できる粒度で記述する。自動適用と参照案内の区別は見出しに明示する（外部参照のみの代用は不可）。
見込み行数の併記義務は`agent-toolkit:agent-standards`「文書サイズ上限」節の対象文書
（`.md`・`.md.tmpl`）に限定する（`wc -l`実測義務は全ファイルへ適用する）。
対象文書は`- [ ] <ファイルパス>（現行N行, 見込みM行）`形式（新設は`（新設, 見込みN行）`）で記載する。
上記以外のファイル（`.py`とそのテストファイル等）は`- [ ] <ファイルパス>`形式（行数記載なし）で列挙する。
`.md`・`.md.tmpl`で事前算出困難な場合は`（現行N行, 実装後未確定）`表記を許容する（`check_wc_projection.py`は同表記を受理）。
220行超過ファイル対象時の`#### 縮減対象`H4の限定用途化は「6-1. 参照テンプレート読み込み」節を参照する。
例外形式として同一パターン置換時の集約H3（`### 置換パターン: <旧> → <新>（対象: <sublist>）`）と`quality-sweep`配下計画の分担バッチ単位列挙を許容する。
集約H3の適用条件は同一の旧文言から新文言への置換が3ファイル以上に及ぶ場合とし、文脈依存の変更を伴うファイルは個別差分ブロックへ切り出す。
差分ラベル規定は[plan-file-diff-labels.md](plan-file-diff-labels.md)をSSOTとする
（同ファイルが仕様・fence内側配置・機械適用対象・記載範囲・単体テスト追加規定を集約する）。
220行到達済みファイル対象の変更は`[現行]`/`[置換後]`ペアもしくは`[追記]`ラベル（同一文面を複数箇所へ配置する場合は`[追記×N]`）で書く（乖離を防ぐ検証経路へ切り替えるため）。

複雑な既存モジュール（既存API関数の拡張・UIコンポーネントのProps拡張等）を対象とする場合、
詳細スケッチを埋め込む代わりに「目的・対象・設計要件・実装者がReadすべき参照コード」の各要素で記述してよい。
詳細スケッチは新規ファイルや純粋機械置換のように既存契約への依存が薄い場合に限る。

他ファイル文言との「完全一致」「同一表現」宣言では、原典パス・引用元節を明示し、原文`text`ブロック直後に追記文言案を配置する。

計画がスキル廃止・ファイル削除・識別子改名を含む場合、`#### 廃止・改名対象一覧`H4節を配置する。
配下にバッククォート囲みで削除・改名する識別子（スキル名・パス・関数名・変数名を含む）を1行ずつ列挙する。
機械可読形式は`` - `<identifier>` ``形式のリスト項目とする。
`check_deprecated_identifier_coverage.py`が本節を抽出し照合に用いる。

### 実行方法（`## 実行方法`）

実装からレビューまでを実行順に箇条書きで記述する。
冒頭にAgentツールで`agent-toolkit:plan-impl-executor`を起動する旨の記載を必須として、作業種別に応じて呼び出すスキルを列挙する。
列挙には配置先スキル（対象リポジトリの`AGENTS.md`・`CLAUDE.md`が編集対象ディレクトリに対して指定するスキル）も必ず含める。
対象リポジトリの`AGENTS.md`・`CLAUDE.md`・`.claude/rules/`配下から
検証・レビュー・コミット・ドキュメント確認の順序を抽出して反映する。
標準順序は実装→検証→`agent-toolkit:commit`の呼び出し→コミット→レビュー→pushとし、各過程を1ステップとして置く。
`agent-toolkit:careful-review`は`git push`より前に配置する
（レビュー指摘で修正が必要な場合に未レビューの変更を公開しないため）。
コミットの実施ステップには件名案をサブバレットへ記す。レビュー既定は`agent-toolkit:careful-review`の呼び出しとし、
ユーザーが明示的にレビュー不要と指示した場合のみ「レビューは実施しない」と記述する。
計画ファイルの変更対象に配置先スキル（`agent-toolkit-edit`等）が管理する配布物が含まれる場合、
当該配置先スキルの規定に従い`## 実行方法`へ版更新手順を明記する（bump粒度判定と実行コマンドを含む）。
規範文書（`AGENTS.md`・`CLAUDE.md`・`.claude/rules/`配下・`agent-toolkit/`配下）を対象ファイル一覧に含む計画がある。
この場合は`agent-toolkit:plan-impl-executor`起動プロンプトへ改訂許可を明記する。
許可対象は規範文書・スキル定義・サブエージェント定義とする。
典拠は`agent-toolkit/rules/03-claude-code.md`「サブエージェントの活用」節の例外規定とする。
UIテンプレートなどの画面変更を含む場合、`## 実行方法`を書く前にプロジェクトのAGENTS.mdで規定されている変更種別別の必須ステップを参照して必要なステップを洗い出す。対象例はコミット前のpyfltr全体実行、Playwright MCPによる動作確認、VRTベースライン更新とする。

### 進捗ログ（`## 進捗ログ`）

区切りの節目ごとに`### <ラベル>`見出しと箇条書きで追記する（初版作成時は空でよい）。
記述粒度は判断・成果・残課題・失敗試行を1行ずつ程度にとどめる。
`### <ラベル>`は時系列ラベル（`### 着手`・`### 検証完了`・`### コミット`・`### レビュー反映`等）または`### 発見`に限定する。
`### 発見`は計画前提と異なる挙動・想定外の副作用・設計判断に影響する観測事実を`<観測事象>。<計画への反映>`形式で記録する。
実装確定形が計画本文と乖離した場合は`### <時系列ラベル>`配下に`[乖離]`サブバレットで記録する。
書式は`plan-file-diff-labels.md`「進捗ログの乖離注記」節に従い、`## 変更内容`本文の同期更新可否を判断する。

### 計画ファイル（本ファイル）のパス（`## 計画ファイル（本ファイル）のパス`）

末尾に絶対パスを記述する。

- 標準保存先は`~/.claude/plans/<stem>.md`直下。プロジェクトリポジトリ配下への保存はしない

## 計画ファイル全体の遵守事項

- 計画ファイル自立性を満たし、判断・調査・ユーザー確認はすべて計画ファイル内で確定する（実装段階等への先送り表現は書かない）
- 計画方針を転換した場合は各セクションを連動更新し、`## 調査結果`・`## 変更内容`前に対象ファイルを全体通読する
- 計画ファイル記載の既存パスは計画提出前に実在確認し、新規作成予定パスは`## 変更内容`で明示し親ディレクトリの実在を確認する
  - 既存パスはプロジェクトルート相対の完全パスで記述する（短縮形は`check_line_ref.py`が実在確認できず反復を招く）
- 追記先として記述する`<file>:<H2/H3節名>`の組合せは、確定前に`grep -nE '^#+ ' <file>`等で実在を確認する
  - 節参照はカギ括弧で節名を囲む形式で書き`##`等の見出しマーカーを含めない（`check_line_ref.py`は見出しマーカー無しで抽出するため参照側にマーカーが混入すると必ず節名不在違反となる）
  - 複数候補ファイルの同名節は各ファイルで有無を確認し、bullet位置は`grep -n`照合で参照する（行番号直書きは避ける）
- 計画ファイル本文の頻出textlint違反予防策は`references/textlint-violations.md`「頻出違反パターン予防策」節に従う
- 検証は自動実行を最優先し、手動検証は自動化困難な場合に限り理由を明記する
  （検証結果が想定と異なる場合は「環境差異」等と解釈せず進捗ログに乖離を記録して前提を見直す）
- 計画本文を編集した契機ごとに本節末尾の書き込み後チェックを再実行する
- 計画ファイル本文で対象位置・対象量を示す際、改訂で変動する絶対数値（行番号・行範囲・行数・件数）の直書きを避け、
  節名・小見出し・冒頭文引用などの相対参照で特定する（対象は本文位置に加え他ファイル参照箇所の行番号・行範囲・行数も含む）
  - 機械検査対象は行番号の参照全般。`wc -l`実測値は`## 調査結果`配下で自由記載可（`<!-- line-ref-ok -->`で抑止、他節は違反）
- 計画本文の参照先は自リポジトリ内で完結させ、`/tmp/`・scratchpad・他セッションパスを書かず削除文言案・計算結果は本文へ記載する（6-1節の`wc -l`用途は対象外）
- `text`コードブロック化義務・機械検査コマンド一式は[plan-file-write-checks.md](plan-file-write-checks.md)参照

## 計画段階の調査・検証細則

各節の詳細手順は[investigation-rules.md](investigation-rules.md)に集約する。

## 6-1. 参照テンプレート読み込み

計画ファイルの初版Write直前に本ファイルをReadで読み込む（未実施時はブロックされるため簡易対応でも省略しない）。
対象ファイル一覧が`agent-toolkit:agent-standards`「文書サイズ上限」節の対象ファイルを含む計画を対象とする。
新規節・大幅追記は目安200行以内に収める設計を優先する（220行到達後の行数調整は実装時の後戻りコストが大きいため）。
対象ファイル全件は拡張子（`.md`・`.md.tmpl`・`.py`・`.json`等）を問わず現行の`wc -l`実測値を
`## 調査結果`配下の`### 対象ファイルの現状`H3へ列挙する（改訂後見込み行数は`.md`・`.md.tmpl`のみ併記する）。
220行超過ファイルはscratchpad配下の一時ファイルへ追記文面案と縮減対象文面を実際に貼り込み`wc -l`で実測する。
追記量と縮減量の相殺を`### エージェント判断`へ必須記載する（見た目行数と実物理行数の乖離を計画段階で解消するため）。
算定規則は削除・統合・見出し挿入（前後空行含む）の実物理行数とし、単文内の文言短縮は寄与しない。
縮減対象の選定では規範述語（「〜しない」「〜とする」等）を含む文の削除・弱体化をせず、意味を保存した統合・冗長表現の圧縮で行数を確保する。
