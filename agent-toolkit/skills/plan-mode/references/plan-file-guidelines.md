# 計画ファイル作成ガイドライン

## 全セクションを通した記述例

サンプル例は[sample.md](sample.md)を参照する。

## セクション構成と記述要件

### 判断材料セクションの共通禁止事項（`# タイトル`・`## 変更履歴`・`## 背景`・`## 対応方針`配下）

- プログラムレベルの実装詳細（関数名・変数名・行番号・コード片・設定キー等の識別子）は
  `## 調査結果`と`## 変更内容`にのみ記述する（ファイル名・パス単独は禁止対象外）
- 実装方法・対象ファイル列挙・ファイル別作業項目は`## 変更内容`または`## 実行方法`へ集約する
- 他セクションと記述を重複させない（ドキュメント修正の変更後文面は実装詳細禁止の対象外）

### タイトル（`# タイトル`）

計画の主題を1行で示す。

### 変更履歴（`## 変更履歴`）

更新単位ごとに見出し
（`- 初版`・`- codexレビュー指摘反映（N回目）`・`- ユーザー判断反映（N回目）`・`- 方針転換（N回目）`）を
時系列順で並べる。
各項目は`<指摘要約> — <修正方針>`形式で1〜2行記述する。
却下は`却下: <指摘要約> — <却下理由>`、
方針転換は`<転換前→転換後> — <契機となった観測事象> — <判断理由>`の形式とする。
`方針転換（N回目）`は実装中・検証中の自発的な方針転換を記録する目的で用いる。
全面改訂（計画の根幹が覆る規模の方針転換）時は変更履歴を「初版」へ戻し旧履歴を残さない
（codexレビューも初回扱いで再実施）。

### 背景（`## 背景`）

なぜこの変更をするか（人間向け）。
情報源はユーザー発話・対話で確認した動機・経緯・期待する成果、またはユーザー提示文面の原文に限定する。
提示素材本文は転記し、文意を保存する書式調整（引用記号「」をbacktickへ変える等の表記統一）は許容する
（語の置換・要約・順序変更などの本質的改変は禁止）。
長文素材は`text`コードブロックで包む。背景を埋めるためだけの追加質問はしない。

バグ対応時は本節末尾に表を必ず含め、
根本原因・修正方針・再発防止策・類似見直し結果・知見の記録の各項目を記入する
（複数バグ対応時はバグごとに表を作成）。

### 対応方針（`## 対応方針`）

変更内容・影響範囲・段取り・リスク扱い（人間向け）。

### ユーザー合意済み事項（`### ユーザー合意済み事項`）

ユーザーが発話または確認で合意した事項のみを記述する。
エージェントの推測・既定判断・CLAUDE.md等の指示内容は`### エージェント判断`に分離する。

### エージェント判断（`### エージェント判断`）

ユーザー指示外でエージェントが既定した事項。
撤廃・削除を伴う変更では対応範囲（依存先含むか）を必ず明示する。
並列サブエージェントへ実装委譲する計画では、
各サブエージェントの担当ファイル集合と、
当該担当ファイルが参照する既存テストファイル集合の対応関係を本節へ列挙する。
並列サブエージェント計画では、検証・git commit・git push等の担当主体
（既定はメイン側完結、個別サブエージェントが担当する場合は`git worktree`による作業ツリー分離の根拠）を
本節へ明記する。
各タスク委譲プロンプトには上記対応関係を前提条件として埋め込み、
独立コンテキストで起動するサブエージェントが他タスクの影響を考慮できる状態にする。

### 却下した代替案（`### 却下した代替案`）

選択肢になり得たが採らなかった案を残す。
該当案が無い場合は「なし」と明記する。

### 恒久化・リファクタリング内容（`### 恒久化・リファクタリング内容`）

工程4・5で決定した内容を記載する。
対象が無い場合も「なし」と明記する。
バグ対応時に`## 背景`の調査結果表で記載済みの場合は「背景表に記載」と参照する。

### 調査結果（`## 調査結果`）

既存コードの挙動・依存・制約（codex向け照合材料）を、後続工程で参照できる粒度で記述する。
`agent-toolkit:agent-standards`「文書サイズ上限」節の対象ファイルを編集する計画では、
当該ファイルの現行行数（`wc -l`実測値）を本節へ記録する。

### 変更内容（`## 変更内容`）

実装するファイルと変更方針を、実装者が再調査せず着手できる粒度で記述する（コード片を含めてよい）。
冒頭に「対象ファイル一覧」H3を必須配置し、
変更対象ファイル全件をチェックボックス形式（`- [ ]`）で列挙する。
続けて各ファイルの変更方針をH3見出しで記述する。
各ファイルの変更方針H3配下では、
変更後の最終文面または差分（unified diff形式）を`text`コードブロックで埋め込み、
実装者が計画ファイル本文のみで変更を再現できる粒度で記述する。
scratchpad配下や一時ファイルへの外部参照のみで代用する形式は採用しない。
自動適用される差分（`patch`コマンドで直接適用する用途）と
参照案内目的の差分（読み手が変更箇所を理解する用途）の区別は見出しか前置きに明示する。
対象ファイル一覧のチェックボックス項目と各ファイル変更方針のH3見出しは1対1で対応させ、
複数ファイルを1つのH3にまとめる形式は採用しない。
自動bumpスクリプトで連動更新される`plugin.json`・`marketplace.json`等を含めて、
各ファイルを個別H3で記述する。
公開関数の追加・変更を含む計画では、
対応する単体テストファイルへの追加・修正項目を含め、
同値分割・境界値分析および正規表現の選択肢（`|`）の各分岐でケースを抽出する。
`agent-toolkit:agent-standards`「文書サイズ上限」節の対象ファイルを編集する計画では、
実装段階での`wc -l`確認手順を本節へ記述する。
改訂後の見込みが215行以上に達する場合は、`### エージェント判断`へ試算行数を併記する。

### 実行方法（`## 実行方法`）

実装からレビューまでを実行順に箇条書きで記述する。
冒頭に呼び出すスキルを列挙し、先頭の`agent-toolkit:plan-impl`を必須として作業種別に応じたスキルを続ける。
列挙には配置先スキル
（対象リポジトリの`AGENTS.md`・`CLAUDE.md`が編集対象ディレクトリに対して指定するスキル）も必ず含める。
対象リポジトリの`AGENTS.md`・`CLAUDE.md`・`.claude/rules/`配下から
検証・レビュー・コミット・ドキュメント確認の順序を抽出して反映する。
実装の実施・検証の実施・`agent-toolkit:commit`の呼び出し・コミットの実施・レビューの実施を
各1ステップとして置く。
コミットの実施ステップには件名案をサブバレットへ記す。
レビュー既定は`agent-toolkit:careful-review`の呼び出しとし、
ユーザーが明示的にレビュー不要と指示した場合のみ「レビューは実施しない」と記述する。
標準順序は実装→検証→コミット→レビュー→pushとし、
`agent-toolkit:careful-review`は`git push`より前に配置する
（レビュー指摘で修正が必要な場合に未レビューの変更を公開しないため）。

### 進捗ログ（`## 進捗ログ`）

区切りの節目ごとに`### <ラベル>`見出しと箇条書きで追記する（初版作成時は空でよい）。
記述粒度は判断・成果・残課題・失敗試行を1行ずつ程度にとどめる。
確定形が計画内容と乖離した場合
（`scratchpad`構築結果がトリム内訳・抽出方針と乖離する場合を含む）は
計画ファイル本文も最終確定形へ更新する。
`### <ラベル>`見出しは時系列ラベル
（`### 着手`・`### 検証完了`・`### コミット`・`### レビュー反映`等）または`### 発見`に限定する。
`### 発見`は計画前提と異なる挙動・想定外の副作用・設計判断に影響する観測事実を
`<観測事象> — <計画への反映>`形式で記録する。
本文の確定形に影響する場合は本節前項に従い当該本文も同時更新する。

### 計画ファイル（本ファイル）のパス（`## 計画ファイル（本ファイル）のパス`）

末尾に絶対パスを記述する。

## 計画ファイル全体の遵守事項

- 計画ファイル自立性を満たし、計画時の判断・調査・ユーザー確認はすべて計画ファイル内で確定する
  （実装段階や計画提出後への先送り表現は本体に書かない）
- ユーザー判断で計画方針を転換した場合は各セクションを連動更新し、
  `## 調査結果`・`## 変更内容`の記述に入る前に変更対象ファイルを原則全体通読する
- 計画ファイル記載の既存パスは計画提出前に実在確認し、
  新規作成予定パスは`## 変更内容`で明示し親ディレクトリの実在を確認する
- 検証は自動実行を最優先し、手動検証は自動化困難な場合に限定して理由を明記する
  （検証結果が想定と異なる場合は「環境差異」等と解釈せず進捗ログに乖離を記録して前提を見直す）
- 計画ファイル本文で対象位置・対象量を示す際、
  改訂で変動する絶対数値（行番号・行範囲・行数・件数）の直書きを避け、
  節名・小見出し・冒頭文引用などの相対参照で特定する
  - 対象は計画ファイル自体の本文位置に加え、他ファイル参照箇所の行番号・行範囲・行数も含む
  - 該当例は改訂対象の節範囲・引用元位置・改訂による行数変化量、
    `## 変更内容`配下のdiff前置きで対象行を示す`Lxx-yy`・`Lxx`形式の行番号参照など
  - 代替表現として該当節名・該当バレット冒頭文・対象識別子
    （クラス名・関数名・メソッド名・`if __name__ == "__main__":`配下など）で位置を特定する
  - `## 調査結果`の確定値は対象外
  - 計画作成中に集計する件数表記（「既存違反N件」「指摘N件」「修正N件」等）も対象に含み、
    内訳列挙が同一文書内にある場合は
    「対象ファイル一覧の全件」「`## 変更内容`配下に列挙したファイル全件」などの相対参照で記述する
- `## 調査結果`・`## 変更内容`配下の追加文面案・反映文面案など
  他文書本文を転記する長文段落（120字を超え得る段落）は`text`コードブロックで包んで記述する。
  textlintのsentence-length違反を初版`Write`段階で回避する
  - `## 調査結果`配下の既存docstring要約・既存関数の説明文・既存規範文の引用などの長文段落も対象に含める
    （行数・件数等の単純な箇条書きの確定値は対象外）
- 新規規範または新規機械検査の導入計画では計画ファイル本文自身が当該新規対象に該当しないかを
  初版`Write`前に確認する。
  該当する場合は当該文面を対象外形式（コードフェンス内転記・抽象表現での参照など）へ事前変換してから
  初版`Write`に進む
- 計画ファイル本文の地の文（コードブロック外の全節本文）は`Write`（初版・改版とも）の前にscratchpad配下の一時ファイルへ出力する
  - 出力対象から`## 背景`配下の`text`コードブロック内容のみを除外する
    - 除外理由は、ユーザー提示素材の原文に口語表現が含まれる場合に事前検査が失敗するため
  - 出力後に`uvx pyfltr run-for-agent --commands=textlint,markdownlint,typos,colloquial <一時ファイル>`で違反を検出する
  - 対象節は`## 変更内容`・`## 調査結果`・`### エージェント判断`等を含む計画ファイル全節とし、節ごとの限定はしない
  - 検出した違反は本文反映前に解消する
  - 工程7の機械チェックを事前検証の代替としてはならない
  - 未実施のまま`Write`を試みた場合はPreToolUseフックが内容ハッシュ照合でブロックする
    - PostToolUseは事前検査の成功後にscratchpadファイル内容の全文SHA256と除去後SHA256をセッション状態へ記録する
    - PreToolUseは`Write`対象の全文SHA256と除去後SHA256を算出し、いずれかが登録済みなら通過する

## 6-1. 参照テンプレート読み込み

計画ファイルの初版Write直前に本ファイルをReadで読み込む
（未実施時はPreToolUseフックがブロックするため簡易対応でも省略しない）。
`## 変更内容`の対象ファイル一覧が`agent-toolkit:agent-standards`「文書サイズ上限」節の対象ファイルを含む計画では、
次のいずれかに該当する場合に改訂後の最終形を一時ファイルへ出力する。
出力後に`wc -l`で実測し`### エージェント判断`へ反映する。

- 現行行数が200行以上
- 追加見込みが現行行数の20%以上
- 対象ファイル全節を跨ぐ改訂

いずれにも該当しない場合は試算行数を`### エージェント判断`へ概算値で記載する（実測は不要）。

## 6-2. 計画ファイル本体の作成

テンプレートに従い所定パスへ計画ファイルを作成し、
工程2〜5の合意事項・解釈・恒久化文面・周辺対応をテンプレート規定の各セクションへ転記して計画ファイル自立性を満たす。
