# 計画ファイル作成ガイドライン

## 全セクションを通した記述例

サンプル例は[sample.md](sample.md)を参照する。

## セクション構成と記述要件

### 判断材料セクションの共通禁止事項（`# タイトル`・`## 変更履歴`・`## 背景`・`## 対応方針`配下）

- 実装詳細（関数名・変数名・行番号・コード片・設定キー等の識別子）は`## 調査結果`・`## 変更内容`のみに記述する。
  実装方法は`## 変更内容`・`## 実行方法`にのみ記述し、他セクションと重複させない
  （ファイル名・パス単独とドキュメント修正の変更後文面は禁止対象外）

### タイトル（`# タイトル`）

計画の主題を1行で示す。

### 変更履歴（`## 変更履歴`）

更新単位ごとに見出し（`- 初版`・`- codexレビュー指摘反映（N回目）`・
`- サブエージェントレビュー指摘反映（N回目）`・`- ユーザー判断反映（N回目）`・`- 方針転換（N回目）`）を時系列順で並べる。
各項目は`<指摘要約>。<修正方針>`形式で1〜2行記述する。
却下は`却下: <指摘要約>。<却下理由>`、方針転換（自発的転換含む）は`<転換前→転換後>。<契機>。<理由>`の形式とする。
全面改訂（計画の根幹（目的・対応方針・成功条件）の転換を伴う方針転換）時は変更履歴を「初版」へ戻し旧履歴を残さない
（工程7の各種レビューも初回扱い）。
変更履歴節への記述義務は方針転換・全面改訂・却下の履歴保持に限定する。
通常の指摘反映（`- codexレビュー指摘反映（N回目）`・`- サブエージェントレビュー指摘反映（N回目）`・
`- ユーザー判断反映（N回目）`の各見出し）は`## 変更内容`本文をSSOTとして直接更新する。
変更履歴節への同時記録は必須としないが、エージェントが経緯を残す判断をした場合は該当見出しを追加してよい。
対応照合の機械検査は「変更内容」節末尾の記載に従う。

### 背景（`## 背景`）

なぜこの変更をするか（人間向け）。
情報源はユーザー発話・対話で確認した動機・経緯・期待する成果、またはユーザー提示文面の原文に限定する。
提示素材本文は転記し、文意を保存する書式調整（引用記号「」をbacktickへ変える等の表記統一）は許容する
（語の置換・要約・順序変更などの本質的改変は禁止）。
長文素材は`text`コードブロックで包む。背景を埋めるためだけの追加質問はしない。

バグ対応時は本節末尾に表を必ず含め、根本原因・修正方針・再発防止策・類似見直し結果・
知見の記録の各項目を記入する（複数バグ対応時はバグごとに表を作成）。

### 対応方針（`## 対応方針`）

変更内容・影響範囲・段取り・リスク扱い（人間向け）。

### ユーザー合意済み事項（`### ユーザー合意済み事項`）

ユーザーが発話または確認で合意した事項のみを記述する。
エージェントの推測・既定判断・CLAUDE.md等の指示内容は`### エージェント判断`に分離する。
フィードバック原文・レビュー指摘等の提示素材に含まれる反映先案・反映内容案・具体的な文面提案は、
ユーザー発話・確認で合意した事項に含めない。素材の提示は`## 背景`へ原文転記し、
採否・具体位置・具体文面はエージェント判断として`### エージェント判断`側へ記述する。

### エージェント判断（`### エージェント判断`）

ユーザー指示外でエージェントが既定した事項。撤廃・削除を伴う変更では対応範囲（依存先を含むか）を必ず明示する。
`plan-implementer`委譲の記述では担当範囲を実装のみに限定する（検証・コミット・レビュー・pushは呼び出し元の担当）。
`agent-toolkit/rules/02-claude-code.md`「サブエージェントの活用」節の例外規定に従う。
並列サブエージェントへ実装委譲する計画では、各サブエージェントの担当ファイル集合と
当該担当ファイルが参照する既存テストファイル集合の対応関係を、担当サブエージェント別のマトリクス形式で本節へ列挙する。
列挙後、同一ファイルが複数サブエージェントへ割り当てられていないかを計画確定前に点検し、
1ファイルが複数エージェントへ割り当てられる場合は当該ファイル全体を単一エージェントまたはメインへ集約する再設計を完了する。
並列サブエージェント計画では、検証・git commit・git push等の担当主体
（既定はメイン側完結、個別サブエージェントが担当する場合は`git worktree`分離の根拠）を本節へ明記する。
各タスク委譲プロンプトには上記対応関係を前提条件として埋め込み、独立コンテキストで
起動するサブエージェントが他タスクの影響を考慮できる状態にする。

### 却下した代替案（`### 却下した代替案`）

選択肢になり得たが採らなかった案を残す（該当案が無い場合は「なし」と明記する）。

### 恒久化・リファクタリング内容（`### 恒久化・リファクタリング内容`）

工程4・5で決定した内容を記載する（対象が無い場合は「なし」と明記する）。
バグ対応時に`## 背景`の調査結果表で記載済みの場合は「背景表に記載」と参照する。

### 調査結果（`## 調査結果`）

既存コードの挙動・依存・制約（codex向け照合材料）を、後続工程で参照できる粒度で記述する。
`wc -l`実測値は「対象ファイルの現状」H3節に列挙する（記録手順は「6-1. 参照テンプレート読み込み」節）。

### 変更内容（`## 変更内容`）

実装するファイルと変更方針を実装者が再調査せず着手できる粒度で記述する（コード片可）。
冒頭に「対象ファイル一覧」H3を必須配置し、変更対象ファイル全件をチェックボックス形式（`- [ ]`）で列挙し、
続けて各ファイル変更方針をH3見出しで記述する。
各ファイル変更方針H3配下では、変更後の最終文面または差分（unified diff形式）を`text`コードブロックで埋め込み、
計画ファイル本文のみで変更を再現できる粒度で記述する。
自動適用と参照案内の区別は見出しに明示する（外部参照のみの代用は不可）。
対象ファイル一覧のチェックボックス項目と各ファイル変更方針のH3見出しは1対1で対応させる。
チェックボックス項目は`- [ ] <ファイルパス>（現行N行, 見込みM行）`形式で見込み行数を併記する。
対象は`agent-toolkit:agent-standards`「文書サイズ上限」節の対象ファイル・新設ファイル（新設は`（新設, 見込みN行）`）とする。
例外形式として同一パターン置換時の集約H3（`### 置換パターン: <旧> → <新>（対象: <sublist>）`）と、
`quality-sweep`配下計画の分担バッチ単位列挙を許容する。
公開関数の追加・変更を含む計画では対応する単体テストファイルへの追加・修正項目を含め、
同値分割・境界値分析および正規表現の選択肢分岐（`|`）の各分岐でケースを抽出する。
コーディングエージェント向け文書の改訂タスクでは、置換・追記・削除のいずれかを明示し、
置換は対比形式、追記は最終文面、削除は現行文言と削除根拠をそれぞれ`text`コードブロックへ記載する。
`[現行]`/`[置換後]`対比ブロック形式は`check_wc_projection.py`の機械適用対象となり、
対応する`### <ファイルパス>`H3見出し配下に配置する。概念レベルの改訂方針記述だけでは委譲しない。
記載する現行・置換後・追記後の最終文面には既存の並列列挙要素・既存参照リンク・既存の代表例列挙を漏れなく含める。
`## 変更内容`本節をSSOTとして扱う。`## 変更履歴`側の各項目要約に含まれる対象ファイルパス・
節名アンカーは本節の対象ファイル一覧・各ファイルH3見出しと一致させる。
一致は`pretooluse.py`の`_check_plan_file_history_content_sync`が機械検査する。

既存節の間に新規節を挿入する差分を記載する場合、挿入位置直前・直後の既存節ヘッダを少なくとも1つずつdiff文脈として含める。
省略する場合は省略記号`...`で明示し挿入位置を1文で説明し、
JSON・YAML等の構造化ファイルでは挿入位置の親要素とその開閉ブラケットを文脈に含める形式を必須とする。

複雑な既存モジュール（既存API関数の拡張・UIコンポーネントのProps拡張等）を対象とする場合、
詳細スケッチを埋め込む代わりに「目的・対象・設計要件・実装者がReadすべき参照コード」の4要素で記述してよい。
詳細スケッチは新規ファイルや純粋機械置換のように既存契約への依存が薄い場合に限る。

他ファイルの規定文言と「完全一致」「同一表現」を宣言する場合は当該原典ファイルパス・引用元節を明示し、
原文を`text`コードブロックで併記し、追記文言案は原文コードブロックの直後に配置してレビュアーが1画面内で照合可能な構成にする。

### 実行方法（`## 実行方法`）

実装からレビューまでを実行順に箇条書きで記述する。
冒頭に呼び出すスキルを列挙し、先頭の`agent-toolkit:plan-impl`を必須として作業種別に応じたスキルを続ける。
列挙には配置先スキル（対象リポジトリの`AGENTS.md`・`CLAUDE.md`が編集対象ディレクトリに対して指定するスキル）も必ず含める。
対象リポジトリの`AGENTS.md`・`CLAUDE.md`・`.claude/rules/`配下から
検証・レビュー・コミット・ドキュメント確認の順序を抽出して反映する。
標準順序は実装→検証→`agent-toolkit:commit`の呼び出し→コミット→レビュー→pushとし、
各過程を1ステップとして置く。
`agent-toolkit:careful-review`は`git push`より前に配置する
（レビュー指摘で修正が必要な場合に未レビューの変更を公開しないため）。
コミットの実施ステップには件名案をサブバレットへ記す。
レビュー既定は`agent-toolkit:careful-review`の呼び出しとし、
ユーザーが明示的にレビュー不要と指示した場合のみ「レビューは実施しない」と記述する。
計画ファイルの変更対象に配置先スキル（`agent-toolkit-edit`等）が管理する配布物が含まれる場合、
当該配置先スキルの規定に従い`## 実行方法`へ版更新手順を明記する（bump粒度判定と実行コマンドを含む）。

### 進捗ログ（`## 進捗ログ`）

区切りの節目ごとに`### <ラベル>`見出しと箇条書きで追記する（初版作成時は空でよい）。
記述粒度は判断・成果・残課題・失敗試行を1行ずつ程度にとどめる。
確定形が計画内容と乖離した場合
（`scratchpad`構築結果がトリム内訳・抽出方針と乖離する場合を含む）は計画ファイル本文も最終確定形へ更新する。
`### <ラベル>`見出しは時系列ラベル
（`### 着手`・`### 検証完了`・`### コミット`・`### レビュー反映`等）または`### 発見`に限定する。
`### 発見`は計画前提と異なる挙動・想定外の副作用・設計判断に影響する観測事実を
`<観測事象>。<計画への反映>`形式で記録する。
本文の確定形に影響する場合は本節前項に従い当該本文も同時更新する。

### 計画ファイル（本ファイル）のパス（`## 計画ファイル（本ファイル）のパス`）

末尾に絶対パスを記述する。scratchpad上で本文を組み立てた後に正規パスへ複写する場合は本節配下のパス値も正規パスへ揃える。

## 計画ファイル全体の遵守事項

- 計画ファイル自立性を満たし、計画時の判断・調査・ユーザー確認はすべて計画ファイル内で確定する
  （実装段階や計画提出後への先送り表現は本体に書かない）
- ユーザー判断で計画方針を転換した場合は各セクションを連動更新し、
  `## 調査結果`・`## 変更内容`の記述に入る前に変更対象ファイルは全体通読する
- 計画ファイル記載の既存パスは計画提出前に実在確認し、新規作成予定パスは`## 変更内容`で明示し親ディレクトリの実在を確認する
- 計画ファイル本文で追記先として記述する`<file>:<H2/H3節名>`の組合せは、
  計画ファイル本文確定前に`grep -nE '^#+ ' <file>`などで実在を確認する
  - 複数候補ファイルで同名節が存在する場合は、各ファイルでの該当節の有無をそれぞれ確認する
  - bullet位置は該当bullet冒頭文言の`grep -n`照合で参照する（行番号直書きは避ける）
- 計画ファイル本文の頻出textlint違反パターン予防策は`agent-toolkit:writing-standards`配下`references/textlint-violations.md`の
  「頻出違反パターン予防策」節に従う
- 検証は自動実行を最優先し、手動検証は自動化困難な場合に限定して理由を明記する
  （検証結果が想定と異なる場合は「環境差異」等と解釈せず進捗ログに乖離を記録して前提を見直す）
- 計画本文を編集した契機ごと（初版・レビュー指摘反映・改訂委譲の検収後を含む）に本節の機械チェックを再実行し、
  検出違反は本文反映前に解消する（工程7の反映直後の再編集起因の違反もレビュー往復へ持ち込まない）
- 計画ファイル本文で対象位置・対象量を示す際、改訂で変動する絶対数値（行番号・行範囲・行数・件数）の直書きを避け、
  節名・小見出し・冒頭文引用などの相対参照で特定する。
  対象は計画ファイル自体の本文位置に加え、他ファイル参照箇所の行番号・行範囲・行数も含む
  - 代替表現は該当節名・該当バレット冒頭文・対象識別子で位置を特定する
  - 機械検査対象は行番号への参照全般で、`wc -l`実測値等の平文数値は`## 調査結果`配下で自由に記載できる
  - 行番号への参照を`## 調査結果`配下で確定値として書く場合は`<!-- line-ref-ok -->`コメントを
    該当行の行末に同一行で付与して抑止する。マーカーは`## 調査結果`H2配下でのみ有効とし、
    他節へ付与しても`check_line_ref.py`・`pretooluse.py`の検出パターン全般に対し違反として報告される
  - 計画作成中に集計する件数表記（「既存違反N件」「指摘N件」「修正N件」等）も対象に含み、内訳列挙が同一文書内にある場合は
    「対象ファイル一覧の全件」「`## 変更内容`配下に列挙したファイル全件」などの相対参照で記述する
- `## 調査結果`・`## 変更内容`配下の追加文面案・反映文面案など
  他文書本文を転記する長文段落（120字を超え得る段落）は`text`コードブロックで包んで記述する。
  textlintのsentence-length違反を初版`Write`段階で回避する
  - `## 調査結果`配下の既存docstring要約・既存関数の説明文・既存規範文の引用などの長文段落も対象に含める
    （行数・件数等の単純な箇条書きの確定値は対象外）
- 計画本文で機械検査の検出パターン代表例・検出対象語彙・禁止語彙を記述する場合、
  地の文ではなく必ず`text`コードブロック内に配置する（走査対象と代表例が同一節で自己抵触するリスクを事前予防する）
- 新規規範または新規機械検査の導入計画では、計画ファイル本文自身へ先行適用してから初版`Write`へ進み、
  対象該当時は対象外形式（コードフェンス内転記・抽象表現での参照など）へ事前変換する（規範導入計画は初回適用事例）
- 計画ファイル本文全域を`Write`前にscratchpad配下の一時ファイルへ出力する
  - 出力対象は`## 背景`配下のフェンス付きコードブロックのみ除外する
   （ユーザー提示素材原文の口語表現で事前検査が失敗するため）。`## 変更内容`配下のコードブロックは
    対象に含め、unified diffはフェンスと行頭`+`・`-`を除去して出力する
  - 出力後に次のコマンドを実行する。
    `uvx pyfltr run-for-agent --commands=textlint,markdownlint,typos,colloquial-check --enable=colloquial-check <一時ファイル>`
    続けて`check_line_width.py`・`check_dash.py`・`check_line_ref.py`・`check_self_ref.py`も実行し、
    本文全域の違反を反映前に解消する
   （工程7を代替としない。`writing-standards`の厳格適合対象は`## 変更内容`配下のコードブロックに限定）

## 計画段階の調査・検証細則

各節の詳細手順は[investigation-rules.md](investigation-rules.md)に集約する。

## 6-1. 参照テンプレート読み込み

計画ファイルの初版Write直前に本ファイルをReadで読み込む（未実施時はブロックされるため簡易対応でも省略しない）。
対象ファイル一覧が`agent-toolkit:agent-standards`「文書サイズ上限」節の対象ファイルを含む計画を対象とする。
対象ファイル全件の現行と改訂後見込みの`wc -l`実測値を`## 調査結果`と`### エージェント判断`へ列挙する。
境界近接（同節が定める200〜219行）のファイルはscratchpad配下の一時ファイルへ
追記文面案と縮減対象文面を実際に貼り込み`wc -l`で実測する。
追記量と縮減量の相殺を`### エージェント判断`へ必須記載する（文面案の見た目行数と実物理行数の乖離を計画段階で解消するため）。
初版Write後の`check_wc_projection.py`実行契機・検査内容・乖離超過時の対処は
`integrity-checks.md`「メイン側担当部分」節を参照する。

## 6-2. 計画ファイル本体の作成

テンプレートに従い所定パスへ計画ファイルを作成し、
工程2〜5の合意事項・解釈・恒久化文面・周辺対応をテンプレート規定の各セクションへ転記して計画ファイル自立性を満たす。
