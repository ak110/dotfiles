# 計画ファイル作成ガイドライン

計画ファイル作成時に参照する記述例と、構成・注意点を収録する。

## 全セクションを通した記述例

以下は単一フェーズ計画の記述例。

````markdown
# ファイルアップロード上限を10MBから50MBへ引き上げ

## 変更履歴

- 初版
- codexレビュー指摘反映（1回目）
  - 却下: 新旧の上限値を切り替えるfeature flagを用意すべき — ロールバックは設定値の差し替えで十分対応でき、feature flagを追加する運用複雑度の増加に見合う利点がないため

## 背景

高解像度画像をアップロードできないという要望が継続して提示されている。
ストレージと転送帯域の増強が完了し運用コストの懸念が解消されたため、上限値を引き上げて要望に応える。

## 対応方針

### ユーザー合意済み事項

- 上限値は50MBに変更
- 超過時のエラーメッセージ文言は既存のものを流用
- 経路上の各層に個別設定されているサイズ制限やタイムアウトを事前に確認し、整合確認を検証範囲に含める

### エージェント判断

- 上限値の定義を一元化してSSOTを確立
- 既存の超過判定とエラー通知の挙動は維持し、参照値のみを差し替え
- `docs/architecture/limits.md` の「設定値管理」節に以下を追記する

  ```markdown
  上限定義はサーバー側で一元管理し、クライアントは設定APIから取得する。
  上限値を変更する際は、サーバー設定・クライアント定義・リバースプロキシ設定の3層を必ず整合させる。
  ```

### 却下した代替案

- ファイル種別ごとに上限を変える案 — 運用が複雑化する割に現時点で必要性が明確でないため
- 上限を撤廃する案 — 想定外のリソース消費や不正利用のリスクが残るため

## 調査結果

- サーバー側の上限定義: `server/config.py:34`
- クライアント側の上限定義: `client/limits.ts:12`（サーバーと独立して保持されており不整合が発生しやすい）
- リバースプロキシのbody size上限: `infra/nginx.conf:58` で20MB
- 関連テスト: `tests/upload_test.py:40-78`

## 変更内容

- 上限値定義の引き上げ: `server/config.py`・`infra/nginx.conf`の上限を50MBへ更新
- クライアント側の重複定義撤去: `client/limits.ts`の独立保持を解消し、設定API経由で取得する形へ統一
- 境界値テストの追従: `tests/upload_test.py`の境界値ケースを新上限基準で改訂
- 副次: `client/limits.ts`のクライアント側上限値定義を設定API経由の取得に統一し、二重管理を解消する
- 副次: `tests/upload_test.py`の境界値テストを`@pytest.mark.parametrize`に集約し、上限値変更時の影響範囲を縮小する

## 実行方法

- 実装方式: careful-impl
- 実装時に事前呼び出しが必要なスキル
  - `agent-toolkit:careful-impl`
  - `agent-toolkit:coding-standards`
- 検証手順
  - `uvx pyfltr run-for-agent tests/upload_test.py`
  - 追加したテストが修正前は失敗し修正後は成功することを確認する
- コミット方針: コミットする。subject: `feat(upload): ファイルサイズ上限を50MBへ引き上げる`

## 計画ファイル（本ファイル）のパス

`~/.claude/plans/upload-limit-increase-concurrent-hickey.md`
````

## セクション構成と記述要件

計画ファイルは以下のセクション構成を厳守する。
人間は判断に関わる重点セクションを冒頭で確認でき、codexは実装レベルの照合材料を後半でまとめて参照できる。
両者の関心が混ざらない構造にするため。

セクション・サブセクションの追加や順序の変更は禁止。
例外: `## 変更内容`・`## 実行方法`節内のフェーズ分割サブ節（`### フェーズN: 題目`）は許容する。

以下の各サブ節で「目的」「記述要件」「禁止事項」「例外」を示す。
まず全体に共通する禁止事項を記述し、その後に各セクション固有の要件を上から順に整理する。

### 判断材料セクション全体に共通する禁止事項

`# タイトル`・`## 変更履歴`・`## 背景`・`## 対応方針`・
`### ユーザー合意済み事項`・`### エージェント判断`・`### 却下した代替案`の
判断材料セクション群には以下の禁止事項が共通して適用される。

- プログラムレベルの実装詳細を書いてはならない
  - 禁止対象: 関数名・メソッド名・クラス名・変数名・行番号・file:line・コード片・
    外部APIの具体値・特定ミドルウェアや設定キーなど、識別子レベルの記述全般
   （ファイル名・パス単独は禁止対象外。実装詳細を伴わない参照は許容する）
  - 理由: これらの節は「なぜ」「どの方向へ」の判断材料を提示する場であり、実装詳細が混入すると
    方針の抽象度が崩れてユーザーが妥当性を評価しにくくなる。
    また実装エージェントが計画記述を過剰な拘束として受け取り、より適切な実装選択肢を排除する
  - 違反例:「`parse_plan()`で解析処理を一本化する」「`src/foo.py`の`Handler`クラスを修正する」
  - 正例:「計画ファイル解析処理のエントリを単一化する」「リクエスト処理層の入力検証を強化する」
   （`CHANGELOG.md`を作成しない方針、のようなファイル名単独の参照はOK）
  - 実装詳細は`## 調査結果`と`## 変更内容`にのみ記述する
- 実装方法・対象ファイル列挙・ファイル別作業項目を書いてはならない
  - これらは方針の方向性ではなく実行手順に属する情報のため、`## 変更内容`または`## 実行方法`へ集約する
  - 違反例:「`src/foo.py`と`src/bar.py`を順に修正する」「フェーズ1で既存テスト整備、フェーズ2で本実装」
  - 正例:「リクエスト処理層の入力検証を強化する」（具体的な対象ファイルは`## 変更内容`側で記述する）
- 他セクションと記述を重複させてはならない
  - 同一内容を複数セクションへ展開せず、各節の役割に沿った最小限の記述に留める
  - 例えば`### エージェント判断`の方針を`## 変更内容`で言い換え列挙したり、
    恒久化検討の永続化先文面を他節にも書いたりしない
- 例外: ドキュメント修正の変更後文面は上記の実装詳細禁止の対象外とする。記述ルールは後述の`### 対応方針`節に従う

### タイトル（`# タイトル`）

- 目的: 計画の主題を1行で示す

### 変更履歴（`## 変更履歴`）

- 目的: 変更の概要と理由を記録し、レビューアーが変更点を把握しやすくする（人間・codex向け）
  - 変更内容自体を事細かに書くのではなく、指摘内容と修正の方向性を簡潔にまとめる
  - 却下した指摘についても本節に残すことで、codexの再指摘回避と却下理由の参照容易性を両立する
- 記述要件:
  - 更新単位ごとに見出しを設け、見出し配下のbullet項目に指摘と修正方針を記述する
  - 1項目あたりの粒度は1〜2行で、`<指摘要約> — <修正方針>`の二要素のみで端的に書く
   （背景説明・経緯・補足・実装詳細は書かない。レビュー時に所在と方針が一目で分かる状態を保ち、
    更新を重ねても本セクションが長文化しないようにするため）
  - 却下した指摘は`却下: <指摘要約> — <却下理由>`の形式で書く
  - 見出しは下記の表記例に従う。運用上揺れやすい更新単位の表記を統一するため
- 表記例（各表記例の配下に実例を1つずつ示す）:

  ```markdown
  ## 変更履歴

  - 初版

  - codexレビュー指摘反映（1回目）
    - エラー型を`ValueError`から`InvalidUploadError`へ — 上位ハンドラーの分岐に合わせるため
    - 却下: 新旧の上限値を切り替えるfeature flagを用意すべき — ロールバックは設定値の差し替えで十分対応でき、feature flagを追加する運用複雑度の増加に見合う利点がないため

  - ユーザー判断反映（1回目）
    - リバースプロキシのbody size上限を50MBから60MBへ変更 — 余裕を持たせる方針をユーザーが選択したため
  ```

- 表記例ごとの使い分け:
  - `- 初版` — 計画ファイルの初回作成
  - `- codexレビュー指摘反映（N回目）` — codexレビュー指摘を反映した更新（却下含む）
  - `- ユーザー判断反映（N回目）` — ユーザー指示で方針・スコープ・対応可否を変更した更新

### 背景（`## 背景`）

- 目的: なぜこの変更をするか。経緯と期待する成果（人間向け）

#### バグ対応サマリーの記録

バグ対応時は`## 背景`節末尾に以下の表を必ず含める。経緯・目的の本文を先に書き、その下に表を置く。
表形式で1か所に集約することで、別セッションへの引き継ぎと振り返りが容易になる。

| 項目 | 内容 |
| --- | --- |
| 根本原因 | 起点まで追跡した一次原因を1〜2文で記述 |
| 修正方針 | 直接の修正内容を1〜2文で記述（再発防止策と対で検討する） |
| 再発防止策 | 自動で機能する仕組み（テスト・CI・型制約など）を優先して記述 |
| 類似見直し結果 | 根本原因に基づき類似パターンの他箇所を確認した範囲と結果 |
| 知見の記録 | 恒久化先（CLAUDE.md・ルール・スキル等）と転記内容 |

本表は対症療法ではなく根本対応に至ったことを示すチェックリストでもある。
`references/bugfix-process.md`の手順で得た結論を本表に集約する。
「類似見直し結果」欄は計画ファイル作成前に類似箇所の調査を完了させ、確定した結果を記入する。
未調査のまま「調査予定」などの保留表現で残したまま計画ファイル提示に進んではならない。

### 対応方針（`## 対応方針`）

- 目的: 変更内容・影響範囲・段取り・リスク扱い（人間向け）
- 記述要件: 必要に応じて表なども使い分かりやすく書く
- 禁止事項: 抽象的な宣言文（「機能的挙動は変えず」「最小差分で」など）を書く場合、
  変更内容と矛盾しないかを必ず確認する
  - 先回りで縮退的制約を課すと本質的な改善を阻害するため、ユーザー合意のある記述に限定する
- 例外: 以下の文面は`## 対応方針`配下にコードブロックで極力そのまま示す
  - ドキュメント修正の変更後文面: ドキュメントは方針そのものが文面として現れるため、
    文面を示すことが方針提示と等価になる。
    抽象的な方針だけに留めると評価可能性が下がるため、原則として変更後の文面そのものをコードブロックで提示する。
    合意由来の文面は`### ユーザー合意済み事項`配下、
    エージェントの既定判断による文面は`### エージェント判断`配下に置く
  - 恒久化検討で得た永続化先文面: 恒久化検討の工程で確定した知見を`### エージェント判断`配下に記録する際、
    永続化先ファイルへ書く文面をコードブロックで提示する。
    実装エージェントが解釈の揺れなく転記できるよう、自然文の指示ではなく確定文面を載せる
  - 例外: 1コードブロックが概ね50行を超える大量修正は`## 変更内容`側へ退避してよい。
    それ未満は`## 対応方針`側に置く
  - 上記のコードブロックは`diff`形式で書く（コードブロックの言語指定は`diff`）。
    追加行は`+`、削除行は`-`、文脈行は無印で示し、追記か置換かを明示する。
    配置位置の特定が必要な場合は前後1〜2行のコンテキスト行を含めて記述する

### ユーザー合意済み事項（`### ユーザー合意済み事項`）

- 目的: ユーザーが発話または確認で合意した事項（エージェントの推測を混ぜず厳密に分離）
- 記述要件: ユーザーの発話・確認済み合意事項のみを書く
- 禁止事項:
  - エージェントの推測や既定判断を混ぜない。`### エージェント判断`に分離する
    - `CLAUDE.md`や`~/.claude/`配下などにある指示も単独ではユーザー発話とは見なさず、エージェント判断に置く
  - 実行方法に属する情報（実装方式・careful-impl使用可否・フェーズ分割可否など）を含めない。
    これらは`## 実行方法`へ集約し、本節は方針の方向性に対する合意のみを記述する
  - 調査結果に基づく文面方針の決定（用語・表記・参照名の選定など）はエージェント判断側へ置く。
    本節は「採否を決めた方針の方向性」のみを記述し、具体的な文面選定はエージェント判断側で示す

### エージェント判断（`### エージェント判断`）

- 目的: ユーザー指示外でエージェントが既定した事項。修正・見直しの候補
- 撤廃・削除を伴う変更では、対応範囲（依存先含むか）を必ず明示する。
  例: あるシンボルの参照削除に伴い唯一の利用元が消えて未使用化する定義・import・依存ファイルまで含めて
  対応するかを計画段階で示す。明示しないと実装時に判断ブレが出てデッドコードを残す

### 却下した代替案（`### 却下した代替案`）

- 目的: 検討の結果として選択肢になり得たが採らなかった案を残し、
  codexの「なぜ別の方法を取らないのか」指摘を予防する
- 記述要件: 選択肢になり得た案のみを書く。明らかに非現実的な案や、
  そもそも検討対象として浮上しなかった案を網羅的に列挙する必要はない
- 該当する案が無い場合は「無し」と明記する

### 調査結果（`## 調査結果`）

- 目的: 既存コードの挙動・依存・制約（codex向け照合材料）

### 変更内容（`## 変更内容`）

- 目的: 変更方針と変更対象ファイルを示し、計画モードを知らない読者（codex等）へ網羅性を提示する
- 記述要件:
  - 同一方針の変更はファイル単位で羅列せず、方針単位でまとめて対象ファイルを併記する。
    ファイル数や恒久化先が増えた計画でも一覧が長大化しないようにするため
   （ファイル数が少なく方針単位が分散する場合はファイル単位の記述でも可）
  - `## 調査結果`との役割分担を保つ。詳細なfile:lineは`## 調査結果`に集約し、
    `## 変更内容`は方針と対象ファイルを簡潔に示すにとどめる
  - 副次対応項目（リファクタリング検討で決定した既存不備修正）は項目先頭に`副次:`プレフィクスを付け、
   「対象（ファイルパス・関数名・節名）」と「実施内容」の2要素を記述する。
    本来の変更項目と区別し、読み手が一目で識別・対応できる粒度を保つため
  - 公開関数の追加・変更を含む計画では、対応する単体テストファイルへの追加・修正項目が含まれているかを確認する。
    テストファイル名はテスト同居方式では`<module>_test.py`、`tests/`分離方式では`tests/<module>_test.py`を採る
- 禁止事項:
  - コード修正内容をコードブロックで転記しない
    - 方針・境界・挙動・外部インターフェース・確定済み制約のみを書き、
      変数名・エラーメッセージ文言・ループ構造・条件分岐の内訳・小規模な制御フローは書かない
     （計画ファイル由来の具体記述が誤読されて実装されると、本来不要な制約として固定化されるため）
    - 粒度の目安:「この記述を読んだ別の実装者が方針を誤解せず進められる」最低限まで
  - `### エージェント判断`に書いた方針を言い換えて列挙しない
    - 本節は方針を「対象ファイルとセットで具体化」する位置付けのため、対象を伴わない方針表明だけの項目は書かない
    - 違反例:`### エージェント判断`に「サーバーAPIをドメイン単位に分割」と書いたうえで、
      `## 変更内容`に「APIをドメイン別ファイルに分割する」と方針のみを再列挙すること
    - 正例:`## 変更内容`では「サーバーAPI分割: `server/api.ts`をドメイン別ファイルへ再配置」のように
      対象と組み合わせた具体化単位で1度だけ記述する

### 実行方法（`## 実行方法`）

- 目的: `ExitPlanMode`後の実装・検証・コミットに必要な計画固有の情報を記述する
- 記述要件: 以下の項目を上から順に記述する
  - `- 実装方式`（1行記述）
    - 値は`careful-impl`または`careful-impl不使用`のいずれか
    - `careful-impl`の場合、メインから`agent-toolkit:careful-impl`スキルを呼び出して実装・検証・レビュー・コミットを行う
    - `careful-impl不使用`の場合、メインが実装・検証・コミットを直接行う
  - `- 実装時に事前呼び出しが必要なスキル` リスト
    - `実装方式: careful-impl`の場合、リスト先頭に`agent-toolkit:careful-impl`を必須として列挙する
     （セッション断絶時にも呼び出すべきスキル名が計画ファイルから一意に特定できるようにするため）
    - 上記に続けて、変更対象に応じて`agent-toolkit:coding-standards`・`agent-toolkit:writing-standards`・
      `agent-toolkit:spec-driven`などを列挙する（対象変更に含まれなければ列挙しない）
  - `- 検証手順` リスト
    - その計画を完了してコミットする前に必要な検証を具体的に記述する
    - プロジェクト方針として定められた手順（format/lint/test一式など）と、計画固有の観測手順の両方を含める
    - 自動化不可な手動確認が必要な場合は理由付きで明示する
  - `- コミット方針`（1行記述）
    - コミット有無とコミットメッセージのsubject行を記述する
    - 本文（subject行以降）は任意で2行目以降に添える
    - ユーザー判断などにより複数コミットに分割する場合はここに分割方針も記述する
      - ただし必要以上に細かく分割はしない。基本的には1計画1コミットで十分
       （細かすぎるコミット分割はレビュアーの負担増加・コミット履歴の煩雑化につながるため）

### 計画ファイル（本ファイル）のパス（`## 計画ファイル（本ファイル）のパス`）

- 目的: 末尾に計画ファイルの絶対パスを記述する
  - コンテキストのクリア時にClaude Codeがパスを渡さず内容だけ渡す問題に対するワークアラウンド

## 計画ファイル全体の注意点

複数セクションに横断する記述方針。

- 計画ファイルは実装工程で単独参照される前提で書く。
  実装工程はコンテキスト圧縮・セッション境界などで計画時の会話履歴を失った状態から開始されることがあり、
  計画ファイルが唯一の入力源になりうるため、計画時点の判断はすべて計画ファイル内に記述しきる
- plan modeの制約で必要な調査が実行できない場合、ユーザーに確認の上で先に実行する
- プラグイン編集を伴う計画では、バージョン更新方針（要否・PATCH/MINOR/MAJOR種別・対象プラグイン）を
  `### エージェント判断`配下に記述する。
  計画時点で種別を確定させ、実装時はプラグインリポジトリで定められたbump手順に従う。
- `## 調査結果`・`## 変更内容`の記述に入る前に、変更対象ファイルと周辺ファイルを通読し、
  既存の章構成・記述粒度・抽象度・命名規則を把握する
  - 例えばドキュメント追記では、対象ファイルを通読せずに書き始めると既存記述との粒度不整合を招きやすい
- 横断的な置換・参照除去・キーワード削除を計画するときは、対象キーワードを `grep -rn` などで全件列挙し、
  該当箇所を`## 調査結果`に列挙してから`## 変更内容`の方針をまとめる
 （網羅漏れを防ぎ、codexレビュー時の追加指摘や実装後の漏れを減らすため）
- ソースコードだけでなく、ドキュメントへの影響も必ず確認する。設計意図などの追記も検討する
- 検証は自動実行できる手順を最優先する（手動検証は再現性が低く、回帰テストとしても機能しないため）
  - 既存の自動チェック基盤で不足する場合は新たな仕組みの導入も検討する（大規模な変更になる場合はユーザー確認要）
  - 手動検証は自動化が技術的・コスト的に困難な場合に限定し、理由を計画ファイル内に明記する
  - 「修正前は失敗し、修正後は成功する」ことを観測可能にする手段を用意する
- 要件が曖昧で仕様を確定しにくい場合は、詳細な仕様化よりも軽量なプロトタイプ反復を提案する選択肢を検討する
  - 仕様化コストより動作例の方が意思決定を早める場合があるため
- 方針転換はユーザーへ可否を確認してから行う
- 計画ファイルもMarkdownのため、`agent-toolkit:writing-standards`スキルを呼び出す
- コード修正を伴う計画では、実装方法検討のため`agent-toolkit:coding-standards`スキルも呼び出す
- ユーザー判断で計画方針を転換した場合、関連セクションを連動して更新する。
  1セクションだけ更新すると整合性が崩れて再指摘・再修正の対象になる。
  以下の連動更新先を網羅的に確認する
  - `## 変更履歴`に`ユーザー判断反映（N回目）`を追記
  - `## 対応方針`配下の関連節を更新（合意内容の変更は`### ユーザー合意済み事項`、
    既定判断の変更は`### エージェント判断`、検討して採らなかった案は`### 却下した代替案`）
  - `## 変更内容`の対象ファイル・項目を方針に沿って更新
  - `## 実行方法`の検証手順・コミット方針（subject行を含む）を新方針と整合させる
  - 文面の追加・削除を伴う場合は`## 対応方針`内のコードブロック（変更後文面）も併せて更新する

## 複数フェーズ計画の記述方法

フェーズ分割は原則として行わない。
フェーズ分割はコスト増大・実装時間の長期化・全体最適の阻害につながるため、
独立した複数の大規模な機能変更が同一計画に含まれる場合に限り許容する。
分割を検討する前に、まず単一フェーズで成立しないかを必ず確認する。
例えば「実装と、その実装のためのリファクタリングと、それらに関するドキュメント更新」は
分割すべきでない（下記の例は説明のための例外）。

- フェーズ分割の記述場所: `## 変更内容`と`## 実行方法`の2節の内側に
  `### フェーズ1: 題目`・`### フェーズ2: 題目` …のサブ節を並べる
  - フェーズ題目は端的に表す名詞句（例: `フェーズ1: 既存テスト整備`）
  - フェーズ番号は連番で付ける
- 他の大見出し（`## 変更履歴`・`## 背景`・`## 対応方針`・`## 調査結果`）はフェーズ分割せず全体で記述する
- フェーズごとに、`## 変更内容`サブ節には当該フェーズで触る変更対象ファイルと変更方針を、
  `## 実行方法`サブ節には当該フェーズの検証手順・コミット方針を書く。
  全フェーズ共通の手順や方針（実装方式・事前呼び出しスキルなど）は親節（フェーズサブ節の外側）にまとめる
  - `実装方式`は計画ファイル全体で単一の値とし、フェーズごとに切り替えない。
    一度`careful-impl`に入ると同一セッションで全フェーズが`careful-impl`配下で進行するため、
    途中フェーズだけ`careful-impl不使用`に切り替える運用は成立しない
- レビューは全フェーズ・全コミット完了後に計画全体を対象として一度だけ実施するため、フェーズ単位のレビュー入力は使わない

`## 実行方法`節では全フェーズ共通の「実装方式」「実装時に事前呼び出しが必要なスキル」を親節に置き、
フェーズ別サブ節に「検証手順」「コミット方針」を記述する。
`## 変更内容`と`## 実行方法`のみ抜粋する。

```markdown
## 変更内容

### フェーズ1: 既存テスト整備

- 修正: `tests/upload_test.py` のフィクスチャを共通化

### フェーズ2: 上限値の引き上げ

- 修正: `server/config.py` の上限定義を50MBへ
- 修正: `infra/nginx.conf` のbody size上限を50MBへ

## 実行方法

- 実装方式: careful-impl
- 実装時に事前呼び出しが必要なスキル
  - `agent-toolkit:careful-impl`
  - `agent-toolkit:coding-standards`

### フェーズ1: 既存テスト整備

- 検証手順
  - `uvx pyfltr run-for-agent`
- コミット方針: コミットする。subject: `test(upload): フィクスチャを共通化する`

### フェーズ2: 上限値の引き上げ

- 検証手順
  - `uvx pyfltr run-for-agent`
  - 50MB境界のテストが追加されており修正前は失敗・修正後は成功することを確認する
- コミット方針: コミットする。subject: `feat(upload): ファイルサイズ上限を50MBへ引き上げる`
```
