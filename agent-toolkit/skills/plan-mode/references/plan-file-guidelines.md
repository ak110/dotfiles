# 計画ファイル作成ガイドライン

計画ファイルの記述例・セクション構成・注意点を示す。

## 全セクションを通した記述例

以下は単一フェーズ計画の記述例。

````markdown
# ファイルアップロード上限を10MBから50MBへ引き上げ

## 変更履歴

- 初版
- codexレビュー指摘反映（1回目）
  - 却下: 新旧の上限値を切り替えるfeature flagを用意すべき — ロールバックは設定値の差し替えで十分対応でき、feature flagを追加する運用複雑度の増加に見合う利点がないため

## 背景

高解像度画像をアップロードできないという要望が継続して提示されている。
ストレージと転送帯域の増強が完了し運用コストの懸念が解消されたため、上限値を引き上げて要望に応える。

## 対応方針

### ユーザー合意済み事項

- 上限値は50MBに変更
- 超過時のエラーメッセージ文言は既存のものを流用
- 経路上の各層に個別設定されているサイズ制限やタイムアウトを事前に確認し、整合確認を検証範囲に含める

### エージェント判断

- 上限値の定義を一元化してSSOTを確立
- 既存の超過判定とエラー通知の挙動は維持し、参照値のみを差し替え
- `docs/architecture/limits.md` の「設定値管理」節に以下を追記する

  ```markdown
  上限定義はサーバー側で一元管理し、クライアントは設定APIから取得する。
  上限値を変更する際は、サーバー設定・クライアント定義・リバースプロキシ設定の3層を必ず整合させる。
  ```

### 却下した代替案

- ファイル種別ごとに上限を変える案 — 運用が複雑化する割に現時点で必要性が明確でないため
- 上限を撤廃する案 — 想定外のリソース消費や不正利用のリスクが残るため

## 調査結果

| ファイル | 内容 |
| --- | --- |
| `server/config.py:34` | サーバー側の上限定義 |
| `client/limits.ts:12` | クライアント側の上限定義。サーバーと独立して保持されており不整合が発生しやすい |
| `infra/nginx.conf:58` | リバースプロキシのbody size上限が20MB |
| `tests/upload_test.py:40-78` | 関連テスト |

## 変更内容

| ファイル | 操作 | 内容 |
| --- | --- | --- |
| `server/config.py`・`infra/nginx.conf` | 修正 | 上限値定義を50MBへ更新する |
| `client/limits.ts` | 修正 | 独立保持を解消し、設定API経由で取得する形へ統一する |
| `tests/upload_test.py` | 修正 | 上限値判定の境界値ケース（49.9MB成功・50MB成功・50.1MB失敗・0MB失敗）を`@pytest.mark.parametrize`へ集約し、新上限基準で改訂する。上限値変更時の影響範囲を縮小する |

## 実行方法

- レビュー方式: レビュー有り
- 実装時に事前呼び出しが必要なスキル
  - `agent-toolkit:plan-impl`
  - `agent-toolkit:coding-standards`
- 検証手順
  - `uvx pyfltr run-for-agent tests/upload_test.py`
  - 追加したテストが修正前は失敗し修正後は成功することを確認する
- コミット方針: コミットする。subject: `feat(upload): ファイルサイズ上限を50MBへ引き上げる`

## 計画ファイル（本ファイル）のパス

`~/.claude/plans/upload-limit-increase-concurrent-hickey.md`
````

## セクション構成と記述要件

計画ファイルは以下のセクション構成を厳守する。
セクション・サブセクションの追加や順序の変更は禁止。
例外: `## 変更内容`・`## 実行方法`節内のフェーズ分割サブ節（`### フェーズN: 題目`）は許容する。

### 判断材料セクション全体に共通する禁止事項

`# タイトル`・`## 変更履歴`・`## 背景`・`## 対応方針`・
`### ユーザー合意済み事項`・`### エージェント判断`・`### 却下した代替案`には以下の禁止事項が共通して適用される。

- プログラムレベルの実装詳細を書いてはならない
  - 禁止対象: 関数名・メソッド名・クラス名・変数名・行番号・コード片・外部APIの具体値・設定キーなど識別子レベルの記述全般
   （ファイル名・パス単独は禁止対象外）
  - 実装詳細は`## 調査結果`と`## 変更内容`にのみ記述する
- 実装方法・対象ファイル列挙・ファイル別作業項目を書いてはならない。これらは`## 変更内容`または`## 実行方法`へ集約する
- 他セクションと記述を重複させてはならない
- 例外: ドキュメント修正の変更後文面は上記の実装詳細禁止の対象外とする

### タイトル（`# タイトル`）

計画の主題を1行で示す。

### 変更履歴（`## 変更履歴`）

- 更新単位ごとに見出しを設け、見出し配下のbullet項目に指摘と修正方針を記述する
- 1項目あたりの粒度は1〜2行で、`<指摘要約> — <修正方針>`の二要素のみで端的に書く
- 却下した指摘は`却下: <指摘要約> — <却下理由>`の形式で書く
- 見出しの表記例（運用上揺れやすいため統一する）:
  - `- 初版` — 計画ファイルの初回作成
  - `- codexレビュー指摘反映（N回目）` — codexレビュー指摘を反映した更新（却下含む）
  - `- ユーザー判断反映（N回目）` — ユーザー指示で方針・スコープ・対応可否を変更した更新

### 背景（`## 背景`）

なぜこの変更をするか。経緯と期待する成果（人間向け）。

#### バグ対応サマリーの記録

バグ対応時は`## 背景`節末尾に以下の表を必ず含める。

| 項目 | 内容 |
| --- | --- |
| 根本原因 | 起点まで追跡した一次原因を1〜2文で記述 |
| 修正方針 | 直接の修正内容を1〜2文で記述（再発防止策と対で検討する） |
| 再発防止策 | 自動で機能する仕組み（テスト・CI・型制約など）を優先して記述 |
| 類似見直し結果 | 根本原因に基づき類似パターンの他箇所を確認した範囲と結果 |
| 知見の記録 | 恒久化先（CLAUDE.md・ルール・スキル等）と転記内容 |

「類似見直し結果」欄は計画ファイル作成前に調査を完了させ、確定した結果を記入する。

### 対応方針（`## 対応方針`）

変更内容・影響範囲・段取り・リスク扱い（人間向け）。

- 禁止事項: 抽象的な宣言文（「機能的挙動は変えず」「最小差分で」など）を書く場合、
  変更内容と矛盾しないかを必ず確認する
- 例外: 以下の文面は`## 対応方針`配下にコードブロックで極力そのまま示す
  - ドキュメント修正の変更後文面（文面が方針と等価になるため）
  - 恒久化検討で得た永続化先文面
  - コードブロックは`diff`形式で書く（追加行`+`・削除行`-`・文脈行無印）
  - 1コードブロックが概ね50行を超える大量修正は`## 変更内容`側へ退避してよい

### ユーザー合意済み事項（`### ユーザー合意済み事項`）

ユーザーが発話または確認で合意した事項のみを書く。エージェントの推測・既定判断・CLAUDE.md等の
指示内容は単独ではユーザー発話と見なさず、`### エージェント判断`に分離する。
実行方法に属する情報（実装方式・フェーズ分割可否など）は含めない。

### エージェント判断（`### エージェント判断`）

ユーザー指示外でエージェントが既定した事項。修正・見直しの候補。
撤廃・削除を伴う変更では、対応範囲（依存先含むか）を必ず明示する。

### 却下した代替案（`### 却下した代替案`）

選択肢になり得たが採らなかった案を残し、codexの「なぜ別の方法を取らないのか」指摘を予防する。
該当する案が無い場合は「無し」と明記する。

### 調査結果（`## 調査結果`）

既存コードの挙動・依存・制約（codex向け照合材料）。
表形式（「ファイル」「内容」の2列固定）で記述する。1行=1観測事項。

### 変更内容（`## 変更内容`）

表形式（「ファイル」「操作」「内容」の3列固定）で記述する。1行=1方針。
副次対応は「操作」列に`副次`と記載する。

- コード修正内容をコードブロックで転記しない。方針・境界・挙動・外部インターフェース・確定済み制約のみを書く
- `### エージェント判断`に書いた方針を対象ファイルを伴わずに言い換えて再列挙しない
- 公開関数の追加・変更を含む計画では、対応する単体テストファイルへの追加・修正項目が含まれているかを確認する
- テスト設計は計画段階で具体化する。同値分割と境界値分析（JSTQB準拠）でテストケースを抽出する
  - 同値分割: 同等の振る舞いをするとみなせる入力範囲ごとに代表値を1つ選ぶ
  - 境界値分析: 各範囲の上限値・下限値とその直前直後を選ぶ
  - 抽出したケースはフレームワーク機能（`@pytest.mark.parametrize`等）でケース集約する。
    1ケース1関数で書き並べると関数数が増えてケース数の経済性が下がるため、
    パラメーター化で1関数へ集約しテストデータの読みやすさとケース数の経済性を両立する
- テスト計画は`## 変更内容`の表内に明示する。
  新規・変更対象の公開関数等に対し、抽出したケース・境界値を「内容」列へ列挙する。
  境界値（上限値・下限値・直前直後）はパラメーター化テストへ集約し、
  上限・下限変更時の影響範囲を縮小する旨も対象テストファイルの行に記述する

### 実行方法（`## 実行方法`）

以下の項目を上から順に記述する。

- `- レビュー方式`: `レビュー無し`または`レビュー有り`のいずれか
- `- 実装時に事前呼び出しが必要なスキル` リスト:
  先頭に`agent-toolkit:plan-impl`を必須として列挙する
- `- 検証手順` リスト: 計画固有の観点と自動チェック手順の両方を含める
- `- コミット方針`: コミット有無とコミットメッセージのsubject行を記述する

### 計画ファイル（本ファイル）のパス（`## 計画ファイル（本ファイル）のパス`）

末尾に計画ファイルの絶対パスを記述する（コンテキストクリア時のワークアラウンド）。

## 計画ファイル全体の注意点

- 計画ファイルは実装工程で単独参照される前提で書く。計画時の判断はすべて計画ファイル内に記述しきる
- `## 調査結果`・`## 変更内容`の記述に入る前に、変更対象ファイルを原則全体通読する。
  部分読み込みのまま進めると不適切な箇所への追記や粒度不整合が起きやすい
- 横断的な置換・参照除去・キーワード削除を計画するときは、対象キーワードを`grep -rn`などで全件列挙し、
  該当箇所を`## 調査結果`に列挙してから`## 変更内容`の方針をまとめる
- 計画ファイルに記載するファイルパス（`## 調査結果`・`## 変更内容`の表内のファイル列、
  対応方針節で言及するパス等）のうち、既存ファイルとして参照するパスは計画提出前に
  Readまたはlsで実在を必ず確認する。
  新規作成予定のパスは`## 変更内容`表の「操作」列に「新規」と明示し、
  親ディレクトリの実在確認を行う。
  非実在のパスを既存扱いで記載すると調査・実装フェーズで気付かれずに進行する事故が起きやすい
- 検証は自動実行できる手順を最優先する。手動検証は自動化が困難な場合に限定し、理由を明記する
- ユーザー判断で計画方針を転換した場合、関連する以下のセクションを連動して更新する
  - `## 変更履歴`に`ユーザー判断反映（N回目）`を追記
  - `## 対応方針`配下の関連節を更新
  - `## 変更内容`の対象ファイル・項目を方針に沿って更新
  - `## 実行方法`の検証手順・コミット方針を新方針と整合させる

## 複数フェーズ計画の記述方法

フェーズ分割は原則として行わない。独立した複数の大規模な機能変更が同一計画に含まれる場合に限り許容する。

- フェーズ分割の記述場所: `## 変更内容`と`## 実行方法`の2節の内側に
  `### フェーズ1: 題目`・`### フェーズ2: 題目` …のサブ節を並べる
- 他の大見出し（`## 変更履歴`・`## 背景`・`## 対応方針`・`## 調査結果`）はフェーズ分割せず全体で記述する
- `レビュー方式`は計画ファイル全体で単一の値とし、フェーズごとに切り替えない
- レビューは全フェーズ・全コミット完了後に計画全体を対象として一度だけ実施する

`## 変更内容`と`## 実行方法`のみ抜粋する。

```markdown
## 変更内容

### フェーズ1: 既存テスト整備

| ファイル | 操作 | 内容 |
| --- | --- | --- |
| `tests/upload_test.py` | 修正 | フィクスチャを共通化する |

### フェーズ2: 上限値の引き上げ

| ファイル | 操作 | 内容 |
| --- | --- | --- |
| `server/config.py` | 修正 | 上限定義を50MBへ更新する |
| `infra/nginx.conf` | 修正 | body size上限を50MBへ更新する |

## 実行方法

- レビュー方式: レビュー有り
- 実装時に事前呼び出しが必要なスキル
  - `agent-toolkit:plan-impl`
  - `agent-toolkit:coding-standards`

### フェーズ1: 既存テスト整備

- 検証手順
  - `uvx pyfltr run-for-agent`
- コミット方針: コミットする。subject: `test(upload): フィクスチャを共通化する`

### フェーズ2: 上限値の引き上げ

- 検証手順
  - `uvx pyfltr run-for-agent`
  - 50MB境界のテストが追加されており修正前は失敗・修正後は成功することを確認する
- コミット方針: コミットする。subject: `feat(upload): ファイルサイズ上限を50MBへ引き上げる`
```
