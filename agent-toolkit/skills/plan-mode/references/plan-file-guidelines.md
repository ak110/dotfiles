# 計画ファイル作成ガイドライン

## 全セクションを通した記述例

````markdown
# ファイルアップロード上限を10MBから50MBへ引き上げ

## 変更履歴

- 初版
- codexレビュー指摘反映（1回目）
  - 却下: 新旧の上限値を切り替えるfeature flagを用意すべき — ロールバックは設定値の差し替えで十分対応でき、追加運用の複雑度に見合う利点がないため

## 背景

高解像度画像をアップロードできないという要望が継続して提示されている。
ストレージと転送帯域の増強が完了し運用コストの懸念が解消されたため、上限値を引き上げて要望に応える。

## 対応方針

### ユーザー合意済み事項

- 上限値は50MBに変更
- 超過時のエラーメッセージ文言は既存のものを流用

### エージェント判断

- 上限値の定義を一元化してSSOTを確立
- `docs/architecture/limits.md`の「設定値管理」節に3層整合ルールを追記する

### 却下した代替案

- ファイル種別ごとに上限を変える案 — 運用が複雑化する割に現時点で必要性が明確でないため

### 恒久化・リファクタリング内容

- 恒久化: 3層を整合させる運用方針を`docs/architecture/limits.md`へ追記する
- リファクタリング: クライアント側の独立保持を解消し設定API経由のSSOTへ統一する

## 調査結果

- `server/config.py:34`: サーバー側で上限値を10MBとして定義する
- `client/limits.ts:12`: クライアント側で同じ値を独立保持し、サーバーとの整合機構は無い
- `infra/nginx.conf:58`: リバースプロキシの`client_max_body_size`が20MB
- `tests/upload_test.py:40-78`: 上限境界のテストがあり、いずれも10MB前提

## 変更内容
````

## セクション構成と記述要件

### 判断材料セクションの共通禁止事項（`# タイトル`・`## 変更履歴`・`## 背景`・`## 対応方針`配下）

- プログラムレベルの実装詳細（関数名・変数名・行番号・コード片・設定キー等の識別子）は`## 調査結果`と`## 変更内容`にのみ記述する（ファイル名・パス単独は禁止対象外）
- 実装方法・対象ファイル列挙・ファイル別作業項目は`## 変更内容`または`## 実行方法`へ集約する
- 他セクションと記述を重複させない（ドキュメント修正の変更後文面は実装詳細禁止の対象外）

### タイトル（`# タイトル`）

計画の主題を1行で示す。

### 変更履歴（`## 変更履歴`）

更新単位ごとに見出しを設け、箇条書きの各項目に指摘と修正方針を記述する。
1項目1〜2行で書き、`<指摘要約> — <修正方針>`の形式とする。
却下指摘は`却下: <指摘要約> — <却下理由>`の形式とする。
エントリは時系列順（古い→新しい）に並べ、新しい更新は末尾へ追記する。
見出し表記は`- 初版`・`- codexレビュー指摘反映（N回目）`・`- ユーザー判断反映（N回目）`・`- 方針転換（N回目）`を用いる。
`方針転換（N回目）`は実装中・検証中に自発的に発生した方針転換を記録する目的で用いる。
`方針転換（N回目）`の各項目は`<転換前→転換後> — <契機となった観測事象> — <判断理由>`形式で記述する。
レビュー指摘起点の改訂では見出し表記として`codexレビュー指摘反映`・`ユーザー判断反映`を用いる。
全面改訂（計画の根幹が覆る規模の方針転換）時は変更履歴を「初版」へ戻し旧履歴を残さない（codexレビューも初回扱いで再実施）。

### 背景（`## 背景`）

なぜこの変更をするか（人間向け）。
情報源はユーザーの発話・対話で確認した動機・経緯・期待する成果、またはユーザーが提示した文面の原文に限定する。
提示素材の本文は転記する。
文意を保存する書式調整（引用記号「」をbacktickへ変える等の表記統一）は許容する。
原文不変が要件となるのは素材原文との照合根拠が失われる本質的な改変（語の置換・要約・順序変更など）に限る。
長文素材は`text`コードブロックで包む。
背景を埋めるためだけの追加質問はしない。

バグ対応時は本節末尾に表を必ず含め、根本原因・修正方針・再発防止策・類似見直し結果・知見の記録の5項目を記入する（複数バグ対応時はバグごとに表を作成）。

### 対応方針（`## 対応方針`）

変更内容・影響範囲・段取り・リスク扱い（人間向け）。

### ユーザー合意済み事項（`### ユーザー合意済み事項`）

ユーザーが発話または確認で合意した事項のみを記述する。
エージェントの推測・既定判断・CLAUDE.md等の指示内容は`### エージェント判断`に分離する。

### エージェント判断（`### エージェント判断`）

ユーザー指示外でエージェントが既定した事項。
撤廃・削除を伴う変更では対応範囲（依存先含むか）を必ず明示する。
並列サブエージェントへ実装委譲する計画では、各サブエージェントの担当ファイル集合と、当該担当ファイルが参照する既存テストファイル集合の対応関係を本節へ列挙する。
並列サブエージェント計画では、検証・git commit・git push等の担当主体（既定はメイン側完結、個別サブエージェントが担当する場合は`git worktree`による作業ツリー分離の根拠）を本節へ明記する。
各タスク委譲プロンプトには上記対応関係を前提条件として埋め込み、独立コンテキストで起動するサブエージェントが他タスクの影響を考慮できる状態にする。

### 却下した代替案（`### 却下した代替案`）

選択肢になり得たが採らなかった案を残す。
該当案が無い場合は「なし」と明記する。

### 恒久化・リファクタリング内容（`### 恒久化・リファクタリング内容`）

工程4・5で実施すると決めた内容を記載する。
対象が無い場合も「なし」と明記する。
バグ対応時に`## 背景`の調査結果表で記載済みの場合は「背景表に記載」と参照する。

### 調査結果（`## 調査結果`）

既存コードの挙動・依存・制約（codex向け照合材料）を、後続工程で参照できる粒度で記述する。`agent-toolkit:agent-standards`「文書サイズ上限」節の対象ファイルを編集する計画では、当該ファイルの現行行数（`wc -l`実測値）を本節へ記録する。

### 変更内容（`## 変更内容`）

実装するファイルと変更方針を、実装者が再調査せず着手できる粒度で記述する（コード片を含めてよい）。
冒頭に「対象ファイル一覧」H3を必須配置し、変更対象ファイル全件をチェックボックス形式（`- [ ]`）で列挙する。
続けて各ファイルの変更方針をH3見出しで記述する。
対象ファイル一覧のチェックボックス項目と各ファイル変更方針のH3見出しはファイル名で対応関係を保つ。
公開関数の追加・変更を含む計画では、対応する単体テストファイルへの追加・修正項目を含め、同値分割・境界値分析および正規表現の選択肢（`|`）の各分岐でケースを抽出する。
`agent-toolkit:agent-standards`「文書サイズ上限」節の対象ファイルを編集する計画では、改訂後の最終形の物理行数の目標値と実装段階での`wc -l`確認手順を本節へ記述する。

### 実行方法（`## 実行方法`）

実装からレビューまでを実行順の手順として箇条書きで記述する。
冒頭に呼び出すスキルを列挙し、先頭の`agent-toolkit:plan-impl`を必須として作業種別に応じたスキルを続ける。
列挙には配置先スキル（対象リポジトリの`AGENTS.md`・`CLAUDE.md`が編集対象ディレクトリに対して指定するスキル）も必ず含める。
対象リポジトリの`AGENTS.md`・`CLAUDE.md`・`.claude/rules/`配下から検証・レビュー・コミット・ドキュメント確認の順序を抽出して反映する。
実装の実施・検証の実施・`agent-toolkit:commit`の呼び出し・コミットの実施・レビューの実施を各1ステップとして置く。
コミットの実施ステップには件名案をサブバレットへ記す。
レビュー既定は`agent-toolkit:careful-review`の呼び出しとし、ユーザーが明示的にレビュー不要と指示した場合のみ「レビューは実施しない」と記述する。
標準順序は実装→検証→コミット→レビュー→pushとし、`agent-toolkit:careful-review`は`git push`より前に配置する（レビュー指摘で修正が必要な場合に未レビューの変更を公開しないため）。

### 進捗ログ（`## 進捗ログ`）

区切りの節目ごとに`### <ラベル>`見出しと箇条書きで追記する（初版作成時は空でよい）。
記述粒度は判断・成果・残課題・失敗試行を1行ずつ程度にとどめる。
確定形が計画内容と乖離した場合は計画ファイル本文も最終確定形へ更新する。
`### <ラベル>`見出しは時系列ラベル（`### 着手`・`### 検証完了`・`### コミット`・`### レビュー反映`等）または`### 発見`に限定する。
`### 発見`は計画前提と異なる挙動・想定外の副作用・設計判断に影響する観測事実を分離記録する目的で用い、各項目は`<観測事象> — <計画への反映>`形式で記述する。
発見が`## 変更内容`・`## 対応方針`の確定形に影響する場合は本節前項（計画内容との乖離時に計画ファイル本文を最終確定形へ更新する規定）に従い当該本文も同時に更新する。

### 計画ファイル（本ファイル）のパス（`## 計画ファイル（本ファイル）のパス`）

末尾に絶対パスを記述する。

## 計画ファイル全体の遵守事項

- 計画ファイル自立性を満たし、計画時の判断・調査・ユーザー確認はすべて計画ファイル内で確定する（実装段階や計画提出後への先送り表現は本体に書かない）
- `## 調査結果`・`## 変更内容`の記述に入る前に変更対象ファイルを原則全体通読する
- 計画ファイル記載の既存パスは計画提出前に実在確認する。新規作成予定のパスは`## 変更内容`で明示し親ディレクトリの実在を確認する
- ユーザー判断で計画方針を転換した場合、各セクションを連動更新する
- 検証結果が想定と異なる場合は「環境差異」等と解釈せず進捗ログに乖離を記録して計画の前提を見直す
- 検証は自動実行を最優先し、手動検証は自動化困難な場合に限定して理由を明記する
- 計画ファイル本文で対象位置・対象量を示す際、改訂で変動する絶対数値（行番号・行範囲・行数・件数）の直書きを避け、節名・小見出し・冒頭文引用などの相対参照で特定する（`## 調査結果`の確定値は対象外）
- 計画ファイル本文のdiffコードブロック外側fenceは3バッククォート既定とし、内側に3バッククォートを含む場合のみ4バッククォートを採用する
- 計画ファイル本文は作文段階で`agent-toolkit:writing-standards`「Markdown記述スタイル」節の規定を事前適用する
- `## 変更内容`配下の追加文面案・反映文面案など他文書本文を転記する長文段落（120字を超え得る段落）は`text`コードブロックで包んで記述し、textlintのsentence-length違反を初版Write段階で回避する
