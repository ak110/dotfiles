# 計画ファイル作成ガイドライン

## 全セクションを通した記述例

````markdown
# ファイルアップロード上限を10MBから50MBへ引き上げ

## 変更履歴

- 初版
- codexレビュー指摘反映（1回目）
  - 却下: 新旧の上限値を切り替えるfeature flagを用意すべき — ロールバックは設定値の差し替えで十分対応でき、feature flagを追加する運用複雑度の増加に見合う利点がないため

## 背景

高解像度画像をアップロードできないという要望が継続して提示されている。
ストレージと転送帯域の増強が完了し運用コストの懸念が解消されたため、上限値を引き上げて要望に応える。

## 対応方針

### ユーザー合意済み事項

- 上限値は50MBに変更
- 超過時のエラーメッセージ文言は既存のものを流用
- 経路上の各層に個別設定されているサイズ制限やタイムアウトの整合確認を検証範囲に含める

### エージェント判断

- 上限値の定義を一元化してSSOTを確立
- 既存の超過判定とエラー通知の挙動は維持し、参照値のみを差し替え
- `docs/architecture/limits.md` の「設定値管理」節に以下を追記する

  ```markdown
  上限定義はサーバー側で一元管理し、クライアントは設定APIから取得する。
  上限値を変更する際は、サーバー設定・クライアント定義・リバースプロキシ設定の3層を必ず整合させる。
  ```

### 却下した代替案

- ファイル種別ごとに上限を変える案 — 運用が複雑化する割に現時点で必要性が明確でないため
- 上限を撤廃する案 — 想定外のリソース消費や不正利用のリスクが残るため
````

## セクション構成と記述要件

セクション構成を厳守する。セクション・サブセクションの追加や順序変更は禁止。

### 判断材料セクション全体に共通する禁止事項

`# タイトル`・`## 変更履歴`・`## 背景`・`## 対応方針`・
`### ユーザー合意済み事項`・`### エージェント判断`・`### 却下した代替案`に共通の禁止事項。

- プログラムレベルの実装詳細を書かない
  - 禁止対象: 関数名・メソッド名・クラス名・変数名・行番号・コード片・外部APIの具体値・設定キーなど識別子レベルの記述全般
   （ファイル名・パス単独は禁止対象外）
  - 実装詳細は`## 調査結果`と`## 変更内容`にのみ記述する
- 実装方法・対象ファイル列挙・ファイル別作業項目は`## 変更内容`または`## 実行方法`へ集約する
- 他セクションと記述を重複させない
- 例外: ドキュメント修正の変更後文面は実装詳細禁止の対象外

### タイトル（`# タイトル`）

計画の主題を1行で示す。

### 変更履歴（`## 変更履歴`）

- 更新単位ごとに見出しを設け、箇条書きの各項目に指摘と修正方針を記述する
- 1項目1〜2行、`<指摘要約> — <修正方針>`の二要素のみで書く
- 却下した指摘は`却下: <指摘要約> — <却下理由>`の形式で書く
- 見出し表記:
  - `- 初版` — 計画ファイルの初回作成
  - `- codexレビュー指摘反映（N回目）` — codexレビュー指摘を反映した更新（却下含む）
  - `- ユーザー判断反映（N回目）` — ユーザー指示で方針・スコープ・対応可否を変更した更新
- 全面改訂時の取り扱い:
  - 計画の根幹（背景・対応方針）が覆る規模の方針転換が発生した場合、変更履歴を「初版」へ戻す。
    旧履歴の指摘・却下事由は新方針と対応せずレビュー時の混乱要因となるため残さない
  - 全面改訂判定はエージェントが自律的に行ってよい。
    判断に迷う場合は部分修正（`ユーザー判断反映（N回目）`）として扱う
  - 全面改訂後はcodexレビューも初回扱いで再実施する（`references/codex-review.md`の該当節を参照）

### 背景（`## 背景`）

なぜこの変更をするか。経緯と期待する成果（人間向け）。
背景に書く情報源は以下の2系統に限定する。

- ユーザーが発話または対話で確認した動機・経緯・期待する成果
- ユーザーが指示の素材として提示した文面の原文
  - 例: レビュー指摘・エラー出力・スタックトレース・要件抜粋・ログ・参考リンクの引用

提示素材は要約・抜粋せず原文を転記し、計画ファイル単独で判断材料が揃う情報量を確保する。
上記2系統に紐づかない動機・経緯・成果はエージェントの推測で補わない。
上記2系統が共に端的な場合に限り、背景章も端的な記述を許容する。
背景を埋めるためだけの追加質問はしない（要件対話工程で目的・スコープ等を確認した内容は転記してよい）。
本制約は叙述部分のみが対象で、バグ対応時の調査結果表（根本原因・修正方針・再発防止策など）は対象外とする。

バグ対応時は`## 背景`節末尾に以下の表を必ず含める。

| 項目 | 内容 |
| --- | --- |
| 根本原因 | 起点まで追跡した一次原因を1〜2文で記述 |
| 修正方針 | 直接の修正内容を1〜2文で記述（再発防止策と対で検討する） |
| 再発防止策 | 自動で機能する仕組み（テスト・CI・型制約など）を優先して記述 |
| 類似見直し結果 | 根本原因に基づき類似パターンの他箇所を確認した範囲と結果 |
| 知見の記録 | 恒久化先（CLAUDE.md・ルール・スキル等）と転記内容 |

「類似見直し結果」欄は計画ファイル作成前に調査を完了させ、確定した結果を記入する。

### 対応方針（`## 対応方針`）

変更内容・影響範囲・段取り・リスク扱い（人間向け）。

- 抽象的な宣言文（「機能的挙動は変えず」「最小差分で」など）を書く場合、変更内容と矛盾しないか必ず確認する
- 例外: 以下の文面は`## 対応方針`配下にコードブロックで極力そのまま示す
  - ドキュメント修正の変更後文面（文面が方針と等価になるため）
  - 恒久化検討で得た永続化先文面
  - コードブロックは`diff`形式で書く（追加行`+`・削除行`-`・文脈行無印）
  - 1コードブロックが概ね50行を超える大量修正は`## 変更内容`側へ退避してよい

### ユーザー合意済み事項（`### ユーザー合意済み事項`）

ユーザーが発話または確認で合意した事項のみを書く。
エージェントの推測・既定判断・CLAUDE.md等の指示内容は`### エージェント判断`に分離する。
実行方法に属する情報（実装方式など）は含めない。

### エージェント判断（`### エージェント判断`）

ユーザー指示外でエージェントが既定した事項。修正・見直しの候補。
撤廃・削除を伴う変更では、対応範囲（依存先含むか）を必ず明示する。

### 却下した代替案（`### 却下した代替案`）

選択肢になり得たが採らなかった案を残し、codexの「なぜ別の方法を取らないのか」指摘を予防する。
該当案が無い場合は「無し」と明記する。

### 調査結果（`## 調査結果`）

既存コードの挙動・依存・制約（codex向け照合材料）。
表形式（「ファイル」「内容」の2列固定）で記述する。1行=1観測事項。

### 変更内容（`## 変更内容`）

表形式（「ファイル」「操作」「内容」の3列固定）で記述する。1行=1方針。
副次対応は「操作」列に`副次`と記載する。

- コード修正内容をコードブロックで転記しない。方針・境界・挙動・外部インターフェース・確定済み制約のみを書く
- `### エージェント判断`の方針を対象ファイルを伴わずに言い換えて再列挙しない
- 公開関数の追加・変更を含む計画では、対応する単体テストファイルへの追加・修正項目を含める
- テスト設計は計画段階で具体化する。同値分割と境界値分析（JSTQB準拠）でテストケースを抽出する
  - 同値分割: 同等の振る舞いをするとみなせる入力範囲ごとに代表値を1つ選ぶ
  - 境界値分析: 各範囲の上限値・下限値とその直前直後を選ぶ
  - 抽出したケースは`@pytest.mark.parametrize`等のパラメーター化機能で1関数へ集約する
- テスト計画は`## 変更内容`の表内に明示する
  - 新規・変更対象の公開関数等に対し、抽出したケース・境界値を「内容」列へ列挙する
  - 境界値はパラメーター化テストへ集約し、上限・下限変更時の影響範囲を縮小する旨を記述する

### 実行方法（`## 実行方法`）

以下の項目を上から順に記述する。

- `- 実装時に事前呼び出しが必要なスキル` リスト: 先頭に`agent-toolkit:plan-impl`を必須として列挙する
- `- 検証手順` リスト: 計画固有の観点と自動チェック手順の両方を含める
- `- コミット方針`: コミット有無とコミットメッセージのsubject行を記述する
- `- レビュー方式`: 既定は`agent-toolkit:careful-review`スキルを使用
  - プロジェクト固有のレビュースキル・エージェントを併用する場合は
    `agent-toolkit:careful-review`スキルと`<エージェント名>`エージェントを使用のように併記する
  - ユーザーが明示的にレビュー不要と指示した場合のみ`レビュー無し`とする

### 進捗ログ（`## 進捗ログ`）

実装フェーズの作業進捗を残す。コンテキストコンパクションを介してもセッション継続できるよう、
区切りの節目（1タスク完了・1コミット作成直前・レビュー指摘反映の節目など）ごとに
`### <ラベル>`見出しと箇条書きで追記する。

- 計画ファイル初版作成時点では本節は空でよい
- 各箇条書きは1項目1行を原則とする
- `<ラベル>`はタスクIDや作業区切りを識別できる短い見出しとする
  - 例:「タスク#3完了」「コミット前検証」「レビュー指摘#1反映」
- 記述粒度は判断・成果・残課題を1行ずつ程度にとどめ、コード差分や長文の調査結果は転記しない
- 失敗した試行・方針転換は理由とともに1行で残し、続けて採用した方針を別項目に書く

### 計画ファイル（本ファイル）のパス（`## 計画ファイル（本ファイル）のパス`）

末尾に計画ファイルの絶対パスを記述する（コンテキストクリア時のワークアラウンド）。

## Gotchas（計画ファイル全体）

- 計画ファイル自立性を満たす。計画時の判断はすべて計画ファイル内に記述しきる
- `## 背景`の動機・経緯・期待する成果の叙述部分には、以下の2系統のみを書く。エージェントの推測で補わない
  - ユーザー発話・対話結論
  - ユーザー提示素材の原文（例: レビュー指摘・エラー出力・要件抜粋）
  - 提示素材は要約・抜粋せず原文を転記し、計画ファイル単独で判断材料が揃う情報量を確保する
  - 上記2系統が共に端的な場合に限り、背景章も端的な記述を許容する
  - 背景を埋めるためだけの追加質問はしない（要件対話工程で目的・スコープ等を確認した内容のみ転記する）
  - バグ対応時の調査結果表（根本原因・修正方針・再発防止策など）は本制約の対象外
- 計画ファイル本体に「〇〇を確認する」「実装時に〇〇を確認する」のような、
  計画段階で結論を確定すべき調査事項を実装段階へ先送りする表現が混入しやすい。
  計画ファイル作成時点で判断材料が揃っていることを意識し、先送り表現を避ける
  - 例外: `## 実行方法`の検証手順内（テスト実行・動作確認など検証作業自体の計画）は対象外
- `## 調査結果`・`## 変更内容`の記述に入る前に、変更対象ファイルを原則全体通読する。
  部分読み込みのまま進めると不適切な箇所への追記や粒度不整合が起きやすい
- 横断的な置換・参照除去・キーワード削除を計画するときは、対象キーワードを`grep -rn`等で全件列挙し、
  該当箇所を`## 調査結果`に列挙してから`## 変更内容`の方針をまとめる
- 計画ファイルに記載するファイルパスのうち、既存ファイルとして参照するパスは計画提出前に
  Readまたはlsで実在を必ず確認する
  - 新規作成予定のパスは`## 変更内容`表の「操作」列に「新規」と明示し、親ディレクトリの実在を確認する
  - 非実在のパスを既存扱いで記載すると調査・実装段階で気付かれずに進行する事故が起きやすい
- 検証は自動実行できる手順を最優先する。手動検証は自動化が困難な場合に限定し、理由を明記する
- 1計画は1リポジトリあたり1コミットが基本。複数コミットへの分割は極力避ける
  計画粒度の判断はユーザー責務であり、エージェント側でフェーズ分割やサブ節分割を導入しない
- `## 対応方針`配下にコードブロックで提示した文面案を実装段階で対象ファイルへ転記する場合、
  独立した完成文として成立するか確認する。計画文中の文脈に依存した断片を切り出すと、転記先で意味が通らなくなる
- `## 変更内容`の表内記述と`## 対応方針`配下のコードブロックで提示した文面が乖離した場合、対応方針配下の文面を正とする
- ユーザー判断で計画方針を転換した場合、関連するセクションを連動して更新する
  - `## 変更履歴`に`ユーザー判断反映（N回目）`を追記
  - `## 対応方針`配下の関連節を更新
  - `## 変更内容`の対象ファイル・項目を方針に沿って更新
  - `## 実行方法`の検証手順・コミット方針を新方針と整合させる
- 計画でルール・スタイルガイド・運用方針などの記述規範を追記または改訂する場合、
  計画ファイル本文も当該規範の適用対象として整合させる。
  追加・改訂したルールに計画文面自体が違反したまま計画提出すると、実装段階で違反を引き継ぐ
- 計画提出前に計画ファイル本文の機械チェックを実行する
  - プロジェクトがpyfltrを採用している場合は
    `uvx pyfltr run-for-agent --work-dir=. <計画ファイルパス>` を実行する。
    カレントディレクトリをプロジェクトルートにすることで `.textlintrc.yaml`・`.markdownlint-cli2.yaml` が解決され、
    プロジェクト規約のtextlint・markdownlint・colloquialが計画ファイル本文へ一括実行される。
    `pre-commit` コマンドはリポジトリ外パスでエラーになるが、他コマンドの結果を主軸として扱う
  - pyfltr非採用、またはtextlint/markdownlint設定が無いプロジェクトでは
    `agent-toolkit/skills/writing-standards/scripts/check_colloquial.py`と
    `agent-toolkit/skills/writing-standards/scripts/check_line_width.py`を計画ファイル本文へ直接実行する
  - 機械チェック適合性の判定対象は、計画ファイル内のコードブロック内（成果物の想定文面）に限定する
   （コードブロック外の本文中のfalse positiveはエージェント側の判断で除外する）
- 計画範囲外のファイルで違反を検知した場合、自己判断で書き換えず修正方針をユーザーへ提示し、合意を得てから着手する
- ユーザーへの確認事項は計画提出前に対話で結論を確定させる
  - 「実装時に確認する」「コミット時に判断する」のように結論確定を先送りしない
- 計画ファイル内の記述間の単純な不整合（章節間の対象ファイル・対応箇所の食い違いなど）は計画提出前に解消する
  - codexレビューには設計上の論点・矛盾検出を集中させる方針のため、初歩的整合性チェックを肩代わりさせない
