# 体系的なバグ対応手順

対症療法（表面的な症状の解消）と根本対応（原因の除去と再発防止）の違いを常に意識する。

| 事象 | 対症療法 | 根本対応 |
| --- | --- | --- |
| compose.yamlのバージョン不一致 | バージョン番号を揃える | 依存更新が起点と特定し、更新時の自動検証を追加 |
| 本番DBのクエリータイムアウト | タイムアウト値を延長 | データ量増加に伴うインデックス不足を特定し追加 |
| API連携の突然の失敗 | リトライ処理を追加 | 外部APIの仕様変更を特定し、スキーマ検証を導入 |

ユーザーからの報告に対応する際は、plan modeを経由して以下の手順で進める。

## 1. 根本原因の特定

- 事象を再現する。期待値、実際値、発生条件を明確にする
  - 一部環境・一部条件でのみ発生するバグは、再現する環境・条件の差異（テナント・データ・設定・バージョンなど）を先に特定する
  - 差異が確認できない確率的・断続的なバグや再現条件を限定できないバグでは、再現試行に固執せず、
    ログ・メトリクス・関連コミット履歴など事後の痕跡から原因を特定する方針へ切り替える
- 表面的な症状で止めず、その状態が生まれた起点まで追跡する
  - コード変更、データの蓄積・変質、外部サービスの仕様変更、設定の不整合、環境差異など、原因の形態を先入観で限定しない
  - 変更履歴（gitログ、デプロイ履歴）、ログ、メトリクス、設定差分を確認し、事実に基づいて原因を特定する
- 複雑な制御フロー・状態遷移・依存関係は、ASCII図や箇条書きのフローで理解を可視化してから原因特定に進む
  - 頭の中だけで追跡すると気付き漏れが生じるため
- 原因仮説が複数ある場合は、支持する事実と棄却する事実を整理し、採用した原因を明示する
- フロントエンドやWeb UIで再現するバグで、コンソールログ・ネットワーク通信・DOM状態のいずれかの直接観測が必要な場合は、
  Playwright系やChrome DevTools系のMCPツールが利用可能であれば優先する
  - サーバー側のログやレスポンスのみで原因特定できるケースは対象外

## 2. 根本原因への対処

根本原因が判明したら、「直接の修正」「再発防止策」「類似見直し」の3点を同時に検討する。
これらは独立したタスクではなく、同一原因に対する多角的な対処である。
1つでも欠けた状態で計画ファイル提示に進むと、修正漏れや再発を招きやすい。

### 直接の修正

- 修正方針は、修正前に失敗し修正後に成功するテストや再現手順と対で定義する
- 修正方針はユーザーの本来の要求（求める方向性・スコープ・成果像）と常に整合させる。
  副次論点や外部指摘の解消を優先するあまり当初の要求から逸脱した対処へ変更しないよう、
  方針候補を比較するときは「本来の要求からの距離」を判定軸の1つとして必ず含める

### 再発防止策

- 直接の修正のみで再発防止策がなければ、同じ問題が再び起きうる
- 人の注意に依存するものより、自動で機能する仕組み（テスト、CI、バリデーション、型制約など）を優先する
- 過剰な対策は避け、再発確率と影響度に見合った水準にする
- 既存の再発防止策があるのに同じ問題が再発した場合は、
  既存策が機能しなかった原因（カバレッジの抜け・テスト対象と症状の乖離・環境差異など）を特定し、
  同じ手段の追加にとどめず再発防止策の設計自体を見直す

### 類似見直し

- 同じ根本原因やパターンによる問題が他の箇所に残っていないか確認する
- 共通処理、コピー実装、類似条件分岐を重点的に調べる
- 根本原因の修正で自動解消される箇所も、影響範囲を確認して報告する

## 3. 知見の記録

- 根本原因・再発防止策・判断理由を再利用可能な場所（CLAUDE.md、運用手順書、コメントなど）に記録する
- 暗黙知だった判断基準の明文化や、今後も誤りやすい箇所の注意喚起を特に重視する

## 計画ファイルへの集約

上記1〜3で導いた根本原因・直接の修正方針・再発防止策・類似見直しの範囲・知見の記録先を、
計画ファイルの`## 背景`直下の表へ集約する。
（表の具体的な項目定義は`plan-file-guidelines.md`の「バグ対応サマリーの記録」節に従う）
