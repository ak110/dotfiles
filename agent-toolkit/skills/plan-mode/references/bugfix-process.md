# 体系的なバグ対応手順

対症療法（表面的な症状の解消）と根本対応（原因の除去と再発防止）の違いを常に意識する。

| 事象 | 対症療法 | 根本対応 |
| --- | --- | --- |
| compose.yamlのバージョン不一致 | バージョン番号を揃える | 依存更新が起点と特定し、更新時の自動検証を追加 |
| 本番DBのクエリータイムアウト | タイムアウト値を延長 | データ量増加に伴うインデックス不足を特定し追加 |
| API連携の突然の失敗 | リトライ処理を追加 | 外部APIの仕様変更を特定し、スキーマ検証を導入 |

## 1. 根本原因の特定

- 事象を再現する。期待値、実際値、発生条件を明確にする
- 再現困難な場合はログ・メトリクス・関連コミット履歴など事後の痕跡から原因を特定する
- クラッシュ症状（プロセスが即終了しダイアログも残らない場合）は事後の痕跡から診断する。
  - OS標準のクラッシュログ（Windowsイベントログの`アプリケーション`カテゴリー等）から
    例外コード・障害が発生したモジュール・スタックトレース相当情報を取得する
  - ランタイム固有の対処は各プロジェクトの規範文書に従う
- 表面的な症状で止めず、コード変更・データ蓄積・外部サービス変更・設定不整合など起点まで追跡する
- 複雑な制御フロー・状態遷移・依存関係は図や箇条書きで可視化してから原因特定に進む
- フロントエンドでコンソール・ネットワーク・DOM観測が必要な場合はPlaywright系MCPツールを優先する
- 上流ライブラリ・ツール・OSなどの変更（既定変更・仕様変更・破壊的変更・廃止・非推奨化等を含む）
  が原因の候補となる場合、リリースノート・CHANGELOG・関連issueトラッカー・関連PRを必ず参照する。
  挙動差分の観察だけでworkaroundを即時採用せず、上流の意図を確認してから対処方針を選ぶ
- 推測した因果チェーン（観測事象、分岐条件、到達状態の各段）は、該当分岐を実装するコードを
  `Read`で確認して対象パス・対象値・分岐条件を1段ずつ裏付ける。裏付けが取れない段が残る場合は
  `### バグ調査結果`表の「直接原因の候補」欄へ列挙し、断定形の「根本原因」欄には書かない。
  裏付けなしの因果チェーンを計画本文へ確定形で書くと、レビュー工程で撤回や却下判定の反復が発生する

## 2. 根本原因への対処

根本原因が判明したら、「直接の修正」「再発防止策」「類似見直し」の各観点を同時に検討する。
1つでも欠けたまま計画ファイル提示に進むと、修正漏れや再発を招く。

### 直接の修正

- 修正方針は、修正前に失敗し修正後に成功するテストや再現手順と対で定義する
  （テストの位置づけは「再発防止策」節の再現テスト・再発防止テストの区別に従う）
- ユーザーの本来の要求（求める方向性・スコープ・成果像）と整合させる

### 再発防止策

再発防止策は次の手順で導出する。

1. 直接原因の特定
2. その原因を許した設計上・実装上の問題の特定
3. 同じ原因によって発生し得る別の不具合パターンの列挙
4. それらをまとめて防ぐために守るべき不変条件の定義
5. 不変条件を実装上どこで強制するかの決定
6. 不変条件が破られた場合に検出できる汎用的なテストの設計

対策強度は次の序列で選定し、上位層が成立しない場合のみ下位層を選ぶ。
下位層採用時は上位層の不成立理由を計画本文の`### 却下した代替案`節へ直接記録する。
外部ファイル（ワークアラウンドメモ・検討補助メモ等）への委譲は計画ファイル自立性違反として認めない。
適用契機はバグ対応に限定せず、オペレーション上の不備・エージェント自身の判断欠陥・
繰り返す再現事象を含む初回発生時点でも評価する。本項が対策強度序列のSSOTとし、
`agent-toolkit/rules/01-agent.md`「品質最優先」節は本節への参照のみを配置する。

1. 設計変更による誤りの構造的抑止（不正な状態を表現できない型定義・API境界の再設計等）
2. 自動チェック（型制約・lint・テスト・CI・hook等の自動で機能する仕組み）
3. チェックリスト・手順書
4. ドキュメント・注意喚起（選定時は自動化・hook化等の上位層への格上げ条件を1行以上検討済みであることを要する）

層3・層4を選定する場合、新規の禁止バレット追加は`agent-toolkit:agent-standards`
「肯定形優先の記述」節の推奨併記義務に加え、次の理由説明を1文以上添える。
理由説明の対象は、禁止対象の判断がなぜ誤りとして生じるか（誤認知の機構）と、
代替の正規手順がなぜ品質に寄与するかである。

既存の再発防止策があるのに再発した場合は、既存策が機能しなかった原因を特定し設計自体を見直す。

テストを再発防止策として計上する場合は「再現テスト」と「再発防止テスト」を区別する。

- 再現テスト: 今回報告された具体値・ファイル名・ID・API・処理経路にだけ依存するテスト。
  「直接の修正」節が定めた失敗テストと修正後成功テストの役割を担い、
  修正の検証手段として必須だが、それ単独を再発防止策として計上しない
- 再発防止テスト: 「今回とは異なる入力や別の呼び出し元で同じ原因の不具合が起きた場合にも、
  この対策で検出または防止できるか」に肯定で答えられるテスト。
  不変条件の定義手順が特定した不変条件を検証対象とする

肯定で答えられない場合は不変条件レベルの対策（設計変更・型制約・プロパティベーステスト・境界での検証等）を再検討する。
特定事象のみに最適化した対症的な対策を再発防止策として計上しない。

再発防止策のレビューでは次の観点を確認する。

- 有効性: 対策が根本原因（手順2で特定した設計上・実装上の問題）に作用しているか
- 汎用性: 今回と異なる入力・別の呼び出し元でも同一原因の不具合を検出・防止できるか
- 対策分類: 各対策が予防（発生自体を抑止）・検知（発生した事象の早期発見）・緩和（発生後の影響低減）
  のいずれに該当するかを区別し、検知・緩和のみで予防が欠ける構成を警告する
- 横展開: 「類似見直し」節が定める確認結果が計画に含まれているか

規範改善提案（バグの再発防止策を除いた恒久化候補の全般）の対策手段の優先順位は
`agent-toolkit/skills/session-review/SKILL.md`「対策手段の優先順位」の項目に従う。

### 類似見直し

適用契機・深さ基準・記録先のSSOTは`similar-review.md`に集約する。
バグ対応時は`similar-review.md`の「深さ基準」に従い、同じ根本原因やパターンによる問題が他の箇所（共通処理・コピー実装・類似の条件分岐等）に残っていないかを母集団の悉皆点検で確認する。

## 3. 知見の記録

根本原因・再発防止策・判断理由を再利用可能な場所（CLAUDE.md・運用手順書・コメントなど）に記録する。

## 計画ファイルへの集約

上記1〜3で導いた根本原因・直接の修正方針・再発防止策・類似見直しの範囲・知見の記録先を、
計画ファイルの`## 背景`の`### バグ調査結果`H3配下の表へ集約する。
項目定義は`plan-file-guidelines.md`「背景（`## 背景`）」節の`### バグ調査結果`H3に一方向で従う。

バグ対応時の恒久化（知見の記録）は本表の「知見の記録」欄へ集約し、
`### 恒久化・リファクタリング内容`では「バグ調査結果表に記載」と参照して重複させない。
