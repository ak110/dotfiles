# 体系的なバグ対応手順

対症療法（表面的な症状の解消）と根本対応（原因の除去と再発防止）の違いを常に意識する。

| 事象 | 対症療法 | 根本対応 |
| --- | --- | --- |
| compose.yamlのバージョン不一致 | バージョン番号を揃える | 依存更新が起点と特定し、更新時の自動検証を追加 |
| 本番DBのクエリータイムアウト | タイムアウト値を延長 | データ量増加に伴うインデックス不足を特定し追加 |
| API連携の突然の失敗 | リトライ処理を追加 | 外部APIの仕様変更を特定し、スキーマ検証を導入 |

## 1. 根本原因の特定

- 事象を再現する。期待値、実際値、発生条件を明確にする
- 再現困難な場合はログ・メトリクス・関連コミット履歴など事後の痕跡から原因を特定する
- クラッシュ症状（プロセスが即終了しダイアログも残らない場合）は事後の痕跡から診断する。
    - OS標準のクラッシュログ（Windowsイベントログの`アプリケーション`カテゴリー等）から
      例外コード・障害が発生したモジュール・スタックトレース相当情報を取得する。
    - ランタイム固有の対処は各プロジェクトの規範文書に従う。
- 表面的な症状で止めず、コード変更・データ蓄積・外部サービス変更・設定不整合など起点まで追跡する
- 複雑な制御フロー・状態遷移・依存関係は図や箇条書きで可視化してから原因特定に進む
- フロントエンドでコンソール・ネットワーク・DOM観測が必要な場合はPlaywright系MCPツールを優先する
- 上流ライブラリ・ツール・OSなどの変更（既定変更・仕様変更・破壊的変更・廃止・deprecation等を含む）
  が原因の候補となる場合、リリースノート・CHANGELOG・関連issueトラッカー・関連PRを必ず参照する。
  挙動差分の観察だけでworkaroundを即時採用せず、上流の意図を確認してから対処方針を選ぶ

## 2. 根本原因への対処

根本原因が判明したら、「直接の修正」「再発防止策」「類似見直し」の各観点を同時に検討する。
1つでも欠けたまま計画ファイル提示に進むと、修正漏れや再発を招く。

### 直接の修正

- 修正方針は、修正前に失敗し修正後に成功するテストや再現手順と対で定義する
- ユーザーの本来の要求（求める方向性・スコープ・成果像）と整合させる

### 再発防止策

- 人の注意に依存するものより、自動で機能する仕組み（テスト・CI・バリデーション・型制約など）を優先する
- 既存の再発防止策があるのに再発した場合は、既存策が機能しなかった原因を特定し設計自体を見直す

### 類似見直し

- 同じ根本原因やパターンによる問題が他の箇所に残っていないか確認する
- 共通処理、コピー実装、類似条件分岐を重点的に調べる

## 3. 知見の記録

根本原因・再発防止策・判断理由を再利用可能な場所（CLAUDE.md・運用手順書・コメントなど）に記録する。

## 計画ファイルへの集約

上記1〜3で導いた根本原因・直接の修正方針・再発防止策・類似見直しの範囲・知見の記録先を、
計画ファイルの`## 背景`直下の表へ集約する。
（表の具体的な項目定義は`plan-file-guidelines.md`の「背景（`## 背景`）」節に従う）

バグ対応時の恒久化（知見の記録）は本表の「知見の記録」欄へ集約し、
`### 恒久化・リファクタリング内容`では「背景表に記載」と参照して重複させない。
