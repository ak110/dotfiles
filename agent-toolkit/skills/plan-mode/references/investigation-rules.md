# 計画段階の調査・検証細則

## 計画段階の調査原則

- メイン側Read範囲は計画成立に必須なファイルに限定する。
  広範囲な周辺調査・横断検索・列挙系grepは`Explore`サブエージェントへ委譲する
- 新規実装または既存機能拡張を計画する場合、
  既存類似先例を1件以上特定し当該パス・該当節を`### エージェント判断`に明示する
  （先例が見つからない場合はその旨を明記する）
  - 探索は対象ファイルへの`git log -- <path>`、および対象変更パターン
    （依存単発更新・セキュリティ対応など）を含むコミット件名のキーワード検索を併用して網羅的に行う。
    網羅確認なしに「先例なし」と結論しない
  - 既存類似先例から仕様（拡張子フィルタ集合・除外ディレクトリ集合・検出ロジック・コードブロック除外形式等）を
    抽出する場合は当該ファイルを全体Readしたうえで実集合・実ロジックを計画文へ転記する。
    部分参照や名前からの推測で仕様を確定しない
- 他ファイルの節タイトル・節内項目名・キーワードを計画文面に引用する場合は、
  引用元節をReadで原文確認してから転記する
- サブエージェントからの調査報告に含まれる件数・関数名・ファイルパス・行番号などの確定値は、
  計画文・成果物へ転記する前にメイン側で`grep`・Readにより原本照合する
- 変更対象が同種パターンの呼び出し点群
  （同一関数の呼び出し箇所群・同一デコレータ適用箇所群・同一設定キー参照箇所群など）に
  該当する場合は`grep -rn`で全件列挙してから`## 変更内容`の対象範囲を確定する
- 計画本文で対象ファイル群を規模記述で確定させる場合、根拠となる
  `ls | wc -l`または`find ... | wc -l`等の実測コマンドと出力値を
  `## 調査結果`へ併記する（試算値のみでの確定は不可とする）。
  内訳列挙が同一文書内にある場合は`plan-file-guidelines.md`
  「計画ファイル全体の遵守事項」節の相対参照ルールを優先する

## 関連参照確認の細則

- ファイル移動・削除・改名: 旧パス参照と「両ファイル」「N個のファイル」等の件数前提記述
- ファイル追加・コンポーネント追加: 仕様書のサービス一覧・構成図・READMEの機能一覧・対応表・
  「N件」「N観点」等の総数表記
  - 検索キーワードは追加対象の概念名（スキル名・サービス名など）と
    総数表記（「N件」「N観点」など）を用いる
- 概念名・スキル名・見出しラベルのリネーム・新規スキル導入:
  既存スキル側で陳腐化する節見出し・description・mermaid図ラベル等
  - 検索キーワードには変更前の見出し文字列・変更前の参照ラベル文字列を含める
  - 他文書から参照される節タイトル・H1〜H3見出し文字列の変更も対象とする
- スキル本体（SKILL.md）でファイル形式・出力形式・質問雛形等の書式を変更する場合:
  当該書式を読み書きするサブエージェント定義（`agent-toolkit/agents/*.md`）も同時更新対象とする。
  当該書式を継承・拡張・引用する関連スキル
  （拡張章を担うスキル・同フォーマットを引用するスキル等）も同時更新対象に含める。
  対象は次のコマンドで照合する。
  `grep -rn '<関連節名|関連識別子>' agent-toolkit/agents/ agent-toolkit/skills/ .chezmoi-source/dot_claude/skills/`
  同SKILL.md内の起動プロンプトテンプレート（「### サブエージェント起動プロンプト」節等）も
  同時更新対象に含める。
  読取側を更新せず書き手のみ更新するとサブエージェントが独立コンテキストで旧書式のまま動作する
- サブエージェント定義または担当節を新規追加・改訂する計画では、
  同一契機で並列起動される他サブエージェント（`plan-spec-reviewer`と
  `plan-impl-reviewer`の対、`careful-review`のカテゴリ分割による並列起動群等）の
  担当観点との重複・境界を`grep -rn`で洗い出す。
  重複が見込まれる場合は追加を撤回するか、担当境界の切り分け文言を計画へ含める。
  境界判断の根拠は`### エージェント判断`へ明記する
- 編集対象スキル本体・サブエージェント定義のfrontmatterコメント（`#`行）の
  同期指示（「意図的重複」「同期要件」等）を確認し同期先ファイルを`## 変更内容`へ組み込む
- 対称的な役割を持つ兄弟機能群（feedback／TBDの後始末コマンド対、Linux／Windowsペア、
  入出力ペア等）を計画対象に含む場合、両者の公開インターフェースを`grep -rn`で全件列挙する。
  対象はサブコマンド一覧・オプション名・出力形式・引数仕様などとする。
  列挙結果を突き合わせて対称性を照合し、片方に存在し他方に欠落する項目は、計画スコープへ組み込むか
  スコープ外扱いにするかを計画確定前に決定し、判断根拠を`### エージェント判断`へ明記する
- 規範文書（`agent-toolkit/skills/*/references/*.md`・`agent-toolkit/rules/*.md`・
  `agent-toolkit/skills/*/SKILL.md`等）を改訂する計画では、
  当該規範を機械的に強制するhookスクリプト・機械検査スクリプト
  （`agent-toolkit/scripts/pretooluse.py`・`agent-toolkit/skills/*/scripts/check_*.py`等）を
  改訂箇所のキーワードで洗い出す。次のコマンドで検索し
  `## 変更内容`対象ファイル一覧に必ず含める
  - `grep -rn '<改訂箇所キーワード>' agent-toolkit/scripts/ agent-toolkit/skills/*/scripts/`
  - キーワード候補: 改訂節の見出し語・条件式に使う識別子・機械強制の対象概念語など
  - 該当スクリプトが見つからない場合はその旨を`### エージェント判断`へ記載する
- agent-toolkitプラグインのversion bumpを計画する場合は
  `agent-toolkit/.claude-plugin/plugin.json`と`.claude-plugin/marketplace.json`の双方を対象ファイル一覧へ含める。
  両者の整合性は`agent-toolkit/scripts/pretooluse_test.py`の`TestManifestSsot`が強制する

確認結果は`## 調査結果`に、追従更新箇所は`## 変更内容`に反映する。

編集対象ファイル全般に対し、当該ファイルへ適用される既存規範違反パターン
（コーディング規範・命名規則・サイズ規定・参照解決方式など）を`grep -nE`で計画作成前に網羅検査する。
検出した既存違反は`## 調査結果`へ列挙し`## 変更内容`に修正項目を組み込む。
新規規範または既存規範改訂を含む計画では`norm-revision-checklist.md`の点検手順を実施する。

各ファイルパスの実在を`test -e`または`ls`で確認し、`AGENTS.md`・`CLAUDE.md`に
「ロールとファイル群の対応」「ディレクトリ構造の注意」相当の表がある場合は、
当該表で配布元・配布先・配布対象外の区別を照合する。
`## 変更内容`の対象ファイル一覧確定後に、
各ファイルパスを`AGENTS.md`・`CLAUDE.md`記載の配置先スキル指定と照合する。
未起動の配置先スキルがあれば計画ファイルWrite前にSkillツールで起動する。
当該スキルが指定する必須記載事項を計画ファイル本文の各セクションへ計画作成時点で組み込む。
必須記載事項の具体例は、
version bump種別判断・`## 実行方法`テンプレート・SSOTファイル同期指示・配布物固有名チェック等を含む。
照合と起動が未完了の状態で計画ファイルWrite・実装着手しない。

`## 変更内容`の対象ファイル一覧が確定した時点で、
修正対象ファイル群（修正前状態）に対して`plan-impl-reviewer`サブエージェントをバックグラウンド並列起動する。
結果は工程7のステップ3で反映する。

## 連動テスト・参照確認の細則

確認対象は公開関数のシグネチャ・挙動・戻り値型・例外仕様・処理対象範囲、
スキル・サブエージェント・公開ツールの呼び出し規約・前提条件・引数仕様、
リテラル文字列・期待値・出力フォーマット等の利用者観測値を含む。

確認対象は単体テストファイル・ドキュメント・スナップショット・hookメッセージ・サンプル設定等を含む。
特に呼び出し元モジュールのテストファイルは旧挙動前提のテストが混在し得るため、必ず読み込んで再評価する。
旧挙動を前提にしたテストの再評価・修正項目は`## 変更内容`へ明示し、確認結果は`## 調査結果`に反映する。

`agent-toolkit/scripts/`配下のhookスクリプト内のモジュールレベル定数や関数を
移管・改名・削除する場合は、削除前に関連テストファイル群を網羅確認する。
具体的には`grep -rn '<識別子名>' agent-toolkit/scripts/`などで参照箇所を列挙する。

- `pretooluse_*_test.py`・`posttooluse_*_test.py`などのファイルは複数存在する
- `*_test.py`を限定指定せず全件をpyfltrの検証対象に含める
- 削除後の検証コマンドの対象範囲に該当する全テストファイルを含める
- フォーム・入力欄・制御要素の追加・変更・共通化を含む計画では、
  対応するサーバー側ハンドラ（受入処理・保存処理・バリデーション）の現状を計画段階で判定する
  - 判定は該当ハンドラをReadで行い、変更が必要な場合は`## 変更内容`の対象ファイル一覧に含める
  - フロントUIのみの変更で完結できると判断した根拠は、`### エージェント判断`へ明記する
- 共通境界（複数の呼び出し元から利用される関数・ミドルウェア・before_request等）へ
  副作用（validation・throw・状態変更）を追加する計画では前提照合を実施する。
  当該共通境界を経由する既存テストのfixture前提を全件grepで事前確認する
  - 追加される副作用の条件を既存fixtureが満たさない場合、
    fixture更新もしくは条件分岐（フィーチャーフラグ・context属性等）で影響範囲を絞る
  - 事前確認結果と対応方針は`### エージェント判断`へ明記する

## 実現可能性の事前検証の細則

結果を`## 調査結果`へ反映し、想定と異なる結果が出た場合は対応方針を変更してから工程3以降を順に進む。
外部依存物（自身の管理外コードに相当するOSS・ライブラリ・SDK・プラグイン・外部CLI・外部サービス等）に
依存する計画では、
表面的なAPI仕様（公開関数・コマンドフラグ・設定キーの型と既定値）だけで前提を確定しない。
依存物のソース・公式ドキュメントの実装節・既知issue・テストコードのいずれかを読み、
次の観点を実機観測に先行して確認し`## 調査結果`へ転記する。

- 入力値ごとに分岐する内部処理経路
- 関連する他設定・グローバル状態・環境変数への副次書き込み
- 読み込み順・評価順・初期化順への依存
- エラー経路・リトライ・タイムアウトの暗黙仕様

自プロジェクト内CLI・モジュールの内部挙動を計画文の根拠とする場合も対象関数本体のみで前提を確定せず、
エントリポイント・前段フック処理まで読み込んで分岐順序と副作用を確認したうえで`## 調査結果`へ反映する。

実機検証が環境制約で実行不能な場合は当該前提を`### エージェント判断`に「未検証」と明記し依拠範囲を縮める
（前提が崩れた場合の代替方針も併記する）。

設定ファイル変更・CLI挙動変更を伴う計画では、
対象CLIに検証用サブコマンド（`critique`・`lint`・`validate`・`check`・`config`・`defaults`等）があれば、
計画段階で実機実行して出力を`## 調査結果`へ転記する。
検証結果に応じて`## 変更内容`の文面案・`## 実行方法`の検証ステップを確定する。

入力文字列の形式判別を伴う設計（URL対ローカルパス、正規化済み識別子対リテラル等）では、
判別ロジックの候補を3〜5の代表入力に対し机上または実機で適用する。
結果を`## 調査結果`へ転記し、誤判定が無いことを確認してから設計を確定する。

正規表現パターン・テスト入力候補を計画文・テストコードへ転記する場合は
`python -c "import re; print(bool(re.search(<pattern>, <input>)))"`等で計画段階で実機検証する。
ユーザー提示素材原文または既存規範文書のglob記法は計画文へ転記前に対象処理系の公式仕様と照合し、
各構成要素（`**`・`*`・`?`等）に対応する代表入力（任意階層・単一階層など）を検証ケースとして列挙する。
結果を`## 調査結果`へ転記し、パターンと入力の整合を机上推定で確定しない。

pylint採用プロジェクトでPythonファイルの追加・移動・追記を含む計画では、
対象ファイルの現行行数を`wc -l`で確認し、追加後の見込み行数が
`max-module-lines`（既定1000行）を超過するか確認する。
超過時は責務分離または`# pylint: disable=too-many-lines`配置のいずれかを計画段階で確定する。
既存`disable`配置済みの場合は理由コメントの追記・更新の要否も判定する。
