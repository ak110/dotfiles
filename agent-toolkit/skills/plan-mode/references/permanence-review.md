# 恒久化検討の詳細手順

工程4「恒久化検討」の詳細手順を集約する。

本工程は`agent-toolkit:agent-standards`配下`references/quality-first-exceptions.md`「正規追加経路の非対称原則」節の経路1を兼ねる。
加えて`agent-toolkit:careful-review`「指摘の統合と修正依頼」節が定める品質俯瞰棚卸しを含む。
ユーザー対話には明示合意・割り込み指摘・選好・判断材料として提示された文面・指示の前提となった暗黙の方向性を含む。
点検対象のうち当該計画固有でない恒久的な知見を整理する。
確定した文面案・検討結果は工程6-2で計画ファイルの`### 恒久化・リファクタリング内容`へ転記する。
永続化先はコードに近い順で検討する。

- 第一候補: ソースコードのdocstring・インラインコメント（コード固有の知見）
- 第二候補: `CLAUDE.md`・`.claude/rules/`配下（プロジェクト横断の方針・繰り返す誤判断の補強）
- 第三候補: `.claude/skills/`配下（情報量が多く利用頻度の低い手順・リファレンス、トリガー時のみロード）
- `docs/`配下にコーディングレベルの指針を書かない。別リポジトリ・別管理でも永続化を見送らない

ユーザー介入が発生した契機（誤判断・規範未認識・手順抜けなど）に対する恒久化候補は、
規範文面の改善だけでなく機械的に再発を抑止する仕組みでも検討する。
対象は、hook追加・スキル冒頭の警告追記・コマンドラッパー追加・トリガーフレーズ拡張・テストケース追加などとする。
仕組み化可能な場合は仕組み案を恒久化先候補に含め、規範文書側の改訂と並行して導入する。

フィードバック採用計画では、各フィードバック提案の反映と並行して1階層上のカテゴリでの
再発予防の検討結果を`### 恒久化・リファクタリング内容`節へ併記する。
詳細規定は`agent-toolkit:agent-standards`配下
`references/feedback-review-common.md`「規範改善案の併記」節をSSOTとする。

計画ファイル提示後・`plan-impl-executor`実行中の割り込み指摘・是正要求は当該修正と並行して本工程を即時再実施する。
同根本原因の再発時は前回の恒久化結果を参照して本工程を再実施する。

- 既存スキル・エージェント定義・規範文書の改修を指示された場合、改修内容自体が今後の同種改修時に
  適用可能な恒常規範として抽出できないかを本工程の必須点検項目に含める
- 恒久化する規範に数値・閾値・境界値が含まれる場合、規定本文と同段落に判断根拠
  （当該数値の由来・境界を分ける理由・トレードオフ）を併記する。
  欠くと後続セッションで解釈揺れが発生し、session-reviewやtrim-agent-docsで根拠情報が形骸化・喪失するリスクがある
