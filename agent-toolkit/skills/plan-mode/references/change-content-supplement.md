# 変更内容の補足記述パターン

本ファイルは`plan-file-guidelines.md`「変更内容（`## 変更内容`）」節から分離した補足記述パターンの詳細である。

複雑な既存モジュール（既存API関数の拡張・UIコンポーネントのProps拡張等）を対象とする場合、
詳細スケッチを埋め込む代わりに「目的・対象・設計要件・実装者がReadすべき参照コード」の各要素で記述してよい。
詳細スケッチは新規ファイルや純粋機械置換のように既存契約への依存が薄い場合に限る。
いかなる理由（例: 記述量・工数・ファイル規模の見積り）があっても、既存契約への依存の濃淡以外を
適用可否の判断入力に用いない。計画ファイル自立性原則が本許容規定に優越し、代替記述を選ぶ場合も
実装者が再調査せず着手できる情報量を満たす。

他ファイル文言との「完全一致」「同一表現」宣言では、原典パス・引用元節を明示し、原文`text`ブロック直後に追記文言案を配置する。

利用者が観測する挙動（出力・終了コード・エラー経路等）を変更する差分を含める場合、当該経路の検証テストの変更または追加を同一計画の対象へ対で含める。
含めない場合は判断根拠を`### エージェント判断`へ記録する。
テスト改修を対象ファイル一覧に含む計画では、CLI名・関数名・パラメーター名等の識別子リネームがテスト側の構造変更を要する箇所を計画作成前に洗い出す。
対象となる構造変更はパラメトリックテストのデコレーター分割・引数タプルの列数変更・fixture再構成等を含む。
当該の構造変更方針（採用案・呼び出し側の分岐方法）を対応するテストファイルの個別H3配下へ明記する。
適用対象は単純な識別子置換だけでは実装が成立しないテスト構造変更全般とする。

例外形式として同一パターン置換時の集約H3（`### 置換パターン: <旧> → <新>（対象: <sublist>）`）と`quality-sweep`配下計画の分担バッチ単位列挙を許容する。
集約H3の適用条件は同一の旧文言から新文言への置換が3ファイル以上に及ぶ場合とし、文脈依存の変更を伴うファイルは個別差分ブロックへ切り出す。

例外形式としてversion bump対象manifest H3のtext/diff fence省略を許容する。
対象は`_plan_format.py`の`BUMP_MANIFEST_PATHS`定数が示すファイル集合とする。
`scripts/agent_toolkit_bump.py`実行時に`version`値が自動更新され事前確定できないため、
H3配下は自動適用対象である旨と参照案内文（bump種別・対象キー）を記載する。
免除の適用条件は、計画ファイルの`## 実行方法`本文に`agent_toolkit_bump.py`実行への言及がある場合に限る
（機械チェックの判定条件と同一）。言及が無い場合は免除されず、通常どおりtext/diffコードブロックを記載する。
`scripts/agent_toolkit_bump.py`のリテラルとのSSOT一致は`scripts/agent_toolkit_bump_test.py`が検証する。

配下にバッククォート囲みで削除・改名する識別子（スキル名・パス・関数名・変数名を含む）を1行ずつ列挙する。
機械可読形式は`` - `<identifier>` ``形式のリスト項目とする。
