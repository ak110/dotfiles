# 計画ファイル本文の記述義務と書き込み後チェック

`plan-file-guidelines.md`「計画ファイル全体の遵守事項」節から参照される、
`Write`前の記述義務と書き込み後の機械検査手順。

## text コードブロック化義務

- `## 調査結果`・`## 変更内容`配下の追加文面案・反映文面案など
  他文書本文を転記する長文段落（120字を超え得る段落）は`text`コードブロックで包んで記述する。
  textlintのsentence-length違反を`Write`段階で回避する
  - `## 調査結果`配下の既存docstring要約・既存関数の説明文・既存規範文の引用などの長文段落も対象に含める
    （行数・件数等の単純な箇条書きの確定値は対象外）
- 計画本文で機械検査の検出パターン代表例・検出対象語彙・禁止語彙を記述する場合、
  地の文ではなく必ず`text`コードブロック内に配置する（走査対象と代表例が同一節で自己抵触するリスクを事前予防する）
  - `text`コードブロック内で機械検出対象カテゴリ（縮退フレーズ・colloquial等）と語彙が交差するテスト入力の
    具体的な文字列を埋め込まない。代替として命名対応・入力の性質・アサーション意図の抽象記述に留める。
    具体的な入力文字列は実装フェーズで隔離フィクスチャ
    （例: `_scope_escalation_test_inputs.txt`）へ配置しテストコードは同フィクスチャを読み込む
- 新規規範または新規機械検査の導入計画では、計画ファイル本文自身へ先行適用してから`Write`へ進み、
  対象該当時は対象外形式（コードフェンス内転記・抽象表現での参照など）へ事前変換する（規範導入計画は初回適用事例）

## 書き込み後チェック

計画ファイル本文を正規パスへ書き込んだ直後に次のコマンドを計画ファイルへ実行する。

```sh
uv run --script ${CLAUDE_PLUGIN_ROOT}/skills/plan-mode/scripts/check_plan_file.py <計画ファイル>
```

統合ランナーは書き込み後チェックの全検査（検査項目は`check_plan_file.py`のdocstringを典拠とする）を
1プロセスで実行する（`plan-file-creator`の整合性チェックの代替としない）。

- 検出違反への対処は自律判断とする。実装時に成果物へ転記する文面（`## 変更内容`配下のコードブロック）の
  違反は極力解消し、それ以外の計画本文に対する体裁系違反への対処は任意とする
  - 詳細は`agent-toolkit/skills/review-standards/SKILL.md`「計画ファイル文脈での例外」節を参照する
- `## 背景`配下のユーザー提示素材原文に起因する違反は対処不要とする
- 自動修正が適用された場合は計画ファイルを`Read`で再読込してから後続編集へ進む
- `#### 縮減対象`H4節配下に削除文言案の`text`コードブロックが存在するかを検査する（欠落時はエラー）

## 起草前準備・書き込み後チェック再実行・絶対数値直書き回避・参照先の自リポジトリ内完結

- 計画ファイル本体の起草者は起草前準備と起草直後の機械チェックを実施する（メインエージェント・サブエージェント問わず）
  - 起草前に`agent-toolkit/skills/writing-standards/references/textlint-violations.md`「頻出違反パターン予防策」節を`Read`で読み込む
  - 差分ラベルを含む計画では`plan-file-diff-labels.md`「フェンス配置」節と`plan-file-diff-labels.md`「差分ラベル6種」節も`Read`で読み込む
  - 起草直後に統合ランナー`agent-toolkit/skills/plan-mode/scripts/check_plan_file.py`を実行する
  - 続けて`uvx pyfltr run-for-agent --no-fix <対象>`のtextlint検査を実施する（事前に別途fixステージを実行済みとする）
  - サブエージェント委譲時は当該規定を委譲プロンプトへ引用転記する
- 検証は自動実行を最優先し、手動検証は自動化困難な場合に限り理由を明記する（検証結果が想定と異なる場合は「環境差異」等と解釈せず進捗ログに乖離を記録して前提を見直す）
- 計画本文を編集した契機ごとに本ファイル「書き込み後チェック」節を再実行する
- 計画ファイル本文で対象位置・対象量を示す際、改訂で変動する絶対数値（行番号・行範囲・行数・件数・version番号）の直書きを避ける。
  節名・小見出し・冒頭文引用などの相対参照で特定する（対象は本文位置に加え他ファイル参照箇所の行番号・行範囲・行数も含む）
  - 機械検査対象は行番号の参照全般。`wc -l`実測値は`## 調査結果`配下で自由記載可（`<!-- line-ref-ok -->`で抑止、他節は違反）
  - version番号は`x.y.z`形式の具体値を直書きしない（並行セッションのbumpによる陳腐化を回避）
  - bumpステップは「実装時点の現行値からのMINOR/PATCH遷移」のように種別のみを記載する
- 計画ファイル本文の記述量に不安を感じた場合は「長大な計画ファイルの段階的記述」節の正規手順に従う
- 計画本文の参照先は自リポジトリ内で完結させ、`/tmp/`・scratchpad・他セッションパスを書かず削除文言案・計算結果は本文へ記載する（6-1節の`wc -l`用途は対象外）

## 長大な計画ファイルの段階的記述

計画段階の記述省略は判断コストを実装工程へ転嫁し増幅させる。
計画段階は全調査結果が文脈内に揃う唯一の時点であり、詳細文面の全量記述が総コスト最小となる。
この目的のため、いかなる理由（例: 計画ファイル本文の記述量・工数の見積り）があっても、
詳細文面を要旨・参照のみへ代替する判断をしない。不安を感じた場合は次の正規手順を選ぶ。

- 長大な計画ファイルは節単位・変更対象ファイル単位で複数回の`Write`・`Edit`に分けて段階的に書き進める。
  1回の出力で全体を書き終える必要はない
- 変更対象ファイル集合が分離可能で並列化の実効性がある場合は、同一セッション内で複数の計画ファイルへ分割する。
  判定基準は`agent-toolkit/rules/02-collaboration.md`「実装手順の詳細」項に従う。
  本分割は同一セッション内の並行処理として扱う。
  `01-agent.md`「完遂原則」項が禁止する別セッションへの先送りには該当しない
- 記述量の見積りは着手可否の判断入力に用いない。詳細を書き進めた結果として計画ファイルが長くなる状態は正常な成果物である
  （代替記述を選べる条件は`change-content-supplement.md`冒頭の規定に限定する）
