# 変更内容セクションの差分ラベルとフェンス配置

`plan-file-guidelines.md`「変更内容」節の詳細規定。

## 差分ラベル6種

コーディングエージェント向け文書の改訂タスクでは、置換・追記・削除のいずれかを明示し、
置換は対比形式、追記は最終文面、削除は現行文言と削除根拠をそれぞれ`text`コードブロックへ記載する。
差分ラベルは次の6種類とする。
`[現行]`は対比形式（置換前提示）と削除形式（削除対象提示）の双方で共用する。

- 既存節間挿入（置換前）: `[現行]`
- 既存節間挿入（置換後）: `[置換後]`
- 既存節への追加（隣接文言との対比を伴わない場合）: `[追記]`
- 新設ファイル全文: `[新設]`単独
- 既存ファイル全文置換: `[置換後（全文）]`
- 削除: `[現行]`と`[削除根拠]`の組

`[追記]`ラベルの使用条件を次のとおり定める。
`[追記]`ラベルは新規節新設・複数バレット追加・隣接文言に依存しない既存節追加のような、現行位置に対比可能な既存文言が無い場合の書式である。
追記量が既存節の隣接文言と1対1対応する形で表現できる場合は`[現行]/[置換後]`対比形式を必須とする。
行数収支主張（「追記N行、縮減M行」等）は計画本文へ書かない（計画時の行数厳密照合を求めないため）。

`[追記×N]`ラベルは`[追記]`の派生形で、同一の追記文面を1つの対象ファイル内のN箇所へ同時配置する場合に用いる。
主な用途は冒頭frontmatter同期注記が同一ファイル内の複数節を同期対象として宣言している場合で、
当該ファイルの各配置箇所へ同一文面を機械転記するときに使う。他ファイルへの同期が必要な場合は
`### <ファイルパス>`H3ごとに個別の`[追記]`ブロックを配置し、`[追記×N]`とは併用しない。
Nは1以上の整数で、`[追記×2]`・`[追記×3]`のように書く。
集計時は`text`フェンス本文の行数×Nを追記量として`check_wc_projection.py`が加算する。
`[追記×1]`は`[追記]`と同等（デフォルトN=1）で、通常は`[追記]`単独形式を用いる。
`×N`修飾子は本体ラベル`[追記]`専用とし、frontmatter対象の複数箇所配置は`[追記（frontmatter）]`単独形式で表す
（書式解釈の曖昧化を防ぐため）。

`[現行]`ブロック本文は対象ファイルを直接`Read`した直後の実測内容のみを転記する。
`Explore`等のサブエージェント調査結果からの間接転記は採用しない。
実測起源義務は本規範文で担保する。
なお対象ファイル本文との一意一致（不一致・複数箇所マッチの検出）は`check_wc_projection.py`が事後検査する。
対象ファイルが既に本計画の追記変更を含む状態（実装先行・他セッション残置・作業ツリー未確定変更等）の場合、`[現行]`は`Read`実測ではなく追記前状態を転記する。
実測内容が本計画の`[置換後]`と同一になる場合は必ず本手順で追記前状態を復元する（`[現行]`と`[置換後]`が完全同一の対比ブロックは差分不成立となるため）。
同一検知は`check_plan_diff_gates.py`が事後検査する。
追記前状態の取得手順は、追記変更がHEADへ未到達か既到達かで次の2通りに分岐する（相互排他）。
作業ツリーのみに未コミットの追記変更が残る場合（HEAD未到達）、`git show HEAD:<path>`で追記前状態が直接取得できる
（HEAD自体は追記を含まないため）。
HEADが本計画の追記変更を既にコミット済みで含む場合（HEAD到達済み）、`git log --oneline`で当該追記コミットを特定し、
`git show <その1つ前のSHA>:<path>`で追記前状態を取得する。

## frontmatter変更用サブラベル

frontmatter部分（`^---$`と`^---$`区間内）の変更を対象とするサブラベルを次の4種類とする。

- 既存frontmatter追記: `[追記（frontmatter）]`
- 既存frontmatter置換前（対比形式）: `[現行（frontmatter）]`
- 既存frontmatter置換後: `[置換後（frontmatter）]`
- 既存frontmatter削除根拠: `[削除根拠（frontmatter）]`

対比形式は`[現行（frontmatter）]`/`[置換後（frontmatter）]`ペアで書く。
記述位置は通常のラベルと同様に`### <ファイルパス>`H3配下の`text`フェンス直後。
担当スクリプトは「機械適用対象」節を参照する。
`[追記（frontmatter）]`は既存frontmatterを持つファイル専用であり、frontmatter未保有ファイルへの新設は
冒頭見出し行を`[現行]`、frontmatterブロック＋見出し行を`[置換後]`とする通常ペアで記述する。

## フェンス配置

各ラベルは`text`フェンス直後1行目にプレーンテキストで配置する（fence内側形式）。
textlint jtf-style/4.3.2大かっこのauto-fixが半角`[]`を全角`［］`へ変換し検査と衝突するため、fence外側配置は禁止する。
対比形式は隣接見出し行を`[現行]`、隣接見出し＋新規小節を`[置換後]`とし1回一致置換を可能にする。
`[現行]`と`[置換後]`は各自の3バッククォート開閉ペアを持つ独立した`text`フェンスへ配置する。
両ラベルを同一フェンス内へ併記した場合は`_leading_label`（`check_wc_projection.py`）が先頭ラベル種別のみを認識して置換後未検出となるため、フェンス分割を必須とする。
構造化ファイル（JSON・YAML等）では親要素と開閉ブラケットを含める。
対象ファイルが`.py`等のPythonコードの場合、差分ラベル方式（`[現行]/[置換後]/[削除根拠]`）で書き、
差分ラベル外の平文フェンス（言語指定`python`または言語指定なしのフェンス）は避ける。
textlintがフェンス内のPython行を日本語文として誤検出し、`sentence-length`等の違反を報告するため。

## 機械適用対象

- `[現行]`/`[置換後]`対比と`[削除根拠]`は`check_wc_projection.py`が検査する
- `[追記]`は`check_wc_projection.py`と`check_plan_diff_gates.py`の直接検出対象とする
  （ラベル行は集計から除外し、フェンス本文をcolloquial-check・textlint検査へ渡す）
- `[追記×N]`は`[追記]`と同一の担当スクリプト2種で処理する
  - `check_wc_projection.py`が`×N`修飾子を抽出し集計値をN倍加算する
  - `check_plan_diff_gates.py`（実装は`_plan_diff_gates_scan.py`）も`×N`修飾子込みでラベル判定し、
    `[追記]`と同等の縮退フレーズ検出・textlint併走検査を適用する（集計は行わない）
- `[新設]`・`[置換後]`・`[置換後（全文）]`は`check_plan_diff_gates.py`が検査する
- 差分ラベル全種と派生形`[追記×N]`、frontmatterサブラベル4種のfence外側配置は`check_plan_diff_gates.py`が検査する
  - fence配置ミスをtextlint jtf-style/4.3.2大かっこ違反の間接検出に委ねる状態を回避し、起草直後に直接検出する
- `[現行]`/`[置換後]`の同一fence内側配置は`check_plan_diff_gates.py`が検査する
  - `_check_inner_label_coexistence`関数。fence外側配置検査と同層
  - 対比ペア併記検出専用の機能であり他ラベルは対象外とする
- 検出対象はラベル文言単独行に加え、バッククォート囲み形式および直後に注記（半角/全角括弧・コロン）が続く形式も含む
- 全角化ラベル（textlint autofixで閉じ括弧が全角化された`[現行］`等、上記全ラベルに対応）のfence外側出現も同スクリプトが検査する
- 各ラベル付きブロックは`### <ファイルパス>`H3配下へ配置する
- frontmatter変更用サブラベル（`[追記（frontmatter）]`・`[現行（frontmatter）]`・
  `[置換後（frontmatter）]`・`[削除根拠（frontmatter）]`）は既存本体ラベルと同一の
  担当スクリプトで処理する（`check_wc_projection.py`と`check_plan_diff_gates.py`が対応）

## 記載範囲

記載する最終文面には既存の並列列挙要素・既存参照リンク・既存の代表例列挙を漏れなく含める。
`## 記載範囲`の規定をSSOTとする。
`## 変更履歴`側の対象ファイルパス・節名アンカーは同規定と一致させる
（`pretooluse.py`の`_check_plan_file_history_content_sync`が機械検査する）。

## 単体テストファイル追加

公開関数の追加・変更を含む計画では対応する単体テストファイルへの追加・修正項目を含め、
同値分割・境界値分析および正規表現の選択肢分岐（`|`）の各分岐でケースを抽出する。

## 進捗ログの乖離注記

`## 進捗ログ`配下の`### <時系列ラベル>`
（`### 着手`・`### 検証完了`・`### コミット`・`### レビュー反映`等）配下で用いる。
実装工程で発生した随時是正が計画本文（`## 変更内容`・`### エージェント判断`等）と乖離した箇所は、
`[乖離]`サブバレット形式で記録する。
1件1行で`<箇所>（実装確定形: <確定形>）`形式とする。

乖離集約のみで対応するか、`## 変更内容`ブロック自体を実装確定形へ更新するかは、
乖離規模と再利用可能性で判断する。
局所的な文言縮減・軽微な整形など再利用可能性が低い乖離はサブバレット記録のみで完結してよい。
複数箇所への波及・後続計画で再参照される確定形・仕様面の変更を伴う乖離は
`## 変更内容`ブロック本文も同時更新する。

`## 進捗ログの乖離注記`は`## 差分ラベル6種`（fence直後配置）とは配置文脈が異なる（箇条書きサブバレット用途）。
`plan-implementer`完了報告の`changed`欄サブバレットで用いる`[乖離]`ラベル
（`launch-prompts-drafting.md`「起草・改訂委譲雛形」節の`## 完了報告要件`欄）と`## 進捗ログの乖離注記`の書式は共通する。
