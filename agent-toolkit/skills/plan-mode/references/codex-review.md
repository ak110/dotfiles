# 計画ファイルのcodexレビュー

## 実行経路

MCPツールが利用可能な環境では常時MCP版を使う。
CLI版はMCPが利用不可な環境専用のフォールバックとする。
CLI実行が失敗した場合は再試行可否のユーザー確認で判断停止せず、MCP利用可能性を再点検したうえで利用可能なら経路を切り替えて続行する。
本方針はClaude Code環境と他のコーディングエージェント環境の双方に共通して適用する。

- Claude Code環境: Skillツールで`agent-toolkit:plan-codex-review`を起動して委譲する
  - `args`に計画ファイルの絶対パス・プロジェクトルートの絶対パス・初回か継続かの別・
    継続時は`threadId`（MCP版）または`SESSION_ID`（CLI版）を渡す
- 他のコーディングエージェント環境（Skillツールが利用できない場合）:
  `plan-codex-review`のSKILL.mdを読み込み、記載のテンプレートに従いMCP版を直接実行する（MCP不在環境のみCLI版へフォールバックする）
  - `${CLAUDE_SKILL_DIR}`は置換されないため、SKILL.mdの所在ディレクトリへ読み替える

## MCPもCLIも利用できない場合

計画ファイル提示前にユーザーに報告し、以下の選択を仰ぐ。

- codex CLIを別途セットアップし、codex MCPをインストールしてやり直す（推奨）
  - インストールコマンド: `claude mcp add --scope=user codex codex mcp-server`
- codex CLIを別途セットアップしてやり直す
- 当該環境ではcodexレビューを永続的にスキップする旨をプロジェクトのローカルメモファイルに記載してスキップする

## codexレビューの進め方

- codexの指摘にエージェント単独では採否を判断できないものが含まれる場合、その場でユーザーに確認して結論を確定させる
- codexの指摘がなくなるまで更新とレビューを繰り返す（指摘が的外れなものしかない場合は合格扱いでよい）
  - 規範違反主張を伴う指摘は実機検証で規範違反の成立を確認し、規範違反が成立する場合は対応する。
    成立しない場合は指摘内容を計画ファイルの`### 却下した代替案`に却下理由と実機検証結果を明記したうえで却下する
  - 規範違反が成立する場合でも、字面適用が次の発生条件のいずれかに該当する場合は本質要件を満たす代替修正手段の有無を検討する
    - 計画ファイルの肥大化・自立性の阻害・可読性の低下
  - 代替修正手段が成立する場合は当該代替修正を採用する
  - 採用時は本質要件との対比・採用した代替手段・実機検証結果を計画ファイルの`### 却下した代替案`へ明記する
- 重大指摘の反映後は修正規模・反映範囲・観測事象の有無を問わず継続codexレビューを実施する。
  指摘が計画ファイル粒度（試算値の精度・サンプル整合性等）であっても省略しない
- 一度指摘ゼロになった後の修正でも、修正がある限り再レビューを実施する
- codex指摘1件が複数セクションに伝播する場合、関連セクションすべてを網羅的に更新する
- 計画ファイルを全面改訂した場合、過去のcodexセッション（MCPの`threadId`またはCLIの`SESSION_ID`）を破棄し、
  新規セッションで初回プロンプトからレビューをやり直す。
  旧セッションには改訂前の指摘文脈が残り、新方針と整合しない誤指摘の原因になるためである
- 計画文面のコードブロックに実装案コードが含まれる場合、
  同一モジュール内の既存類似関数の実装方式と整合するかを観点に含める旨を初回プロンプトに記載する

## Windows環境での注意

codexはPowerShell経由でファイルを読み書きする際、デフォルトでShift-JISが使われ日本語が文字化けする。
対処としてプロンプトに以下を追記する。
「ファイルの読み書きはUTF-8エンコーディングを明示すること（例: `Get-Content -Encoding UTF8`）」
