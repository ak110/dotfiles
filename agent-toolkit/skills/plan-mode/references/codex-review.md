# 計画ファイルのcodexレビュー

## 実行経路

MCPツールが利用可能な環境では常時MCP版を使う。
CLI版はMCPが利用不可な環境専用のフォールバックとする。
CLI実行が失敗した場合は再試行可否のユーザー確認で判断停止せず、
MCP利用可能性を再点検したうえで利用可能なら経路を切り替えて続行する。
本方針はClaude Code環境と他のコーディングエージェント環境の双方に共通して適用する。

- Claude Code環境: Agentツールで`subagent_type: agent-toolkit:plan-codex-reviewer`を起動して委譲する
  - プロンプト本文に計画ファイルの絶対パス・プロジェクトルートの絶対パス・初回か継続かの別を渡す。
    継続時は`threadId`（MCP版）または`SESSION_ID`（CLI版）を渡す。
    追加のレビュー観点（サブエージェント定義との整合等、必要時のみ付与、任意）は初回・継続いずれでも渡してよい
  - サブエージェント経由のCLI版がauto mode下でブロックされた場合、
    メインが`mcp__codex__codex`（初回、新規`threadId`が発行される）と
    `mcp__codex__codex-reply`（継続、同`threadId`を使う）を直接呼び出して継続する
- 他のコーディングエージェント環境（Agentツールが利用できない場合）:
  `agent-toolkit/agents/plan-codex-reviewer.md`を読み込み、記載のテンプレートに従いMCP版を直接実行する
  （MCP不在環境のみCLI版へフォールバックする）

## MCPもCLIも利用できない場合

計画ファイル提示前にユーザーに報告し、以下の選択を仰ぐ。

- codex CLIを別途セットアップし、codex MCPをインストールしてやり直す（推奨）
  - インストールコマンド: `claude mcp add --scope=user codex codex mcp-server`
- codex CLIを別途セットアップしてやり直す
- 当該環境ではcodexレビューを永続的にスキップする旨をプロジェクトのローカルメモファイルに記載してスキップする

## codexレビューの進め方

- codexの指摘にエージェント単独では採否を判断できないものが含まれる場合、
  確認が必要と判断した時点でユーザーに確認して結論を確定させる
- 継続呼び出しは初回レビュー1回と、重大以上（`致命的`・`重大`）の指摘を反映した場合の
  反映確認レビュー1回までとする。初回レビューで重大以上の指摘が無い場合は反映確認レビューを実施しない
  - 反映確認レビューの対象は計画ファイル本文への初回指摘反映であり、実装コード・対象ファイル現状の状態は評価対象外である
    （呼び出し元プロンプトへ本旨を明示的に含める）
  - 反映確認レビューで新たな重大以上の指摘が残る場合も追加の継続レビューは行わない。
    メイン側で採否を確定し、計画本文への反映または明示的な不対応判定の記録を完了させてから
    工程7完了と扱う（`SKILL.md`工程7節の完了条件と同一基準とする）
  - 規範違反主張を伴う指摘は実機検証で規範違反の成立を確認し、成立しない場合は
    `### 却下した代替案`へ却下理由と実機検証結果を明記して却下する
  - 規範違反が成立する場合でも、計画ファイルの肥大化・自立性の阻害・可読性の低下のいずれかに
    該当する字面適用は本質要件を満たす代替修正手段を検討する。
    採用時は本質要件との対比・採用手段・実機検証結果を`### 却下した代替案`へ明記する
- codex指摘1件が複数セクションに伝播する場合、関連セクションすべてを網羅的に更新する。
  `## 変更履歴`への追記だけで完了扱いにせず`## 変更内容`本文への転記を同一ターン内で完遂する。
  転記漏れは計画ファイル自立性違反にあたる
- 計画ファイルを全面改訂した場合、過去のcodexセッション（MCPの`threadId`またはCLIの`SESSION_ID`）を破棄し、
  新規セッションで初回プロンプトからレビューをやり直す
- 計画文面のコードブロックに実装案コードが含まれる場合、
  同一モジュール内の既存類似関数の実装方式と整合するかを観点に含める旨を初回プロンプトに記載する
- 実装案コードが文字列走査・入力ストリーム処理・状態機械・トークン切り出しなど境界条件が
  挙動を変える処理を含む場合、境界条件の網羅観点を初回codexプロンプトへ含める。
  網羅観点は空入力・単一行入力・改行文字有無・EOF終端・未閉じ構造・区切り不在時の残余処理を含む
- 時刻・日付・タイムゾーンを扱う実装（cron・スケジュール・タイマー・UNTIL/DTSTART・rrule等）の
  計画レビューでは、次の投入観点を初回プロンプトへ含める
  - 変換関数の入出力タイムゾーン契約（UTC／ローカル／特定TZID）
  - `new Date().getFullYear/Month/Date()`系のブラウザローカルTZ依存の混入
  - `Date.UTC()`と`Date.now()`の意味的違い
  - 外部ライブラリのDate表現規約（rruleのUTC-markedローカル時刻等）

## Windows環境での注意

codexはPowerShell経由でファイルを読み書きする際、デフォルトでShift-JISが使われ日本語が文字化けする。
対処としてプロンプトに以下を追記する。
「ファイルの読み書きはUTF-8エンコーディングを明示すること（例: `Get-Content -Encoding UTF8`）」
