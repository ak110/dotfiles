# 計画ファイルのcodexレビュー

## 実行経路

MCPツールが利用可能な環境では常時MCP版を使う。
CLI版はMCPが利用不可な環境専用のフォールバックとする。
CLI実行が失敗した場合は再試行可否のユーザー確認で判断停止せず、
MCP利用可能性を再点検したうえで利用可能なら経路を切り替えて続行する。
本方針はClaude Code環境と他のコーディングエージェント環境の双方に共通して適用する。

- Claude Code環境: Skillツールで`agent-toolkit:plan-codex-review`を起動して委譲する
  - `args`に計画ファイルの絶対パス・プロジェクトルートの絶対パス・初回か継続かの別・
    継続時は`threadId`（MCP版）または`SESSION_ID`（CLI版）を渡す
- 他のコーディングエージェント環境（Skillツールが利用できない場合）:
  `plan-codex-review`のSKILL.mdを読み込み、記載のテンプレートに従いMCP版を直接実行する
 （MCP不在環境のみCLI版へフォールバックする）
  - `${CLAUDE_SKILL_DIR}`は置換されないため、SKILL.mdの所在ディレクトリへ読み替える

## MCPもCLIも利用できない場合

計画ファイル提示前にユーザーに報告し、以下の選択を仰ぐ。

- codex CLIを別途セットアップし、codex MCPをインストールしてやり直す（推奨）
  - インストールコマンド: `claude mcp add --scope=user codex codex mcp-server`
- codex CLIを別途セットアップしてやり直す
- 当該環境ではcodexレビューを永続的にスキップする旨をプロジェクトのローカルメモファイルに記載してスキップする

## codexレビューの進め方

- codexの指摘にエージェント単独では採否を判断できないものが含まれる場合、
  確認が必要と判断した時点でユーザーに確認して結論を確定させる
- codexの指摘がなくなるまで更新とレビューを繰り返す（指摘が的外れなものしかない場合は合格扱いでよい）
  - 規範違反主張を伴う指摘は実機検証で規範違反の成立を確認し、規範違反が成立する場合は対応する。
    成立しない場合は指摘内容を計画ファイルの`### 却下した代替案`に却下理由と実機検証結果を明記したうえで却下する
  - 規範違反が成立する場合でも、字面適用が次の発生条件のいずれかに該当する場合は本質要件を満たす代替修正手段の有無を検討する
    - 計画ファイルの肥大化・自立性の阻害・可読性の低下
  - 代替修正手段が成立する場合は当該代替修正を採用する
  - 採用時は本質要件との対比・採用した代替手段・実機検証結果を計画ファイルの`### 却下した代替案`へ明記する
- 重大指摘の反映後は修正規模・反映範囲・観測事象の有無を問わず継続codexレビューを実施する。
  指摘が計画ファイル粒度（試算値の精度・サンプル整合性等）であっても省略しない
  - 継続codexレビューで指摘0件を確認するまで、plan-implementer委譲・実装工程移行を発行しない。
    auto mode下・単独foreground委譲下でも本規範は省略しない
- 一度指摘ゼロになった後の修正でも、修正がある限り再レビューを実施する
- 3ラウンド連続で新規の重大指摘が出る場合の扱い
  - 計画ファイルの基本設計に根本的な問題がある可能性として扱う
  - 計画方針自体の再検討（テンプレート再選定・スコープ再分割・アプローチ転換）を実施する
  - auto mode下でも計画方針の再検討を省略せず、必ず実施する
  - 再検討せず4ラウンド目に進むことは禁ずる
  - 「新規」の判定基準: 前ラウンドの指摘集合に同等観点が含まれていなかった指摘のみを新規とし、
    既往指摘の派生・展開は新規に数えない
  - 判定は指摘性質を区別する
    - 「設計判断・方針転換の対象」（アプローチ選択の誤り・要件解釈の齟齬・
      アーキテクチャレベルの不整合など）: 計画方針の再検討を必要とする
    - 「実装完成度の反復」（規範記述の網羅性・記述整合性・連動先追随・軽微な用語揺れの精緻化など）:
      指摘収束を優先しサイクル継続でよい
  - いずれと判断したかの根拠を計画ファイル本文または`## 進捗ログ`へ1行以上残す
  - 新規重大指摘が同一ドメイン（同一節・同一機能・同一概念モデル）内で観点を変えながら連発する場合の扱い。
    指摘ごとに個別対応せず、元フィードバック原文または元計画方針の設計判断まで遡って再検討する。
    同ドメイン多サイクル発散の兆候例は次のとおり。
    判定ラベル・分類区分の意味論不整合、列定義・データモデルの自己参照性、
    規範記述の入れ子階層の粒度不揃い、既存語彙との重複定義。
    これらが2サイクル以上で連発した時点で個別修正を停止し、設計判断へ遡及する
  - 発散判定発動を機械化する補助チェックリスト
    （既存の2サイクル発散判定を狭めない。
    既存判定が発動する場面をチェックリスト3条件不成立で除外しない）。
    ここでの「細部プロトコル」は既述「実装完成度の反復」に含まれる
    対象のうち、数値パラメーター・列挙条件・API呼び出し粒度など
    プロトコル仕様の精緻化を指す
    - 直前2ラウンドの重大指摘が全て同一節・
      同一概念モデル・同一機能内に留まっているか
    - 直前2ラウンドの重大指摘が細部プロトコルの精緻化に集中しているか
    - フィードバック原文の主旨が細部プロトコル指定を求めていないか
- codex指摘1件が複数セクションに伝播する場合、関連セクションすべてを網羅的に更新する
  - 重大以上の指摘反映は`## 変更履歴`への追記だけで完了扱いにせず、
    `## 変更内容`本文（対象ファイル一覧・変更方針の見出し・差分ブロック）への転記を同一ターン内で完遂する。
    転記完了前にplan-implementer委譲・実装工程移行を発行しない。
    転記漏れは`## 変更履歴`と`## 変更内容`の同期不備として計画ファイル自立性違反にあたる
- 計画ファイルを全面改訂した場合、過去のcodexセッション（MCPの`threadId`またはCLIの`SESSION_ID`）を破棄し、
  新規セッションで初回プロンプトからレビューをやり直す。
  旧セッションには改訂前の指摘文脈が残り、新方針と整合しない誤指摘の原因になるためである
- 計画文面のコードブロックに実装案コードが含まれる場合、
  同一モジュール内の既存類似関数の実装方式と整合するかを観点に含める旨を初回プロンプトに記載する

## Windows環境での注意

codexはPowerShell経由でファイルを読み書きする際、デフォルトでShift-JISが使われ日本語が文字化けする。
対処としてプロンプトに以下を追記する。
「ファイルの読み書きはUTF-8エンコーディングを明示すること（例: `Get-Content -Encoding UTF8`）」
