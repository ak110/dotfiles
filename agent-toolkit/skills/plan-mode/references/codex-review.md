# 計画ファイルのcodexレビュー

## 実行経路

MCPツールが利用可能な環境では常時MCP版を使う。
CLI版はMCPが利用不可な環境専用のフォールバックとする。
CLI実行が失敗した場合は再試行可否のユーザー確認で判断停止せず、
MCP利用可能性を再点検したうえで利用可能なら経路を切り替えて続行する。
本方針はClaude Code環境と他のコーディングエージェント環境の双方に共通して適用する。

- Claude Code環境: Agentツールで`subagent_type: agent-toolkit:plan-codex-delegate`
  （`用途: 計画レビュー`）を起動して委譲する
  - プロンプト本文に計画ファイルの絶対パス・プロジェクトルートの絶対パス・初回か継続かの別を渡す。
    継続時は`threadId`（MCP版）または`SESSION_ID`（CLI版）を渡す。
    追加のレビュー観点（サブエージェント定義との整合等、必要時のみ付与、任意）は初回・継続いずれでも渡してよい
  - 起動時は`model`パラメーターを省略し、エージェント定義frontmatterの`model`指定に従う。
    レビュー推論はcodex側が担い、本エージェントの処理はcodexへの委譲と応答転記が中心であるため、
    タスク種別に基づくモデル選択基準の「レビュー」区分を適用しない
  - サブエージェント経由のCLI版がauto mode下でブロックされた場合、
    メインが`mcp__codex__codex`（初回、新規`threadId`が発行される）と
    `mcp__codex__codex-reply`（継続、同`threadId`を使う）を直接呼び出して継続する
- 他のコーディングエージェント環境（Agentツールが利用できない場合）:
  `agent-toolkit/agents/plan-codex-delegate.md`を読み込み、記載のテンプレートに従いMCP版を直接実行する
  （MCP不在環境のみCLI版へフォールバックする）

## codex利用可否の3段階判定

計画レビュー・実装レビュー双方で共通する判定手順とする。両者の呼び出し元は本節を参照し、
個別ファイルで同旨の条件文を重複記述しない（プロンプト共通化の一環）。

- 段階1（MCP優先）: `mcp__codex__codex`・`mcp__codex__codex-reply`が呼び出し可能な環境。
  判定は`plan-codex-delegate`起動時にツール解決エラーが発生しないことで確認する
- 段階2（CLIフォールバック）: 段階1が不可でcodex CLIが利用可能な環境。
  判定は`command -v codex`の終了コード0とする
- 段階3（claudeフォールバック）: 段階1・段階2がいずれも不可の環境。
  「MCPもCLIも利用できない場合」節でユーザーが
  「当該環境ではcodexレビューを永続的にスキップする」を選択した場合に限り遷移する。
  この場合、計画レビューは`plan-reviewer`、実装レビューは`plan-impl-reviewer`を代替として起動する

段階3への遷移はユーザー選択を経由する。`plan-codex-delegate`起動がauto mode下でブロックされた場合
（段階1・段階2は技術的に利用可能）は段階3に該当せず、「plan-file-creatorからの起動」節の
`needs_escalation`経路に従う。

## MCPもCLIも利用できない場合

計画ファイル提示前にユーザーに報告し、以下の選択を仰ぐ。

- codex CLIを別途セットアップし、codex MCPをインストールしてやり直す（推奨）
  - インストールコマンド: `claude mcp add --scope=user codex codex mcp-server`
- codex CLIを別途セットアップしてやり直す
- 当該環境ではcodexレビューを永続的にスキップする旨をプロジェクトのローカルメモファイルに記載してスキップする。
  この場合「codex利用可否の3段階判定」節の段階3に遷移する
  （以降の代替起動先は「codex利用可否の3段階判定」節の段階3の記述に従う）

## codexレビューの進め方

- codexの指摘にエージェント単独では採否を判断できないものが含まれる場合、
  確認が必要と判断した時点でユーザーに確認して結論を確定させる
- 継続呼び出しは初回レビュー1回と、重大以上（`致命的`・`重大`）の指摘を反映した場合の
  反映確認レビュー1回までとする。初回レビューで重大以上の指摘が無い場合は反映確認レビューを実施しない
  - 反映確認レビューの対象は計画ファイル本文への初回指摘反映であり、実装コード・対象ファイル現状の状態は評価対象外である
    （呼び出し元プロンプトへ本旨を明示的に含める）
  - 反映確認レビューで新たな重大以上の指摘が残る場合も追加の継続レビューは行わない。
    メイン側で採否を確定し、計画本文への反映または明示的な不対応判定の記録を完了させる。
    完了後は整合性チェック完了と扱う（`agent-toolkit/agents/plan-file-creator.md`「進め方」節の完了条件と同一基準とする）
  - 規範違反主張を伴う指摘は実機検証で規範違反の成立を確認し、成立しない場合は
    `### 却下した代替案`へ却下理由と実機検証結果を明記して却下する
  - 規範違反が成立する場合でも、計画ファイルの肥大化・自立性の阻害・可読性の低下のいずれかに
    該当する字面適用は本質要件を満たす代替修正手段を検討する。
    採用時は本質要件との対比・採用手段・実機検証結果を`### 却下した代替案`へ明記する
- codex指摘1件が複数セクションに伝播する場合、関連セクションすべてを網羅的に更新する。
  `## 変更履歴`への追記だけで完了扱いにせず`## 変更内容`本文への転記を同一ターン内で完遂する。
  転記漏れは計画ファイル自立性違反にあたる
- 計画ファイルを全面改訂した場合、過去のcodexセッション（MCPの`threadId`またはCLIの`SESSION_ID`）を破棄し、
  新規セッションで初回プロンプトからレビューをやり直す
- 計画文面のコードブロックに実装案コードが含まれる場合、
  同一モジュール内の既存類似関数の実装方式と整合するかを観点に含める旨を初回プロンプトに記載する
- 実装案コードが文字列走査・入力ストリーム処理・状態機械・トークン切り出しなど境界条件が
  挙動を変える処理を含む場合、境界条件の網羅観点を初回codexプロンプトへ含める。
  網羅観点は空入力・単一行入力・改行文字有無・EOF終端・未閉じ構造・区切り不在時の残余処理を含む
- 時刻・日付・タイムゾーンを扱う実装（cron・スケジュール・タイマー・UNTIL/DTSTART・rrule等）の
  計画レビューでは、次の投入観点を初回プロンプトへ含める
  - 変換関数の入出力タイムゾーン契約（UTC／ローカル／特定TZID）
  - `new Date().getFullYear/Month/Date()`系のブラウザローカルTZ依存の混入
  - `Date.UTC()`と`Date.now()`の意味的違い
  - 外部ライブラリのDate表現規約（rruleのUTC-markedローカル時刻等）

## 初回プロンプト雛形

`用途: 計画レビュー`の初回プロンプトは以下のテンプレートで構成する（MCP・CLI共通）。

```text
{plan_full_path} この計画ファイルをレビューして。

レビュー観点:

- 発見した指摘は重大度ラベルを付けて全件報告する（採否の選別はメイン側で行う）
- 全件報告の中で、同種カテゴリの違反が同ファイル内に複数出現する場合は代表例1件に集約し、残箇所は箇所数のみ示す
- `agent-toolkit:*`スキルや`plan-*-reviewer`などの識別子はClaude Code側の仕組みでcodex側からは存在確認できないため、
  これらの実在性に関する指摘は不要
- 構文の合法性は機械チェック担当領域のため指摘対象外とする。
  対象言語バージョンで有効化された新構文（Python 3.14のPEP 758による`except`括弧省略等）を
  学習知識との齟齬で構文エラーと誤判定しないこと
- 計画ファイル自体は実装着手後に廃棄される一時的な作業文書であり、永続的なプロジェクト成果物ではない。
  通常のコード・ドキュメントレビューでは指摘すべき以下の観点も、計画ファイルでは指摘対象外とする
  - 記述スタイル（章構成・段落構成・表現選択・書き方）への指摘。記述間の矛盾は対象に含む
  - 実装時にエージェントが判断可能な細部（変数名・エラーメッセージ文言・小規模なループ構造・局所的な制御フローなど）への指摘
- 指摘対象は計画ファイル自立性が崩れる重大不備に限定する。対象例は次の通り
  - 参照先実体の不在
  - 対象ファイル一覧の欠落
  - 矛盾する判断記述
  - 実装段階で再現できない曖昧な手順
- 形式適合は成立可能な実装計画である限り指摘対象外とする。対象外の例は次の通り
  - 節配置の最適化・追記欄の網羅性・確認結果の記録粒度
  - 現行行数記載・ハードリミット試算
  - 縮減根拠の節配置・遡及スキャン結果の節配置
```

継続プロンプトは`計画ファイルを更新したので再レビューして。対象は計画ファイル本文への初回指摘反映であり、
実装コード・対象ファイル現状の状態は評価対象外である。`とする。

## Windows環境での注意

codexはPowerShell経由でファイルを読み書きする際、デフォルトでShift-JISが使われ日本語が文字化けする。
対処としてプロンプトに以下を追記する。
「ファイルの読み書きはUTF-8エンコーディングを明示すること（例: `Get-Content -Encoding UTF8`）」

## plan-file-creatorからの起動

`agent-toolkit:plan-file-creator`が内部でcodexレビューを起動する場合、
`subagent_type: agent-toolkit:plan-codex-delegate`（`用途: 計画レビュー`）のみ使う。
観点分担で並列起動してよい（詳細は下記「並列起動方針」）。
`mcp__codex__codex`直接呼び出しへ自律フォールバックしない。
`plan-codex-delegate`起動がauto mode下でブロックされた場合は`needs_escalation`で呼び出し元
（トップレベルセッション）へ返却する。呼び出し元が`mcp__codex__codex`を直接呼び出して継続する。
理由は`isSidechain`真の状態で`plan-file-creator`自身が直接フォールバックすると
`codex_review_invoked`フラグが記録されないためである。
`agent-toolkit/scripts/posttooluse.py`の`isSidechain`分岐は`plan-codex-delegate`内部の`用途: 実装`呼び出しとの
誤記録回避のため維持し、本エージェントでは改修しない。

### 並列起動方針

計画ファイルレビューは`plan-codex-delegate`を既定の第一経路とし、観点分担で並列起動する。
基本は「構造・整合性」「実現可能性・技術妥当性」の2並列とする。
計画が規範文書（`AGENTS.md`・`CLAUDE.md`・`.claude/rules/`配下・`agent-toolkit/`配下）の
改訂を含む場合は「規範整合」を加えた3並列とする。各インスタンスの`追加のレビュー観点`欄へ担当ラベルをそのまま渡す。
`plan-reviewer`（claude）は既定経路から外し、「codex利用可否の3段階判定」節の段階3が
成立した場合に限り代替として起動する。
