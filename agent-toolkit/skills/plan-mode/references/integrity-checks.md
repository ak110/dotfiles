# 計画ファイル整合性チェック

`agent-toolkit:plan-file-creator`が実施する整合性チェックで適用する観点を節ごとに定義する（節名は本ファイルの見出しを典拠）。

## ユーザー発話・提示素材との照合

`## 背景`の許容情報源はユーザー発話・対話確認内容と提示素材原文に限定する。提示素材は原文転記する。
調査結果・既定値・推測・エージェント解釈は`### エージェント判断`へ移動する。
`### ユーザー合意済み事項`の各項目はユーザー発話・質問回答へ1対1で遡及照合し、遡れない項目は`### エージェント判断`へ移動する。
照合対象は原文の合意発話・`AskUserQuestion`回答・確認応答を含み、推測・既定判断・素材原文内の未合意提案は合意欄に含めない。
起動プロンプト・フィードバック原文・スキル定義はユーザー発話に該当せず、`### エージェント判断`側へ出所明示のうえ記載する。
`### 計画メタ情報`・`### 現状`・`### バグ調査結果`調査由来要約はエージェント記載を許容する例外として本節の限定対象から除外する。

## 計画文内・他ファイルとの整合

サブエージェント担当部分:

- 計画ファイル本文全域で同一概念を指す用語が複数表現で混在しないか照合する
- 「置換パターン: <旧> → <新>（対象: ...）」形式のH3配下で列挙されたsublistファイルすべてが
  対象ファイル一覧に含まれるかを`plan-reviewer`が照合する。対象ファイル一覧とH3見出しの1対1対応自体は、
  PreToolUseフック（`_check_plan_file_target_files_h3_correspondence`）が編集時にブロックレベルで機械検査する。
  このため`plan-reviewer`側では重複照合をしない
- `## 調査結果`で影響を受けるとした既存ファイルが対象ファイル一覧に網羅反映されているかも併せて照合する
- 追記文面の表記が追記先の既存表記と整合しているかも併せて照合する
- 計画ファイル本文の追記文面と既存規範文書の間で同一概念の判定基準を2箇所以上に記述する場合、`grep -rn`で全箇所を抽出して文言完全一致を照合する
  - 代替案として定義元1箇所へ集約し、読取側は委譲記述（「判定基準は定義元に従う」相当）のみへ統一する形を優先する
  - 集約が公開インターフェース互換性の理由等で成立せず読取側で再掲する場合は定義元と一字一句同一の表現に揃える
- `## 変更履歴`と`## 変更内容`の対応照合は「変更履歴と変更内容の対応照合」節の機械検査と並行して、
  `plan-reviewer`が意味の同期確認を担う
- 既存規範の周辺品質サブ観点: 追加・改訂・拡張・削除・統合・移動時は同節・隣接節の該当記述抽出と統合可否を確認する。
  周辺文面の不自然な踏襲・複数概念マージ時の手順脱落・新ルール適用漏れ・横断指針か（agent-standards節参照）を照合する
- 工程4「恒久化検討」の出力検査: 計画ファイルの`### 恒久化・リファクタリング内容`へ転記された恒久化候補が
  既存規範との衝突・上位概念への統合可否・指摘原因の遡及対象の各観点に適合するかを照合する
- 複数規範改訂の論理交差確認: 同一計画内でN件（N≧2）の規範改訂を含む場合、
  リスクベース条件に該当するペアを机上シミュレーションする
  - 具体手順とリスクベース条件は`norm-revision-checklist.md`「複数規範改訂の論理交差確認」節に従う
- 確定ラベルの転記照合: 計画`### エージェント判断`で複数選択肢ラベル
 （「採用／部分採用／不採用」「採否区分」「重大度ラベル」等）を確定した場合、地の文・書式例（コードフェンス内）・
  AskUserQuestion条件文・置換指示の旧→新文面の各箇所で確定用語通りの転記か、派生表現の混入がないかを照合する
- 意図的重複の同期注記照合: 同一趣旨の規範を2ファイル以上に追加または既存規範を改訂する計画では、
  `grep -n '^# 同期注記:' <対象ファイル>`で検出される全ファイルが計画本文の対象ファイル一覧に含まれるかをplan-file-creatorの整合性チェックで照合する
- 追記文言案の自己参照する指示語との照合:
  計画本文の`## 変更内容`配下の追記文言案（`[置換後]`ブロック内文面・frontmatter注記文面等）を対象とする。
  「本節」「本ファイル」「上記」「本バレット」等の指示語が
  追記対象の実ファイル適用後も適切な指示先を持つかを実ファイルの節構造と照合する
  - `plan-reviewer`の担当観点として明示する
  - 既存の`check_self_ref.py`が検出する限定パターン以外の一般語尾を人的レビューで補完する位置付けとする
- サブエージェント担当観点として宣言された観点の実在照合:
  計画本文でサブエージェント（`plan-reviewer`・`plan-spec-reviewer`・`plan-impl-reviewer`・
  `agent-doc-validator`等）の担当観点として宣言された観点を対象とする。
  当該サブエージェント定義ファイル（`agent-toolkit/agents/<name>.md`）の
  `## 担当節`または`## 担当観点`H2配下に該当観点が実在するかをレビュー時に照合する
  - 実在しない場合は担当宣言側の修正か、当該サブエージェント定義側への担当観点追記かのいずれかを重大指摘として報告する
  - 静的リスト運用のサブエージェントへ`references/`配下の個別バレットが担当として宣言されている場合は重大指摘として報告する。
    静的リスト運用のサブエージェントは`agent-doc-validator`のように`agent-standards`共通と`01-agent.md`品質最優先で運用する形式を指す
- 既存節へのバレット追加時は追記先の既存バレット述部を`grep -n`で抽出し新規述部が既存パターンに揃うか照合する
- 機械検査関連スクリプト（`check_plan_diff_gates.py`・`pretooluse.py`等）への新規warn/check関数追加を
  対象ファイル一覧に含む場合、次の各項が計画本文に反映済みかを照合する
  - `check_plan_diff_gates.py`が`_check_plan_file`と`_extract_diff_blocks`の双方から呼び出される。
    `_plan_format.py`配下SSOT判定関数の新設・改訂時も両スクリプトから呼び出される
  - モジュールdocstringの検査項目列挙（`pretooluse.py`の「統合しているチェック:」等）に追記される
  - 対応テスト（`check_plan_diff_gates_test.py`・`pretooluse_test.py`等）が対象ファイル一覧に含まれる
  - `_plan_format.py`配下等のSSOT関数の新設・改訂を対象ファイル一覧に含む場合、
    当該SSOT関数の呼び出し元全経路（本番スクリプト・対応テスト）を`grep -rn '<関数名>'`で列挙する。
    全経路が対象ファイル一覧へ網羅されているかを照合する
  - `hooks.json`のmatcher拡張を対象ファイル一覧に含む場合、
    当該matcherが起動するハンドラ本体（`pretooluse.py`・`posttooluse.py`等）と対応テストも対象ファイル一覧へ含まれているかを照合する
- 変更対象の節・文言が他文書から準用・参照される場合（「準用範囲」節・委譲記述・SSOT参照を含む）、参照元の全文脈を列挙し、変更後文面が各文脈で成立するかを照合する。
  単一文脈の前提（状態・時系列・主体）を全文脈へ一般化していないかを確認する
- 他ファイルへ実体委譲する縮減案（`caller-reception.md`等への委譲）を計画に含める場合、参照先ファイルの該当節を`Read`で確認し、
  逆方向の参照が存在しないか（循環参照が発生しないか）を計画段階で照合する

呼び出し元（メイン）が`plan-file-creator`起動前に実施する部分（会話コンテキスト依存のため委譲不可）:

- 追記・改訂トピック関連の別ファイル記述（集約先・対象範囲・列挙・節ラベル・典拠先等）の追従更新要否を`grep -rn`で確認する
- ファイル一覧で得たファイルは本文まで読み込んで対応関係を確認する
- 計画ファイル本文に「集約する」「委ねる」「従う」「参照する」等の参照型表現を含む追記文面を書く場合、
  参照先（ファイル・節・コメントブロック・関数docstring等）の実体存在を計画提出前に検査する
  - 実在しない場合は参照先の新規作成を`## 変更内容`の対象ファイル一覧へ含めるか参照型表現を撤回するかのいずれかを取る
- レビュー指摘反映で対象ファイル一覧・変更内容が拡張・縮小・改訂される場合、`### 却下した代替案`・`### エージェント判断`欄の記述と反映後の状態との整合を反映直後に確認する。矛盾する却下記述は削除するか、当該レビュー指摘反映により却下判断を変更した旨の注記を追加する
- 意図的重複の同期注記の逆方向照合: 「意図的重複の同期注記照合」バレット
  （前方向照合、`plan-file-creator`が対象ファイル一覧網羅を確認）が特定した同期先列挙ファイル全件の本体を
  `grep`または`Read`で確認し、規定本文を保持しないファイルは既存矛盾として同一計画で是正対象へ含める
  - 節名実在照合は`pretooluse.py`の`_check_frontmatter_sync_note_body_exists`が警告として担い、
    規定本文の意味論的な実在確認は逆方向照合が担う

`plan-file-creator`が自身のBashツールで実施する部分（会話コンテキスト非依存のため委譲可能）:

- 各サブエージェント起動（条件成立時の`agent-doc-validator`含む）前に、次を機械的にセルフチェックする
  - `uv run --script agent-toolkit/skills/plan-mode/scripts/check_plan_file.py <計画ファイルパス>`を実行する。
    検査項目は同スクリプトのdocstringを典拠とする
  - `## 変更内容`H3節配下の`[現行]/[置換後]`テキスト対比ブロックが対象ファイル実体と一意一致するか
    （転記の陳腐化防止）を検査対象とする。不一致検出時は`[現行]`ブロックへ周辺行を含めて一意化する
  - 対象ファイルに`.md`規範文書が含まれる場合、`plan-implementer`委譲のmodel/effort指定規範と突合する。
    新規規範のスコープ限定表現（固有名詞限定の有無）がコーディングエージェント向け文書全般へ拡張可能かも点検する
  - `## 変更内容`配下の対象ブロック（ラベル体系は`plan-file-guidelines.md`「変更内容」小節の定義に従う）を検査対象とする。
    縮退フレーズは違反0件を必須とする。
    textlint違反は転記文面として極力解消する（残存は自律判断で記録）
  - 計画ファイルに`#### 廃止・改名対象一覧`H4が存在する場合、リポジトリ横断grep結果と対象ファイル一覧の差集合が空か確認する
  - `agent-doc-validator`の起動要否について、後述「整合性チェック・codexレビューの実施手順」節
    「サブエージェント`agent-doc-validator`（条件付き起動）」バレットの起動条件に該当するかを確認する
  - 改訂委譲時の既存記述の整合性チェック: 反映方針で変更・撤回・移動が指示された概念について、
    改訂適用後に次の検査表を作成し、機械検索と意味的照合の2段階で残置を検知する。
    - 検査表の作成: 改訂指示ごとに「旧表記」「関連識別子」「対称箇所」を1行ずつ列挙する。
      旧表記は撤回する文字列・移動前の節名・変更前の識別子等、
      関連識別子は同一概念を指す別表記・略記・カタカナ表記、
      対称箇所は実装案コードとdocコメント・複数節にわたる同一定義などを指す
    - 機械検索: 検査表の「旧表記」「関連識別子」を対象に
      `grep -n <文字列> <計画ファイルパス>`で計画ファイル本文の全セクション
      （`### バグ調査結果`表・`### 提示素材`・`### エージェント判断`・`### 却下した代替案`・
      `### 恒久化・リファクタリング内容`・`## 調査結果`・`## 変更内容`・`## 実行方法`）を検索し、
      ヒット箇所を全件列挙する
    - 意味的照合: 機械検索で0ヒットの場合も、検査表の「対称箇所」欄を人手で読み比べて、
      実装案コードとdocコメント・複数箇所で定義される概念・順序依存の記述の整合を確認する
    - 合格条件: 機械検索のヒットが「改訂後方針を積極的に参照する記述」のみで構成され、
      意味的照合での対称箇所照合で矛盾が検出されない状態

## 機械チェック適合性

計画ファイル本文全域が実装後の機械チェック（sentence-length・口語表現・漢字7字連続）を通過する形か確認する。
`## 背景`配下の原文転記領域は本節の例外として違反を許容する。
本文全域の機械チェックのうち和文ハイフンは`plan-file-guidelines.md`「計画ファイル全体の遵守事項」節の書き込み後チェック（`check_plan_file.py`側）で実施済みとなる。警告出力のみでexit codeへ算入しない。
textlint（sentence-length等の各ルールを含む）・markdownlint・typos・口語表現の全文検査はplan-file-creatorの整合性チェックステップ4で行う。
実行コマンドは`uvx pyfltr run-for-agent`とし、1回のみ実行する。
`plan-reviewer`は補完照合のみ担当し、残存する体裁系違反は軽微指摘として扱う。
`## 変更内容`・`## 対応方針`配下のコードブロック内文面は次で点検する。

- `agent-toolkit/rules/04-styles.md`「日本語の品質を保つ」節の上位原則
- `agent-toolkit:writing-standards`「Markdown記述スタイル」節

対象ファイル一覧に統合ランナー（`check_plan_file.py`）またはPreToolUseハンドラ（`pretooluse.py`）自身を含み、修正で機械チェックの失敗が解消される類型の場合は許容する。
当該事象は`### エージェント判断`へ定型注記で記録する。

## 編集対象スキル固有規定の事前適用

本節は`agent-toolkit:agent-standards`「文書サイズ上限」節の対象ファイルを扱う。
対象ファイルは`01-agent.md`・`03-claude-code.md`・各`SKILL.md`・サブエージェント定義・`references/`配下などとする。
対象ファイルが計画ファイル`## 変更内容`の対象ファイル一覧へ含まれる場合は、計画ファイル本文の以下を機械的に検査する。

- `## 変更内容`の`wc -l`現行の実測値記載が`## 調査結果`配下の`### 対象ファイルの現状`H3へ記載済みかを検査する。
  欠落時は重大指摘として報告する。
  `agent-toolkit:agent-standards`「文書サイズ上限」節のトリガー行数超過が見込まれる場合の`#### 縮減対象`H4配置要否は、
  `plan-file-guidelines.md`「6-1. 参照テンプレート読み込み」節の判定条件に従い併せて検査する。
  行数の厳密な再計算はレビュー観点に含めない
- `agent-toolkit/`配下のファイル変更を含む計画では対象ファイル×改訂対象H2/H3節数のマトリクスを
  `## 変更内容`または`### エージェント判断`「版更新」項へ表形式で列挙する
  （5列: ファイル・改訂節数・節名・判定・該当基準。集計対象は`agent-toolkit/`配下に限定し`.chezmoi-source/`は対象外）。
  MINOR判定基準への該当を計画段階で照合し、PATCH判定時は根拠を再確認して記録する。
  基準は`agent-toolkit-edit`「バージョン更新」節に従う。
  `plan-reviewer`はマトリクスと`## 実行方法`の`scripts/agent_toolkit_bump.py {種別}`記述の整合を照合する
  - 「判定」列は`bump不要`ラベルも許容する（コメント・docstring・非配布ファイルのみの変更等でbump不要と確定した場合）。
    `_plan_format.py`のSSOT関数が本ラベルを検出してbumpステップ記載を求める警告を抑止する
  - `check_plan_file.py`の`_check_version_bump_matrix_consistency`は次の条件で警告を出力する
    （exit codeへ算入しない）
    - 改訂節数1かつ該当基準列に「新設」を含まない行で判定列が`MINOR`・`MAJOR`の場合
    - `## 実行方法`本文（コードフェンス除外）のbump種別が判定列最大値
      （MAJORをMINORより大、MINORをPATCHより大、PATCHをbump不要より大とする順）と不一致の場合
