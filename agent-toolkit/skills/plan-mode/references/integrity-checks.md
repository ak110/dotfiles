# 計画ファイル整合性チェック

`agent-toolkit:plan-mode`スキル工程6-2の整合性チェックで適用する観点を節ごとに定義する。
関連スキル・サブエージェントが参照する節名は本ファイルの見出しを典拠とする。

## ユーザー発話・提示素材との照合

`## 背景`の許容情報源はユーザー発話・対話確認内容と提示素材原文に限定する。
提示素材は原文転記する。
調査結果・既定値・推測・エージェント解釈は`### エージェント判断`へ移動する。

## 計画文内・他ファイルとの整合

サブエージェント担当部分:

- 計画ファイル本文全域で同一概念を指す用語が複数表現で混在しないか照合する
- `## 変更内容`の対象ファイル一覧チェックボックス項目と各ファイル変更方針H3見出しが完全に対応するかを双方向照合する
- `## 調査結果`で影響を受けるとした既存ファイルが対象ファイル一覧に網羅反映されているかも併せて照合する
- 追記文面の表記が追記先の既存表記と整合しているかも併せて照合する
- 計画ファイル本文の追記文面と既存規範文書の間で同一概念の判定基準を2箇所以上に記述する場合、
  `grep -rn`で全箇所を抽出して文言完全一致を照合する
  - 代替案として定義元1箇所へ集約し、読取側は委譲記述（「判定基準は定義元に従う」相当）のみへ統一する形を優先する
  - やむを得ず読取側で再掲する場合は定義元と一字一句同一の表現に揃える
- `## 変更履歴`に記録された方針転換・修正内容が`## 変更内容`配下の文面・対象ファイル一覧・差分指示・サブエージェント起動プロンプト文面に同期反映されているか照合する
- 既存規範の周辺品質サブ観点: 既存規範の改訂・拡張時は周辺の既存文面に不自然な表現の踏襲がないかを照合する
  - 複数概念を1述句へマージする場合は各概念の手順・条件・対象の差異が脱落していないか照合する
  - 新ルール・規範（計画策定プロセス自体の試算手順・調査手順・整合性チェック手順を含む）追加・改訂時は当該規範を計画ファイル本文にも適用して整合させる
  - 新規規範を追加する場合、同節・隣接節の既存箇条のうち新規規範の対象事象に該当する記述を全件抽出し、新規規範で置換・統合する形になるか確認する
  - 構造的・形式的な規範改訂も既存違反遡及スキャンの対象として等しく扱う
- 工程4「恒久化検討」の出力検査: 計画ファイルの`### 恒久化・リファクタリング内容`へ転記された恒久化候補が既存規範との衝突・上位概念への統合可否・指摘原因の遡及対象の各観点に適合するかを照合する
- 複数規範改訂の論理交差確認: 同一計画内でN件（N≧2）の規範改訂を含む場合、C(N,2)=N(N-1)/2ペアを全件机上シミュレーションする
  - 具体手順は`agent-toolkit/skills/plan-mode/references/norm-revision-checklist.md`「複数規範改訂の論理交差確認」節に従う
- 確定ラベルの転記照合: 計画`### エージェント判断`で複数選択肢ラベル（「採用／部分採用／不採用」「採否区分」「重大度ラベル」等）を確定した場合、計画本文への転記を照合する
  - 照合対象: 地の文・書式例（コードフェンス内）・AskUserQuestion条件文・置換指示の旧→新文面の各箇所
  - 確定用語そのままで転記されているか、また確定用語と異なる派生表現（語尾拡張・修飾語付与）が地の文に混入していないかを照合する

メイン側担当部分:

- 追記・改訂対象トピックに関連する別ファイルの記述（集約先案内文・対象範囲記述・列挙構造・節ラベルの転記箇所・典拠先記述・docstring内の同義説明など）に追従更新の要否を`grep -rn`で網羅確認する
- ファイル一覧で得たファイルは本文まで読み込んで対応関係を確認する

## 機械チェック適合性

計画ファイル本文全域が実装後の機械チェック（sentence-length・line-length・口語表現・漢字7字連続）を通過する形か確認する。
`## 背景`配下の原文転記領域は本節の例外として違反を許容する。
`## 変更内容`・`## 対応方針`配下のコードブロック内文面はagent.md「日本語の品質を保つ」節の上位原則と`agent-toolkit:writing-standards`「Markdown記述スタイル」節で点検する。
diff `+`側文面は計画ファイル本体の機械チェックでは実質的に検査対象外となるため、`agent-toolkit:plan-mode`スキル本文の工程7の反映後機械チェック手順で検査する。

## 編集対象スキル固有規定の事前適用

本節は`agent-toolkit:agent-standards`「文書サイズ上限」節の対象ファイル（`agent.md`・`claude-code.md`・各`SKILL.md`・サブエージェント定義・`references/`配下など）を扱う。
対象ファイルが計画ファイル`## 変更内容`の対象ファイル一覧へ含まれる場合は、計画ファイル本文の以下2点を機械的に検査する。

- `agent-toolkit:plan-mode`スキル工程6-1の発火条件（現行行数200行以上・追加見込み20%以上・全節跨ぐ改訂のいずれか）に該当する計画では、`## 変更内容`に実装段階での`wc -l`確認手順の記載が存在するかを検査する。欠落している場合は重大指摘として報告する
- 上記いずれかに該当する場合は`### エージェント判断`に`wc -l`実測の試算行数が、いずれにも該当しない場合は概算値の記載が、それぞれ存在するかを検査する。欠落している場合は重大指摘として報告する

## 担当分担

各節の担当先を次の通り定める。

- メイン側担当:「ユーザー発話・提示素材との照合」全体と、「計画文内・他ファイルとの整合」のうち横断`grep -rn`確認・関連ファイル本文の通読部分
- `plan-integrity-checker`サブエージェント担当: 上記以外の節（「計画文内・他ファイルとの整合」の計画ファイル内部照合部分・「機械チェック適合性」・「編集対象スキル固有規定の事前適用」）

サブエージェント担当節とメイン側担当節の対応関係は呼び出し元コンテキスト参照の要否で分かれる。

## 工程7の実施手順

1. 計画ファイル初版Write完了後、`## 変更内容`配下のdiff +側を編集対象ファイル単位の改訂後最終形として
   一時ファイル群へ出力し、同一メッセージ内で次を並列起動する
   - codexレビュー: `codex-review.md`を読み込んで手順に従う
   - サブエージェント`plan-integrity-checker`:
     「計画文内・他ファイルとの整合」「機械チェック適合性」「編集対象スキル固有規定の事前適用」の
     独立コンテキスト点検を委譲する
     - 起動プロンプトに本ファイルの読み込み指示、計画ファイルパス、
       「計画文内・他ファイルとの整合」節の既存規範の周辺品質サブ観点点検対象の既存規範ファイルパスを記載する
   - サブエージェント`naive-executor`: 計画ファイルの愚直読解を委譲する
     - 起動プロンプトに対象テキストとして計画ファイルパスを渡す
     - 実行コンテキストとして「これは計画ファイルである。あなたはこの計画に従って実装する実行者である」を渡す
     - 抽出観点として「暗黙補完・参照不能・手順抜け・複数解釈の4カテゴリ全件」を渡す
   - サブエージェント`plan-impl-reviewer`:
     上記一時ファイル群を対象とし、コード・ドキュメント・日本語表現の単体品質を事前レビューする
     - 起動プロンプトに各一時ファイル絶対パスと正本パスの対応表、差分基準として計画着手前コミットSHAを記載する
2. 同一メッセージ内で「ユーザー発話・提示素材との照合」と
   「計画文内・他ファイルとの整合」のうちユーザー発話照合と横断grep確認部分をメイン側で並行実施する
3. 全ての指摘が出揃った時点で
   （工程2.5および本ステップで起動した`plan-impl-reviewer`の結果も含む）、
   重大度ラベルに基づき対応要否を判断する。
   軽微指摘はメイン判断で取捨する。対応する指摘を計画ファイルの該当セクションへ反映する
4. 反映後にメイン側で機械チェックを実行する
   - コマンド: `uvx pyfltr run-for-agent --no-fix --work-dir=. <計画ファイルパス>`
     （cwdをプロジェクトルートに設定）
   - `--no-fix`必須（`## 背景`原文転記領域の無断整形を避けるため）。
     `pre-commit`はリポジトリ外パスでエラーになるが他コマンド結果を主軸とする
   - 判定対象は計画ファイル本文全域だが`## 背景`配下の原文転記領域は違反許容
     （「ユーザー発話・提示素材との照合」節既定）
   - ステップ1で出力した`## 変更内容`配下のdiff +側一時ファイル群に対しても機械チェックを実行する
     - 実行対象: 計画ファイル向けと同じ`pyfltr`コマンドに加え、
       `agent-toolkit:writing-standards`配下`scripts/check_line_width.py`・`scripts/check_dash.py`
   - 並列チェッカーが構造違反を検出していた場合は、本ステップで指摘を反映し機械チェック違反を同時解消する
   - 検出違反は計画ファイル本文へ反映する。
     Claude Code側permission modeが`plan`なら`ExitPlanMode`を呼び出し、
     `plan`でなければ`ExitPlanMode`を呼ばず工程8へ進む
