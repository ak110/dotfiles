# 計画モード・計画ファイル作成

工程1〜8を、以降の同番号の節（`## 1.`〜`## 8.`、工程2は`## 2.5`〜`## 2.7`を含む）の定義に従い順に実施する。

## 工程の進め方

- 工程の登録: 工程が未登録ならバグ対応該当時は工程1〜8、非該当時は工程2〜8を`TaskCreate`で一括登録する
  - `TaskCreate`が利用不可な環境（サブエージェント下等でdeferred toolから除外される場合等）では、
    計画ファイル`## 進捗ログ`に工程チェックリスト（未着手・進行中・完了の状態を示す箇条書き）を作成して代替する
  - 代替時は各工程の遷移報告を`## 進捗ログ`への追記で行う
- 各工程は`TaskUpdate`で`in_progress`・`completed`へ遷移し、完了時に決定事項を箇条書きでユーザーへ報告する
- 次工程への移行は`AskUserQuestion`発行時のみ応答を待ち、完了報告への応答は待たない
- 分割実行・再入時: スキル再読み込み時に工程登録済みなら未完了工程を先頭から再走査する（登録済みは再登録しない）
- 暗黙スキップ・明示バイパスの禁止: いかなる理由でも登録された全工程を実施する。
  前段で満たし済みの工程も根拠を1行報告してから`completed`にする
  - 実施を回避する根拠として採用しない例:
    変更規模（対象ファイル数・変更行数・単一ファイル性など）・
    auto modeでの自律実行下・機械チェック通過・作業量見込み・所要時間見込み等
  - サブエージェント下でも本規範は等しく適用される。
    委譲プロンプトが実装・コミット・pushまでを一括指示する場合でも、全工程の省略は禁止する。
    `agent-toolkit/rules/01-agent.md`「品質最優先」節および
    「サブエージェント設計・起動形態を理由に本節の規範を自身へ適用しない判断はしない」規定と一体で適用する
  - 多段委譲経路（`process-feedbacks`経由・`apply-feedback`経由等）でネスト起動された本スキルは、
    上位スキルの工程遷移目標（振り返り工程の到達等）を各工程の完遂の免除根拠にしない。
    上位スキルの後続工程は本スキルの全工程が完了して初めて実施可能となる
  - `plan-impl`実行中の割り込みで工程4のみを再実施する場合も本規範を等しく適用する。
    当該の工程4を完遂して初めて`plan-impl`の後続工程へ復帰可能となる
  - 登録済み工程を実施せず先へ進める判断を発話・記録した時点で当該判断を無効とし、
    直ちに正規の工程へ復帰する。当該発話・記録を`session-review`や恒久化の検討等の観測事象として扱う場合、
    `agent-toolkit:agent-standards`配下`references/quality-first-exceptions.md`の
    例外条件（`atk fb status`有効・根本原因の独立性・成果物品質への非影響・対応要件充足）を個別に確認する。
    代表フレーズ集は`agent-toolkit:agent-standards`配下`references/scope-escalation-phrases.md`の
    `process-omission`カテゴリを参照する。当該パターンの宣言を規範違反続行の正当化に用いない
- plan mode移行の前提: 本スキルはplan mode下でも外でも起動できる。
  plan mode下の場合は`ExitPlanMode`が計画ファイル承認ゲートとして機能する。
  plan mode外の場合はユーザー承認を介さず、工程8で`agent-toolkit:plan-impl`へ引き継ぐ。
  Codex環境ではAGENTS.mdの「plan modeのエミュレーション」節に従う
  - `process-feedbacks`・`apply-feedback`スキル経由で本スキルが呼び出される場合はplan mode外で実行する。
    メイン側で`EnterPlanMode`を発行しない。
    plan mode下での起動はユーザーの明示的な`EnterPlanMode`指示があった場合に限る
- 本スキル未起動のまま計画ファイル（`~/.claude/plans/*.md`）の編集に至った場合は、
  PreToolUseフックがブロックするため本スキルを呼び出したうえで工程1または2から順に実施する
- 例外: plan mode移行後にユーザーから単純な即時実行の指示があった場合に限り、
  `AskUserQuestion`で即時実行への切替可否を確認してよい（選択時は`ExitPlanMode`で抜けて直接実行する）

## 1. バグ対応手順（最優先で確認）

ユーザーの指示がバグ・障害・エラー・デグレードなどの対応に該当する場合、
`references/bugfix-process.md`を読み込み手順に従い確実に対応する。
判定が即座につかない場合はバグ対応に該当するものとして扱う。

判定はユーザー指示時に加え工程2以降の調査・レビューでバグ・デグレード・不具合を発見した時点でも行い、
発見時に工程1が未登録なら追加してバグ対応手順へ移行する。

## 2. 計画ファイル作成前の要件対話

ユーザー指示が曖昧・複数の解釈が成り立つ場合は、先に要件を対話的に明確化してから計画ファイル作成に進む。

- 対話で確定させる観点: 目的・ゴール、制約・前提、成功条件、スコープ境界、方針の選択肢、
  ユーザーが直接利用する新設の識別子（スキル名・コマンド名・CLIオプション名など公開インターフェース）の命名。
  加えて運用面の要望（自動起動・デプロイ・既存運用への統合）。
  類似ツールに運用面の仕組みがある場合はその踏襲有無を含む
- 決定の依存関係を順に解消する形で計画の各分岐を網羅的に質問し、
  各質問にはエージェントの推奨回答を併記してユーザーが上書きできる形にする
  - 各論点が確定するまで対話を継続する
- コードベース調査で判明する事項は先に調査し、
  `AskUserQuestion`で複数の論点をバッチでまとめて提示する（質問ごとの応答待ちで分断しない）
- 対話で得た結論は`## 背景`と`### ユーザー合意済み事項`へ転記する
  （合意済み事項はユーザー発話・確認で合意した方針のみ）
  - ユーザーから複数の改訂候補を一括で見直すよう指示された場合の扱い。
    確定改訂は`### ユーザー合意済み事項`へ転記する。
    評価判断を要する候補は判定基準とともに`### エージェント判断`へ分離する。
    各候補の採否を当該基準に照らした根拠とともに1行ずつ示す
- ユーザーが指示の素材として提示した文面（レビュー指摘・エラー出力・要件抜粋等）は`## 背景`へ原文転記する

計画段階の調査原則は`references/investigation-rules.md`の「計画段階の調査原則」節に従う。

## 4. 恒久化検討

当該計画に至るまでの本セッションのユーザー対話と、`apply-feedback`等で採用したフィードバック原文の主旨は、
当該計画固有の事情を除きすべてを恒久化候補として点検対象に含める。
ユーザー対話には明示合意・割り込み指摘・選好・判断材料として提示された文面・指示の前提となった暗黙の方向性を含む。
点検対象のうち当該計画固有でない恒久的な知見を整理する。
永続化先と文面をいずれも工程6-2「計画ファイル本体の作成」前に確定する（実装工程・工程6以降への先送りはしない）。
確定した文面案・検討結果は工程6-2で計画ファイルの`### 恒久化・リファクタリング内容`へ転記する。
永続化先はコードに近い順で検討する。

- 第一候補: ソースコードのdocstring・インラインコメント（コード固有の知見）
- 第二候補: `CLAUDE.md`・`.claude/rules/`配下（プロジェクト横断の方針・繰り返す誤判断の補強）
- 第三候補: `.claude/skills/`配下（情報量が多く利用頻度の低い手順・リファレンス、トリガー時のみロード）
- `docs/`配下にコーディングレベルの指針を書かない。別リポジトリ・別管理でも永続化を見送らない

ユーザー介入が発生した契機（誤判断・規範未認識・手順抜けなど）に対する恒久化候補は、
規範文面の改善だけでなく機械的に再発を抑止する仕組みでも検討する。
対象は、hook追加・スキル冒頭の警告追記・コマンドラッパー追加・トリガーフレーズ拡張・テストケース追加などとする。
仕組み化可能な場合は仕組み案を恒久化先候補に含め、規範文書側の改訂と並行して導入する。

計画ファイル提示後・`plan-impl`実行中に新たな割り込み指摘・是正要求が発生した時点で、
当該指摘の追加修正と並行して本工程を即時に実施する（`plan-impl/SKILL.md`の規定に従う）。
後発で同根本原因の指摘が再発した場合は、前回の恒久化結果を参照したうえで本工程を再実施する。

## 5. リファクタリング検討

本来の対応が既存コード・ドキュメントの理想的な状態を崩す場合は、
それを補うリファクタリングを終えて初めて修正完了とみなす
（検討結果は工程6で`### 恒久化・リファクタリング内容`へ記載する）。

- 本計画の追記・削除により編集対象ファイル全体（または周辺ファイル群）の構成・節順・粒度バランスが崩れる場合は
  構成全体の再設計を検討する。追記時は当該ファイルの章構成が新節を吸収できる形になっているか、
  削除時は不要となる節・冗長化する記述・整合性を失う参照を全件洗い出す
- コード・ドキュメント変更を契機に発見した周辺対応（責務肥大化・粒度不整合・陳腐化・テスト不足など）と、
  計画作成中に発見した既存記述の不備を洗い出し、`## 変更内容`に本来の変更と並べて列挙する
- 責務帰属・抽象化層の妥当性を`agent-toolkit:coding-standards`「責務帰属（SRP）分析の観点」節に従い点検する
- コーディングエージェント向け文書（ルール・スキル等）に新節・新項目を追加する計画では、
  既存節・既存項目の自明事項・冗長記述の削除観点も合わせて洗い出す
- 編集対象スキル本体・関連ファイル群の過去コミット履歴に小改訂が累積している場合は`git log -p`で経緯確認し、
  上位スキルへの責務吸収など過去の関連変更へ追従しきれず温存された記述がないかを点検する
- 編集対象ファイル群が複数ディレクトリ・複数サブシステムへ跨る場合は、
  `Explore`サブエージェントへ既存類似先例・責務肥大・陳腐化箇所の横断調査を委譲する。
  委譲時は対象範囲・調査観点（責務帰属・抽象化層・既存規範違反など）・期待出力（ファイルパスと該当節の一覧）を明示する

## 5.5 既存運用データ・運用仕組みの移行検討

運用中のシステム・仕組み（既存運用ディレクトリ・既存ファイル形式・既存設定ファイル・個人運用メモなど）の改修時は、
既存データを新仕様へ移行する手段を検討する。

- 既存データの新形式への変換手順（リネーム・ディレクトリ移動・スキーマ変換など）
- 旧形式の残置可否と互換性保持の必要性
  （git履歴に残せば作業ツリーから除去してよいか、利用側コードが旧形式を解釈できなくなる影響など）
- 移行検証手順の組み込み（移行前後の件数照合・サンプル抽出比較・移行後の動作確認）

移行が必要と判断した場合は移行手順を`## 変更内容`へ列挙し`## 実行方法`の検証ステップ直前へ記述する。
不要と判断した場合は理由を`### エージェント判断`へ明記する。

## 8. 計画の実行

plan mode下では`ExitPlanMode`前に記述漏れを最終確認しユーザー承認後に
メインが`agent-toolkit:plan-impl`を呼び出して計画ファイルへ引き継ぐ。
plan mode外ではユーザー承認を介さずメインが直接`agent-toolkit:plan-impl`を呼び出す
（レビュー実施有無は`## 実行方法`のレビューステップで分岐する）。
