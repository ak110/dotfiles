# 計画モード・計画ファイル作成

以下の工程を順に実施する。

1. バグ対応手順
2. 計画ファイル作成前の要件対話
3. ユーザー要求の認識合わせ
4. 恒久化検討
5. リファクタリング検討
6. 計画ファイルの作成
7. 計画ファイルのcodexレビュー
8. 計画の実行

## 工程の進め方

- 起動直後: バグ対応該当時は工程1〜8、非該当時は工程2〜8を`TaskCreate`で一括登録する
- 各工程: 開始時に`TaskUpdate`で`in_progress`、完了時に`completed`へ遷移する
- 各工程完了時: 決定事項を箇条書きでユーザーへ報告する
- 次工程への移行: `AskUserQuestion`発行時のみ応答を待つ。完了報告への応答は待たない

## 1. バグ対応手順（最優先で確認）

ユーザーの指示がバグ・障害・エラー・デグレードなどの対応に該当する場合、
`references/bugfix-process.md`を読み込み、手順に従い確実に対応する。
判定が即座につかない場合はバグ対応に該当するものとして扱う。

## 2. 計画ファイル作成前の要件対話

ユーザー指示が曖昧・複数の解釈が成り立つ場合は、先に要件を対話的に明確化してから計画ファイル作成に進む。

- 対話で確定させる観点: 目的・ゴール、制約・前提、成功条件、スコープ境界、方針の選択肢
- `AskUserQuestion`で複数の論点をバッチでまとめて提示する
- 対話で得た結論は`## 背景`（経緯と目的）と`### ユーザー合意済み事項`（合意した方針）へ転記する
- ユーザーが指示の素材として提示した文面（レビュー指摘・エラー出力・要件抜粋等）は`## 背景`へ原文転記する

「レビューして」「検討して」「見直して」のような指示は、
調査→改善案提示→ユーザー選択取得→計画ファイル反映までを本工程内で完結させる。
明示的に「調査だけ」と指示された場合のみ報告で完結させる。

## 2.5 ファイル移動・削除を伴う変更の旧パス参照確認

ファイル移動・削除・改名（ルール→スキル化、ファイル統合、モジュール再配置など）を伴う場合は、
計画ファイル作成前に旧パス参照を`grep -rn`で網羅確認する。
確認対象はコード内docstring・インラインコメント・他Markdown内のリンク・スクリプト引数等を含む。
確認結果は`## 調査結果`に反映する。

## 3. ユーザー要求の認識合わせ

工程2で曖昧さを除去した後、推測で埋めた前提・対応範囲・スコープ境界を箇条書きで提示する。

## 4. 恒久化検討

当該計画固有でない恒久的な知見（ユーザー合意事項・選択理由・プロジェクト方針的な情報）を整理し、
永続化先と文面を確定する。永続化先への書き込みは実装工程で行い、文面案は工程6で計画ファイルへ転記する。

永続化先はコードに近い場所から順に検討する。

- 第一候補: ソースコードのdocstring・インラインコメント。
  対象モジュール固有の判断（閾値の選定理由・ワークアラウンド・関数間の連携意図など）に向く
- 第二候補: `CLAUDE.md`・`.claude/rules/`配下。
  プロジェクト横断の方針や、エージェントが繰り返し誤った判断をする場合の補強に向く
- 第三候補: `.claude/skills/`配下。
  利用頻度は低いが情報量が多い手順・リファレンスに向く（トリガー時のみロード）
- `docs/`配下にはコーディングレベルの指針を書かない（人間の開発者向け情報のみ）
- 永続化先が別リポジトリ・別管理であることを理由に永続化を見送らない。
  別計画やTODO等への切り離しは避け当該セッション内で完了させる

## 5. リファクタリング検討

本来の対応によって理想的な状態が崩れるなら、それを補うリファクタリングまで完了して初めて修正完了とみなす。
「今回の本筋から外れるためスコープ外」として必要なリファクタリングを先送りしてはならない。

コード・ドキュメント変更を契機に発見した周辺対応（責務肥大化・粒度不整合・陳腐化・テスト不足など）と、
計画作成中に発見した既存記述の不備を洗い出し、`## 変更内容`に本来の変更と並べて列挙する。

コーディングエージェント向け文書（ルール・スキル等）に新節・新項目を追加する計画では、
追加と同時に既存節・既存項目の自明事項・冗長記述の削除観点も合わせて洗い出す。

## 6. 計画ファイルの作成

### 6-1. 参照テンプレート読み込み

計画ファイルの骨子を書き始める直前に、`references/plan-file-guidelines.md`を読み込む。

### 6-2. 計画ファイル本体の作成

テンプレートに従い所定パスへ計画ファイルを作成する。
工程2〜5の合意事項・解釈・恒久化文面・周辺対応をテンプレート規定の各セクションへ転記し、計画ファイル自立性を満たす。

作成完了後、以下の6観点で整合性をチェックする。
矛盾を検出したらいずれが正かを確定し、両セクションの記述を一致させてから次工程へ進む。

- 観点A（ユーザー発話・提示素材との照合）:
  下記のとおり限定されているか照合する。許容情報源は対象ごとに異なる。（バグ対応時の調査結果表は除く）
  - `## 背景`の叙述部分（動機・経緯・期待する成果）の許容情報源は次の2系統とする
    - ユーザーが直接発話・対話で確認した内容
    - ユーザーが指示の素材として提示した文面の原文
  - `### ユーザー合意済み事項`配下の各項目の許容情報源はユーザーが発話または確認で合意した事項とする
  - 提示素材の例: レビュー指摘・エラー出力・スタックトレース・要件抜粋・ログ・参考リンクの引用
  - 提示素材は要約・抜粋せず原文を転記し、計画ファイル単独で判断材料が揃う情報量を確保する
  - 調査結果・既定値・推測・エージェントの解釈で埋めた事項は`### エージェント判断`へ移動する
  - `## 背景`の叙述部分で許容情報源に紐づかない動機・経緯・成果は削除する
  - `## 背景`の2系統（発話・対話結論および提示素材原文）が共に端的な場合に限り、背景章も端的な記述を許容する
- 観点B（変更内容とエージェント判断の相互照合）:
  - `## 変更内容`表の対象ファイルが`### エージェント判断`で対応外と書かれていないか
  - `### エージェント判断`の方針が`## 変更内容`表に対象ファイルを伴って反映されているか
- 観点C（計画対象ファイル内の方針整合）:
  `## 変更内容`で計画対象ファイルに新たな方針記述を追記する場合、
  同一ファイル・同一章・同一節内の既存方針と衝突しないか照合する。
  衝突する場合は既存方針の改訂を伴う変更として`## 変更内容`へ含める
- 観点D（計画文面の機械チェック適合性）:
  `## 変更内容`配下のコードブロックに転記する成果物の最終文面は、
  実装後の機械チェックを通過する形で書く。検査対象は次の2系統とする。
  - textlint `ja-technical-writing/sentence-length`の120字上限
  - `agent-toolkit/skills/writing-standards/scripts/check_line_width.py`の半角換算127幅上限

  括弧書きを含む長文や、半角識別子・コード片を多用する行は計画策定時点で分割する。
  実装段階で違反が発覚した場合、文意を変えない範囲での分割は許容する
- 観点E（計画文面内部の用語統一）:
  計画ファイル本文内で同一概念を指す用語が複数の表現で混在していないか照合する。
  混在したまま実装に反映されると配布物の用語が不整合となる起点になるため、いずれかへ統一する
- 観点F（既存規範の改訂箇所周辺における日本語表現の再点検）:
  既存のルール・スタイルガイド・運用方針を改訂・拡張する変更を含む場合、
  改訂箇所周辺の既存文面の日本語表現を再点検する。
  既存文面の不自然な表現（格助詞の誤用・冗長な修飾など）を踏襲したまま追記を重ねると、
  改訂後に違和感のある文面が拡大する
- 観点G（集約先案内との整合）:
  追記・改訂対象のトピックに関連する集約先案内（「〜は〜スキルに集約する」「〜は〜節に集約する」など）が
  別ファイルに存在しないか`grep -rn`で確認する。
  案内文の集約範囲と整合しない追記となる場合、案内文と追記のいずれが正かを確定し、両者が整合する形へ調整する。
  検索キーワードは追記内容の中核語句（スキル名・概念名・対象モジュール名等）を用いる

## 7. 計画ファイルのcodexレビュー

ユーザーに計画ファイルを提示する前にcodexレビューする。
先に`agent-toolkit:review-standards`スキルを呼び出し、続いて`references/codex-review.md`を読み込んで手順に従う。
指摘があった場合は計画ファイルの該当セクションへ反映してから`ExitPlanMode`を呼び出す。

## 8. 計画の実行

`ExitPlanMode`でユーザーに計画ファイルを提示し、承認を得てから実行に進む。
ExitPlanMode前に記述漏れがないか最終確認し、メインが`agent-toolkit:plan-impl`を呼び出して計画ファイルへ引き継ぐ。
レビュー実施有無は計画ファイルの`レビュー方式`項目で分岐し、`agent-toolkit:plan-impl`側で処理される。
