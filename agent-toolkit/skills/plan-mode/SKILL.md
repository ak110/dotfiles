# 計画モード・計画ファイル作成

工程1〜8を、以降の同番号の節（`## 1.`〜`## 8.`、工程2は`## 2.5`〜`## 2.7`を含む）の定義に従い順に実施する。

## 工程の進め方

Claude Code固有ツール（`TaskCreate`・`TaskUpdate`・`AskUserQuestion`・`EnterPlanMode`・`ExitPlanMode`等）を前提とする。
他のコーディングエージェントは同等機能へ読み替える。

- 工程の登録: 工程が未登録なら、バグ対応該当時は工程1〜8、非該当時は工程2〜8を`TaskCreate`で一括登録する
- 各工程: 開始時に`TaskUpdate`で`in_progress`、完了時に`completed`へ遷移する
- 各工程完了時: 決定事項を箇条書きでユーザーへ報告する
- 次工程への移行: `AskUserQuestion`発行時のみ応答を待つ。完了報告への応答は待たない
- 分割実行・再入時: スキル再読み込み時に工程が登録済みなら、未完了工程を先頭から再走査して続行する（登録済みは再登録しない）
- 暗黙スキップの禁止: いずれの工程も省略しない
  前段のメッセージや過去の調査・レビューで満たし済みの工程も、満たした根拠を1行報告してから`completed`にする
- plan mode移行の前提: pretooluseフックで強制される。呼び出し前に`EnterPlanMode`でplan modeへ移行する
  - Codex環境ではAGENTS.mdの「plan modeのエミュレーション」節に従いエミュレーション開始を宣言する
- 本スキル未起動のまま計画ファイル（`~/.claude/plans/*.md`）の編集に至った場合、
  pretooluseフックがブロックする。本スキルを呼び出したうえで工程1（バグ対応該当時）または
  工程2（非該当時）から順に実施する

システムプロンプトのPlan Workflowが定義する調査・設計の段階は工程2〜5で実施する。
「計画して」「plan-modeに従って計画して」のような指示でも、plan mode中に調査・レビュー済みでも、
工程6へ直行せず工程1〜5を順に実施する。

## 1. バグ対応手順（最優先で確認）

ユーザーの指示がバグ・障害・エラー・デグレードなどの対応に該当する場合、
`references/bugfix-process.md`を読み込み、手順に従い確実に対応する。
判定が即座につかない場合はバグ対応に該当するものとして扱う。

判定はユーザー指示時に加え、工程2以降の調査・レビューでバグ・デグレード・不具合を発見した時点でも行う。
発見時に工程1が未登録なら追加し、バグ対応手順へ移行する。

## 2. 計画ファイル作成前の要件対話

ユーザー指示が曖昧・複数の解釈が成り立つ場合は、先に要件を対話的に明確化してから計画ファイル作成に進む。

- 対話で確定させる観点: 目的・ゴール、制約・前提、成功条件、スコープ境界、方針の選択肢、
  ユーザーが直接利用する新設の識別子の命名
  - 対象はスキル名・コマンド名・CLIオプション名など、公開インターフェースの命名
- 決定の依存関係を順に解消する形で、計画の各分岐について網羅的に質問する
  浅い理解で計画フェーズへ移らず、各論点が確定するまで対話を継続する
- 各質問にはエージェントの推奨回答を併記する。ユーザーが選択肢から選ぶか上書きできる形にする
- コードベース調査で判明する事項は質問せず、先に調査して回答を確定してから対話を進める
- `AskUserQuestion`で複数の論点をバッチでまとめて提示する（質問ごとの応答待ちで分断しない）
- 対話で得た結論は`## 背景`（経緯と目的）と`### ユーザー合意済み事項`（合意した方針）へ転記する
- ユーザーが指示の素材として提示した文面（レビュー指摘・エラー出力・要件抜粋等）は`## 背景`へ原文転記する

「レビューして」「検討して」「見直して」のような指示は、調査→改善案提示→ユーザー選択取得→計画ファイル反映までを
本工程内で完結させる。明示的に「調査だけ」と指示された場合のみ報告で完結させる。

## 2.5 ファイル増減を伴う変更の関連参照確認

ファイル移動・削除・改名（ルール→スキル化、ファイル統合、モジュール再配置など）を伴う場合は、
計画ファイル作成前に旧パス参照を`grep -rn`で網羅確認する。
確認対象はコード内docstring・インラインコメント・他Markdown内のリンク・スクリプト引数等を含む。
旧ファイル数・構成を前提とする間接記述（「両ファイル」「2ファイル」「N個のファイル」など）も検出対象に含める。

ファイル追加・コンポーネント追加（spec追加・サービス追加・モジュール追加など）を伴う場合は、
計画ファイル作成前に関連する差分外ファイルの件数表記・列挙を`grep -rn`で網羅確認する。
確認対象は仕様書のサービス一覧・構成図・READMEの機能一覧・対応表などを含む。
検索キーワードは追加対象の概念名（スキル名・サービス名など）と総数（「N件」「N観点」など）を用いる。
追従更新が必要な箇所は`## 変更内容`へ含める。

いずれも確認結果は`## 調査結果`に反映する。

## 2.6 コード変更時の連動テスト・参照確認

公開インターフェース仕様（利用者が直接観測する識別子・値・呼び出し規約の総称）を変更する場合は、
計画ファイル作成前に変更前の表記を`grep -rn`で網羅確認する。対象は公開関数のシグネチャ・挙動・
戻り値型・例外仕様・処理対象範囲、スキル・サブエージェント・公開ツールの呼び出し規約・前提条件・
引数仕様、リテラル文字列・期待値・出力フォーマット等の利用者観測値を含む。

確認対象は単体テストファイル・ドキュメント・スナップショット・hookメッセージ・サンプル設定等を含む。
特に呼び出し元モジュールのテストファイルは旧挙動前提のテストが混在し得るため、必ず読み込んで再評価する。
旧挙動を前提にしたテストの再評価・修正項目は`## 変更内容`へ明示し、確認結果は`## 調査結果`に反映する。

違反件数・該当箇所数などの集計値を計画ファイルへ記載する場合は、関数呼び出し・メソッド呼び出し・属性差し替えなど
複数の表記パターンを列挙して`grep -nE`等で網羅集計し、計画記載値と一致を確認してから記載する。

## 2.7 実現可能性の事前検証

計画の成立可否を左右する前提（外部機能・ツール挙動・ライブラリAPI・公式仕様・想定する環境条件など、
推測のまま計画へ採用すると計画全体が破綻し得る前提）は計画ファイル作成前に実機検証する。
結果を`## 調査結果`へ反映し、想定と異なる結果が出た場合は対応方針を変更してから工程3以降を順に進む。

## 3. ユーザー要求の認識合わせ

工程2で曖昧さを除去した後、推測で埋めた前提・対応範囲・スコープ境界を箇条書きで提示する。

## 4. 恒久化検討

当該計画固有でない恒久的な知見（ユーザー合意事項・選択理由・プロジェクト方針的な情報）を整理し、永続化先と文面を確定する。
永続化先への書き込みは実装工程で行い、文面案・検討結果は工程6で`### 恒久化・リファクタリング内容`へ記載する。

セッション内で受領したユーザー指摘事項自体も恒久化検討の対象に含める。
指摘の対応反映と並行して、再発防止策の永続化先と文面を確定する。
対応反映のみで完結させず、根本原因を規範・プロセスへ反映できないか必ず検討する。

永続化先はコードに近い場所から順に検討する。

- 第一候補: ソースコードのdocstring・インラインコメント
  - 対象モジュール固有の判断（閾値の選定理由・ワークアラウンド・関数間の連携意図など）に向く
  - コード固有の知見はここに留め、第二候補以降に持ち込まない
- 第二候補: `CLAUDE.md`・`.claude/rules/`配下（プロジェクト横断の方針、繰り返す誤判断の補強に向く）
- 第三候補: `.claude/skills/`配下（情報量が多く利用頻度の低い手順・リファレンスに向く。トリガー時のみロード）
- `docs/`配下にはコーディングレベルの指針を書かない（人間の開発者向け情報のみ）
- 永続化先が別リポジトリ・別管理でも永続化を見送らず、別計画やTODO等へ切り離さず当該セッション内で完了させる

## 5. リファクタリング検討

本来の対応が既存コード・ドキュメントの理想的な状態を崩す場合は、それを補うリファクタリングを終えて初めて修正完了とみなす。
「今回の本筋から外れるためスコープ外」などとして必要なリファクタリングを先送りしてはならない。
検討結果は工程6で`### 恒久化・リファクタリング内容`へ記載する。

- コード・ドキュメント変更を契機に発見した周辺対応（責務肥大化・粒度不整合・陳腐化・テスト不足など）と、
  計画作成中に発見した既存記述の不備を洗い出し、`## 変更内容`に本来の変更と並べて列挙する
- コーディングエージェント向け文書（ルール・スキル等）に新節・新項目を追加する計画では、
  既存節・既存項目の自明事項・冗長記述の削除観点も合わせて洗い出す
- バグ・不統一な実装・ドキュメントの誤り・テスト未整備・観測不能な挙動を見つけたら、指示外でもユーザーへ提案する

## 6. 計画ファイルの作成

### 6-1. 参照テンプレート読み込み

計画ファイルの初版Write直前に、`references/plan-file-guidelines.md`をReadで読み込む。
Read未実施のままWriteへ進むとテンプレート規定のH2セクション構成から外れ、
PostToolUseフックの構造検査で構造違反として警告される。簡易な指摘対応で計画規模が小さい場合でも省略しない。

### 6-2. 計画ファイル本体の作成

テンプレートに従い所定パスへ計画ファイルを作成する。
工程2〜5の合意事項・解釈・恒久化文面・周辺対応をテンプレート規定の各セクションへ転記し、計画ファイル自立性を満たす。

作成完了後、以下の観点で整合性をチェックする。
矛盾は両セクション一致まで解消してから次工程へ進む。
以下は間違いやすいポイントの例示集であり、網羅規定ではない。

- 観点A（ユーザー発話・提示素材との照合）:
  `## 背景`の叙述部分（動機・経緯・期待する成果）の許容情報源は、ユーザーが直接発話・対話で確認した内容と、
  指示の素材として提示した文面の原文の2系統に限定する。提示素材は要約・抜粋せず原文転記する。
  調査結果・既定値・推測・エージェントの解釈で埋めた事項は`### エージェント判断`へ移動する。
  `### ユーザー合意済み事項`は発話または確認で合意した事項のみとする
- 観点B（変更内容とエージェント判断の相互照合）:
  `## 変更内容`の対象ファイルと`### エージェント判断`の方針が双方向に整合しているか、
  `### 恒久化・リファクタリング内容`の各検討の結論が記載されているか（対象が無ければ「なし」と明記）、
  永続化先への書き込みや周辺対応が`## 変更内容`に反映されているかを照合する。
  方針・変更内容に数値（合計・差分・最終値・件数など）を含む場合は、算出式と結果値の整合、
  および集計値と`grep -nE`実測の整合も確認する。
  規範文書の改訂を含む場合、改訂後の物理行数と縮減で除去する行数の試算欄が`## 変更内容`に存在するかも確認する
- 観点C（計画対象ファイル内の方針整合）:
  `## 変更内容`で計画対象ファイルに新たな方針記述・節を追記する場合、同一ファイル内の既存方針と
  文書冒頭の目次的記述（章番号・節番号の一覧、サマリーテーブル、変更履歴の対応表など）との
  整合を照合する。衝突する場合は既存方針の改訂、目次的記述への追従更新を`## 変更内容`へ含める
- 観点D（計画文面の機械チェック適合性）:
  `## 変更内容`・`## 対応方針`配下のコードブロックに転記する成果物の文面が実装後の機械チェックを通過する形か。
  主な検査対象は120字sentence-length・127幅line-length・口語表現・漢字7字連続とする。
  括弧書きを含む長文や半角識別子を多用する行は計画策定時点で分割する。
  実装段階で違反が発覚した場合、文意を変えない範囲での分割は許容する
- 観点E（計画文面内部・追記先ファイルとの用語統一）:
  計画ファイル本文内で同一概念を指す用語が複数の表現で混在していないか照合し、混在時は統一する。
  追記・改訂を含む変更では追記文面の表記が追記先ファイルの既存表記と整合しているか`grep -rn`で確認する
- 観点F（既存規範の改訂箇所周辺における日本語表現の再点検）:
  既存規範の改訂・拡張時は改訂箇所周辺の既存文面の日本語表現を再点検し、不自然な表現を踏襲したまま追記しない
- 観点G（集約先案内との整合）:
  追記・改訂対象のトピックに関連する集約先案内（「〜は〜スキルに集約する」「〜は〜節に集約する」など）が
  別ファイルに存在しないか`grep -rn`で確認する。整合しない追記は案内文と追記のいずれが正かを確定し両者を整合させる。
  既存記述を別ファイル（参照先スキルのfrontmatter description・参照先節など）の記述に委ねる形で
  省略・縮約する場合、省略する内容が省略先で実際にカバーされているか`grep -rn`等で確認する。
  カバーされていない場合は省略を取りやめるか、省略先へ先に追記してから省略する
- 観点H（計画文面の機械チェック非対象の文面品質）:
  `## 変更内容`・`## 対応方針`配下のコードブロックの文面を、
  agent.md「日本語の品質を保つ」節の上位原則と`agent-toolkit:writing-standards`スキルの「Markdown記述スタイル」節、
  および`agent-toolkit:agent-standards`スキルの「既にコンテキストに読み込み済みと想定されるルールやスキルに対する
  言及を繰り返さない」原則で点検する。特に日本語文中への英字直挿入は機械チェックで検出されないため
  `diff`ブロック内も確認する。
  規範本文への追加文面案に会話文脈依存の指示語（「今回」「先ほど」「当該セッション」など）が含まれていないかも点検する。
  追記・改訂を含む変更では、追記先ファイルの既存表記との整合（観点E）を本観点より優先し、
  既存表記からの逸脱に当たる修正提案は採用しない
- 観点I（新規・改訂ルールの計画ファイル本文への自己適用）:
  計画ファイル内で新ルール・規範を追加または改訂する場合、当該規範を計画ファイル本文にも適用して整合させる

## 7. 計画ファイル整合性チェック・codexレビュー

ユーザーに計画ファイルを提示する前に、整合性チェックとcodexレビューを並列実行する。

1. 計画ファイル初版Write完了後、同一メッセージ内で次の2件を並列起動する
   - codexレビュー: `references/codex-review.md`を読み込んで手順に従う
   - サブエージェント`plan-integrity-checker`: 観点B・C・D・E・F・H・Iの独立コンテキスト点検を委譲する
     - 同サブエージェントは観点記号のみ参照し定義本文を持たないため、工程6-2の該当観点の定義本文を起動プロンプトへ転記する
     - あわせて計画ファイルパスと観点Fで点検対象とする既存規範ファイルのパスを渡す
2. 同一メッセージ内で観点A（ユーザー発話・提示素材との照合）と観点G（集約先案内との整合）をメイン側で並行実施する
3. 同一メッセージ内でメイン側から計画ファイル本文の機械チェックを実行する
   - コマンド: `uvx pyfltr run-for-agent --no-fix --work-dir=. <計画ファイルパス>`
   - `--no-fix`を付ける。fixステージは引用箇条書き内の句点除去など計画ファイル本文を無断整形し、
     `## 背景`のユーザー原文転記領域まで書き換え得るため、検査のみ実行する
   - fix相当の修正が必要な場合は差分を確認したうえで該当セクションへ手動反映する
   - カレントディレクトリをプロジェクトルートにすることで設定ファイル群を解決する
   - `pre-commit`コマンドはリポジトリ外パスでエラーになるが、他コマンドの結果を主軸として扱う
   - 機械チェック適合性の判定対象は計画ファイル内のコードブロック（成果物の想定文面）に限定する
4. 全ての指摘が出揃ったら、重大度ラベルに基づき対応要否を判断し（軽微な形式上の指摘はメイン判断で取捨してよい）、
   対応する指摘を計画ファイルの該当セクションへ反映してから`ExitPlanMode`を呼び出す

## 8. 計画の実行

`ExitPlanMode`でユーザーに計画ファイルを提示し、承認を得てから実行に進む。
ExitPlanMode前に記述漏れがないか最終確認し、メインが`agent-toolkit:plan-impl`を呼び出して計画ファイルへ引き継ぐ。
レビュー実施有無は計画ファイルの`## 実行方法`のレビューステップで分岐し、`agent-toolkit:plan-impl`側で処理される。
