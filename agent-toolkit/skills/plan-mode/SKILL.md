# 計画モード・計画ファイル作成

工程1〜8を、以降の同番号の節（`## 1.`〜`## 8.`、工程2は`## 2.5`〜`## 2.7`を含む）の定義に従い順に実施する。

## 工程の進め方

- 工程の登録: 工程が未登録ならバグ対応該当時は工程1〜8、非該当時は工程2〜8を`TaskCreate`で一括登録する
- 各工程は`TaskUpdate`で`in_progress`・`completed`へ遷移し、完了時に決定事項を箇条書きでユーザーへ報告する
- 次工程への移行は`AskUserQuestion`発行時のみ応答を待ち、完了報告への応答は待たない
- 分割実行・再入時: スキル再読み込み時に工程登録済みなら未完了工程を先頭から再走査する（登録済みは再登録しない）
- 暗黙スキップ・明示バイパスの禁止: いずれの工程も省略しない。前段で満たし済みの工程も根拠を1行報告してから`completed`にする
  - auto modeでの自律実行下や機械チェック通過のみを理由として工程7（整合性チェック・codexレビュー・plan-integrity-checker）を省略しない
- plan mode移行の前提: Claude Code側permission modeが`plan`の場合は呼び出し前に`EnterPlanMode`でplan modeへ移行する。
  `plan`でない場合は`EnterPlanMode`を呼ばず本スキルから直接着手する（Codex環境ではAGENTS.mdの「plan modeのエミュレーション」節に従う）
- 本スキル未起動のまま計画ファイル（`~/.claude/plans/*.md`）の編集に至った場合は、PreToolUseフックがブロックするため本スキルを呼び出したうえで工程1または2から順に実施する
- 例外: plan mode移行後にユーザーから単純な即時実行の指示があった場合に限り、`AskUserQuestion`で即時実行への切替可否を確認してよい（選択時は`ExitPlanMode`で抜けて直接実行する）

## 1. バグ対応手順（最優先で確認）

ユーザーの指示がバグ・障害・エラー・デグレードなどの対応に該当する場合、`references/bugfix-process.md`を読み込み手順に従い確実に対応する。判定が即座につかない場合はバグ対応に該当するものとして扱う。

判定はユーザー指示時に加え工程2以降の調査・レビューでバグ・デグレード・不具合を発見した時点でも行い、発見時に工程1が未登録なら追加してバグ対応手順へ移行する。

## 2. 計画ファイル作成前の要件対話

ユーザー指示が曖昧・複数の解釈が成り立つ場合は、先に要件を対話的に明確化してから計画ファイル作成に進む。

- 対話で確定させる観点: 目的・ゴール、制約・前提、成功条件、スコープ境界、方針の選択肢、ユーザーが直接利用する新設の識別子（スキル名・コマンド名・CLIオプション名など公開インターフェース）の命名
- 決定の依存関係を順に解消する形で計画の各分岐を網羅的に質問し、各質問にはエージェントの推奨回答を併記してユーザーが上書きできる形にする
  - 各論点が確定するまで対話を継続する
- コードベース調査で判明する事項は先に調査し、`AskUserQuestion`で複数の論点をバッチでまとめて提示する（質問ごとの応答待ちで分断しない）
- 対話で得た結論は`## 背景`と`### ユーザー合意済み事項`へ転記する（合意済み事項はユーザー発話・確認で合意した方針のみ）
  - ユーザーから複数の改訂候補を一括で見直すよう指示された場合、`### ユーザー合意済み事項`には確定改訂のみを記載する。
    評価判断を要する候補は`### エージェント判断`へ判定基準とともに分離し、各候補の採否は当該基準に照らした判定根拠を1行ずつ示す
- ユーザーが指示の素材として提示した文面（レビュー指摘・エラー出力・要件抜粋等）は`## 背景`へ原文転記する

### 計画段階の調査原則

- メイン側Read範囲は計画成立に必須なファイルに限定する。広範囲な周辺調査・横断検索・列挙系grepは`Explore`サブエージェントへ委譲する
- 新規実装または既存機能拡張を計画する場合、既存類似先例を1件以上特定し当該パス・該当節を`### エージェント判断`に明示する（先例が見つからない場合はその旨を明記する）
- 他ファイルの節タイトル・節内項目名・キーワードを計画文面に引用する場合は、引用元節をReadで原文確認してから転記する

## 2.5 ファイル増減を伴う変更の関連参照確認

ファイル増減・コンポーネント増減・概念名のリネーム・新規スキル導入時は、計画ファイル作成前に対象表記を`grep -rn`で網羅確認する。

- ファイル移動・削除・改名: 旧パス参照と「両ファイル」「N個のファイル」等の件数前提記述
- ファイル追加・コンポーネント追加: 仕様書のサービス一覧・構成図・READMEの機能一覧・対応表・「N件」「N観点」等の総数表記
  - 検索キーワードは追加対象の概念名（スキル名・サービス名など）と総数表記（「N件」「N観点」など）を用いる
- 概念名・スキル名のリネーム・新規スキル導入: 既存スキル側で陳腐化する節見出し・description・mermaid図ラベル等

確認結果は`## 調査結果`に、追従更新箇所は`## 変更内容`に反映する。

編集対象ファイル全般に対し、当該ファイルへ適用される既存規範違反パターン（コーディング規範・命名規則・サイズ規定・参照解決方式など）を`grep -nE`で計画作成前に網羅検査する。
検出した既存違反は`## 調査結果`へ列挙し`## 変更内容`に修正項目を組み込む。
新規規範や新規規定を導入する計画では、規範対象範囲のスクリプト群・コード群も同じ手順で網羅確認する。

編集対象が特定プラグイン・サブシステム配下のファイル群を含む場合、当該ディレクトリの編集規範を集約する配置先スキルを計画作成前に起動する（所在はプロジェクト直下の`AGENTS.md`・`CLAUDE.md`から特定）。
配置先スキルが指定する必須記載事項（バージョン管理・配布物固有名チェック・固有の検証手順など）を計画ファイル本文へ計画作成時点で組み込む。

`## 変更内容`に書く対象ファイル一覧を確定した直後に、各ファイルパスをプロジェクト直下の`AGENTS.md`・`CLAUDE.md`に記載された配置先スキル指定と1ファイルずつ照合する。
未起動の配置先スキルがあれば計画ファイルWrite前に起動する。

## 2.6 コード変更時の連動テスト・参照確認

公開インターフェース仕様（利用者が直接観測する識別子・値・呼び出し規約の総称）を変更する場合は、計画ファイル作成前に変更前の表記を`grep -rn`で網羅確認する。
対象は公開関数のシグネチャ・挙動・戻り値型・例外仕様・処理対象範囲、スキル・サブエージェント・公開ツールの呼び出し規約・前提条件・引数仕様、リテラル文字列・期待値・出力フォーマット等の利用者観測値を含む。

確認対象は単体テストファイル・ドキュメント・スナップショット・hookメッセージ・サンプル設定等を含む。
特に呼び出し元モジュールのテストファイルは旧挙動前提のテストが混在し得るため、必ず読み込んで再評価する。
旧挙動を前提にしたテストの再評価・修正項目は`## 変更内容`へ明示し、確認結果は`## 調査結果`に反映する。

## 2.7 実現可能性の事前検証

計画の成立可否を左右する前提（外部機能・ツール挙動・ライブラリAPI・公式仕様・想定する環境条件など、推測のまま計画へ採用すると計画全体が破綻し得る前提）は計画ファイル作成前に実機検証する。
結果を`## 調査結果`へ反映し、想定と異なる結果が出た場合は対応方針を変更してから工程3以降を順に進む。
外部依存物（自身の管理外コードに相当するOSS・ライブラリ・SDK・プラグイン・外部CLI・外部サービス等）に依存する計画では、
表面的なAPI仕様（公開関数・コマンドフラグ・設定キーの型と既定値）だけで前提を確定しない。
依存物のソース・公式ドキュメントの実装節・既知issue・テストコードのいずれかを読み、
次の4観点を実機観測に先行して確認し`## 調査結果`へ転記する。

- 入力値ごとに分岐する内部処理経路
- 関連する他設定・グローバル状態・環境変数への副次書き込み
- 読み込み順・評価順・初期化順への依存
- エラー経路・リトライ・タイムアウトの暗黙仕様

公式リファレンス・公式ブログの記述だけで根本前提を確定せず、外部ツール挙動・hook環境変数・CLI出力・ライブラリ実行時挙動などは実機観測してから採用する。
実機検証が環境制約で実行不能な場合は当該前提を`### エージェント判断`に「未検証」と明記し依拠範囲を縮める（前提が崩れた場合の代替方針も併記する）。

設定ファイル変更・CLI挙動変更を伴う計画では、対象CLIに検証用サブコマンド（`critique`・`lint`・`validate`・`check`・`config`・`defaults`等）があれば、計画段階で実機実行して出力を`## 調査結果`へ転記する。
検証結果に応じて`## 変更内容`の文面案・`## 実行方法`の検証ステップを確定する。

## 3. ユーザー要求の認識合わせ

工程2で曖昧さを除去した後、推測で埋めた前提・対応範囲・スコープ境界を箇条書きで提示する。

## 4. 恒久化検討

当該計画に至るまでの本セッションのユーザー対話（明示合意・割り込み指摘・選好・判断材料として提示された文面・指示の前提となった暗黙の方向性）は、当該計画固有の事情を除き全てを恒久化候補として点検対象に含める。
点検対象のうち当該計画固有でない恒久的な知見を整理し、永続化先と文面を確定する。
永続化先への書き込みは実装工程で行い、文面案・検討結果は工程6で`### 恒久化・リファクタリング内容`へ記載する。
永続化先はコードに近い順で検討する。

- 第一候補: ソースコードのdocstring・インラインコメント（コード固有の知見）
- 第二候補: `CLAUDE.md`・`.claude/rules/`配下（プロジェクト横断の方針・繰り返す誤判断の補強）
- 第三候補: `.claude/skills/`配下（情報量が多く利用頻度の低い手順・リファレンス、トリガー時のみロード）
- `docs/`配下にコーディングレベルの指針を書かない。別リポジトリ・別管理でも永続化を見送らない

## 5. リファクタリング検討

本来の対応が既存コード・ドキュメントの理想的な状態を崩す場合は、それを補うリファクタリングを終えて初めて修正完了とみなす。
検討結果は工程6で`### 恒久化・リファクタリング内容`へ記載する。

- コード・ドキュメント変更を契機に発見した周辺対応（責務肥大化・粒度不整合・陳腐化・テスト不足など）と、計画作成中に発見した既存記述の不備を洗い出し、`## 変更内容`に本来の変更と並べて列挙する
- 責務帰属・抽象化層の妥当性を`agent-toolkit:coding-standards`「責務帰属（SRP）分析の3観点」節に従い点検する
- コーディングエージェント向け文書（ルール・スキル等）に新節・新項目を追加する計画では、既存節・既存項目の自明事項・冗長記述の削除観点も合わせて洗い出す
- 編集対象スキル本体・関連ファイル群の過去コミット履歴に小改訂が累積している場合は`git log -p`で経緯確認し、上位スキルへの責務吸収など過去の関連変更へ追従しきれず温存された記述がないかを点検する

## 5.5 既存運用データ・運用仕組みの移行検討

運用中のシステム・仕組み（既存運用ディレクトリ・既存ファイル形式・既存設定ファイル・個人運用メモなど）の改修時は、既存データを新仕様へ移行する手段を検討する。

- 既存データの新形式への変換手順（リネーム・ディレクトリ移動・スキーマ変換など）
- 旧形式の残置可否と互換性保持の必要性（git履歴に残せば作業ツリーから除去してよいか、利用側コードが旧形式を解釈できなくなる影響など）
- 移行検証手順の組み込み（移行前後の件数照合・サンプル抽出比較・移行後の動作確認）

移行が必要と判断した場合、移行手順を`## 変更内容`に本来の変更と並べて列挙し、`## 実行方法`の検証ステップ直前に移行コマンドを記述する。
不要と判断した場合は理由を`### エージェント判断`へ明記する。

## 6. 計画ファイルの作成

### 6-1. 参照テンプレート読み込み

計画ファイルの初版Write直前に`references/plan-file-guidelines.md`をReadで読み込む（未実施時はPostToolUseフックが構造違反として警告するため簡易対応でも省略しない）。

### 6-2. 計画ファイル本体の作成

テンプレートに従い所定パスへ計画ファイルを作成し、工程2〜5の合意事項・解釈・恒久化文面・周辺対応をテンプレート規定の各セクションへ転記して計画ファイル自立性を満たす。
作成完了後、`references/integrity-checks.md`を読み込んで観点A〜Jの定義に従い整合性をチェックする。観点記号・名称は同referencesの記述を典拠とし、関連スキル・サブエージェントが参照する呼称もこれに統一する。

## 7. 計画ファイル整合性チェック・codexレビュー

ユーザーに計画ファイルを提示する前に、機械チェックを先行させたうえで整合性チェックとcodexレビューを実施する。

1. 計画ファイル初版Write完了後、メイン側で機械チェックを実行する
   - コマンド: `uvx pyfltr run-for-agent --no-fix --work-dir=. <計画ファイルパス>`（cwdをプロジェクトルートに設定）
   - `--no-fix`必須（`## 背景`原文転記領域の無断整形を避けるため）。fix相当は差分確認後に手動反映する。`pre-commit`はリポジトリ外パスでエラーになるが他コマンド結果を主軸とする
   - 判定対象は計画ファイル本文全域だが`## 背景`配下の原文転記領域に限り違反を許容する（観点A既定）
   - 検出された違反のうち自前で解消できるもの（textlint・127幅・sentence-length等）はこの時点で計画ファイルへ反映してから次ステップへ進む
2. 機械チェック反映後、同一メッセージ内で次の2件を並列起動する
   - codexレビュー: `references/codex-review.md`を読み込んで手順に従う
   - サブエージェント`plan-integrity-checker`: 観点B・C・D・E・F・H・I・Jの独立コンテキスト点検を委譲する
     - 起動プロンプトに`references/integrity-checks.md`の読み込み指示、計画ファイルパス、観点F点検対象の既存規範ファイルパスを記載する
3. 同一メッセージ内で観点A（ユーザー発話・提示素材との照合）と観点G（ファイル間対応関係の追従整合）をメイン側で並行実施する
4. 全ての指摘が出揃ったら、重大度ラベルに基づき対応要否を判断し（軽微指摘はメイン判断で取捨）、対応する指摘を計画ファイルの該当セクションへ反映する
5. ステップ1と同じ機械チェックを再実行し検出された違反を解消する。
   Claude Code側permission modeが`plan`なら`ExitPlanMode`を呼び出し、`plan`でなければ`ExitPlanMode`を呼ばず工程8へ進む

## 8. 計画の実行

plan mode下では`ExitPlanMode`前に記述漏れを最終確認しユーザー承認後にメインが`agent-toolkit:plan-impl`を呼び出して計画ファイルへ引き継ぐ。
plan mode外ではユーザー承認を介さずメインが直接`agent-toolkit:plan-impl`を呼び出す（レビュー実施有無は`## 実行方法`のレビューステップで分岐する）。
