# 計画モード・計画ファイル作成

工程1〜7を、以降の同番号の節（`## 1.`〜`## 7.`、工程2は`## 2.5`〜`## 2.7`を含む）の定義に従い順に実施する。

## 工程の進め方

- 工程の登録: 工程が未登録ならバグ対応該当時は工程1〜7、非該当時は工程2〜7を`TaskCreate`で一括登録する
  - `TaskCreate`が利用不可な環境（サブエージェント下等でdeferred toolから除外される場合等）では、
    計画ファイル`## 進捗ログ`に工程チェックリスト（未着手・進行中・完了の状態を示す箇条書き）を作成して代替する
  - 代替時は各工程の遷移報告を`## 進捗ログ`への追記で行う
- 各工程は`TaskUpdate`で`in_progress`・`completed`へ遷移し、完了時に決定事項を箇条書きでユーザーへ報告する
- 次工程への移行は`AskUserQuestion`発行時のみ応答を待ち、完了報告への応答は待たない
- 分割実行・再入時: スキル再読み込み時に工程登録済みなら未完了工程を先頭から再走査する（登録済みは再登録しない）
- 暗黙スキップ・明示バイパスの禁止: いかなる理由でも登録された全工程を実施する。
  前段で満たし済みの工程も根拠を1行報告してから`completed`にする
  - 実施を回避する根拠として採用しない例:
    変更規模（対象ファイル数・変更行数・単一ファイル性など）・
    auto modeでの自律実行下・機械チェック通過・作業量見込み・所要時間見込み等
  - 実行形態（サブエージェント下・多段委譲経路でのネスト起動・`plan-impl-executor`実行中の割り込みでの
    工程4再実施等）を問わず本規範を適用する。
    初回起動時は登録済みの全工程を完遂するまで呼び出し元・後続工程へ復帰しない。
    割り込みで工程4のみを再実施する場合は当該工程4の完遂のみで復帰可能とする。
    本規定は`agent-toolkit/rules/01-agent.md`冒頭の、
    サブエージェント設計・起動形態を理由に本ファイルの規範を自身へ適用しない判断をしない旨の規定と一体で適用する。
    - 委譲プロンプトが実装・コミット・pushまでを一括指示する場合でも全工程の省略は禁止する
    - 上位スキルが工程遷移目標（振り返り工程の到達等）を提示する場合でも全工程完遂を免除しない
  - 登録済み工程を実施せず先へ進める判断を発話・記録した時点で当該判断を無効とし直ちに正規の工程へ復帰する。
    復帰後に当該発話・記録を観測事象として扱いたい場合のみ、`agent-toolkit:agent-standards`配下
    `references/quality-first-exceptions.md`「正規追加経路の非対称原則」節の適用順序に従い経路2の
    条件充足を判定する（順序の入れ替えは不可）。代表フレーズ集は`references/scope-escalation-phrases.md`の
    `process-omission`カテゴリを参照する。当該パターンの宣言を規範違反続行の正当化に用いない
- plan mode移行の前提: 本スキルはplan mode下でも外でも起動できる。
  plan mode下は`ExitPlanMode`が計画ファイル承認ゲートとして機能する。
  plan mode外はユーザー承認を介さず工程7で`plan-impl-executor`へ引き継ぐ。
  Codex環境ではAGENTS.mdの「plan modeのエミュレーション」節に従う。<!-- section-ref-ok -->
  - `process-feedbacks`・`plan-and-add-feedback`経由で呼び出される場合はplan mode外で実行し、メイン側は
    `EnterPlanMode`を発行しない
    （PreToolUseフックが`process_feedbacks_skill_invoked`または`plan_and_add_feedback_skill_invoked`
    真時にブロックする）
- 本スキル未起動のまま計画ファイル（`~/.claude/plans/*.md`）の編集に至った場合は、
  PreToolUseフックがブロックするため本スキルを呼び出したうえで工程1または2から順に実施する
- 例外: plan mode移行後にユーザーから単純な即時実行の指示があった場合に限り、
  `AskUserQuestion`で即時実行への切替可否を確認してよい（選択時は`ExitPlanMode`で抜けて直接実行する）
- 計画作成プロセス自体（本スキル・`references/`配下・スクリプト・レビューサブエージェント）を改訂する場合、
  計画ファイル向けチェック・レビューの目的を「計画が成立しない致命的な問題の検出」に置き体裁・表記の指摘は警告に留める。
  工程・チェック項目・レビュー経路・再レビュー契機の追加時は同一改訂内で同等以上の削減・軽量化を検討し、所要時間を増やす一方向の改訂をしない

## 1. バグ対応手順（最優先で確認）

ユーザーの指示がバグ・障害・エラー・デグレードなどの対応に該当する場合、
`references/bugfix-process.md`を読み込み手順に従い確実に対応する。
判定が即座につかない場合はバグ対応に該当するものとして扱う。

判定はユーザー指示時に加え工程2以降の調査・レビューでバグ・デグレード・不具合を発見した時点でも行い、
発見時に工程1が未登録なら追加してバグ対応手順へ移行する。

## 2. 計画ファイル作成前の要件対話

ユーザー指示が曖昧・複数の解釈が成り立つ場合は、先に要件を対話的に明確化してから計画ファイル作成に進む。

- 対話で確定させる観点: 目的・ゴール、制約・前提、成功条件、スコープ境界、方針の選択肢、
  ユーザーが直接利用する新設の識別子（スキル名・コマンド名・CLIオプション名など公開インターフェース）の命名。
  加えて運用面の要望（自動起動・デプロイ・既存運用への統合）。
  類似ツールに運用面の仕組みがある場合はその踏襲有無を含む
  - 表示・出力フォーマット系機能では、入力データとして利用可能な項目の全量を調査して一覧提示し、
    表示対象の取捨・配置・整形方針を一括確認する
  - 永続化フィールド（DBカラム・JSONに保存されるリスト/辞書等）を新設する場合、陳腐化を検知する契機（同期契機・除去タイミング）・自動削除契機（親要素削除時の連動）・肥大化の抑止方針（上限値・自動整理）を確定する。永続データのライフサイクル管理は後日ユーザー介入で追加指示が発生しやすい典型論点であり、計画時に一括で確定させることで介入往復を削減できる
- 決定の依存関係を順に解消する形で計画の各分岐を網羅的に質問し、
  各質問にはエージェントの推奨回答を併記してユーザーが上書きできる形にする
  - 各論点が確定するまで対話を継続する
- コードベース調査で判明する事項は先に調査し、
  `AskUserQuestion`で複数の論点をバッチでまとめて提示する（質問ごとの応答待ちで分断しない）
- 対話で得た結論は`## 背景`と`### ユーザー合意済み事項`へ転記する
  （合意済み事項はユーザー発話・確認で合意した方針のみ）
  - ユーザーから複数の改訂候補を一括で見直すよう指示された場合、確定改訂は`### ユーザー合意済み事項`へ転記し、
    評価判断を要する候補は判定基準とともに`### エージェント判断`へ分離する。
    各候補の採否を当該基準に照らした根拠とともに1行ずつ示す
- ユーザーが指示の素材として提示した文面（レビュー指摘・エラー出力・要件抜粋等）は`## 背景`へ原文転記する

計画段階の調査原則は`references/investigation-rules.md`の「計画段階の調査原則」節に従う。

## 2.5 編集対象の関連参照確認

ファイル増減・コンポーネント増減・概念名・スキル名・見出しラベルのリネーム・新規スキル導入時は、
計画ファイル作成前に対象表記を`grep -rn`で網羅確認する。
詳細手順は`references/investigation-rules.md`の「関連参照確認の細則」節に従う。

## 2.6 コード変更時の連動テスト・参照確認

公開インターフェース仕様（利用者が直接観測する識別子・値・呼び出し規約の総称）を変更する場合は、
計画ファイル作成前に変更前の表記を`grep -rn`で網羅確認する。
詳細手順は`references/investigation-rules.md`の「連動テスト・参照確認の細則」節に従う。

## 2.7 実現可能性の事前検証

計画の成立可否を左右する前提（外部機能・ツール挙動・ライブラリAPI・公式仕様・想定する環境条件など、
推測のまま計画へ採用すると計画全体が破綻し得る前提）は計画ファイル作成前に実機検証する。
詳細手順は`references/investigation-rules.md`の「実現可能性の事前検証の細則」節に従う。

工程2で曖昧さを除去した後、推測で埋めた前提・対応範囲・スコープ境界を箇条書きで提示する。

## 4. 恒久化検討

当該計画に至るまでの本セッションのユーザー対話と、`process-feedbacks`等で採用したフィードバック原文の主旨は、
当該計画固有の事情を除きすべてを恒久化候補として点検対象に含める。
永続化先の検討順序・仕組み化検討・数値根拠併記の詳細手順は
[references/permanence-review.md](references/permanence-review.md)に集約する。
永続化先と文面をいずれも工程6-2「計画ファイル本体の作成」前に確定する（実装工程・工程6以降への先送りはしない）。

## 5. リファクタリング検討

本来の対応が既存コード・ドキュメントの理想的な状態を崩す場合は、
それを補うリファクタリングを終えて初めて修正完了とみなす
（検討結果は工程6で`### 恒久化・リファクタリング内容`へ記載する）。

- 本計画の追記・削除により編集対象ファイル全体（または周辺ファイル群）の構成・節順・粒度バランスが崩れる場合は
  構成全体の再設計を検討する。追記時は当該ファイルの章構成が新節を吸収できる形かを確認し、
  削除時は不要となる節・冗長化する記述・整合性を失う参照を全件洗い出す
- コード・ドキュメント変更を契機に発見した周辺対応（責務肥大化・粒度不整合・陳腐化・テスト不足など）と、
  計画作成中に発見した既存記述の不備を洗い出す。
  洗い出した内容は`## 変更内容`に本来の変更と並べて列挙する。
  - 発見した既存規範違反の別課題化を許容する基準は`references/step5-supplement.md`「別課題化の許容基準」節に従う
- 責務帰属・抽象化層の妥当性を`agent-toolkit:coding-standards`「責務帰属（SRP）分析の観点」節に従い点検する
- コーディングエージェント向け文書（ルール・スキル等）に新節・新項目を追加する計画では、
  既存節・既存項目の自明事項・冗長記述の削除観点も合わせて洗い出す
- 編集対象スキル本体・関連ファイル群の過去コミット履歴に小改訂が累積している場合は`git log -p`で経緯確認し、
  上位スキルへの責務吸収など過去の関連変更へ追従しきれず温存された記述がないかを点検する
- 母集団の悉皆点検（類似見直し）: 詳細な適用契機・深さ基準・記録先は`references/similar-review.md`に従う
- 編集対象ファイル群が複数ディレクトリ・複数サブシステムへ跨る場合は、
  `Explore`サブエージェントへ既存類似先例・責務肥大・陳腐化箇所の横断調査を委譲する。
  委譲時は対象範囲・調査観点（責務帰属・抽象化層・既存規範違反など）・期待出力（ファイルパスと該当節の一覧）を明示する
- 症状の局所対処案と構造改善案の双方が成立し得る場合、両案を計画段階で比較し
  長期保守性・アーキテクチャ全体の整合を基準に採否を確定する。
  比較結果と選定根拠は`### エージェント判断`へ記述する

## 6. 計画ファイルの作成・整合性チェック

工程2〜5で確定した要件対話の結果・認識合わせの内容・恒久化検討の結果・リファクタリング検討の結果を
起動プロンプトへ過不足なく埋め込む。
Agentツールで`agent-toolkit:plan-file-creator`を起動し、計画ファイル本体の作成から
整合性チェック・codexレビューまでを委譲する。
起動前に`references/integrity-checks.md`「ユーザー発話・提示素材との照合」節をメイン側で実施し、
結果を起動プロンプトへ埋め込む。
起動プロンプトの雛形は`references/launch-prompts-plan-file-creator.md`に集約する。
完了報告に`needs_escalation`が含まれる場合は、報告された論点をメイン側で確定する。
確定後に縮減した起動プロンプトで`agent-toolkit:plan-file-creator`を再起動する。
plan mode下でサンドボックスパスへ書き込まれた場合の検収手順は`references/launch-prompts-plan-file-creator.md`「検収手順」節に従う。
工程6完了時にユーザー可視のテキスト応答へ計画ファイル絶対パスを1行で明示する
（`計画ファイル: <絶対パス>`形式）。

## 7. 計画の実行

plan mode下では`ExitPlanMode`前に記述漏れを最終確認しユーザー承認後に
メインがAgentツールで`agent-toolkit:plan-impl-executor`を起動して計画ファイルへ引き継ぐ。
plan mode外ではユーザー承認を介さずメインが直接`agent-toolkit:plan-impl-executor`を起動する。
レビュー実施有無は`## 実行方法`のレビューステップで分岐する。
起動前準備と完了報告の受領手順は`agent-toolkit/references/plan-impl/caller-reception.md`に従う。
