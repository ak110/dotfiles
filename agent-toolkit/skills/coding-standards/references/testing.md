# テスト方針（全言語共通）

新しいpublicな関数などを書く際は必ずテストコードも書く。簡潔さ・安定性・速度を重視する。

## 簡潔さ

- テストダブル（モック・スタブ・モンキーパッチ等）は、外部サービス・時間・乱数・グローバル状態など
  実コード経由では決定論的に検証できない依存に限定する。依存差し替えは引数注入
 （テスト対象にオプション引数追加・デフォルト値で本番動作維持）を優先し、
  テストダブルはグローバル状態や言語ランタイム機構のテストに限定する
- 単純過ぎるテストは避け、まとまった機能（CRUD流れ等）のテストを1つの関数にまとめ、セットアップオーバーヘッドを抑える
- テストデータのセットアップ・後始末は、3件以上のテストで同じ準備を繰り返す場合にfixtureやヘルパーへ共通化する。
  1〜2件だけで使うデータはテスト内に置き、読み手が入力と期待値を確認できる形を保つ

## private関数の直接テスト禁止

- private関数（モジュール・クラス内でのみ参照される関数）を直接テストせず、公開インターフェース経由で
  まとまった機能・一連の流れの形で書く
- private関数内のロジックの検証手段は以下の優先順で検討する（新規テスト作成・既存直接テストの移行とも共通）
  1. 公開インターフェース経由の結合テスト（呼び出し元と、同一モジュール内の同等機能を提供する他公開関数も網羅探索）
  2. 引数注入による制御（「簡潔さ」節の方針通り）
  3. 設計判断としてのpublic化（当該関数が単体で公開する価値を持つ場合に限り採用する）
     - テスト都合のみのpublic化は実装詳細の公開となり、リファクタリング耐性低下とSSOT違反を招くため禁止する
  4. 直接テスト（言語ランタイム機構など上記のいずれも成立しない場合に限り、最小限の範囲で許容する）
- テストのリファクタリング・移行・共通ヘルパー化では、検証内容（観測可能なステータス・分類・分岐網羅など）を維持する
  - 検証粒度の縮小（冗長な検証の削減など）が必要な場合は、副作用としてでなく意図した変更として実施する
- 同ファイル内に既存違反パターンが存在しても、新規追加は本節の禁止に従う。既存パターンとの整合は新規違反の正当化にならない

## 失敗テストの除外による回避禁止

失敗するテストを通すために対象機能をテスト対象から除外する回避策は採用しない。対象は次のとおり。

- テストケース削除・`@pytest.mark.skip`・`skipif`・`xfail`等のスキップマーカー付与
- 対象集合の縮小（テスト対象リストからの除外・別定数への分離など）
- アサーション緩和（失敗観点の削除・期待値を実出力に合わせる書き換えなど）

テストが失敗する場合は対象機能側・テスト設計側のいずれかに根本原因があり、根本原因を特定して修正する。
環境依存・タイミング依存などのflakyテストは「安定性: flakyテスト防止」節に従い決定論的に修正する。

## 安定性: flakyテスト防止

- 現在時刻・乱数・UUIDなど非決定的な値は引数やDIで注入可能にしてテストでは固定値を渡す。
  sleepはイベント・コールバック・ポーリングなど確定的な待機手段を実装できない場合の最終手段とする
- ファイルシステム・ネットワーク・外部サービスへの依存は一時ディレクトリやインメモリー実装で代替し、
  並行実行を前提にポート番号・ファイルパスの競合が起きない設計にする
- 環境依存（OS、ロケール、タイムゾーン等）で結果が変わるテストは、テスト内で条件を固定する

## 速度

- コストの高いセットアップはテストスイート内で1回だけ実行し共有する（テスト間の状態依存を持ち込まないよう注意）
- テストデータは最小限にとどめ、sleepを使う場合は0.1秒単位の最小限とし、関数全体でも1秒以内を目標とする
- 実インタプリタ・外部プロセスの反復起動を避ける設計を優先し、
  CLIスクリプトを検証するテストが多数の呼び出しを要する場合は共有ランナー等で起動回数自体を抑える設計を検討する
  - 外部プロセス起動を完全に避けることまでは求めない
- テストスイート全体・個別テストの実行時間には固定上限を設けない。プロジェクト規模・テスト性質で適正値が変わり、
  閾値固定は誤検出を招くため機械チェック化はしない（上記のsleep使用時の1秒以内目安は個別関数の設計指針であり、
  本項が指す上限方針とは別枠として扱う）
  - `--durations`出力の上位に現れる極端に遅いテストは、個別に原因を調査し設計を見直す
