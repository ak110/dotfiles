# Svelte／SvelteKit記述スタイル

対象バージョン: Svelte 5系・SvelteKit 2系（参考実利用バージョン: Svelte 5.56・SvelteKit 2.65）。

## runesの使い分け

- リアクティブな値は`$state`で宣言する（runesモードではトップレベル`let`の単独宣言はリアクティブにならず、`$state`が明示的な宣言経路となるため）
- 他の状態から導出できる値は`$derived`（複雑な導出は`$derived.by`）を用い、`$effect`内での代入で同期しない（`$effect`によるstate同期は循環更新・不要な再計算・無限ループを招く公式のアンチパターンであるため）
- `$effect`は分析ログ送信・DOM直接操作・外部ライブラリ同期など副作用の逃げ道に限定し、状態の同期・派生値の計算には用いない。Svelte公式ベストプラクティスが`$effect`を「escape hatch」と明記し、頻用を避けるよう求めているためである
- コンポーネントのprops宣言は`$props()`分割代入を使い、Svelte 4の`export let`を使わない（`export let`はレガシーモード専用の構文であり、runesモードでは`$props`が唯一の宣言経路のため）
- リアクティブ文`$:`は使わず、派生値は`$derived`、副作用は`$effect`に置き換える（`$:`は依存追跡が暗黙的でrunesモードでは非対応のため）
- ストアは複数コンポーネント間で共有すべき真にグローバルな状態にのみ使い、コンポーネントローカルな状態管理へストアを乱用しない（runes導入によりローカル状態は`$state`で完結し、ストア経由の間接参照は不要な複雑さを増やすため）
- 外部ライブラリとのDOM同期は`$effect`内の手続き的な同期処理よりも`{@attach ...}`を優先する（公式ベストプラクティスが`$effect`より`{@attach}`を明示的に推奨しているため）

## スニペットとコンポーネント分割

- テンプレート内で繰り返す表示断片は別コンポーネントに分割せず`{#snippet}`で定義し`{@render}`で描画する。コンポーネント分割はモジュール境界・propsの型定義・importコストを伴うのに対し、スニペットは同一ファイル内で完結し軽量なためである
- コンポーネント分割は状態のカプセル化・再利用範囲がファイルを越える場合・独立したライフサイクル（`$effect`等）を持つ場合に限定する。スニペットは呼び出し元のスコープを共有するに過ぎず、独自の状態やライフサイクルを持てないためである
- コンポーネントタグ内に直接記述した内容は暗黙の`children`スニペットとして扱われることを理解し、複数の挿入位置が必要な場合のみ名前付きスニペットを追加する。暗黙`children`のみで完結する場面まで名前付きスニペットを増やすと、呼び出し側の記述が冗長になるためである
- ネストコンテンツは暗黙の`children`スニペットとして`$props()`で受け取る。明示的に渡す通常propへ`children`という名前を使わない（ネストコンテンツ由来の暗黙`children`と割り当てが衝突するため）
- 受け取ったスニペットは必ず`{@render name()}`で描画し、`{name}`のような直接展開をしない（直接展開はスニペット本体を文字列化するだけで意図した描画にならず、公式が明示するエラーパターンであるため）

## SvelteKitのload関数

- データベース・ファイルシステムへの直接アクセスや秘密の環境変数を要する処理は`+page.server.ts`/`+layout.server.ts`のserver load関数に置く。server loadはサーバー上でのみ実行され、秘密情報がクライアントへ漏洩しないためである
- 外部APIから秘密情報なしで取得できるデータや、シリアライズ不可な値（コンポーネントコンストラクタ等）を返す場合は`+page.ts`/`+layout.ts`のuniversal load関数を使う。universal loadは初回SSR・hydration後はブラウザでも実行され、サーバー往復を省略できるためである
- 同一ルートにserver loadとuniversal loadが共存する場合、server loadの戻り値がuniversal loadの`data`引数として渡される順序を前提に設計する。SvelteKitの実行順序仕様であり、誤った前提は`data`未定義エラーを招くためである
- フォーム送信・データ変更操作は`fetch`によるAPI呼び出しではなくform actions（`+page.server.ts`の`actions`）を優先する。form actionsはプログレッシブエンハンスメントとCSRF対策に標準で対応し、JavaScript無効環境でも動作するためである
- APIキー・DB接続文字列などの秘密情報はサーバー専用モジュール（`+page.server.ts`・`+layout.server.ts`・`+server.ts`・サーバーフック・`$lib/server`配下）でのみ扱う。universal load（`+page.ts`）・クライアント側コードへ秘密情報を渡さない（サーバー専用モジュール以外はブラウザでも実行され得るためである）

## SSR前提の実装

- `window`・`document`・`localStorage`等のブラウザ専用APIへのアクセスは、実行環境を判定してからにする。コンポーネントはSSR時にNode環境で最初にレンダリングされ、ブラウザAPI不在によりサーバー側で例外が発生するためである
- ブラウザAPIアクセスを`$effect`内に置く場合でも`if (browser) {...}`のような判定で囲まない。`$effect`はサーバーでは実行されないため判定自体が不要であり、公式ベストプラクティスが明示的に非推奨としているためである
- `$effect`外でブラウザAPIに触れる箇所（トップレベルスクリプト・`$derived.by`等）は`$app/environment`の`browser`定数で判定する。トップレベルコードとテンプレート評価はSSR時にも実行されるためである

## セキュリティ

- `{@html}`へユーザー入力・外部データ由来の未検証文字列を渡さない（`{@html}`は既定のエスケープ処理を回避するため、渡した文字列に含まれる`<script>`等がそのまま実行されXSSにつながる）
- `{@html}`を使う場合は信頼できるソース（自身が生成した静的コンテンツ・サニタイズ済みMarkdown変換結果等）に限定し、サニタイズ処理を経由しないユーザー入力を直接渡さない。サニタイズ未経由の経路はレビュー時に見落とされやすい。後日の入力経路変更で脆弱性化しやすいためでもある

## 状態共有

- サーバー側のモジュールスコープ変数・グローバル変数にユーザー固有データを保持しない。サーバーはステートレスかつ長寿命プロセスとして複数ユーザーに共有されるため、あるユーザーが送信したデータを別ユーザーが閲覧する事故につながる
- load関数内でストア・グローバル状態への書き込みを行わない。load関数は純粋関数として扱うべきであり、SSR中の書き込みは全ユーザーで共有される領域を汚染するためである
- ユーザー固有の状態は`cookies`で認証したうえでデータベースへ永続化し、サーバーのメモリー上の変数に保持しない。サーバー再起動でのデータ消失とユーザー間のデータ漏洩の両方を避けるためである
- SSRを伴うcontextベースの状態は子から親への値伝播に依存する設計を避け、状態は親から子へ渡す。SSR中の深い階層での更新はレンダリング済みの親コンポーネントへ反映されず、hydration時の表示ちらつきの原因になるためである
