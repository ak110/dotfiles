# Alpine.js記述スタイル

対象バージョン: Alpine.js v3系。

## 適用範囲の判断

- Alpine.jsは既存HTMLへの軽量な振る舞い付与に限定して使う（マークアップに直接ロジックを組み込む設計思想であり、ルーティング・状態管理基盤を要するSPA的用途ではVue/Reactを選びAlpineの使用を避ける根拠となるため）
- ページ全体を1つの巨大な`x-data`で覆わず、機能単位で`x-data`ブロックを分割する（Alpineは軽量な振る舞い付与用の粒度を想定しており、コンポーネント境界が曖昧だとリアクティブ依存の把握が困難になるため）

## x-data・Alpine.data()の使い分け

- 単純・単発のインライン状態は`x-data="{ open: false }"`のオブジェクトリテラルで直接記述する（記述量が少なく可読性を保てるため）
- 複数箇所で再利用するロジックや、フィルタ処理・複雑な条件式を伴うロジックは`Alpine.data()`で名前付き登録し`x-data="dropdown"`のように参照する。重複記述を避けテスト可能な単位に分離できるためである。公式CSPガイドも複雑な式のx-data直書きを避け、`Alpine.data()`への抽出を推奨している
- 複雑な真偽値判定はインライン式に書かず`Alpine.data()`内のgetterへ切り出す（`x-show="users.filter(...).length > 0"`のような式は可読性・保守性を損なうため）
- `Alpine.data()`の登録は`alpine:init`イベントリスナー内で行う（Alpine起動前に登録を完了させる必要があるため）

## x-if/x-showの使い分け

- CSSの`display: none`のみで実現できる高頻度トグル（ドロップダウン・タブ切替等）は`x-show`を使う（DOM追加削除コストを避け表示切替を軽量に保てるため）
- 要素をDOMから完全に排除したい場合（初期化コストの高い子要素・`x-if`内でのみ有効化したいプラグイン連携等）は`<template x-if="...">`を使う（`x-show`はCSSで隠すだけでDOM上に要素が残り続けるため）
- `x-if`は対象要素へ直接付与せず`<template>`タグで囲んで使う（Alpineが除去済み要素を記録し再出現時に復元する仕組み上、`template`ラップが前提となるため）

## x-forとkey指定

- `x-for`は`<template>`タグに付与し、直下に単一のルート要素を置く（Alpineの反復描画機構が`template`要素を前提とするため）
- 配列の並び替え・挿入・削除が発生し得るループでは`:key`にID等の一意な値を指定する（キー未指定だとAlpineがDOM要素を誤って再利用し、フォーム入力値やイベントリスナーの紐付けがずれる不具合を招くため）

## CSP対応・セキュリティ

- Content Security Policyで`unsafe-eval`を許可できない環境では標準ビルドではなく`@alpinejs/csp`ビルドを使う。標準ビルドはテンプレート式の評価に`eval`相当の機構を使うため、CSPの`script-src`に`nonce`ベースの`strict-dynamic`のみを許可する構成と両立しない
- CSPビルド利用時は複雑なインライン式を避け、ロジックを`Alpine.data()`で登録した関数・getterへ抽出する（CSPビルドの評価器はJavaScript式の安全なサブセットのみを解釈し、未対応の複雑な式は動作しないため）
- `x-html`はユーザー入力・外部由来の信頼できないデータへ使わない（`innerHTML`へ直接反映するためXSSに直結し、公式ドキュメントもサードパーティ由来のHTMLを動的レンダリングしないよう明記しているため）。テキストの表示には自動エスケープされる`x-text`・テキスト補間を使う
- `$el.insertAdjacentHTML()`等のDOM直接操作による代替HTML注入も同様に避ける（`x-html`と同一のXSSリスクを持ち、CSPビルドでは実際にサポート対象外とされているため）。HTML挿入が必要な場合はサニタイズ処理を経由した信頼済み文字列に限定する

## 初期化タイミングとFOUC対策

- Alpineの読み込みスクリプトタグには`defer`属性を付与する（DOM解析完了後に初期化させ、未定義要素への早期アクセスを防ぐため）
- Alpine起動前に外部プラグイン登録・`Alpine.data()`登録・`Alpine.store()`登録をする場合は`alpine:init`イベントで実行する。当該イベントはAlpineが内部初期化を始める直前に発火し、登録漏れなく反映されるためである
- 初期状態でAlpine未処理のDOMがちらつく場合は対象要素に`x-cloak`属性を付与し、`[x-cloak] { display: none !important; }`をCSSへ用意する。Alpineが初期化完了時に`x-cloak`属性を除去する仕組みを利用し、未処理DOMの一瞬の露出＝FOUCを防げるためである

## ストア（Alpine.store）とコンポーネント間通信

- コンポーネント境界を越えて共有する状態は個別コンポーネントの`x-data`に持たせず`Alpine.store()`へ集約する。Alpineにはコンポーネント間の直接的なprops/emit機構がなく、ストアが公式に案内される唯一のグローバル状態共有手段であるためである
- ストアはオブジェクト単位（状態＋操作メソッド）で登録し、`$store.name.method()`のようにメソッド経由で更新する。値の直接代入だけに頼ると更新ロジックが呼び出し側へ分散し保守性が低下するためである。単一値のみで完結する場合は`Alpine.store('darkMode', false)`のような単一値登録も許容する
- ストア登録も`alpine:init`イベント内で行う（コンポーネント初期化時点でストアが未登録だと参照エラーになるため）
