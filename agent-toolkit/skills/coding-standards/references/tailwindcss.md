# Tailwind CSS記述スタイル

対象バージョン: Tailwind CSS v4系（参考実利用バージョン: 4.3）。

## セットアップ・設定（v4のCSS-first設定）

- `tailwind.config.js`を新規作成せず、CSSファイル内の`@theme`ディレクティブでデザイントークンを定義する（v4のCSS-first設定が標準であり、JS設定ファイルは互換目的の`@config`経由でのみ許容されるため）
- エントリCSSは`@tailwind base/components/utilities`ではなく`@import "tailwindcss";`の1行で読み込む（v3以前の3行構文はv4で廃止されたため）
- 色・フォント・ブレークポイント・イージング等のトークンは`@theme { --color-brand-500: oklch(...); }`のようにCSS変数として定義する。生成される全ユーティリティとネイティブCSS変数の双方から同一トークンを参照できるためである
- 既定のカラーパレットを丸ごと置き換える場合は`--color-*: initial;`で初期化してから独自パレットを定義する（既定色との混在による命名衝突・意図しない残存を避けるため）
- ダークモードの判定基準を変更する場合は`@custom-variant dark (&:where(.dark, .dark *));`のようにCSSで`dark`バリアントを再定義する。v4では`darkMode`設定キーではなくCSS側で変体を制御するためである
- プレフィックスが必要な場合は`@import "tailwindcss" prefix(tw);`を使う（新規設定はCSS側の`prefix()`へ統一し、`@config`経由の互換設定には頼らないため）

## ユーティリティファースト運用

- スタイリングはユーティリティクラスの直接記述を第一選択とし、`@apply`によるコンポーネントクラス抽出は多用しない（ユーティリティの意図が失われ、HTMLとCSSの二重管理が発生するため）
- 同一の見た目パターンが3箇所以上で反復する場合に限り、`@apply`ではなくコンポーネント分割（Reactコンポーネント・Vueコンポーネント等）で共通化する（マークアップとスタイルの対応関係を1箇所に保てるため）
- CSSモジュールやVue/Svelteの`<style>`ブロックなど別バンドル経路のスタイルでは、`@apply`より`var(--color-blue-500)`のようなCSS変数の直接参照を優先する。当該経路で`@apply`・`@variant`を使うには`@reference`によるテーマ読み込みが別途必要になり、ビルド時のTailwind処理でビルド性能も下がるためである（CSS変数の直接参照は`@reference`不要）

## クラス整列

- `prettier-plugin-tailwindcss`を導入し、クラス順序の整形をフォーマッタに委ねる（クラス順序に関する議論・レビュー往復をなくし、diffノイズを削減するため）
- 手動でのクラス順序調整はしない（フォーマッタの出力と衝突し、保存のたびに差分が発生するため）

## 動的クラス名の禁止

- `` `bg-${color}-600` ``のような文字列連結・テンプレートリテラルでクラス名を生成しない（Tailwindはソースファイルをプレーンテキストとしてスキャンするため、実行時にしか確定しないクラス名はCSS生成の対象から漏れるため）
- 条件分岐でクラス名を切り替える場合は完全なクラス名を分岐の両方に書く（例: `color === "red" ? "bg-red-600" : "bg-green-600"`）（クラス名全体が静的に出現していればスキャナが検出できるため）
- コンポーネントのバリアント管理には、完全なクラス名の対応表（オブジェクトリテラル等）を用いる（動的合成を避けつつ再利用性を確保できるため）

## レスポンシブ・ダークモード・カスタムテーマの推奨パターン

- レスポンシブは`md:`・`lg:`等のブレークポイントプレフィックスを基準スタイルに追記する形で書く（モバイルファーストの既定ブレークポイントに合わせ、上書き対象を明確にするため）
- ダークモードは`dark:`バリアントをベースユーティリティへ直接併記する（`dark:lg:hover:bg-indigo-600`のように他バリアントとも合成できるため、専用のダークモード用CSSファイルを分離しない）
- カスタムブレークポイント・カスタムイージング等は`@theme`内で`--breakpoint-3xl: 120rem;`のように命名規則（`--breakpoint-*`・`--color-*`等）に沿って追加する。既定トークンと同じ生成規則に乗り、命名の一貫性を保てるためである

## アンチパターン

- `[mask-type:luminance]`のような任意値記法を汎用スタイルの常用手段にしない（デザイントークンから外れた値が増えると、テーマの一貫性が失われ再利用性も下がるため）。同じ値を繰り返し使う場合は`@theme`のトークンへ昇格する
- `!important`相当の`!`修飾子（例: `text-red-500!`）を特異度問題の恒常的な回避策にしない（根本原因であるセレクタ設計・読み込み順序を見直さずに済ませてしまうため）。特異度問題が生じた場合はセレクタ設計・読み込み順序を見直す
- 巨大なクラス文字列を1行にまとめて記述したままレビューしない（`prettier-plugin-tailwindcss`の整形結果を前提に、レイアウト・配色・状態という意味のある単位で改行・グルーピングして可読性を保つ）
