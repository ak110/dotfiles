# Bash記述スタイル

- 基本構造
  - 冒頭に`set -euo pipefail`を記述する（エラー終了・未定義変数の検出・パイプ失敗の伝播を有効化する）
  - 実行シェルの指定は`#!/usr/bin/env bash`とする（PATHからbashを解決して移植性を確保するため）
  - スクリプト自身の位置は`BASH_SOURCE`から解決する（カレントディレクトリや`$0`に依存しないため）
    - 例: `script_dir="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"`
- 変数とクォート
  - 変数参照とコマンド置換はダブルクォートで囲む（語分割とグロブ展開を防ぐため）
  - 未クォート参照はshellcheckが`SC2086`等で検出する
- エラーハンドリング
  - 一時ファイル・一時ディレクトリは`trap ... EXIT`で確実に後始末する
  - `exec cmd1 || cmd2`は`cmd1`の起動失敗（バイナリ不在等）の場合のみ`cmd2`へフォールバックする。
    `exec`がシェル自体を`cmd1`へ置換するため、`cmd1`の終了コードでは分岐できない。
    終了コードに応じてフォールバックする場合は`exec`を使わず`cmd1 || cmd2`の形で書く
- 安全な削除
  - `rm -rf`はパス取り違え時の被害範囲が広いため、削除範囲を限定できる場合は代替を優先する
    - 空ディレクトリの削除は`rmdir`を使う（想定外のファイルが残ると失敗し安全弁になるため）
    - git管理下のファイル削除は`git rm -r`を使う（ステージング経由で復元できるため）
    - 一時ディレクトリの後始末など対象が明確な場合は、変数の空チェックと`trap`を併用して`rm -rf`を使う
- セキュリティ上の危険パターン
  - 外部入力を`eval`に渡さない（コマンドインジェクションの危険があるため）
  - 外部入力を含むコマンド文字列を組み立てて実行しない
- 他で指定が無い場合のツール推奨:
  - 静的解析: shellcheck
  - フォーマッター: shfmt
