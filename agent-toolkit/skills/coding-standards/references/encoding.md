# 文字エンコーディングの詳細

`coding-standards`スキルの「文字エンコーディング」項目の詳細リファレンス。
日本語環境（Windows/Linux混在）で開発する際の落とし穴と対処をまとめる。

## 基本原則

- 入出力境界でエンコーディングを明示する。内部処理ではUTF-8を基本とする
- ランタイム既定への非依存: 日本語WindowsのACPはCP932であり、
  明示しない場合に文字化けや例外が起きる
- ライブラリ提供フィールドの暗黙加工: stdlib・サードパーティライブラリのフィールドには
  プラットフォーム依存の暗黙加工が含まれうる。
  生データ復元の用途では、加工済みフィールドではなく未加工のフィールド／APIが
  別途提供されていないかを必ず確認する

## BOMの扱い

- 読み込み時はBOM有無を両方許容する
- 書き出し時はBOMを付けない（Excelに渡すCSV等、受け手が要求する場合のみ付ける）

## 日本語テキストの自動判定順序

BOM検出→ISO-2022-JP特有エスケープシーケンスの整合性検証→UTF-8厳格復号の試行→EUC-JP・CP932の順で判定する。
ISO-2022-JPはASCII系バイト列で構成されUTF-8厳格復号にも成功するため、必ずUTF-8より先に検出する。
入手元情報から判断できる場合は自動判定を使わない。

## ZIPファイル名の扱い

General Purpose Bit FlagのBit 11がセットされている場合はUTF-8で復号する。
未設定時の仕様上の基本はCP437であり、作成元が日本語Windows等である根拠がある場合のみCP932をフォールバック候補とする。
Info-ZIP Unicode Path Extra Fieldなど補助情報も確認対象に含める。

Pythonの`zipfile.ZipInfo`を扱う場合、`filename`は`_sanitize_filename`を経由する。
Windowsでは`os.sep`の`\`が`/`に置換されるため、
CP932の2バイト目に`0x5C`を含む文字（ソ・ポ・表など）で往復復元に失敗する。
生バイト列の復元には未加工の`orig_filename`を使う。

## Unicode正規化

比較・検索・一意性判定に使うキーはドメイン要件に応じてNFC等の正規化形式を用いる。
認可・識別キー・パスワード等でNFKCを使うと合字・互換文字経由のバイパスを許す恐れがある。
大文字小文字の畳み込みが必要な場合はNFCで正規化してから`casefold`を適用する
（逆順にすると言語固有の畳み込みが不完全になる）。
HFS+はApple変種のNFD相当でファイル名を保存し、APFSは正規化形式を保持するため、
ファイルシステム・API境界をまたぐ比較では正規化形式の差を考慮する。

## その他

- バイト長と文字長の混同防止: バイト長と文字数（コードポイント数・書記素クラスター数）は別物として扱う
- Unicode特殊文字のセキュリティ: Bidi制御文字（Trojan Source）・ゼロ幅文字・ホモグラフは
  ソースコード・識別子・ユーザー入力・ログで禁止または警告する
