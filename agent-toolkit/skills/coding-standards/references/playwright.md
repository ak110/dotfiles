# Playwrightテスト記述スタイル

対象バージョン: Playwright Test v1系（参考実利用バージョン: 1.60）。

## ロケーター戦略

- 要素選択には`getByRole`・`getByLabel`・`getByText`等のユーザー視点ロケーターを最優先で使う（CSSクラス・XPathベースのロケーターはDOM構造の変更に追従できずテストの安定性が損なわれやすいため）
- `getByRole`を第一候補とし、アクセシビリティツリー上のroleとaccessible nameで要素を特定する（スクリーンリーダー利用者が認識する情報と一致し、実装詳細から独立するため）
- ロケーターの手書きが困難な場合はcodegenツールまたはVS Code拡張機能で生成する（Playwrightが一意性の高いロケーターを自動生成するため）
- ユーザー視点ロケーターで特定できない要素は`getByTestId`（`data-testid`属性）を使う（専用テスト属性はスタイル変更・マークアップ変更の影響を受けないため）。CSSセレクタ・XPathは組み込みロケーターで特定できない場合の最終手段に限定する

## アサーション

- `expect(locator).toBeVisible()`等のweb-first assertionsを使う（ロケーターの状態を条件成立まで自動的にポーリングし、余分な待機コードなしでflakyさを抑えられるため）
- `expect(await locator.isVisible()).toBe(true)`のような非同期待機を伴わない手動アサーションを書かない（判定が1時点のスナップショットに固定され、非同期なUI変化に追従できないため）
- 複数の検証を1テスト内で継続したい場合は`expect.soft()`を使う（1件目の失敗でテストを即座に打ち切らず、以降の検証結果もまとめて収集できるため）
- `page.waitForTimeout()`による固定時間待機は使わない。実行環境の速度差でタイムアウト超過や過剰待機を招き、flakyの根本原因となるためである。状態変化を待つ場合はロケーターの自動待機または`expect().toPass()`を使う

## テスト分離

- 各テストは他テストの実行結果に依存せず独立して実行できる状態にする（並列実行時の順序非決定性に対して再現性を保ち、失敗時の原因切り分けを容易にするため）
- テスト間で共有する前提状態は`beforeEach`・`afterEach`等のフックで都度構築・破棄する（テスト間の暗黙的な依存を避けるため）
- ログイン等の認証状態は各テストで都度実行せず、専用のsetup projectで1回だけ認証し`storageState`へ保存して他のprojectから再利用する。認証処理の重複実行を避けつつテスト独立性を維持できるためである。`playwright.config.ts`の`projects`配下で`dependencies: ['setup']`を指定する構成が公式に推奨される
- 自身が制御しない外部サイト・サードパーティサーバーへのリンク遷移をテストしない（可用性・応答時間が自チームの管理外でありテストの信頼性を損なうため）。必要な場合はNetwork APIでモックする

## flaky対策

- Playwrightのロケーターが持つ自動待機・actionabilityチェック（可視性・有効状態等の確認後に操作する仕組み）に処理を委ね、独自の待機ループを実装しない
- `retries`設定はCI環境でのみ有効化し、ローカル実行では0のままにする（ローカルでの再試行はflakyの発見を妨げるため）
- `trace: 'on-first-retry'`を設定する（失敗時のみトレースを記録しパフォーマンスへの影響を抑えつつ、CI上のデバッグに必要な情報を確保できるため）
- CI上のデバッグ手段は動画・スクリーンショットよりトレースビューアーを優先する（トレースはローカルPWAとして操作ログ・DOM・ネットワークを時系列で参照でき、失敗原因の特定が容易なため）

## Page Object Model採用の判断基準

- テストスイートが小規模、またはロケーター・操作列の再利用箇所が少ない場合はPage Object Modelを導入せずテスト本文へ直接記述する（抽象化層の維持コストが再利用効果を上回るため）
- テストスイートが大規模化し同一画面へのロケーター参照・操作列が複数テストで重複する場合はPage Object Modelを導入する（要素セレクタを1箇所へ集約し、UI変更時の修正箇所をページオブジェクト側に限定できるため）
- 公式ドキュメントはPage Object Modelを唯一の正解とはせず、構造化するための1つの手法として位置づける。採用有無はコードベースの規模と重複度に応じて判断する

## 並列実行・CI設定

- Playwright Testは既定で並列実行される。テスト同士の独立性（前節「テスト分離」参照）を保つことで並列実行時の相互干渉を防ぐ
- CI環境ではコミット単位など高頻度で実行する（回帰の検知を早め、原因切り分けの対象コミット数を限定できるため）
- 実行時間が長いスイートは`--shard=1/N`形式のシャーディングで複数マシンへ分割する（CI全体の所要時間を短縮できるため）
- Chromium・Firefox・WebKitへ対応する場合は`projects`設定でブラウザごとにプロジェクトを分離する（単一設定へ条件分岐を持ち込まず構成を宣言的に保てるため）
- `@typescript-eslint/no-floating-promises`等のリントルールを適用する（`await`の付け忘れによるアサーション未実行・操作の競合状態を静的に検出できるため）
- Playwright本体は定期的に最新版へ更新する（同梱ブラウザの更新を継続的に取り込み、実運用ブラウザとの挙動差を防ぐため）
