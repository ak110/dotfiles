# コーディング品質・テスト方針

コード編集に着手する前に、編集対象の拡張子に対応する`references/<言語>.md`を必ずReadで読み込む。
軽微な変更でも省略しない。対応ファイルが無い場合は共通品質のみで進める。

言語別リファレンス:

- Python: `references/python.md`
- TypeScript/TSX: `references/typescript.md`
- Rust: `references/rust.md`
- C#: `references/csharp.md`
- Bash/sh・bash・sh.tmpl: `references/bash.md`
- PowerShell/ps1・ps1.tmpl・psm1・psd1: `references/powershell.md`
- Windowsバッチ/cmd・bat: `references/windows-batch.md`
- Dockerfile: `references/dockerfiles.md`
- GitHub Actionsワークフロー（`.github/workflows/*.yaml`）: `references/github-actions.md`

トピック別リファレンス:

- 文字エンコーディング詳細（日本語環境・ZIPファイル・Unicode正規化等）: `references/encoding.md`

## コーディング品質（全言語共通）

プロジェクトや言語の特性に応じて適切に判断する一般論。

- トップダウンの記述順序を遵守する
  - 型定義や上位の関数から先に書き、関数Aから関数Bを呼び出す場合はAを前にBを後ろに定義する
  - 関連するものは近くにまとめ、定義順を保つ
  - 追記時は機械的に末尾へ追加せず、既存の定義順序に沿った位置へ挿入する
- 計算量: データ構造とアルゴリズムの計算量を意識し、入力サイズの増大に耐える実装を選ぶ
- 浮動小数点: 金額など正確さが必要な計算には整数演算や専用の十進型を使う。等値比較は許容誤差で行う
- 日時: タイムスタンプはUTC、市民時刻はタイムゾーン情報を保持する（プロジェクト方針が無い場合）。日時演算にはライブラリを使う
- 文字エンコーディング: 入出力境界でエンコーディングを明示し、内部処理ではUTF-8を基本とする。
  詳細は`references/encoding.md`を参照
- 乱数: 言語標準APIを使い、シードに予測可能な値を使わない。
  - 範囲変換には専用API（`randint`等）を使い、剰余での変換は避ける
  - セキュリティ用途ではCSPRNGを使う
- 整数オーバーフロー: 固定幅整数を扱う言語では、大きな値の演算・型変換時にオーバーフローを考慮する
- リソース管理: ファイルハンドル・DB接続・ロックなどはRAIIやコンテキストマネージャーで確実に解放する
- 並行処理・非同期処理: 共有状態へのアクセスは適切な同期機構で保護する（可能なら共有状態自体を避ける）。
  I/O待ちにはasync/awaitを優先し、awaitの付け忘れに注意する
- セキュリティ: 信頼境界（外部入力・API応答・ユーザー操作など）を常に意識する
  - 入力検証は型・範囲・形式を検証し、許可リスト方式を優先する
  - インジェクション防止: SQL・コマンド・HTMLにユーザー入力を直接結合しない
  - シークレットをソースコードにハードコードしない。ログにも出力しない
  - パストラバーサル: ユーザー入力からのパス構築は正規化して基準ディレクトリ内に収まることを検証する
  - デシリアライゼーション: 信頼できないデータの復元には安全な手段を使う
  - 最小権限: プロセス・トークン・DB接続などの権限は必要最小限にする
  - エラー情報: スタックトレース・内部パスなどの内部情報をエンドユーザーに露出しない
- ロギング: 障害時の原因究明を前提に書く。
  - 構造化ログ・相関IDの伝播を推奨する
  - シークレット・個人情報はログに出力しない
- エラー処理: 例外の種別ごとに方針を決めて扱う
  - 一時的エラー（ネットワーク・ロック競合等）はリトライ対象。指数バックオフ + ジッタを基本とする
  - 永続的エラー（入力不正・権限不足等）は即座に失敗させ、呼び出し元に原因を返す
  - プログラマエラー（不変条件違反）はfail-fastで例外を送出し、捕捉して無視してはならない
  - catchは具体的な型に限定し、広域キャッチは境界層に限る。リトライ回数・タイムアウトは設定可能にする
- 設定ファイルへの変更に専用CLIがある場合は手編集より優先する（`uv add`/`pnpm`/`mise`等）
- 依存管理: サプライチェーン攻撃対策とセキュリティ維持の観点から運用する
  - 依存追加・更新は上記の専用CLIに従い、ロックファイルとマニフェストを同期させる
  - 公開直後の新バージョンは一定期間（目安1日）待ってから導入する
   （ツール例: uvの`exclude-newer`、pnpmの`minimum-release-age`）
  - バージョン固定は管理コスト増大のため原則採用しない。上記の待機設定で代替する
  - SCAツールで定期的に脆弱性を確認する（`pip-audit`/`pnpm audit`/`cargo-audit`等）
  - 依存追加時はメンテナンス状況・代替の有無を確認する。使われていない依存は削除する
  - ロックファイルを尊重する: CI/Makefileでは依存解決を再実行せずロックファイルをそのまま使う
    - 環境変数で一括適用する（uvの`UV_FROZEN=1`、pnpmの`--frozen-lockfile`等）
    - 依存更新コマンドのみ一時解除する。開発者個人のシェルには設定しない
  - 依存追加・更新コマンドの出力警告を見逃さない。
    パッケージマネージャーの仕様変更・非推奨化が警告で告知されることがあり、放置すると設定が無効化される
    - pnpm v11以降は`package.json`の`pnpm`フィールドが読み込まれない。
      `overrides`等の設定は`pnpm-workspace.yaml`へ移行する。
      該当時は`pnpm install`実行中にフィールド非読み込みの警告が出るため、依存を更新するときに確認する
- 自動生成ファイルは直接編集しない
  - 対象例: 依存ロックファイル（`uv.lock`・`pnpm-lock.yaml`・`Cargo.lock`等）、
    自動生成コード（`*_pb2.py`・OpenAPI生成クライアント等）、ビルド成果物、補完スクリプト
  - 更新は生成元（`pyproject.toml`・`package.json`・`.proto`等）または専用CLI
   （`uv add`・`pnpm add`・`cargo update`等）経由で行う
  - 直接編集が必要と判断される場合は、生成元側の設定変更で表現できないかをまず検討する
- lintエラーへの対応は以下の優先順で検討する
  1. コード修正 -- ルールが正当な問題を指摘している場合
  2. コードの書き方の変更 -- del・assert・型の明示など、軽微な変更で解消できる場合
  3. インライン無視コメント -- ユーザーによる確認必須。
     コードが意図的で、その箇所だけルールの当てはまらない場合に使う。
     理由が自明でなければ理由を併記する
  4. 設定ファイルでの抑制 -- ユーザーによる確認必須。
     ルールがプロジェクト方針と合わない、または同一の抑制が繰り返し必要になる場合
  - 振る舞いを変えずlintの指摘だけを回避するコード変更は避ける
  - プロジェクト方針でlinter／formatterのファイル個別実行がサポートされている場合は、
    変更対象ファイルへ事前実行し自動修正可能な指摘をその場で対応する（具体コマンドはプロジェクト方針に従う。例: `pyfltr`等）
- コマンドラインオプションの書式（プロジェクト方針が無い場合）:
  - ロング形式（`--foo`）を優先する。
    除外: ロング形式が存在しないもの、シェル組み込みの慣用的オプション（`set -euo pipefail`等）、
    広く定着した複合フラグ（`curl -fsSL`等）
  - 引数付きの場合はスペース区切りより`=`区切りを優先する
    - 例: `--config=path/to/config`、`--config="path/to/config with space"`
    - subprocessのargv配列形式も同方針（`["cmd", "--scope=user"]`）
    - 除外条件:
      - ショートオプションで`=`形式を取らないもの（`git commit -m "..."`等）
      - `=`を受け付けないBSD系コマンド
      - フラグの直後に位置引数が続くパターン（`uv tool install --editable .`等）

## テスト方針（全言語共通）

新しいpublicな関数などを書く際は必ずテストコードも書く。簡潔さ・安定性・速度を重視する。

### 簡潔さ

- テストダブル（モック・スタブ・モンキーパッチ等）は極力避けて実コードを通す
  - 依存の差し替えが必要な場合は引数注入を優先する（テスト対象にオプション引数を追加し、デフォルト値で本番動作を維持する）
  - テストダブルの使用は、引数注入では制御できないグローバル状態や言語ランタイム機構のテストに限定する
- 単純過ぎるテストは避け、まとまった機能のテストを重視する。
  CRUDなどの一連の流れや、やむを得ない単体テストは原則1つのテスト関数にまとめ、
  フィクスチャ初期化・接続確立などのセットアップオーバーヘッドを抑える
- テストデータのセットアップ・後始末は可能な限り共通化する

### private関数の直接テスト禁止

- private関数（モジュール・クラス内でのみ参照される関数）を直接テストしない。
  テストは公開インターフェースを通じて、まとまった機能・一連の流れの形で書く
- テストのために関数をpublic化することも禁止する。
  モジュール内で完結する実装詳細を公開すると、リファクタリング耐性が下がりSSOT違反を招く
- 既存テストがprivate関数を直接テストしている場合、public化ではなく公開インターフェース経由の結合テストへ移行する
- 引数注入で制御できる場合は引数注入を優先する（`### 簡潔さ`の方針通り）
- 例外: 言語ランタイム機構や引数注入では到達不能なロジックに限り、最小限の範囲で許容する

### 安定性: flakyテスト防止

- 現在時刻・乱数・UUIDなど非決定的な値は、引数やDIで注入可能にしてテストでは固定値を渡す
- sleepは極力避け、イベント・コールバック・ポーリングなど確定的な待機手段を優先する
- ファイルシステム・ネットワーク・外部サービスへの依存は、一時ディレクトリやインメモリー実装で代替する
- 並行実行を前提に、ポート番号・ファイルパスの競合が起きない設計にする
- 環境依存（OS、ロケール、タイムゾーン等）で結果が変わるテストは、テスト内で条件を固定する

### 速度

- コストの高いセットアップはテストスイート内で1回だけ実行し共有する（テスト間の状態依存を持ち込まないよう注意）
- テストデータは最小限にとどめる
- sleepを使う場合は0.1秒単位の最小限とし、関数全体でも1秒以内を目標とする
