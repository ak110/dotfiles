# コーディング品質

判断の一貫性を維持するため、コード・テストコードの品質規範を集約する。

## 適用範囲

全プロジェクト・全言語横断のコーディング規範のみを記述する。
特定言語・特定フレームワーク・特定ライブラリ固有の事項は記述せず、`references/<言語>.md`または`references/<フレームワーク・ライブラリ名>.md`へ置く。
プロジェクト固有の規約は当該プロジェクトの`CLAUDE.md`・`.claude/rules/`へ置く。

コード編集に着手する前に、編集対象の拡張子に対応する`references/<言語>.md`を必ずReadで読み込む
（変更規模を問わず省略しない。対応ファイルが無い場合に限り共通品質のみで進める）。
プロジェクトの依存定義（`package.json`等）または`<script>`読み込み等の技術利用痕跡から対象技術の利用が判明する場合は、対応する`references/<フレームワーク・ライブラリ名>.md`も読み込む。

言語別リファレンス:

- Python: `references/python.md`と`references/python-references.md`
- TypeScript/TSX: `references/typescript.md`
- Rust: `references/rust.md`
- C#: `references/csharp.md`
- Bash/sh・bash・sh.tmpl: `references/bash.md`
- PowerShell/ps1・ps1.tmpl・psm1・psd1: `references/powershell.md`
- Windowsバッチ/cmd・bat: `references/windows-batch.md`
- Dockerfile: `references/dockerfiles.md`
- GitHub Actionsワークフロー（`.github/workflows/*.yaml`）: `references/github-actions.md`

トピック別リファレンス:

- 文字エンコーディング詳細（日本語環境・ZIPファイル・Unicode正規化等）: `references/encoding.md`
- 単体HTML成果物（ユーザーへ単体で提示するレポート・ダッシュボード等）: `references/independent-html.md`
- テスト方針（テストコードの作成・修正時に必読）: `references/testing.md`

フレームワーク・ライブラリ別リファレンス:

- Tailwind CSS（v4系、依存名`tailwindcss`）: `references/tailwindcss.md`
- Alpine.js（v3系、依存名`alpinejs`または`<script>`読み込み）: `references/alpinejs.md`
- Playwright Test（v1系、依存名`@playwright/test`）: `references/playwright.md`
- Drizzle ORM/Drizzle Kit（0.x系、依存名`drizzle-orm`・`drizzle-kit`）: `references/drizzle.md`
- Svelte 5/SvelteKit 2（依存名`svelte`・`@sveltejs/kit`）: `references/svelte.md`

## コーディング品質（全言語共通）

以下の規範は、01-agent.md「品質最優先」原則の具体例である（網羅規定ではなく、列挙外の場面でも原則に従う）。

- 実装に着手する前に、以下の順で検討する実装ラダーを適用する
  - 上位で成立する選択肢が見つかった時点で下位の検討は打ち切る
  - 検討順序:
    1. そもそも必要か（YAGNI）
    2. 標準ライブラリで実現できるか
    3. プラットフォーム標準機能（ブラウザAPI・OS提供機能等）で実現できるか
    4. インストール済みの依存で実現できるか
    5. 1行で書くか
    6. 上記いずれも成立しない場合のみ最小限の独自実装
  - ラダー対象外: 信頼境界検証・データ損失防止・セキュリティ・アクセシビリティ・
    ハードウェア較正に関わる実装はラダーで省略せず必要な堅牢性を確保する
- 実装ラダーで意図的に簡略化した箇所には、簡略化マーカーをコメントで残す
  - 形式: `# 簡略化: <簡略化の内容>, 既知の限界: <制約>, 見直し契機: <再評価のトリガー>`
   （行コメント記号は言語慣用に従う。例: TypeScriptは`// 簡略化:`、PowerShellは`# 簡略化:`）
  - 既知の限界と見直し契機は、コードの読み手が後から再評価するための情報であり省略しない
  - 恒久放置を意図せず、見直し契機の事象が発生した時点で再評価する明示的負債として扱う
  - 棚卸しは`grep -rn '# 簡略化:'`等で代替し、専用CLI・モード切替は導入しない
- トップダウンの記述順序を遵守する。型定義や上位の関数から先に書き、関連するものは近くにまとめ、追記時は
  機械的に末尾へ追加せず既存の定義順序に沿った位置へ挿入する
- 既存ファイルへ新規関数・節・テストケース等を追加する場合、追加先ファイル全体の既存命名規約を観察してから追加する。
  観察対象は関数名のASCII・非ASCIIの別、節名の命名パターン、用語選択を含む。同一ファイル内で命名形式を混在させない
- 計算量を意識し、入力サイズの増大に耐える実装を選ぶ
- 浮動小数点: 金額など正確さが必要な計算には整数演算や専用の十進型を使い、等値比較は許容誤差で行う
- 日時: タイムスタンプはUTC、市民時刻はタイムゾーン情報を保持する（プロジェクト方針が無い場合）。日時演算にはライブラリを使う
- 文字エンコーディング: 入出力境界でエンコーディングを明示し、内部処理ではUTF-8を基本とする（詳細は`references/encoding.md`）
- 端末表示・固定幅レイアウト向けの文字列の幅計算・切り詰め・揃え処理は、文字数ではなく表示幅を基準に実装する。
  表示幅の判定基準はEast Asian WidthでW/F/A（曖昧幅を含む）を全角2セル換算、他を1セルとする
- 乱数: 言語標準APIを使い、シードに予測可能な値を使わない。範囲変換には専用API（`randint`等）を使い、
  剰余での変換は避ける。セキュリティ用途ではCSPRNGを使う
- 整数オーバーフロー: 固定幅整数を扱う言語では、大きな値の演算・型変換時にオーバーフローを考慮する
- リソース管理: ファイルハンドル・DB接続・ロックなどはRAIIやコンテキストマネージャーで確実に解放する
- 並行処理・非同期処理: 共有状態へのアクセスは適切な同期機構で保護し（可能なら共有状態自体を避け）、
  I/O待ちにはasync/awaitを優先する
- フレームワーク固有・ライブラリ固有の実装パターンを採用する場合、
  根拠とした公式ドキュメントのURLをコード内コメントへ付記する
- セキュリティ: 信頼境界（外部入力・API応答・ユーザー操作など）を常に意識する
  - 入力検証は型・範囲・形式を検証し許可リスト方式を優先する。SQL・コマンド・HTMLへの直接結合、
    未正規化のパス構築によるパストラバーサル、信頼できないデータの安全でない復元は避ける
  - シークレットはソースコード・ログに出力しない。プロセス・トークン・DB接続などの権限は必要最小限にし、
    スタックトレース・内部パスなどの内部情報をエンドユーザーへ露出しない
- ロギング: 障害時の原因究明を前提に書き、構造化ログ・相関IDの伝播を推奨する
  - シークレット・個人情報はログに出力しない
- エラー処理: 例外の種別ごとに方針を決めて扱う
  - 一時的エラー（ネットワーク・ロック競合等）はリトライ対象。指数バックオフ + ジッタを基本とする
  - 永続的エラー（入力不正・権限不足等）は即座に失敗させ、呼び出し元に原因を返す
  - プログラマエラー（不変条件違反）はfail-fastで例外を送出し、捕捉して無視してはならない
  - catchは具体的な型に限定し、広域キャッチは境界層に限る。リトライ回数・タイムアウトは設定可能にする
- 設定ファイルへの変更に専用CLIがある場合は手編集より優先する（`uv add`/`pnpm`/`mise`等）
- 依存管理: サプライチェーン攻撃対策とセキュリティ維持の観点から運用する
  - 依存追加・更新は上記の専用CLIに従い、ロックファイルとマニフェストを同期させる。
    バージョン固定は管理コスト増大のため原則採用せず、公開直後の新バージョンを一定期間（目安1日）待つ設定で代替する
   （ツール例: uvの`exclude-newer`、pnpmの`minimum-release-age`）
  - SCAツールで定期的に脆弱性を確認する（`pip-audit`/`pnpm audit`/`cargo-audit`等）
  - 依存追加時はメンテナンス状況・代替の有無を確認する。使われていない依存は削除する
  - ロックファイルを尊重する: CI/Makefileでは依存解決を再実行せずロックファイルをそのまま使う。
    環境変数で一括適用し（uvの`UV_FROZEN=1`、pnpmの`--frozen-lockfile`等）、
    依存更新コマンドのみ一時解除する（開発者個人のシェルには設定しない）
  - 依存追加・更新コマンドの出力警告を見逃さない
    パッケージマネージャーの仕様変更・非推奨化が警告で告知されることがあり、放置すると設定が無効化される
    - pnpm v11以降は`package.json`の`pnpm`フィールドが読み込まれない
      `overrides`等の設定は`pnpm-workspace.yaml`へ移行する。
      該当時は`pnpm install`実行中にフィールド非読み込みの警告が出るため、依存を更新するときに確認する
- 自動生成ファイル（依存ロックファイル・`*_pb2.py`等の自動生成コード・ビルド成果物・補完スクリプト等）は直接編集しない
  - 更新は生成元（`pyproject.toml`・`package.json`・`.proto`等）または専用CLI
   （`uv add`・`pnpm add`・`cargo update`等）経由で行う
  - 直接編集が必要と判断される場合は、生成元側の設定変更で表現できないかをまず検討する
- 既存コードの意図的な制約（型制約・アクセス制限・lint設定・不変条件のアサーション等）を新規コードの都合で除去・緩和しない。
  制約の変更が必要と判断した場合は先にその必要性を検証する
- lintエラーへの対応は以下の優先順で検討する
  1. コード修正（ルールが正当な問題を指摘している場合）
  2. コードの書き方の変更（del・assert・型の明示など、軽微な変更で解消できる場合）
  3. インライン無視コメント（ユーザーによる確認必須）: コードが意図的で、その箇所だけルールの当てはまらない場合に使う。
     理由が自明でなければ理由を併記する
  4. 設定ファイルでの抑制（ユーザーによる確認必須）: ルールがプロジェクト方針と合わない、
     または同一の抑制が繰り返し必要になる場合
  - 振る舞いを変えずlintの指摘だけを回避するコード変更は避ける
  - ファイル全体一括の抑制コメント（行数・複雑度・メソッド数等の上限を一括無視するコメント）の
    新規追加は採用しない。該当ファイルが過大化している場合は責務境界・シナリオ群でのファイル分割を優先する
  - 抑制コメントの新規追加時は、同ファイル内・同種ファイルにある既存の抑制コメントとは独立に本節の判断基準を適用する
  - プロジェクト方針でlinter／formatterのファイル個別実行がサポートされている場合は、
    変更対象ファイルへ事前実行し自動修正可能な指摘をその場で対応する（具体コマンドはプロジェクト方針に従う。例: `pyfltr`等）
- コマンドラインオプションの書式（プロジェクト方針が無い場合）:
  - ロング形式（`--foo`）を優先する。除外はロング形式が存在しないもの、シェル組み込みの慣用的オプション
    （`set -euo pipefail`等）、広く定着した複合フラグ（`curl -fsSL`等）
  - 引数付きの場合はスペース区切りより`=`区切りを優先する（例: `--config=path/to/config`、
    subprocessのargv配列形式も同方針で`["cmd", "--scope=user"]`）。
    除外はショートオプションで`=`形式を取らないもの（`git commit -m "..."`等）、`=`を受け付けないBSD系コマンド、
    フラグの直後に位置引数が続くパターン（`uv tool install --editable .`等）
- CLIメッセージ設計: 利用者向けエラーメッセージ・案内文は次の原則に従う
  - 現行仕様のみで理解できる文面とし、過去仕様・変更経緯・廃止済み引数名に依存する表現を書かない
  - 引数誤用の検出はパーサ標準のusage表示（argparseなら`parser.error()`相当）で終了する。
    独自の使用例文字列はオプション追加時に実際のusageと乖離するため組み立てない
  - エラー本文にはmetavar・内部識別子ではなく利用者の操作に即した平易な語を使う。
    ヘルプを参照しなければmetavarの意味が分からないため、エラー本文への持ち込みを避ける。
    usage行・ヘルプ内のmetavar標準表示はmetavarの説明そのものであるため対象外とする
- モジュール内および隣接モジュール（同ディレクトリ配下、または共通の公開インターフェースを介する
  呼び出し関係にあるモジュール群）の命名規約・エラー処理方針・返却値構造の一貫性を保つ
- 公開インターフェースを変更する場合、連動追従が必要な次の対象を全件洗い出す。
  公開インターフェースの詳細は`agent-toolkit/rules/05-terminology.md`の「公開インターフェース」定義を参照する
  - 対応する単体テスト・統合テスト・スナップショット
  - 対応するドキュメント（利用者向け・開発者向け）・サンプルコード・出力例
  - 対応する利用元（配布物を利用する側のコード・スクリプト、または内部で公開インターフェースを呼び出すモジュール群）
  - 利用者向けエラーメッセージ・案内文

### 責務帰属（SRP）分析の観点

責務肥大化・SRP違反を点検する場面（リファクタリング検討・改善提案の主提案確定・コード設計レビュー等）で共通基準として用いる。

- 責務帰属: 当該機能の責務がどのコンポーネントに帰属すべきかを問い、現在の配置が責務上正しいかを確認する
- 直接アクセス: 他コンポーネントが対象コンポーネントの内部状態（設定ファイル・フラグファイル・内部API等）へ
  直接アクセスしている構造があるかを点検する
- 抽象化層: 利用側が低レベルな副作用（ファイル存在確認・パス直書き・shellコマンド組み立て等）へ依存している箇所があれば、
  判定API・専用サブコマンドへ集約し抽象化層を1段上げる選択肢を検討する

表層整理（内部重複の撤去のみ・既存実装の再利用のみ）を選択する場合は判定確定前に上記観点の分析を一段実施し、
「なぜ責務帰属の見直しが不要か」を判断記録に併記する。戻り値型・引数順の差異のみで本体ロジック同一なら
共通化検討時に温存せずラッパー化・引数化で集約し、類似責務の新規関数を追加する時は既存関数との重複行数を
`grep`で確認して20行以上重複時は共通化を実装前に組み込む。
