# 判定四分類の境界事例

`SKILL.md`ステップ2の四分類（保持／除去／コード移動／根本対応の併発）が割れやすい典型パターンを列挙する。

- 「公式ドキュメントに書かれているが頻繁に参照する」: 都度取得のコストと文書サイズのトレードオフで判断する
  原則は除去と判定し、保持する場合は判定理由を想定ミス列に併記する
- 「コード移動か保持か迷う」: コード変更時にのみ意味を持つならコード移動、
  読み取り時にも誤行動を防ぐ必要があるなら保持とする
- 「SSOT違反の重複だが意図的に重複させている」（コミットメッセージ方針と`.gitmessage`など）:
  重複が意図的であることが他文書側に明記されている場合は除去しない
- 「根本対応の併発か除去か保持か迷う」: 根本対応の実装コストが大きく、
  運用ノウハウとして残すべき側面も併せ持つ場合は保持を選ぶ。
  根本対応が小規模で確実に再発を抑止できる場合は根本対応の併発を選ぶ。
  根本対応が成立せず文書側のみで処理可能な場合は除去を選ぶ。
  判断に迷う場合は呼び出し元へ判定を仰ぐ
- 「除去と根本対応の併発が両方該当する」: 根本対応が成立する場合は根本対応の併発を優先する。
  文書側削除のみでは再発抑止に至らないためである
- 「frontmatter無しルール（`.claude/rules/`配下・`agent-toolkit/rules/`配下のpaths未指定ルール）の分割」:
  ファイル名順に全件自動で読み込まれるため参照漏れリスクは生じない。
  分割の要否は行数上限を判断根拠にせず、主題のまとまり（1ファイル1主題・節間の凝集度）で判断する。
  常時ロード対象のため、まず記述量の縮減（判定四分類の適用）を試み、
  縮減で構成上の不自然さが解消しない場合に限り分割する
  - 基本原則を担う中核規範ファイル（`01-agent.md`等）の分割では、節間の論理接続や
    全称規定の適用範囲（「本ファイル以下の全規範」等の宣言が指す範囲）が分割後も維持されるか照合してから実施する
  - 分割後のファイル名は、読み込み順（ファイル名順）が規範間の依存順と整合するよう採番する
- 「スキルの`references/`分離」:
  `references/`配下は自動読み込みされないため、分離判定は行数上限を判断根拠にせず、
  手順として自然な粒度（工程単位・トピック単位で独立して理解できるまとまり）への再構成として設計する。
  分離判定は次の全てを満たす場合とする
  1. 当該節が`SKILL.md`本体で必ず読まれる規範に該当しない
  2. 手順として自然な粒度（工程単位・トピック単位）で独立して理解できる
  3. 分離後も`SKILL.md`本体からの参照ポインタが手順進行上の自然な位置にある
  - 分離同時に`SKILL.md`本体側へ読み込み指示（「`references/<name>.md`を読み込んで従う」等の形式）を明記し、
    使用箇所の直前でコンテキストへ読み込まれる状態を保つ
  - 冒頭の一括読み込みや末尾の参考リンク列挙は使用時点との距離が開いて参照漏れを招くため避ける
  - 新規分離時は既存`references/`構成の全体を再設計対象に含め、同一主題は1ファイルへ集約する
- 限定条件付き規範文（適用条件を限定する表現を含む記述）: 削除前に規範性の独立性を確認する。
  縮減根拠の四分類（SSOT違反・自明導出・公式ドキュメント代替可能・上位指針への統合）へ該当するかを判定する
- 発動条件付き規範文（直後・契機を明示する表現を含む記述）: 削除前に同上の四分類判定を実施する
- 否定形規範文（禁止規定として動詞を打ち消す語尾を含む記述）: 削除前に同上の四分類判定を実施する
- 肯定的な完遂義務（完遂を明示する語尾を含む記述）: 削除前に同上の四分類判定を実施する
