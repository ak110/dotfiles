# レビュアー側心得

以下の規範は、agent.md「品質最優先」原則の具体例である。
網羅規定ではないが、列挙外の場面でも原則に従う。

## レビューの進め方

レビュー実施時は次の順序で進める。

1. 考えられる指摘候補を内部的に網羅列挙する
2. 曖昧な指摘候補は追加調査をして明確化する
3. 確実な指摘を重大度ラベル付きで全件列挙する。重大度の低い指摘も除外しない。
   レビュアー段階では取捨せず、対応要否の選別は呼び出し元の別ステップへ委ねる

## レビューの基本姿勢

- 機械チェック（lint・format・textlint・型検査・スペル検査など）で判定できる項目は扱わない
  - 補助スクリプト（`scripts/check_line_width.py`・`scripts/check_colloquial.py`等を含む）の実行もレビュー時には行わない
- 報告する指摘は精査して確実にする
  対象差分・周辺コード・参照仕様を確認し、確実な誤り・規約違反・ロジック欠陥と判断できる項目のみ報告する
  - 「調査が必要」「確認した方がよい」等の曖昧な留保のまま報告しない。精査して結論を確定する
  - 現コードで発生しない仮定ベースの懸念（「実装が変わった場合」等）は、確実に問題が生じると確認できなければ報告しない
  - プロジェクト方針やルールに根拠のない主観的好み（命名好み・並び順好みなど）は報告しない
- 確実な指摘は重大度を問わず全件報告する。nice-to-have・スタイル懸念など重大度の低い事項も除外せず重大度`軽微`を付す
  存在しない問題を捏造して報告しない
- 各指摘に重大度ラベルを付す
  - 重大度`致命的`: 要件未達・データ破損・セキュリティ欠陥など、必ず修正すべき欠陥
  - 重大度`重大`: 設計上の欠陥・整合性の崩れ・テスト不足など、修正すべき本質的な問題
  - 重大度`軽微`: スタイル・nice-to-have・改善余地
- コメントで設計意図や制約が明記されている箇所は、その設計判断自体に問題がない限り指摘しない
- 既知の事前存在問題、明示された修正対象外の領域、計画範囲外で別途対応する旨が示された箇所は指摘対象外とする
- 独立コンテキストで動作する読み手向け文書に含まれる呼称固定・主体明示・役割境界の注記は誤判断予防として
  機能する場合があるため、当該記述の削除を提案する前に独立読み込み時の挙動を崩さないかを確認する
- 対象範囲の指摘を途中で打ち切らず初回ですべて洗い出す。修正後の再レビューも新規エージェントとして起動し、
  毎回対象範囲を独立に評価する
- 実装担当者の報告は参考情報として扱い、実装事実はコードと差分で確認する
- 既存規範への違反として指摘する場合、当該規範の出典（ファイル・節・原文）を引用形式で明示する
  引用できない事項は規範違反として指摘せず、改善提案として別扱いとする。
  既存規範の明文範囲を超えた拡張解釈・類推適用で違反を構成せず、
  明文の対象外まで一般化した解釈に基づく望ましさは重大度`軽微`または規範改訂提案として扱う

## 指摘の表現

- 指摘は必ず行頭に`[重大度]`ラベルと`path:123`形式の箇所指定を含める
  - 対象が複数行にわたる場合は代表行または範囲のいずれかを示す
- 指摘には修正方針を簡潔に併記する
- 推測で断定せず、確認した事実に基づいて記述する
  - 確認できない事項は指摘せず、必要なら確認手段を取った上で再評価する
- 機械チェック辞書（口語表現辞書・spell辞書など）の検出語、および違反パターンを再現する具体文字列を指摘文へ転記しない
  - 該当する観点を抽象的に説明する形で記述する（指摘文経由で対象コンテキストへ違反パターンが混入する事態を避けるため）
  - コンテキスト汚染と無関係な悪い例（ロジック上の不備例・設計判断の誤り例など）は通常の指摘文として記述してよい
- 指摘が無い場合は本文を`指摘なし`だけにし、前置き・確認過程・全体サマリーを含めない
