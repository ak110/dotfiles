# レビュアー側心得

以下の規範は、01-agent.md「品質最優先」原則の具体例である。
網羅規定ではないが、列挙外の場面でも原則に従う。

## レビューの進め方

レビュー実施時は次の順序で進める。

1. 考えられる指摘候補を内部的に網羅列挙する
   - 対象が複数プロバイダ・複数バックエンド・複数コンポーネントで類似処理を共有する場合、
     同じ抽象化階層に位置する他コンポーネントに同じ根本原因の事象がないかを能動的に確認し、
     該当する箇所も列挙対象へ含める
2. 曖昧な指摘候補は追加調査をして明確化する
3. 確実な指摘を重大度ラベル付きで全件列挙する。重大度の低い指摘も除外しない。
   レビュアー段階では取捨せず、対応要否の選別は呼び出し元の別ステップへ委ねる

## レビューの基本姿勢

- 機械チェック（lint・format・textlint・型検査・スペル検査など）で判定できる項目は扱わない
  - 補助スクリプト（`scripts/check_line_width.py`等を含む）の実行もレビュー時には行わない
  - 行幅・行数など機械チェック・補助スクリプトで判定可能な定量項目は、
    レビュアーが推定値を算出して指摘しない
  - 行長判定は機械チェック（markdownlint MD013・`scripts/check_line_width.py`等）の判定結果に従う。
    レビュアー独自のbyte長計測による行長判定はしない
   （日本語マルチバイトはbyte長が文字長の約3倍となり恒常的な誤検出を招くため）
- 報告する指摘は精査して確実にする
  対象差分・周辺コード・参照仕様を確認し、確実な誤り・規約違反・ロジック欠陥と判断できる項目のみ報告する
  - 「調査が必要」「確認した方がよい」等の曖昧な留保のまま報告しない。精査して結論を確定する
  - 現コードで発生しない仮定ベースの懸念（「実装が変わった場合」等）は、確実に問題が生じると確認できなければ報告しない
  - プロジェクト方針やルールに根拠のない主観的好み（命名好み・並び順好みなど）は報告しない
  - 日本語の文末述部の見た目（理由節終止「〜のため」「〜から」など接続表現を含む形）のみを根拠に
    文構造不完全と判定しない。当該構文も独立した1文として文意が完結する場合があるため、
    文意の完結性を確認したうえで判定する
- 確実な指摘は重大度を問わず全件報告する。nice-to-have・スタイル懸念など重大度の低い事項も除外せず重大度`軽微`を付す
  存在しない問題を捏造して報告しない
- 各指摘に重大度ラベルを付す
  - 重大度`致命的`: 要件未達・データ破損・セキュリティ欠陥など、必ず修正すべき欠陥
  - 重大度`重大`: 設計上の欠陥・整合性の崩れ・テスト不足など、修正すべき本質的な問題
  - 重大度`軽微`: スタイル・nice-to-have・改善余地
  - 上記以外のラベル（「中程度」「警告」「軽度」等）を出力しない。
    いずれにも自己分類できない指摘は追加調査でいずれかを確定する。
    追加調査で確定できない指摘は取り下げる
- コメントで設計意図や制約が明記されている箇所は、当該設計判断の妥当性のみを評価対象とする
- 既知の事前存在問題、明示された修正対象外の領域、計画範囲外で別途対応する旨が示された箇所は指摘対象外とする
- 独立コンテキストで動作する読み手向け文書に含まれる呼称固定・主体明示・役割境界の注記は誤判断予防として
  機能する場合があるため、当該記述の削除を提案する前に独立読み込み時の挙動を崩さないかを確認する
- 対象範囲の指摘を途中で打ち切らず初回ですべて洗い出す。修正後の再レビューも新規エージェントとして起動し、
  毎回対象範囲を独立に評価する
- 実装担当者の報告は参考情報として扱い、実装事実はコードと差分で確認する
- 既存規範への違反として指摘する場合、当該規範の出典（ファイル・節・原文）を引用形式で明示する
  引用できない事項は規範違反として指摘せず、改善提案として別扱いとする。
  既存規範の明文範囲を超えた拡張解釈・類推適用で違反を構成せず、
  明文の対象外まで一般化した解釈に基づく望ましさは重大度`軽微`または規範改訂提案として扱う
- 構文の合法性はパーサー・lint・型検査で確実に検出できる機械チェック担当領域のため、机上レビューでは指摘対象外とする
  （学習知識と対象言語バージョンで有効化された新構文の齟齬が誤指摘の主因のため）
- 残る言語仕様・標準ライブラリの仕様への依存指摘（非推奨・削除など）を挙げる前に、
  対象プロジェクトの実行環境のバージョンをReadで確認する
  - 参照先は`.python-version`・`pyproject.toml`の`requires-python`・
    `package.json`の`engines`など
  - 当該バージョンの公式リリースノート・PEP番号・言語仕様の変更履歴で裏付けを取る

## 計画ファイル文脈での例外

計画ファイル本文（`~/.claude/plans/*.md`）は廃棄される一時作業文書である。次を指摘対象外とする。

- 記述スタイル（章構成・段落構成・表現選択・書き方）への指摘。記述間の矛盾は対象に含む
- 実装時にエージェントが判断可能な細部（変数名・エラーメッセージ文言・小規模なループ構造・局所的な制御フローなど）への指摘

詳細は次の規定に従う。

- `agent-toolkit/skills/plan-codex-review/SKILL.md`の規定
- `agent-toolkit/skills/plan-mode/references/launch-prompts-integrity.md`のplan-reviewer雛形の規定

## 指摘の表現

- 指摘は必ず行頭に`[重大度]`ラベルと`path:123`形式の箇所指定を含める
  - 対象が複数行にわたる場合は代表行または範囲のいずれかを示す
- 指摘には修正方針を簡潔に併記する
- 推測で断定せず、確認した事実に基づいて記述する
  - 確認できない事項は指摘せず、必要なら確認手段を取った上で再評価する
- 機械チェック辞書（口語表現辞書・spell辞書など）の検出語、および違反パターンを再現する具体文字列を指摘文へ転記しない
  - 該当する観点を抽象的に説明する形で記述する（指摘文経由で対象コンテキストへ違反パターンが混入する事態を避けるため）
  - コンテキスト汚染と無関係な悪い例（ロジック上の不備例・設計判断の誤り例など）は通常の指摘文として記述してよい
- 指摘が無い場合は本文を`指摘なし`だけにし、前置き・確認過程・全体サマリーを含めない
- 最終的な指摘リストは1回のみ出力する
  - 試行錯誤の途中過程で同一指摘を複数回列挙したり、整理過程の中間版リストを最終出力に含めたりしない
- 出力前に重複行・部分重複した同一指摘は集約する
- gitignore対象ファイル（`CLAUDE.local.md`・`.env`等）の内容を根拠とする指摘は、
  本文表現をgitignore対象内容の直接引用ではなく公開可能な観点表現に抽象化する。
  指摘対応の記録（計画ファイル`## 変更履歴`・進捗ログ・PR説明）にも非公開内容を転記せず、抽象観点で記述する。
  gitignore対象ファイルを参照する場合は指摘本文に「gitignore対象方針との整合」等のラベルを明示し、
  統合側が公開レビュー成果物への転記可否を判断できるようにする
