# フィードバック投入

## ステップ0: 事前準備の確認

最初に`atk fb status`を実行する。
非ゼロ終了の場合は標準エラー出力をユーザーへ提示して終了する。
正常終了の場合は出力を破棄して次のステップへ進む。

## ステップ1: 入力の確認

ユーザーが提示したフィードバック本文を読み取る。
本文が提示されていない場合はユーザーに提供を求める。

本文に「確認したい論点」「事前確認が必要」「〜でよいか」「未解決」「未確定」等の
未確定表現・確認要求文言が含まれる場合は、`atk fb add`実行前にユーザーと当該論点を解消する。
解消後は本文から確認要求文言を除き反映結果として書き直したうえで投入する。
ユーザー判断で残置と決めた項目は「保留事項」として本文へ明記したうえで投入する。

本文に含まれる前提・影響範囲のうちコードベース調査（Read・grep等）で確定できる事項は、
投入前に調査して事実として確定した記述へ書き直す。
未調査の条件分岐表現（「〜している場合は」等）のまま投入しない。
ユーザー選好・採否確定後の設計判断に属する論点のみを保留事項として明記して投入する。

プロセス・規範の改善提案では、静的な文書調査に加えて観測可能な実績データで事象の頻度・傾向を裏付けてから本文を確定する。
実績データはセッションログ・進捗ログ・レビュー指摘の統合表などを指す。
対象は「プロセス・規範の改善提案」に限定し、軽微な提案への過剰調査は避ける。

原因記述に推測表現を含むフィードバックの取り扱いは
`agent-toolkit:session-review`スキル「ステップ2」の「推測表現の投入抑止」バレットに従う。

TBD投入（`atk fb tbd-add`）を伴う場合の書式は
`agent-toolkit/skills/apply-feedback/references/hold-with-tbd-inject.md`を正本として参照する。
選択肢形式の優先方針・`--scope`引数の検証手順も同ファイルに集約する。

### エージェント自律投入時の妥当性条件

コーディングエージェント自身が自律判断で`atk fb add`によりフィードバックを投入する場合の妥当性条件は
`agent-toolkit/rules/01-agent.md`「品質最優先」節の条件付き例外の全条件に従う。

## ステップ2: 対象リポジトリの判別

各提案項目の主題から`target_repo`のリポジトリパスを判定する。
判定はフィードバック本文が言及する識別子・パス・スキル名・ファイル名等から対象repoを推定する原則に従う。

- 特定のプロジェクトの成果物・機能・ドキュメントへの提案は当該プロジェクトのリポジトリパス
- 現在作業中のプロジェクトへの提案はカレントディレクトリ

判別が困難な項目はユーザーへ`AskUserQuestion`で確認する。

## ステップ3: メッセージの整形

各メッセージは単体で読んで意味が完結する形に整える。
同一出典から派生する複数提案では、本文へ次の情報を必ず含める。
直近コミット文脈・対話文脈への暗黙参照を含む提案でも同様とする。

- 出典URL・参照元（外部リポジトリ・記事・PR等）
- 関連する派生提案との関係（同一出典・代替案・前提依存など）
- 対象を一意に特定するために必要な前置き

対象ファイル・対象節が特定できる場合はメッセージ冒頭で明示する。
項目間に共通する前置きは各メッセージ本文へ複写する。
フィードバックは個別ファイルへ分割して保管されるためファイル間参照は機能しない。

## ステップ4: 投入

target_repoごとに、対応するメッセージ群を位置引数で渡して`atk fb add`を呼び出す。
投入直後の状態は`inbox`となる。
`processing`（`start-processing`後の途中状態）・`adopted`・`rejected`（最終処理済み）への
移動は`atk fb`の対応サブコマンド経由で行われ、本スキルは常に`inbox`への追加のみを担う。

```sh
atk fb add <repo-path> "<message1>" ["<message2>" ...]
```

- `<repo-path>`: target_repoの絶対パスまたは`~`展開可能パス
- `<message>`: 自由本文。改行を含めてよい。
  シェルメタ文字（バッククォート・`$`・`!`等）を含む場合は
  シングルクォート（`'...'`）または`$'...'`で囲む。
  本文中にシングルクォートを含む場合はシェルのエスケープ規則に従う
- 複数のtarget_repoを投入する場合はリポジトリ別に複数回呼び出す

### 禁止する起動形式

本文は位置引数として単一の文字列で渡す。以下は補足の禁止規定として扱う。

- ヒアドキュメント（`<<EOF`）・パイプ（`echo ... |`）・ファイルリダイレクト（`< file`）で本文を渡さない
  - `atk fb add`は位置引数で本文を受け取り、位置引数を省略すると`$EDITOR`（vim等）を対話的に起動する
  - 標準入力は読み取らない仕様のため、上記の起動形式ではvimが起動して本文が空のまま失敗する
- 本文をファイルから読みたい場合は呼び出し側で`"$(cat path/to/file)"`等で位置引数へ展開する

### 起動失敗時の対処

- `Vim: 警告: 端末への出力ではありません`等のvim起動メッセージが返った場合は、本文が位置引数で渡されていない
  - 起動コマンドを再構成し、本文全体をクォート付き位置引数として渡し直す

## ステップ5: 結果の提示

`atk fb add` CLIの標準出力をそのままユーザーへ提示する。
