# 保留判定・部分採用の除外要求のTBD投入

保留判定時、フィードバック原文内に事前確認が必要な論点・確認事項・仕様上の未確定項目が列挙されている場合は、
当該事項を`atk fb tbd-add`で個別投入してから保留する。

## 準用範囲

本節の書式（`atk fb tbd-add`のオプション指定・`--scope`検証等）は、
保留判定を伴わない場合にも準用する。準用先は次のとおり。

- `agent-toolkit:add-feedback`のTBD投入
- `agent-toolkit:process-feedbacks`ステップ3「複数要求の分解判定」で部分採用と判定した際の除外要求投入

`add-feedback`のTBD投入は`atk fb add`実行前に発生するため、
対象フィードバックファイルは未生成であり、
`--scope`値の事前ファイル存在検証（`ls`・`grep`照合）は適用対象外とする。
部分採用の除外要求投入は既に生成済みの対象フィードバックファイルを`--scope`値に指定するため、
事前ファイル存在検証は通常どおり適用する。
`atk fb add`が生成するファイル名`{timestamp}-{counter:03d}.md`の`{timestamp}`部分は
`atk fb add`実行時点で確定し、秒単位で異なる値になり得るため事前予測は不可能である。
TBD投入時は暫定値（投入セッションの現在時刻等）を`--scope`に指定し、
`atk fb add`実行後に事後照合する。
不一致が確認された場合は`atk fb tbd-edit <ファイル名>`で`$EDITOR`を起動し、
frontmatterの`scope:`値を実ファイル名の日付連番部分と一致するよう手動で書き換える。
1回の`atk fb add`呼び出しで複数`counter`のファイルが生成された場合は、
論点が全ファイル共通なら共通の`{timestamp}`部分のみを`--scope`に指定する。
論点がcounter個別ファイル固有なら全`counter`値へ個別のTBDを紐付ける。

## 保留判定を確定する前の観測ソース精査

確定前に、原因調査に利用可能な観測ソースを全件洗い出したうえで精査を試みる。

- 対象観測ソースは`~/.claude/projects/`配下のセッションjsonlログ・実行トレース・作業ツリー痕跡（`git status`・`git log`・`git reflog`）などを含む
- 精査で源が特定できた場合は保留判定を再検討する
- 特定できない場合のみ保留判定を確定し、精査対象と検索パターン・結果件数を保留理由へ添える

## 対象論点の判定基準

次のいずれかを満たす記述を対象とする。

- 「事前確認が必要な論点」「未確定事項」「要確認」等の見出し配下の項目
- 質問形式で明記された項目

## 投入コマンド

各論点を個別に次の書式で投入する。

`atk fb tbd-add --question-type=choice --choices "<選択肢1>,<選択肢2>,..." --scope=<提案識別子> "<論点本文>"`

対象リポジトリは`atk fb tbd-add`実行時のカレントディレクトリから自動解決される。

選択肢化可能な論点は`--question-type=choice`と`--choices`を用いた選択肢形式を優先する。
選択肢化が困難な自由記述質問のみ`--question-type=free`（既定）を用いる。

`<提案識別子>`はフィードバックファイル名の日付連番部分などを使う。
`--scope`引数の値はフィードバックファイル名の日付連番部分と一致することを投入前に検証する。
検証は`ls`で該当ファイル名を確認するか、`--scope`値を含むファイル名を`grep`で照合する。

投入する論点本文は`agent-toolkit/rules/02-collaboration.md`「3. TBD記録の書式」節の
疑問文の必須要件に従い、疑問文を1文含める。後述の「保留判定時のTBD投入」節が定める
保留理由・実機検証済みの内容・次回再評価時の確認事項の3項目は状態記録であり、本要件の規範上の対象外とする。
ただし`_atk_fb_tbd.py`の機械検出は投入元（`--source`）で分岐しないため、
疑問文を欠く場合は機械検出上の警告が出る場合がある（警告のみで投入は成功するため運用上の支障はない）。

## 投入後の扱い

TBD投入はフィードバック本体の配置を変更しない。
本体の所在は`agent-toolkit:add-feedback`のTBD投入では`atk fb add`実行後のinbox配下、
それ以外の場合（保留判定・部分採用の除外要求の投入時）はprocessing配下となる。
次回起動時、回答済みTBDと合わせて再評価する。

## 保留判定時のTBD投入

`agent-toolkit:process-feedbacks`ステップ2.5で保留判定を確定した時点で、
次のコマンドを実施する。
本節はfeedback原文内に事前確認の対象が列挙されない場合にも必須ゲートとして適用する。

`atk fb tbd-add --scope=hold --source=session-hold --question-type=free "<本文>"`

`--scope=hold`は固定値運用とし、本ファイル「対象論点の判定基準」節の`--scope`値検証手順
（`ls`・`grep`によるファイル名照合）は本用途では適用しない。

feedback原文内に事前確認の対象が列挙される場合は、本節の投入と本ファイル「投入コマンド」節の
個別論点投入を併用する（保留全体の記録と個別論点の記録の両方を残す）。

本文には次の3項目を含める。

- 保留理由（何を待っているか・何が未確定か）
- 実機検証済みの内容（前提条件を確認した手段・確認結果）
- 次回再評価時の確認事項（何が判明すれば採用可能になるか）
