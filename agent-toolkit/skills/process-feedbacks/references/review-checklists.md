# 採否判定チェックリスト

`agent-toolkit:process-feedbacks`ステップ2.5「採否判定前の網羅調査」・
ステップ3「批判的検討」から参照する調査観点・判定観点の全件を集約する。

## 網羅調査チェックリスト

- 追記先候補ファイルの該当節を読み、既存記述または既存実装で同等内容が既にカバー済みでないかを確認する。
  併せて同ドメインの既存規範が`agent-toolkit/rules/`・`agent-toolkit/skills/`配下に存在しないかを`grep -rn`で網羅確認する。
  既存規範が存在する場合はスコープ拡張・条件補強による一本化案を優先候補とする。
  原文に複数の提案項目が含まれる場合のマッピング手順は
  `agent-toolkit/skills/process-feedbacks/references/explore-template.md`「Explore委譲雛形」節観点1に従う
- 禁止規定・推奨規定との照合:
  フィードバック原文が示す反映パターン（キーワード・実装手法・禁止/推奨の主張）を
  `agent-toolkit/rules/`・`agent-toolkit/skills/`配下の既存禁止規定・推奨規定と照合する。
  照合手順は原文キーワードで`grep -rn`し、禁止規定・推奨規定として書かれた節を確認する。
  抵触する場合はステップ3の批判的検討へ引き渡し、既存規範との衝突可否を判定する
- 関連スキル・関連ルールファイルに重複・矛盾する規範が無いかを確認する
- 追記先候補ファイルのスコープ宣言（目的記述・SSOT宣言・冒頭導入文の対象範囲）を読み、
  追記内容の主題が同スコープ内かを検証する。
  スコープ外の場合は追記先を変更するか、当該ファイルのスコープ宣言拡張を`## 変更内容`へ含める
- 反映先が上位規範文書（01-agent.md・agent-standards/SKILL.md・plan-mode/references/配下等）の場合は、
  反映先ファイルの該当節を`Read`で直接確認する。
  確認対象は反映内容と衝突しうる既存上位規定（「品質最優先」節・fabricated-metricsカテゴリ・
  「不採用根拠の限定」節・「先送り・縮退表現の混入検査」節等）とする。
  衝突検出時は反映内容の適用範囲を限定するか、上位規定側にも例外参照を追加する統合案へ再設計する
- 追記先候補が配置先スキルの管轄範囲（プロジェクト直下`AGENTS.md`・`CLAUDE.md`が編集対象
  ディレクトリに対して指定する範囲）に該当する場合は、当該配置先スキルをSkillツールで起動する。
  起動は採否判定・計画作成に着手する前とし、配布物固有の規範を採否判定段階で適用する
  - 起動失念時に発生し得る事象の例: version bump対象ファイルの`## 変更内容`欠落・配布物境界の規範の未適用・固有名混入の事後検出
- 提案の主旨を原文転記から再確認し、自身の解釈誤りが無いかを確認する
- 追記後の最終形に近い文面を組み立てて`wc -l`で実測し、`agent-toolkit:agent-standards`「文書サイズ上限」節の規定に従う
- フィードバック原文の取得・解釈が完遂できない場合は、規範改訂・代替反映の判定より前に次で進める。
  対象事例は外部URL取得失敗・素材参照ができない場合・前提仕様の確定ができない場合などとする。
  原文が問いかけ調・抽象的で具体反映内容を一意に確定できない場合は具体化候補を2〜4案生成し、
  `AskUserQuestion`で提示してユーザーに選ばせる（多肢を併せて採る想定の場合は`multiSelect: true`を許容）。
  候補生成が不可能な場合に限り作業を中断してユーザーへ取扱（保留・不採用・原文貼付要求）を確認する
- 対象機能に既存の仕様書・設計書・要件定義書等でロール定義・受領者定義
 （受領者・利用者・負担先管理者などの区分）が存在する場合、当該定義箇所を調査対象に含め、
  実装するロールの受領者・参照範囲が定義と一致することを確認してから採否判定へ進む
- 実装挙動異常の観測時は採否判定前に、
  `agent-toolkit/skills/agent-standards/references/feedback-review-common.md`「事前調査の徹底」節に従う。
  調査範囲が広い場合は`Explore`サブエージェントへ委譲する

## 批判的検討チェックリスト

- 批判的検討の観点: 詳細は
  `agent-toolkit/skills/agent-standards/references/feedback-review-common.md`「判断軸の順序」節を参照する
- 批判的検討の対象限定:
  単一要求内は採用・不採用の二値評価のみを対象とする。
  複数の独立要求が並列に含まれる場合の分解判定は本節「複数要求の分解判定」項に従う。
  原文の反映内容の量・粒度・実装レベルへの裁量介入は対象外とする
- 背景方針の推定: ユーザーが背景方針を直接表現していない場合でも、
  フィードバック群の傾向から推定して判定軸に含める。
  規範緩和に関するフィードバックが複数同時に提示された場合は、
  採否判定前に`AskUserQuestion`でユーザー方針確認を発行してよい
- 汎用観点の判定: `agent-toolkit:critical-thinking`スキル「批判的検討の汎用観点」節
 （既存記述との重複・矛盾、コンテキストコスト対効果、適用範囲との整合、論理整合性）に従い判定する
- 振り返り由来フィードバックの再帰検討:
  frontmatterに`source: session-review`を持つフィードバックは、生成元スキル側の改善案も併記検討する対象に含める。
  提案内容またはその根拠に問題が見つかった場合、`agent-toolkit:session-review`スキル本文の修正案を併せて提示する
- 仕様不明を理由とした不採用判定時の検証義務:
  反映先設定ファイル・CLIなどの当該仕様が不明であることを理由に不採用判定する場合は、まず実機検証する。
  検証手段は対象CLIのhelp出力・サブコマンド実機実行・対象設定ファイルの現在値などとする。
  検証手段が思い当たらない場合は不採用確定せず「未検証」として保留し、ユーザーへ照会する
- 候補技術主張の実機検証: 詳細は
  `agent-toolkit/skills/agent-standards/references/feedback-review-common.md`「候補技術主張の実機検証」節を参照する
- 複数要求の分解判定:
  フィードバック原文に複数の要求が並列で含まれる場合は原文を要求単位に分解する。
  並列要求の例として例示と方向性の両立、具体例と一般原則の両立、特定対応と全体評価の両立などがある。
  分解後、採否を**採用／部分採用／不採用**で判定する。
  部分採用と判定した場合は採用する要求と除外する要求を明示し、ステップ4の提示書式の採用範囲・除外範囲欄に転記する
  - 除外する要求は`atk fb tbd-add`で個別投入してユーザーが未対応要求を認識できる状態にする。
    投入手順は`references/hold-with-tbd-inject.md`「準用範囲」節に従う
- 不採用根拠の限定: 詳細は
  `agent-toolkit/skills/agent-standards/references/feedback-review-common.md`「不採用根拠の限定」節を参照する。
  実測結果はステップ4の提示書式「理由」欄へ具体数値・観測事実として転記し、
  実効性が対象範囲の一部にとどまる場合は「採用」へ再判定する
- 採用時の反映内容の縮小禁止:
  原文の反映内容を「過剰」「実効性に対して負担が大きい」「実装の手間が見合わない」等の自己判断で薄めた形の採用や、
  過剰と判断した箇所だけを除いて反映する部分採用は禁止する。
  原文の反映内容が過剰と判断した場合も、原文どおり採用するか不採用とする二択とする。
  既存の「部分採用」語彙は複数の独立要求が並列に含まれる場合の要求単位分割にのみ用いる
- 重複禁止規定との衝突時の例外規定:
  ステップ2「既存SSOT規範との衝突回避」の書き換えで解消しない場合に限り本例外を発動する。
  原文どおりの採用が既存規範の重複禁止（`agent-toolkit:agent-standards`「記述の簡潔さ」節等が禁じる
  言い換え・冗長記述）と衝突する場合は重複禁止を優先する。
  原文どおりの採用が成立しない場合は不採用とする。
  要求単位の分解の成立条件は「複数要求の分解判定」バレットに従う
- 識別子帰属誤り発見時の第1候補:
  フィードバック原文が言及する識別子（ファイル名・節名・欄名など）に
  帰属誤り・実在不備が判明した場合、除外・削除を第1候補とせず、
  正しい帰属先への訂正と原文列挙内容の維持を第1候補とする。
  除外は原文主旨が当該識別子を含まなくても達成可能であることを、
  原文文面と対象ファイル現状の突合で確認できた場合に限る。
  判断過程は`### エージェント判断`へ1行で記録する
- 自己整合性の検査: 詳細は
  `agent-toolkit/skills/agent-standards/references/feedback-review-common.md`「自己整合性の検査」節を参照する
