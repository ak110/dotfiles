# ステップ4詳細: 検討結果の提示書式と適用手順

本ファイルは`agent-toolkit:process-feedbacks`スキルのステップ4での検討結果の提示専用の書式を定義する。
`agent-toolkit:plan-mode`の計画ファイル書式（`### エージェント判断`等のH3見出し）は別体系である。
plan-mode側の定義は
`agent-toolkit/skills/plan-mode/references/plan-file-guidelines.md`にある。

## 提示書式

各提案の採否判定と理由をユーザーに提示する。
フィードバック1件ごとに`### <ファイル名>: <提案要約>`の見出しを設け、
配下に箇条書きで内訳を並べる。
採用提案は「採否」「理由」「反映先」「反映内容」を記述する。
部分採用提案は「採否」「理由」「採用範囲」「除外範囲」「反映先」「反映内容」を記述する。
不採用・保留提案は「採否」のみ記述し、採否欄に不採用理由または保留理由を併記する。

```markdown
## 検討結果

### <ファイル名>: <提案要約>

- 採否: 採用
- 理由: <採用する根拠>
- 反映先: <ファイル＋節または該当箇所>
- 反映内容: <反映する具体的な内容>

### <ファイル名>: <提案要約>

- 採否: 部分採用
- 理由: <採用する範囲の根拠>
- 採用範囲: <採用する要求の列挙>
- 除外範囲: <除外する要求と除外理由>
- 反映先: <ファイル＋節または該当箇所>
- 反映内容: <採用範囲のみの反映内容>
- 投入したTBDファイル名: <ファイル名1>, <ファイル名2>（除外要求のTBD投入時のみ）

### <ファイル名>: <提案要約>

- 採否: 不採用。<不採用理由>

### <ファイル名>: <提案要約>

- 採否: 保留。<保留理由>
- 投入したTBDファイル名: <ファイル名1>, <ファイル名2>（TBD投入時のみ）
```

ファイル名は呼び出し元エージェントが結合したmarkdown内へ付与した`## <filename>`見出しの値をそのまま用いる。

見出し直下は箇条書き項目のみとし、地の文の補足は箇条書き項目内へ統合する。

フィードバックが複数件の場合は、
計画本文でフィードバックを参照する際は元プロンプトの通し番号（`[1]`から`[N]`）を用い、
採否ごとの連番とは独立させる。

- 通し番号は採用・不採用を混在した順に付与し、途中スキップを許容する
- 1件のみの場合は通し番号運用は不要

フィードバック原題を`decision-format.md`「提示書式」節の見出し（`### <ファイル名>: <提案要約>`H3）へ転記する場合、
原題内の縮退フレーズ語彙の混入有無を確認する。
該当語彙を含む見出しは、該当語彙を含まない要約表現へ書き換える。
代表フレーズは`agent-toolkit:agent-standards`配下`scope-escalation-phrases.md`の各該当カテゴリを参照する。
原題の完全転記は計画ファイルの`## 背景`配下の`text`コードブロックでのみ行う。
呼び出し元の`agent-toolkit:process-feedbacks`スキルは原題入りフィードバック本文を渡す責務を持つ。
計画ファイル本体の作成・追記は`agent-toolkit:plan-mode`スキルが担う。

## AskUserQuestion発行条件と除外条件

採用1件以上を含み、不採用判定と部分採用の除外要求が0件の場合は、
ユーザー応答を待たず計画作成へ移行する（保留の混在は許容、保留は`AskUserQuestion`発行対象外）。
不採用判定と部分採用の除外要求が1件以上含まれる場合は`AskUserQuestion`を発行する。
発行対象は、ステップ3「不採用根拠の限定」に該当する根拠を持つものに限定する。
規模・作業量・対象件数・スコープ整理を根拠とする不採用・除外は、
`agent-toolkit:agent-standards`配下`scope-escalation-phrases.md`の縮退誘発カテゴリ扱いで発行対象外とする。
当該1件ごとに発行し、不採用は採否確認、部分採用は除外要求の妥当性確認をユーザーから得る。
例外: frontmatterに`source: session-review`を持つ不採用判定は発行対象から除外してよい
（自己生成起点で信頼性が低く、不採用によるリスクが小さいため確認を省略できる）。
必要と判断した場合は通常通り発行する。

不採用根拠が「既存記述または既存実装で同等内容が既にカバー済み」の場合は、
`AskUserQuestion`発行対象から除外してよい。該当箇所を検討結果本文へ引用して提示する。

- 確認は不採用判定の見落とし防止が目的のため、フィードバック1件ごとの個別承認とする
- 1問あたり最大4件の上限を超える場合は複数問へ分割する
- ユーザーが「採用へ転換」を選んだ場合は採否を採用へ更新し、ステップ2・ステップ3の判定を再評価して提示書式も更新する
- 部分採用の適用範囲は`agent-toolkit/skills/process-feedbacks/references/review-checklists.md`
  「批判的検討チェックリスト」節の「採用時の反映内容の縮小禁止」項に従う

## レビュー指摘による反映先変更の扱い

採用判定後にレビュー指摘で反映先の変更が必要と判明した場合の扱いは次のとおりとする。

- 対象レビューは`agent-toolkit:plan-mode`工程5以降のcodexレビュー・
  `plan-reviewer`・`agent-toolkit:careful-review`が該当する
- 不採用転換ではなく反映先変更として扱い、変更後の反映先・反映内容を計画ファイルの`## 変更履歴`へ記録する
- ユーザーへの追加採否確認は不要とし、フィードバック原文の主旨が同等に達成されている限り採用扱いの後始末を継続してよい

## 後始末順序

計画ファイル`## 実行方法`での後始末コマンドの順序は次のとおりとする。

- 後始末は対象リポジトリのレビュー完遂・`git push`完了後に配置する。
  後始末コマンドがフィードバック管理側へcommit/pushを伴う場合、
  対象リポジトリ側がレビュー指摘で巻き戻るとフィードバック管理側だけが先行公開され整合性が崩れるため
- 標準順序は実装→検証→`agent-toolkit:commit`→コミット→
  `agent-toolkit:careful-review`まで`plan-impl-executor`側で完遂し、
  以降の後続工程（`git push`から振り返り・`exit-session`まで）は
  `agent-toolkit:process-feedbacks-finish`スキル本文の工程順序へ引き継ぐ
  - push後CI通過確認は`agent-toolkit:commit`スキル「push後のCI通過確認」節に従う
- ステップ3時点は採否ラベルの確定に留める。`atk fb adopt`・`atk fb reject`・`atk tb adopt`の
  実行タイミングは、「後始末順序」節の「標準順序は実装→検証→…」で始まるバレットが定める
  `process-feedbacks-finish`への引き継ぎ工程に従う

## 後始末コマンドの引数

`atk fb adopt`・`atk fb reject`・`atk tb adopt`は`--note`と`--commit`を渡して結果を対象ファイルへ追記する。

- `--note`: 採用時は反映概要、不採用時は不採用理由。1行に収める
  - シェルメタ文字混入をその都度検知せず、常に`$'...'`のANSI-Cクォートで囲む。
    引用符規範は`agent-toolkit/skills/add-feedback/SKILL.md`「ステップ4: 投入」節の`<message>`引用符規範と同一とする
  - 本文中にシングルクォートを含む場合は`'`を`\'`へエスケープする
  - 二重引用符囲みでバッククォート囲み識別子（例: `` `_plan_diff_gates_scan.py` ``）を渡すとbashがコマンド置換として解釈する。
    該当文字列が実在コマンドと一致する場合は誤実行の危険があり、一致しない場合は`--note`値から
    バッククォート内文字列が空文字置換された意図しない内容が記録される
  - 値の指定形式は`--note=VALUE`形式（`=`区切り）で統一する。値の内容によらず`=`区切りへ揃えることで、値が`-`で始まる場合の`expected one argument`エラーと、後始末例示との表記不一致とを併せて回避する
- `--commit`: 対象リポジトリの短縮commit hash（`git rev-parse --short HEAD`の値）。
  管理側リポジトリのhashではなく対象リポジトリのhashを渡す
  - 値の指定形式は`--commit=VALUE`形式（`=`区切り）で統一する（理由は上記`--note`のサブバレットに同じ）
- 対象ファイル末尾の`## 処理結果`節は`--note`・`--commit`の指定有無にかかわらず必ず追記される。
  必須項目として採否と処理日時を記録し、指定された任意項目（`--commit`・`--note`）のみを追加行として書き込む
- `atk fb adopt`・`atk fb reject`・`atk tb adopt`は対象を`<filename>`引数で直接解決する。
  `--target-repo`オプションを指定した場合はfrontmatter`target_repo`との一致検証を行い、
  不一致でexit 2として誤対象リポジトリでの操作事故を予防する。
  ただし前掲`--commit`値の取得は対象リポジトリの作業ツリー内で行う必要があるため、
  実務上は当該作業ツリー内から一連の後始末をする

## 計画作成移行ルール

全件の承認を得たうえで計画作成へ移行する。
