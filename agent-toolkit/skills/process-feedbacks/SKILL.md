# フィードバックの取得・批判的検討・適用

`atk fb`と`atk tb`の全サブコマンドは内部で`git pull --ff-only`を実行する。
対象は`atk fb`側の`add`・`list`・`show`・`adopt`・`reject`・`rm`・`edit`・`commit`と、
`atk tb`側の`add`・`list`・`answer`・`edit`・`adopt`・`rm`を含む。
手動での`git pull`実行は不要とする。
`adopt`・`reject`は採否確定を管理側へ反映する（管理側リポジトリの操作は`atk fb`が内部で完結する）。
対象リポジトリ側のcommit/pushは別途必要とする。
自律実行モード下で起動された場合は`agent-toolkit/rules/02-collaboration.md`「自律実行モード」節の運用を本スキルの処理単位へ適用する。
適用前提として本スキルは次を自身の文脈で確立する。

- 処理対象: 「ステップ1: 入力の確定」で取得したフィードバック全件
- 完了判定基準: 全件の`atk fb adopt`・`atk fb reject`・`atk tb adopt`反映と対象リポジトリのcommit・push完了
- ユーザー確認事項の記録先: `atk tb add`（本スキルは`atk fb`・`atk tb`の存在を前提とするため`atk`不在分岐はない）
- 追加停止契機: なし（同節既定の前提崩れのみで停止する）

協調モード下では同節を適用せず、ユーザー確認は通常の確認規範に従う。

## 基本方針

各採否判定の最優先基準は、当該フィードバックの適用が成果物および今後の作業の品質向上に寄与するかとする。
手順を機械的に踏襲して採否を確定しない。
形式整合のみを根拠とした採用・不採用判定など、品質寄与の評価を伴わない判定を避ける。不採用根拠の限定はステップ3の規定に従う。

## ステップ1: 入力の確定

`/process-feedbacks <repo-path>`形式で引数を受け取った場合、引数が実在ディレクトリなら対象リポジトリパスとして扱う。
実在ディレクトリでない引数はフィードバック本文の直接入力として扱う（`/apply-feedback <本文>`相当の利用パターンを吸収する）。
引数なしの場合は`git rev-parse --show-toplevel`で取得した現リポジトリパスを対象リポジトリとして扱う（既定）。

対象リポジトリ確定後、`atk fb list --status=processing --target-repo=<repo>`でprocessing残存を確認する。
残存時は前セッションの中断とみなし、実装状況を確認したうえで再実施対象へ最優先で組み込む。
続いて`atk fb show --all --status=active --target-repo=<repo>`を実行する。
feedback（inbox・processing）と回答済みTBDを一括取得する。
出力は`# feedback`・`# tbd`種別ヘッダで区分けされ、`### <filename> [<state>]`形式で
各エントリの状態ラベルが併記される。
`--status=active`はfeedback側の`inbox`・`processing`とtbd側の`answered`を指す。
`<repo>`は対象リポジトリパスまたは正規化リモートURL（`host/owner/repo`形式）を指す。
出力が空（`### <filename>`見出しが1件も存在しない）場合は「処理対象なし」と示して終了する。
1件以上の場合は`### <filename>`見出しの件数を1文でユーザーに提示する。
取得した全件を本セッションの処理対象とする（件数・規模による選定は行わない）。
`atk fb show`が非ゼロ終了する場合（feedback-inbox無効化等）は、標準エラー出力のエラーメッセージを提示して終了する。
本ステップ取得のスナップショット以降にinboxへ追加されたファイルは、次回起動時の処理対象とする。
処理対象の確定後、今回取得したfeedback分（`# feedback`見出し配下、`# tbd`側は対象外）を`atk fb start-processing <filename...>`でprocessingへ遷移させてから着手する。

フィードバック本文の直接入力の場合は上記`atk fb show`取得をスキップし、ユーザーメッセージから該当箇所を抽出する。
フィードバックが含まれない場合はユーザーに提供を求める。

`atk fb`が扱う操作（inbox・processing・adopted・rejected本文の取得、投入、状態遷移など）はCLI経由で実施する
（管理側の抽象化を保ち、Windows等の環境依存での表示破損を避けるため）。
CLI未提供の参照用途（`atk fb show --include-processed`で拾えない横断検索・複数ファイル参照・特殊な集計用途など）は
直接Readで参照してよい。
管理repoのrootは`AGENT_TOOLKIT_PRIVATE_NOTES`環境変数で指定する（詳細は`agent-toolkit:agent-standards`スキル「識別子と環境変数」節）。
frontmatterは出力に保持されており`source: session-review`などの投入元情報を後続ステップで参照できる。
自己生成起点（`source: session-review`等）のフィードバックの信頼性と採否判定の扱いは、`agent-toolkit/rules/02-collaboration.md`「協調モード」節の優先順位の段落の直後の規定に従う。

取得した全件を「計画実装型」「通常型」へ分類する。
判定基準・混在時の並行制御・後始末分担は`references/plan-impl-feedback-flow.md`に従う。

フィードバック本文の反映先ファイル群がtarget_repoで指定されたリポジトリの外にある場合、target_repoの誤設定として扱う。
frontmatterのtarget_repoを実際の反映先リポジトリの値へ書き換えて`atk fb commit`で確定させる。
書き換え後のinboxは本セッションでは処理対象から外し、書き換え先リポジトリの次回process-feedbacks起動で扱う。
書き換え手段は`atk fb edit`の対話編集または直接編集後`atk fb commit`の2系統とする。

## ステップ2: 内容の調整

フィードバックの記述にとらわれず、各提案の追記先と抽象度を最適化する。
適用順は次のとおりとする。最初に「主提案の確定」で主提案を決める。
確定した主提案に対して「追記先の検討」「既存文書き換え時の根拠段落の維持照合」「抽象度の調整」
「規範側の再発予防案の併記検討」「既存SSOT規範との衝突回避」「原文列挙の意味的完全性の精査」
「引用素材の原文の表記点検」を順に適用する。「既存文書き換え時の根拠段落の維持照合」以降の
各項目の詳細は`references/review-checklists.md`「内容調整チェックリスト」節に従う。

- 主提案の確定: 提案がリカバリー手順の文書化・
  観測主体（レビュアー・lint・型チェック・hook等）の誤検出への適用除外コメント・前提明示の追記・
  例外規定の新設など場当たり対処に該当する場合の判定軸とする。
  この場合、根本対応（コード修正・設定変更・自動化スクリプトの整備・CI制約の追加・
  誤検出を生む規範側の改訂や撤廃など）が成立しないかを最初に検討する。
  誤検出由来の事象は規範側の改訂や撤廃を優先し、成立する場合は根本対応案を主提案とし元の追記提案は不採用扱いとする。
  成立しない場合は元の追記提案をそのまま主提案とする
  - 根本対応案を表層レベルの整理（内部重複の撤去のみ・既存実装の再利用のみなど）に限定した場合、
   `agent-toolkit:coding-standards`「責務帰属（SRP）分析の観点」節に従い判定確定前に分析する
  - フィードバック原文がワークアラウンド表現（フォールバック・回避策・迂回・失敗時対処など）を含む場合、
    元の追記提案を主提案として確定する前に根本対応を検討し、検討結果を
   `~/.claude/plans/<計画ファイルstem>-workaround-check.md`へ記録する必須ゲートとする。
    記録項目は「根本原因の候補」「根本対応が成立するか」「成立しない場合の理由」の3項目とし、
    複数フィードバック統合時も1計画1メモとする。
    メモ不在または未記入の場合はpretooluse.pyのWrite検査がブロックし採用確定できない
- 追記先の検討: 提案が示す対象ファイルにとらわれず、最適な追記先を選ぶ。
  コードに近い側（docstring・インラインコメント）や既存の関連セクションも候補に含める。
  フィードバックが既存追記そのものを否定する内容のときは削除候補も含める。
  追記先候補の節タイトルが定める性質（実装時契約・計画時手順・注意点・運用ノウハウ等）と
  追記内容の性質が整合するかを事前検証する。
  不整合時は既存節の別候補・意味論拡張・新設節を比較して追記先を確定する。節導入文との適用範囲の照合手順は`agent-toolkit:plan-mode`配下`references/norm-revision-checklist.md`「追記先節の選定」節に従う。
  追記先候補が既存bulletの子bullet位置となる場合は、追記前に親bulletの適用条件
  （前提・トリガー・対象事象）をReadで確認し、追記内容の前提と照合する。
  前提が異なる場合は子bullet位置での従属配置を採用せず、独立の親bulletとして配置するか、
  親bulletの前提条件文言を追記内容の前提を含む形に改める
  - 反映内容が複数スキル・複数エージェントへ一般化できる規範の場合、上位カテゴリへの反映を第一候補とする。
    上位カテゴリの例は上位ルールファイル（`agent-toolkit/rules/01-agent.md`）と横断スキル（`agent-toolkit:agent-standards`）
  - 1階層上のカテゴリで規範として成立するかを固有名詞除外テストで点検する。
    テスト手順は対象スキル名・対象CLIコマンド名を伏せて規範文が成立するかを確認する
  - 成立する場合は上位カテゴリへ昇格させてから反映先を確定する

## ステップ2.5: 採否判定前の網羅調査

ステップ3の批判的検討に入る前に、各提案の追記先となる既存記述と関連スキルを横断調査する。
調査観点全件は`references/review-checklists.md`「網羅調査チェックリスト」節に従う
（列挙時は実リストを再抽出し、要約列挙による項目漏れを避ける）。
調査結果を踏まえて初めてステップ3の批判的検討へ移行する。調査未了のまま採否を断定しない。

フィードバックが複数件の場合は、各フィードバックの追記先調査を`Explore`サブエージェントへ
個別に並列委譲する（必須ゲート）。
起動プロンプトは`references/explore-template.md`「Explore委譲雛形」節を機械転記して構築する。
同節の調査観点リスト全件を転記漏れなく含める。
メイン側で順次調査を進めるとコンテキストが膨張し、plan-mode工程6を実質的にバイパスした直接編集を誘発するため、
メイン側は返却された調査結果を集約してステップ3の内容を批判的に検討する。

フィードバック原文にリリース待ち・別計画完了待ち等の前提条件があれば実機検証で充足確認のうえ採否判定する。
前提が未充足なら「保留」として扱い`atk fb reject`せずprocessing配下へ残置する。
保留と判定したフィードバックはステップ4提示で「採否: 保留。<保留理由>」と表記し反映先・反映内容欄は
省略する。今回の計画作成対象から除外し次回起動時に再評価対象とし、全件保留の場合は計画作成に進まない
（`process-feedbacks-finish`への遷移は「ステップ5: 計画作成と実行」末尾の規定に従う）。

- 保留判定を確定した時点で`references/hold-with-tbd-inject.md`「保留判定時のTBD投入」節に従い
  `atk tb add --scope=hold --source=session-hold`によりTBD投入を必須ゲートとして実施し保留理由・確認手段・再評価契機を記録する
- 原文が`ClassName.field_name`・`ClassName.method_name`形式の識別子を言及する場合、
  `grep`ヒットに加えて対象ファイルReadでクラス所有・型・実在を意味論照合する
  （観点全件は`references/explore-template.md`観点7に集約）

## ステップ3: 批判的検討

調整後の内容で、各提案が対応に値するかを評価する。
判定観点全件は`references/review-checklists.md`「批判的検討チェックリスト」節に従う。
単一要求内は採用・不採用の二値評価のみを対象とし、複数の独立要求が並列に含まれる場合の分解判定は
同節「複数要求の分解判定」項に従い採用／部分採用／不採用で判定する。
原文の反映内容の量・粒度・実装レベルへの裁量介入は対象外とし、
原文の反映内容を自己判断で薄めた形の採用や、過剰と判断した箇所だけを除いて反映する部分採用は禁止する
（過剰と判断した場合も原文どおり採用するか不採用の二択とする）。

## ステップ5: 計画作成と実行

`agent-toolkit:plan-mode`スキルに従い計画を作成して実行する。
採用した全フィードバックへ同スキル工程5の母集団の悉皆点検（類似見直し）を必ず実施する。計画実装型のみの場合、
または通常型の採用0件（全件不採用・全件保留等）の場合は本ステップの計画作成を実施しない
（`references/plan-impl-feedback-flow.md`に従う）。

plan-impl-executor起動プロンプトの`## 追加指示`欄には計画ファイルに閉じた指示のみを書き、
後続スキル名（`process-feedbacks-finish`等）や当該スキル配下の工程内訳を含めない。
後続工程はメイン側が完了報告の受領後に別途起動する運用とする。

plan-modeスキル本文で規定される全工程の遵守を、auto mode下・自律実行系CLI経由の委譲下でも維持する。
各工程の実施義務は`agent-toolkit/skills/plan-mode/SKILL.md`「工程の進め方」節の「暗黙スキップ・明示バイパスの禁止」バレットの規定に従う。
plan mode下の場合の最終承認は`agent-toolkit:plan-mode`工程7の`ExitPlanMode`で得る。
全件採用扱いで計画作成へ進んだ後に採否判定を変更したい場合は、ユーザー側の割り込みで提示する前提とする。
計画ファイルの`## 実行方法`へ`agent-toolkit:process-feedbacks-finish`呼び出しを書かない。
記載すると`plan-impl-executor`がメイン専任の後始末（`atk fb adopt`等）を誤実行する。
後続工程はメイン側が完了報告の受領後に別途起動する。
`## 実行方法`本文と後続スキルの引き継ぎ範囲の区切りは`references/decision-format.md`「後始末順序」節に従う。

### 計画分割基準

基本は取得した全件を1度の計画作成で1計画へ統合し、1コミット・1レビューで完遂する。
以下のいずれかに該当する場合のみ複数計画へ分割する。分割後も各計画は1計画1コミットの原則を維持し、
`agent-toolkit:process-feedbacks-finish`工程1「全計画のコミット完了確認」で複数計画を順次処理する。
分割の根拠は観測可能な事実（外部issue参照の有無、フィードバック本文中の処理順序・グルーピング指示の記述）
のみとする。規模・作業量・所要時間・コンテキスト量等の自己推定量は分割根拠に含めない
（`agent-toolkit/rules/01-agent.md`「品質最優先」節fabricated-metricsカテゴリ参照）。

- 外部管理issue由来: フィードバック本文にissue番号・トラッカーURL等の外部参照があり、
  プロジェクト運用上コミット・PRでのissue紐づけが求められる場合は、自律判断で1イシュー1計画1コミットへ分離できる
- 処理順序・グルーピング指示: フィードバック本文中の処理順序・グルーピング指示（「ほかのフィードバックの処理完了後に」
  「ほかより優先して独立に」「〇〇のフィードバックと一緒に」など、記述形式は例示に限定しない）を解釈する。解釈結果は
  計画の分割・統合・実行順序へ反映し、「ほかの処理完了後に」に類する指示は独立の最終計画として全計画完了後に実行する形で扱う
- 既存計画ファイル参照: `references/plan-impl-feedback-flow.md`の判定基準に該当する場合は独立コミットで実施する
- 上記いずれにも該当しない場合は分割せず1計画へ統合する

計画作成後（計画実装型のみの場合は直列実装の消化後、採用0件の場合は採否の判定確定後）、
採用N件・不採用N件・保留N件のサマリーをユーザーに提示する。
サマリー提示後、コード変更有無に関わらず`agent-toolkit:process-feedbacks-finish`スキルへ進み後続工程を完遂する
（振り返り工程の実行主体規定は同スキル「工程5: 振り返り」節に従う）。
振り返り工程完遂前に完了を示す応答は発行しない。
