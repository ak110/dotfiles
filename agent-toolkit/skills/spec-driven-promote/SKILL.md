# spec-driven昇格ワークフロー

リリース後に次版ドキュメントを現行版ドキュメントへ統合する。

ワークフロー全体像は`agent-toolkit:spec-driven`スキルの「ワークフロー全体像」節を参照する。

次版ドキュメントを現行版ドキュメントへ統合し、次版ドキュメント配下を削除する。
現行版・次版ドキュメントの構成は`CLAUDE.md`に記録された配置・命名規則に従う。
配置は`agent-toolkit:spec-driven-init`および`agent-toolkit:spec-driven`で確定する。
未記録の場合は`references/spec-driven-framework.md`の既定
（現行版`docs/features/`・`docs/topics/`、次版`docs/v{next}/`）を用いる。

## 参照ファイル

- `agent-toolkit:spec-driven`の`references/spec-driven-framework.md`:
  用語定義・配置規約・現行版ドキュメントの記述レベル（手順1で使用）

## 手順

1. `agent-toolkit:spec-driven`スキルと`references/spec-driven-framework.md`を呼び出し、
   用語・配置規約・現行版の記述レベルを把握する
   - `CLAUDE.md`記録または既定値から次版ドキュメントの配置先（以降「次版配置」）を解決しておく
2. 次版配置を確認する
   - 次版総合ドキュメント（`OVERVIEW.md`）・作業テーマドキュメント（`{作業テーマ名}.md`）・
     計画ファイル（`plans/{作業テーマ名}.md`）の3種として扱う
   - 計画ファイルは統合作業の参照材料として活用する
    （設計判断と却下した代替案の根拠を確認するため）
3. 各作業テーマドキュメントの「現行版ドキュメントの対応先」節から統合先を特定する
   - 対応先のパスを`CLAUDE.md`記録の構成（機能ドキュメント／横断ドキュメント等）に照らして種別を判別する
4. 次版ドキュメントの内容を、判別した種別の現行版ドキュメントへ統合する（後述の統合方針に従う）
5. コード中の参照コメントを現行版ドキュメントのパスへ書き換える
6. 次版配置を削除する
   - 削除対象には次版総合ドキュメント・作業テーマドキュメント・計画ファイル群（`plans/*.md`）を含む
   - 先に手順5を済ませておくことで、参照元と新パスの突き合わせが容易になる
7. `grep -r`などで次版配置のパスへの参照が残っていないことを確認する
8. リンクと参照コメントの指し先が実在のファイルを指していることを検証する

## 統合方針

- 新規機能は機能ドキュメントとして新規ファイルを作成する
- 既存機能の改修は対応先の現行版ドキュメントへ統合する
  - 現行版ドキュメントのタイトル・ファイル名は保持し、新規追加分は節として吸収する
  - 既存節がある場合は追記または上書き、無い場合は節を新設する（「却下した代替案」節も対象）
  - 採用基準を満たす代替案が1件もない担当ファイルでは、空の節を新設しない
    （完了報告でその旨を明示する）
  - 事実が並立する場合（背景・スコープ・受け入れ基準など）は追記する
  - 判断が衝突する場合は新判断で上書きし、旧決定は「却下した代替案」へ廃止理由を一行添えて移す
- 横断ドキュメントは横断種別の現行版ドキュメントへ統合する
  - 横断/機能の判定は作業テーマドキュメントの「現行版ドキュメントの対応先」節を根拠とする
- 「主対応」「関連」のうち「関連」として参照されているドキュメントは、
  新たな共通決定が含まれる場合のみ追記し、それ以外は更新対象外とする
- 主要設計判断は現行の判断として整理する
- 却下した代替案は判断履歴として残す
- 次版ドキュメントに含まれる内部実装詳細（クラス・関数・変数名、ファイルパス、モジュール分割など）は、
  現行版へ統合する際に外部仕様レベルへ整理して除去する
  - 記述レベルの規定は`references/spec-driven-framework.md`を参照
- 次版ドキュメントの「現行版ドキュメントの対応先」節は統合先を案内する次版固有の記述なので、
  現行版へ持ち込まない
- 次版総合ドキュメントは次版作業の進行管理用ファイルなので統合せず、
  手順6での次版配置の削除時に併せて破棄する
- 計画ファイルは統合作業の参照材料として活用し、現行版へは持ち込まず手順6で削除する

### 統合先の選定

作業テーマドキュメントの「現行版ドキュメントの対応先」節を起点として、
次の順で統合先を確定する。新規ファイル作成は最後の選択肢とする。

- 対応先として明示された既存ファイル内の既存節へ吸収できないかを最初に検討する
- 既存節に収まらない場合のみ新規節を当該ファイル内に新設する
- 当該ファイル内に収まらない場合のみ、対応先と同一ディレクトリへ新規ファイルを作成する
- 単一機能内で閉じる仕様は機能ドキュメント側に記述し、複数機能で共有される共通仕様のみ横断ドキュメントへ置く
 （`agent-toolkit:spec-driven-init`スキルのドキュメント単位の方針に従う）
- 対応先候補が複数挙がる場合は、利用者視点で読み手が当該仕様を探す入口に近い側を採用する
- いずれの既存ファイルにも収まらず新規ファイル化も妥当でない判断履歴は、
  完了報告でテーマ名・除外理由・対応する次版ドキュメント節を明示したうえで、
  git履歴のみでの保全を許容する

### 統合後の肥大化防止

統合は追記だけでなく、現行版ドキュメント全体の簡潔さを保つ整理として実施する。
記述レベル（外部仕様レベル限定）は`references/spec-driven-framework.md`の規定に従う。

- 追記の前に統合先ファイル全体を通読し、既存記述と内容が重複する箇所を特定する
  - 重複時は新記述に統一して旧記述を除去し、節として成立しない断片は併せて削除する
- 上書き対象の旧決定を「却下した代替案」へ退避する際も、退避理由が既存の却下案と重複する場合は
  既存項目への追記に留め、同等項目を別個に並べない
- 公式ドキュメント・規格・上流ライブラリのURLで同等情報に到達できる記述は、
  該当URLへの参照に置き換えて本文の再掲を避ける
- リポジトリ内の設定ファイル・コード本体に同等記述が存在する場合は、
  当該ファイルへの参照（パスと該当箇所の見出し）に置き換えて本文の再掲を避ける
- ワークアラウンド手順・運用回避策として記述されていた次版ドキュメントの内容が、
  本実装で根本対応として解消される場合は、現行版へ手順を持ち込まず解消済みの旨だけを残す
- 既存ドキュメントに残る旧仕様の説明・移行措置の案内が本実装で不要になった場合は、
  統合と同じセッション内で除去する
- 手順5で扱うコード中の参照コメントも肥大化対象とする
  - 同一ファイルから複数の現行版ドキュメントを参照する形になった場合は、宛先を最も適切な1箇所へ統合する
  - 旧次版ドキュメントへの参照は除去する
