# ユーザー確認後回し処理の汎用基盤

## 1. 引数と呼び出し前提

呼び出し元は次の情報を引数として渡す。

- `tbd_path`: TBD.md保存先パス（既定はカレントディレクトリ直下の`TBD.md`）
- `scope`: 呼び出し元固有のスコープ説明（処理単位の境界・1コミット境界・対象ファイル集合など）
- `additional_stop_triggers`: 追加停止契機の列挙（任意。本スキル既定の停止契機に追加する）

呼び出し元は本スキルを呼び出す前に次の前提を確立する。

- 処理対象（計画ファイル群・タスクリスト等）が存在する
- 処理単位ごとの完了判定基準が定義されている
- 本スキル既定のオーバーライド対象外となる固有の確認契機があれば、呼び出し元側で明示する

## 2. 起動前提チェック

本スキルはauto mode下での自律実行を前提とする。
起動直後にシステムプロンプトの`Auto Mode Active`注入を確認し、含まれない場合は
「本スキルはauto mode専用・再起動はauto mode有効化後に行う」旨を1行通知して即時終了する。
当該チェックは「5. 停止判定」の前提崩れ停止と別経路として扱う。

## 3. ユーザー確認規範のオーバーライド対象

本スキル実行中は次の確認契機をTBD.md追記＋暫定判断＋続行へ置換する。

- `01-agent.md`「品質最優先」節および「ユーザーとともに考える」節の確認要求全般
- `02-claude-code.md`「auto mode下での挙動」節「迷ったら作業を止めて確認」
- `agent-toolkit:plan-mode`の要件対話・認識合わせ・恒久化検討・リファクタリング検討の各ユーザー確認
- `agent-toolkit:plan-impl`「2.5サブエージェント完了報告の検収」のうちユーザー確認を要する分岐

サブエージェント（呼び出し元から起動される実装系・配下の`plan-implementer`等）は
`01-agent.md`「縮退表明は発行しない」項の規定に従う。
呼び出し元が定義した処理単位の全変更を実装・検証・コミットまで完遂する。
処理対象に記載された破壊的操作・外部送信（`git push`・外部APIへの送信・データ削除等）も
通常工程として実行し、停止理由としない。
確認事項はTBD.mdへQブロックとして記録し続行根拠とする（停止判断の根拠としない）。
本スキル全体の実行前提が崩れる場合の停止規定は「5. 停止判定」に従う。

## 4. TBD.md書式

ユーザー確認事項の記録先は次の順序で判定する。

- `command -v atk`が成功する場合は`atk fb tbd-add <repo_path> --scope=<scope> <message>`で記録する
  - `tbd_path`引数は使用しない
  - 重複防止・ユーザー指摘の追記は呼び出し元が`atk fb tbd-list`・`atk fb tbd-edit`で扱う
- `atk`が存在しない場合は以下のTBD.md書式に従う

TBD.mdの目的はユーザー確認事項の記録に限定する。
次の用途では使用しない。

- 縮退範囲・未着手範囲・段階分割・実機検証スキップ宣言の記録
- 発見済み既存バグの記録（同セッション内で対処する。次サイクル改修への先送り記録としない）
- ユーザー確認を要しない実装未完了の範囲の記録（同セッション内で対処する）
- 並列実行制約・作業ツリー全体への影響など、呼び出し元（メイン側または委譲先）が
  追加判断なしに実行・委譲できる検証・確認の記録（完了報告で呼び出し元へ返却する）。
  ユーザー判断・ユーザー操作を要する事象は本項目の対象外であり、引き続きTBD.mdへQブロック化する

章構成: 冒頭に本ファイルの説明文、呼び出し元のスコープ区分（テーマ・タスク・モジュール等）ごとに
`## {scope name}`配下に`### Q{n}: {要約}`ブロックを並べ、末尾に`## ユーザー指摘`節を設ける。

質問ブロックは選択式・YES/NO・自由記述の3類型から質問の性質に合わせて選ぶ。
各Qブロックには次の項目を含める。

- 発生工程（計画・実装・検証・コミット等）
- 関連箇所（ファイルパスまたは計画文書の節名）
- 背景（何が決められず何を仮置きしたか）
- 選択肢または質問本文
- 暫定採用（採用ラベルと根拠）
- 回答欄（HTMLコメントで「ユーザーはこの行以降に回答を追記する」を明記）

呼び出し元が固有の質問雛形・追記欄雛形（ユーザー指摘欄など）を持つ場合は、当該雛形の参照先を呼び出し元側で示す。

回答欄判定: 直後のHTMLコメント行を除き非空行があれば「回答済み」、非空行がなければ「未回答」と判定する。

回答済み質問の反映タイミングは呼び出し元が処理単位の特性に応じて定義する。

## 5. 停止判定

auto mode未活性時の起動拒否は本節の停止契機と別経路で扱う（「2. 起動前提チェック」節に従う）。

停止契機は次のいずれかとする。

- 本スキル全体の実行前提崩れ（処理対象不在・呼び出し元から渡された`scope`の根拠喪失等）
- 呼び出し元から`additional_stop_triggers`として明示された契機

検知時は本スキル既定のオーバーライドを解除し、呼び出し元へ停止理由と再開時の確認事項を返却する。

処理対象の実行中に発生する破壊的操作・外部送信は、呼び出し元の通常工程として扱い停止理由としない。

返却メッセージには、停止契機・対象処理単位の識別子・原因の要約・再開時の確認事項を含める。
