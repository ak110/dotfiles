# ドキュメント品質

一般ドキュメント（Markdown・README・技術文書・API文書、コメント類）の品質指示。
本スキルの各指針は、agent.md「言語表現」章の最上位原則（「自然で簡潔な言い回し・文章スタイルで書く」）と、
その文体面の具体化であるJIS規格スタイル・NHKスタイルを、表記・構成の面でさらに具体化・補足するものとする。
コーディングエージェント向け文書も対象に含むが、その編集時は`agent-toolkit:agent-standards`を併用する。

## 読み手別の追加注意点

成果物の対象読者ごとの追加注意点（対象読者と文体の基本方針はagent.md「言語表現」章を参照）。

- エンドユーザー向け: 内部実装は知らない前提とし、専門用語は初出時に補足する
  - 対話型UI: 操作・状態・対処を端的に伝え、エラーは原因と対処を一組で示す
  - ドキュメント:「何ができるか」「どう使うか」を中心に利用者の操作目線で書く
- 開発者向け: 前提知識を共有する開発者を想定し初歩説明は省くが、コード識別子レベルの詳細は扱わない
  - 「なぜそう設計したか」「どこに注意が必要か」を中心に書く
- コーディングエージェント向け文書: 前置き・自己紹介・書き手目線の解説は不要とし、判断に必要な情報のみ直接記述する

## 構成・構造

- 対象読者を明確にし、冒頭に目的・概要を置いて読者が「読むべきか」を即座に判断できるようにする
- 前提条件（必要な知識・ツール・環境）は本題の前に配置する
- 見出しの階層は3段（`##` / `###` / `####`）までを目安とする
- 関連する内容は近くにまとめ、読者が文書内を往復しなくて済むようにする
- コマンド例やコードスニペットはコピー&ペーストでそのまま動くことを前提に書く
- 外部リソースへのリンクは、リンク切れリスクを考慮して要点を本文にも記載する
- トップダウン（段階的詳細化）で構成し、追記時も末尾追加でなく既存記述との粒度・配置の整合性を確認する

## 日本語の書き方

- 観測事実ベースで書く。動作・状態・構造は誰でも同じ結論に至る客観的・中立的な表現を採用する
- 推奨形を先に列挙し、避ける表現を排除する順序で書く（禁止条件から発想すると不自然な言い換えに陥りやすい）
- 一段詳細な話を続けるときは「例えば」などの接続表現を付けて論理の飛躍を回避する
- 直接的な書き言葉で一文一義に書く
- 断定的で安定した語尾を使う（常体では「〜する」「〜である」「〜となる」「〜とする」「〜ものとする」）
- 一語で意味が通る名詞は単体で表記する（後ろに別の名詞を付け足して二語化しない）
  - 非推奨例:「設定値」「構成情報」「入力データ」「出力結果」／推奨例:「設定」「構成」「入力」「出力」「結果」
  - 単体で意味が通らない場合に限り、必要な修飾語・複合語を組み合わせる
- 比喩動詞を避け、観測できる事実を表す標準的な技術用語を使う
  - 推奨例:「例外を送出する」「プロセスを終了する」「設定を読み込む」「実行する」「上書きする」「無効化する」「書き込む」
- 危険語彙・禁止パターンを含む観点の対比例は`references/tone-examples.md`を参照する
  - 参照時は必ずサブエージェントを使用する（コンテキスト汚染回避のためメインから直接Readしない）

### 行動指示の述部

- 行動指示を終止形で結ぶと主体が不明確になり、説明文と混同されやすい
  - 常体の推奨例:「〜を参照」／敬体の推奨例:「〜を参照してください」
  - 例外: コーディングエージェント向け文書は常体の指示終止形（「〜する」「〜を確認する」）を許容する
- 同一ファイル内で述部形式が混在する場合は優勢な形式へ統一する
  - 拮抗時はコーディングエージェント向け文書で指示終止形、それ以外で体言止めに揃える
  - 追記時は対象節の既存述部を確認してから整える

### 語順と主述

- 結論・要点・規定の中身を文の前方に置く
  - 種別や分類のラベル（「最上位の原則は」など）を機械的に前置きしない
  - 規定の中身が長い引用・列挙のときは前方に置き規範述語で締めてよい（「Xを〜とする」の形を崩さない）
- 主語と述語を近づける（主語直後に長い括弧列挙や挿入句を置かず、サブバレットや後続文へ分ける）
- 同一の話題・対になる概念は隣接させて並べる
- 規定は動作主の曖昧な能動表現を避け、規範述語（「〜とする」「〜ものとする」）で結ぶ
- 説明文を体言止めの断片で分断せず、主語・述語のそろった1文に統合する（行動指示の述部は前節に従う）
  - 例外: 用語定義の見出し（「用語名: 〜の状態」など辞書形式）は体言止めで統一してよい

## 排除すべき癖・AIスロップの抑止

JIS規格スタイル指定下でもコーディングエージェントの出力には以下の癖・冗長（AIスロップ）が混入しやすい。意識的に排除する。

| 合理化されがちな判断 | 対処 |
| --- | --- |
| 主観的修飾の混入 | 程度や評価を主観で示す形容詞・形容動詞を名詞句に付けない |
| 営業文書調・宣伝調の混入 | 特徴・理由・魅力を宣伝句で要約する書き方を使わない |
| 程度副詞による強調 | 強調や勧誘を意図する副詞を技術文書に持ち込まない |
| 比喩動詞での表現 | 観測動詞で記述する。推奨例:「適用する」「採用する」「導入する」「反映する」「踏襲する」「整合する」 |
| 日本語文中への英字の混入 | 訳出回避の英字直挿入を採用せず日本語・外来語（カタカナ）に置き換える。バッククォート付き識別子・ツール名・コマンド名は対象外 |
| ユーザー指示・対話履歴の文体への追従 | 入力の口語・俗語・略語・敬体・絵文字を持ち込まず、意味のみ取り出して本スキル方針で書く |
| 引用コード・既存コメントの文体の流用 | 引用元の古い表記・慣用句に引きずられず、本スキル方針へ書き換えてから取り込む |
| 前置き・導入文の付与 | 章名・節名で示した目的を本文で再宣言せず、本題から記述を始める |
| 同義言い換えによる補強 | 同じ趣旨を別表現で並べず、最も短い1表現に集約する |
| 抽象主語・抽象副詞による一般化 | 対象を曖昧化する冒頭表現を採用せず、対象を直接指す |
| 体裁を整える背景・歴史・展望の付加 | 求められた粒度を超える背景・歴史・展望を機械的に補わない |
| 既存節の構造を機械的に流用する | 情報量に応じて構造を選び、同趣旨が複数形式に重複するなら最も短い1形式に集約する |
| 「網羅性のため」の例示・周辺列挙 | 主目的に必要な代表例のみ書き、自明な周辺を補足列挙しない |
| 比喩名詞での表現 | 観測可能な事象名で記述する。推奨例:「原因」「契機」「経路」 |
| 分析過程の段階的展示 | 結論を冒頭に置く。推論手順を段階的に並べる小見出しは設けない |
| 判定の曖昧化 | 観測事実か明確な判定かを直接書き、判定を曖昧化する留保表現を避ける |

## 日本語の表記ルール

プロジェクト方針が無い場合の方針（JTF日本語標準スタイルガイドおよびJIS Z 8301:2019を参考）。
原則としてJTF日本語標準スタイルガイドに準拠する。textlintの`preset-jtf-style`で検査される項目はそちらに従う。

- 日本語と全角括弧の境界にスペースを挿入しない（例: `設定ファイル（既定値あり）`）
- 全角丸括弧は同一論理行内で開きと閉じを完結させる。リスト項目内で改行する場合は閉じてから改行する
- 全角丸括弧の入れ子は避ける。内側の補足にはカンマや読点で区切る表現を使う
- 英文脈の半角括弧は英数字との間にスペース1つ（例: `Python (CPython 3.12)`）
- 外来語の語尾には長音符を付ける（例:「ユーザー」「サーバー」「パラメーター」。3音以上の語は省略しない）
- 技術用語・固有名詞・コード識別子は原典の表記を優先する
- 文面案を執筆する前に`references/textlint-violations.md`を読み込み、頻出違反パターンを確認する
 （メインエージェント・サブエージェントとも）

## 改訂・保守

- 追記内容の粒度が既存記述と整合しているか確認し、必要に応じて既存部分や章構成も変更する
- 陳腐化した記述は積極的に削除する。「古い情報が残る」ことは「情報がない」より有害
- SSOTを維持する。重複が避けられない場合はどちらが正かを明記する
- ドキュメントの変更がコードの変更と連動する場合は、同じコミットまたはPRに含める
- 本文から数えて把握できる件数・列挙（「N件」「N観点」「N項目」など）は明記しない
  - 対象一覧と並べてサービス・コンポーネント・対応項目を再列挙する形式も冗長なため避ける
  - 仕様書の網羅性チェックなどで意図的に明示する場合は、本文との整合を保つ責務を負う
- 本文と連動する従属箇所（フォーマット例・コードブロック・目次・件数表記・要約・mermaid図のラベル等）は、
  本文変更時に合わせて更新し片方だけ更新の矛盾を残さない

## コメントの記述粒度

- ソースコードには必要最小限のコメントを書き、コードから自明な処理にコメントを付けない
  - 書く場合は「なぜそうするか」か「何をしているか」に限る
    - 「なぜ」: 設計判断の背景・根拠・閾値選定理由・トレードオフ・バグ回避理由。ワークアラウンドはその旨と参照先を明記
    - 「何を」: 数行以上の塊の短い要約・非自明な制御フロー
- 実装詳細はインラインコメントに書き、モジュール冒頭のdocstringへ集約しない（冒頭は役割・概要のみ）
- 同概念を扱う複数の関数で説明が重複する場合、本質的な説明は呼び出し関係の入口側1箇所に集約する
  - 内部関数側は1行サマリーか入口関数への参照で済ませる
- ドキュメントはコードからわからない情報や意図・設計理由を書き、現在の状態を直接記述する
  - 「〜へ集約済み」「〜へ移行済み」のような時系列表現は使わない（履歴はコミットメッセージに委ねる）

## 口語表現チェック

口語的な日本語表現の混入を防ぐため、`scripts/check_colloquial.py`の実行を必須とする。

```sh
uv run --script path/to/writing-standards/scripts/check_colloquial.py path/to/file.md
```

- 検出範囲: `.md`・`.py`・`.txt`・`.yaml`・`.yml`・`.toml`（コードブロック内のコメントも含む）

コンテキスト汚染を避けるため、以下のファイルはメインエージェントから直接Readしない。

- `agent-toolkit/skills/writing-standards/references/tone-examples.md`
- `agent-toolkit/scripts/_colloquial_words.txt`

確認が必要な場合はExploreサブエージェント経由で参照し、修正が必要な場合は`plan-implementer`経由で行う。

## Markdown記述スタイル

行幅・句点位置などの細則はmarkdownlint・textlintで自動検証される。
頻出textlint違反パターンは`references/textlint-violations.md`を参照する。

- `**`は強調したい箇所のみとし、箇条書きの見出しなどでの使用は禁止する
- markdownlintが通るように書く。特に注意するルール:
  - `MD022`: 見出しの前後に空行を置く／`MD031`: フェンスドコードブロックの前後に空行を置く
  - `MD040`: フェンスドコードブロックに言語を指定する
- 1行の表示幅は半角換算で127を上限とする（全角=2、半角=1。例外はコードブロック・表）
  - frontmatterが長くなる場合はYAML複数行記法（`>`など）で収める
  - 1文ごと（`。`ごと）に改行し、`scripts/check_line_width.py`で検査する
- 箇条書きは1項目1文で書き、複数文は別項目分割かサブバレット階層化で表す
  - 120字超の単文や半角識別子・コマンド名を含む列挙は`ja-technical-writing/sentence-length`・
    `preset-jtf-style 1.1.3.箇条書き`違反を招くため、最初から別項目化・サブバレット化で予防する
- 図はMermaid記法で書く
- 別のMarkdownファイルへのリンクは用途で書き分ける
  - Markdownソースのまま読まれる想定（`README.md`・GitHub閲覧前提のdocs等）は
    `[ルートからのパス](相対パス)`形式（例: `[docs/guide/setup.md](guide/setup.md)`）
  - HTMLに変換される想定（mkdocs・Sphinx等）は自然な表現で書く（例: `[セットアップ手順](guide/setup.md)`）
- テーブルは列数が多い場合や内容が長い場合は箇条書きへの変換を検討する
  - 特定列のみ長文化する場合はキー単語に留めて表下の補足文へ退避する

## 技術文書の書き方

- 手順書（セットアップ・デプロイ・移行など）: 番号付きリストで1ステップ1動作。
  前提条件・事前準備を冒頭に、成功時の期待結果を各ステップか末尾に、失敗時の対処・ロールバックも記載する
- トラブルシューティング: 症状 → 原因 → 対処の3要素で構成し、よくある問題から並べる。
  エラーメッセージは検索しやすいよう正確に引用する
- API文書: エンドポイント・パラメーター・レスポンスを網羅し、具体例とエラーケースも記載する
- コミットメッセージ: 読者はエンドユーザー。内部リファクタリング等の詳細は不要。プロジェクト方針があれば従う

## ディレクトリ構成

OSSプロジェクトの典型的なドキュメント配置（プロジェクト方針がある場合はそちらに従う）。

- ルート直下: `README.md`、`LICENSE`
- `docs/`配下: 種別ごとにサブディレクトリ（`docs/guide/`・`docs/development/`・`docs/api/`等）
- ファイル粒度: 1ファイル1トピック。1ファイル300行超で分割を検討する
- ファイル命名: ケバブケース（`setup-guide.md`）。定番ファイルは大文字の慣例に従う
- サブディレクトリの入口index: nav機能で目次を自動生成できる場合は不要（手動メンテのindex.mdは陳腐化しやすい）

## README規約

READMEはプロジェクトのトップレベルに配置し、初見の読者が最初に参照する。

- 標準構成（プロジェクト方針がない場合の例。規模と性質に応じて取捨選択）:
  プロジェクト名と一文の概要 → 主な機能・特徴 → インストール・セットアップ → 基本的な使い方 → ドキュメントへのリンク
- `CLAUDE.md`との役割分担: READMEは人間の読者向け、`CLAUDE.md`はClaude Code向け。
  重複情報はREADMEを正とし、`CLAUDE.md`からは参照するか要点のみ再掲する
