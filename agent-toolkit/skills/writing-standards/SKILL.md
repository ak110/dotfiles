# ドキュメント品質

一般ドキュメント（Markdown・README・技術文書・API文書、コメント類）の品質指示。
コーディングエージェント向け文書も対象に含むが、その編集時は`agent-toolkit:agent-standards`を併用する。

## 読み手別の書き分け

対象読者と文体の基本方針（読者・文書の対応表、JIS規格／NHKスタイルの定義、コーディングエージェントの発話、個別注意）は
`agent-toolkit/rules/styles.md`に置く。読み手別の追加注意点は次の通り。

- エンドユーザー向け: 内部実装は知らない前提、専門用語は初出時に補足する
  - 対話型UI: 操作・状態・対処を端的に伝え、エラーは原因と対処を一組で示す
  - エンドユーザー向けドキュメント:「何ができるか」「どう使うか」を中心に利用者の操作目線で書く
- 開発者向け: プロジェクト前提知識を共有する開発者を想定し、初歩説明は省くがコード識別子レベルの詳細は扱わない
  - 「なぜそう設計したか」「どこに注意が必要か」を中心に書く
- コーディングエージェント向け文書: 前置き・自己紹介・書き手目線の解説は不要。判断に必要な情報のみを直接記述する
  - 冗長な説明はトリガー精度低下とコンテキスト消費を招く

## 構成・構造

- 対象読者を明確にする
- 冒頭にドキュメントの目的・概要を置き、読者が「読むべきか」を即座に判断できるようにする
- 前提条件（必要な知識・ツール・環境）は本題の前に配置する
- 見出しの階層は3段（`##` / `###` / `####`）までを目安とする
- 関連する内容は近くにまとめ、読者が文書内を往復しなくて済むようにする
- コマンド例やコードスニペットはコピー&ペーストでそのまま動くことを前提に書く
- 外部リソースへのリンクは、リンク切れリスクを考慮して要点を本文にも記載する
- トップダウン（段階的詳細化）で構成する。全体の構成・見出しから書き始め、徐々に肉付けする。
  追記時も無条件に末尾追加せず、既存記述との粒度・配置の整合性を確認する

## 日本語の書き方

- 観測事実ベースで書く。動作・状態・構造は誰でも同じ結論に至る客観的・中立的な表現を採用する
- 冗長な前置き・修飾・言い換えを排除し、最も短い1表現に集約する
- 推奨形を先に列挙し、当てはまらない場合に避ける表現を排除する順序で書く
  - 禁止条件から発想すると不自然な言い換えに陥りやすい
- 文章の並びは粒度を揃える。追記時は対象個所の記述粒度と整合しているか確認し、必要に応じて既存部分や章構成も変更する
  - 一段詳細な話を続けるときは「例えば」などの接続表現を付けて論理の飛躍を回避する
- 各節の対比例（書き換え前後）は`references/tone-examples.md`を参照する
  - 参照時は必ずサブエージェントを使用する（コンテキスト汚染回避のためメインエージェントから直接Readしない）

### 語尾と語彙

- 直接的な書き言葉で一文一義に書く
- 断定的で安定した語尾を使う（常体では「〜する」「〜である」「〜となる」「〜とする」「〜ものとする」）

### 名詞句

- 一語で意味が通る名詞は単体で表記する（後ろに別の名詞を付け足して二語化しない）
  - 非推奨例:「設定値」「構成情報」「入力データ」「出力結果」「処理結果」
  - 推奨例:「設定」「構成」「入力」「出力」「結果」
  - 単体で意味が通らない場合に限り、必要な修飾語・複合語を組み合わせる

### 動詞

- 標準的な技術用語を使う（比喩動詞は使わない）。
  推奨例:「例外を送出する」「プロセスを終了する」「処理を呼び出す」「メモリーを解放する」「接続を確立する」「設定を読み込む」
- 処理・状態遷移・データ変更は技術的に観測できる事実を表す動詞で書く。
  推奨例:「実行する」「起動する」「上書きする」「置き換える」「無効化する」「書き込む」「読み込む」

### 行動指示の述部

- 行動指示を終止形で結ぶ（例：「～を参照する」）と、主体が不明確になり、説明文と混同されやすい
  - 常体の場合の推奨例:「〜を参照」
  - 敬体の場合の推奨例:「〜を参照してください」
- 例外: コーディングエージェント向け文書は常体の指示終止形を許容する（許容例:「〜する」「〜を確認する」）
- 同一ファイル内に複数の述部形式が混在している場合、優勢な形式へ統一する。
  優勢度が拮抗する場合は、コーディングエージェント向け文書では指示終止形（〜する）、
  それ以外の文書では体言止め（〜を参照）に揃える。
  追記時は対象節の既存述部を確認してから追記文面を整える

## 排除すべき癖

JIS規格スタイル指定下でもコーディングエージェントの出力には以下の癖が混入しやすい。意識的に排除する。

| 合理化されがちな判断 | 対処 |
| --- | --- |
| 主観的修飾の混入 | 程度や評価を主観で示す形容詞・形容動詞を名詞句に付けない |
| 営業文書調・宣伝調の混入 | 特徴・理由・魅力を宣伝句で要約する書き方を使わない |
| 程度副詞による強調 | 強調や勧誘を意図する副詞を技術文書に持ち込まない |
| 比喩動詞での表現 | 観測動詞で記述する。推奨例:「適用する」「採用する」「導入する」「反映する」「踏襲する」「整合する」 |
| 日本語文中への英字の混入 | 訳出を回避するための英字直挿入を採らず、日本語または外来語（カタカナ表記）に置き換える。`grep`・`diff`等のバッククォート付きコード識別子・ツール名・コマンド名は対象外 |
| ユーザー指示・対話履歴の文体への追従 | 入力に含まれる口語表現・俗語・略語・敬体・絵文字を成果物に持ち込まず、指示の意味のみ取り出して本スキル方針で書く |
| 引用コード・既存コメントの文体の流用 | 引用元の古い表記・慣用句に引きずられず、本スキル方針へ書き換えてから取り込む |

## AIスロップの抑止

AIスロップ（AIが情報量に対して冗長に生成する文章）を避ける。

| 合理化されがちな判断 | 対処 |
| --- | --- |
| 前置き・導入文の付与 | 章名・節名で示した目的を本文で再宣言せず、本題から記述を始める |
| 同義言い換えによる補強 | 同じ趣旨を別表現で並べず、最も短い1表現に集約する |
| 抽象主語・抽象副詞による一般化 | 対象を曖昧化する冒頭表現を採らず、対象を直接指す |
| 体裁を整える背景・歴史・展望の付加 | 求められた粒度を超える背景・歴史・展望を機械的に補わない |
| 既存節の構造（本文・表・サブバレットなど）を機械的に流用する | 情報量に応じて構造を選ぶ。同趣旨が複数形式に重複するなら最も短い1形式に集約する |
| 「網羅性のため」例示・周辺ケースを列挙する | 主目的に必要な代表例のみ書き、自明な周辺を補足列挙しない |

## 日本語の表記ルール

プロジェクト方針が無い場合の方針（JTF日本語標準スタイルガイドおよびJIS Z 8301:2019を参考）。
原則としてJTF日本語標準スタイルガイドに準拠する。textlintの`preset-jtf-style`で検査される項目はそちらに従う。

- 日本語と全角括弧の境界にスペースを挿入しない（例: `設定ファイル（既定値あり）`）
- 英文脈の半角括弧は英数字との間にスペース1つ（例: `Python (CPython 3.12)`）
- 外来語の語尾には長音符を付ける（例:「ユーザー」「サーバー」「パラメーター」。3音以上の語は省略しない）
- 技術用語・固有名詞・コード識別子は原典の表記を優先する
- 頻出違反パターンは`references/textlint-violations.md`を参照する

## 改訂・保守

- 追記する内容の粒度が既存記述と整合しているか確認し、必要に応じて既存部分や章構成も変更する
- 陳腐化した記述は積極的に削除する。「古い情報が残る」ことは「情報がない」より有害
- SSOTを維持する。重複が避けられない場合はどちらが正かを明記する
- ドキュメントの変更がコードの変更と連動する場合は、同じコミットまたはPRに含める
- 同一ファイル内で本文と連動する従属箇所（フォーマット例・コードブロック・目次・
  項目数表記（「N観点」「N項目」など）・要約記述）が併存する場合、本文を変更したら従属箇所も合わせて更新し、
  片方だけ更新されて矛盾が残らないようにする

## コメントの記述粒度

- ソースコードには必要最小限のコメントを書き、コードから自明な処理にコメントを付けない
  - コメントを書く場合は以下のいずれかに限る
    - なぜそうするか（設計判断の背景・根拠、閾値選定理由、トレードオフ、バグ回避理由）。
      ワークアラウンドの場合はその旨と参照先を明記する
    - 何をしているか（数行以上のコードの塊に対する短い要約、非自明な制御フロー）
- 細かい判断や実装詳細はインラインコメントに書き、モジュール冒頭のdocstringに集約しない。
  冒頭はモジュールの役割・概要のみに留める
- 同概念を扱う複数の関数（公開ラッパーと内部実装、薄いオーバーロード等）でdocstring・説明コメントが重複する場合、
  本質的な説明は呼び出し関係の入口側1箇所に集約する。
  内部関数側は1行サマリーに留めるか入口関数への参照で済ませる
- ドキュメントはコードを見ただけではわからない情報や意図・設計理由を書く
- ドキュメントは現在の状態を直接記述し、「〜へ集約済み」「〜へ移行済み」のような時系列表現は使わない。
  履歴はコミットメッセージなどに委ねる

## 口語表現チェック

- 口語的な日本語表現の混入を防ぐため、`scripts/check_colloquial.py`の実行を必須とする

    ```sh
    uv run --script path/to/writing-standards/scripts/check_colloquial.py path/to/file.md
    uv run --script path/to/writing-standards/scripts/check_colloquial.py path/to/dir
    ```

  - 検出範囲: `.md`・`.py`・`.txt`・`.yaml`・`.yml`・`.toml`
    - コードブロック内のコメントも含む

コンテキスト汚染を避けるため、以下のファイルはメインエージェントから直接Readしない。

- `agent-toolkit/skills/writing-standards/references/tone-examples.md`
- `agent-toolkit/scripts/_colloquial_words.txt`

確認が必要な場合はExploreサブエージェント経由で参照する。修正が必要な場合は`plan-implementer`経由で行う。

## Markdown記述スタイル

行幅・句点位置などの細則はmarkdownlint・textlintで自動検証される。
頻出textlint違反パターンは`references/textlint-violations.md`を参照する。

- `**`は強調したい箇所のみとし、箇条書きの見出しなどでの使用は禁止する
- markdownlintが通るように書く。特に注意するルール:
  - `MD022`: Headings should be surrounded by blank lines
  - `MD031`: Fenced code blocks should be surrounded by blank lines
  - `MD040`: Fenced code blocks should have a language specified
- 1行の表示幅は半角換算で127を上限とする（全角=2、半角=1）
  - frontmatterが長くなる場合はYAML複数行記法（`>`など）で収める
  - 1文ごと（`。`ごと）に改行する
  - 機械的チェック: `scripts/check_line_width.py`で検査できる
    - `uv run --script path/to/writing-standards/scripts/check_line_width.py path/to/file.md`
  - 例外: コードブロック・表

- 箇条書きは1項目1文で書く
  - 複数文が必要なときは1項目内に句点で並べず、別項目に分割するかサブバレットへ階層化する
  - 句点で複数文を1項目に並べるとtextlintの`ja-technical-writing/sentence-length`違反を招きやすい
- 120字を超える単文は別項目に分割するかサブバレットへ階層化する
  - textlintの`ja-technical-writing/sentence-length`や`preset-jtf-style 1.1.3.箇条書き`の違反を避けるため
- 図はMermaid記法で書く
- 別のMarkdownファイルへのリンクは用途によって書き分ける
  - Markdownソースのまま読まれる想定（`README.md`・`CLAUDE.md`・GitHub閲覧前提のdocsなど）:
    `[プロジェクトルートからのパス](記述個所からの相対パス)`形式（閲覧者がパスを一目で把握できる）
    - 例: `docs/api.md`から`docs/guide/setup.md`へリンク → `[docs/guide/setup.md](guide/setup.md)`
  - HTMLに変換される想定（mkdocs・Sphinx・Docusaurus等）: 自然な表現でリンクテキストを書く
    - 例: `[セットアップ手順](guide/setup.md)`
- テーブルは列数が多い場合や内容が長い場合は箇条書きへの変換を検討する
  - 特定列のみ長文化する場合はキー単語に留めて表下の補足文へ退避する

## 技術文書の書き方

- 手順書（セットアップ・デプロイ・移行など）:
  - 番号付きリストで手順を示す。各ステップは1つの動作に限定する
  - 前提条件・事前準備を冒頭に明記する
  - 成功時の期待結果を各ステップまたは末尾に記載する
  - 失敗時の対処やロールバック手順も含める
- トラブルシューティング: 症状 → 原因 → 対処の3要素で構成する。よくある問題から順に並べ、
  エラーメッセージは検索しやすいように正確に引用する
- API文書: エンドポイント・パラメーター・レスポンスを網羅し、リクエスト／レスポンスの具体例とエラーケースも記載する
- コミットメッセージ: 読者はエンドユーザー
  - 内部リファクタリングなどの詳細は不要
  - プロジェクト方針があればそれに従う

## ディレクトリ構成

OSSプロジェクトの典型的なドキュメント配置（プロジェクト方針がある場合はそちらに従う）。

- ルート直下: `README.md`、`LICENSE`
- `docs/`配下: 種別ごとにサブディレクトリ（`docs/guide/`・`docs/development/`・`docs/api/`等）
- ファイル粒度: 1ファイル1トピック。1ファイル300行超で分割を検討する
- ファイル命名: ケバブケース（`setup-guide.md`）。定番ファイルは大文字の慣例に従う
- サブディレクトリの入口index: 静的サイトジェネレーター（mkdocs等）のnav機能で目次を自動生成できる場合は不要。
  手動メンテのリンク集index.mdは陳腐化しやすいため設けない

## README規約

READMEはプロジェクトのトップレベルに配置し、初見の読者が最初に参照する。

- 標準構成パターン（プロジェクト方針がない場合の例。規模と性質に応じて取捨選択する）:
  1. プロジェクト名と一文の概要
  2. 主な機能・特徴（箇条書き）
  3. インストール・セットアップ手順
  4. 基本的な使い方（コード例やコマンド例）
  5. ドキュメントへのリンク
- `CLAUDE.md`との役割分担: READMEは人間の読者向け、`CLAUDE.md`はClaude Code向け。
  両者で重複する情報はREADMEを正とし、`CLAUDE.md`からはREADMEを参照するか要点のみ再掲する
