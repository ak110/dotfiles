# 頻出textlint違反パターン

- 日本語と半角文字（英数字・記号・インラインコード）が隣接する場合、間にスペースを挿入しない（JTF 3.1.1）。
  例: `Pythonの設定`、`chezmoiで管理する`、「`uv add`を使う」
- 和文中にハイフン`-`を使わない（JTF 4.2.6）。接頭辞名や識別子はインラインコード内に含める
- 半角括弧`()`を日本語本文に使わない。代わりに全角の`（）`を使う（JTF 4.3.1）
- かっこ類と隣接する文字の間のスペース（JTF 3.3）は、
  改行折り返しで全角括弧が行頭に位置した場合や、`（`と`）`が別の論理行に分かれる書き方でも違反と判定される
  - 括弧句を行末側で完結させる、文を分割する、改行位置を前文末側へずらす等で回避する
  - Markdown箇条書きの継続行（2行目以降）でも同違反として検出される
  - 行頭インデント（箇条書き継続行・サブバレットなど）の直後に`「`を置いてかぎ括弧で文を始めると、
    前文末の句点と隣接する位置でJTF 3.3違反と長文判定（sentence-length違反）を同時に誘発する。
    前文と論理接続する書き方（「〜に加え」等）か、かぎ括弧を外したパラフレーズに書き換えて回避する
- 一文は120文字以内、読点は1文あたり4個未満に抑える（ja-technical-writing/sentence-length、max-ten）
- 中黒（`・`）区切りで多項目を1文に並列列挙すると、句点までを1文として連結判定するため120字を超過しやすい。
  複数文への分割または箇条書きへの展開で解消する
- 同じ漢字の7文字連続を避ける（ja-technical-writing/max-kanji-continuous-len）
  - `進行状況可視化目的`・`実行中状態確認処理`のような語は分割や言い換えで回避する
  - 4文字＋3文字の結合でも検出される。助詞挿入で分割する例: `修正対象外領域`→`修正対象外の領域`
  - プロジェクト固有名詞（画面名・機能名など）で分割や言い換えが不自然な場合は、
    `.textlintrc.yaml`の`allow`に当該語句を登録して対応する
  - バッククォート囲みはインラインコードの意味を持つため、漢字連続回避の手段として使わない
- 同じ単語の連続使用は避ける（`ja-technical-writing/ja-no-successive-word`）
  - 同一の全角記号を連続させたプレースホルダー（`○○`・`××`・`◎◎`など）は本ルールで検出される
  - 代名詞的表現（「該当〜」「特定〜」など）か、`<対象>`形式のインラインコード（JTF 4.3.7）へ書き換える
- 対応のない山括弧`<>`を本文に書かない。プレースホルダーはインラインコード内に含める（JTF 4.3.7）
- 敬体と常体を混在させない（ja-technical-writing/no-mix-dearu-desumasu）
- 箇条書き項目を複数行に分けて書く場合（`preset-jtf-style 1.1.3.箇条書き`）:
  最終行以外の各論理行は句点で終える
  - 最終行末は`shouldUsePoint`設定に従う。`false`の場合は句点を付けない
  - 頻出違反は途中行末の句点抜けと最終行末への句点付与の両方
- 冗長表現（`ja-no-redundant-expression`の`dict5`系）はサ変動詞単独形へ置き換える
  - 「〜を行う」→「〜する」
  - 「〜を実施する」→「〜する」
  - 「〜を実行する」→「〜する」（対象が処理・機能を担う主体である場合は「〜を担う」も使う）
- `preset-jtf-style`は半角コロン+全角開きかぎ括弧（`「`）の境界スペースを自動削除する
  - 境界スペースを保持しようとして手動で挿入してもlintで削除されるため、形式統一が必要なら表現自体を書き換える
