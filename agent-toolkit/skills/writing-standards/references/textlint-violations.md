# 頻出lint違反パターン

textlintを中心としつつ、`pyfltr`経由で同時実行される他のツール（markdownlint・typos等）の違反対処も含む。

- 日本語と半角文字（英数字・記号・インラインコード）が隣接する場合、間にスペースを挿入しない（JTF 3.1.1）
  例: `Pythonの設定`、`chezmoiで管理する`、「`uv add`を使う」
  - 実装側のソースコード文字列リテラル（ログ・エラー・CLIヘルプなど）をMarkdown本文へ引用する場合、
    Markdown側のJTF 3.1.1 auto-fixで全角・半角間スペースが除去される。
    ソースコード側のスペース保持と非対称になる。
  - 利用者が`grep`等で文字列の完全一致照合をする想定がある場面では、
    引用文を地の文に置かず`text`または言語指定付きコードフェンスで包んでauto-fix対象外にする。
- 和文中にハイフン`-`を使わない（JTF 4.2.6）。接頭辞名や識別子はインラインコード内に含める
- 半角括弧`()`を日本語本文に使わない。代わりに全角の`（）`を使う（JTF 4.3.1）
- 一文は120文字以内、読点は1文あたり4個未満に抑える（ja-technical-writing/sentence-length、max-ten）
- 中黒（`・`）区切りで多項目を1文に並列列挙すると、句点までを1文として連結判定するため120字を超過しやすい
  複数文への分割または箇条書きへの展開で解消する
- 同じ漢字の7文字連続を避ける（ja-technical-writing/max-kanji-continuous-len）
  - `進行状況可視化目的`・`実行中状態確認処理`のような語は分割や言い換えで回避する
  - 4文字＋3文字の結合でも検出される。助詞挿入で分割する例: `修正対象外領域`→`修正対象外の領域`
  - プロジェクト固有名詞（画面名・機能名など）で分割や言い換えが不自然な場合は、
    `.textlintrc.yaml`の`allow`に当該語句を登録して対応する
    - 緩和採用の判断基準・検証要件は`references/lint-relax-criteria.md`を参照する
  - 計画タイトル・件名案・節タイトルで頻出する**N対象＋必須化／緩和／確定／推定／追加**型連結語は漢字連続違反になりやすい。
    助詞挿入で分割する。例: `自己検証必須化`→`自己検証の必須化`、`220行接近条件緩和`→`220行接近条件の緩和`
  - バッククォート囲みはインラインコードの意味を持つため、漢字連続回避の手段として使わない
- 対応のない山括弧`<>`を本文に書かない。プレースホルダーはインラインコード内に含める（JTF 4.3.7）
- 敬体と常体を混在させない（ja-technical-writing/no-mix-dearu-desumasu）
- 箇条書きの記述単位規定は`agent-toolkit:writing-standards`
  「Markdown記述スタイル」節に集約する。
  1項目1文・1項目1行を原則とし、1文が長い場合や複数論点を含む場合はサブバレット階層化または項目分割で表す
- 箇条書き項目末尾に句点を付けない
  （`preset-jtf-style 1.1.3.箇条書き`の`shouldUsePoint: false`規定に整合。
  当リポジトリでは`shouldUsePoint`を`false`で運用する）
- 冗長表現（`ja-no-redundant-expression`の`dict5`系）はサ変動詞単独形へ置き換える
  - 「〜を行う」→「〜する」
  - 「〜を実施する」→「〜する」
  - 「〜を実行する」→「〜する」（例外の詳細は`references/tone-examples.md`「冗長表現」節を参照する）
- Markdownのコードブロック外側fenceに4バッククォートを採用するとtextlintは外側fenceを認識しない。
  内側文面は地の文として処理されsentence-length等の違反として誤検出される
  - 外側fenceは3バッククォートを既定とする。内側に3バッククォートを含むコードブロックを書く場合のみ4バッククォートを採用する
- 列挙の分類ラベルに英字（`a`・`b`・`c`等）を使うと違反が連鎖する
  - 半角の`(a)`は`preset-jtf-style/4.3.1.丸かっこ（）`違反を招く
  - 全角の`（a）`は`ja-technical-writing/ja-unnatural-alphabet`違反を招く
  - バッククォート囲み表記は`agent-toolkit:writing-standards`「Markdown記述スタイル」の
    バッククォート用途規約（コードリテラル・コマンド・ファイルパス専用）違反を招く
  - バレット内番号ラベル方式（`- 1. xxx` `- 2. yyy`）はmarkdownlint-cli2の自動正規化で全番号が`- 1.`へ統一され識別性が失われる
  - 分類ラベルには順序付きリスト記法（`1. xxx`形式の直接記述）または意味ラベル（「仕様変更系」「追認系」等）を使う

## 頻出違反パターン予防策

計画ファイル本文の事前textlint検査で頻発する違反パターンを次のように予防する。

- バッククォート付き識別子（定数名・変数名・関数名など）を3つ以上連続して並べる場面では、
  テーブル形式や「`<定数名>などを典拠とする`」形式の参照記述で代替する
  （`sentence-length`違反予防）
- 1文は目安80字（推奨値、機械チェック上限は既存120字）を目指し、100字を超える場合は句点で分割する
  （`sentence-length`違反予防。上限120字との関係は上記のとおり）
- 漢字7文字連続違反になりやすいキーワード連結語（`修正対象外領域`・`220行接近条件緩和`など）は
  本ファイル「漢字連続」項の分解例に従い読点・助詞・送り仮名で分解する
- 口語的な動詞は同ファイル「冗長表現」項のJIS規格スタイル相当へ書き換える
- フィードバック原文・レビュー指摘・エラー出力の引用フレーズを規範文へ埋め込む前に、
  `ja-no-weak-phrase`と`max-kanji-continuous-len`に該当する語を機械的に書き換える
  - 書き換え例: 弱い自信を示す語尾相当を観測事象化した表現へ、
    「〜と思われる」を「〜と観測される」へ書き換える
  - 漢字が7つ以上連続する語は本ファイル「漢字連続」項の分解例に従い読点・助詞・送り仮名で分解する
  - 原文の主旨を保った表現へ調整したうえで転記する
- `## 変更内容`配下の差分ラベル直後の`text`フェンス内本文は、
  ラベル種別（`[追記]`・`[置換後]`・`[置換後（全文）]`・`[新設]`等、`[削除根拠]`・`[現行]`を除く）を問わず
  textlint検査対象となる。ブロック内容案の作成時に本ファイル冒頭の頻出違反パターンに
  該当しない形へ事前整形する
- 複数の`#### 縮減対象`H4見出しはmarkdownlint MD024違反を招くため、
  `#### 縮減対象（<ファイル名>）`形式でユニーク化する（対象ファイル名を末尾へ付与）
- 半角スペースを含む節名を`「」`で囲んで参照するとJTF 3.1.1 auto-fixが日本語と半角文字の隣接スペースを削除し、
  計画チェック側の節名不在検査で誤検出される。節名参照はバッククォート囲み（例: `` `text コードブロック化義務` ``）で書きauto-fix対象外にする

## typos擬陽性回避

typos linterは`text`コードブロック内も検査対象とする。固有名や末尾が切り詰められたファイル名で擬陽性が発生しやすい。
除外語登録・プレースホルダー化のいずれも緩和判断に該当する。
採用可否は`references/lint-relax-criteria.md`の判断フレーム・検証要件に従う。

- 対象語句が固有名・意味のある語（プロジェクト名・OSS名等）の場合は除外語登録を選ぶ
  - 登録先は`pyproject.toml`の`[tool.typos.default.extend-words]`
  - `.typos.toml`採用環境では当該ファイルの`[default.extend-words]`を使う
- 対象語句が廃棄前提の識別子（自動生成名・テスト用ダミー値等）の場合はプレースホルダー化を選ぶ
  - `XXX`等への置換と、原文の趣旨を示す注釈の併記とで対処する
