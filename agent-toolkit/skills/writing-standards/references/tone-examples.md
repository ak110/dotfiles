# 危険語彙・禁止パターンの対比集

本ファイルはコンテキスト汚染リスクの高い悪い例の隔離専用ファイル。
口語の検出語・主観的修飾・比喩動詞・冗長表現など、生成確率を上げる危険語彙・禁止パターンを収録する。
メインエージェントが対比を確認したいときに条件付きで参照する。

## 運用注記

- メインエージェントから直接Readしない。コンテキスト汚染を避けるため、
  確認はExploreサブエージェント経由で行い、修正が必要な場合は`plan-implementer`経由で実施する
- `plan-impl-reviewer`はReadツールで直接読み込む（日本語表現観点で参照する）
  読み込んだ悪い例の文面は指摘テキストへ転記しない
- 口語表現チェックの検査対象から除外される（`pyproject.toml`の`[tool.pyfltr] extend-exclude`で対象パスを列挙する運用）
  そのため悪い例には辞書検出語が含まれてよい
- 悪い例の文面はwriting-standardsの規定違反のままで構わない。観点を示すために必要なため意図的に違反させている
  書き換え案・解説文・観点説明文はwriting-standardsに準拠する

## 修飾過多

主観的・感情的な修飾語を名詞句に付けると、観測事実から離れて宣伝的な響きになる。
フラットに対象名詞のみで表記する。

```text
悪い例: 強力なバリデーション機能により、堅牢なコード品質を実現します。
書き換え: バリデーション機能を提供する。設定したルールに違反するコードを検出する。
解説: 「強力な」「堅牢な」は主観的修飾語のため削除し、何を提供するかを観測事実で記述する。
```

```text
悪い例: シンプルで使いやすいAPIを備えた、優れたライブラリです。
書き換え: APIは関数3つで構成される。`load`・`save`・`validate`の各関数を提供する。
解説: 「シンプル」「使いやすい」「優れた」は主観評価のため削除し、APIの構造を具体名で示す。
```

## 不要なアピール

営業的・宣伝的な記述は技術ドキュメントの読者の判断材料にならない。
「何ができるか」「どう動くか」を観測事実として書く。

```text
悪い例: 推奨は uvx での実行。事前のインストールやdev依存への追加は不要で、常に最新のpyfltrを利用できる。
書き換え: `uvx pyfltr`で実行する。事前インストールやdev依存への追加なしで最新版を取得して実行する。
解説: 「推奨は」「常に最新の…を利用できる」は宣伝的表現のため削除し、コマンドと挙動のみを記述する。
```

```text
悪い例: エージェント向け出力（run-for-agent）も用意しており、CIとの連携も容易です。
書き換え: エージェント向け出力モードとして`run-for-agent`サブコマンドを提供する。CIから呼び出すことができる。
解説: 「容易です」は主観評価のため削除し、サブコマンド名と利用可能性を事実として記述する。
```

## 擬人化副詞

警告や標準出力なしで処理が進む動作を「静かに」「黙って」のような副詞で擬人化すると、
動作内容が観測事実から離れる。何が起き何が起きないかを観測動詞や明示的な振る舞い記述に置き換える。

```text
悪い例: 設定が見つからない場合は静かにスキップされる。
書き換え: 設定が見つからない場合は処理を継続する。警告ログは出力しない。
解説: 「静かに」は処理の不可視性を擬人化する副詞のため、何が起きるか・何が起きないかを観測事実で記述する。
```

```text
悪い例: ロックが取れないときは黙ってリトライする。
書き換え: ロックが取れないときはログ出力なしでリトライする。最大3回まで再試行する。
解説: 「黙って」は擬人化副詞のため、観測可能な振る舞い（ログ出力なし・リトライ回数）に置き換える。
```

## 五段動詞の縮約形・ら抜き言葉

可能を表す表現には次の2系統の崩れがある。書き言葉では`〜することができる`形式または別の能動表現に置き換える。

- 五段動詞の縮約形: 五段動詞の能動形を1語に縮約した可能形で文を結ぶ書き方
  口語では定着しているが、書き言葉では能動形＋`〜することができる`に展開する方が安定する
- ら抜き言葉: 一段動詞・カ変動詞の可能形から本来必要な「ら」を脱落させる書き方
  文法的に誤りとされるため、正しい可能形（「〜られる」）または別の能動表現に置き換える

```text
悪い例: 本ツールでスクリプトが書ける。設定値を使える状態にする。
書き換え: 本ツールでスクリプトを書くことができる。設定値を利用可能な状態にする。
解説: 「書ける」「使える」は五段動詞の縮約形のため、「書くことができる」「利用可能な〜」など能動表現に置き換える。
```

```text
悪い例: 出力ログを見れる。設定の挙動が見れる。
書き換え: 出力ログを見られる。設定の挙動を確認できる。
解説: 「見れる」はら抜き言葉のため、本来の可能形「見られる」または別の動詞「確認できる」に置き換える。
```

## 比喩動詞

物理動作や行為の比喩を技術文書に持ち込むと、観測可能な動作から離れて読者が挙動を確認できない。
標準的な技術用語の動詞（「分類する」「振り分ける」「同等に扱う」「適用する」など）で書く。

```text
悪い例: `is_symlink()`のOSError例外をsymlink扱いへ寄せて返す。
書き換え: `is_symlink()`のOSError例外をsymlink扱いに分類して返す。
解説: 「寄せる」は物理動作の比喩のため、観測動詞「分類する」で置き換える。
```

```text
悪い例: 設定値を既定値の挙動へ寄せる。
書き換え: 設定値を既定値と同等に扱う。
解説: 「寄せる」は擬人的な誘導表現のため、観測可能な動作「同等に扱う」で置き換える。
```

```text
悪い例: 不正な入力を弾く。許可されない操作を蹴る。
書き換え: 不正な入力を拒否する。許可されない操作を拒否する。
解説: 「弾く」「蹴る」は物理動作の比喩のため、観測動詞「拒否する」で置き換える。
```

```text
悪い例: 子プロセスを叩いて結果を受け取る。サブコマンドを叩く。
書き換え: 子プロセスを呼び出して結果を受け取る。サブコマンドを呼び出す。
解説: 「叩く」は打撃動作の比喩で外部コマンド実行の口語表現のため、観測動詞「呼び出す」で置き換える。
```

```text
悪い例: 入力値がスキーマに刺さるか確認する。型定義に噛み合うか検査する。
書き換え: 入力値がスキーマに合致するか確認する。型定義に整合するか検査する。
解説: 「刺さる」「噛み合う」は物理動作の比喩のため、観測動詞「合致する」「整合する」で置き換える。
```

```text
悪い例: 旧APIを潰す。互換シムを殺す。
書き換え: 旧APIを無効化する。互換シムを削除する。
解説: 「潰す」「殺す」は破壊・暴力の比喩のため、観測動詞「無効化する」「削除する」で置き換える。
```

```text
悪い例: 既存スタイルを引きずって書き換える。先行プロジェクトの方針を被せる。
書き換え: 既存スタイルを踏襲して書き換える。先行プロジェクトの方針を採用する。
解説: 「引きずる」「被せる」は物理動作の比喩のため、観測動詞「踏襲する」「採用する」で置き換える。
```

```text
悪い例: 新ルールを差し込む。設定を流し込む。
書き換え: 新ルールを導入する。設定を適用する。
解説: 「差し込む」「流し込む」は物理動作の比喩のため、観測動詞「導入する」「適用する」で置き換える。
```

```text
悪い例: 修正をmainブランチに乗せる。指摘内容を本文に取り込む。
書き換え: 修正をmainブランチへ反映する。指摘内容を本文へ反映する。
解説: 「乗せる」「取り込む」は物理動作の比喩のため、観測動詞「反映する」で置き換える。
```

## 冗長表現

動作名詞に「を行う」「を実施する」を後接させると、動詞化を二段重ねた冗長表現になる。
動作名詞をそのまま動詞化（サ変動詞化）するか、構造を組み替えるか、観測動詞へ置き換える。
`ja-no-redundant-expression`のdict5系が判定対象とする頻出パターン。

```text
悪い例: サブエージェントがレビューを実施する。設定の読み込みを行う。
書き換え: サブエージェントがレビューする。設定を読み込む。
解説: 「レビューを実施する」「読み込みを行う」は動作名詞と「行う／実施する」を
二段で重ねた冗長表現のため、サ変動詞化または元の動詞へ戻す。
```

```text
悪い例: 整合チェックを行う。事前確認を実施する。
書き換え: 整合をチェックする。事前に確認する。
解説: 動作名詞が複合語の場合はサ変動詞化が不自然になることがある。
分解可能な場合は「整合をチェックする」「事前に確認する」のように構造を組み替える。
```

```text
悪い例: ファイル出力のみを実施する。GitLab CIの評価までを行う。
書き換え: ファイル出力のみを実行する。GitLab CIの評価まで実行する。
解説: サ変動詞化と分解のいずれも不自然な場合は「実行する」など標準的な汎用動詞に置き換える。
```

動作名詞「実施」が単体動詞として使われる場合（例「全項目を実施する」のように
「を実施する」の直前に動作名詞が無いケース）は冗長判定外。

## LLM口調

論点を増やさず装飾として用いる型を排除する。
術語・概念を議論や説明に使うこと自体は対象外とする。

### 予告と総括

主張内容を述べずに姿勢・構成を宣言する前置きや言い換えだけの結びは、
主張そのものを直接書くことで代替できる。

```text
悪い例: 重要なのは、テストを設計段階から考慮することである。まとめると、境界値を外さないことが大切だ。
書き換え: テスト設計は実装前の仕様策定段階から開始する。テストケースは境界値を網羅する。
解説: 「重要なのは〜である」は姿勢の宣言であり主張を含まないため削除する。
「まとめると」が直前の言い換えとして機能する場合はその文ごと削除する。
```

### 正面から系

中身の代わりに対象への取り組み姿勢を宣言する型は、
対象範囲を具体化した記述に置き換える。

```text
悪い例: エラー処理を正面から扱う。例外設計の問題を正面から回収する。
書き換え: 呼び出し側の想定外入力を起点とする例外クラスの設計方針を示す。
解説: 「正面から扱う」「正面から回収する」は扱う内容を示さない姿勢宣言のため、
対象範囲と述べる内容を具体化した文に置き換える。
```

### 空虚な形容

中身を述べずに強調や概観だけを示す形容詞類は、
観点と帰結を具体化した記述に置き換える。

```text
悪い例: 不可欠な性能改善として、キャッシュ戦略が核心的な役割を担う。
書き換え: キャッシュを導入するとDBクエリ回数が減少し、レスポンスタイムが短縮される。
解説: 「不可欠な」「核心的な」は改善の内容や帰結を述べない形容のため、観測可能な指標と因果関係で記述する。
```

```text
悪い例: 本UIは直感的に操作できる。
書き換え: 本UIはメニューを2階層に抑え、主要操作を1クリックで完了する構造とする。
解説: 「直感的」は読者ごとに解釈が異なる形容のため、観測可能な構造（階層数・操作ステップ数）で記述する。
```

### 空虚な動詞

何をどう記述したかを示さず終わる動詞類は、
述べる対象と内容を具体化した動詞句に置き換える。

```text
悪い例: APIのレスポンス仕様について深掘りする。認証エラーの挙動に触れる。
書き換え: APIのレスポンスボディの構造とステータスコードの対応を示す。
解説: 「深掘りする」「触れる」は何を述べるかを示さないため、具体的な対象を明示した動詞句に置き換える。
```

### 接続の型

新情報を持たない接続表現の連打は、複数の事項を1文に集約することで代替できる。

```text
悪い例: 設定において、タイムアウト値を指定する。また、リトライ回数を指定する。さらに、ログレベルを設定するという側面から挙動を変更できる。
書き換え: タイムアウト値・リトライ回数・ログレベルの3項目を設定ファイルで指定する。
解説: 「において」「また」「さらに」「という側面から」で分割した各文が独立情報を持たない場合は、1文へ集約して接続表現を削除する。
```

### 弱い緩和と称賛

根拠のない述部で主張を弱めたり、観測値を持たない強調語で記述を膨らませたりしない。
推量・仮定・読者の疑念を示す場面では不確実性を示す述部を保持する。

```text
悪い例: バリデーションを追加すると、非常に安全になると言えるだろう。
書き換え: 入力バリデーションを追加すると、不正値による処理中断を防止できる。
解説: 「非常に」は観測値を持たない強調語のため削除する。「〜と言えるだろう」は根拠のある判断を弱める述部のため「〜できる」に置き換える。
ただし推量・仮定・読者の疑念を表す場面では「かもしれない」「だろう」を保持して不確実性を示す。
```
