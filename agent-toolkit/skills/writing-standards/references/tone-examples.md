# 自然な日本語表現の対比集

本ファイルは観点別に悪い例と書き換え案を対比形式で収録する補助資料。
メインエージェントが対比を確認したいときに条件付きで参照する。

## 運用注記

- メインエージェントから直接Readしない。コンテキスト汚染を避けるため、
  確認はExploreサブエージェント経由で行い、修正が必要な場合は`plan-implementer`経由で実施する
- `plan-impl-reviewer`は本ファイルを必須で読み込む（日本語表現観点で参照する）。
  読み込んだ悪い例の文面は指摘テキストへ転記しない
- `check_colloquial.py`の検査対象から除外される（スクリプト側でハードコード除外する運用）。
  そのため悪い例には辞書検出語が含まれてよい
- 悪い例の文面はwriting-standardsの規定違反のままで構わない。観点を示すために必要なため意図的に違反させている。
  書き換え案・解説文・観点説明文はwriting-standardsに準拠する

## 修飾過多

主観的・感情的な修飾語を名詞句に付けると、観測事実から離れて宣伝的な響きになる。
フラットに対象名詞のみで表記する。

```text
悪い例: 強力なバリデーション機能により、堅牢なコード品質を実現します。
書き換え: バリデーション機能を提供する。設定したルールに違反するコードを検出する。
解説: 「強力な」「堅牢な」は主観的修飾語のため削除し、何を提供するかを観測事実で記述する。
```

```text
悪い例: シンプルで使いやすいAPIを備えた、優れたライブラリです。
書き換え: APIは関数3つで構成される。`load`・`save`・`validate`の各関数を提供する。
解説: 「シンプル」「使いやすい」「優れた」は主観評価のため削除し、APIの構造を具体名で示す。
```

## 不要なアピール

営業的・宣伝的な記述は技術ドキュメントの読者の判断材料にならない。
「何ができるか」「どう動くか」を観測事実として書く。

```text
悪い例: 推奨は uvx での実行。事前のインストールやdev依存への追加は不要で、常に最新のpyfltrを利用できる。
書き換え: `uvx pyfltr`で実行する。事前インストールやdev依存への追加なしで最新版を取得して実行する。
解説: 「推奨は」「常に最新の…を利用できる」は宣伝的表現のため削除し、コマンドと挙動のみを記述する。
```

```text
悪い例: エージェント向け出力（run-for-agent）も用意しており、CIとの連携も容易です。
書き換え: エージェント向け出力モードとして`run-for-agent`サブコマンドを提供する。CIから呼び出すことができる。
解説: 「容易です」は主観評価のため削除し、サブコマンド名と利用可能性を事実として記述する。
```

## 粒度ミスマッチ

冒頭でいきなり細部を提示すると、読者は全体像を把握できないまま個別仕様を読まされることになる。
全体像から徐々に詳細へ掘り下げる。

```text
悪い例: 最小設定から段階的に導入する手順は…を参照する。
書き換え: 本ツールは静的解析・テスト・フォーマットを統合実行する。
  導入手順は段階的な設定例とともに後述する（…節を参照）。
解説: 細部の参照リンクを冒頭に置かず、まず本ツールが何を行うかを示す。詳細は後続節へ移す。
```

```text
悪い例: 実行するツールはpyproject.tomlの`[tool.pyfltr]`セクションで指定する。
書き換え: 実行対象のツールは設定ファイルで指定する。
  具体的には`pyproject.toml`の`[tool.pyfltr]`セクションで列挙する（詳細は後述）。
解説: 設定キーをいきなり提示せず、まず「設定ファイルで指定する」という抽象レベルで提示し、具体名を補足として続ける。
```

## 不正確な要約

観測事実から外れた要約は読者を誤誘導する。実装の挙動と一致する範囲で要約する。

```text
悪い例: pyfltrは全てのPythonプロジェクトで動作し、あらゆる静的解析を統合する。
書き換え: pyfltrは設定ファイルで指定したツールを順に実行する。対応ツールはruff・mypy・pytest等。
解説: 「全ての」「あらゆる」は事実を超えた誇張のため削除し、対応範囲を具体的に示す。
```

```text
悪い例: 本コマンドは設定ファイルを読み込み、最適な実行順を自動で決定する。
書き換え: 本コマンドは設定ファイルの`commands`配列の順序でツールを実行する。
解説: 「最適な実行順を自動で決定する」は実装に存在しない動作のため、実際の挙動（配列順実行）に置き換える。
```

## 行動指示述部

エンドユーザー向けドキュメント・開発者向けドキュメントで行動指示に常体終止を使うと不自然になる。
行動指示は体言止めまたは敬体で結ぶ。
コーディングエージェント向け文書（`CLAUDE.md`・`SKILL.md`・サブエージェント定義など）は
エージェントへの直接指示として書くため、常体の指示終止形を許容する。
本節の対比はエンドユーザー向けドキュメント・開発者向けドキュメントを対象とする。

```text
悪い例: 詳細はドキュメントを参照する。
書き換え: 詳細はドキュメントを参照（敬体採用ドキュメントでは「ドキュメントを参照してください」）。
解説: 「参照する」は説明文の語尾だが、行動指示の文脈では体言止めまたは敬体に切り替える。
```

```text
悪い例: 設定ファイルにキーを追加する。再起動後に反映される。
書き換え: 設定ファイルにキーを追加。再起動後に反映される。
解説: 1文目は行動指示のため体言止め。2文目は説明文のため常体終止を維持する。
```

## 抽象動詞

観測できない抽象的・宣伝的な動詞は、読者が動作を確認できない。
標準的な技術用語の動詞で書く。

```text
悪い例: 本ツールはコード品質を向上させ、開発体験を改善する。
書き換え: 本ツールはlintとフォーマットを実行する。違反箇所を出力する。
解説: 「向上させる」「改善する」は観測不能な抽象動詞のため、実行する処理と出力結果に置き換える。
```

```text
悪い例: 設定を最適化して、パフォーマンスを引き出す。
書き換え: 設定値を調整する。`workers=4`の指定で並列実行する。
解説: 「最適化する」「引き出す」は観測不能な動詞のため、何を変更するか・どう動くかを具体的に書く。
```

## 擬人化副詞

警告や標準出力なしで処理が進む動作を「静かに」「黙って」のような副詞で擬人化すると、
動作内容が観測事実から離れる。何が起き何が起きないかを観測動詞や明示的な振る舞い記述に置き換える。

```text
悪い例: 設定が見つからない場合は静かにスキップされる。
書き換え: 設定が見つからない場合は処理を継続する。警告ログは出力しない。
解説: 「静かに」は処理の不可視性を擬人化する副詞のため、何が起きるか・何が起きないかを観測事実で記述する。
```

```text
悪い例: ロックが取れないときは黙ってリトライする。
書き換え: ロックが取れないときはログ出力なしでリトライする。最大3回まで再試行する。
解説: 「黙って」は擬人化副詞のため、観測可能な振る舞い（ログ出力なし・リトライ回数）に置き換える。
```

## メタ記述の冒頭混入

参照ファイル冒頭に「〇〇をまとめる。」「〇〇についてまとめる。」のように
ファイル全体を要約する書き手目線の記述を置くと、エージェントへの指示としては冗長で、
本文の内容自体から把握すべき情報を上書きしてしまう。
冒頭は実行・判断に必要な情報を直接記述する。

```text
悪い例: GitLab CIキーワード仕様・典型パターン・lintのリファレンスをまとめる。
書き換え: GitLab CIキーワード仕様・典型パターン・lintのリファレンス。
解説: 「〜をまとめる」は書き手目線の管理方針の言い回しで、エージェントへの指示として機能しない。
  本文内容を端的に示す名詞句に置き換える。
```

```text
悪い例: 計画ファイル作成時に参照する記述例と、構成・注意点を収録する。
書き換え: 計画ファイル作成時の参照用。記述例・構成・注意点を含む。
解説: 「収録する」は管理方針の記述で、内容自体の説明にならない。
  「参照用」と用途を直接示し、含む要素を端的に列挙する。
```

## 五段動詞の縮約形・ら抜き言葉

可能を表す表現には次の2系統の崩れがある。書き言葉では`〜することができる`形式または別の能動表現に置き換える。

- 五段動詞の縮約形: 五段動詞の能動形を1語に縮約した可能形で文を結ぶ書き方。
  口語では定着しているが、書き言葉では能動形＋`〜することができる`に展開する方が安定する
- ら抜き言葉: 一段動詞・カ変動詞の可能形から本来必要な「ら」を脱落させる書き方。
  文法的に誤りとされるため、正しい可能形（「〜られる」）または別の能動表現に置き換える

```text
悪い例: 本ツールでスクリプトが書ける。設定値を使える状態にする。
書き換え: 本ツールでスクリプトを書くことができる。設定値を利用可能な状態にする。
解説: 「書ける」「使える」は五段動詞の縮約形のため、「書くことができる」「利用可能な〜」など能動表現に置き換える。
```

```text
悪い例: 出力ログを見れる。設定の挙動が見れる。
書き換え: 出力ログを見られる。設定の挙動を確認できる。
解説: 「見れる」はら抜き言葉のため、本来の可能形「見られる」または別の動詞「確認できる」に置き換える。
```

## 比喩動詞

物理動作や行為の比喩を技術文書に持ち込むと、観測可能な動作から離れて読者が挙動を確認できない。
標準的な技術用語の動詞（「分類する」「振り分ける」「同等に扱う」「適用する」など）で書く。

```text
悪い例: `is_symlink()`のOSError例外をsymlink扱いへ寄せて返す。
書き換え: `is_symlink()`のOSError例外をsymlink扱いに分類して返す。
解説: 「寄せる」は物理動作の比喩のため、観測動詞「分類する」で置き換える。
```

```text
悪い例: 設定値を既定値の挙動へ寄せる。
書き換え: 設定値を既定値と同等に扱う。
解説: 「寄せる」は擬人的な誘導表現のため、観測可能な動作「同等に扱う」で置き換える。
```
