# 現行版ドキュメント構成サンプル集

初版構成を決める際に参考にするサンプル。
各テンプレートの書式詳細は`agent-toolkit:spec-driven`の`references/templates.md`に従う。

## 共通方針

- 配置先・命名規則は確定後に`CLAUDE.md`へ記録する
- ファイル種別は「機能ドキュメント」「横断ドキュメント」の2種を基本とし、
  必要に応じて画面・APIなどの索引ドキュメントを併設する
- 機能数や公開面の有無に応じて、以下のサンプルから近いものを起点にカスタマイズする
- 命名はケバブケース（`feature-name.md`）を基本とする。
  プロジェクトに既存の命名慣習があればそれを優先する

## サンプル1: 最小構成

小規模ライブラリ・CLIツール向け。機能単位の分割が過剰となる規模で採用する。

配置ツリー:

```text
docs/
├── overview.md       # プロジェクト全体の背景・スコープ・受け入れ基準
├── design.md         # 主要設計判断と却下した代替案
├── usage.md          # 利用者向けの基本的な使い方（任意）
└── glossary.md       # 用語集（任意）
```

適用条件:

- 機能が1〜2個に閉じている
- 利用者から確認できる振る舞いを1ファイルで記述しきれる
- `docs/features/`のような分割を導入すると索引性より冗長性が勝る

運用ヒント:

- 機能が増えて`overview.md`が肥大化したら「サンプル2: トピック追加構成」へ切り替える
- `design.md`と`overview.md`の境界が曖昧になりがちな場合は、
 「外部から確認できる仕様（overview）」「設計判断と理由（design）」で割る

## サンプル2: トピック追加構成

複数機能を持つシステム向け。`spec-driven`既定の`docs/features/`・`docs/topics/`構成。

配置ツリー:

```text
docs/
├── features/
│   ├── _template.md        # 機能ドキュメントのテンプレート（任意）
│   └── {機能名}.md         # 機能ごと（例: sso.md・billing.md）
└── topics/
    ├── _template.md        # 横断ドキュメントのテンプレート（任意）
    └── {トピック名}.md     # 横断ごと（例: error-response.md・retry-policy.md）
```

適用条件:

- 機能が3個以上ある
- 複数機能で共有される共通仕様（エラー応答・認証・データ型など）が存在する
- 機能追加・改修が継続的に発生する

運用ヒント:

- `_template.md`を置いておくと新規ファイル作成時の出発点になる（書式詳細は`spec-driven`の`templates.md`に従う）
- 横断/機能の判定に迷う仕様は、まず機能ドキュメント側に書き、複数機能で参照され始めた時点で
  横断ドキュメントへ抽出する
- `docs/v{next}/`は`spec-driven`スキルが管理するため、本構成では設けない

## サンプル3: Webシステム構成

画面・APIなど明示的な公開面を持つシステム向け。索引ドキュメントを併設して網羅性を担保する。

配置ツリー:

```text
docs/
├── features/
│   └── {機能名}.md         # 機能ごと
├── topics/
│   └── {トピック名}.md     # 横断ごと
├── screens.md              # 画面一覧（画面名・用途・主要機能ドキュメントへのリンク）
└── api.md                  # API一覧（エンドポイント・用途・主要機能ドキュメントへのリンク）
```

適用条件:

- ユーザーインターフェース（画面・公開API）が明示的な公開面として存在する
- 利用者が「どの画面でどの機能を使うか」「どのAPIでどの機能を呼ぶか」を索引で辿りたい
- 画面・APIごとの仕様詳細は機能ドキュメント側に集約したい（`screens.md`・`api.md`は索引に徹する）

運用ヒント:

- `screens.md`・`api.md`は索引に徹し、機能仕様は機能ドキュメントへリンクで委ねる
 （仕様の二重管理を避けるため）
- 画面・APIに対する変更は、まず該当機能ドキュメントを更新し、必要に応じて索引行を追記する
- 画面・APIの粒度が大きい場合は`docs/screens/`・`docs/api/`配下に分割するなど、
  本構成を起点にカスタマイズする
