# 完了報告の検収手順

`agent-toolkit:spec-driven-impl`「4. 計画ファイル処理の委譲」配下「完了報告の検収」節から参照される詳細手順。

## 正常完了報告の検収

正常な完了報告を受け取った場合は次の計画ファイルへ進む。

- 全モード共通: 完了報告のOVERVIEW.md更新項目と`git show --name-only --format= HEAD`の実差分の双方で
  更新を照合し、未更新時は別途追加コミットで更新する
- 段1（並列実装専従モード）: 各サブエージェント報告から
  OVERVIEW.md更新要否・編集ファイルパス・段2への引き継ぎ事項を集約し、
  段2サブエージェントの起動プロンプトに渡せる形で保持する
- 段2（検証・コミット専従モード）: 上記共通照合に加え、サブエージェント報告の「検証結果」「作成コミットのSHA」を確認する
- 通常モード・段2では`review_handoff`欄を検収する。
  「レビューは実施しない」以外は記載のスキル・エージェントを
  `agent-toolkit:plan-impl`配下`references/fork-reception.md`の手順で起動する

## needs_escalation・未解決事象を伴う完了報告

- `needs_escalation`報告: `4.5 エスカレーション受領時の調停`節の手順で処理する
- 未解決事象を伴う完了報告は、振り分け前に当該事象の技術的な解消可能性を評価する
  （メイン側または別`spec-driven-implementer`起動での対処可否を判定）。
  `references/qblock-templates.md`「ユーザー作業必須雛形」適用範囲外の事象はTBD化せず、
  対処コミットおよび`needs_escalation`のいずれかで処理する
  - メイン側で技術的に解消可能な事象は、対処専用の新規`spec-driven-implementer`を
    別途起動して対処コミットまで完遂する。原テーマの`spec-driven-implementer`を再起動して続行する。
    原テーマ計画ファイル範囲外への変更（既存バグの修正・前提環境の整備など）が必要な場合も同経路で扱う
  - 上記経路でも解消しない事象は`4.5 エスカレーション受領時の調停`節で扱う
  - ユーザー判断・ユーザー操作を要する事象
    （`references/qblock-templates.md`「ユーザー作業必須雛形」適用範囲規定）はTBD.mdへQブロックとして記録し、
    当該テーマの実装はそのまま続行する
  - 本スキル全体の実行前提が崩れる事象（「6. 停止判定」の停止契機に該当）のみ、
    停止理由を本スキル最終メッセージへ記録して終了する

## 解決済み情報の削除

解決済み事象（解決済みQブロック・確定済み暫定判断・解決済み発見事項・解決済み既知制約）は
解決確定時点で削除して最新化する。
削除対象は計画ファイル本文・TBD.md・テーマ仕様・OVERVIEW.md。
残置範囲は`references/escalation.md`末尾の規定に従う。

## TBD.mdの回答済み質問の反映

TBD.mdの回答済み質問は回答内容の性質で反映タイミングを分岐する（優先順位: 1→2→3）。

- 1（仕様・方針見直し系）: 計画ファイル本文を最終確定形へ更新し、本セッション内で実装サブエージェントを再起動して反映する
- 2（追加実装系）: 既存方針内に収まる局所的な追加実装は計画ファイル末尾「追加実装タスク」へ追記し、
  本セッション内で実装サブエージェントを再起動して反映する（次セッション以降への先送りは行わない）
- 3（暫定追認系）: 暫定判断の追認のみで追加要求を含まない回答は、
  計画ファイル進捗ログへ追記しTBD.mdの該当ブロックを削除して完了とする

## 未解決状態を残す文面の禁止

計画ファイル本文・実装コード・コミットメッセージ・OVERVIEW.md・最終メッセージ・完了報告サマリーに
未解決状態を文面として残さない（「制限がありスキップ」「未対応」「次サイクル送り」「未着手」等）。

- 「6. 停止判定」の停止契機による正規の停止理由記載は本項目の対象外
- 技術的に解消可能な事象は同セッション内で完遂し、残存する事象は前項各分岐
  （完了報告返却・TBD.mdのQブロック化・「6. 停止判定」による最終メッセージ記録）へ収容する
