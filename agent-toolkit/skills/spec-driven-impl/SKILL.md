# 自律実行による計画ファイル巡回実装

## 1. 位置付けと前提

入力前提:

- `docs/v{next}/plans/*.md`（計画ファイル群）が1件以上存在する
- 上記の配置は`agent-toolkit:spec-driven`配下`references/spec-driven-framework.md`の既定または
  プロジェクト指定に従う

起動方法:

- 引数なしで起動する（`/agent-toolkit:spec-driven-impl`または`agent-toolkit:spec-driven-plan`からの誘導）
- `docs/v{next}/plans/*.md`を全件巡回する
- 起動時に入力前提を満たさない場合の停止規定は「6. 停止判定」に従う

### 起動前提チェック

本スキルはauto mode下での自律実行を前提とする。
起動直後に次のチェックを実施し、未活性の場合は本スキルを即時終了する。

- メインエージェントへ注入されるシステムプロンプトに`Auto Mode Active`が含まれるかを確認する
- 含まれない場合、ユーザー宛に次の旨を1行で通知し本スキルを即時終了する
  - 本スキルはauto mode下での自律実行を前提とする
  - 再起動はauto modeを有効化してから行う
- 当該チェックは「6. 停止判定」の前提崩れ停止と別経路として扱う

## 2. ユーザー確認規範のオーバーライド

本スキル全体はauto mode下での実行を前提とする（「1. 位置付けと前提」配下「起動前提チェック」サブ節を典拠とする）。
本スキル実行中は次の確認契機をTBD.md追記＋暫定判断＋続行へ置換する。

- `agent.md`「品質最優先」節および「ユーザーとともに考える」節の確認要求全般
- `claude-code.md`「auto mode下での挙動」節「迷ったら作業を止めて確認」
- `agent-toolkit:plan-mode`の要件対話・認識合わせ・恒久化検討・リファクタリング検討の各ユーザー確認
- `agent-toolkit:plan-impl`「2.5サブエージェント完了報告の検収」のうちユーザー確認を要する分岐

サブエージェント（`spec-driven-implementer`および配下の`plan-implementer`等）は`agent.md`「縮退表明は発行しない」項の規定に従う。
計画ファイル記載の全変更を実装・検証・コミットまで完遂する。
計画ファイルに記載された破壊的操作・外部送信（`git push`・外部APIへの送信・データ削除等）も通常工程として実行し、停止理由としない。
確認事項はTBD.mdへQブロックとして記録し続行根拠とする（停止判断の根拠としない）。
本スキル全体の実行前提が崩れる場合の停止規定は「6. 停止判定」に従う。

## 3. 計画ファイル巡回ループ

1. `docs/v{next}/plans/*.md`の一覧を取得する
   - `docs/v{next}/OVERVIEW.md`に「テーマ間依存関係」節（`依存元 → 依存先`記法）があれば、依存先が依存元より先に並ぶよう並べ替えて処理対象とする
   - 節がなければファイル名順とする
2. `docs/v{next}/TBD.md`を読み込み、回答済み質問の反映方針を確定する
   - 未作成ならテンプレで新規作成する
   - 別テーマで記載された未回答事項のうちメイン側で対処可能なものは、当該テーマ着手前に対処する
   - 「ユーザー指摘」欄を走査し、当該テーマに関する指摘があれば反映方針を確定する
3. 計画ファイル1件ごとに「4. 計画ファイル処理の委譲」を実施する（1テーマ1コミット）
4. 未処理テーマがあれば継続可否をユーザーへ確認せず全件を自動継続する
5. 全計画ファイルを処理し終えたら、TBD.mdの未回答件数と冒頭リストを最終メッセージで提示して終了する

## 4. 計画ファイル処理の委譲

メイン側のコンテキスト消費を抑えるため、計画ファイル1件あたり1サブエージェント（`spec-driven-implementer`）を起動する。
当該サブエージェント内で`agent-toolkit:plan-impl`の工程（タスク分解・実装・検証・コミット・レビュー）を完遂する。

メイン側はオーケストレーションに専念し、次の作業のみを担う。

- 計画ファイル一覧の取得と依存解決による並べ替え
- TBD.md読み込みと`## ユーザー指摘`走査
- `spec-driven-implementer`の起動と完了報告検収
- 並列起動の可否判定（編集対象ファイル衝突判定など）
- `needs_escalation`受領時の調停（`4.5 エスカレーション受領時の調停`節）
- 最終メッセージの提示
- サブエージェント完了通知後の`git status --short`・`git diff --stat`による作業ツリー実態照合（`agent.md`「サブエージェントの活用」節の事務的照合）

メイン側で次の作業は行わない（全て`spec-driven-implementer`へ委譲する）。

- 計画ファイル・テーマ仕様・OVERVIEW.mdの内容編集
- 検証コマンドの実行
- `git commit`の実行

### サブエージェント起動時のモデル指定

メイン側モデルに応じて`spec-driven-implementer`起動時のモデル指定を切り替える。

- 本スキル起動時、メイン側自身が動作するモデル名をシステムプロンプトの`You are powered by the model named ...`から読み取る
- モデル系統がopus未満（Sonnet・Haiku等）の場合、`spec-driven-implementer`起動時のパラメーターで`model: opus`を明示指定する
- モデル系統がopus以上の場合は親モデル継承のままとする（明示指定不要）
- 段1（並列実装専従モード）・段2（検証・コミット専従モード）・直列モード（通常モード）のいずれでも同じ判定を適用する

### テーマ間並列実装の判定と起動

次の条件をすべて満たす複数テーマは`spec-driven-implementer`を2段委譲構成で並列起動して効率化する。

- `docs/v{next}/OVERVIEW.md`「テーマ間依存関係」節で当該テーマ間に依存（`依存元 → 依存先`）が記載されていない
- 当該テーマ間で編集対象ファイル集合が無衝突である（各計画ファイル`### 対象ファイル一覧`の和集合に重複がない）
- 各テーマの計画ファイル`## 実行方法`が`git push`など作業ツリー全体へ影響する操作を含まない

2段委譲構成の動作は次のとおり。

- 段1（並列実装専従モード）。同一メッセージ内で複数の`spec-driven-implementer`を並列起動し、起動プロンプトで検証・OVERVIEW.md編集・`git commit`の実行を明示禁止する
- 段2（検証・コミット専従モード）。段1完了後、テーマごとに`spec-driven-implementer`を直列起動し、検証・OVERVIEW.md更新・`git commit`の工程を委譲する
  - 1テーマ1コミット境界は段2側で維持する
  - 段2は依存順に直列実行する
- 段1の事前準備として`git rev-parse HEAD`で着手前SHAを取得し、段1完了後に`git diff --name-only <着手前SHA>..HEAD`で編集対象ファイル集合と実差分を照合する
- 段1の全件の完了報告を待ち合わせてから段2へ進む
- 段1でいずれかが`needs_escalation`を返した場合は当該テーマを`4.5 エスカレーション受領時の調停`節へ引き渡し、他テーマの並列処理は続行する
- 段1の事前判定後に編集対象ファイル衝突が発覚した場合は並列断念し当該グループを順次処理（通常モード）へ戻す

計画ファイル内の「## 変更内容」が多論点に跨る場合、起動する`spec-driven-implementer`サブエージェントの内部で
`plan-impl`規定の論点単位タスク分解を当初から細かく適用するよう、起動プロンプトへ明示する。
1計画ファイル1`spec-driven-implementer`サブエージェント1コミットの対応関係は変えず、タイムアウト検知後の事後対応より事前指示を優先する。

### サブエージェント起動プロンプト

通常モード・段1（並列実装専従モード）・段2（検証・コミット専従モード）の3種テンプレートは
`references/launch-prompts.md`の各節を参照し、対象モードのテンプレートをそのまま使う。

### 完了報告の検収

サブエージェント完了報告を検収し、次のいずれかへ分岐する。

- 正常完了報告: 次の計画ファイルへ進む
  - 通常モード:
    - サブエージェント完了報告の「OVERVIEW.md『次に行うべき作業』節を更新したか」項目を確認する
    - `git show --name-only --format= HEAD`で当該コミットの実差分にOVERVIEW.mdが含まれているかも併せて照合する
    - 報告と実差分のいずれかが未更新を示す場合は別途追加コミットで更新する
  - 段1（並列実装専従モード）:
    - 各サブエージェント完了報告の「OVERVIEW.md更新の要否と追記内容案」「編集したファイルパス一覧」「実機検証・VRT更新などメイン側または段2で実施すべき引き継ぎ事項」を集約する
    - 段2サブエージェントの起動プロンプトに集約結果を渡せる形で保持する
  - 段2（検証・コミット専従モード）:
    - サブエージェント完了報告の「OVERVIEW.md更新の実施」「検証結果」「作成コミットのSHA」を確認する
    - `git show --name-only --format= HEAD`で当該コミットの実差分にOVERVIEW.mdが含まれていることを確認する
- `needs_escalation`報告: `4.5 エスカレーション受領時の調停`節の手順で処理する
- 未解決事象を伴う完了報告は次のいずれかへ振り分ける
  - メイン側で技術的に解消可能な事象は`4.5 エスカレーション受領時の調停`節で扱うか、メイン側で当該検証ステップを単独実行して解消する
  - ユーザー判断・ユーザー操作を要する事象はTBD.mdへQブロックとして記録し、当該テーマの実装はそのまま続行する
  - 本スキル全体の実行前提が崩れる事象（「6. 停止判定」の停止契機に該当）のみ、停止理由を本スキル最終メッセージへ記録して終了する
- TBD.mdの回答済み質問は回答内容の性質で反映タイミングを分岐する
  - 判定優先順序は1（最優先）→2→3とし、上位分岐に該当しない場合のみ下位分岐を適用する
  - 1（仕様・方針見直し系）: 仕様・計画本文・対応方針の変更を要する回答（例: 仕様変更指示・要件追加・対応方針の見直し）は、対応する計画ファイル本文を最終確定形に更新したうえで本セッション内で実装サブエージェントを再起動して反映する
  - 2（追加実装系）: 既存方針内に収まる局所的な追加実装要求は、計画ファイル末尾「追加実装タスク」へ追記する
    - 対象は`〜にして`・`〜を直して`・`〜を変更して`等の指示形を含み、計画本文の改訂を要しない回答とする
    - 追記と同時に本セッション内で実装サブエージェントを再起動して反映する（次セッション以降への先送りは行わない）
  - 3（暫定追認系）: `暫定採用でOK`・`採用案でOK`など暫定判断追認のみで追加要求を含まない回答は、計画ファイル進捗ログへ追記しTBD.mdの該当ブロックを削除して完了とする
- 計画ファイル本文（`## 変更内容`等）・実装コード・コミットメッセージ・OVERVIEW.mdに「制限がありスキップ」「未対応」「次サイクル送り」等の未解決状態を文面として残さない。残ると判定される未解決事象は前項「未解決事象を伴う完了報告」の3分岐に従い、完了報告返却・TBD.mdのQブロック化・最終メッセージ記録のいずれかへ収容する

## 5. TBD.md書式

TBD.mdの目的はユーザー確認事項の記録に限定する。
次の用途では使用しない。

- 縮退範囲の記録
- 未着手範囲の記録
- 段階分割の記録
- 実機検証スキップ宣言の記録
- 発見済み既存バグの記録（同セッション内で対処する。次サイクル改修への先送り記録としない）
- ユーザー確認を要しない実装未完了の範囲の記録（同セッション内で対処する）
- 並列実行制約・作業ツリー全体への影響など、呼び出し元（メイン側または段2）が追加判断なしに実行・委譲できる検証・確認の記録（完了報告で呼び出し元へ返却する）。ユーザー判断・ユーザー操作を要する事象は本項目の対象外であり、引き続きTBD.mdへQブロック化する

章構成: 冒頭に本ファイルの説明文、`## {theme name}`配下に`### Q{n}: {要約}`ブロックを並べ、末尾に`## ユーザー指摘`節を設ける。
質問ブロックは3類型から質問の性質に合わせて選ぶ（雛形は`references/qblock-templates.md`参照）。
Qブロックの重複防止規定とユーザー指摘の追記欄雛形は`references/tbd-format-details.md`に従う。

回答欄判定: 直後のHTMLコメント行を除き非空行があれば「回答済み」と判定する。
非空行がなければ「未回答」と判定する。

## 6. 停止判定

auto mode未活性時の起動拒否は本節の停止契機と別経路で扱う（「1. 位置付けと前提」配下「起動前提チェック」サブ節に従う）。
停止契機は「本スキル全体の実行前提崩れ（`docs/v{next}/plans/`不在等の処理対象不在）」のみとし、検知時は本スキルを即時終了して終了理由を最終メッセージに記載する。
計画ファイル実行中の破壊的操作・外部送信は、計画記載の通常工程として扱い停止理由としない。

最終メッセージへ記載する内容。

- 停止契機（前提崩れ）
- 対象計画ファイルパス
- 原因の要約
- 再開時にユーザーへ確認したい事項
