# 自律実行による計画ファイル巡回実装

## 1. 位置付けと前提

入力前提:

- `docs/v{next}/plans/*.md`（計画ファイル群）が1件以上存在する
- 上記の配置は`agent-toolkit:spec-driven`配下`references/spec-driven-framework.md`の既定または
  プロジェクト指定に従う

起動方法:

- 引数なしで起動する（`/agent-toolkit:spec-driven-impl`または`agent-toolkit:spec-driven-plan`からの誘導）
- `docs/v{next}/plans/*.md`を全件巡回する
- 起動時に入力前提を満たさない場合の停止規定は「6. 停止判定」に従う

## 2. ユーザー確認規範のオーバーライド

本スキル実行中は次の確認契機をTBD.md追記＋暫定判断＋続行へ置換する。

- `agent.md`「品質最優先」節および「ユーザーとともに考える」節の確認要求全般
- `claude-code.md`「auto mode下での挙動」節「迷ったら作業を止めて確認」
- `agent-toolkit:plan-mode`の要件対話・認識合わせ・恒久化検討・リファクタリング検討の各ユーザー確認
- `agent-toolkit:plan-impl`「2.5サブエージェント完了報告の検収」のうちユーザー確認を要する分岐

サブエージェント（`spec-driven-implementer`および配下の`plan-implementer`等）は`agent.md`「縮退表明は発行しない」項の規定に従う。
計画ファイル記載の全変更を実装・検証・コミットまで完遂する。
計画ファイルに記載された破壊的操作・外部送信（`git push`・外部APIへの送信・データ削除等）も通常工程として実行し、停止理由としない。
確認事項はTBD.mdへQブロックとして記録し続行根拠とする（停止判断の根拠としない）。
本スキル全体の実行前提が崩れる場合の停止規定は「6. 停止判定」に従う。

## 3. 計画ファイル巡回ループ

1. `docs/v{next}/plans/*.md`の一覧を取得する。
   `docs/v{next}/OVERVIEW.md`に「テーマ間依存関係」節（`依存元 → 依存先`記法）があれば、依存先が依存元より先に並ぶよう並べ替えて処理対象とする。
   節がなければファイル名順とする
2. `docs/v{next}/TBD.md`を読み込み、回答済み質問の反映方針を確定する
   - 未作成ならテンプレで新規作成する
   - 別テーマで記載された未回答事項のうちメイン側で対処可能なものは、当該テーマ着手前に対処する
   - 「ユーザー指摘」欄を走査し、当該テーマに関する指摘があれば反映方針を確定する
3. 計画ファイル1件ごとに「4. 計画ファイル処理の委譲」を実施する（1テーマ1コミット）
4. 未処理テーマがあれば継続可否をユーザーへ確認せず全件を自動継続する
5. 全計画ファイルを処理し終えたら、TBD.mdの未回答件数と冒頭リストを最終メッセージで提示して終了する

## 4. 計画ファイル処理の委譲

メイン側のコンテキスト消費を抑えるため、計画ファイル1件あたり1サブエージェント（`spec-driven-implementer`）を起動する。
当該サブエージェント内で`agent-toolkit:plan-impl`の工程（タスク分解・実装・検証・コミット・レビュー）を完遂する。
メイン側は委譲・完了報告検収・次計画への遷移に専念する。

### テーマ間並列実装の判定と起動

次の条件をすべて満たす複数テーマは`spec-driven-implementer`を実装専従モードで並列起動して効率化する。

- `docs/v{next}/OVERVIEW.md`「テーマ間依存関係」節で当該テーマ間に依存（`依存元 → 依存先`）が記載されていない
- 当該テーマ間で編集対象ファイル集合が無衝突である（各計画ファイル`### 対象ファイル一覧`の和集合に重複がない）
- 各テーマの計画ファイル`## 実行方法`が`git push`など作業ツリー全体へ影響する操作を含まない

並列起動時の動作は次のとおり。

- 並列起動するサブエージェントは実装専従とし、起動プロンプトで検証および`git commit`の実行を明示禁止する
- メイン側は並列実装の完了後に各テーマの検証および`git commit`を順次実施し、1テーマ1コミット境界を維持する
- 並列起動前に`git rev-parse HEAD`で着手前SHAを取得し、各サブエージェント完了後に`git diff --name-only <着手前SHA>..HEAD`で編集対象ファイル集合と実差分を照合する
- 並列起動は同一メッセージ内で複数の`spec-driven-implementer`を起動し、全件の完了報告を待ち合わせる
- 並列起動中にいずれかが`needs_escalation`を返した場合は当該テーマを「4.5エスカレーション受領時の調停」節へ引き渡し、他テーマの並列処理は続行する
- 並列起動の事前判定後に編集対象ファイル衝突が発覚した場合は並列断念し当該グループを順次処理へ戻す

計画ファイル内の「## 変更内容」が多論点に跨る場合、起動する`claude`サブエージェントの内部で
`plan-impl`規定の論点単位タスク分解を当初から細かく適用するよう、起動プロンプトへ明示する。
1計画ファイル1`claude`サブエージェント1コミットの対応関係は変えず、タイムアウト検知後の事後対応より事前指示を優先する。

### サブエージェント起動プロンプト

```text
以下の計画ファイルを`agent-toolkit:plan-impl`スキルに従って実装してください。

- 計画ファイル: `docs/v{next}/plans/{theme}.md`
- 多論点計画の場合は`plan-impl`のタスク分解を論点単位で細かく行ってください
- 実装過程でテーマ仕様（`docs/v{next}/{theme}.md`）・次版総合ドキュメント（`docs/v{next}/OVERVIEW.md`）側の追記・修正が必要と判明した場合は、同サブエージェント内で当該仕様ファイルも更新してください
- `docs/v{next}/OVERVIEW.md`の「次に行うべき作業」節をテーマ実装結果に合わせて更新してください（完了テーマの除去・新発覚事項の追記）。当該更新は計画ファイルのコミットと同一コミットに含めてください
- 完了報告には「OVERVIEW.md『次に行うべき作業』節を更新したか」項目を含めてください
```

並列実装専従モードで起動する場合は上記テンプレートを使わず、次の実装専従テンプレートを使う。

```text
以下の計画ファイルを`agent-toolkit:plan-impl`スキルに従って実装してください。ただし本起動は並列実装専従モードであり、検証および`git commit`の実行は禁止します。

- 計画ファイル: `docs/v{next}/plans/{theme}.md`
- 多論点計画の場合は`plan-impl`のタスク分解を論点単位で細かく行ってください
- 実装過程でテーマ仕様（`docs/v{next}/{theme}.md`）・次版総合ドキュメント（`docs/v{next}/OVERVIEW.md`）側の追記・修正が必要と判明した場合は、同サブエージェント内で当該仕様ファイルも更新してください
- `docs/v{next}/OVERVIEW.md`の「次に行うべき作業」節をテーマ実装結果に合わせて更新してください
- 検証コマンドの実行および`git commit`は呼び出し元（メイン）が並列実装の完了後に順次実施するため、本サブエージェント内では実行しないでください
- 完了報告には「OVERVIEW.md『次に行うべき作業』節を更新したか」項目と、編集したファイルパス一覧を含めてください
```

### 完了報告の検収

サブエージェント完了報告を検収し、次のいずれかへ分岐する。

- 正常完了報告: 次の計画ファイルへ進む
  - 「OVERVIEW.md『次に行うべき作業』節を更新したか」項目を確認し、未更新と報告された場合は別途追加コミットで更新する
- 停止理由を伴う報告: 停止理由を本スキル最終メッセージへ記録し、次の計画ファイルへ進む
- `needs_escalation`報告: `4.5 エスカレーション受領時の調停`節の手順で処理する
- TBD.mdに既存の回答済み質問があれば、暫定採用との一致／不一致を判定し、不一致なら次サイクル以降の
  改修タスクとして当該計画ファイルへ追記する

## 4.5 エスカレーション受領時の調停

サブエージェントが`needs_escalation`で返した場合、本節の手順でメイン側調停を実施する。
対象はTBD.md追記のみで解消しない事象に限る。

1. 受領情報の確認: サブエージェントの完了報告から「エスカレーション原因」「該当ステップ」「TBD.mdの集約Qブロック識別子」を抽出する
2. 原因分類: 次の3類型へ分類する
   - 環境制約系（並列実行不可・外部サービス未起動・権限不足等）
   - 計画不整合系（前段テーマ未完了・前提仕様の食い違い等）
   - 入力不足系（テストデータ不在・設定値の未定義等）
3. 分類別の調停手順を実施する
   - 環境制約系: メイン側で当該検証ステップを逐次実行へ切り替えるか、検証コマンドを単独実行する。解消した場合は当該テーマのサブエージェント再起動で続行する
   - 計画不整合系: 当該計画ファイルの`## 変更内容`または依存テーマ計画ファイルを修正してから再起動する
   - 入力不足系: 必要素材を準備または取得し、計画ファイルへ反映してから再起動する
4. 調停結果の記録: TBD.mdの当該集約Qブロックの`回答`欄に調停内容と再起動結果を記入し、Qブロックを「回答済み」化する
5. 再起動可否: 再起動でも同一原因が3回以上発生した場合は調停不能として最終メッセージへ記録し次テーマへ進む（「6. 停止判定」の停止契機には該当しない）

調停は技術的解消が見込める範囲に限定し、「2. ユーザー確認規範のオーバーライド」節を覆さない。ユーザー判断を要する事項はTBD.mdへ残して次テーマへ進む。

## 5. TBD.md書式

TBD.mdの目的はユーザー確認事項の記録に限定する。
次の用途では使用しない。

- 縮退範囲の記録
- 未着手範囲の記録
- 段階分割の記録
- 実機検証スキップ宣言の記録
- 発見済み既存バグの記録（同セッション内で対処する。次サイクル改修への先送り記録としない）
- ユーザー確認を要しない実装未完了の範囲の記録（同セッション内で対処する）

章構成: 冒頭に本ファイルの説明文、`## {theme name}`配下に`### Q{n}: {要約}`ブロックを並べ、末尾に`## ユーザー指摘`節を設ける。
質問ブロックは3類型から質問の性質に合わせて選ぶ（雛形は`references/qblock-templates.md`参照）。

### 重複防止

新規Qブロック追記前に、同テーマ配下（`## {theme name}`配下）の既存Qブロックを走査して重複有無を確認する。

- 要約・関連箇所・背景がほぼ同一の質問が存在する場合は新規Qブロックを追加せず、既存ブロックの末尾サブバレットへ`- 検知回数: {回数}`を更新する
- E2Eテスト・動作確認などの環境制約由来で実行不能な事象は、サブエージェント実行のたびに同一原因で発生し得るため原因単位で1Qブロックへ集約する
- 重複判定基準は関連箇所のパスまたは節名が一致し、かつ背景・暫定採用の要旨が同義であること
- 同一原因事象を3回以上検知した場合は重複追記を打ち切り、サブエージェントは`needs_escalation`を返してメイン側調停（`4.5 エスカレーション受領時の調停`節）に引き渡す

回答欄判定: 直後のHTMLコメント行を除き非空行があれば「回答済み」と判定する。
非空行がなければ「未回答」と判定する。

### ユーザー指摘の追記欄

質問ブロックと並置できる形でユーザーが任意のタイミングで指摘を書き込めるよう、TBD.mdの末尾に`## ユーザー指摘`節を設ける。
雛形は次の通り。

````markdown
## ユーザー指摘

### U{n}: {一行で要約}

- 関連箇所: `path/to/file` または `docs/v{next}/plans/{theme}.md`の節名
- 指摘内容:
  <!-- ユーザーはこの行以降に指摘を記入する -->
- 反映方針: {サブエージェントが計画ファイル巡回時に記入する}
````

サブエージェントは計画ファイル巡回ループの冒頭で本節を走査し、未対応の指摘について反映方針を確定する。
対応完了後も履歴として削除しない。

## 6. 停止判定

停止契機は「本スキル全体の実行前提崩れ（`docs/v{next}/plans/`不在等の処理対象不在）」のみとし、検知時は本スキルを即時終了して終了理由を最終メッセージに記載する。
計画ファイル実行中の破壊的操作・外部送信は、計画記載の通常工程として扱い停止理由としない。

最終メッセージへ記載する内容。

- 停止契機（前提崩れ）
- 対象計画ファイルパス
- 原因の要約
- 再開時にユーザーへ確認したい事項
