# 自律実行による計画ファイル巡回実装

## 1. 位置付けと前提

入力前提:

- `docs/v{next}/plans/*.md`（計画ファイル群）が1件以上存在する
- 上記の配置は`agent-toolkit:spec-driven`配下`references/spec-driven-framework.md`の既定または
  プロジェクト指定に従う

起動方法:

- 引数なしで起動する（`/agent-toolkit:spec-driven-impl`または`agent-toolkit:spec-driven-plan`からの誘導）
- `docs/v{next}/plans/*.md`を全件巡回する
- 起動時に入力前提を満たさない場合の停止規定は「6. 停止判定」に従う

## 2. ユーザー確認規範のオーバーライド

本スキル実行中は次の確認契機をTBD.md追記＋暫定判断＋続行へ置換する。

- `agent.md`「品質最優先」節および「ユーザーとともに考える」節の確認要求全般
- `claude-code.md`「auto mode下での挙動」節「迷ったら作業を止めて確認」
- `agent-toolkit:plan-mode`の要件対話・認識合わせ・恒久化検討・リファクタリング検討の各ユーザー確認
- `agent-toolkit:plan-impl`「2.5サブエージェント完了報告の検収」のうちユーザー確認を要する分岐

サブエージェント（`spec-driven-implementer`および配下の`plan-implementer`等）は`agent.md`「縮退表明は発行しない」項の規定に従う。
計画ファイル記載の全変更を実装・検証・コミットまで完遂する。
計画ファイルに記載された破壊的操作・外部送信（`git push`・外部APIへの送信・データ削除等）も通常工程として実行し、停止理由としない。
確認事項はTBD.mdへQブロックとして記録し続行根拠とする（停止判断の根拠としない）。

本スキル全体の実行前提が崩れる状態（`docs/v{next}/plans/`不在・走査結果が空など）を検知した場合は、
処理対象が成立しないため本スキル自体を即時終了する。
終了理由は本スキル最終メッセージに記載する。

## 3. 計画ファイル巡回ループ

1. `docs/v{next}/plans/*.md`の一覧を取得する。
   `docs/v{next}/OVERVIEW.md`に「テーマ間依存関係」節（`依存元 → 依存先`記法）があれば、依存先が依存元より先に並ぶよう並べ替えて処理対象とする。
   節がなければファイル名順とする
2. `docs/v{next}/TBD.md`を読み込み、回答済み質問の反映方針を確定する
   - 未作成ならテンプレで新規作成する
   - 別テーマで記載された未回答事項のうちメイン側で対処可能なものは、当該テーマ着手前に対処する
   - 「ユーザー指摘」欄を走査し、当該テーマに関する指摘があれば反映方針を確定する
3. 計画ファイル1件ごとに「4. 計画ファイル処理の委譲」を実施する
4. 全計画ファイルを処理し終えたら、TBD.mdの未回答件数と冒頭リストを最終メッセージで提示して終了する

テーマ1件（計画ファイル1件）ごとに1コミットとする。

テーマ間の継続可否確認をユーザーへ問わず、未処理テーマがあれば全件を自動継続する。

## 4. 計画ファイル処理の委譲

メイン側のコンテキスト消費を抑えるため、計画ファイル1件あたり1サブエージェント（`spec-driven-implementer`）を起動する。
当該サブエージェント内で`agent-toolkit:plan-impl`の工程を完遂する。
工程はタスク分解・実装・検証・コミット・レビューを含む。
メイン側は委譲・完了報告検収・次計画への遷移に専念する。

計画ファイル内の「## 変更内容」が多論点に跨る場合、起動する`claude`サブエージェントの内部で
`plan-impl`規定の論点単位タスク分解を当初から細かく適用するよう、起動プロンプトへ明示する。
1計画ファイル1`claude`サブエージェント1コミットの対応関係は変えず、タイムアウト検知後の事後対応より事前指示を優先する。

### サブエージェント起動プロンプト

```text
以下の計画ファイルを`agent-toolkit:plan-impl`スキルに従って実装してください。

- 計画ファイル: `docs/v{next}/plans/{theme}.md`
- 多論点計画の場合は`plan-impl`のタスク分解を論点単位で細かく行ってください
- 実装過程でテーマ仕様（`docs/v{next}/{theme}.md`）・次版総合ドキュメント（`docs/v{next}/OVERVIEW.md`）側の追記・修正が必要と判明した場合は、同サブエージェント内で当該仕様ファイルも更新してください
- `docs/v{next}/OVERVIEW.md`の「次に行うべき作業」節をテーマ実装結果に合わせて更新してください（完了テーマの除去・新発覚事項の追記）。当該更新は計画ファイルのコミットと同一コミットに含めてください
- 完了報告には「OVERVIEW.md『次に行うべき作業』節を更新したか」項目を含めてください
```

### 完了報告の検収

サブエージェント完了報告を検収し、次のいずれかへ分岐する。

- 正常完了報告: 次の計画ファイルへ進む
  - 「OVERVIEW.md『次に行うべき作業』節を更新したか」項目を確認し、未更新と報告された場合は別途追加コミットで更新する
- 停止理由を伴う報告: 停止理由を本スキル最終メッセージへ記録し、次の計画ファイルへ進む
- TBD.mdに既存の回答済み質問があれば、暫定採用との一致／不一致を判定し、不一致なら次サイクル以降の
  改修タスクとして当該計画ファイルへ追記する

## 5. TBD.md書式

TBD.mdの目的はユーザー確認事項の記録に限定する。
次の用途では使用しない。

- 縮退範囲の記録
- 未着手範囲の記録
- 段階分割の記録
- 実機検証スキップ宣言の記録

章構成: 冒頭に本ファイルの説明文、`## {theme name}`配下に`### Q{n}: {要約}`ブロックを並べる。
別途、ユーザー指摘記入用の`## ユーザー指摘`節を設ける。

質問ブロックは次の3類型から質問の性質に合わせて選ぶ。

選択式雛形（複数選択肢から1つ選ぶ場合）:

````markdown
### Q{n}: {一行で要約}

- 発生工程: {計画|実装|検証|コミット}
- 関連箇所: `path/to/file` または `docs/v{next}/plans/{theme}.md`の節名
- 背景: {何が決められず何を仮置きしたか}
- 選択肢:
  - A: {...}
  - B: {...}
- 暫定採用: {ラベル}（理由: {根拠}）
- 回答:
  <!-- ユーザーはこの行以降に回答を追記する -->
````

YES/NO雛形（二択で回答できる場合）:

````markdown
### Q{n}: {一行で要約}

- 発生工程: {計画|実装|検証|コミット}
- 関連箇所: `path/to/file` または `docs/v{next}/plans/{theme}.md`の節名
- 背景: {何が決められず何を仮置きしたか}
- 質問: {YES/NOで回答できる問い}
- 暫定採用: {YES または NO}（理由: {根拠}）
- 回答:
  <!-- ユーザーはYESまたはNOを記入する -->
````

ユーザー作業必須雛形（権限不足等でユーザー操作が必要な場合）:

````markdown
### Q{n}: {一行で要約}

- 発生工程: {計画|実装|検証|コミット}
- 関連箇所: `path/to/file` または `docs/v{next}/plans/{theme}.md`の節名
- 背景: {何が決められず何を仮置きしたか}
- ユーザー操作内容: {必要な操作の具体手順}
- 権限要件: {sudo、root権限、外部サービスアカウント等}
- 完了確認方法: {ユーザーが完了を本スキル側へ伝える手段}
- 回答:
  <!-- ユーザーは操作完了後にこの行以降に記入する -->
````

回答欄判定: 直後のHTMLコメント行を除き非空行があれば「回答済み」と判定する。
非空行がなければ「未回答」と判定する。

### ユーザー指摘の追記欄

質問ブロックと並置できる形でユーザーが任意のタイミングで指摘を書き込めるよう、TBD.mdの末尾に`## ユーザー指摘`節を設ける。
雛形は次の通り。

````markdown
## ユーザー指摘

### U{n}: {一行で要約}

- 関連箇所: `path/to/file` または `docs/v{next}/plans/{theme}.md`の節名
- 指摘内容:
  <!-- ユーザーはこの行以降に指摘を記入する -->
- 反映方針: {サブエージェントが計画ファイル巡回時に記入する}
````

サブエージェントは計画ファイル巡回ループの冒頭で本節を走査し、未対応の指摘について反映方針を確定する。
対応完了後も履歴として削除しない。

## 6. 停止判定

停止契機は「本スキル全体の実行前提崩れ（`docs/v{next}/plans/`不在等の処理対象不在）」のみとする。
前提崩れを検知した場合は本スキル自体を即時終了し、終了理由を最終メッセージに記載する。
計画ファイル実行中の破壊的操作・外部送信は、計画記載の通常工程として扱い停止理由としない。

最終メッセージへ記載する内容。

- 停止契機（前提崩れ）
- 対象計画ファイルパス
- 原因の要約
- 再開時にユーザーへ確認したい事項
