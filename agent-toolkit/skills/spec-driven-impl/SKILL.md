# 自律実行による計画ファイル巡回実装

## 1. 位置付けと前提

入力前提:

- `docs/v{next}/plans/*.md`（計画ファイル群）が1件以上存在する
- 上記の配置は`agent-toolkit:spec-driven`配下`references/spec-driven-framework.md`の既定または
  プロジェクト指定に従う

起動方法:

- 引数なしで起動する（`/agent-toolkit:spec-driven-impl`または`agent-toolkit:spec-driven-plan`からの誘導）
- `docs/v{next}/plans/*.md`を全件巡回する
- 起動時に入力前提を満たさない場合の停止規定は「6. 停止判定」に従う

### 起動前提チェック

本スキルは`agent-toolkit/rules/02-collaboration.md`が定義する自律実行モード下での
実行を前提とする。`Auto Mode Active`注入が無い場合、
「本スキルは自律実行モード専用・再起動はauto mode有効化後に行う」旨を1行通知して即時終了する。

## 2. ユーザー確認規範のオーバーライド

本スキル実行中は`agent-toolkit/rules/02-collaboration.md`の「自律実行モード」節を適用する。
本スキルが確立する前提は次のとおり。

- TBD保存先: `atk`不在時は`docs/v{next}/TBD.md`
- スコープ: 「テーマ単位の計画ファイル巡回（1テーマ1コミット境界）」
- 追加停止契機: なし（同節既定の前提崩れのみで停止する）

本スキル固有のオーバーライド対象として、
`plan-impl-executor`手順「2.5サブエージェント完了報告の検収」のうち
ユーザー確認を要する分岐も含める。

サブエージェント（`spec-driven-implementer`および配下の`plan-implementer`等）も
`agent-toolkit/rules/02-collaboration.md`の「自律実行モード」節の同規定に従う。
ただし`agent-toolkit/rules/03-claude-code.md`「サブエージェントの活用」節のサブエージェント`git push`禁止規定が優先される。
計画ファイルに`git push`が記載されていてもメイン側で全タスク完了後にまとめて実施する。
本スキル全体の実行前提が崩れる場合の停止規定は「6. 停止判定」に従う。

## 3. 計画ファイル巡回ループ

1. `docs/v{next}/plans/*.md`の一覧を取得する
   - `docs/v{next}/OVERVIEW.md`に「テーマ間依存関係」節（`依存元 → 依存先`記法）があれば、
     依存先が依存元より先に並ぶよう並べ替えて処理対象とする
   - 節がなければファイル名順とする
2. `docs/v{next}/TBD.md`を読み込み、回答済み質問の反映方針を確定する
   - 未作成ならテンプレで新規作成する
   - 別テーマで記載された未回答事項のうちメイン側で対処可能なものは、当該テーマ着手前に対処する
   - 「ユーザー指摘」欄を走査し、当該テーマに関する指摘があれば反映方針を確定する
3. 計画ファイル1件ごとに「4. 計画ファイル処理の委譲」を実施する（1テーマ1コミット）
4. 下記「完遂条件と再実行責務」の条件達成まで全件を自動継続する。
   達成時、TBD.mdの未回答件数と冒頭リストを最終メッセージで提示し終了する

### 完遂条件と再実行責務

巡回は1巡で終了せず、次のいずれかに達するまで続行する。

- 全計画ファイルの正常完了
- 残未解決事象がユーザー判断・操作必須（`references/qblock-templates.md`「ユーザー作業必須雛形」適用範囲規定）として
  `docs/v{next}/TBD.md`へ集約済みであること

中断テーマは原因解消後にサブエージェントへ委譲して再実行する（メイン側で直接実行しない）。
同一原因が3回以上の場合は「4.5節エスカレーション受領時の調停」の規定に従い
原因分類（環境制約系・計画不整合系・入力不足系）を再評価し、試行履歴はTBD.mdへ追記する。
全分類で解消不可能と確定した場合のみ調停不能として最終メッセージへ記録し次テーマへ進む（停止契機には該当しない）。

## 4. 計画ファイル処理の委譲

メイン側のコンテキスト消費を抑えるため計画ファイル1件あたり1サブエージェント（`spec-driven-implementer`）を起動する。
当該サブエージェント内で`plan-impl-executor`工程（タスク分解・実装・検証・コミット・レビュー）を完遂する。
メイン側はオーケストレーションに専念し、次の作業のみを担う。

- 計画ファイル一覧の取得と依存解決による並べ替え
- TBD.md読み込みと`## ユーザー指摘`走査
- `spec-driven-implementer`の起動と完了報告検収
- 並列起動の可否判定（編集対象ファイル衝突判定など）
- `needs_escalation`受領時の調停（`4.5 エスカレーション受領時の調停`節）
- 最終メッセージの提示
- サブエージェント完了通知後の`git status --short`・`git diff --stat`による作業ツリー実態照合
  （`01-agent.md`「サブエージェントの活用」節の事務的照合）

メイン側では計画ファイル・テーマ仕様・OVERVIEW.mdの編集、検証コマンド、`git commit`は実行せず、
全て`spec-driven-implementer`へ委譲する。

### サブエージェント起動時のモデル指定

メイン側モデルに応じて`spec-driven-implementer`起動時のモデル指定を切り替える。

- 本スキル起動時、メイン側自身が動作するモデル名をシステムプロンプトの`You are powered by the model named ...`から読み取る
- モデル系統がopus未満（Sonnet・Haiku等）の場合、`spec-driven-implementer`起動時のパラメーターで`model: opus`を明示指定する
- モデル系統がopus以上の場合は親モデル継承のままとする（明示指定不要）
- 段1（並列実装専従モード）・段2（検証・コミット専従モード）・直列モード（通常モード）のいずれでも同じ判定を適用する

### ユーザー割り込み指示の対応

本スキル実行中の割り込み指示は「4. 計画ファイル処理の委譲」節冒頭の役割分担を維持して扱う。

- 既定として現在実行中の`spec-driven-implementer`（並列起動中は全件）の完了を待ってから対処へ着手する。
  割り込み趣旨が現在テーマの即時停止を明示する場合のみ未完了サブエージェントへ`TaskStop`を発行してから進める
- 状況確認・原因切り分け・対象テーマ特定はメイン側で行ってよい
- 実作業と対処完了後の原テーマ再開は新規`spec-driven-implementer`起動で実施する。
  `completed`状態のサブエージェントへの`SendMessage`は失敗するため、
  割り込み趣旨・切り分け結果・対象ファイル集合・原テーマ計画ファイル位置を起動プロンプトに含める

### テーマ間並列実装の判定と起動

次の条件をすべて満たす複数テーマは`spec-driven-implementer`を2段委譲構成で並列起動して効率化する。

- `docs/v{next}/OVERVIEW.md`「テーマ間依存関係」節で当該テーマ間に依存（`依存元 → 依存先`）が記載されていない
- 当該テーマ間で編集対象ファイル集合が無衝突である（各計画ファイル`### 対象ファイル一覧`の和集合に重複がない）
- 各テーマの計画ファイル`## 実行方法`が`git push`など作業ツリー全体へ影響する操作を含まない

2段委譲構成の動作は次のとおり。

- 段1（並列実装専従モード）: 同一メッセージ内で複数の`spec-driven-implementer`を並列起動し、
  検証・OVERVIEW.md編集・`git commit`を明示禁止する
- 段2（検証・コミット専従モード）: 段1完了後にテーマごとへ直列起動し検証・OVERVIEW.md更新・`git commit`を委譲する。
  1テーマ1コミット境界と依存順の直列実行は段2で維持する
- 段1着手前に`git rev-parse HEAD`でSHAを取得し、段1完了後に`git diff --name-only <SHA>..HEAD`で照合する
- 段1完了を全件待ち合わせてから段2へ進む。`needs_escalation`受領時は当該テーマを4.5節へ引き渡し他テーマ続行する
- 段1事後に衝突が発覚した場合は並列断念し当該グループを通常モードへ戻す

「## 変更内容」が多論点に跨る場合、論点単位タスク分解を当初から適用するよう起動プロンプトへ明示する。
1計画ファイル1`spec-driven-implementer`1コミットの対応関係は変えず、タイムアウト後の事後対応より事前指示を優先する。

### サブエージェント起動プロンプト

通常モード・段1（並列実装専従モード）・段2（検証・コミット専従モード）の3種テンプレートは
`references/launch-prompts.md`の各節を参照し、対象モードのテンプレートをそのまま使う。

### 完了報告の検収

サブエージェント完了報告を検収し、次のいずれかへ分岐する。

- 正常完了報告: 次の計画ファイルへ進む
- `needs_escalation`報告: `4.5 エスカレーション受領時の調停`節の手順で処理する
- 未解決事象を伴う完了報告: 事象の技術的な解消可能性を評価し、対処専用の新規`spec-driven-implementer`起動・
  `needs_escalation`調停・TBD.mdへのQブロック記録・停止判定のいずれかへ振り分ける

モード別の照合手順・未解決事象の振り分け詳細・解決済み情報の削除・TBD.md回答済み質問の反映タイミング・
未解決状態の文面禁止規定は`references/completion-report-review.md`を参照する。

## 4.5 エスカレーション受領時の調停

詳細手順は`references/escalation.md`を参照する。

## 5. TBD.md書式

記録先判定（`atk`有無分岐）・目的・用途制限・章構成の基本・質問ブロック雛形・回答欄判定は
`agent-toolkit/rules/02-collaboration.md`の「自律実行モード」節「3. TBD記録の書式」（SSOT）に従う。
本スキル固有事項は次のとおり。

- `atk`不在時の保存先パスは`docs/v{next}/TBD.md`とする（「2. ユーザー確認規範のオーバーライド」で
  確立済み）
- `atk fb`使用時のスコープ（`--scope=`）・回答済み判定は計画ファイル単位（`<theme name>`）とする
- スコープ区分は計画ファイル単位とし、`## {theme name}`配下に`### Q{n}: {要約}`ブロックを並べる
- 質問ブロックの3類型の雛形は`references/qblock-templates.md`を参照する
- Qブロックの重複防止規定と`## ユーザー指摘`欄の追記欄雛形は`references/tbd-format-details.md`に従う
- TBD.md未作成時はテンプレートで新規作成する（「3. 計画ファイル巡回ループ」ステップ2参照）

## 6. 停止判定

本スキルの停止判定は`agent-toolkit/rules/02-collaboration.md`の「自律実行モード」節
「4. 停止判定」に従う。本スキル固有の停止契機は次のとおり。

- 本スキル全体の実行前提崩れ（`docs/v{next}/plans/`不在等の処理対象不在）

計画ファイル実行中の破壊的操作・外部送信は、計画記載の通常工程として扱い停止理由としない。
個別計画ファイル処理中のエスカレーション・調停不能事象は本節の停止契機に該当せず、
`4.5 エスカレーション受領時の調停`節および「3. 計画ファイル巡回ループ」配下「完遂条件と再実行責務」節に従う。
最終メッセージには、停止契機・対象計画ファイルパス・原因の要約・再開時の確認事項を記載する。
