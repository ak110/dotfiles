# フィードバックの批判的検討と適用

## 基本方針

各採否判定の最優先基準は、当該フィードバックの適用が成果物および今後の作業の品質向上に寄与するかとする。
手順を機械的に踏襲して採否を確定しない。
形式整合のみを根拠とした採用・不採用判定など、品質寄与の評価を伴わない判定を避ける。
不採用根拠の限定はステップ3の規定に従う。

## ステップ1: 入力の確認

ユーザーのメッセージからフィードバック項目を抽出する。
フィードバックが含まれていない場合はユーザーに提供を求める。

## ステップ2: 内容の調整

フィードバックの記述にとらわれず、各提案の追記先と抽象度を最適化する。
適用順は次のとおりとする。最初に「主提案の確定」で主提案を決め、確定した主提案に対して
「追記先の検討」「抽象度の調整」「規範側の再発予防案の併記検討」「既存SSOT規範との衝突回避」
「引用素材の原文の表記点検」を順に適用する。

- 主提案の確定: 提案がリカバリー手順の文書化・
  観測主体（レビュアー・lint・型チェック・hook等）の誤検出への適用除外コメント・前提明示の追記・
  例外規定の新設など場当たり対処に該当する場合の判定軸とする。
  この場合、根本対応（コード修正・設定変更・自動化スクリプトの整備・CI制約の追加・
  誤検出を生む規範側の改訂や撤廃など）が成立しないかを最初に検討する。
  誤検出由来の事象は規範側の改訂や撤廃を優先し、成立する場合は根本対応案を主提案とし元の追記提案は不採用扱いとする。
  成立しない場合は元の追記提案をそのまま主提案とする
  - 根本対応案を表層レベルの整理（内部重複の撤去のみ・既存実装の再利用のみなど）に限定した場合、
   `agent-toolkit:coding-standards`「責務帰属（SRP）分析の観点」節に従い判定確定前に分析する
  - フィードバック原文がワークアラウンド表現
   （フォールバック・回避策・迂回・失敗時対処・「〜が失敗する場合は〜する」など）を含む場合は、
    根本対応検討を経ずに元の追記提案を主提案へ確定しない
  - ワークアラウンド表現を含む場合、根本対応の検討結果を
   `${TMPDIR:-/tmp}/scratchpad/workaround-check-<計画ファイルstem>.md`へ記録する必須ゲートとする。
    記録項目は「根本原因の候補」「根本対応が成立するか」「成立しない場合の理由」の3項目、
    複数フィードバック統合時も1計画1メモとする。
    メモ不在または未記入の場合はpretooluse.pyのWrite検査がブロックし採用確定できない
- 追記先の検討: 提案が示す対象ファイルにとらわれず、最適な追記先を選ぶ。
  コードに近い側（docstring・インラインコメント）や既存の関連セクションも候補に含める。
  フィードバックが既存追記そのものを否定する内容のときは削除候補も含める。
  追記先候補の節タイトルが定める性質（実装時契約・計画時手順・注意点・運用ノウハウ等）と
  追記内容の性質が整合するかを事前検証する。
  不整合時は既存節の別候補・意味論拡張・新設節を比較して追記先を確定する。
  追記先候補が既存bulletの子bullet位置となる場合は、追記前に親bulletの適用条件
  （前提・トリガー・対象事象）をReadで確認し、追記内容の前提と照合する。
  前提が異なる場合は子bullet位置での従属配置を採用せず、独立の親bulletとして配置するか、
  親bulletの前提条件文言を追記内容の前提を含む形に改める
  - 反映内容の抽象度を上位カテゴリへ昇格できるかを点検する
  - 反映内容が複数スキル・複数エージェントへ一般化できる規範の場合、上位カテゴリへの反映を第一候補とする。
    上位カテゴリの例は上位ルールファイル（`agent-toolkit/rules/01-agent.md`）と横断スキル（`agent-toolkit:agent-standards`）
  - 1階層上のカテゴリで規範として成立するかを固有名詞除外テストで点検する。
    テスト手順は対象スキル名・対象CLIコマンド名を伏せて規範文が成立するかを確認する
  - 成立する場合は上位カテゴリへ昇格させてから反映先を確定する
- 抽象度の調整: 特定セッションの固有事象に特化しすぎている場合は一般化し、逆に抽象化しすぎて実効性を欠く場合は具体化する。
  事前にユーザー提示素材の原文の主旨を再確認し、批判的検討の前提となる解釈に誤読が無いことを確認する。
  既存節への新ケース追加・列挙拡張の判定基準は`agent-toolkit:agent-standards`「既知情報・冗長記述の排除」節に従う
  - 全体点検指示（「点検して」「見直して」等）と例示（「例えば〜」「〜などは」形式）を併せ持つ原文では、
    原文の範囲を例示に限定せず点検対象全体を表形式で洗い出し`## 調査結果`へ転記する。
    表はステップ3の批判的検討へ引き渡すべき情報を網羅し、判定は既存採否語彙（採用／部分採用／不採用）に従う
- 規範側の再発予防案の併記検討:
  事象の根本原因が既存規範文書の曖昧さに帰せられる場合は規範への改善案も提案候補に含める
 （観点の詳細は`agent-toolkit:session-review`スキルの「規範改善案の併記」項目に従う）
- 既存SSOT規範との衝突回避:
  フィードバック原文の改善方針案を機械的に反映する前に、既存SSOT規範との衝突有無を点検する。
  対象は`agent-toolkit:review-standards`の重大度ラベル規定・
  `agent-toolkit:writing-standards`のMarkdown記述スタイル規定などとする。
  点検時は対象SSOT規範文書の関連節をReadで原文確認してから提案本文を確定する。
  衝突する場合は既存SSOTを維持したうえで、参照誘導強化・上位指針への昇格・既存記述の表現強化などの
  代替案から衝突を最小化する反映案へ書き換えてから提案する
  - フィードバック原文が独自の分類ラベル（判定区分・重大度ラベル等）・
    独自の構造定義（対応表・分類テンプレート・列定義等）・
    独自の処理フロー（工程順序・判定基準等）を提示する場合の扱い。
    追加対象スキルの既存採否語彙・既存ステップ構成・既存テンプレートとの、
    概念モデルおよび語彙の一致を事前検証する。
    既存の枠組みへ委譲可能な要素は、原文の独自定義を破棄し既存枠組みへ統合する形へ書き換えてから提案する
- 引用素材の原文の表記点検:
  フィードバック原文や引用素材の表記をそのまま規範文面へ転記する前に、
  JIS Z 8301:2019等の日本語の表現規範に照らして数学記号・略記・英字混在の有無を点検する。
  違反がある場合は転記前に自然な日本語表現へ書き換える。
  対比例: `≤`→「以下」、`≥`→「以上」、`<`→「未満」、`>`→「超」

## ステップ2.5: 採否判定前の網羅調査

ステップ3の批判的検討に入る前に、各提案の追記先となる既存記述と関連スキルを横断調査する。
調査は提案ごとに次の各点を確定する。

- 追記先候補ファイルの該当節を読み、既存記述または既存実装で同等内容が既にカバー済みでないかを確認する。
  併せて同ドメインの既存規範が`agent-toolkit/rules/`・`agent-toolkit/skills/`配下に存在しないかを`grep -rn`で網羅確認する。
  既存規範が存在する場合はスコープ拡張・条件補強による一本化案を優先候補とする
- 禁止規定・推奨規定との照合:
  フィードバック原文が示す反映パターン（キーワード・実装手法・禁止/推奨の主張）を
  `agent-toolkit/rules/`・`agent-toolkit/skills/`配下の既存禁止規定・推奨規定と照合する。
  照合手順は原文キーワードで`grep -rn`し、禁止規定・推奨規定として書かれた節を確認する。
  抵触する場合はステップ3の批判的検討へ引き渡し、既存規範との衝突可否を判定する
- 関連スキル・関連ルールファイルに重複・矛盾する規範が無いかを確認する
- 追記先候補が配置先スキルの管轄範囲に該当する場合、採否判定・計画作成に着手する前に当該配置先スキルをSkillツールで起動する
  - 管轄範囲はプロジェクト直下`AGENTS.md`・`CLAUDE.md`で編集対象ディレクトリに対して指定された範囲とする
  - 配布物境界・version bumpの基準・固有名の混入抑止などの配置先固有の規範を採否判定段階で適用する
  - 起動失念時に発生し得る事象の例: version bump対象ファイルの`## 変更内容`欠落・配布物境界の規範の未適用・固有名混入の事後検出
- 提案の主旨を原文転記から再確認し、自身の解釈誤りが無いかを確認する
- 追記後の最終形に近い文面を組み立てて`wc -l`で実測し、`agent-toolkit:agent-standards`「文書サイズ上限」節の規定に従う
- フィードバック原文の取得・解釈が完遂できない場合は、規範改訂・代替反映の判定より前に次で進める。
  対象事例は外部URL取得失敗・素材参照ができない場合・前提仕様の確定ができない場合などとする。
  原文が問いかけ調・抽象的で具体反映内容を一意に確定できない場合は具体化候補を2〜4案生成し、
  `AskUserQuestion`で提示してユーザーに選ばせる（多肢を併せて採る想定の場合は`multiSelect: true`を許容）。
  候補生成が不可能な場合に限り作業を中断してユーザーへ取扱（保留・不採用・原文貼付要求）を確認する
- 実装挙動異常の観測時は採否判定前に`agent-toolkit:session-review`スキル「事前調査の徹底」項目に従う。
  調査範囲が広い場合は`Explore`サブエージェントへ委譲する

調査結果を踏まえて初めてステップ3の批判的検討へ移行する。
調査未了のまま採否を断定しない。

フィードバックが複数件の場合は、
各フィードバックの追記先調査を`Explore`サブエージェントへ個別に並列委譲する（必須ゲート）。
Explore起動プロンプトは`references/explore-template.md`「Explore委譲雛形」節を機械転記して構築する。
本節の点検項目全件（追記先の候補確認・既存規範のgrep網羅・関連スキル重複確認・
配置先スキル起動要否判断・原文再確認・`wc -l`実測）を委譲プロンプトへ転記漏れなく含める。
メイン側で順次調査を進めるとメイン側コンテキストが膨張し、
後続のplan-mode工程（工程6の初版ファイル作成・工程7の各種レビュー）を実質的にバイパスした
直接編集への切替を誘発する。
メイン側は返却された調査結果を集約してステップ3の内容を批判的に検討する。

フィードバック原文にリリース待ち・別計画完了待ち等の前提条件が明記されている場合は、
当該前提の充足を実機検証してから採否判定する。
前提が未充足なら「保留」として扱い、`atk fb reject`せずinbox配下へ残置する。
実機検証手段の例として、対象パッケージの公開バージョン確認・対象CLIのhelp出力確認・
対象モジュールのimport可否確認などとする。
保留と判定したフィードバックはステップ4提示で「採否: 保留。<保留理由>」と表記し、
反映先・反映内容欄は省略する。今回の計画作成対象から除外し、次回起動時に再評価対象となる。
全件保留の場合は計画作成に進まず処理を終了する。

- 保留判定時の事前確認事項のTBD投入は`references/hold-with-tbd-inject.md`に従う

## ステップ3: 批判的検討

調整後の内容で、各提案が対応に値するかを評価する。

- 批判的検討の観点:
  `(a)` plan-impl以降に発生する問題をplan-mode以前に前もって対策できるような改善提案は、
  品質改善効果が見込まれるため最優先で採用する。
  `(b)` 数回のツール呼び出しを削減できるような改善提案（事前知識の追加など）は、
  基本的にメリットとデメリットが見合わないため不採用とする。
  ただしツール側の改善などで（コマンド出力の素直化・既存機能の使い方明確化等）より素直な挙動になるものは採用する
- 批判的検討の対象限定:
  単一要求内は採用・不採用の二値評価のみを対象とする。
  複数の独立要求が並列に含まれる場合の分解判定は後述「複数要求の分解判定」項に従う。
  原文の反映内容の量・粒度・実装レベルへの裁量介入は対象外とする
- 背景方針の推定: ユーザーが背景方針を直接表現していない場合でも、
  フィードバック群の傾向から推定して判定軸に含める。
  規範緩和に関するフィードバックが複数同時に提示された場合は、
  採否判定前に`AskUserQuestion`でユーザー方針確認を発行してよい
- 汎用観点の判定: `agent-toolkit:critical-thinking`スキル「批判的検討の汎用観点」節
 （既存記述との重複・矛盾、コンテキストコスト対効果、適用範囲との整合、論理整合性）に従い判定する
- 振り返り由来フィードバックの再帰検討:
  frontmatterに`source: session-review`を持つフィードバックは、生成元スキル側の改善案も併記検討する対象に含める。
  提案内容またはその根拠に問題が見つかった場合、`agent-toolkit:session-review`スキル本文の修正案を併せて提示する
- 仕様不明を理由とした不採用判定時の検証義務:
  反映先設定ファイル・CLIなどの当該仕様が不明であることを理由に不採用判定する場合は、まず実機検証する。
  検証手段は対象CLIのhelp出力・サブコマンド実機実行・対象設定ファイルの現在値などとする。
  検証手段が思い当たらない場合は不採用確定せず「未検証」として保留し、ユーザーへ照会する
- 複数要求の分解判定:
  フィードバック原文に複数の要求が並列で含まれる場合は原文を要求単位に分解する。
  並列要求の例として例示と方向性の両立、具体例と一般原則の両立、特定対応と全体評価の両立などがある。
  分解後、採否を**採用／部分採用／不採用**で判定する。
  部分採用と判定した場合は採用する要求と除外する要求を明示し、ステップ4の提示書式の採用範囲・除外範囲欄に転記する
- 不採用根拠の限定:
  不採用根拠は技術的不成立・公開インターフェース破壊・破壊的操作・
  既存記述または既存実装で同等内容が既にカバー済みに限定する。
  完遂困難・規模・スコープ整理・集約難易度・コミット履歴での過去再現有無などを不採用根拠としない
  例外1: 観察事象の原因が既存規範の文面解釈ミスに帰せられる場合は不採用判定の対象外とし、当該規範の文面明確化案へ再構成する。
  例外2: 判断軸`(b)`該当の提案はメリット・デメリット観点で不採用としてよい
  - 「既存記述または既存実装で同等内容が既にカバー済み」を不採用根拠とする場合、
    当該既存記述・既存実装が現実の対象範囲でどの程度実効性を持つかを実測する。
    実測手段は既存機械チェックの対象範囲実行・サブエージェントの起動記録の網羅性照合・`grep`実測での対象件数計上などとする。
    実測結果はステップ4の提示書式「理由」欄へ具体数値・観測事実として転記する。
    実効性が対象範囲の一部にとどまる場合は「採用」へ再判定する
- 採用時の反映内容の縮小禁止:
  原文の反映内容を「過剰」「実効性の割にコスト高」「実装コストが見合わない」等の自己判断で縮小した
  「採用（縮退版）」「部分採用（過剰部分除外）」は禁止する。
  原文の反映内容が過剰と判断した場合も、原文どおり採用するか不採用とする二択とする。
  既存の「部分採用」語彙は複数の独立要求が並列に含まれる場合の要求単位分割にのみ用いる
- 自己整合性の検査: 不採用根拠に規範を挙げた場合、以降の実装・応答で当該規範への違反が発生していないか点検する
  （実施要件と撤回操作の詳細は横断節「自己整合性の検査（ステップ3〜5および事後レビュー横断）」に集約）

## ステップ5: 計画作成と実行

`agent-toolkit:plan-mode`スキルに従い計画を作成して実行する。
plan-modeスキル本文で規定される全工程を遵守する。
auto mode下・自律実行系CLI経由の委譲下でも本規定を維持する。
多段委譲経路（`process-feedbacks`経由等）でも本規定を維持する。
各工程の実施義務は`plan-mode/SKILL.md`
「暗黙スキップ・明示バイパスの禁止」節の規定に従う。
plan mode下の場合の最終承認は`agent-toolkit:plan-mode`工程8の`ExitPlanMode`で得る。
全件採用扱いで計画作成へ進んだ後に採否判定を変更したい場合は、ユーザー側の割り込みで提示する前提とする。

## 自己整合性の検査（ステップ3〜5および事後レビュー横断）

不採用根拠として明示された規範が存在する場合、メインエージェントは点検を実施する。
実施時点はステップ3の批判的検討時、ステップ5「計画作成と実行」の実装段階、および事後のレビュー・進捗ログ確認時とする。
応答テキスト・思考メモ・工程遷移の報告等で当該規範への違反行為が発生していないか確認する。
違反発生時は矛盾指摘として扱い、当該判断を直ちに撤回する。
撤回操作は当該フィードバックの採否再判定と反映内容の是正、
または当該規範への違反行為の即時中止を指す。
