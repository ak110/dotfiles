# フィードバックの批判的検討と適用

## 基本方針

各採否判定の最優先基準は、当該フィードバックの適用が成果物および今後の作業の品質向上に寄与するかとする。
手順を機械的に踏襲して採否を確定しない。
形式整合のみを根拠とした採用・不採用判定など、品質寄与の評価を伴わない判定を避ける。
不採用根拠の限定はステップ3の規定に従う。

## ステップ1: 入力の確認

ユーザーのメッセージからフィードバック項目を抽出する。
フィードバックが含まれていない場合はユーザーに提供を求める。

## ステップ2: 内容の調整

フィードバックの記述にとらわれず、各提案の追記先と抽象度を最適化する。
適用順は次のとおりとする。最初に「主提案の確定」で主提案を決め、確定した主提案に対して
「追記先の検討」「抽象度の調整」「規範側の再発予防案の併記検討」を順に適用する。

- 主提案の確定: 提案がリカバリー手順の文書化・
  観測主体（レビュアー・lint・型チェック・hook等）の誤検出への適用除外コメント・前提明示の追記・
  例外規定の新設など場当たり対処に該当する場合の判定軸とする。
  この場合、当該事象を発生させない根本対応（コード修正・設定変更・
  自動化スクリプトの整備・CI制約の追加・誤検出を生む規範側の改訂や撤廃など）が
  成立しないかを最初に検討する。
  特に誤検出由来の事象では「誤検出を生む規範側の改訂や撤廃」を最初に検討する。
  根本対応が成立する場合は根本対応案を主提案とし、元の追記提案は不採用扱いとする。
  成立しない場合は元の追記提案をそのまま主提案とする
  - 根本対応案を表層レベルの整理（内部重複の撤去のみ・既存実装の再利用のみなど）に限定した場合、
   `agent-toolkit:coding-standards`「責務帰属（SRP）分析の観点」節に従い判定確定前に分析する
- 追記先の検討: 提案が示す対象ファイルにとらわれず、最適な追記先を選ぶ
  コードに近い側（docstring・インラインコメント）や既存の関連セクションも候補に含める。
  フィードバックが既存追記そのものを否定する内容のときは、追記先候補の検討と並行して
  当該既存記述を削除候補に含める
- 抽象度の調整: 特定セッションの固有事象に特化しすぎている場合は一般化する。
  逆に抽象化しすぎて実効性を欠く場合は具体化する。
  事前にユーザー提示素材の原文の主旨を再確認し、批判的検討の前提となる解釈に誤読が無いことを確認する。
  既存節への新ケース追加・列挙拡張の判定基準は`agent-toolkit:agent-standards`「既知情報・冗長記述の排除」節に従う
- 規範側の再発予防案の併記検討:
  事象の根本原因が既存規範文書の曖昧さに帰せられる場合は規範への改善案も提案候補に含める
 （観点の詳細は`agent-toolkit:session-review`スキルの「規範改善案の併記」項目に従う）
- 既存SSOT規範との衝突回避:
  フィードバック原文の改善方針案を機械的に反映する前に、既存SSOT規範との衝突有無を点検する。
  対象は`agent-toolkit:review-standards`の重大度ラベル規定・
  `agent-toolkit:writing-standards`のMarkdown記述スタイル規定などを含む。
  点検時は対象SSOT規範文書の関連節
 （追加する規範が言及する具体ツール・観測値・実行方法を扱う節）をReadで原文確認してから提案本文を確定する。
  衝突する場合は既存SSOTを維持したうえで衝突を最小化する反映案へ書き換えてから提案する。
  代替案の選択肢は参照誘導の強化・列挙拡張の代替としての上位指針への昇格・既存記述の表現強化などを含む

## ステップ2.5: 採否判定前の網羅調査

ステップ3の批判的検討に入る前に、各提案の追記先となる既存記述と関連スキルを横断調査する。
調査は提案ごとに次の各点を確定する。

- 追記先候補ファイルの該当節を読み、既存記述で同等内容が既にカバーされていないかを確認する
- 関連スキル・関連ルールファイルに重複・矛盾する規範が無いかを確認する
- 追記先候補が配置先スキルの管轄範囲に該当する場合、当該配置先スキルをSkillツールで起動する
  - 管轄範囲はプロジェクト直下`AGENTS.md`・`CLAUDE.md`で編集対象ディレクトリに対して指定された範囲とする
  - 配布物境界・version bumpの基準・固有名の混入抑止などの配置先固有の規範を採否判定段階で適用する
- 提案の主旨を原文転記から再確認し、自身の解釈誤りが無いかを確認する
- 追記後の最終形に近い文面を組み立てて`wc -l`で実測し、`agent-toolkit:agent-standards`「文書サイズ上限」節の規定に従う
- フィードバック原文の取得・解釈が完遂できない場合は、規範改訂・代替反映の判定より前に、以下のいずれかで進める
  - 該当場面には外部URLの取得失敗・原文内の引用素材の参照不能・前提仕様の確定不能などを含む
  - 原文が問いかけ調・抽象的で具体反映内容を一意に確定できない場合は、原文の主旨から想定される具体化候補を2〜4案生成する
  - 生成した候補は`AskUserQuestion`で提示してユーザーに選ばせる
  - 多肢併採が想定される場合は`multiSelect: true`を許容する
  - 候補を生成できる場合は同一ターン内で候補提示まで進め、ユーザー往復回数を最小化する
  - 候補生成が不可能と判断した場合に限り、作業を中断してユーザーへ取扱（保留・不採用・原文貼付要求のいずれか）を確認する

調査範囲が広い場合は`Explore`サブエージェントへ委譲する。
調査結果を踏まえて初めてステップ3の批判的検討へ移行する。
調査未了のまま採否を断定しない。

## ステップ3: 批判的検討

調整後の内容で、各提案が対応に値するかを評価する。

- 背景方針の推定: ユーザーが背景方針を直接表現していない場合でも、
  フィードバック群の傾向から推定して判定軸に含める。
  規範緩和に関するフィードバックが複数同時に提示された場合は、
  採否判定前に`AskUserQuestion`でユーザー方針確認を発行してよい
- 汎用観点の判定: `agent-toolkit:critical-thinking`スキル「批判的検討の汎用観点」節
 （既存記述との重複・矛盾、コンテキストコスト対効果、適用範囲との整合、論理整合性）に従い判定する
- 振り返り由来フィードバックの再帰検討:
  frontmatterに`source: session-review`を持つフィードバックは、生成元スキル側の改善案も併記検討する対象に含める。
  提案内容またはその根拠に問題が見つかった場合、`agent-toolkit:session-review`スキル本文の修正案を併せて提示する
- 仕様不明を理由とした不採用判定時の検証義務:
  反映先設定ファイル・CLIなどの当該仕様が不明であることを理由に不採用判定する場合は、まず実機検証する。
  検証手段は対象CLIのhelp出力・サブコマンド実機実行・対象設定ファイルの現在値などとする。
  検証手段が思い当たらない場合は不採用確定せず「未検証」として保留し、ユーザーへ照会する
- 複数要求の分解判定:
  フィードバック原文に複数の要求が並列で含まれる場合は原文を要求単位に分解する。
  並列要求の例として例示と方向性の両立、具体例と一般原則の両立、特定対応と全体評価の両立などがある。
  分解後、採否を**採用／部分採用／不採用**の3区分で判定する。
  部分採用と判定した場合は採用する要求と除外する要求を明示し、ステップ4の提示書式の採用範囲・除外範囲欄に転記する
- 不採用根拠の限定:
  不採用根拠は技術的不成立・公開インターフェース破壊・破壊的操作・既存記述で同等内容が既にカバー済みの4種に限定する。
  完遂困難・規模・スコープ整理・集約難易度・コミット履歴での過去再現有無などを不採用根拠としない
  例外: 観察事象の原因が既存規範の文面解釈ミスに帰せられる場合は不採用判定の対象外とし、
  当該規範の文面明確化案へ再構成して提案する

## ステップ4: 検討結果の提示と適用

各提案の採否判定と理由をユーザーに提示する。
フィードバック1件ごとに`### <ファイル名>: <提案要約>`の見出しを設け、
配下に箇条書きで内訳を並べる。
採用提案は「採否」「理由」「反映先」「反映内容」、
部分採用提案は「採否」「理由」「採用範囲」「除外範囲」「反映先」「反映内容」、
不採用提案は「採否」のみ（採否欄に不採用理由を併記する）とする。

```markdown
## 検討結果

### <ファイル名>: <提案要約>

- 採否: 採用
- 理由: <採用する根拠>
- 反映先: <ファイル＋節または該当箇所>
- 反映内容: <反映する具体的な内容>

### <ファイル名>: <提案要約>

- 採否: 部分採用
- 理由: <採用する範囲の根拠>
- 採用範囲: <採用する要求の列挙>
- 除外範囲: <除外する要求と除外理由>
- 反映先: <ファイル＋節または該当箇所>
- 反映内容: <採用範囲のみの反映内容>

### <ファイル名>: <提案要約>

- 採否: 不採用。<不採用理由>
```

ファイル名は呼び出し元エージェントが結合したmarkdown内へ付与した`## <filename>`見出しの値をそのまま用いる。

見出し直下は箇条書き項目のみとし、項目外の地の文を追加しない。
補足は全て箇条書き項目内へ統合する。

不採用判定も部分採用の除外要求も含まれない（全件採用）場合は、ユーザー応答を待たず続けて計画作成へ移行する。
不採用判定または部分採用の除外要求が1件以上含まれる場合は`AskUserQuestion`を発行する。
発行対象は、ステップ3「不採用根拠の限定」に該当する根拠を持つものに限定する。
規模・作業量・対象件数・スコープ整理を根拠とする不採用・除外は、
`agent-toolkit:agent-standards`配下`scope-escalation-phrases.md`の縮退誘発カテゴリに該当する判定として扱う。
当該根拠を持つ判定はステップ3で除外され、本ステップの発行対象に含めない。
当該1件ごとに発行し、不採用は採否確認、部分採用は除外要求の妥当性確認をユーザーから得る。

- 確認は不採用判定の見落とし防止が目的のため、フィードバック1件ごとの個別承認とする
- 1問あたり最大4件の上限を超える場合は複数問へ分割する
- ユーザーが「採用へ転換」を選んだ場合の処理
  - 当該フィードバックの採否を採用へ更新する
  - ステップ2・ステップ3の判定を再評価する
  - 本ステップの提示書式を更新する
- 採用判定後にレビュー指摘で追記先の変更が必要と判明した場合の扱い
  - 対象レビューは`agent-toolkit:plan-mode`工程5以降のcodexレビュー・
    `plan-integrity-checker`・`agent-toolkit:careful-review`が該当する
  - 不採用転換ではなく追記先変更として扱い、変更後の追記先・反映内容を計画ファイルの`## 変更履歴`へ記録する
  - ユーザーへの追加採否確認は不要とし、フィードバック原文の主旨が同等に達成されている限り採用扱いの後始末を継続してよい
- 計画ファイル`## 実行方法`での後始末コマンドの順序
  - 後始末は対象リポジトリのレビュー完遂・`git push`完了後に配置する。
    後始末コマンドがフィードバック管理側へcommit/pushを伴う場合、
    対象リポジトリ側がレビュー指摘で巻き戻ったときに
    フィードバック管理側だけが先行公開され整合性が崩れるため
  - 標準順序は実装→検証→`agent-toolkit:commit`→コミット→
    `agent-toolkit:careful-review`→`git push`→後始末（呼び出し元スキルが指定する具体コマンド）とする
- 全件の承認を得たうえで計画作成へ移行する

Claude Code側permission modeが`plan`でない場合は`EnterPlanMode`を呼ばず直接`agent-toolkit:plan-mode`スキルへ進む。
permission modeが`plan`の場合は`EnterPlanMode`でplan modeへ移行してから`agent-toolkit:plan-mode`スキルへ進む。
移行後は`agent-toolkit:plan-mode`スキルに従い計画を作成して実行する。
plan mode下の場合の最終承認は`agent-toolkit:plan-mode`工程8の`ExitPlanMode`で得る。
全件採用扱いで計画作成へ進んだ後に採否判定を変更したい場合は、ユーザー側の割り込みで提示する前提とする。
