# 計画分割時の並列実装委譲

`agent-toolkit:apply-feedback`ステップ4「計画分割・統合の判断基準」で2計画以上に分割した場合の
実装工程の並列化手順を定める。

## 委譲先と処理単位

- 委譲先ツール: `Agent`ツール
- サブエージェント種別: `subagent_type: claude`
- 起動形態: 個別foreground並列委譲
- 処理単位: 委譲先が`agent-toolkit:plan-impl`工程まで完遂する
- 起動プロンプト: `agent-toolkit:plan-mode`工程6で作成した計画ファイルパスを埋め込む
- 1計画に統合した場合: 並列委譲せずメイン側で順次実装する

## 合流点

`git push`と後始末（`dotfiles-fb adopt`または`dotfiles-fb reject`）は、
全計画完遂後にメイン側で一括実施する。
呼び出し元スキルの後始末順序を維持する。

## 判断基準

- 採用フィードバック群の変更対象ファイル集合が分離可能な塊を2つ以上持つ場合は分割する。
  ただし各塊の変更対象ファイル数が実効性ある単位に満たない塊は他の塊へ統合する
- 各計画あたり変更対象ファイル数は3件以上とし、3件未満の塊は他の塊へ統合する
- 分割目的は計画レビュー・実装の並列化による時間効率化に限り、品質は分割・統合いずれでも同等とする
- hookスクリプト新設・セッション状態ファイル項目追加・作業ツリー競合を起こす改修などを含むフィードバックは、
  他のフィードバックと同一計画へ混載せず、単独計画として先行分離する
  - 単独計画として分離する対象の例:
    PreToolUse・PostToolUse・Stop等のhookスクリプトの新規追加または既存hook判定分岐の大幅な変更、
    セッション状態ファイルの項目追加を伴うhook間連携
- フィードバックが複数種別（テキスト追記系・hook実装系・構造改訂系など）を含む場合は、
  種別別に事前分類してから計画数を確定する
- コミットはユーザー明示指示なき限り1計画1コミットへ統合する
- 分割・統合は技術判断としてエージェント自律で決定し、`AskUserQuestion`は発行しない
- 分割は同一セッション内での計画ファイル分離を指す
  - `agent-toolkit/rules/01-agent.md`「セッション分割・別計画化は禁止する」節の対象外
