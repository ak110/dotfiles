# フィードバックの独立計画分離

以下のいずれかに該当する場合、単一および複数フィードバック横断のいずれでも、
他フィードバックから分離した独立計画として扱う。
本ファイルは`agent-toolkit:apply-feedback`ステップ4での分離規定を定める。

## 判断基準

- 単一フィードバックが対象50ファイル以上の大規模な一括処理を要求する場合は、
  当該フィードバックを他フィードバックから分離した独立計画として扱う
- 複数フィードバックを1計画へ統合する場合、フィードバック件数が5件を超え、
  かつ変更対象ファイル集合が分離可能な塊を2つ以上持つ場合は複数計画への分割を必須とする。
  密結合な複数フィードバックはエージェント判断でまとめてよく、
  その場合は`### エージェント判断`へ根拠を1行で明示する
- 境界近接ファイル（`agent-toolkit:agent-standards`「文書サイズ上限」節が定める200〜219行）への
  追記または縮減が同一ファイル内で2箇所以上を含む場合は複数計画への分割を必須とする。
  相殺算術の複雑度と、追記位置管理ミスによる相殺不成立を予防するためである。
  同一ファイル内で1箇所のみの場合は分割不要
  - 例外: `agent-toolkit:process-feedbacks`スキルが複数フィードバックを1回の
    `apply-feedback`呼び出しへ統合起動した場合は境界近接ファイル分割規定の対象外とする。
    フィードバック横断で同一ファイルが2回以上の編集対象になる状況が発生し得るため、
    `process-feedbacks`側SKILL.mdの「1度のapply-feedback呼び出しで処理する」規定と整合させる。
    統合起動時も縮減の同時実施で境界超過を回避する義務は維持し、
    同一計画内で境界超過を検知した場合は縮減対象を追加選定または追記文言を圧縮して許容幅内に収める
- hookスクリプト新設・セッション状態ファイル項目追加・作業ツリー競合を起こす改修などを含むフィードバックは、
  他のフィードバックと同一計画へ混載せず、単独計画として先行分離する
- 認可・認証・CSRF等のフレームワーク基盤層改修を含む要求は、独立計画として先行分離する。
  複合要求内に権限判定・ガード条件の変更を必要とする要素が含まれる場合、当該要素を独立計画として先行分離する
- 他フィードバック群を先行計画で完遂したうえで、独立計画で別途扱う
- 分離判定は`agent-toolkit:apply-feedback`ステップ4「検討結果の提示と適用」で行い、
  ステップ2.5「採否判定前の網羅調査」原則と両立する

## 委譲先と処理単位

- 委譲先ツール: `Agent`ツール
- サブエージェント種別: `subagent_type: claude`
- 起動形態: 個別foreground起動
- 処理単位: 委譲先が`agent-toolkit:plan-impl`工程まで完遂する
- 起動プロンプト: `agent-toolkit:plan-mode`工程6で作成した計画ファイルパスを埋め込む

## 合流点

`git push`と後始末（`atk fb adopt`または`atk fb reject`）は、
独立した計画の完遂後に呼び出し元スキル（`process-feedbacks`など）の後始末順序で実施する。

## その他

- コミットはユーザー明示指示なき限り1計画1コミットへ統合する
- 分割・統合は技術判断としてエージェント自律で決定し、`AskUserQuestion`は発行しない
- 分割は同一セッション内での計画ファイル分離を指し、
  `agent-toolkit/rules/01-agent.md`「セッション分割・別計画化は禁止する」節の対象外
