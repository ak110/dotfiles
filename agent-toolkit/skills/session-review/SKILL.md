# セッション振り返り

本スキルはdotfiles未利用・agent-toolkitのみ利用している環境向けの振り返りスキルとする。
dotfiles利用環境では拡張スキル`session-review-dotfiles`を必ず併用する
（章の置換範囲は`## ユーザードキュメント改善提案`節を参照）。

## 上位目的

本スキルの上位目的は次のセッション以降でのユーザー介入（指摘・修正指示・確認質問への応答介入など）を最小化することにある。
レビュー指摘・手戻り・確認応答のようにセッション内で反復して消費される是正コストを、
恒久対策で将来にわたり削減することも同目的に含む。
各ステップの判断はすべてこの上位目的に従属する。

## 起動方針

ユーザーから手動起動された場合は本節の終了判定をせず、そのままステップ1へ進む。

Stopフックから呼ばれた場合は、直前のアシスタントターンを読み返して
末尾が作業完了の言い切り（過去形・現在形）で締めくくられているかを判定する。
ユーザーへの質問・確認・承認待ちや、バックグラウンド待機・非同期処理待ちの表明を含む場合は、
作業完了の言い切りに該当しないものとして扱う。

Stopフック誘導かつ言い切り判定を満たす場合は、他工程
（plan-reviewer・codexレビュー・careful-review等を含む全工程）で処理済みとの理由で
ステップ1への進行を省略しない。

条件を満たす場合のみステップ1へ進み、満たさない場合は振り返りをしない。
質問・確認・承認待ちで終えていた場合は`AskUserQuestion`で当該の問いを再発行してターンを終え、それ以外は追加発話もせず即座にターンを終了する。

## ステップ1: Reflect

本ステップの目的は、恒久対策の種となる観測事象をセッション全体から漏れなく収集することにある。

本セッション中のユーザー入力は、タスク固有の事情を除き、その全てを恒久化候補として漏れなく点検対象に含める。
対象には明示の指示・割り込み・指摘・選好に加え、対話文脈で示された判断材料を含む。
「割り込みがあった場面」「明確な指摘があった場面」など個別契機へ矮小化せず、本セッションのユーザー入力を網羅的に再点検する。

候補を挙げる対象として、まず以下の観察源をセッションから洗い出す。

ユーザーからの割り込み・指摘・修正・確認質問は`agent-toolkit/rules/01-agent.md`「品質最優先」節
「ユーザーの指摘・割り込み・是正要求が発生した段階で」の項に従い観察する。
これに加え、セッション振り返り固有の観察対象として以下を洗い出す。

- Edit/Writeが期待通り反映されなかった事象
- hookでブロックされた事象
- `careful-review`等サブエージェント出力の統合表で不対応注記が付いた指摘
  - `（メイン判断: 不対応 — <理由>）`または`（ユーザー判断: 不対応 — <理由>）`形式の注記
- 計画ファイル`~/.claude/plans/`配下の`## 進捗ログ`章（`### レビュー反映`ラベル配下）に記録された指摘対応結果
- plan-reviewer・codexレビュー・careful-review等のレビュー工程で多数の指摘が発生した事象
  - 指摘の多発自体はレビュー工程の正常動作であり問題としない。
    着眼点は事前に防げたはずの同型指摘への対応を毎セッション繰り返す非効率にある
  - 指摘群を類型化し、少量の規範追加・既存規範の表現強化など追加のツール実行・工程追加を
    伴わない対策で事前に防げる類型がある場合は、当該対策を改善提案へ含める

そのうえで、判断ミスを防ぎ、ユーザー確認に頼らず自律判断できるようにするため、
どのような情報・規範が不足していたか確認する。

- プロジェクト固有の知見の不足
- コーディングエージェントが判断に迷った場面の存在
- CLAUDE.mdなどによるコーディングエージェントへの指示の不足・矛盾
- 誤った合理化・確認漏れ・推測による断定など、判断ミスを誘発した要因の存在

## ステップ2: Draft Additions

ステップ1の観察源と確認内容をもとに、プロジェクトドキュメントの改善候補（新規追加または既存記述の修正）を洗い出す。
本ステップの目的は、観測事象を将来のセッションでも効果が続く最小コストの恒久対策へ変換することにある。
個別事象への対症対処の列挙や、再発防止効果に見合わない恒常コンテキストの消費はこの目的に反する。

各候補は以下を自己検証し、すべて通過したものだけを提示する。

- 根本原因への遡及: 表層の現象ではなく根本原因まで遡って分析し、特定の再現条件に依存しない一般化した規範として提案する。
  詳細手順は[references/generation-criteria-detail.md](references/generation-criteria-detail.md)の該当節を参照する
- 生成基準（層1〜層3の順に適用し先に該当した層で確定する）: 層1はユーザー介入（明示的な指摘・修正指示・
  確認質問への応答介入）・致命的ミス（手戻り・破壊的結果・誤成果物確定）・後戻り・再設計要の計画ファイル
  不備を対象とする。単発でも優先採用し、事前検出する規範改訂を併記する（自動チェックで再検出される事象・
  局所修正で解消できる体裁・表記不備は実装段階の対応へ残す）。層1候補は「既存記述でカバー済み」を
  単独の不採用根拠にしない。介入へ至った経路を特定し、原因が文面の誤読・曖昧さ・他文書からの参照欠落と
  特定できる場合は文面修正・参照追加のみとする。文面が明確で参照もされたのに守られなかった場合は、
  同種の文面強化を既定とせず下記「対策手段の優先順位」の第2候補以降を第一候補として評価し、
  文面追加のみで終える場合は当該の別階層手段の不成立根拠を明記する。層2はコンテキスト効率改善・計画段階の
  摘出率強化系を効果大の根拠付きのみ候補化する対象とする。層3はその他の細かい単発ミスを対象とし、
  反復証跡（過去採用フィードバック・git履歴・同一セッション内の複数回発生のいずれか）を理由欄へ明示できない限り候補化しない
- 対策手段の優先順位: 同じ再発防止を実現できる手段のうち恒常コンテキストを消費しない手段から順に検討する
  （コーディング作業中のバグ・障害の再発防止策は本バレット全体に代えて
  `agent-toolkit/skills/plan-mode/references/bugfix-process.md`「再発防止策」節に従う）。
  詳細な候補順位は
  [references/generation-criteria-detail.md](references/generation-criteria-detail.md)の該当節を参照する
- 判断軸の順序・既存対処済みの確認・候補技術主張の実機検証・ユーザー過去方針の一次資料確認・方針固定化は
  `agent-toolkit/skills/agent-standards/references/feedback-review-common.md`の該当節を参照する（肥大化抑止の
  判定基準は`agent-toolkit:agent-standards`「既知情報・冗長記述の排除」節に従う）
- 既採用フィードバックとの意向照合・既存条文の制定経緯確認・肯定形優先チェック・推測表現の投入抑止・
  緩和側の検出は[references/generation-criteria-detail.md](references/generation-criteria-detail.md)の該当節を参照する
- 反映先の優先順位: 反映先はコードに近い順（docstring/インラインコメント → CLAUDE.md/.claude/rules → .claude/skills）で最適なものを選び、「書きやすい場所」に倒さない
- 規範改善案の併記: 詳細は`agent-toolkit/skills/agent-standards/references/feedback-review-common.md`「規範改善案の併記」節を参照する
- 自己検証通過の内部記録: 上記の各点の通過結果を候補ごとに`scratchpad`配下等の内部メモへ記録してから提案を確定する。判定根拠を未記録のまま提案へ進めない

## ステップ3: Show Proposed Changes

`## プロジェクトドキュメント改善提案`章は全環境で必須の章で、対象は現在作業中のプロジェクトの
`CLAUDE.md`・`README.md`・`docs/`配下のうち、本セッションで判明したプロジェクト固有の知見のみとする。
観点はbashコマンド・コードスタイル・テスト手法・環境設定・警告／陥穽・繰り返された修正指示・
コーディングエージェントの振る舞い改善（判断・確認・計画粒度等のルール化で再発防止できた場面）などとする。

`## ユーザードキュメント改善提案`章はdotfiles未利用環境のみで出力する。
対象は`~/.claude/`配下のルール・スキル・設定ファイル全般とする。
dotfiles利用環境では拡張スキル`session-review-dotfiles`の`## dotfiles改善提案`章に置換される。

ユーザーへ改善提案を以下のフォーマットで提示する。

````text
----

# 改善提案

[総括コメント]

----

## プロジェクトドキュメント改善提案

### [提案要約]

- 理由: [提案する根拠]
- 反映先: [ファイル＋節または該当箇所]
- 反映内容: [反映する具体的な内容]
- メリット: [採用時に得られる効果]
- デメリット: [採用時のコスト・副作用・運用上の負担]

----

## ユーザードキュメント改善提案

### [提案要約]

- 理由: [提案する根拠]
- 反映先: [ファイル＋節または該当箇所]
- 反映内容: [反映する具体的な内容]
- メリット: [採用時に得られる効果]
- デメリット: [採用時のコスト・副作用・運用上の負担]

----
````

各提案は `### [提案要約]` 見出し配下に「理由」「反映先」「反映内容」「メリット」「デメリット」を箇条書きで記述する。
メリット・デメリットは外部仕様レベルで記述し、実装識別子（特定ファイルパス・関数名・行番号など）は含めない。
ユーザーは外部仕様の採否判断者であり、実装細部を見ない前提とする。
提案が無い場合は同見出し配下に「提案無し」とのみ書く。
各提案文は本セッションの会話履歴を知らない読者が単独で実装に変換できる粒度で自己完結させる。

見出し直下は箇条書き項目のみとし、項目外の地の文を追加しない。
理由・補足は全て理由欄へ統合する。

- 含めるべき情報
  - 観測した具体事象（実行コマンド名・エラー文言・対象ファイルパス・誤った判断内容など）
  - 改善後の振る舞いとその根拠
- 禁止する暗黙参照表現
  - 「先ほどの〜」「上記の〜」「当該セッションで〜」「同種の〜」「その際の〜」など
  - 会話履歴を参照しなければ対象が確定しない表現
- 自己完結できる粒度に収まらない提案は分割するか、観測事象を具体化できない場合は提案そのものを除外する

## ステップ4: 反映の案内

拡張スキルが本ステップを上書きする場合は当該スキルの指示に従う。
上書きがない場合は次の手順で改善提案を投入する。

`atk fb status`を実行し、feedback-inboxが有効な場合のみ以下の投入手順を実行する。
非ゼロ終了時は改善提案の投入を省略し進捗ログのみ記録する。

ステップ3の改善提案本文をユーザー可視の応答として発行した直後、同一ターン内で
`atk fb add --source=session-review`を各改善提案について本文をそのまま連続実行する。
本文の範囲は`### <提案要約>`見出しから次の`###`直前までとし、要約・省略・書式変換をしない。引数形式は`agent-toolkit:add-feedback`スキルの`ステップ4: 投入`節を参照する。
引数へはtempファイルの生パス文字列をそのまま渡さない。本文をtempファイル経由で構築した場合は`$(cat /tmp/xxx.md)`のようにシェル展開で本文へ変換してから渡す。生のファイルパス文字列を位置引数として渡すと`atk fb add`のCLI側で拒否される（呼び出し側は本文文字列を位置引数として受け取る仕様のため）。
投入完了後は`agent-toolkit:exit-session`スキルへ遷移せず、通常の完了応答へ戻る。
「以上の改善提案を検討します」等の承認待ちと解釈される表現をステップ3応答末尾へ置かない。
