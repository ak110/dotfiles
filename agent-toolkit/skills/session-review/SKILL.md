# セッション振り返り

## 起動方針

ユーザーから手動起動された場合は本節の終了判定をせず、そのままステップ1へ進む。

Stopフックから呼ばれた場合は、直前のアシスタントターンを読み返して
末尾が作業完了の言い切り（過去形・現在形）で締めくくられているかを判定する。
ユーザーへの質問・確認待ちや、バックグラウンド待機・非同期処理待ちの表明を含む場合は、
作業完了の言い切りに該当しないものとして扱う。

条件を満たす場合のみステップ1へ進む。
満たさない場合は振り返りをせず、追加発話もせずに即座にターンを終了する。

## ステップ1: Reflect

本セッション中のユーザー入力は、タスク固有の事情を除き、その全てを恒久化候補として漏れなく点検対象に含める。
対象には明示の指示・割り込み・指摘・選好に加え、対話文脈で示された判断材料を含む。
「割り込みがあった場面」「明確な指摘があった場面」など個別契機へ矮小化せず、本セッションのユーザー入力を網羅的に再点検する。

候補を挙げる対象として、まず以下の観察源をセッションから洗い出す。

- ユーザーからの割り込み・指摘・修正
- ユーザーへの確認・質問
- Edit/Writeが期待通り反映されなかった事象
- hookでブロックされた事象

そのうえで、判断ミスを防ぎ、ユーザー確認に頼らず自律判断できるようにするため、
どのような情報・規範が不足していたか確認する。

- プロジェクト固有の知見の不足
- コーディングエージェントが判断に迷った場面の存在
- CLAUDE.mdなどによるコーディングエージェントへの指示の不足・矛盾
- 誤った合理化・確認漏れ・推測による断定など、判断ミスを誘発した要因の存在

## ステップ2: Draft Additions

ステップ1の観察源と確認内容をもとに、プロジェクトドキュメントの改善候補（新規追加または既存記述の修正）を洗い出す。

各候補は以下の6点を自己検証し、6点すべて通過したものだけを提示する。

- (a) 表層の現象ではなく根本原因まで遡って分析し、特定の再現条件に依存しない一般化した規範として提案する
  - 事象固有の識別子（ツール名・デコレーター名・設定名など）に依存した狭い規則ではなく、
    同じ根本原因から生じる類似事象を包含する上位の規範に一般化する
  - 抽象化階層を1段上で止めない。具体的には、複数階層がある場合は各階層で当該規範文書（スキル・ルール）の
    適用範囲と整合するかを順に照合する
- (b) 採否優先順位を以下の3段階で判定する
  - (b1) ユーザー介入（明示的な指摘・修正指示・確認質問への応答介入）を招いた事象は最優先で採用する
  - (b2) ExitPlanMode後・実装段階で計画ファイル不備として検出された事象は採用する。
    可能な限り計画段階で先回り検出する規範改訂を併記し、実現困難な場合は実装段階の検証で残してよい
  - (b3) セルフレビュー・機械チェック・サブエージェントのレビュー・コマンド実行失敗などで自動是正できる事象は、
    規範追加で消費する恒常コンテキスト量が事象再発時の自動是正で発生するトークン消費量を下回ると見積もれる場合のみ採用する
- (c) 既存のCLAUDE.md・ルール・スキルで対処済み、または同じ根本原因を部分的に扱う記述がないか確認する
  - 該当記述がある場合は新規追加でなく既存記述の修正・補強を優先し、細かい禁止事項の羅列による肥大化を避ける
  - 既存記述が存在するにもかかわらずセッション中に違反が発生した場合は「対処済み」と見なさず、
    既存記述の強化・明確化を提案対象とする
  - 対比表（Red flags／Gotchas形式）や禁止事項の羅列を新設する提案は行わない
    上位原則の補強として既存節へ統合する形に変換してから提案する
  - 既存節の列挙へ新ケースを追記する提案も、列挙の拡張ではなく上位指針への昇格・統合で
    吸収できないかを先に検討する
- (d) 反映先はコードに近い順（docstring/インラインコメント → CLAUDE.md/.claude/rules → .claude/skills）で
  最適なものを選び、「書きやすい場所」に倒さない
- (e) 規範の曖昧条件が原因の場合は規範改善案を併記する: 事象の根本原因が既存規範文書の
  適用範囲の明示漏れ・条件記述漏れに帰せられる場合、当該規範への改善案を併記する。
  対症対処のみを提案候補とせず、規範側の明確化で同種事象の再発を予防する案を併記する
- (f) 事象自体を発生させない根本対応を最優先する:
  対象事象を発生させない根本対応（コード修正・設定変更・自動化スクリプトの整備・
  CI制約の追加など）が成立する場合の優先順位を定める。
  この場合は、規範文の追加・ドキュメント追記・構造改善より根本対応案を提案として優先する。
  観測事象がワークアラウンドの文書化や運用ノウハウの蓄積にしか結びつかない場合は、
  当該の文書追記提案を取り下げ、代替として根本対応案そのものを改善提案へ列挙する。
  根本対応が成立しない場合に限り、対象資産の構造改善（セクション名の改名・命名統一・
  工程順序の再編・必須経路化など）で根本対策できないかを次に検討する。
  構造改善も成立しない場合のみ、規範文の追加を提案する。
  規範文の追加はコンテキストを恒常的に消費するため、コード・設定側または構造側の誘導で
  同種事象を予防できる場面では当該誘導を主提案とする

## ステップ3: Show Proposed Changes

ユーザーへ改善提案を以下のフォーマットで提示する。

```markdown
----

## プロジェクトドキュメント改善提案

- 対象ファイル — 提案内容
- 対象ファイル — 提案内容
- ...

----
```

各項目はセクション見出し配下に「- 対象ファイル — 提案内容」の形で1項目1行・1コンセプトで簡潔に書く。
提案が無い場合は同見出し配下に「提案無し」とのみ書く。

各提案文は本セッションの会話履歴を知らない読者が単独で実装に変換できる粒度で自己完結させる。

- 含めるべき情報
  - 観測した具体事象（実行コマンド名・エラー文言・対象ファイルパス・誤った判断内容など）
  - 改善後の振る舞いとその根拠
- 禁止する暗黙参照表現
  - 「先ほどの〜」「上記の〜」「当該セッションで〜」「同種の〜」「その際の〜」など
  - 会話履歴を参照しなければ対象が確定しない表現
- 自己完結できる粒度に収まらない提案は分割するか、観測事象を具体化できない場合は提案そのものを除外する

### プロジェクトドキュメント

CLAUDE.md、README.md、docs/配下など、プロジェクト固有のドキュメント全般。

今後の作業に必要な、本セッションで判明したプロジェクト固有の知見のみを対象とする。
観点はbashコマンド・コードスタイル・テスト手法・環境設定の傾向・警告／陥穽・繰り返された修正指示・
コーディングエージェントの振る舞い改善（判断・確認・計画粒度等のルール化で再発防止できた場面）など。

## ステップ4: 反映の案内

改善提案の反映には`agent-toolkit:apply-feedback`スキルを使う旨をユーザーに案内する。
本スキルでは適用しない。
