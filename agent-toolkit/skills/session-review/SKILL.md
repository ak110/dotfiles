# セッション振り返り

## 上位目的

本スキルの上位目的は次のセッション以降でのユーザー介入（指摘・修正指示・確認質問への応答介入など）を最小化することにある。
各ステップの判断はすべてこの上位目的に従属する。

## 起動方針

ユーザーから手動起動された場合は本節の終了判定をせず、そのままステップ1へ進む。

Stopフックから呼ばれた場合は、直前のアシスタントターンを読み返して
末尾が作業完了の言い切り（過去形・現在形）で締めくくられているかを判定する。
ユーザーへの質問・確認待ちや、バックグラウンド待機・非同期処理待ちの表明を含む場合は、
作業完了の言い切りに該当しないものとして扱う。

Stopフック誘導かつ言い切り判定を満たす場合は、他工程
（plan-integrity-checker・codexレビュー・careful-review等を含む全工程）で処理済みとの理由で
ステップ1への進行を省略しない。

条件を満たす場合のみステップ1へ進む。
満たさない場合は振り返りをせず、追加発話もせずに即座にターンを終了する。

## ステップ1: Reflect

本セッション中のユーザー入力は、タスク固有の事情を除き、その全てを恒久化候補として漏れなく点検対象に含める。
対象には明示の指示・割り込み・指摘・選好に加え、対話文脈で示された判断材料を含む。
「割り込みがあった場面」「明確な指摘があった場面」など個別契機へ矮小化せず、本セッションのユーザー入力を網羅的に再点検する。

候補を挙げる対象として、まず以下の観察源をセッションから洗い出す。

ユーザーからの割り込み・指摘・修正・確認質問は`agent-toolkit:agent.md`「品質最優先」節
「ユーザーの指摘・割り込み・是正要求が発生した段階で」の項に従い観察する。
これに加え、セッション振り返り固有の観察対象として以下を洗い出す。

- Edit/Writeが期待通り反映されなかった事象
- hookでブロックされた事象

そのうえで、判断ミスを防ぎ、ユーザー確認に頼らず自律判断できるようにするため、
どのような情報・規範が不足していたか確認する。

- プロジェクト固有の知見の不足
- コーディングエージェントが判断に迷った場面の存在
- CLAUDE.mdなどによるコーディングエージェントへの指示の不足・矛盾
- 誤った合理化・確認漏れ・推測による断定など、判断ミスを誘発した要因の存在

## ステップ2: Draft Additions

ステップ1の観察源と確認内容をもとに、プロジェクトドキュメントの改善候補（新規追加または既存記述の修正）を洗い出す。

各候補は以下を自己検証し、すべて通過したものだけを提示する。

- 根本原因への遡及: 表層の現象ではなく根本原因まで遡って分析し、特定の再現条件に依存しない一般化した規範として提案する
  - 事前調査の徹底: 候補事象に関連する既存規範文書の関連節を網羅的にReadしてから抽象化階層を確定する
    - 対象は`01-agent.md`・`02-claude-code.md`・該当作業のスキル本体・対応する`references/`配下を含む
    - 当該事象を特殊ケースとして包含する既存上位節があるかを先に確認する
    - ある場合は新規追加ではなく既存節の表現強化・対象類型追加・例示拡張へ変換してから提案する
    - 観測事象が実装挙動の異常（フック発火漏れ・状態フラグ立て漏れ・非同期処理の判定誤り・
      スラッシュコマンド起動時のツール呼び出し扱いなど）に該当する場合は、
      当該セッション内で実装レベルの調査を完遂してから提案を書く
      - 調査対象はセッション状態ファイル`{tempdir}/claude-agent-toolkit-{session_id}.json`のダンプと
        transcript走査とする。加えて実装コードのReadによるロジック追跡・条件フラグ状態確認・ログ出力可否確認も含む
      - 抽象的な「再点検する」「発火条件を確認する」等の調査提案のまま投入しない
      - 原因が実装外（Claude Code側の仕様など）に帰せられる場合は、
        その旨と依拠範囲・代替回避策を提案内に併記する
  - 事象固有の識別子（ツール名・デコレーター名・設定名など）に依存した狭い規則ではなく、
    同じ根本原因から生じる類似事象を包含する上位の規範に一般化する
  - 抽象化階層を1段上で止めない。具体的には、複数階層がある場合は各階層で当該規範文書（スキル・ルール）の
    適用範囲と整合するかを順に照合する
    - 提案ラベル・本文確定前に、提案中の固有名詞
     （個別ツール名・関数名・設定キー名・実装パターン名・言語名など）を1つずつ除外しても
      規範として成立するかを点検する
    - 成立する場合は当該固有名詞を「具体例」欄へ分離し、規範本体は固有名詞を含まない抽象表現に置く
    - 固有名詞を除外すると意味が崩れる場合は当該名詞のさらに上位カテゴリを試し、
      規範として通用する最上位カテゴリへ昇格させてから本文を確定する
    - 上位カテゴリの試行例: 個別プラグイン名 → 個別ツール種別 → 外部依存物
- 採否優先順位の判定: 以下の3段階で判定する
  - ユーザー介入優先: ユーザー介入（明示的な指摘・修正指示・確認質問への応答介入）を招いた事象は最優先で採用する
  - 計画段階での事前検出: ExitPlanMode後・実装段階で計画ファイル不備として検出された事象は採用する。
    計画段階で事前検出する規範改訂を併記する
   （実装段階の自動チェックやサブエージェントレビューで再検出されることを
    事前確認できた事象に限り当該実装段階の検証へ残してよい）
  - コスト比較判定: セルフレビュー・機械チェック・サブエージェントのレビュー・コマンド実行失敗などで
    自動是正できる事象を扱う。規範追加より先に、次の代替手段の可否を優先検討する
    - Hook（PreToolUse・PostToolUse・Stopなど）で機械的に検出・阻止可能か
    - サブエージェント・機械チェック（textlint・pyfltr等）で自動是正可能か
    - 既存Hook・既存機械チェックの検出範囲拡張で吸収可能か
    代替手段が成立する場合は当該実装案を主提案とし規範追記案は取り下げる。
    代替手段が成立せず、かつ規範追加で消費する恒常コンテキスト量が事象再発時の
    自動是正で発生するトークン消費量を下回ると見積もれる場合のみ規範追加を採用する。
    限定場面にしか適用されない規範追記は取り下げる
- 既存対処済みの確認: 既存のCLAUDE.md・ルール・スキルで対処済み、または同じ根本原因を部分的に扱う記述がないか確認する
  - 該当記述があり機能している場合は新規追加ではなく既存記述の表現強化・条件追加・例示拡張・配置見直しを優先する
  - 既存記述が存在しても違反が発生した場合は「対処済み」と見なさず、機能していない兆候として既存記述の強化を提案する
  - 観察事象の原因が既存規範の文面解釈ミスに帰せられる場合は対処済み扱いとせず、当該規範の文面明確化案を提案候補に含める
  - 列挙拡張・対比表新設・禁止事項羅列での肥大化抑止の判定基準は
    `agent-toolkit:agent-standards`「既知情報・冗長記述の排除」節に従う
- 反映先の優先順位: 反映先はコードに近い順（docstring/インラインコメント → CLAUDE.md/.claude/rules → .claude/skills）で
  最適なものを選び、「書きやすい場所」に倒さない
- 規範改善案の併記: 規範の曖昧条件が原因の場合は規範改善案を併記する。
  事象の根本原因が既存規範文書の適用範囲の明示漏れ・条件記述漏れに帰せられる場合、当該規範への改善案を併記する。
  対症対処のみを提案候補とせず、規範側の明確化で同種事象の再発を予防する案を併記する
- 根本対応の最優先:
  対象事象を発生させない根本対応（コード修正・設定変更・自動化スクリプトの整備・
  CI制約の追加など）が成立する場合の優先順位を定める。
  この場合は、規範文の追加・ドキュメント追記・構造改善より根本対応案を提案として優先する。
  観測事象がワークアラウンドの文書化や運用ノウハウの蓄積にしか結びつかない場合は、
  当該の文書追記提案を取り下げ、代替として根本対応案そのものを改善提案へ列挙する。
  根本対応が成立しない場合に限り、対象資産の構造改善（セクション名の改名・命名統一・
  工程順序の再編・必須経路化など）で根本対策できないかを次に検討する。
  構造改善も成立しない場合のみ、規範文の追加を提案する。
  規範文の追加はコンテキストを恒常的に消費するため、コード・設定側または構造側の誘導で
  同種事象を予防できる場面では当該誘導を主提案とする
- 自己検証通過の内部記録: 上記の各点の通過結果を候補ごとに`scratchpad`配下等の内部メモへ
  記録してから提案を確定する。判定根拠を未記録のまま提案へ進めない

## ステップ3: Show Proposed Changes

ユーザーへ改善提案を以下のフォーマットで提示する。

````text
----

# 改善提案

[総括コメント]

----

## プロジェクトドキュメント改善提案

### [提案要約]

- 理由: [提案する根拠]
- 反映先: [ファイル＋節または該当箇所]
- 反映内容: [反映する具体的な内容]
- メリット: [採用時に得られる効果]
- デメリット: [採用時のコスト・副作用・運用上の負担]

----

## ユーザードキュメント改善提案

### [提案要約]

- 理由: [提案する根拠]
- 反映先: [ファイル＋節または該当箇所]
- 反映内容: [反映する具体的な内容]
- メリット: [採用時に得られる効果]
- デメリット: [採用時のコスト・副作用・運用上の負担]

----
````

各提案は `### [提案要約]` 見出し配下に「理由」「反映先」「反映内容」「メリット」「デメリット」を箇条書きで記述する。
メリット・デメリットは外部仕様レベルで記述し、実装識別子（特定ファイルパス・関数名・行番号など）は含めない。
ユーザーは外部仕様の採否判断者であり、実装細部を見ない前提とする。
提案が無い場合は同見出し配下に「提案無し」とのみ書く。
各提案文は本セッションの会話履歴を知らない読者が単独で実装に変換できる粒度で自己完結させる。

見出し直下は箇条書き項目のみとし、項目外の地の文を追加しない。
理由・補足は全て理由欄へ統合する。

- 含めるべき情報
  - 観測した具体事象（実行コマンド名・エラー文言・対象ファイルパス・誤った判断内容など）
  - 改善後の振る舞いとその根拠
- 禁止する暗黙参照表現
  - 「先ほどの〜」「上記の〜」「当該セッションで〜」「同種の〜」「その際の〜」など
  - 会話履歴を参照しなければ対象が確定しない表現
- 自己完結できる粒度に収まらない提案は分割するか、観測事象を具体化できない場合は提案そのものを除外する

### プロジェクトドキュメント

CLAUDE.md、README.md、docs/配下など、プロジェクト固有のドキュメント全般。

今後の作業に必要な、本セッションで判明したプロジェクト固有の知見のみを対象とする。
観点はbashコマンド・コードスタイル・テスト手法・環境設定の傾向・警告／陥穽・繰り返された修正指示・
コーディングエージェントの振る舞い改善（判断・確認・計画粒度等のルール化で再発防止できた場面）など。

### ユーザードキュメント

`~/.claude/`配下のルール・スキル・設定ファイル全般（プロジェクト横断で参照されるグローバルユーザー設定）。

## ステップ4: 反映の案内

拡張スキルが本ステップを上書きする場合は当該スキルの指示に従う。
上書きがない場合は改善提案の反映に`agent-toolkit:apply-feedback`スキルを使う旨を案内する。
