# ステップ2生成基準の詳細

`agent-toolkit/skills/session-review/SKILL.md`「ステップ2: Draft Additions」自己検証チェックリストの
詳細手順を集約する。対象は「根本原因への遡及」「対策手段の優先順位」「既採用フィードバックとの意向照合」
「既存条文の制定経緯確認」「肯定形優先チェック」「推測表現の投入抑止」「緩和側の検出」とする。

## 根本原因への遡及

- 事前調査の徹底は`agent-toolkit/skills/agent-standards/references/feedback-review-common.md`「事前調査の徹底」節を参照する
- 事象固有の識別子（ツール名・デコレーター名・設定名など）に依存した狭い規則ではなく、
  同じ根本原因から生じる類似事象を包含する上位の規範に一般化する
- 抽象化階層を1段上で止めず、複数階層がある場合は各階層で当該規範文書の適用範囲と整合するかを順に照合する
  - 提案ラベル・本文確定前に、提案中の固有名詞を1つずつ除外しても規範として成立するかを点検する
  - 成立する場合は当該固有名詞を「具体例」欄へ分離し、規範本体は固有名詞を含まない抽象表現に置く
  - 意味が崩れる場合は上位カテゴリ（例: 個別プラグイン名 → 個別ツール種別 → 外部依存物）へ昇格させて確定する

## 対策手段の優先順位

同じ再発防止を実現できる手段のうち、恒常コンテキストを消費しない手段から順に検討する。
コーディング作業中のバグ・障害の再発防止策は、本節に代えて
`agent-toolkit/skills/plan-mode/references/bugfix-process.md`「再発防止策」節に従う。

- 第1候補: 事象を発生させない根本対応（コード修正・設定変更・自動化スクリプト整備・CI制約追加等）。
  ワークアラウンド文書化・運用ノウハウ蓄積にしか結びつかない場合は文書追記案を取り下げ、
  根本対応案そのものを改善提案へ列挙する
- 第2候補: 対象資産の構造改善（セクション名改名・命名統一・工程順序再編・必須経路化等）
- 第3候補: 機械的検出・自動是正（Hookでの検出・阻止、サブエージェント・機械チェック等での自動是正、既存検出範囲の拡張を含む）
- 規範文の追加は上記いずれも成立しない場合の最終手段とし、第1〜第3候補が成立しないことを提案の「理由」欄へ根拠付きで明示する
  （トークン消費量・コンテキスト残量の見積は判断根拠にしない）

## 既採用フィードバックとの意向照合

生成候補は対象target_repoの既採用フィードバック（adopted配下）で示された意向と衝突しないかを確認する。

- 衝突が確認された候補は不採用として扱い候補化しない
- 既採用意向を強化する形の候補へ組み替え可能な場合は組み替える
- 既採用フィードバックの参照は`atk fb show --include-processed <FILENAME>`で実施する

## 既存条文の制定経緯確認

既存規範文の改訂を提案する前に対象条文の`git log -p`による制定経緯と既存根拠段落を確認し、
提案文面が既存根拠を引き継ぐか根拠変更の理由を明記する。本規定は改訂候補全般を対象とする最も広い粒度で扱う。

- 姉妹規定`agent-toolkit:process-feedbacks`「既存文書き換え時の根拠段落の維持照合」は数値・閾値・境界の書き換えに限定する
- 姉妹規定`agent-toolkit:plan-mode`配下`references/norm-revision-checklist.md`の該当節は短縮・言い換え時の要素照合に限定する

## 肯定形優先チェック

禁止規定の追加を提案候補とする前に、同じ再発防止を「実施すべき正規フローの肯定的な記述」で表現できないか先に試みる。
肯定形で成立する場合は禁止形を採用せず、肯定規定が無い領域へ禁止列挙のみを追加する提案をしない
（詳細判定は`agent-toolkit:agent-standards`「肯定形優先の記述」節に従う）。

## 推測表現の投入抑止

原因記述に推測表現（「〜が疑われる」「〜の可能性がある」「原因候補」「と思われる」「調査すべき点」等）を
含む提案候補は投入前に原因を確定する。
確定手段は観測可能な診断ソース（常時ログ・セッション状態ファイル・transcript・対象スクリプトの実機照合など）とする。
バグ・障害・不具合報告が外部仕様・外部挙動に由来する場合、確定手段には一次資料
（公式ドキュメント・公式CHANGELOG等）照合も含める。
いずれの手段でも確定できない場合のみ調査依頼として投入し、未確定である旨と調査済み範囲を本文へ明記する。

## 緩和側の検出

既存規範の緩和・免除・例外条項の新設を伴う候補は、緩和対象規範の完遂義務が同一セッション内で
成立していたかを観測事実（実装差分・進捗ログ）で確認する。作業量・所要時間・残工程数を根拠にした縮退表明が
観測される場合は緩和側候補として採用せず対応する縮退防止側の規範強化候補を併記する。
