# 計画後の計画実行

`ExitPlanMode`後にコンテキストがクリアされた状態で呼び出される可能性がある。
計画ファイルが唯一の入力源として自立している必要がある。
サブエージェントへの委譲でも計画ファイルのパスを省略しない。

レビュー実施有無は計画ファイル側の`レビュー方式`項目で確定済みである前提で動作する。
本スキルでは当該項目を読み取って`レビュー無し`／`レビュー有り`に分岐する。

1. タスク分解
2. 実装
3. 検証
4. コミット
5. レビュー（`レビュー有り`の場合のみ）

フェーズ分けされた計画では「2. 実装」「3. 検証」「4. コミット」を各フェーズ分繰り返す。
前フェーズのコミット完了を確認してから次フェーズへ進む。
全フェーズ・全コミット完了後に「5. レビュー」を一度だけ実施する（`レビュー有り`の場合のみ）。

## サブエージェント起動方針

`completed`状態のサブエージェントへ`SendMessage`を送ると失敗する。
修正再実装・followupレビューなど追加指示が必要な場面はすべて新規`Agent`起動で行う。
新規起動のたびに計画ファイル位置・元タスク・修正指摘を省略せずプロンプトに含める。

### 実装サブエージェント委譲（plan-implementer）の判断指針

実装工程の判定は「`plan-implementer`を起動するか」と「起動するサブエージェントのモデル」の
2階層で行う。メインモデルとサブエージェントのモデルは独立軸として扱う。

第1階層 — `plan-implementer`を起動するか:

- 並列タスクが無い場合は委譲のオーバーヘッドのみ生じるため、メインモデルを問わずメインが直接実装する
- 独立した複数タスクを同時に進める場合のみ`plan-implementer`を並列起動する

第2階層 — 起動するサブエージェントのモデル:

- 既定はOpus
- `レビュー方式: レビュー有り`の計画はレビュー工程で品質を担保できるため、Sonnet以下で起動してもよい

### `run_in_background`既定方針

サブエージェント起動時の`run_in_background`は以下を既定とする。

- 並列起動以外はforeground（`run_in_background`未指定）
- 独立した複数タスクを同時に進める並列起動時のみbackground（`run_in_background=true`）

### 並列サブエージェント実行時の注意

並列起動時は以下を実施する。

- 起動前に各タスクの編集対象ファイル集合の重複を点検し、重複があればタスク境界を再設計する
- 各完了報告は`git diff`と必ず照合し、報告に含まれない差分や報告と実差分の不一致があれば内容を確認する
- 「他タスクの中間状態が原因」とする推定は鵜呑みにせず、
  メイン側で実状（他タスクの差分・追加・削除）を確認し過剰削除や誤修正に起因しないか判断する

## 1. タスク分解

計画ファイルを起点にタスク分解し、`TaskCreate`で登録する（検証・コミット・レビュータスクも含む）。
過度な細分化は避ける。ファイル削除（`rm`等）はサブエージェントへ委譲しない。
レビュータスクは観点ごとに別サブエージェントを起動する（実装の経緯を持ち込まないため）。

## 2. 実装

分解したタスクを実装する。
上記「実装サブエージェント委譲（plan-implementer）の判断指針」に従い、
メイン直接実装または`plan-implementer`への委譲を選ぶ。

```text
以下のタスクを{実装 | 修正再実装}してください。

- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`

## タスク

{タスクの具体的な記述}

## 修正指摘（修正再実装時のみ）

{レビュアー指摘テキストとメインが具体化した修正方針（対象ファイルパス・該当位置・修正方針）を
 まとめて記述する。具体化方針は欄内に統合し欄外に分散させない}
```

## 3. 検証

全実装タスク完了後に検証を実施する（実装タスクへの統合も可）。
検証コマンドは計画ファイルの「検証手順」の対象範囲全体を1コマンドで実行する。

## 4. コミット

計画ファイルの「コミット方針」に従ってコミットする。
全コミットが`HEAD`に反映されたら「5. レビュー」へ進む（`レビュー有り`の場合のみ）。

## 5. レビュー

`レビュー方式: レビュー有り`の計画でのみ実施する。`レビュー無し`の場合は本節をスキップして作業を終了する。

### 初回レビュー（2種別並列）

`plan-spec-reviewer`と`plan-impl-reviewer`を常に並列で起動する。
担当観点は種別ごとに異なるため、テンプレートの`{担当観点}`を下記の種別定義で差し替えてプロンプトを組み立てる。

- `spec`（`plan-spec-reviewer`）: 計画ファイルと成果物の仕様適合性。
  修正再実装を経た差分の場合は前回指摘文面が改変・要約・省略されずに反映されているかの照合を含む
- `impl`（`plan-impl-reviewer`）: コード単体品質・ドキュメント単体品質・成果物間の整合性

```text
以下の実装について初回モードでレビューを実施してください。

- 担当観点: {上記種別定義を参照}
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
- 対象範囲: 計画全体（全コミット完了後の差分）
- 対象外ファイル: {一時ファイル一覧}
- 差分取得: `git diff {基準点}..HEAD`と、未追跡ファイルを含む作業ツリー参照で行う
  - `{基準点}`は計画着手前コミットを指す不変参照点（`origin/master`相当の上流ブランチHEADまたは計画着手前SHA）を選ぶ。
    複数コミット計画でも基準点は変えない。
    amend/fixupを伴う場合はHEAD相対参照（`HEAD~1`等）が刻々と変わるため、コミットSHAを優先する

## 制約

- 指摘を断片的にせず、一度のレビューですべての問題点を洗い出すこと
```

### 指摘の統合と修正依頼

各レビュアーの戻り値を受け取りメインが指摘を統合する。
計画に照らして妥当な修正であれば確認無しで修正してよい。必要に応じてユーザーへ確認して対応方針を決定する。

対応不要と判断した指摘には行末尾に注記する。注記付き指摘は「修正指摘」欄に含めない。

- `（ユーザー判断: 不対応 — <理由>）`
- `（メイン判断: 不対応 — <理由>）`

対応が必要な指摘が1件以上あれば修正内容をユーザーに簡潔に報告したうえで、
修正再実装モードで`plan-implementer`へ委譲する。
修正の再実装が完了したら検証タスクを新規起動して合格を確認し、
直前の未プッシュコミットへ`git commit --amend`で統合する。
amend完了後は`git status`で作業ツリーが空であることを確認する。
pre-commit hookのstash処理で意図しない差分が残ることがあるため、取り残しを検出した場合は内容を確認して再amendする。

### 再レビュー（followupモード）

amend統合後、`plan-spec-reviewer`・`plan-impl-reviewer`を並列でfollowupモードとして起動する。

```text
以下の実装についてfollowupモードで再レビューを実施してください。

- 担当観点: {初回レビュー節の種別定義を参照。初回と同一の値を使う}
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
- 対象範囲: 計画全体（全コミット完了後の差分）
- 差分取得: `git diff {基準点}..HEAD`（基準点は初回レビューと同一）

## 前回指摘

{統合済み指摘テキストを行を抜かず転記する（注記付きの指摘も含む）}

## 制約

- 新規指摘は追加しないこと（初回レビューで網羅されている前提）
- 前回指摘ごとに`addressed`／`partial`／`missing`を判定すること
- 修正の副作用で生じた明らかな回帰のみ`regression`欄に記載すること
- 全件`addressed`の場合は短文応答で合格を伝えてよい（例: `全件addressed。回帰無し。`）
```

`partial`または`missing`が1件以上あれば修正対象として扱う。`regression`検出時も同様。
この状態が連続2回発生した場合は修正ループを中断してユーザーへエスカレーションする。

全レビュー合格で作業終了。
