# 計画後の計画実行

`ExitPlanMode`後にコンテキストがクリアされた状態で呼び出される可能性がある。計画ファイル自立性を前提とする。
計画ファイルの`## 実行方法`のレビューステップを読み取り、「レビューは実施しない」と記載されていればレビュー工程をスキップする。
それ以外は記載のスキル・エージェントへ引き継いでレビューを実施する。

`## 変更内容`に列挙された全項目を初回実装で完遂する（「初回必須」「将来拡張」など段階化ラベルを付与した先送りは禁止する）。

`ExitPlanMode`によるユーザー承認は、計画ファイル全文の精読を保証しない。
ユーザーは`## 背景`・`## 対応方針`を中心に確認したうえで承認している場合が多く、
`## 変更内容`・`### エージェント判断`の細部や周辺対応までは未読の可能性がある。
実装段階で計画ファイル本文に判断指針との衝突やオーバーエンジニアリング的な記述を発見した場合、
「計画ファイルに書いてある」「ユーザーが承認した」を理由に判断指針を後退させてはならない。
ここでの判断指針とは、`agent.md`の
「コーディングエージェント（あなた）の振る舞い」節（特に「品質最優先」「ユーザーとともに考える」）を指す。
ここでの衝突は、計画ファイル本文の明示記述と判断指針の明示記述が文面上矛盾する場合に限る。
自己の不安・疑念・主観的な違和感・自己推定上の制約は衝突に該当しない。
衝突を発見した時点で実装を止め、計画ファイル本文と判断指針のいずれが正かをユーザーへ確認する。

`ExitPlanMode`はユーザー承認応答の取得まで含めてワンステップで完結する。
承認応答を受領した時点で実装着手の合意が成立しているため、
「実装に進めてよいか」等の追加確認文を発さず、直ちに以下の工程1へ着手する。

以下の手順は省略不可の工程である。
手順中の規範は、agent.md「品質最優先」原則の具体例である。網羅規定ではないが、列挙外の場面でも原則に従う。

1. タスク分解
2. 実装
2.5. サブエージェント完了報告の検収
3. 検証
4. コミット
5. レビュー（レビューステップでレビュー実施が指定されている場合のみ。記載のスキル・エージェントへ引き継ぐ）

各フェーズの区切りの節目（1タスク完了・1コミット作成直前・レビュー指摘反映の節目など）ごとに、
計画ファイルの`## 進捗ログ`章へ`### <ラベル>`見出しと箇条書きで追記する。
書式詳細は`agent-toolkit:plan-mode`スキル配下の`references/plan-file-guidelines.md`の「進捗ログ」節に従う。
いずれのフェーズでも、計画ファイル本文（`## 変更内容`・`## 対応方針`等）の記載と乖離する確定形を採用した場合は、`## 進捗ログ`への追記だけでなく当該本文を最終確定形へ更新する。
適用対象にはレビュー指摘対応によるdiffコードブロック内文面案の意味的変更も含む。

実装着手前に計画ファイル本文の段取り記述（`## 対応方針`配下の`### 改訂作業の段取り`等）を読み返し、
「記録する」「反映する」等の進捗ログ追記要件を抽出してフェーズ末の記録対象に組み込む。
計画ファイル本文側で明示した個別の追記要件は、上記のフェーズ節目記録と並行して履行する。
計画ファイル本文の自立性が成立しているかの愚直読解は`agent-toolkit:plan-mode`スキル工程7で`naive-executor`サブエージェントが担当する。
実装段階の本スキルでは再実施しない。

## 1. タスク分解

計画ファイルを起点にタスク分解し、`TaskCreate`で登録する（検証・コミット・レビュータスクも含む）。
過度な細分化は避ける。ファイル削除（`rm`等）はサブエージェントへ委譲しない。
レビュータスクは観点ごとに別サブエージェントを起動する（実装の経緯を持ち込まないため）。

## 2. 実装

分解したタスクを下記「実装サブエージェント委譲（plan-implementer）の判断指針」に従って実装する。

### 実装サブエージェント委譲（plan-implementer）の判断指針

計画確定後の実装は方針が固まっているため、原則として`plan-implementer`への委譲を優先しメイン直接実装を限定する。
委譲可否は自己推定上のタスク規模・対象範囲・所要量で判断せず、タスク内容の確定度（修正方針・対象ファイル・期待結果の明確さ）で判断する。
タスクごとに次の手順で委譲可否を決める。

1. タスクが修正方針自体の判断（opus相当）を要するか見積もる
2. opus相当の判断を要するタスクのみメイン直接実装を選ぶ（品質担保のため）
3. それ以外は`plan-implementer`へ委譲する（既定はsonnet、機械的作業はhaikuを`model`引数で指定）
4. 委譲したタスクが`needs_escalation`を返したらメイン直接実装へ引き上げる
5. 委譲タスクのうち、編集対象ファイルが独立する2件以上は`run_in_background=true`で並列起動する

タスク内容の確定度に迷うタスクは手順3に従ってまず委譲する。

### サブエージェント起動プロンプト

```text
以下のタスクを{実装 | 修正再実装}してください。

- 計画ファイル: `{呼び出し元から渡された計画ファイルパス}`

## タスク

{タスクの具体的な記述}

## 修正指摘（修正再実装時のみ）

{レビュアー指摘テキストとメインが具体化した修正方針（対象ファイルパス・該当位置・修正方針）を
 まとめて記述する。具体化方針は欄内に統合し欄外に分散させない}
```

## 2.5 サブエージェント完了報告の検収

サブエージェント完了報告にunplanned項目が含まれる場合、次のレビューまで保留せずメインで即時に検収する。
unplanned項目とは即時相談候補・カテゴリ変更・追加変更など計画外で発生した判断を指す。

- 完了報告のchanged一覧が計画ファイル`## 変更内容`当該節の独立変更項目（複数行・複数識別子の更新指示等）を網羅するかを照合する
  - 1ファイル内に複数箇所の更新指示がある場合は一部のみ更新して報告するケースがあるため`git diff`で差分実体を確認する
- 計画方針および判断指針へ照らして即時採否判定する
- 採否を保留すべき事項のみユーザーへ確認する

## 3. 検証

全実装タスク完了後に検証する（実装タスクへの統合も可）。
検証コマンドは計画ファイルの`## 実行方法`の検証ステップの対象範囲全体。

## 4. コミット

計画ファイルの`## 実行方法`のコミットステップに従ってコミットする。

## 5. レビュー

計画ファイルの`## 実行方法`のレビューステップに記載されたスキル・エージェントへ引き継ぐ。
複数のスキル・エージェントが併記されている場合は並列起動する。
対象範囲は計画全体（全コミット完了後の差分）、計画ファイルパスは本計画のものを渡す。
不変参照点（`origin/<base-branch>`相当または計画着手前SHA）と一時ファイル一覧を併せて渡す。
以降の起動・指摘統合・修正再実装・amend統合・再レビューは引き継ぎ先の手順に従う。
