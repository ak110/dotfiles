# コーディングエージェント向け文書品質

コーディングエージェント向け文書固有の記述原則。
一般ドキュメント品質方針は`agent-toolkit:writing-standards`が担当し、本スキルと併用する。

## 読者像

対象文書はコーディングエージェントが直接読み込む。人間の読者を想定した説明文・前置き・装飾は書かない。

## 共通の記述原則

### 記述の簡潔さ

- 各行について「削除するとコーディングエージェントの判断が誤る」と言えなければ削除する
- 冗長な指示はトリガー精度の低下とコンテキスト消費の増大を招く
- 同じ趣旨を複数の表現で繰り返さず、最も短い1表現に集約する
- コーディングエージェントが既に知っていることや、ツール経由で調べればすぐ分かることは書かない
- 既にコンテキストに読み込み済みと想定されるルールやスキルに対する言及を繰り返さない
  - 対象は自動ロードされる`AGENTS.md`・`CLAUDE.md`・`.claude/rules/`配下と、呼び出し済みスキル内の自ファイル間参照
  - 「詳細は～スキルを参照」などの表現は避ける（すでにコンテキストに読み込まれている場合、自動的に参照されている）
  - 自動ロード対象外の外部ドキュメント（公式ドキュメント・他リポジトリのファイル・未ロードのMarkdownなど）へのリンクは対象外
  - 適用範囲などは極力参照される側に書く

### メタ記述の禁止

スキル・エージェント・ルールなどコーディングエージェントへ直接読み込まれる文書は、エージェントへの直接指示として書く。
編集・レビューする人間向けのメモ・メタ情報は本文に書かず、frontmatter内コメント（`#`行）に書く。
実行時にエージェントへ渡らないため本文を汚さずに記述できる。

「本スキル」「本ドキュメント」などの自己言及そのものは許容する。
本文へ書かずfrontmatter内コメントへ書くべきメタ記述は以下の4種類である。

- 管理方針 — ドキュメント自身がどこに何を集約しているかを語る言い回し
  - 例:「〜をSSOTとしてまとめている」「本スキルでは再掲しない」「対象をまとめる」
  - そのままセクション本文に直接指示を書く（読み手は内容自体から理解する）
  - 対象外:「（手順1で読み込む）」「（本スキルに先立って読み込む）」のような使用タイミング注記は、
    コーディングエージェントの実行手順を直接示す情報のため許容する
- 編集判断 — 書き手がどのように記述を取捨選択したかを語る言い回し
  - 例:「〜のSSOT」「本節では〜を記述する」
  - 必要な指示はそのまま本文に書く
- 運用変化 — 過去・現在の状態の変化や廃止・移行を語る言い回し
  - 例:「〜へ集約済み」「以前は〜だった」「判定ラベルは廃止」
  - 現在守るべき運用・現在のフォーマットを直接記述する
- 連携先・編集者向け参照 — 他ファイルとの連携・同期手順、修正時に併せて更新すべき他ファイルへの参照、
  他ファイル規範との意図的重複（SSOT違反を承知で強調目的に再掲する旨）の宣言。
  例:「修正後に他ファイルへ反映する手順」「対応する他ファイル群への参照」「本原則は〜と同趣旨」

### 横断指針の配置

複数フェーズ・複数文脈に跨る指針は、独立節として手順リスト直後など適用範囲が明確な位置に置く。
特定フェーズ・特定文脈の見出し配下に置くと、コーディングエージェントは見出し階層を強く参照するため、
当該フェーズ・当該文脈に限定された指針と狭く解釈する。

### サンプル・テンプレート本文の純度

サンプル計画ファイル・テンプレート・記述例コードブロック内には、成果物本文に書くべき内容のみを置く。
書き方の手引き（メタ説明文）をサンプル本文に混入させると、
コーディングエージェントが手引き文ごと成果物へ転記する誤読を招く。
手引きはサンプル外（節見出し配下の地の文・別節）に分離する。

### コンテキスト汚染の回避

- 不適切な表現や禁止パターンの具体例をそのまま書かない（コンテキストに混入し生成されやすくなるため）
  - 具体例が必要な場合は抽象化した表現にとどめる
- 機械チェック用辞書（NGワード・正規表現リストなど）の検出語そのものを、
  スキルの本文・テストコード・hookの警告メッセージへ転記しない（生成確率を上げる逆効果になるため）
- 例外: 観点別の対比集（悪い例と書き換え案の対比が学習目的で必要な`references/`配下の隔離ファイル）に限り、
  悪い例を直接記述してよい。隔離ファイルは以下の運用条件を満たす
  - メインエージェントから直接読み込ませない（確認はExploreサブエージェント経由、
    修正は`plan-implementer`経由で行うルールを呼び出し元プロンプトおよびエージェント本文で明示する）
  - サブエージェント側で読み込む場合は悪い例の文面を成果物・指摘・出力へ転記しない
  - 機械チェックスクリプト（`check_colloquial.py`等）の検査対象から除外する

### 既知情報・冗長記述の排除

- 学習データやシステムプロンプトで既知の一般指針は書かない
- 細則の列挙はそれを束ねる上位指針に集約する。
  詳細手順は対応するスキルのreferences配下へ退避し、トリガー時のみロードする
- 同一概念を複数箇所で長文展開しない。
  用語定義は1箇所で短く示し、他箇所では用語のみで参照する
- 短い権威表現（用語・規範用語）で長文説明を置き換える場合、
  独立した第三者のコーディングエージェントが用語1語のみから同一行動を取れることを
  事前にサブエージェント検証で確認する。
  未検証または検証不合格の表現は採用しない

## Claude Code固有事項

Claude Code固有の機能・制約に依存する記述。
他のコーディングエージェントを利用する場合は読み替える前提とする。

### `.claude/rules/`配下のpaths活用

`.claude/rules/`配下のルールファイルはYAML frontmatterの`paths`フィールドで条件付きロードできる。
パターンに一致するファイルをClaude Codeが読むときにのみコンテキストへロードされ、起動時の常時ロードを避けられる。
パスパターン記法と詳細はMemory公式ドキュメントを参照する。

新規ルール作成・既存ルール見直し時は以下を順に評価する。

- 常にコンテキストに必要かを判断する。タスク固有で読み込み頻度が低いものは`.claude/skills/`配下へ移す
- 特定モジュール・特定ディレクトリ群でのみ参照すれば十分なものは`paths`で対象を限定する
- 全コード・全ドキュメントに横断的に関わるものはfrontmatter無しのまま残す（無条件ロード）
- 既存ルールへ節を追加する場合、追記内容の適用範囲（対象ファイル群・ディレクトリ群）が
  `paths`の対象範囲と一致しているかを確認する。
  対象外まで広がる場合は別ルールへ分離するか`paths`を見直す

paths未指定のルールは`CLAUDE.md`と同等の優先度で起動時に常時ロードされ、コンテキストを継続的に消費する。

## Gotchas

- システムプロンプト用語と訳語が揺れやすい。
  - システムプロンプトで定義された用語を優先する
  - 文章が大幅に変わる場合はユーザーへ確認する
- システムプロンプトと矛盾しない、かつ重複しない指示にする。やむを得ず矛盾する指示を書く場合はその旨注記する
- 複数ファイルに同一トピックを書くと矛盾が生まれやすい。一箇所を正規の定義元とし、他のファイルは参照に留める
