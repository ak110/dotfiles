# コーディングエージェント向け文書品質

コーディングエージェント向け文書固有の記述原則。
一般ドキュメント品質方針は`agent-toolkit:writing-standards`が担当し、本スキルと併用する。

## 適用範囲

対象はコーディングエージェント向け文書一般に共通する記述原則とする。
コーディングエージェント向け文書はClaude Code環境を前提として書いてよい。
特定のコーディングエージェント実装に固有の機能・制約・ツール名（`AskUserQuestion`・hookなど）も本文へ直接書いてよい。
`## Claude Code固有事項`節への分離は要しない。
他のコーディングエージェントを利用する場合は読み替える前提とする。

## 読者像

対象文書はコーディングエージェントが直接読み込む。人間の読者を想定した説明文・前置き・装飾は書かない。

## 共通の記述原則

以下はagent.md「品質最優先」「日本語の品質を保つ」原則の具体例である。網羅規定ではないが、列挙外の場面でも原則に従う。

### 記述の簡潔さ

- 各行について「削除するとコーディングエージェントの判断が一貫しなくなる」と言えなければ削除する
- 同じ趣旨を複数の表現で繰り返さない。ただし文章のリズム・文意の遠近感に寄与する反復・接続は保持する
- コーディングエージェントが既に知っていることや、ツール経由で調べればすぐ分かることは書かない
- 既にコンテキストに読み込み済みと想定されるルールやスキルに対する言及を繰り返さない
  - 対象は自動ロードされる`AGENTS.md`・`CLAUDE.md`・`.claude/rules/`配下、
    呼び出し済みスキルの自ファイル間参照、サブエージェント定義のfrontmatter `skills`でプリロードされるスキル
  - 「詳細は～スキルを参照」などの参照誘導は避ける（読み込み済みの内容への誘導は冗長なため）。
    ただしサブエージェントが独立コンテキストで読む場面（スキル本文・サブエージェント定義）は
    参照先スキル名・節名のパスを明示する
  - 自動ロード対象外の外部ドキュメント（公式ドキュメント・他リポジトリのファイル・未ロードのMarkdownなど）へのリンクは対象外

### 文書サイズ上限

対象はコーディングエージェントが常時または高頻度で本文を直接読み込む文書
（`agent.md`・`claude-code.md`・各`SKILL.md`・サブエージェント定義・スキル配下の`references/`など）。
作業ごとに作成・廃棄される計画ファイル（`~/.claude/plans/*.md`・`docs/v{next}/plans/*.md`）は本上限の対象外とする。
本文の物理行数はハードリミット220行を厳守する。
ソフトリミット200行を超える見込みの場合は対策（縮減・統合・分割）まで計画段階で確定する。
ハードリミット220行未満に収まっている既存ファイルは新規違反として扱わず、レビュー指摘の対象としない。
frontmatterおよびYAMLヘッダ行も行数に算入する。
コーディングエージェントによっては文書先頭の一定行数（観測例: 先頭220行までしか読み込まない実装）のみを読み込む。
そのためハードリミット超過は重要な指示の読み込み範囲外への漏れを招く。

- 縮減手順・判定基準・縮減根拠4類型は`agent-toolkit:trim-agent-docs`を参照する
- 改訂の計画作成時は、改訂後の物理行数がハードリミット220行未満に収まるかを計画段階で見積もる
  - 改訂後の見込みが215行以上に達する場合は、改訂後の最終形を組み立てて`wc -l`で実測する
  - 具体手順: `cp`で正本をscratchpadへ複製し、`Edit`で改訂を適用してから`wc -l`で実測する
  - 計画ファイルの`## 調査結果`に現行行数（`wc -l`実測値）、`## 変更内容`に実装段階での`wc -l`再確認手順を記載する
  - 既存ハードリミット超過時または改訂で超過する場合は、既存節縮減を本計画スコープへ含めて220行未満へ収める
- 計画段階で改訂後の最終形を組み立て、行数増減を伴わない末尾追記計画も含めて
  `uvx pyfltr run-for-agent --no-fix`でtextlintの`sentence-length`（120字上限）等の違反を検査し解消する
- 改訂後最終形の機械検査では、検査が参照する設定ファイル・本体内の相対パス参照先を含む関連ファイルをscratchpadへ同時コピーする
  - コピー漏れは行幅違反・リンク切れの擬陽性を招く
- 本上限は対象ファイル単体の評価基準であり、計画の縮退・分割・先送りの根拠としない

### メタ記述の禁止

スキル・エージェント・ルールなどコーディングエージェントへ直接読み込まれる文書は、エージェントへの直接指示として書く。
編集経緯・SSOT宣言・履歴メモなど実行判断に寄与しないメタ情報は本文に書かず、frontmatter内コメント（`#`行）に書く。
他スキルとの併用関係・優先順位・適用範囲・呼び出し関係など、実行判断に直接寄与する関係性情報は本文へ書いてよい。

「本スキル」「本ドキュメント」などの自己言及そのものは許容する。
本文へ書かずfrontmatter内コメントへ書くべきメタ記述を次に挙げる。

- 管理方針: ドキュメント自身がどこに何を集約しているかを語る言い回し
  - 違反例:「〜をSSOTとしてまとめている」「本スキルでは再掲しない」「対象をまとめる」
  - 改善例: 本文には集約後の内容をそのまま直接指示として書く（読み手は内容自体から理解する）
- 編集判断: 書き手がどのように記述を取捨選択したかを語る言い回し
  - 違反例:「〜のSSOT」「本節では〜を記述する」「以下に〜の規定を列挙する」
  - 改善例: 必要な指示はそのまま本文に書き、選択理由は本文に含めない
- 運用変化: 過去・現在の状態の変化や廃止・移行を語る言い回し
  - 違反例:「〜へ集約済み」「以前は〜だった」「判定ラベルは廃止」「旧仕様では〜」
  - 改善例: 現在守るべき運用・現在のフォーマットを直接記述する
- 編集者向け同期手順: 修正後に他ファイルへ反映する手順、編集経緯、履歴メモ、SSOT違反承知の重複再掲宣言など
  - 違反例:「修正後に他ファイルへ反映する手順」「本原則は〜と同趣旨」
  - 改善例: 編集者向けメモはfrontmatter内コメントへ移し、本文には現行の規範のみを残す
- 意図的重複・同期要件を新設する場合のfrontmatter同期注記の同時追加
  - 違反例: 重複対象の双方のファイルがあるのに、frontmatter内同期注記コメントを片側のみへ追加する
  - 改善例: 同期注記コメントは重複対象の全ファイルへ同時に追加する。
    計画段階で対象ファイル一覧へ全件組み込み、片側のみの追加で同期破綻リスクを残さない

### 横断指針の配置

複数フェーズ・複数文脈に跨る指針は独立節として手順リスト直後など適用範囲が明確な位置に置く。
特定フェーズ・特定文脈の見出し配下では、コーディングエージェントが見出し階層から限定指針として狭く解釈する。

### 適用範囲・条件の明示

規範を新規作成・改訂する際は、当該規範の適用範囲・除外対象・前提条件を明示する。曖昧なまま規範化しない。

- 複数のディレクトリ・モジュール・配布物が並ぶ文脈で規範を書く場合、どの対象に適用しどの対象に適用しないかを明示する
- 配布物（プラグイン・ライブラリ）と非配布物（個人環境・専用設定）が混在する場合は配布物境界を明示し、
  非配布物固有の依存を配布物側へ持ち込まない原則を併記する
- 親項目が複数の対象を束ねるサブ項目を持つ場合、サブ項目の適用対象が親項目と一致するか・狭まるかを明示する
- 適用範囲が後から拡張・縮小する余地のある規範は、改訂時に範囲表現を再確認する
- 規範を新規作成・改訂する際の適用範囲は、最も広く取れる範囲を初期案とし、
  対象を限定する場合は限定の根拠を本文に併記する。
  狭く取った結果として限定理由を書かないまま放置すると、後続の改訂で対象漏れが発生する
- 規範本文は独立コンテキストで参照されるため、追加文面案に
  会話文脈依存の指示語（「今回」「先ほど」「当該セッション」など）を含めない。
  具体対象を確定できる名詞句で記述する

### サンプル・テンプレート本文の純度

サンプル計画ファイル・テンプレート・記述例コードブロック内には、成果物本文に書くべき内容のみを置く。
書き方の手引き（メタ説明文）をサンプル本文に混入させると、
コーディングエージェントが手引き文ごと成果物へ転記する誤読を招く。
手引きはサンプル外（節見出し配下の地の文・別節）に分離する。

### コンテキスト汚染の回避

- 悪い例の記述可否は、コンテキスト汚染リスクの有無で判断する
  - 汚染リスクが高い悪い例（生成確率を上げる語彙）は本文に書かず`references/`配下の隔離ファイルへ置く
  - 汚染リスクが低い対比例（語順・主述・構成・冗長性など、危険な語彙に依存しない構造レベルの良否）は本文に直接書いてよい
- 機械チェック用辞書の検出語そのものをスキル本文・テストコード・hookメッセージへ転記しない（生成確率を上げる逆効果のため）
- `references/`配下の隔離ファイルは、危険語彙・禁止パターンを含む悪い例の格納専用とする
  - メインエージェントから直接読み込ませない（確認はExploreサブエージェント・修正は`plan-implementer`経由）
  - サブエージェントは悪い例文面を成果物・指摘・出力へ転記しない
  - 機械チェックスクリプトの検査対象から除外する
  - 隔離ファイルを新規追加する場合は既存例（`scope-escalation-phrases.md`等）と同じく
    対応する配置先スキルの`references/`配下へ配置し、機械チェック除外を同時に整備する
    - `check_colloquial.py`の`_EXCLUDED_FILES`と対応する`check_colloquial_test.py`の2件テストパターン
    - `pyproject.toml`の`[tool.pyfltr] extend-exclude`

### 既知情報・冗長記述の排除

- 学習データやシステムプロンプトで既知の一般指針は書かない
- システムプロンプトと矛盾しない、かつ重複しない指示にする
  システムプロンプトで定義された用語を優先し、訳語が揺れる場合は原語を使う。
  やむを得ず矛盾する指示を書く場合はその旨注記する
- 細則の列挙はそれを束ねる上位指針に集約する
  詳細手順は対応するスキルのreferences配下へ退避し、トリガー時のみロードする
  - 既存節の列挙へ新ケースを追記する提案があった場合は、列挙の拡張ではなく上位指針への昇格・統合で
    吸収できないかを先に検討する。同種事象を含む上位概念へリライトできる場合は、列挙拡張を選ばない
  - 同類事象か新種事象かの判別は、新ケースが既存列挙の対象集合と同一の親カテゴリに属するかで機械的に判断する
  - 規範改訂計画での「遡及スキャン結果」「論理交差の確認結果」「根拠記録」など同種の追記が累積する場合も同基準を適用する
    - 複数事象を統合する上位概念への昇格を優先する
- 同一概念を複数箇所で長文展開しない
  用語定義は1箇所で短く示し、他箇所では用語のみで参照する。
  複数ファイルで同一トピックを扱う場合は一箇所を正規の定義元とし、他のファイルは参照に留める
- 短い権威表現（用語・規範用語）で長文説明を置き換える場合、
  独立した第三者のコーディングエージェントが用語1語のみから同一行動を取れるかを確認する。
  エージェントの行動に影響する規範の置き換えではサブエージェント検証を推奨する
- 当然実行・解釈できる手順詳細を1段ずつ書き下さず、目的・対象の指示にとどめる

### 目的記述の明示

スキル・規範本文は冒頭で目的（適用判断の核となる動機）を1〜2文で明示する。
本文冒頭またはfrontmatter `description`のいずれかで目的が明示されていれば充足とする。
手順のみで目的を欠く本文は改訂時に目的記述を追加する。

### メカニズム選択

新規挙動・制約は次の対応で振り分ける。
指示文は確率的にしか従われないため、決定論的強制が必要な制約は地の文で書かずhookで機械的に実装する。

- 常時参照される横断的標準: `CLAUDE.md`・`AGENTS.md`本文、または`.claude/rules/`配下のpaths未指定ルール
- 特定パス・特定モジュール限定の規則: `.claude/rules/`配下のルールファイル（frontmatterで`paths`指定）
- 手順・チェックリスト・段階的ワークフロー: スキル
- 決定論的強制・コマンドブロック・自動lint・通知: hook（不可逆禁止は`permissions`併用）
- 重い調査・冗長出力の隔離: サブエージェント

`.claude/rules/`配下の新規作成・見直し時の評価手順（paths未指定は起動時に常時ロードされる）。

- 常にコンテキストに必要かを判断する。タスク固有で頻度が低いものは`.claude/skills/`配下へ移す
- 特定モジュール・特定ディレクトリ群でのみ参照すれば十分なものは`paths`で対象を限定する
- 全コード・全ドキュメントに横断的に関わるものはfrontmatter無しのまま残す
- 既存ルールへ節を追加する場合、追記内容の適用範囲が`paths`と一致するかを確認し、不一致なら別ルールへ分離するか`paths`を見直す
