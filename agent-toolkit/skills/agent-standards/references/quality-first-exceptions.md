# 「品質最優先」原則の先送り例外規定

`agent-toolkit/rules/01-agent.md`「品質最優先」節の詳細補足として、
`atk fb`連携の先送り例外規定と、非該当時のデフォルト完遂義務を集約する。

## フィードバック自動継続機構の例外

フィードバック自動継続機構（`atk fb process-loop`等の外部処理系）が導入された環境では、
処理系設計上の分割（inbox残存を引き継ぎ媒体とする自動継続）を「完遂原則」の例外として認める。
未導入環境では本例外は適用対象外となり、処理単位の確定後の対象削減は縮退表明として禁止する。

- 本例外の適用範囲は着手前の処理単位の確定タイミングでの全件取得のみとし、
  処理単位の確定後およびネストスキル実行中は先送り・中断・呼び出し元スキルへの復帰の根拠に転用しない
- 途中中断は`atk fb process-loop`側で異常終了扱いとなり自動継続自体が停止する
  （次セッションの自動起動は`agent-toolkit:exit-session`の正常終了コード検知後にのみ発生）

## `atk fb`連携の先送り例外

- 例外: `atk fb status`が有効で、かつ次の全条件を満たす事象は`atk fb add`による
  フィードバック登録を正規経路として認める（無効時は当該計画内での完遂義務を維持する）
  - 根本原因が現在の作業・計画と独立すること（自セッション導入の不良・当該計画スコープ内の問題は対象外）
  - 現在のセッションの成果物品質（テスト成功・lint適合・規範適合・コミット可能性）に影響しないこと
  - 対応にユーザー合意（公開インターフェース変更・規範方針判断など）・別リポジトリ横断作業・
    独立した新規計画の立案のいずれかを要すること
  （上記全条件は`agent-toolkit/rules/01-agent.md`「完遂原則」項の
  自己推定に関する禁止規定に従う）
- 先送り時は事象・根本原因分析・対応案を含む完結したフィードバック本文で登録し、
  登録した旨をユーザーへ報告する（登録をもって「正規経路への引き渡し完了」と扱う）。
  自セッション由来のlint違反・テスト失敗・品質ゲート指摘・自セッション導入の問題は先送り禁止のまま維持し、
  同一根本原因のフィードバックが既にinboxまたは処理履歴にあれば重複登録せず既存項目を見直す。
  判断に迷う場合は先送りせず同一セッション内で対応する（既定は完遂側）
- 既存不良（flaky test・既知バグ・既存lint違反・既存規範違反・既存の陳腐化記述等）も本例外規定の対象に含む。
  例外の全条件を満たさない場合は計画スコープ外を理由に対処を見送らず、発見時点で同一計画内に対処項目として組み込む。
  対処不能の根拠は公式仕様・実機検証で観測可能な事実に限り、成立しない場合は判断を保留せずメイン側へ即時報告する。
  既存コード・既存ファイルの規範違反パターンや「既存の同種ファイルと同運用」等の類似性主張も
  新規追加分の正当化根拠にせず、新規追加分・既存違反自体ともに同一計画内で是正する
- 機械検出（`_scope_escalation.py`によるカテゴリ判定、`pretooluse.py`・`stop_advisor.py`双方の検出経路を含む、
  `pattern-conformance`カテゴリ）は本例外の条件充足を判定しない。
  `process-omission`カテゴリの`session-review`・振り返り・観測事象を主語とする記録型フレーズ、
  および正規のフィードバック登録報告文（`atk fb add`実行後の登録通知など）も同じ扱いとする。
  当該事象で機械検出がフレーズを拾った場合は、本例外の条件
  （`atk fb status`有効・根本原因の独立性・成果物品質への非影響・対応要件充足）を個別に確認する。
  条件充足を確認したうえで、事象そのものを`atk fb add`による正規経路の
  完結したフィードバック本文として登録する（上記のとおり引き渡し完了として扱う）
