# 「品質最優先」原則の先送り例外規定

`agent-toolkit/rules/01-agent.md`「品質最優先」節の詳細補足として、
`atk fb`連携の先送り例外規定・非該当時のデフォルト完遂義務・
登録済み手順の省略禁止と品質作業追加の正規経路（非対称原則）を集約する。

## 正規追加経路の非対称原則

いかなる理由（例: 変更規模・所要時間見込み等）があっても登録済み手順を省略しない。
ここでの登録済み手順とは、規範文書が明示し発火条件が成立した必須の工程・ステップを指す。
条件不成立で対象外となった工程・説明目的の節は含まない。
代表例は`agent-toolkit:plan-mode`工程1〜7・`agent-toolkit:process-feedbacks`ステップ1〜5である。
一方、登録済み手順に列挙のない品質作業（品質最優先の原則・critical-thinking・review-standardsが求める
自発判断型の点検を含む）の追加実施は常時許可する。
ただし公開インターフェース変更・破壊的操作・他利用者への影響を伴う追加実施は、
`agent-toolkit/rules/01-agent.md`「品質最優先」節の既存合意要件を維持したまま実施する。
追加実施の帰結は次の2経路と1例外のいずれかへ収束させる。

- 経路1: 現在進行中の計画または現在のセッションの作業へ取り込む
  （`## 変更内容`・`### 恒久化・リファクタリング内容`等へ反映し同一計画内で完遂する。
  計画ファイル不在時は同一セッション内で対応を完遂するものとする）
- 経路2: 「`atk fb`連携の先送り例外」節の全条件を満たす場合に限り`atk fb add`で正規登録する
- 例外: `agent-toolkit:plan-mode`工程5の別課題化の限定条件を満たす場合は、
  `agent-toolkit/rules/01-agent.md`「完遂原則」項が定める別セッションへの移管（別計画化）を
  第三の対応区分として認める。
  限定条件は`agent-toolkit/skills/plan-mode/references/step5-supplement.md`
  「別課題化の許容基準」節に従う

上記2経路・1例外のいずれも伴わない先送り（登録を伴わない停止・省略・確認待ち等）は本原則の対象外であり、
`agent-toolkit/rules/01-agent.md`「完遂原則」項の縮退表明として禁止される。

本原則は次の緊張点の解消基準を兼ねる。

- 定性的な状況説明と数値化された作業量主張の境界: 品質最優先の原則に基づき対応範囲を拡大した旨を説明する場合、
  実施済み観測事象（点検した観点・発見した事象・反映先）のみを記述する。
  件数・所要時間・残作業量等の自己推定量は記述しない
  （`agent-toolkit/rules/01-agent.md`「完遂原則」項の定性的な進捗記述の許容範囲と同一基準を用いる）
- `agent-toolkit:plan-mode`「工程の進め方」節の暗黙スキップ・明示バイパスの禁止バレットと
  本ファイルの適用順序: 登録済み工程の省略発話・記録を検知した場合、
  まず即時無効化・正規工程への復帰を先に完了させる。
  復帰後に当該発話・記録を観測事象として扱いたい場合のみ、本ファイル
  「`atk fb`連携の先送り例外」節の条件充足を判定する（順序の入れ替えは不可）

## フィードバック自動継続機構の例外

フィードバック自動継続機構（`atk fb process-loop`等の外部処理系）が導入された環境では、
処理系設計上の分割（inbox残存を引き継ぎ媒体とする自動継続）を「完遂原則」の例外として認める。
未導入環境では本例外は適用対象外となり、処理単位の確定後の対象削減は縮退表明として禁止する。

- 本例外の適用範囲は着手前の処理単位の確定タイミングでの全件取得のみとし、
  処理単位の確定後およびネストスキル実行中は先送り・中断・呼び出し元スキルへの復帰の根拠に転用しない
- 途中中断は`atk fb process-loop`側で異常終了扱いとなり自動継続自体が停止する
  （次セッションの自動起動は`agent-toolkit:exit-session`の正常終了コード検知後にのみ発生）

## `atk fb`連携の先送り例外

- 例外: `atk fb status`が有効で、かつ次の全条件を満たす事象は`atk fb add`による
  フィードバック登録を正規経路として認める（無効時は当該計画内での完遂義務を維持する）。
  本条件を満たすフィードバック登録は縮退表明に該当しない
  - 根本原因が現在の作業・計画と独立すること（自セッション導入の不良・当該計画スコープ内の問題は対象外）
  - 現在のセッションの成果物品質（テスト成功・lint適合・規範適合・コミット可能性）に影響しないこと
  - 対応にユーザー合意（公開インターフェース変更・規範方針判断など）・別リポジトリ横断作業・
    独立した新規計画の立案のいずれかを要すること
  （上記全条件は`agent-toolkit/rules/01-agent.md`「完遂原則」項の
  自己推定に関する禁止規定に従う）
- 先送り時は事象・根本原因分析・対応案を含む完結したフィードバック本文で登録し、
  登録した旨をユーザーへ報告する（登録をもって「正規経路への引き渡し完了」と扱う）。
  自セッション由来のlint違反・テスト失敗・品質ゲート指摘・自セッション導入の問題は先送り禁止のまま維持し、
  同一根本原因のフィードバックが既にinboxまたは処理履歴にあれば重複登録せず既存項目を見直す。
  判断に迷う場合は先送りせず同一セッション内で対応する（既定は完遂側）
- 既存不良（flaky test・既知バグ・既存lint違反・既存規範違反・既存の陳腐化記述等）も本例外規定の対象に含む。
  例外の全条件を満たさない場合は計画スコープ外を理由に対処を見送らず、発見時点で同一計画内に対処項目として組み込む。
  対処不能の根拠は公式仕様・実機検証で観測可能な事実に限り、成立しない場合は判断を保留せずメイン側へ即時報告する。
  既存コード・既存ファイルの規範違反パターンや「既存の同種ファイルと同運用」等の類似性主張も
  新規追加分の正当化根拠にせず、新規追加分・既存違反自体ともに同一計画内で是正する
- 機械検出（`_scope_escalation.py`によるカテゴリ判定、`pretooluse.py`・`stop_advisor.py`双方の検出経路を含む、
  `pattern-conformance`カテゴリ）は本例外の条件充足を判定しない。
  `process-omission`カテゴリの`session-review`・振り返り・観測事象を主語とする記録型フレーズ、
  および正規のフィードバック登録報告文（`atk fb add`実行後の登録通知など）も同じ扱いとする。
  当該事象で機械検出がフレーズを拾った場合は、本例外の条件
  （`atk fb status`有効・根本原因の独立性・成果物品質への非影響・対応要件充足）を個別に確認する。
  条件充足を確認したうえで、事象そのものを`atk fb add`による正規経路の
  完結したフィードバック本文として登録する（上記のとおり引き渡し完了として扱う）
- 段階的移行の継続例外を認める。対象計画とテーマ・設計意図が同一の後続段階であっても、
  `atk fb status`が有効で、かつ次の全条件を満たす場合、`atk fb add`によるフィードバック登録を
  正規経路として認める（独立事象向けの既存条件とは適用対象を異にし、
  同一の先送り事象が両条件を同時に満たすとは扱わない）
  - 対象計画の成果物が、引き継ぎ分を含まずに単独完結すること
    （テスト成功・状態の一貫性・コミット可能性を満たす）
  - 引き継ぎ分が独立した計画として成立する分離性を持つこと
  - 完結したフィードバック本文の登録を伴うこと
- 段階的移行の継続例外の適用時は、登録した旨の報告のみで足り、
  協調モード・自律実行モードとも事前確認・事後確認は求めない
- いかなる理由（例: 段階的移行の継続例外に該当すること）があっても、
  フィードバック登録を伴わない先送り宣言を発行しない
