# Agent Skills実装ガイドライン

Skill仕様のうちコーディングエージェント横断で適用できる汎用部分を扱う。
Claude Code固有の挙動には節ごとに「Claude Code固有」の注記を付ける。

## YAML frontmatter

- `description`には、概要とトリガーすべき条件の説明のみを書く
  - スキルの実装の詳細は書かない
  - `name`と合わせて100語程度までを目安に。長すぎるとトリガー判定のノイズになる
  - 複数行に分けたい場合はYAML折り畳みスカラー（`>`）を使う。改行は半角スペースに変換される
  - 起動条件の重複表現を避ける
    - 例:「明示指示でのみ起動する。`/name`での手動トリガー専用」のように同じ意味の文を並べない
    - 重複はモデルがどちらを優先するかの揺れを招き、後の改訂時に片側更新漏れが起きやすい
  - トリガー発話例を複数列挙する書き方（「Aと言われたら／Bと言われたら」など）は重複ではない
- `disable-model-invocation: true`（Claude Code固有）: Claudeによる自動起動を無効化し、`/name`の手動起動のみに限定する。
  副作用の大きいワークフロー（デプロイ・引き継ぎファイル出力など）で意図しない起動を防ぐ用途に使う

## SKILL.md本体とreferences/の役割分担

言語・トピック非依存の汎用規範はSKILL.md本体へ置き、言語固有・トピック固有の詳細はreferences/配下へ置く。

- 本体は全対象に共通する原則を記述する
- references/は対象拡張子・トピックごとの細則を記述し、トリガー時のみ読み込む
- ある規範の所在を調べる際は本体とreferences/配下の両方を確認する
  - 片方だけを確認して有無を判断すると、本体側の汎用規範を看過する

## その他

- `description`や本文にはfrontmatterで設定済みの仕様（例えば`disable-model-invocation: true`による
  自動起動の無効化など）を書かない。
  システム側で保証される挙動を繰り返すとコンテキスト消費が増え、
  frontmatterと本文で表現に齟齬が生じた時の矛盾リスクも生じる
- スキルの運用で繰り返し発生した失敗事例は`## Gotchas`セクションに蓄積する。
  再発防止に有効で、スキル本文の抽象的な指示を具体例で補う
  - Gotchasの記述は禁止事項ではなく、「陥りやすい判断パターンと回避策」の形で書く（禁止事項の列挙はコンテキスト汚染を招くため）
