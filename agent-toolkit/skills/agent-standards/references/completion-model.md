# 完遂成立モデル

`agent-toolkit/rules/01-agent.md`「品質最優先」節冒頭が参照する完遂成立モデルの本体として、
誤ったモデルと正しいモデルの対比、および各根拠の背景を集約する。
本ファイルの要点は`01-agent.md`「品質最優先」節冒頭にinlineで併記される。両者の趣旨が乖離しないよう、改訂時は同時更新する。

## 誤ったモデルと正しいモデル

セッション完遂に関する判断は、次の4組の対比で誤ったモデルから正しいモデルへ置き換える。

| 観点 | 誤ったモデル | 正しいモデル |
| --- | --- | --- |
| セッションの性質 | 有限予算の会計単位であり消費量で完遂可否を判定する | 継続機構が作業量を外部化し任意規模の完遂を担保する |
| 評価対象 | セッション全体の遂行可能性を見積り懸念として報告する | 評価対象は常に次の1工程の技術的成立性のみとする |
| 規模と分割の関係 | 大規模な変更ほど分割した方が安全である | 規模が大きいほど一括計画・一括レビューが品質維持の条件となる |
| 工程列挙の意味 | 後続工程列挙は残量（残債）情報であり多いほど困難になる | 工程列挙は実施順序の定義であり作業量見積りの入力ではない |

## 各根拠の背景

### 継続機構による作業の外部化

- 自動コンパクションによりコンテキストは自動的に要約・継続され、セッションが単純な有限予算として尽きることはない
- 計画ファイル自立性により、計画ファイル本文のみで判断材料が揃う耐久状態を構築できる。
  コンテキストクリアやサブエージェント委譲を経ても計画ファイル本文から作業を再開できる
- サブエージェント委譲によりメインのコンテキストを消費せず並列・分割実行できる
- 大規模開発を1計画ファイルで完遂した実績が十分にある（`git log --grep`で参照可能）

### 次の1工程のみを評価対象とする根拠

セッション全体の遂行可能性は、公式仕様が保証する`max_tokens`・コンテキスト窓以外の
観測可能な量的指標を持たない。トークン消費量・コンテキスト残量・経過時間の実測値は
コーディングエージェントが自力で取得できないため、これらに基づく完遂可否の見積りは
実測に基づかない主観的推定にすぎず、判断の根拠として成立しない。
statuslineの値はhook間で共有されず、hook経由でメインエージェントへ橋渡しする経路も
公式の仕様保証外のため導入しない。
成立する評価は「次の1工程が技術的に実行可能か」のみであり、この評価の反復により
任意規模の作業を完遂まで進められる。

### 規模拡大時に一括計画・一括レビューが必要な根拠

本運用（agent-toolkitが管理する規範・スキル・サブエージェント定義群）は相互参照が密で、
1箇所の変更が関連する複数箇所（判定基準・参照リンク・列挙構造・遡及対象）に波及する。
関連変更を分割すると、分割単位ごとのレビューでは他の変更単位との整合性検討が欠落し、
統合時に不整合（同一概念の複数表現・参照の欠落・判定基準の齟齬）が生じやすい。
一般的なソフトウェア開発では大規模変更を分割してレビューする直観が働く。
本運用の密結合な構造にはこの直観を適用できない。

### 工程列挙が順序定義である根拠

多段スキル（`agent-toolkit:process-feedbacks`・`agent-toolkit:plan-mode`等）が提示する工程列挙は実施順序の定義である。
「これだけの残量がある」という作業量見積りの入力ではない。
工程数の多寡は当該多段スキルの設計上の分業構造を反映するにすぎず、完遂の困難さを表す指標ではない。
