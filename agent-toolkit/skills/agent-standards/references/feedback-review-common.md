# フィードバック処理・振り返り共通観点

`agent-toolkit:session-review`・`agent-toolkit:process-feedbacks`両スキルから参照するSSOT。
両スキルで独立に定義していた採否判定観点・調査観点を集約する。

## 事前調査の徹底

候補事象に関連する既存規範文書の関連節を網羅的に`Read`してから抽象化階層を確定する。
対象は`agent-toolkit/rules/01-agent.md`・`agent-toolkit/rules/03-claude-code.md`・該当作業のスキル本体・
対応する`references/`配下を含む。当該事象を特殊ケースとして包含する既存上位節がある場合は
新規追加ではなく既存節の表現強化・対象類型追加・例示拡張へ変換する。

観測事象が実装挙動の異常（フック発火漏れ・記録漏れ等）に該当する場合は次の3項を実施する。

1. 該当機能の実装コードを`grep -n`で追跡し現行実装で対処済みかを確認する
2. セッション状態JSON（`{tempdir}/claude-agent-toolkit-{session_id}.json`）をダンプし該当契機の記録を確認する
3. 該当機能のテストコードを`grep -n`で確認し再現ケースが既存かを確認する

いずれか1項でも既対処が確認できた場合は投入せず処理済扱いとする。
原因が実装外（Claude Code側の仕様など）に帰せられる場合は、
その旨と依拠範囲・代替回避策を提案内に併記する。

## ユーザー過去方針の一次資料確認

規範改訂・スコープ判断・方針衝突の解釈でユーザーの過去方針・選好を参照する場合、
adopted配下のユーザー起草フィードバック（frontmatterに`source: session-review`を持たないもの）を
一次資料とする。確認は`atk fb show --include-processed <FILENAME>`で実施する。

## 候補技術主張の実機検証

候補の技術的主張（フィードバック原文・観測事象の因果解釈など）が実装挙動・ツール挙動・
ライブラリ仕様の因果関係を含む場合、批判的検討ステップで当該主張の実機再現を1回実施する。
実機再現手段は低コスト検証可能な範囲
（対象CLIの`--help`出力・単体コマンド実機実行・対象設定ファイルの現在値照合など）に限定する。

- 該当する場合は実機再現に加えて計画段階で該当実装コードの`Read`で裏付ける。
  実装ソースパス・確認関数名・照合結果を
  `agent-toolkit/skills/plan-mode/references/plan-file-guidelines.md`が定める
  `### 事実確認済み事項`へ記述してから反映内容を確定する
- 観測事象が真だが因果解釈のみ誤っている場合は、原主旨（観測事象の解消）を保ちつつ
  因果部分を正確な実装挙動に整えた表現へ書き換えて反映する（縮小ではなく事実訂正扱いとする）
- 主張どおりに再現できた場合は裏付け情報として採用判定材料へ組み込む
- 観測事象自体が再現不可で主張が成立しない場合は書き換えて反映する扱いに含めず、
  「不採用根拠の限定」節の例外3で扱う
- 実機再現手段が思い当たらない主張
  （環境固有事象・過去1回の観測事象のみで再現不能なものなど）は不採用確定せず
  「未検証」として保留し、ユーザーへ照会する

## 追記文言案の表記統一の事前照合

追記文言案を計画ファイル`## 変更内容`へ転記する前に、追記文言に含まれる技術用語について、
追記対象ファイル内で同概念の既存表記が存在しないかを`grep -n`で照合し、
既存表記があれば追随する。照合対象は次の4種類とする。

- 英字識別子（バッククォート囲みのコード識別子は対象外）
- 頻出のカタカナ語
- 専門用語の漢字表記候補
- 和語同義語の候補

検出時は追記文言側を既存表記へ揃える。既存表記自体が不適切と判断する場合は
既存表記の改訂も本計画スコープへ含める。

## 完全性主張の実在照合

追記対象箇所またはその近傍に「全件」「全項目」「網羅」「漏れなく」等の完全性主張が含まれる場合、
当該主張が指す実在の列挙・実装項目を対象節内で全件抽出し、主張と実在の一致を目視照合する。
乖離を検出した場合は本計画のスコープへ乖離解消を含める。

- 抽出手順は対象節を`grep -n`または`Read`で走査し、実バレット・実列挙要素を全件列挙する
- 主張の代表項目列挙が過去改訂時点で更新が停止し実バレット追加に追従できていない場合は、
  代表項目列挙を実在の一次情報源への参照（「対象節に列挙された全項目」等）へ置換して以降の乖離発生源を除去する

## 判断軸の順序

`(a)` 計画実行（`plan-impl-executor`）以降に発生する問題をplan-mode以前に前もって対策できる改善提案は、
後戻り・再設計を要する不備の事前防止に限り品質改善効果が見込まれるため最優先で採用する。
実装時の局所修正（改行追加・表記統一・命名調整等）で解消できる体裁・表記上の不備は、
計画時チェック追加の対象とせず実装段階の対応へ残す（往復コストが品質改善効果を上回るため）。

`(b)` 数回のツール呼び出しを削減できるような改善提案（事前知識の追加など）は、
基本的にメリットとデメリットが見合わないため不採用とする。
ただしツール側の改善（コマンド出力の素直化・既存機能の使い方明確化など）でより素直な挙動になるものは採用する。

## 既存対処済みの確認

既存のCLAUDE.md・ルール・スキルで対処済み、または同じ根本原因を部分的に扱う記述がないか確認する。

- 該当記述があり機能している場合は新規追加ではなく既存記述の表現強化・条件追加・例示拡張・配置見直しを優先する
- 既存記述が存在しても違反が発生した場合は「対処済み」と見なさず、機能していない兆候として既存記述の強化を提案する
- 観測事象の原因が既存規範の文面解釈ミスに帰せられる場合は対処済み扱いとせず、当該規範の文面明確化案を提案候補に含める

## 規範改善案の併記

対症対処のみで終えず、当該規範の明確化案または1階層上のカテゴリでの再発予防の検討結果を併記する。
これにより同種事象の再発を予防する。本節は`agent-toolkit:plan-mode`工程4「恒久化検討」と
同一のSSOTを構成する。条件を問わず全採否判定で実施する。
検討対象は、複数フィードバックに共通する上位カテゴリの規範化候補・仕組み化候補・
既存規範との衝突回避パターンなどとする。
上位カテゴリでの独立節新設が実効性を欠く場合はその判断根拠を明記し、
各フィードバック反映内容の相互参照で代替する。

フィードバック採用計画の`### 却下した代替案`節では、対策強度の序列に従い上位層から順に成立可否を評価する。
対策強度の序列は`agent-toolkit/skills/plan-mode/references/bugfix-process.md`「再発防止策」節が定義する。
不成立の場合は根拠を明記したうえで下位層を選ぶ。上位層評価を省略して下位層のみを列挙する形は認めない。

## 総量規制の努力目標

規範追記を伴うフィードバック採用計画は、同一計画内で既存規範の削減候補の検討を必須とする。
削減候補が成立しない場合は不成立の根拠を明記したうえで追記のみを許容する。

## 方針固定化

`agent-toolkit:session-review`ステップ2「Draft Additions」の生成基準バレットと、
本ファイル「総量規制の努力目標」節は、ユーザー明示方針（2026-07-21確定）である。
`agent-toolkit:agent-standards`「文書サイズ上限」節の文書サイズ2段構造方針も同様にユーザー明示方針
（出典はadopted配下の採用済みユーザー起草フィードバック）として扱う。
緩和・撤回・例外新設はsession-review由来フィードバック・process-feedbacks自動処理では実施せず、
ユーザーの明示指示に限る。

## 同一カテゴリ累積時の規範化ゲート

`atk fb list --type=feedback --status=adopted --category=<カテゴリ名>`で
過去の同一カテゴリ採用数を確認する。判定対象は定数`_CATEGORY_GATE_THRESHOLD`
（`agent-toolkit/scripts/_atk_fb_mutations.py`定義）である。
`atk fb adopt --category`実行時、CLIが同カテゴリの累積件数を集計し閾値到達時に標準エラーへ警告を出力する。
警告出力時は、上位カテゴリでの規範化・仕組み化の検討を「規範改善案の併記」の必須項目へ格上げする。

## 不採用根拠の限定

不採用根拠は技術的不成立・公開インターフェース破壊・破壊的操作・
既存記述または既存実装で同等内容が既にカバー済みに限定する。
完遂困難・規模・スコープ整理・集約難易度・コミット履歴での過去再現有無などを不採用根拠としない。

- 例外1: 観測事象の原因が既存規範の文面解釈ミスに帰せられる場合は不採用判定の対象外とし、当該規範の文面明確化案へ再構成する
- 例外2: 判断軸`(a)`のうち後戻り・再設計を要さず局所修正で解消できる体裁・表記上の不備、
  および判断軸`(b)`該当の提案は、いずれも品質改善効果と実装時修正コストの比較衡量観点で不採用としてよい
- 例外3: 「候補技術主張の実機検証」節に従い実機検証で
  観測事象自体が再現不可で主張が成立しないと判明した場合は
  「実機検証で観測事象自体が再現不可と判明」を不採用根拠として明示できる
- 「既存記述または既存実装で同等内容が既にカバー済み」を不採用根拠とする場合、当該既存記述・既存実装の実効性を実測する
  （実測手段は既存機械チェックの対象範囲実行・サブエージェントの起動記録の網羅性照合・`grep`実測での対象件数計上など）
- 不採用根拠が「既存規範との方向性衝突」の場合、不採用確定前に「例外規定として反映可能か」を必ず検討する。
  例外規定として反映する形は「既存規定は基本ルールとして維持し、フィードバック原文の指定契機に限定した
  例外条項を末尾へ追記する」構造を指す。当該例外規定が更に他の既存規範と衝突する場合、または
  例外条項が主規定を実質的に無効化する場合に限り不採用を確定してよい。不採用確定時は当該検討結果を
  `### 却下した代替案`へ記録する
- 例外4: `agent-toolkit:session-review`ステップ2の生成基準（層3）は反復証跡の有無を理由に
  候補を生成する前の段階で除外するゲートである。「不採用根拠の限定」節が扱う生成済み候補の採否判定とは
  適用段階が異なるため、両者は衝突しない

## 自己整合性の検査

不採用根拠として明示された規範が存在する場合、応答テキスト・思考メモ・工程遷移の報告等で
当該規範への違反行為が発生していないか点検する。実施時点は批判的検討時、実装段階、
および事後のレビュー・進捗ログ確認時とする。違反発生時は矛盾指摘として扱い、
当該判断を直ちに撤回する。撤回操作は当該フィードバックの採否再判定と反映内容の是正、
または当該規範への違反行為の即時中止を指す。
