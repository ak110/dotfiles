# 機械チェックスクリプトの設計原則

エージェントが反復実行する機械チェックスクリプト（`check_*.py`等、計画ファイル・規範文書に対する
検査スクリプト全般）は、出力がエージェントのコンテキストへ蓄積されるため、
ノイズの抑制と起動回数の削減を設計段階で担保する。

## 出力の簡潔さ

- 成功時（違反なし）は無出力でexit 0とする。違反発見の有無を確認するため出力を読む手間を発生させない
- `error`区分の違反時は`stderr`へ該当箇所と要点のみを出力し、exit 1とする。
  `warning`区分は`stderr`へ`[warn]`等の識別子付きで出力しexit codeへ算入しない
  （区分の判定基準は本ファイル末尾の区分基準節を参照する）
- 外部コマンド（textlint等）の生出力をそのまま転記せず、違反行のみへ要約する
  （JSONL等の構造化出力はパースして違反件数・該当箇所のみ提示する）

## 起動の統合

- 同一契機（例: 計画ファイル書き込み直後）で反復起動される複数の独立スクリプトは、
  1プロセスで実行する統合ランナーへ集約する
- 統合ランナーは各スクリプトが公開済みの検査用関数をimportで再利用する。
  同一検査対象への重複したサブプロセス起動を避ける（配布境界を跨ぐ場合の
  サブプロセス呼び出しは許容する）。`check_plan_file.py`を参考実装とする

## 検査項目のerror・warning区分

- `error`区分（exit codeへ算入）は、計画が成立しない致命的な問題の検出に限定する
  （パス・スキル名・節見出しの実在確認失敗、対象ファイル一覧との不整合、縮退フレーズ混入等）
- 計画の成立可否に影響しない体裁・表現規範は新規の機械チェック対象にしない。
  `04-styles.md`「日本語の品質を保つ」節の件数・列挙を明記しない規定は、
  同節が機械チェック対象化を明示的に禁じているためこの区分に属し、機械チェックを持たない
- 体裁・表記系（textlint・markdownlint・typos・口語表現等）は`warning`出力に限定し、
  exit codeへ算入しない。判断根拠は`agent-toolkit:plan-mode`の
  「計画作成プロセス自体」改訂方針（体裁・表記の指摘は警告に留める）に従う
- 新設検査を追加する場合、`error`・`warning`区分の判定根拠を検査スクリプトのdocstringへ明記する
