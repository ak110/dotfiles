# 修正案の採用前検証

修正案の適用結果が新規違反を生じないかを以下の観点で検証する。
レビュアー代替案・メイン採用判定済み指摘の双方が対象。

1. 文書サイズ上限への影響（`agent-toolkit:agent-standards`「文書サイズ上限」節の対象ファイル）。
   追記採用時は追記後行数を見積もり、同節のハードリミット超過を違反とする判定基準に従う。
   ハードリミット以下に収まらない場合は既存節縮減との抱き合わせ・代替案選択・不対応への転換のいずれかを選び採用方針を再評価する。
2. 既存先例との整合（`agent-toolkit:coding-standards`・`agent-toolkit:writing-standards`・
   `agent-toolkit/rules/01-agent.md`、同節内の他箇所、関連スキルの同観点記述）。
3. 同一ファイル内で隣接する他規範への副次的影響。
4. 原文の論理関係保存（理由節・条件節・並列列挙の関係）。
   文末述語削除・接続表現差し替え・並列要素統合などで文意が変化する案は不採用。
5. 指摘根拠の外部仕様（言語仕様・ファイル名規約・命名慣習等）の事実性の一次資料検証。
   外部仕様参照を伴う指摘では公式仕様・PEP・公式ドキュメントを引いて事実確認し、
   ファイル名・命名規約への指摘は同プロジェクト内の同種既存ファイルを観察して規約妥当性を確認する。
6. 同一ファイル内自参照（節タイトル名・見出し名で他節から参照される箇所）の同期確認。
   タイトル変更を伴う修正案は、変更前タイトル文言を当該ファイル内で`grep`し、参照箇所も同時更新する。
7. 元目的の達成維持。
   修正案が表層事象（テスト失敗・整合性違反・レビュー指摘等）を解消しても元目的を損なう場合は不採用。
   元目的を維持できる代替案（規範側の明文化・指摘不採用・別実装への切替）を選ぶ。
   判定は指摘の表層記述ではなく、修正前実装・計画ファイル記載の目的・要求元コメントに照合して行う。
