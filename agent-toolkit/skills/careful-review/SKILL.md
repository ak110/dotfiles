# レビュー工程

対象範囲（ファイル群または差分）の評価を2エージェントへ分担する。

- `plan-spec-reviewer`: 計画適合性および成果物間の整合性
- `plan-impl-reviewer`: コード単体品質・ドキュメント単体品質・日本語表現

両エージェントへ並列に委ね、指摘を統合する。

## 入力

呼び出し元から計画ファイルパス・対象範囲のいずれか（または両方）を受け取る。
入力形式は厳密に固定せず、提示された情報から対象範囲と計画ファイルの有無を判別する。

- 計画ファイルパス: `~/.claude/plans/{ファイル名}.md`
- 対象範囲: ファイルパス列挙、またはgit差分範囲（`<base>..HEAD`等）

`plan-impl`経由起動時は両方が付与される。
手動起動時は`/agent-toolkit:careful-review <引数>`の形式で計画ファイル・対象範囲・両方のいずれかを受け取る。

## 対象範囲の確定

呼び出し元から受け取った情報を以下の順で扱う。

1. 対象範囲が明示されていればそれを採用する
2. 対象範囲が未指定で計画ファイルのみ提示された場合、計画着手前コミットから`HEAD`までの差分を対象範囲とする
3. 不変参照点は計画ファイル記載のSHA、または上流ブランチHEAD（`origin/<base-branch>`相当）から選ぶ。
   amend／fixupを伴う場合は`HEAD~1`等のHEAD相対参照が刻々と変わるため、コミットSHAを優先する
4. 対象外ファイル（一時ファイル等）があれば呼び出し元から受け取り、起動プロンプトへ列挙する
5. 差分取得は`git diff {基準点}..HEAD`と未追跡ファイルを含む作業ツリー参照で行う

## サブエージェント起動方針

`plan-spec-reviewer`と`plan-impl-reviewer`を初回モードで並列起動する。
計画ファイルの有無は`plan-spec-reviewer`の担当観点のみに影響する。

- 計画ファイルあり: 仕様適合性と成果物間の整合性を評価する
- 計画ファイル無し: 仕様への適合性評価が成立しないため、成果物間の整合性のみ評価する

並列起動時は単一メッセージ内に複数の`Agent`呼び出しブロックを並べる
（個別メッセージで逐次起動すると並列化されないため）。

### plan-impl-reviewerの分割起動

`plan-impl-reviewer`は対象ファイルの件数と種別に応じて分割起動を検討する。
種別ごとに呼び出すスキルが異なるため、件数が多いほど分割の効果が大きい。

- 対象が少ない場合は1件にまとめて起動する
- 対象が多い場合は次の3カテゴリに分けて並列起動する
  - コード・テストコード（`agent-toolkit:coding-standards`＋`agent-toolkit:writing-standards`）
  - 一般ドキュメント（`agent-toolkit:writing-standards`）
  - コーディングエージェント向け文書（`agent-toolkit:writing-standards`＋`agent-toolkit:claude-code-standards`）
- 各カテゴリの件数が少なすぎる（例: 1〜2ファイル）場合は、他のカテゴリと統合して起動する
- 1カテゴリ内の件数が多い場合は、ファイル群をさらに分割して並列起動してよい

## 起動プロンプトテンプレート

両エージェント共通で初回モードに以下を渡す。
`{担当観点}`は種別ごとに差し替える。

- `spec`（`plan-spec-reviewer`）: 計画ファイルと成果物の仕様適合性、および成果物間の整合性。
  計画ファイル未提示の場合は仕様適合性をスキップし整合性のみ評価する。
  修正再実装を経た差分の場合は前回指摘文面が改変・要約・省略されずに反映されているかの照合を含む
- `impl`（`plan-impl-reviewer`）: コード単体品質・ドキュメント単体品質・日本語表現

```text
以下の実装について初回モードでレビューを実施してください。

- 担当観点: {上記種別定義を参照}
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`（計画ファイル無しの場合は欄を省略）
- 対象範囲: {ファイルパス列挙またはgit範囲}
- 対象外ファイル: {一時ファイル一覧}
- 差分取得: `git diff {基準点}..HEAD`と、未追跡ファイルを含む作業ツリー参照で行う
  - `{基準点}`は計画着手前コミットを指す不変参照点（`origin/master`相当の上流ブランチHEADまたは計画着手前SHA）を選ぶ。
    複数コミット計画でも基準点は変えない。
    amend/fixupを伴う場合はHEAD相対参照（`HEAD~1`等）が刻々と変わるため、コミットSHAを優先する

## 制約

- 指摘を断片的にせず、一度のレビューですべての問題点を洗い出すこと
```

## 指摘の統合と修正依頼

各レビュアーの戻り値を受け取りメインが指摘を統合する。
計画やプロジェクトの基準に照らして妥当な修正であれば確認無しで修正してよい。
指摘箇所が今回の計画の範囲外であっても問題点は原則修正する。
必要に応じてユーザーへ確認して対応方針を決定する。

対応不要と判断した指摘には行末尾に注記する。注記付き指摘は「修正指摘」欄に含めない。

- `（ユーザー判断: 不対応 — <理由>）`
- `（メイン判断: 不対応 — <理由>）`

対応が必要な指摘が1件以上あれば修正内容をユーザーに簡潔に報告したうえで、
修正再実装モードで`plan-implementer`へ委譲する。
修正再実装の完了後に検証タスクで合格を確認する。

修正をコミットへ反映する際は、`git log --oneline --decorate -5`で
直前の未プッシュコミットがレビュー対象と一致するかを確認する。
判定基準は次の2点を満たすこと。

1. コミット題目がレビュー対象の計画と整合する
2. コミットに含まれるファイル群がレビュー対象範囲に収まる（範囲外の無関係な変更を含まない）

両方を満たす場合は`git commit --amend`で統合する。
いずれかを満たさない（無関係な未プッシュコミットが直前にある）場合はamendせず新規コミットを作成する。
amend完了後は`git status`で取り残しが無いことを確認する。

## 再レビュー（followupモード）

修正・コミット統合を実施した場合に起動する。

初回レビューと同じエージェント構成（計画ファイル有無に応じて一種別または2種別）を
followupモードで並列起動する。

```text
以下の実装についてfollowupモードで再レビューを実施してください。

- 担当観点: {初回レビュー節の種別定義を参照。初回と同一の値を使う}
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`（計画ファイル無しの場合は欄を省略）
- 対象範囲: {初回と同一}
- 差分取得: `git diff {基準点}..HEAD`（基準点は初回レビューと同一）

## 前回指摘

{統合済み指摘テキストを行を抜かず転記する。
 `（メイン判断: 不対応 — ...）`・`（ユーザー判断: 不対応 — ...）`等の注記が付いた指摘も注記文面ごと残し、
 注記の要約・省略・改変を行わない}

## 制約

- 新規指摘は追加しないこと（初回レビューで網羅されている前提）
- 前回指摘ごとに`addressed`／`partial`／`missing`を判定すること
- 前回指摘内に行番号が含まれる場合、修正で行がズレている可能性があるため、
  対応関係の特定は内容引用で行うこと（必要なら「修正前のL{N}（修正後のL{M}に相当）」の形で併記してよい）
- 修正の副作用で生じた明らかな回帰のみ`regression`欄に記載すること
- 全件`addressed`の場合は短文応答で合格を伝えてよい（例: `全件addressed。回帰無し。`）
```

`partial`／`missing`／`regression`が1件以上あれば、メインで各判定を`git diff {基準点}..HEAD`で再確認する。
実差分が指摘の修正を反映していると判断できる場合は、当該判定を
`（メイン判断: 不対応 — 既に修正済み・サブエージェントの判定誤り）`として扱い、再修正の対象から除外する。
残った真の未対応指摘について「指摘の統合と修正依頼」節へ戻り、
修正再実装→コミット統合→followup再起動を繰り返す。
followupで全件解消できない状態が連続2回続いたら、修正ループを中断してユーザーへエスカレーションする。

全件`addressed`かつ`regression`無しで作業終了。
