# レビュー工程

対象範囲（ファイル群または差分）の評価を複数エージェントへ並列に委ね、指摘を統合する。

- `plan-spec-reviewer`: 計画適合性および成果物間の整合性
- `plan-impl-reviewer`: コード単体品質・ドキュメント単体品質・日本語表現

## 入力

呼び出し元から計画ファイルパス（`~/.claude/plans/{ファイル名}.md`）・対象範囲
（ファイルパス列挙またはgit差分範囲）のいずれか（または両方）を受け取る。
`plan-impl-executor`経由起動時は両方が付与され、手動起動時は`/agent-toolkit:careful-review <引数>`で受け取る。

## 対象範囲の確定

呼び出し元から受け取った情報を以下の順で扱う。

1. 対象範囲が明示されていればそれを採用する
2. 対象範囲が未指定で計画ファイルのみ提示された場合、計画着手前コミットから`HEAD`までの差分を対象範囲とする
3. 不変参照点は計画ファイル記載のSHAまたは上流ブランチHEADから選ぶ
   （amend／fixupを伴う場合はHEAD相対参照が刻々と変わるためコミットSHAを優先する）
4. 対象外ファイル（一時ファイル等）があれば呼び出し元から受け取り起動プロンプトへ列挙する
5. `git diff {基準点}..HEAD`と未追跡ファイルを含む作業ツリー参照で差分を取得する

## サブエージェント起動方針

`plan-spec-reviewer`と`plan-impl-reviewer`を並列起動する。
計画ファイルの有無は`plan-spec-reviewer`の担当観点のみに影響する。

- 計画ファイルあり: 仕様適合性と成果物間の整合性を評価する
- 計画ファイル無し: 成果物間の整合性のみ評価する（仕様への適合性評価が成立しないため）

対象範囲にコーディングエージェント向け文書カテゴリが含まれる場合、
`agent-doc-validator`を`plan-impl-reviewer`・`plan-spec-reviewer`と並列に独立起動する。
`plan-impl-reviewer`は単体品質、`agent-doc-validator`は`01-agent.md`・`agent-standards`方針への適合性を担当する。

### plan-impl-reviewerの分割起動

`plan-impl-reviewer`は対象ファイル件数と種別に応じて分割起動を検討する
（種別ごとに呼び出すスキルが異なるため件数が多いほど分割効果が大きい）。

- 対象が多い場合は次カテゴリに分けて並列起動する
  - コード・テストコード（`agent-toolkit:coding-standards`＋`agent-toolkit:writing-standards`）
  - 一般ドキュメント（`agent-toolkit:writing-standards`）
  - コーディングエージェント向け文書（`agent-toolkit:writing-standards`＋`agent-toolkit:agent-standards`）
- 対象が合計5ファイル以上または合計差分が概ね300行以上の場合はカテゴリ分割を優先する
  （自動bumpで連動更新される`plugin.json`・`marketplace.json`等は閾値カウント対象外）
- 単一カテゴリ内の対象ファイル数が10以上または合計行数が概ね2000行以上の場合は当該カテゴリ内で更に分割起動する。
  APIエラー等の異常終了時の再委譲手順は`agent-toolkit/rules/03-claude-code.md`「サブエージェントの活用」節に従う
- 対象合計ファイル数が30以上または合計差分が2000行以上の場合は`plan-impl-reviewer`の初回並列度を3以下に抑える。
  一般ドキュメント・コーディングエージェント向け文書を統合し2グループへ集約する（併読は`agent-toolkit:writing-standards`・
  `agent-toolkit:agent-standards`）。本条件は単一カテゴリ内分割ルールより優先し、`plan-spec-reviewer`・
  `agent-doc-validator`の独立起動は維持する。重大以上の指摘が集中する領域のみ2サイクル目で追加分割する

## 起動プロンプトテンプレート

両エージェント共通で以下を渡す。`{担当観点}`は種別ごとに差し替える。

- `spec`（`plan-spec-reviewer`）: 計画ファイルと成果物の仕様適合性、および成果物間の整合性
  （計画ファイル未提示の場合は仕様適合性をスキップし整合性のみ評価）
- `impl`（`plan-impl-reviewer`）: コード単体品質・ドキュメント単体品質・日本語表現
- `doc-validator`（`agent-doc-validator`）: `01-agent.md`方針および`agent-standards`スキル方針への適合性
  （コーディングエージェント向け文書カテゴリを含む場合のみ起動）

```text
以下の実装についてレビューを実施してください。

- 担当観点: {上記種別定義を参照}
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`（計画ファイル無しの場合は欄を省略）
- 対象範囲: {ファイルパス列挙またはgit範囲}
- 対象範囲スコープ: 既存ファイル全体（既存違反も指摘してよい）。計画ファイルで別途明示があればそちらに従う
- 対象外ファイル: {一時ファイル一覧}
- 差分取得: `git diff {基準点}..HEAD`と未追跡ファイルを含む作業ツリー参照で差分を取得する
  - `{基準点}`は計画着手前コミットを指す不変参照点を選び、複数コミット計画でも基準点は変えない。
    amend/fixupを伴う場合はHEAD相対参照が刻々と変わるためコミットSHAを優先する

## 制約

- 担当観点ごとに網羅的に列挙すること。各エージェント定義「担当観点」節の全カテゴリを初回レビューで点検し、
  観点ごとに指摘の有無を報告する（指摘なしも明示する）
- 同一ファイル内で同種カテゴリの違反は重大度を問わず初出箇所のみで止めず該当ファイル全体から全件を列挙すること
- 指摘は差分起因（対象計画の改訂で新たに生じた違反）と既存違反を区別して報告すること。両者を別欄に分け、既存違反は補足情報として位置付ける
- 対象範囲のいずれかが1ファイル1000行を超える場合、`autocompact thrashing`予防のため次の制約を適用すること
  - 当該ファイルの全文`Read`を避け、`git diff`出力範囲・該当節（H2/H3単位）・差分周辺のシンボル単位・特定行範囲のみを`Read`する
  - 「該当ファイル全体から全件を列挙」観点は当該ファイル内で差分周辺と該当節範囲に限定される
  - 例外: `plan-impl-reviewer`起動時は本制約を適用せず「plan-impl-reviewerの分割起動」節の運用で担当分を1000行以内へ収める
```

## 指摘の統合と修正依頼

- 各サブエージェントの完了報告冒頭の`## 観点網羅`サマリー欄を検査し、未点検（`[ ]`）観点があれば再起動する

各レビュアー出力（差分起因・既存違反の二区分）をメインが統合対象とし、対応要否は区分によらず重大度で判断する。
`致命的`・`重大`は既存違反でも対応し、`軽微`はメイン判断とする（区分情報は対応順序の参考で不対応根拠にしない）。
選別結果は修正着手前に、
`agent-toolkit/rules/03-claude-code.md`「サブエージェントの活用」節の固定書式表に従い表形式でターン内に提示する。
「重大度／観点」列は`致命的`・`重大`・`軽微`、「区分」列は差分起因・既存違反を記入する。
表後に取り下げ件数と理由パターンを併記する。
ユーザー合意は待たず、提示と同一ターンまたは続くターンで修正実装へ進む。
指摘箇所が対象計画の範囲外でも`致命的`・`重大`は対応し、スコープ拡大等を理由とした不対応は禁止する
（`agent-toolkit/rules/01-agent.md`「品質最優先」原則に従う）。

重大度`致命的`・`重大`の指摘を採用して修正着手する前に、メインが指摘箇所を単体`Read`する。
表層の現象が設計意図上のディフェンシブ実装（初期値埋め直後の`assert`・冗長判定されやすい二重チェック・
呼び出し元前提条件の事前検査等）として成立し得るかを照合する。
成立し得る場合はサブエージェント分類ラベルを採用せず、ユーザー確認またはコード一次資料での再検証へ切り替える。
設計意図上の必要性が確認できた指摘は次の注記で不対応扱いを許容する:
`（メイン判断: 不対応 — 設計意図上のディフェンシブ実装として成立）`。

specとimpl指摘が同一変更箇所で相反する場合、重大度が異なれば高い重大度を優先する。
同一重大度の場合に限りimpl指摘（品質修正）をspec指摘（計画適合性）より優先する。
不対応とした側の指摘はメイン判断で不対応扱いとし理由を注記する。

対応不要と判断した指摘には末尾に注記する。注記付き指摘は「修正指摘」欄に含めない。
注記書式は`（ユーザー判断: 不対応 — <理由>）`または`（メイン判断: 不対応 — <理由>）`とする。

派生・展開系指摘（重複除去・記述冗長削除・SSOT統合等）の対応で原フィードバック・原指摘の明示要求の反映を撤回する場合、
計画ファイル`### エージェント判断`または`## 進捗ログ`「### レビュー反映」ラベルへ撤回の根拠を明示記録する。
撤回の根拠が既存節での機能的等価カバーで説明できない場合は撤回せず、原フィードバック要求の反映を維持する。

修正案の採用前の検証観点と確認手順は`references/modification-checks.md`に従い、
検証結果が不明な場合は当該案を採用せず既存先例と整合する別案を選ぶ。

再レビュー時の指摘が形式上の違い（追記か新節か、節順、見出し名の言い換えなど）に収束する場合は、
対応コストと再発防止効果のバランスを踏まえメイン判断で不対応にしてよい。

修正指摘で追加文面案・置換後文面案を提示する場合、提示前にメイン側で機械チェックを実行し整形する。
機械チェックは`uvx pyfltr run-for-agent --commands=textlint,colloquial-check --enable=colloquial-check`等を指す。

修正再実装で実装ファイルを変更した場合、計画ファイル本文の関連箇所（`## 変更内容`配下のコードブロック・
`### エージェント判断`記載・対象ファイル一覧）も同期更新する。
同期を委譲先で行うかメイン側amend前に行うかは修正指示で明示する。

対応が必要な指摘が1件以上あれば修正内容をユーザーへ簡潔に報告して修正再実装へ進み、完了後に検証タスクで合格を確認する。
codex委譲実装分（進捗ログに`threadId`記録あり）は`agent-toolkit:codex-impl`の継続呼び出しで同一スレッドへ差し戻し、
それ以外は修正再実装モードで`plan-implementer`へ委譲する。

修正のコミット反映は、直前の未プッシュコミットがレビュー対象計画と件名・対象範囲の両面で整合する場合に
`git commit --amend`で統合する。いずれかを満たさない場合はamendせず新規コミットを作成する。
amend後は`git status`で取り残しが無いことを確認する。

## 再レビュー

修正・コミット統合した場合は`重大`以上を含む修正で必ず起動する。
成果物本体（コード・ドキュメント・計画ファイル`## 変更内容`のコードブロック内文面）に
影響しない軽微修正（計画ファイル地の文の日本語表現整え等）のみで構成される場合は再起動しない。
新しい`Agent`で並列起動する（先入観排除のため`SendMessage`再開は採らない）。
対象範囲は修正再実装で書き換えた個別ファイル群とし、
メインが修正方針および完了後の`git status`／`git diff`から確定して起動プロンプトへ列挙する
（計画ファイル・永続化情報には記録しない）。

起動対象は次のいずれかに該当する種別に限定する。

- 初回レビューで対応した指摘が1件以上あった種別（修正後の品質確認のため）
- 修正差分が担当観点に影響し得る種別（回帰確認のため）
  例: impl指摘の修正で計画範囲やドキュメント整合性が変わり得る場合はspec種別も対象に含める

両条件のいずれにも該当しない種別はスキップする。
起動プロンプトは「起動プロンプトテンプレート」節と同一形式とし、対象範囲欄のみ修正で書き換えた個別ファイル群に差し替える。
加えて「再指摘の抑止対象」欄を「対象外ファイル」欄の直後に挿入する。
抑止対象は観点（パターン類型）レベルで記述し既知具体例を併記する。
前ラウンドで採用指摘対応として意図的削除・意図的縮小のために修正した箇所は
「観点＋具体的なファイル名・節名」形式で「再指摘の抑止対象」欄へ全件転記する
（初回レビューで対応済みの指摘は含めない）。
抑止対象は書式レベル（バッククォート等の表記差）と意味レベル（参照や識別子の存否）を分離して記述する。
書式レベルの抑止対象を記述する場合は、対応する意味レベルを別項目として抑止対象外である旨を明示する。

再レビュー結果に対応要の指摘が1件以上あれば「指摘の統合と修正依頼」節へ戻り、
修正再実装→コミット統合→再レビューを繰り返す。
レビュー回数によらず直すべき指摘は確認せず修正する。
メインが対処不可能と判断した場合のみ修正ループを中断してユーザーへエスカレーションする。
判断軸は同一指摘の再発・修正の品質低下・予期せぬ作業量の拡大などとする。

再レビューサイクル2以降の重大度`軽微`指摘は原則メイン判断で不対応とする。
採用は構造的問題（対応規範違反・既存記述齟齬・修正による新規違反の混入など）に限り、
表記揺れ・スタイル懸念・あると望ましい程度の指摘は不採用とし取り下げ理由欄へ記録する。
当該サイクルで採用指摘が0件なら指摘0件扱いとして修正ループを終了し、後続工程へメイン判断で進む。

サイクル3以降で検出された指摘が全て軽微であり、かつ次のいずれかに該当する場合、
当該サイクルで採用指摘なしと即座に判定して修正ループを終了する。

- `(a)` 前サイクル不対応判定の軽微指摘の派生・再指摘
- `(b)` 表記揺れ・コメント精度・重複分岐の指摘
- `(c)` 実装挙動に影響しないスタイル改善提案

`(b)`・`(c)`を適用する前に、対象違反箇所へプロジェクトの機械チェック
（`pyfltr`配下の`pylint`・`colloquial-check`・`textlint`等）を実行する。
機械チェックが違反として検出したものは`(b)`・`(c)`による不対応判定の対象外とし、修正して再レビューを継続する
（機械チェックが検出した違反はpush後CIで顕在化するため「実装挙動に影響しない」の暗黙前提が成立しない）。

サイクル3以降で新規重大指摘が連続する場合の判定基準は
`agent-toolkit/skills/plan-mode/references/codex-review.md`「3ラウンド連続で新規の重大指摘が出る場合の扱い」節に従う。
ユーザー再確認は不要とし不対応の内訳を進捗ログへ記録し、指摘がなければ作業を完了する。
