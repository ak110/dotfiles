# レビュー工程

対象範囲（ファイル群または差分）の評価を2エージェントへ分担する。

- `plan-spec-reviewer`: 計画適合性および成果物間の整合性
- `plan-impl-reviewer`: コード単体品質・ドキュメント単体品質・日本語表現

両エージェントへ並列に委ね、指摘を統合する。

## 入力

呼び出し元から計画ファイルパス・対象範囲のいずれか（または両方）を受け取る。
入力形式は厳密に固定せず、提示された情報から対象範囲と計画ファイルの有無を判別する。

- 計画ファイルパス: `~/.claude/plans/{ファイル名}.md`
- 対象範囲: ファイルパス列挙、またはgit差分範囲（`<base>..HEAD`等）

`agent-toolkit:plan-impl`経由起動時は両方が付与される。
手動起動時は`/agent-toolkit:careful-review <引数>`の形式で計画ファイル・対象範囲・両方のいずれかを受け取る。

## 対象範囲の確定

呼び出し元から受け取った情報を以下の順で扱う。

1. 対象範囲が明示されていればそれを採用する
2. 対象範囲が未指定で計画ファイルのみ提示された場合、計画着手前コミットから`HEAD`までの差分を対象範囲とする
3. 不変参照点は計画ファイル記載のSHA、または上流ブランチHEAD（`origin/<base-branch>`相当）から選ぶ。
   amend／fixupを伴う場合は`HEAD~1`等のHEAD相対参照が刻々と変わるため、コミットSHAを優先する
4. 対象外ファイル（一時ファイル等）があれば呼び出し元から受け取り、起動プロンプトへ列挙する
5. `git diff {基準点}..HEAD`と未追跡ファイルを含む作業ツリー参照で差分を取得する

## サブエージェント起動方針

`plan-spec-reviewer`と`plan-impl-reviewer`を並列起動する。
計画ファイルの有無は`plan-spec-reviewer`の担当観点のみに影響する。

- 計画ファイルあり: 仕様適合性と成果物間の整合性を評価する
- 計画ファイル無し: 仕様への適合性評価が成立しないため、成果物間の整合性のみ評価する

### plan-impl-reviewerの分割起動

`plan-impl-reviewer`は対象ファイルの件数と種別に応じて分割起動を検討する。
種別ごとに呼び出すスキルが異なるため、件数が多いほど分割の効果が大きい。

- 対象が少ない場合は1件にまとめて起動する
- 対象が多い場合は次のカテゴリに分けて並列起動する
  - コード・テストコード（`agent-toolkit:coding-standards`＋`agent-toolkit:writing-standards`）
  - 一般ドキュメント（`agent-toolkit:writing-standards`）
  - コーディングエージェント向け文書（`agent-toolkit:writing-standards`＋`agent-toolkit:agent-standards`）
- 各カテゴリの件数が少なすぎる（例: 1〜2ファイル）場合は、他のカテゴリと統合して起動する
- 1カテゴリ内の件数が多い場合は、ファイル群をさらに分割して並列起動してよい
- 対象が合計5ファイル以上または合計差分が概ね300行以上の場合は、上記の統合条件よりカテゴリ分割を優先する。
  コンテキスト消費の集中を分散し`autocompact thrashing`を予防する
  - 自動bumpで連動更新される`plugin.json`・`marketplace.json`等は閾値カウントの対象外とする
  - 種別を跨ぐ場合はコード・テストカテゴリと文書カテゴリを別エージェントへ分けてコンテキスト消費を分散する

## 起動プロンプトテンプレート

両エージェント共通で以下を渡す。`{担当観点}`は種別ごとに差し替える。

- `spec`（`plan-spec-reviewer`）: 計画ファイルと成果物の仕様適合性、および成果物間の整合性
  計画ファイル未提示の場合は仕様適合性をスキップし整合性のみ評価する
- `impl`（`plan-impl-reviewer`）: コード単体品質・ドキュメント単体品質・日本語表現

```text
以下の実装についてレビューを実施してください。

- 担当観点: {上記種別定義を参照}
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`（計画ファイル無しの場合は欄を省略）
- 対象範囲: {ファイルパス列挙またはgit範囲}
- 対象範囲スコープ: 既存ファイル全体（既存違反も指摘してよい）
  - 計画ファイルで対象範囲スコープが別途明示されている場合はそちらに従う
- 対象外ファイル: {一時ファイル一覧}
- 差分取得: `git diff {基準点}..HEAD`と未追跡ファイルを含む作業ツリー参照で差分を取得する
  - `{基準点}`は計画着手前コミットを指す不変参照点（`origin/master`相当の上流ブランチHEADまたは計画着手前SHA）を選ぶ
    複数コミット計画でも基準点は変えない。
    amend/fixupを伴う場合はHEAD相対参照（`HEAD~1`等）が刻々と変わるため、コミットSHAを優先する

## 制約

- 担当観点ごとに網羅的に列挙すること。
  各エージェント定義「担当観点」節の全カテゴリを初回レビューで点検し、観点ごとに指摘の有無を報告する
  （指摘なしも明示する）
- 同一ファイル内で同種カテゴリの違反は重大度を問わず初出箇所のみで止めず該当ファイル全体から全件を列挙すること
- 指摘は差分起因（対象計画の改訂で新たに生じた違反）と既存違反（改訂前から存在する違反）を区別して報告すること。両者を別欄に分け、既存違反は補足情報として位置付ける
- 対象範囲のいずれかが1ファイル1000行を超える場合、`autocompact thrashing`によるレビュー中断を予防するため次の制約を適用すること
  - 当該ファイルの全文`Read`を避け、`git diff`の出力範囲・該当節（H2/H3単位）・差分周辺のシンボル（関数・クラス定義）単位・特定行範囲のみを`Read`する
  - 広範囲調査は明示的に必要な場合のみ実施する
  - 上記「該当ファイル全体から全件を列挙」観点は当該ファイル内で差分周辺と該当節範囲に限定される
```

## 指摘の統合と修正依頼

各レビュアーの出力を受け取りメインが指摘を統合する。
各レビュアーは差分起因・既存違反の二区分形式で指摘を返す。
メインは両区分の指摘を統合対象とし、対応要否を区分によらず重大度で判断する。
区分情報は対応順序の参考とし不対応の根拠としない。既存違反であっても重大度`致命的`・`重大`は対応する。
重大度`致命的`・`重大`は対応し、重大度`軽微`はメイン判断とする。
選別結果は修正実装に着手する前に、表形式（列: 通番・重大度・区分・箇所・内容・対応方針）でターン内に提示する。
表の後に取り下げた指摘の件数と取り下げ理由のパターンを併記する。
ユーザー合意は待たず、提示と同一ターンまたは続くターンで修正実装へ進む。
指摘箇所が対象計画の範囲外であっても重大度`致命的`・`重大`は対応する。
スコープ拡大等を理由とした不対応は禁止する（`agent-toolkit/rules/agent.md`「品質最優先」原則に従う）。

重大度`致命的`・`重大`の指摘を採用して修正に着手する前に、メインが指摘箇所のコードを単体`Read`し、
表層の現象（型不整合・論理矛盾・冗長記述等）が設計意図上のディフェンシブ実装として成立し得るかを照合する。
対象例は初期値埋め直後の`assert`・冗長と判定されやすい二重チェック・呼び出し元の前提条件の事前検査などとする。
成立し得る場合はサブエージェントの分類ラベルを採用せず、ユーザー確認またはコード一次資料での再検証へ切り替える。
設計意図上の必要性が確認できた指摘は次の注記で不対応扱いを許容する:
`（メイン判断: 不対応 — 設計意図上のディフェンシブ実装として成立）`。

specレビュアーとimplレビュアーの指摘が同一変更箇所で相反する場合は次の通りとする。
両指摘の重大度が異なる場合は高い重大度を優先する。
重大度ラベルは事象の影響範囲・是正緊急度を直接示すため、レビュアー種別の優先順位より優先する。
同一の重大度の場合に限り、品質上の修正（impl指摘への対処）を計画適合性の差分（spec指摘）より優先する。
不対応とした側の指摘はメイン判断で不対応扱いとし、理由を注記する。

対応不要と判断した指摘には末尾に注記する。注記付き指摘は「修正指摘」欄に含めない。
注記書式は`（ユーザー判断: 不対応 — <理由>）`または`（メイン判断: 不対応 — <理由>）`とする。

修正案の採用前の検証観点と確認手順は`references/modification-checks.md`に従う。
検証結果が不明な場合は当該案を採用せず、既存先例と整合する別案を選ぶ。

再レビュー時の指摘が形式上の違い（追記か新節か、節順、見出し名の言い換えなど）に
収束する場合は、対応コストと再発防止効果のバランスを踏まえメイン判断で不対応にしてよい。

修正指摘で追加文面案・置換後文面案を提示する場合、提示前にメイン側で機械チェックを実行する。
整形には`agent-toolkit:writing-standards`配下`scripts/check_line_width.py`等を用い、127幅以内へ揃える。
整形未実施の文面案を委譲するとサブエージェントがそのまま実装へ転記する。
その結果、機械チェック違反でエスカレーションを返し、修正依頼と再実装のやり取り回数が増える。

修正再実装で実装ファイルを変更した場合、計画ファイル本文の関連箇所も同期更新する。
対象は`## 変更内容`配下の対応する文面案コードブロック・`### エージェント判断`記載・対象ファイル一覧とする。
同期更新を修正再実装の委譲先で実施するか、メイン側でamend前に実施するかは修正指示で明示する。
同期未実施のまま再レビューを起動すると計画文と実装の齟齬が重大指摘として再検出される。

対応が必要な指摘が1件以上あれば修正内容をユーザーに簡潔に報告したうえで、
修正再実装モードで`plan-implementer`へ委譲する。
修正再実装の完了後に検証タスクで合格を確認する。

修正のコミット反映は、直前の未プッシュコミットがレビュー対象計画と
件名・対象範囲の両面で整合する場合に`git commit --amend`で統合する。
いずれかを満たさない場合はamendせず新規コミットを作成する。
amend後は`git status`で取り残しが無いことを確認する。

## 再レビュー

修正・コミット統合した場合は、修正規模を問わず必ず起動する。
新しい`Agent`で並列起動する（先入観排除のため既存サブエージェントの`SendMessage`再開は採らない）。
対象範囲は修正再実装で書き換えた個別ファイル群とする。
メインが`plan-implementer`委譲時の修正方針および修正完了後の`git status`／`git diff`から確定し、
amend／新規コミットの粒度に依存しない形で起動プロンプトの対象範囲欄へ列挙する。
計画ファイルや永続化情報には記録しない。

起動対象は次のいずれかに該当する種別に限定する。

- 初回レビューで対応した指摘が1件以上あった種別（修正後の品質確認のため）
- 修正差分が担当観点に影響し得る種別（回帰確認のため）
  例: impl指摘の修正で計画範囲やドキュメント整合性が変わり得る場合はspec種別も対象に含める

両条件のいずれにも該当しない種別はスキップする。
起動プロンプトは「起動プロンプトテンプレート」節と同一形式とし、対象範囲欄のみ修正で書き換えた個別ファイル群に差し替える。
加えて初回レビューで`（メイン判断: 不対応 — <理由>）`・`（ユーザー判断: 不対応 — <理由>）`の注記を付した既存違反箇所を、
「対象外ファイル」欄の直後に「再指摘の抑止対象」欄として箇所と理由を併記する形で挿入する。
抑止対象は箇所単体ではなく観点（パターン類型）レベルで記述する。
例: 「関数docstring内における内部識別子の参照（具体箇所A・B等）」のように類型と既知具体例を併記し、
同根の派生箇所も再指摘対象外となる構成とする。
再レビュアーは当該箇所を再評価対象から除外する。
但し書きを付して再指摘する形（「判断はメインエージェントに委ねる」「検討すること」等）も抑止対象に含む。
再レビューは修正後の品質確認が目的であるため、初回レビューで対応済みの指摘は抑止対象に含めない。

再レビュー結果に対応要の指摘が1件以上あれば「指摘の統合と修正依頼」節へ戻り、
修正再実装→コミット統合→再レビューを繰り返す。
レビュー回数によらず直すべき指摘は確認せず修正する。
メインが対処不可能と判断した場合のみ修正ループを中断してユーザーへエスカレーションする。
判断軸は同一指摘の再発・修正の品質低下・予期せぬ作業量の拡大などとする。

再レビューサイクル2以降の重大度`軽微`指摘は、原則メイン判断で不対応とする。
採用は構造的問題（対応規範違反・既存記述齟齬・修正による新規違反の混入など）に限り、
表記揺れ・スタイル懸念・あると望ましい程度の指摘は不採用とし取り下げ理由欄へ記録する。

指摘がなければ作業を完了する。
