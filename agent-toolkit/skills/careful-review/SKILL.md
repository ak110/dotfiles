# レビュー工程

対象範囲（ファイル群または差分）の評価を複数エージェントへ並列に委ね、指摘を統合する。

- `plan-spec-reviewer`: 計画適合性および成果物間の整合性（claude据え置き）
- `plan-codex-delegate`（`用途: 実装差分レビュー`）: コード単体品質・ドキュメント単体品質・日本語表現
  （モード「codex寄り」の既定経路）
- `plan-impl-reviewer`: 同上観点（モード「claude寄り」の既定経路、または「codex寄り」でcodex利用不可時のフォールバック）

## 入力

呼び出し元から計画ファイルパス（`~/.claude/plans/{ファイル名}.md`）・対象範囲
（ファイルパス列挙またはgit差分範囲）のいずれか（または両方）を受け取る。
`plan-impl-executor`経由起動時は両方が付与され、手動起動時は`/agent-toolkit:careful-review <引数>`で受け取る。

## 対象範囲の確定

呼び出し元から受け取った情報を以下の順で扱う。

1. 対象範囲が明示されていればそれを採用する
2. 対象範囲が未指定で計画ファイルのみ提示された場合、計画着手前コミットから`HEAD`までの差分を対象範囲とする
3. 不変参照点は計画ファイル記載のSHAまたは上流ブランチHEADから選ぶ
   （amend／fixupを伴う場合はHEAD相対参照が刻々と変わるためコミットSHAを優先する）
4. 対象外ファイル（一時ファイル等）があれば呼び出し元から受け取り起動プロンプトへ列挙する
5. `git diff {基準点}..HEAD`と未追跡ファイルを含む作業ツリー参照で差分を取得する

## サブエージェント起動方針

実装レビュー担当は環境のバランスモードで決まる。
`test -f ~/.config/agent-toolkit/review-balance-mode.claude-heavy`が真の場合「claude寄り」、
偽の場合（既定）「codex寄り」とする。
「claude寄り」は`plan-spec-reviewer`と`plan-impl-reviewer`を並列起動する（現行構成を維持する）。
「codex寄り」は`plan-spec-reviewer`と`plan-codex-delegate`（`用途: 実装差分レビュー`）を並列起動する。
計画ファイルの有無は`plan-spec-reviewer`の担当観点のみに影響する。

- 計画ファイルあり: 仕様適合性と成果物間の整合性を評価する
- 計画ファイル無し: 成果物間の整合性のみ評価する（仕様への適合性評価が成立しないため）

`codex-review.md`「codex利用可否の3段階判定」節の段階3が成立した場合は
`plan-codex-delegate`の代わりに`plan-impl-reviewer`を起動する。
本フォールバックはモードによらず共通で適用する。
分担・並列度の詳細は`references/impl-review-launch.md`を参照する。

対象範囲にコーディングエージェント向け文書カテゴリが含まれる場合、
`agent-doc-validator`を上記と並列に独立起動する。
単体品質担当（`plan-codex-delegate`または`plan-impl-reviewer`）と`agent-doc-validator`（`01-agent.md`・`agent-standards`方針への適合性）は担当観点を分離する。

## 起動プロンプトテンプレート

両エージェント共通で以下を渡す。`{担当観点}`は種別ごとに差し替える。

- `spec`（`plan-spec-reviewer`）: 計画ファイルと成果物の仕様適合性、および成果物間の整合性
  （計画ファイル未提示の場合は仕様適合性をスキップし整合性のみ評価）
- `impl`（`plan-impl-reviewer`）: コード単体品質・ドキュメント単体品質・日本語表現
  （バランスモード「claude寄り」時は既定経路、「codex寄り」時は段階3成立時のフォールバック経路として使用）
- `doc-validator`（`agent-doc-validator`）: `01-agent.md`方針および`agent-standards`スキル方針への適合性
  （コーディングエージェント向け文書カテゴリを含む場合のみ起動）

```text
以下の実装についてレビューを実施してください。

- 担当観点: {上記種別定義を参照}
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`（計画ファイル無しの場合は欄を省略）
- 対象範囲: {ファイルパス列挙またはgit範囲}
- 対象範囲スコープ: 既存ファイル全体（既存違反も指摘してよい）。計画ファイルで別途明示があればそちらに従う
- 対象外ファイル: {一時ファイル一覧}
- 差分取得: `git diff {基準点}..HEAD`と未追跡ファイルを含む作業ツリー参照で差分を取得する
  - `{基準点}`は計画着手前コミットを指す不変参照点を選び、複数コミット計画でも基準点は変えない。
    amend/fixupを伴う場合はHEAD相対参照が刻々と変わるためコミットSHAを優先する

## 制約

- 担当観点ごとに網羅的に列挙すること。各エージェント定義「担当観点」節の全カテゴリを初回レビューで点検し、
  観点ごとに指摘の有無を報告する（指摘なしも明示する）
- 同一ファイル内で同種カテゴリの違反は重大度を問わず初出箇所のみで止めず該当ファイル全体から全件を列挙すること
- 指摘は差分起因（対象計画の改訂で新たに生じた違反）と既存違反を区別して報告すること。両者を別欄に分け、既存違反は補足情報として位置付ける
- 対象範囲のいずれかが1ファイル1000行を超える場合、`autocompact thrashing`予防のため次の制約を適用すること
  - 当該ファイルの全文`Read`を避け、`git diff`出力範囲・該当節（H2/H3単位）・差分周辺のシンボル単位・特定行範囲のみを`Read`する
  - 「該当ファイル全体から全件を列挙」観点は当該ファイル内で差分周辺と該当節範囲に限定される
  - 例外: `plan-impl-reviewer`起動時は本制約を適用せず、
    `references/impl-review-launch.md`「plan-impl-reviewerの分割起動（フォールバック時）」節の
    運用で担当分を1000行以内へ収める
- 構文の合法性を根拠とする指摘は機械チェック担当領域のため指摘対象外とすること。対象言語バージョンで有効化された新構文
  （Python 3.14のPEP 758による`except`括弧省略等）を学習知識との齟齬で構文エラーと判定しないこと
- named subagentとして`run_in_background=true`起動される場合、完了時に完了報告本文をSendMessage(to: 'main')で能動送付する義務を必須ゲートとする。
  `idle_notification(available)`のみでメイン要求を待つ挙動は未完遂扱いとし、SubagentStopフックがブロックする
```

## codex実装レビューの起動

`plan-codex-delegate`をAgentツールで`用途: 実装差分レビュー`指定で起動する。
渡す情報（対象範囲・対象外ファイル・計画ファイルパス・差分取得コマンド・担当カテゴリ）と
並列度・カテゴリ分担の閾値は`references/impl-review-launch.md`を参照する。
起動プロンプトの構成自体は`plan-codex-delegate`エージェント定義側が担うため、
本ファイルはカテゴリ分担と並列起動可否の判断のみを担当する。

## 指摘の統合と修正依頼

- 各サブエージェントの完了報告冒頭の`## 観点網羅`サマリー欄を検査し、未点検（`[ ]`）観点があれば再起動する
- 構文の合法性を根拠とする指摘は起動プロンプトの構文除外規定により採用しない。
  取り下げ確定前に、対象プロジェクトの宣言バージョン（`requires-python`等）に適合する処理系・構文チェッカを明示指定して実機検証する。
  対象は例えばPythonの`uv run --python=<適合版>`起点の`ast.parse`、TypeScriptの`tsc --noEmit`、Rustの`cargo check`等とする
- PATH上の既定処理系は旧版が新構文をエラー判定し得るため使わない。真正の構文エラーは指摘採用でなく機械チェックで検出・修正し、再現しない指摘は`（メイン判断: 不対応 — 実機検証で構文エラー不成立）`の注記で取り下げる

各レビュアー出力（差分起因・既存違反の二区分）をメインが統合対象とし、対応要否は区分によらず重大度で判断する。
`致命的`・`重大`は既存違反でも対応し、`軽微`はメイン判断とする（区分情報は対応順序の参考で不対応根拠にしない）。
固定書式表の組み立て前に、レビュー指摘とは別に品質俯瞰棚卸しを1回実施する。
品質俯瞰棚卸しとは、登録済み手順に無い品質作業の点検を指す。
本棚卸しは`agent-toolkit:agent-standards`配下`references/quality-first-exceptions.md`
「正規追加経路の非対称原則」節の経路1に対応する実施地点である。
発見事項は固定書式表へ追加行として統合する。
該当なしの場合は「品質俯瞰棚卸し: 該当なし」と1行報告する。
選別結果は修正着手前に`agent-toolkit/rules/03-claude-code.md`「サブエージェントの活用」節の固定書式表に従い
表形式でターン内に提示する。
「重大度／観点」列は`致命的`・`重大`・`軽微`、「区分」列は差分起因・既存違反を記入し、表後に取り下げ件数と理由パターンを併記する。
ユーザー合意は待たず、提示と同一ターンまたは続くターンで修正実装へ進む。
指摘箇所が対象計画の範囲外でも`致命的`・`重大`は対応し、スコープ拡大等を理由とした不対応は禁止する
（`agent-toolkit/rules/01-agent.md`「品質最優先」原則に従う）。

重大度`致命的`・`重大`の指摘を採用して修正着手する前に、メインが指摘箇所を単体`Read`する。
表層の現象が設計意図上のディフェンシブ実装（初期値埋め直後の`assert`・冗長判定されやすい二重チェック・
呼び出し元前提条件の事前検査等）として成立し得るかを照合する。
成立し得る場合はサブエージェント分類ラベルを採用せず、ユーザー確認またはコード一次資料での再検証へ切り替える。
設計意図上の必要性が確認できた指摘は次の注記で不対応扱いを許容する: `（メイン判断: 不対応 — 設計意図上のディフェンシブ実装として成立）`。

specとimpl指摘が同一変更箇所で相反する場合、重大度が異なれば高い重大度を優先する。
同一重大度の場合に限りimpl指摘（品質修正）をspec指摘（計画適合性）より優先する。
不対応とした側の指摘はメイン判断で不対応扱いとし理由を注記する。

対応不要と判断した指摘には末尾に注記する。注記付き指摘は「修正指摘」欄に含めない。
注記書式は`（ユーザー判断: 不対応 — <理由>）`または`（メイン判断: 不対応 — <理由>）`とする。

派生・展開系指摘（重複除去・記述冗長削除・SSOT統合等）の対応で原フィードバック・原指摘の明示要求の反映を撤回する場合、
計画ファイル`### エージェント判断`または`## 進捗ログ`「### レビュー反映」ラベルへ撤回の根拠を明示記録する。
根拠が既存節での機能的等価カバーで説明できない場合は撤回せず原フィードバック要求の反映を維持する。

修正案の採用前の検証観点と確認手順は`references/modification-checks.md`に従い、
検証結果が不明な場合は当該案を採用せず既存先例と整合する別案を選ぶ。

再レビュー時の指摘が形式上の違い（追記か新節か、節順、見出し名の言い換えなど）に収束する場合は、実質的な再発防止効果がないことを確認したうえでメイン判断で不対応にしてよい。

修正指摘で追加文面案・置換後文面案を提示する場合、提示前にメイン側で機械チェックを実行し整形する。
機械チェックは`uvx pyfltr run-for-agent --commands=textlint,colloquial-check --enable=colloquial-check`等を指す。

修正再実装で実装ファイルを変更した場合、計画ファイル本文の関連箇所（`## 変更内容`配下のコードブロック・`### エージェント判断`記載・対象ファイル一覧）も同期更新する。
同期を委譲先で行うかメイン側amend前に行うかは修正指示で明示する。

対応が必要な指摘が1件以上あれば修正内容をユーザーへ簡潔に報告して修正再実装へ進み、完了後に検証タスクで合格を確認する。
修正実装はcodexを優先する。
詳細は`agent-toolkit/references/plan-impl/execution-process.md`「実装委譲（plan-codex-delegate / plan-implementer）の判断指針」節に従う。
進捗ログに`threadId`記録があればAgentツールで`plan-codex-delegate`（`用途: 実装`）サブエージェントを継続呼び出し
（`threadId`引き渡し）で同一スレッドへ差し戻す。
`threadId`記録が無い場合も`mcp__codex__codex`が利用可能なら新規に`plan-codex-delegate`（`用途: 実装`）サブエージェントへ委譲する。
MCPが利用不可の場合のみ修正再実装モードで`plan-implementer`へ委譲する。

修正のコミット反映は、直前の未プッシュコミットがレビュー対象計画と件名・対象範囲の両面で整合する場合に
`git commit --amend`で統合する（いずれか不一致ならamendせず新規コミット作成）。
amend後は`git status`で取り残しが無いことを確認する。

## 再レビュー

修正・コミット統合した場合は`重大`以上を含む修正で必ず起動する。成果物本体（コード・ドキュメント・計画ファイル
`## 変更内容`のコードブロック内文面）に影響しない軽微修正（計画ファイル地の文の日本語表現整え等）のみで構成される場合は再起動しない。
新しい`Agent`で並列起動する（先入観排除のため`SendMessage`再開は採らない）。対象範囲は修正再実装で書き換えた
個別ファイル群とし、メインが修正方針および完了後の`git status`／`git diff`から確定して起動プロンプトへ列挙する
（計画ファイル・永続化情報には記録しない）。
`plan-codex-delegate`も本項の新規`Agent`起動方式に従う（`threadId`継続は使わない）。
理由は`agent-toolkit/skills/review-standards/SKILL.md`「レビューの基本姿勢」節の独立評価原則を実装レビューへ適用するためである。
本節の上限なしサイクルモデルは`codex-review.md`の反映確認レビュー上限（初回+1回）と不整合であり、
計画ファイルレビューの`threadId`継続方式とも非対称とする。

起動対象は次のいずれかに該当する種別に限定する。

- 初回レビューで対応した指摘が1件以上あった種別（修正後の品質確認のため）
- 修正差分が担当観点に影響し得る種別（回帰確認のため）
  例: impl指摘の修正で計画範囲やドキュメント整合性が変わり得る場合はspec種別も対象に含める

両条件のいずれにも該当しない種別はスキップする。
起動プロンプトは「起動プロンプトテンプレート」節と同一形式とし、対象範囲欄のみ修正で書き換えた個別ファイル群に差し替え、
「再指摘の抑止対象」欄を「対象外ファイル」欄の直後に挿入する。抑止対象は観点（パターン類型）レベルで記述し既知具体例を併記する。
前ラウンドで採用指摘対応として意図的削除・意図的縮小のために修正した箇所は「観点＋具体的なファイル名・節名」形式で
「再指摘の抑止対象」欄へ全件転記する（初回レビューで対応済みの指摘は含めない）。
抑止対象は書式レベル（バッククォート等の表記差）と意味レベル（参照や識別子の存否）を分離して記述し、
書式レベルの抑止対象を記述する場合は対応する意味レベルを別項目として抑止対象外である旨を明示する。

再レビュー結果に対応要の指摘が1件以上あれば「指摘の統合と修正依頼」節へ戻り、修正再実装→コミット統合→再レビューを繰り返す。
レビュー回数によらず直すべき指摘は確認せず修正する。
メインが対処不可能と判断した場合のみ修正ループを中断してユーザーへエスカレーションする。
判断軸は同一指摘の再発・修正の品質低下・予期せぬ作業量の拡大などとする。

再レビューサイクル2以降の重大度`軽微`指摘は原則メイン判断で不対応とする。
採用は構造的問題（対応規範違反・既存記述齟齬・修正による新規違反の混入など）に限り、
表記揺れ・スタイル懸念・あると望ましい程度の指摘は不採用とし取り下げ理由欄へ記録する。
当該サイクルで採用指摘が0件なら指摘0件扱いとして修正ループを終了し、後続工程へメイン判断で進む。

サイクル3以降で検出された指摘が全て軽微であり、かつ次のいずれかに該当する場合、
当該サイクルで採用指摘なしと即座に判定して修正ループを終了する。

- `(a)` 前サイクル不対応判定の軽微指摘の派生・再指摘
- `(b)` 表記揺れ・コメント精度・重複分岐の指摘
- `(c)` 実装挙動に影響しないスタイル改善提案

`(b)`・`(c)`を適用する前に、対象違反箇所へプロジェクトの機械チェック
（`pyfltr`配下の`pylint`・`colloquial-check`・`textlint`等）を実行する。
機械チェックが違反として検出したものは`(b)`・`(c)`による不対応判定の対象外とし、修正して再レビューを継続する
（機械チェックが検出した違反はpush後CIで顕在化するため「実装挙動に影響しない」の暗黙前提が成立しない）。

サイクル3以降で新規重大指摘が連続する場合は継続レビューを打ち切る。メイン側で採否を確定し、
計画本文への反映または明示的な不対応判定の記録を完了させて当該サイクルを終了する
（完了基準は`plan-file-creator.md`「進め方」節と同一）。ユーザー再確認は不要とし、指摘がなければ作業を完了する。
