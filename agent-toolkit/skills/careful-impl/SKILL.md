# 計画後のセルフレビュー付き計画実行

`ExitPlanMode`後にコンテキストがクリアされた状態で呼び出される可能性がある。
そのため、計画ファイルが唯一の入力源として自立している必要がある。
サブエージェントへの委譲でも同様に計画ファイルのパスを省略しない。

1. タスク分解
2. 実装
3. 検証
4. コミット
5. レビュー

これらの工程では、計画ファイルを唯一の入力として参照する。
計画ファイルがフェーズ分けされていた場合は、「2. 実装」「3. 検証」「4. コミット」を各フェーズ分繰り返す。
各フェーズの完了判定と次フェーズの実装タスク起動はメイン側が担当し、
前フェーズのコミット完了を確認してから次フェーズへ進む（複数フェーズを並行起動しない）。
全フェーズ・全コミットが完了したら「5. レビュー」を計画全体に対して一度だけ実施する。

## 共通制約

全工程のサブエージェント呼び出しに共通する制約。

- gitコミット・pushは行わないこと（コミットはメインが直接行う）

## サブエージェント起動方針

careful-implフロー内で起動するサブエージェント（実装担当・初回レビュアー・修正再実装の担当・followupレビュアー）は
結果返却で`completed`状態に入る。
`completed`状態のサブエージェントへ`SendMessage`で追加指示を送ると失敗するため、
追加指示が必要となる場面（修正再実装・followupレビューなど）はすべて新規`Agent`起動で行う。
`agentId`保持や`SendMessage`再開経路は使わない。

- 修正再実装や再レビューが必要になったときは、計画ファイル位置・元タスク・修正指摘を
  省略せずプロンプトに含めて新規`Agent`起動する
- 新規起動のたびにモデルを選び直せるため、対象タスクの性質に応じて`haiku`・`sonnet`・`opus`から選定する

## 1. タスク分解

計画ファイルを起点にタスク分解し、`TaskCreate`で登録し、進捗は`TaskUpdate`で遷移させる。
タスク分解時に依存関係を洗い出し、検証・コミット・レビュータスクまで登録する。

タスク分解時は、サブエージェントの活用を前提とし、以下の点を考慮した適切な粒度に分解する。

- 時間効率（依存関係のない部分は並列化）
- コンテキスト効率（1エージェントあたりの作業量を増やし過ぎない）
- コスト効率（簡単なものは極力sonnet/haikuモデルを活用）

ただし、過度な分割はすべての効率を悪化させるため、細かく分割しすぎないことに注意する。
（それぞれのサブエージェントが計画ファイルや必要スキルなどを読むため）
小規模なタスクであればメイン側で直接実行した方がよい場合もある。
（文脈が限られたり精度の低いモデルで作業すると、成果物の品質が低下し、後の工程で手戻りが発生するため）

検証タスクは上記の効率を考慮して実装タスクとまとめてもよい。

ファイル削除（`rm`等の破壊的Bash操作）はサブエージェントへ委譲しない。
サブエージェントのBash権限は環境依存で拒否される事例があり、再開・新規起動を繰り返してもステータスが`blocked`で停止する。
削除対象が発生した場合は、計画時点で「メイン側のフォローアップタスクで実行する」運用を既定とする。

ただし、レビュータスクは必ず独立させ、観点ごとに別のサブエージェントを起動する。
（実装の経緯を持ち込まず、クリーンな状態でレビューさせるため）
レビュータスクも、上記の効率を考慮し、1観点に対して複数のサブエージェントを起動してもよい。
また、最終判断はメイン側（opus/sonnet）が行う前提でより安価なモデルを使用してもよい。

コミットはメインが直接行う。（通常小規模なタスクのため）

サブエージェント起動時のモデルは、`Agent`ツールの`model`パラメーターで指定する
（`"haiku"`・`"sonnet"`・`"opus"`のいずれか）。
プロンプト本文にモデル名を書いてもサブエージェントの動作モデルは変わらない。
モデルの使い分け基準は次の通り。

- `haiku`: 単純で確定的な作業（明確な条件での検索・抽出、単純なファイル修正・置換など）
- `sonnet`: 軽度の推論が必要な作業（方向性の固まっている小規模なコーディング・文書記述・レビューなど）。既定
- `opus`: 創造的で難度の高い作業（バグの原因調査や方針が未確定のコーディングなど）。コスト最高のため最終手段

### 大規模変更のタスク分割

サブエージェント1回あたりのtool_uses上限に達するとusage limitエラーで作業が中断する。
数千行規模のファイル分割・複数サブモジュールに跨る変更を1サブエージェントで実施すると上限に達しやすいため、
タスク分解時に以下の観点で事前分割する。

- ファイル分割や大規模リファクタリングは、対象ファイル単位またはサブモジュール単位で別タスクへ分ける
- 1タスクの作業量がコード生成量で数百行を超える見込みなら、独立可能な部分を切り出して別タスク化する
- 切り出した各タスクは依存関係を明示し、並列化または順次実行を選択する

## 2. 実装

分解されたタスクに従い、実装タスクを`careful-implementer`へ委譲する。
`careful-implementer`のfrontmatterでは`agent-toolkit:coding-standards`と
`agent-toolkit:writing-standards`を自動ロードする。
そのため呼び出しテンプレートにスキル呼び出し指示は含めない。

下記テンプレートで`Agent`を新規起動する（修正再実装時も同テンプレートを使い、
`## 修正指摘`欄に統合済み指摘テキストを貼り付ける）。

```text
以下のタスクを実装してください。

- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`

## タスク

{タスクの具体的な記述}

## 修正指摘（該当時のみ）

{メインが統合した指摘テキストをここに直接貼り付ける}
```

## 3. 検証

全実装タスク完了後、検証を実施する。
検証は「1. タスク分解」節の効率判断に従い、実装タスクに統合しても独立タスクとして分離してもよい。
ただし検証コマンドは個別ファイル単位ではなく、計画ファイルの「検証手順」に列挙された対象範囲全体を1コマンドで実行する。
部分検証で合格判定すると、計画範囲全体での並列衝突や対称的な変更の不整合を拾い損ねる。
独立タスクとして`careful-implementer`へ委譲する場合は下記テンプレートで`Agent`を新規起動する。

```text
以下のタスクを実施してください。

- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`

## タスク

{検証内容の具体的な記述。例: `uvx pyfltr run-for-agent`を実行し、警告ゼロ・テストグリーンを確認する等}
```

## 4. コミット

各実装・検証タスクが合格したら、計画ファイルの`## 実行方法`節「コミット方針」項目に従ってコミットする。
複数フェーズ計画ではフェーズ単位の中間コミット、複数コミット計画ではコミット単位の中間コミットになる。

「全コミット完了」の判定はメイン側が行う。
判定基準は「計画ファイルの`## 実行方法`節『コミット方針』項目に列挙された全コミットが`HEAD`に反映された状態」である。
単一コミット計画の場合は当該1コミットを終えた時点、複数コミット計画では最終コミット完了時点が該当する。
判定後、次の「## 5. レビュー」へ進む。

## 5. レビュー

全コミット完了後、計画全体の差分を対象として一度だけレビューを実施する。
レビュー工程ではBashは読み取り系操作（`git diff`・`git status`・`git log`・`ls`・`rg`相当）に限定し、
ファイルへの書き込みは行わない。

### 初回レビュー（2種別並列）

`careful-spec-reviewer`と`careful-impl-reviewer`を常に並列で起動する。
計画ファイル内の起動要否指定は参照せず、両種別を必ず起動する。

### 初回レビュー呼び出しテンプレート

共通骨子を使い、種別固有の欄を差し替えて2エージェントを並列起動する。
テンプレートのプレースホルダー`{レビュー種別名}`・`{担当観点}`は次表の値で差し替える。

| 種別 | 起動エージェント | 担当観点 |
| --- | --- | --- |
| `spec` | `careful-spec-reviewer` | 計画ファイルと成果物の仕様適合性 |
| `impl` | `careful-impl-reviewer` | コード単体品質・ドキュメント単体品質・成果物間の整合性 |

`spec`の担当観点には、修正再実装を経た差分の場合、メインから渡された前回指摘文面が
改変・要約・省略されずに反映されているかの照合も含める。

```text
以下の実装について初回モードで{レビュー種別名}レビューを実施してください。

- 担当観点: {担当観点}
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
- 対象範囲: 計画全体（全コミット完了後の差分）
- 対象外ファイル: {一時ファイル一覧}
- 差分取得: `git diff {基準点}..HEAD`と、未追跡ファイルを含む作業ツリー参照で行う
  - `{基準点}`は計画着手前のコミットを指す不変の参照点を選ぶ。例: `origin/master`相当の上流ブランチHEAD、
    または計画着手前に控えておいた特定コミットSHA。複数コミット計画でも基準点は変えない
    - amend/fixupを伴う計画ではHEAD相対参照（`HEAD~1`等）は刻々と意味が変わる。
      基準点はコミットSHA（コミットが存在する限り不変）または計画開始時に明示的に作成した固定ref（タグや別名branch）を優先する。
      `origin/<branch>`を使う場合は計画期間中にfetchで動き得ないことを確認するか、
      その時点のSHAへ事前に固定して扱う

## 制約

- 指摘を断片的にせず、一度のレビューですべての問題点を洗い出すこと
```

### 指摘の統合と修正依頼

各レビュアーの戻り値本文を受け取り、メインが指摘を統合する。
必要に応じてユーザーへ確認して対応方針を決定する（計画に照らし合わせて妥当な修正であれば確認無しで修正してよい）。

最終的に対応が必要な指摘が1件以上あれば、どのような修正をするかをユーザーに簡潔に報告したうえで、
修正再実装を`careful-implementer`へ依頼する。
依頼方法は「## 2. 実装」節のテンプレートに従い、`## 修正指摘`欄に統合済み指摘テキストを貼り付けて
新規`Agent`を起動する。
修正の再実装が完了したら検証タスクを新規起動して合格を確認し、
直前の未プッシュコミットへ`git commit --amend`で統合する
（`agent-toolkit`配布ルールの未プッシュコミット運用に従う）。

### 再レビュー（followupモード）

修正をamend統合した後、初回レビューと同じ`careful-spec-reviewer`・`careful-impl-reviewer`を並列で起動し、
followupモード（前回指摘＋修正差分を入力とする再評価）として呼び出す。
両エージェントは前回指摘ごとに`addressed`／`partial`／`missing`を判定し、
修正の副作用で生じた回帰のみ`regression`欄に記載する。新規指摘は追加しない。

「対応不要」と判断した指摘は、判断者と理由を統合済みテキスト内の当該指摘の行末尾に注記する。
注記書式は判断主体に応じて使い分ける。

- ユーザーへ確認した結果として対応見送りとした場合: `（ユーザー判断: 不対応 — <理由>）`
- メイン側で計画と照らして対応見送りと判断した場合（実害が低い・コストが過大などをメインが判断した場合）:
  `（メイン判断: 不対応 — <理由>）`

どちらの注記も対応確認対象から除外し、「修正指摘」欄から除外する。

### followupモード呼び出しテンプレート

下記テンプレートで`Agent`を新規起動する。
「## 前回指摘」欄に統合済み指摘テキストを転記する（注記付きの指摘も行を抜かず転記する）。

```text
以下の実装についてfollowupモードで再レビューを実施してください。

- 担当観点: {担当観点}
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
- 対象範囲: 計画全体（全コミット完了後の差分）
- 差分取得: `git diff {基準点}..HEAD`と、未追跡ファイルを含む作業ツリー参照で行う
  - `{基準点}`は初回レビューと同一の参照点を使う

## 前回指摘

- `path/to/file.py:L10` — 戻り値の型アノテーションが欠けている
- `path/to/file.py:L20` — 変数名が不明瞭 （ユーザー判断: 不対応 — 現状仕様で問題なし）
- `path/to/file.py:L30` — エラー型を`InvalidUploadError`へ変更する
- `path/to/file.py:L40` — テストの境界値が古い上限を参照したまま （メイン判断: 不対応 — 既存挙動を維持するため）

## 制約

- 新規指摘は追加しないこと（初回レビューで網羅されている前提）
- 前回指摘ごとに`addressed`／`partial`／`missing`を判定すること
- 修正の副作用で生じた明らかな回帰のみ`regression`欄に記載すること
- 全件`addressed`の場合は短文応答で合格を伝えてよい（例: `全件addressed。回帰無し。`）
```

### 修正ループのエスカレーション

`partial`または`missing`を1件以上含む結果が返った場合、修正対象として扱う。
`regression`は修正の副作用で生じた明らかな回帰のみを記録する欄であり、検出された場合は例外的に修正対象に加える。
通常の新規指摘はfollowupモードでは追加しない。
この「妥当な未対応指摘が残った状態」が連続2回発生した場合、修正ループを中断してユーザーへエスカレーションする。

全レビュー合格で作業終了。
