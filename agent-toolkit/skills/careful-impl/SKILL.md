# 計画後のセルフレビュー付き計画実行

本スキルは、`agent-toolkit:plan-mode`での計画ファイル合意後の、計画実行手順を示す。

1. タスク分解
2. 実装
3. 検証
4. コミット
5. レビュー

これらの工程では、計画ファイルを唯一の入力として参照する。
計画ファイルがフェーズ分けされていた場合は、「2. 実装」「3. 検証」「4. コミット」を各フェーズ分繰り返す。
各フェーズの完了判定と次フェーズの実装タスク起動はメイン側が担当し、
前フェーズのコミット完了を確認してから次フェーズへ進む（複数フェーズを並行起動しない）。
全フェーズ・全コミットが完了したら「5. レビュー」を計画全体に対して一度だけ実施する。

## 共通制約

全工程のサブエージェント呼び出しに共通する制約。

- gitコミット・pushは行わないこと（コミットはメインが直接行う）

## サブエージェント再開機能の楽観試行とフォールバック

実験的機能のサブエージェント再開（`SendMessage`ツール）を活用してトークン消費と文脈断絶を抑える。
利用可否は事前判定せず、`agentId`が手元にあれば`SendMessage`を楽観的に試行する。
ツール未提供等で失敗した場合は、同等内容の新規`Agent`起動に切り替えて再依頼する。

- `Agent`ツールでサブエージェントを起動した際、戻り値に含まれる`agentId`を、
  対象（フェーズ単位の実装担当・観点別レビュアーなど）と紐付けてメイン側で保持する
- 同一サブエージェントへ追加指示を渡す場面では、対応する`agentId`を選んで`SendMessage`を呼び出す。
  `SendMessage`の`message`欄には追加指示のみを送る（前回コンテキストに残る計画ファイル位置や元タスクは省略してよい）
- `SendMessage`が失敗した場合は、同等内容を新規`Agent`起動として組み立てて再依頼する。
  新規起動経路では計画ファイル位置・元タスク・修正指摘などを省略せずプロンプトに含める。
  起動した新規エージェントの`agentId`で従来の追跡情報を上書きし、以降の追加指示も新しいIDで行う
- 再開時はモデルを初回起動時から変更できない。異なるモデルで処理する必要が生じた場面では、
  `SendMessage`を使わず新規`Agent`起動を選ぶ

## 1. タスク分解

計画ファイルを起点にタスク分解し、`TaskCreate`で登録し、進捗は`TaskUpdate`で遷移させる。
タスク分解時に依存関係を洗い出し、検証・コミット・レビュータスクまで登録する。

タスク分解時は、サブエージェントの活用を前提とし、以下の点を考慮した適切な粒度に分解する。

- 時間効率（依存関係のない部分は並列化）
- コンテキスト効率（1エージェントあたりの作業量を増やし過ぎない）
- コスト効率（簡単なものは極力sonnet/haikuモデルを活用）

ただし、分割のし過ぎは全ての効率を悪化させるため、細かく分割し過ぎないことに注意。
（それぞれのサブエージェントが計画ファイルや必要スキルなどを読むため）
小規模なタスクであればメイン側で直接実行した方がよい場合もある。
（文脈が限られたり精度の低いモデルで作業すると、成果物の品質が低下し、後の工程で手間が増えるため）

検証タスクは上記の効率を考慮して実装タスクとまとめてもよい。

ただし、レビュータスクは必ず独立させ、観点毎に別のサブエージェントを起動する。
（実装の経緯を持ち込まず、クリーンな状態でレビューさせるため）
レビュータスクも、上記の効率を考慮し、1観点に対して複数のサブエージェントを起動してもよい。
また、最終判断はメイン側（opus/sonnet）が行う前提でより安価なモデルを使用してもよい。

コミットはメインが直接行う。（通常小規模なタスクのため）

サブエージェント起動時のモデルは、`Agent`ツールの`model`パラメーターで指定する
（`"haiku"`・`"sonnet"`・`"opus"`のいずれか）。
プロンプト本文にモデル名を書いてもサブエージェントの動作モデルは変わらない。
モデルの使い分け基準は次の通り。

- `haiku`: 単純で確定的な作業（明確な条件での検索・抽出、単純なファイル修正・置換など）
- `sonnet`: 軽度の推論が必要な作業（方向性の固まっている小規模なコーディング・文書記述・レビューなど）。既定
- `opus`: 創造的で難度の高い作業（バグの原因調査や方針が未確定のコーディングなど）。コスト最高のため最終手段

### 大規模変更のタスク分割

サブエージェント1回あたりのtool_uses上限に達するとusage limitエラーで作業が中断する。
数千行規模のファイル分割・複数サブモジュールに跨る変更を1サブエージェントで実施すると上限に達しやすいため、
タスク分解時に以下の観点で事前分割する。

- ファイル分割や大規模リファクタリングは、対象ファイル単位またはサブモジュール単位で別タスクへ分ける
- 1タスクの作業量がコード生成量で数百行を超える見込みなら、独立可能な部分を切り出して別タスク化する
- 切り出した各タスクは依存関係を明示し、並列化または順次実行を選択する

### サブエージェントagent IDの保持

修正再実装やfollowupレビューで再開経路を選べるようにするため、
タスク分解時には以下の対応関係を想定して設計し、起動時にはその単位で取得した`agentId`を保持する。

- フェーズ分割や複数並列起動した実装担当エージェントは、フェーズ単位で`agentId`を保持する。
  修正指摘の所属フェーズに応じて該当エージェントを再開する
- 仕様適合・単体品質レビュアーは観点別の2エージェントとして起動するため、
  各観点の`agentId`を初回レビュー時に保持する。followup時は対応する`agentId`を再開する
- 再開時はモデルが初回起動から不変である点を踏まえ、初回起動時のモデル選択は将来の修正再実装まで含めて妥当性を判断する

## 2. 実装

分解されたタスクに従い、実装タスクを`careful-implementer`へ委譲する。
`careful-implementer`のfrontmatterでは`agent-toolkit:coding-standards`と
`agent-toolkit:writing-standards`を自動ロードする。
そのため呼び出しテンプレートにスキル呼び出し指示は含めない。

新規起動経路では下記テンプレートで`Agent`を起動する（修正再実装時も同テンプレートを使い、
`## 修正指摘`欄に統合済み指摘テキストを貼り付ける）。
戻り値の`agentId`は「## サブエージェント再開機能の楽観試行とフォールバック」節に従って保持する。

```text
以下のタスクを実装してください。

- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`

## タスク

{タスクの具体的な記述}

## 修正指摘（該当時のみ）

{メインが統合した指摘テキストをここに直接貼り付ける}
```

修正再実装で再開経路を使う場合、保持した`agentId`へ`SendMessage`で次の指示を渡す。
元タスクと計画ファイル位置は前回コンテキストに残っている前提のため、修正指摘テキストだけを送る。

```text
以下の修正指摘に対応してください。

## 修正指摘

{メインが統合した指摘テキストをここに直接貼り付ける}
```

## 3. 検証

全実装タスク完了後、検証を実施する。
検証は「1. タスク分解」節の効率判断に従い、実装タスクに統合しても独立タスクとして分離してもよい。
独立タスクとして`careful-implementer`へ委譲する場合は、新規起動経路または再開経路を選ぶ。

新規起動経路では下記テンプレートで`Agent`を起動する。

```text
以下のタスクを実施してください。

- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`

## タスク

{検証内容の具体的な記述。例: `uvx pyfltr run-for-agent`を実行し、警告ゼロ・テストグリーンを確認する等}
```

実装担当エージェントを再開して検証させる場合、保持した`agentId`へ`SendMessage`で次の指示を渡す。

```text
以下の検証を実施してください。

{検証内容の具体的な記述}
```

## 4. コミット

各実装・検証タスクが合格したら、計画ファイルの`## 実行方法`節「コミット方針」項目に従ってコミットする。
複数フェーズ計画ではフェーズ単位の中間コミット、複数コミット計画ではコミット単位の中間コミットになる。

「全コミット完了」の判定はメイン側が行う。
判定基準は「計画ファイルの`## 実行方法`節『コミット方針』項目に列挙された全コミットが`HEAD`に反映された状態」である。
単一コミット計画の場合は当該1コミットを終えた時点、複数コミット計画では最終コミット完了時点が該当する。
判定後、次の「## 5. レビュー」へ進む。

## 5. レビュー

全コミット完了後、計画全体の差分を対象として一度だけレビューを実施する。
レビュー工程ではBashは読み取り系操作（`git diff`・`git status`・`git log`・`ls`・`rg`相当）に限定し、
ファイルへの書き込みは行わない。

### 初回レビュー（2種別並列）

`careful-spec-reviewer`と`careful-impl-reviewer`を常に並列で起動する。
計画ファイル内の起動要否指定は参照せず、両種別を必ず起動する。

### 初回レビュー呼び出しテンプレート

共通骨子を使い、種別固有の欄を差し替えて2エージェントを並列起動する。
初回レビューでは対象範囲の指摘をすべて洗い出すこと。
再レビュー（followupモード）では新規指摘を追加しない前提で、この初回で見つかった問題をすべて列挙する。

各エージェントの戻り値に含まれる`agentId`は、followup時の再開先として観点別に保持する
（「## サブエージェント再開機能の楽観試行とフォールバック」節）。

テンプレートのプレースホルダー`{レビュー種別名}`・`{担当観点}`は、レビュー種別ごとに次表の値で差し替える。

| 種別 | 起動エージェント | 担当観点 |
| --- | --- | --- |
| `spec` | `careful-spec-reviewer` | 計画ファイルと成果物の仕様適合性 |
| `impl` | `careful-impl-reviewer` | コード単体品質・ドキュメント単体品質・成果物間の整合性 |

```text
以下の実装について初回モードで{レビュー種別名}レビューを実施してください。

- 担当観点: {担当観点}
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
- 対象範囲: 計画全体（全コミット完了後の差分）
- 対象外ファイル: {一時ファイル一覧}
- 差分取得: `git diff {基準点}..HEAD`と、未追跡ファイルを含む作業ツリー参照で行う
  - `{基準点}`は計画着手前のコミットを指す不変の参照点を選ぶ。例: `origin/master`相当の上流ブランチHEAD、
    または計画着手前に控えておいた特定コミットSHA。複数コミット計画でも基準点は変えない

## 制約

- 指摘を小出しにせず、一度のレビューですべての問題点を洗い出すこと
```

### 指摘の統合と修正依頼

各レビュアーの戻り値本文を受け取り、メインが指摘を統合する。
必要に応じてユーザーへ確認して対応方針を決定する（計画に照らし合わせて妥当な修正であれば確認無しで修正してよい）。

最終的に対応が必要な指摘が1件以上あれば、どのような修正をするかをユーザーに簡潔に報告したうえで、
修正再実装を`careful-implementer`へ依頼する。
依頼方法は「## 2. 実装」節の経路選択に従う。
該当フェーズの`agentId`を保持していれば再開経路を選び、文脈の引き継ぎとトークン効率の改善を得る。
修正の再実装が完了したら検証タスクを再起動して合格を確認し、修正反映の追加コミットを行う。

### 再レビュー（followupモード）

修正反映コミット後、初回レビューと同じ`careful-spec-reviewer`・`careful-impl-reviewer`を並列で起動し、
followupモード（前回指摘＋修正差分を入力とする再評価）として呼び出す。
両エージェントは前回指摘ごとに`addressed`／`partial`／`missing`を判定し、
修正の副作用で生じた回帰のみ`regression`欄に記載する。新規指摘は追加しない。

ユーザーが「対応不要」と判断した指摘は、判断理由とともに統合済みテキスト内の当該指摘の行末尾に
`（ユーザー判断: 不対応 — <理由>）`の書式で注記する。
注記付きの指摘は対応確認対象から除外し、「修正指摘」欄から除外する。

### followupモード呼び出しテンプレート

経路選択は「## サブエージェント再開機能の楽観試行とフォールバック」節に従う。
観点別の`agentId`を保持していれば再開経路を試行し、失敗時に新規起動経路へフォールバックする。

新規起動経路では「## 前回指摘」欄に、初回レビューで管理している統合済み指摘テキストを転記する。
注記`（ユーザー判断: 不対応 — <理由>）`が付いている指摘も注記付きのまま転記する
（レビュー側で対応確認対象から除外される前提のため、転記時に行を抜かない）。

```text
以下の実装についてfollowupモードで再レビューを実施してください。

- 担当観点: {担当観点}
- 計画ファイル: `~/.claude/plans/{自動生成ファイル名}.md`
- 対象範囲: 計画全体（全コミット完了後の差分）
- 差分取得: `git diff {基準点}..HEAD`と、未追跡ファイルを含む作業ツリー参照で行う
  - `{基準点}`は初回レビューと同一の参照点を使う

## 前回指摘

- `path/to/file.py:L10` — 戻り値の型アノテーションが欠けている
- `path/to/file.py:L20` — 変数名が不明瞭 （ユーザー判断: 不対応 — 現状仕様で問題なし）
- `path/to/file.py:L30` — エラー型を`InvalidUploadError`へ変更する
- `path/to/file.py:L40` — テストの境界値が古い上限を参照したまま （ユーザー判断: 不対応 — 既存挙動を維持するため）

## 制約

- 新規指摘は追加しないこと（初回レビューで網羅されている前提）
- 前回指摘ごとに`addressed`／`partial`／`missing`を判定すること
- 修正の副作用で生じた明らかな回帰のみ`regression`欄に記載すること
```

再開経路では、初回レビューで提示した指摘テキストはエージェント側のコンテキストに残っている前提のため、
「## 前回指摘」欄の転記を省略する。
ユーザー判断「不対応」の指摘がある場合のみ、対応確認対象から除外する目印として
「## 不対応指摘の所在」欄に短縮形で渡す。不対応指摘がない場合は当該欄ごと省略する。
保持した`agentId`へ`SendMessage`で次の指示を渡す。

```text
followupモードで再レビューを実施してください。

## 不対応指摘の所在

- `path/to/file.py:L20`（ユーザー判断: 不対応 — 現状仕様で問題なし）
- `path/to/file.py:L40`（ユーザー判断: 不対応 — 既存挙動を維持するため）

## 制約

- 新規指摘は追加しないこと（初回レビューで網羅されている前提）
- 前回指摘ごとに`addressed`／`partial`／`missing`を判定すること
- 修正の副作用で生じた明らかな回帰のみ`regression`欄に記載すること
```

### 修正ループのエスカレーション

`partial`または`missing`を1件以上含む結果が返った場合、修正対象として扱う。
`regression`は修正の副作用で生じた明らかな回帰のみを記録する欄であり、検出された場合は例外的に修正対象へ加える。
通常の新規指摘はfollowupモードでは追加しない。
これら「妥当な未対応指摘が残った状態」が連続2回発生した場合、修正ループを中断してユーザーへエスカレーションする。

全レビュー合格で作業終了。
