# styles.md

## 対象読者と文体

成果物の対象読者を考慮し、適切な文体・表現を選ぶ。

| 読み手 | 文書／出力の例 | 文体 |
| --- | --- | --- |
| エンドユーザー | 対話型UI（CLI案内・GUIダイアログ・エラー画面・通知） | NHKスタイル |
| エンドユーザー | README・利用ガイド・`docs/guide/`配下 | JIS規格スタイル |
| 開発者 | アーキテクチャ・CONTRIBUTING | JIS規格スタイル |
| 開発者 | コーディングエージェントの発話（応答本文および拡張思考） | JIS規格スタイル |
| コーディングエージェント | コーディングエージェント向け文書[^1]・コードコメント | JIS規格スタイル |

[^1]: `CLAUDE.md`・`.claude/rules/`・`SKILL.md`・サブエージェント定義など、コーディングエージェントが直接参照する文書

- 横断修正ではファイルごとに対象読者・文体・粒度が変わるため、他ファイルの文体を流用しない。
  コメント・ログ・ドキュメント・コミットメッセージなど全ての文章は日本語で書く。
  例外は当該規範側で明記する

## 日本語の品質を保つ

あらゆる成果物の日本語は自然で簡潔な言い回し・文章スタイルで書くことを最上位の原則とし、以下のスタイルを用いる。

- 常体の場合: JIS規格・公的な標準仕様書のスタイル（以下、「JIS規格スタイル」と呼ぶ）
- 敬体の場合: NHKの案内放送原稿のスタイル（以下、「NHKスタイル」と呼ぶ）

すべての日本語は上記のスタイルに従い、表記の細則はJTF日本語標準スタイルガイドおよびJIS Z 8301:2019に準拠する。
詳細な対比例・違反パターンは`agent-toolkit:writing-standards`スキルを参照する。
日本語規範は3層で構成する。
本節の上位原則（原則層）・`agent-toolkit:writing-standards`の細則（細則層）・textlint等のlint群（機械層）とする。
原則層は横断的な判断基準を扱い、個別の書き方・表記ルールの新設は細則層へ配置する。
書き方の上位原則は以下とする（ジョージ・オーウェルの執筆原則に基づく。番号は原則6からの参照に用いる）。

- 原則1: 陳腐な比喩・決まり文句・借用文体・主観的評価・宣伝調・強調副詞を持ち込まず、観測可能な事実に基づく表現を選ぶ
  - 比喩動詞を避け、観測動詞・観測可能な事象名で記述する（「適用する」「実行する」「反映する」「原因」「契機」等）
  - 入力文体・引用文体へ追従しない。
    LLM特有の傾向（口語・俗語・敬体・絵文字・古い表記・慣用句・誇張表現・冗長な前置き・読み手への呼びかけ等）を持ち込まない
- 原則2: 短い言葉とフォーマル文書で一般的な表記を選び、長い言葉や不必要なカタカナ化・英字混在を避ける
  （バッククォート付き識別子・コマンド名は本規定の対象外）
- 原則3: 除去できる語をすべて除去し、同義言い換え・抽象主語・周辺列挙を排して1表現に集約する
  - 主語と述語を近づけ、主語直後に長い括弧列挙や挿入句を置かず、サブバレットや後続文へ分ける
  - 本文から数えて把握できる件数・列挙（「N件」「N観点」「N項目」など）は明記しない。
    いかなる理由があっても新規の機械チェック対象へ加えない
  - 除去で意味・適用範囲が変わる語（代表例の列挙に付く「など」「等」「を含む」等）は除去しない
- 原則4: 能動態が成立する箇所で受動態を使わず、動作主の明確な能動・規範述語で書く
- 原則5: 外国語・科学用語・専門用語で書かれた内容を平易な日常の日本語で表せる場合は日常語を選ぶ
- 原則6: 原則1〜5および細則層の規則に従うと不自然・不明瞭な文になる場合、自然さと明瞭さを優先し当該規則を破る。
  ただし機械検査対象の規則（textlint・markdownlint等）は破らない。
  不自然な適用が続く場合は`agent-toolkit:writing-standards`の`references/lint-relax-criteria.md`の判断基準に従い設定緩和を検討する

原則の適用にあたり、次の判断基準を本節で直接定める。

- 一段詳細な話を続けるときは「例えば」等の接続表現を付けて論理の飛躍を回避する
- 構成は結論を冒頭に置き、簡潔にまとめる。手順を段階的に分解した小見出しは必要最小限にとどめる
- 規範文書・スキル本文・サブエージェント定義などで例示・列挙・条件記述する場合は必要以上に限定しない
  - 代表例の列挙には「など」「等」「を含む」を付け、同類事象を含む解釈余地を残す
  - 「Aである場合のみ〜」より「Aである場合は〜」を優先し、排他的条件と非排他的条件を区別する
  - 規範の根本原因を捉え、特定の表層事象に限定した狭い記述を避ける
  - 「Xを根拠にYしない」「Xを理由にYしない」形式の禁止規定は、
    「Xでなければ`Y`してよい」と誤読される可能性がある。
    代替として「いかなる理由（例: X）があっても`Y`しない」等の全称否定形で書く。
    Xは代表例として括弧内へ置き、対象事象の全体集合を制約しない形にする

## JIS規格スタイル

常体を使用する文章はJIS規格・公的な標準仕様書の文体に統一する。
規範述語（「〜とする」「〜ものとする」「〜してはならない」「〜できる」）をそのまま踏襲する。

## NHKスタイル

敬体を使用する文章はNHKの案内放送原稿の文体に統一する。
命令調・断定的な責め口調・過度に事務的な表現を避ける。

## コーディングエージェントの発話

進捗・選択判断・成果物状況は情報量を保ち、すべて日本語を使用する。
結論ファーストで構成し、挨拶・社交辞令・絵文字を含めず、率直に記述する。
敬体を用いず常体で記述し、同調・称賛・迎合的な相づちを述べず、事実と判断のみを返す。
不安の喚起・緊急性の演出・選択肢の恣意的なフレーミングなど、ユーザーの判断を誘導する表現を用いない。
推奨を示す場合は根拠となる観測事実を併記する。

長時間の作業や多数のツール呼び出しを経た最終メッセージは、ユーザーが作業経過を確認していない前提で書き直す。

- ツール呼び出し間の作業メモに含まれる略記・略号・矢印チェーン・自作ラベル・実装識別子などを最終メッセージへ持ち込まない
- ファイル名・コミット・フラグなどの識別子は素の日本語で個別の節を設けて説明する
- 冒頭1文で本セッションの担当範囲の完了可否を言い切り、詳細説明を後に配置する（残作業なし、または残作業の列挙）
  - 別セッション・自動処理・CIなど後続処理が残る場合は、言い切りに続けて後続の主体と確認手段を1〜2文で添え、以降が自分の作業でないことを明示する
- 短さと明瞭さが衝突する場合は明瞭さを優先する
- 異常・障害事象（委譲先の成果物不整合・検証失敗・想定外エラー等）の報告は、
  抽象語（主観的・非具体的な状態描写語）でなく観測事実（何が・どの観測手段で・期待とどう相違したか）と
  影響・対処を列挙する構成で書く
