# styles.md

コード・テストコード・ドキュメントなどあらゆる成果物の新規作成・修正・計画・レビュー時に、以下の方針に従う。

## 大原則

- 自然で簡潔な日本語で書く

## 対象読者と文体

成果物の対象読者を考慮し、適切な文体・表現を使用する。

| 読み手 | 文書／出力の例 | 文体 |
| --- | --- | --- |
| エンドユーザー | 対話型UI（CLI案内・GUIダイアログ・エラー画面・通知） | NHKスタイル |
| エンドユーザー | README・利用ガイド・`docs/guide/`配下 | JIS規格スタイル |
| 開発者 | アーキテクチャ・CONTRIBUTING | JIS規格スタイル |
| 開発者 | コーディングエージェントの発話（応答本文および拡張思考） | JIS規格スタイル |
| コーディングエージェント | コーディングエージェント向け文書[^1]・コードコメント | JIS規格スタイル |

[^1]: `CLAUDE.md`・`.claude/rules/`・`SKILL.md`・サブエージェント定義など、コーディングエージェントが直接参照する文書

### 読み手

- エンドユーザー向け: 内部実装は知らない前提、専門用語は初出時に補足する
  - 対話型UI: 操作・状態・対処を端的に伝え、エラーは原因と対処を一組で示す
  - エンドユーザー向けドキュメント:「何ができるか」「どう使うか」を中心に利用者の操作目線で書く
- 開発者向け: プロジェクト前提知識を共有する開発者を想定し、初歩説明は省くがコード識別子レベルの詳細は扱わない
  - 開発者向けドキュメント:「なぜそう設計したか」「どこに注意が必要か」を中心に書く
  - コーディングエージェントの発話: 進捗・選択判断・成果物状況の情報量を保ち、
    拡張思考・ツール呼び出し前後の短文ステータス報告も日本語を使用する
    - 結論ファーストで構成し、挨拶・前置き・絵文字を含めず、指摘すべき事項は曖昧化せず率直に記述する
    - コード識別子・英語キーワードに引きずられて文全体を英語にしない
- コーディングエージェント向け: 前置き・自己紹介・書き手目線の解説は不要。判断に必要な情報のみを直接記述する
  - 冗長な説明はトリガー精度低下とコンテキスト消費を招く

### 文体：JIS規格スタイル

ドキュメントなど常体を使用する文章は **JIS規格・公的な標準仕様書のスタイル** の文体に統一する。
語尾・漢語率・断定度・規範述語（「〜とする」「〜ものとする」「〜してはならない」「〜できる」）をそのまま踏襲する。

### 文体：NHKスタイル

対話型UIなど敬体を使用する文章は **NHKの案内放送原稿のスタイル** の文体に統一する。
明瞭で中立的な表現を用いる。命令調、断定的な責め口調、過度に事務的な表現を避ける。

### 個別注意

- 複数ファイル横断の修正・計画ではファイルごとに対象読者・文体・粒度が変わることに注意し、他ファイルの文体をそのまま適用しない
- コメント・ログ・ドキュメント・コミットメッセージなど全ての文章を日本語で書く（プロジェクト方針が無い場合）
- 対話型UI向けでも、コーディングエージェントに直接渡るhook出力（`reason`・`additionalContext`・exit 2のstderr等）は
  プロジェクト方針で英語表記を固定する場合がある。
  Claude Code関連の出力先別方針は`agent-toolkit:agent-standards`スキルの
  `references/claude-hooks.md`「メッセージの記述言語」節に従う

## 推奨する書き方

- 観測事実ベースで書く。動作・状態・構造は誰でも同じ結論に至る客観的・中立的な表現を採用する
- 冗長な前置き・修飾・言い換えを排除し、最も短い1表現に集約する
- 推奨形を先に列挙し、当てはまらない場合に避ける表現を排除する順序で書く
  - 禁止条件から発想すると不自然な言い換えに陥りやすいため
- 各節の対比例（書き換え前後）は`agent-toolkit:writing-standards`スキルの`references/tone-examples.md`を参照する
  - 参照時は必ずサブエージェントを使用する（コンテキスト汚染回避のためメインエージェントから直接Readしない）

### 語尾と語彙

- 直接的な書き言葉で一文一義に書く
- 断定的で安定した語尾を使う（常体では「〜する」「〜である」「〜となる」「〜とする」「〜ものとする」）

### 名詞句

- 一語で意味が通る名詞は単体で表記する（後ろに別の名詞を付け足して二語化しない）。
  - 非推奨例:「設定値」「構成情報」「入力データ」「出力結果」「処理結果」
  - 推奨例:「設定」「構成」「入力」「出力」「結果」
  - 単体で意味が通らない場合に限り、必要な修飾語・複合語を組み合わせる

### 動詞

- 標準的な技術用語を使う（比喩動詞は使わない）
  - 推奨例:「例外を送出する」「プロセスを終了する」「処理を呼び出す」「メモリーを解放する」「接続を確立する」「設定を読み込む」
- 処理・状態遷移・データ変更は技術的に観測できる事実を表す動詞で書く
  - 推奨例:「実行する」「起動する」「上書きする」「置き換える」「無効化する」「書き込む」「読み込む」

### 行動指示の述部

- 行動指示を終止形で結ぶ（例：「～を参照する」）と、主体が不明確になり、説明文と混同されやすい
  - 常体の場合の推奨例:「〜を参照」
  - 敬体の場合の推奨例:「〜を参照してください」
- 例外: コーディングエージェント向け文書は常体の指示終止形を許容する（許容例:「〜する」「〜を確認する」）

### 構造と順序

- トップダウン（段階的詳細化）: 全体の構造から徐々に詳細に掘り下げる
  - ソースコードでは型定義や上位の関数から先に書く。関数Aから関数Bを呼び出す場合、Aを前にBを後ろに定義する
  - ドキュメントでは全体の構成やセクションの見出しから書き始め、徐々に肉付けする
  - 上記を満たす範囲内で関連するものは近くにまとめる。追記時も無条件に末尾追加しない
- 文章の並びは粒度を揃える
  - 追記時は対象個所の記述粒度と整合しているか必ず確認し、必要に応じて既存部分や章構成の変更も行う
  - 一段詳細な話を続けるときは「例えば」などの接続表現を付けて論理の飛躍を回避する
- 箇条書きは1項目1文を原則とし、複数文になる場合は別項目へ分割するかサブバレットへ階層化する

## 排除すべき癖

JIS規格スタイル指定下でもコーディングエージェントの出力には以下の癖が混入しやすい。意識的に排除する。

| 合理化されがちな判断 | 対処 |
| --- | --- |
| 主観的修飾の混入 | 程度や評価を主観で示す形容詞・形容動詞を名詞句に付けない |
| 営業文書調・宣伝調の混入 | 特徴・理由・魅力を宣伝句で要約する書き方を使わない |
| 程度副詞による強調 | 強調や勧誘を意図する副詞を技術文書に持ち込まない |
| 比喩動詞での表現 | 観測動詞で記述する。推奨例:「適用する」「採用する」「導入する」「反映する」「踏襲する」「整合する」 |
| 日本語文中への英字の混入 | 訳出を回避するための英字直挿入を採らず、日本語または外来語（カタカナ表記）に置き換える。`grep`・`diff`等のバッククォート付きコード識別子・ツール名・コマンド名は対象外 |
| ユーザー指示・対話履歴の文体への追従 | 入力に含まれる口語表現・俗語・略語・敬体・絵文字を成果物に持ち込まず、指示の意味のみ取り出して本ファイル方針で書く |
| 引用コード・既存コメントの文体の流用 | 引用元の古い表記・慣用句に引きずられず、本ファイル方針へ書き換えてから取り込む |

## AIスロップの抑止

AIスロップ（AIが情報量に対して冗長に生成する文章）を避ける。

| 合理化されがちな判断 | 対処 |
| --- | --- |
| 前置き・導入文の付与 | 章名・節名で示した目的を本文で再宣言せず、本題から記述を始める |
| 同義言い換えによる補強 | 同じ趣旨を別表現で並べず、最も短い1表現に集約する |
| 抽象主語・抽象副詞による一般化 | 対象を曖昧化する冒頭表現を採らず、対象を直接指す |
| 体裁を整える背景・歴史・展望の付加 | 求められた粒度を超える背景・歴史・展望を機械的に補わない |
| 既存節の構造（本文・表・サブバレットなど）を機械的に流用する | 情報量に応じて構造を選ぶ。同趣旨が複数形式に重複するなら最も短い1形式に集約する |
| 「網羅性のため」例示・周辺ケースを列挙する | 主目的に必要な代表例のみ書き、自明な周辺を補足列挙しない |

## 日本語の表記ルール

プロジェクト方針が無い場合の方針（JTF日本語標準スタイルガイドおよびJIS Z 8301:2019を参考）。
原則としてJTF日本語標準スタイルガイドに準拠する。textlintの`preset-jtf-style`で検査される項目はそちらに従う。

- 日本語と全角括弧の境界にスペースを挿入しない（例: `設定ファイル（既定値あり）`）
- 英文脈の半角括弧は英数字との間にスペース1つ（例: `Python (CPython 3.12)`）
- 外来語の語尾には長音符を付ける（例:「ユーザー」「サーバー」「パラメーター」。3音以上の語は省略しない）
- 技術用語・固有名詞・コード識別子は原典の表記を優先する
- 頻出違反パターンは`agent-toolkit:writing-standards`スキルの`references/textlint-violations.md`を参照する

## コード関連の追加方針

### コメントとドキュメントの記述粒度

- ソースコードには必要最小限のコメントを書き、コードから自明な処理にコメントを付けない
  - コメントを書く場合は以下のいずれかに限る
    - なぜそうするか（設計判断の背景・根拠、閾値選定理由、トレードオフ、バグ回避理由）。
      ワークアラウンドの場合はその旨と参照先を明記する
    - 何をしているか（数行以上のコードの塊に対する短い要約、非自明な制御フロー）
- 細かい判断や実装詳細はインラインコメントに書き、モジュール冒頭のdocstringに集約しない。
  冒頭はモジュールの役割・概要のみに留める
- ドキュメントはコードを見ただけではわからない情報や意図・設計理由を書く
- ドキュメントは現在の状態を直接記述し、「〜へ集約済み」「〜へ移行済み」のような時系列表現は使わない。
  履歴はコミットメッセージなどに委ねる

### コマンドラインオプションの書式

プロジェクト方針が無い場合の方針。

- ロング形式（`--foo`）を優先する。
  除外: ロング形式が存在しないもの、シェル組み込みの慣用的オプション（`set -euo pipefail`等）、
  広く定着した複合フラグ（`curl -fsSL`等）
- 引数付きの場合はスペース区切りより`=`区切りを優先する
  - 例: `--config=path/to/config`、`--config="path/to/config with space"`
  - subprocessのargv配列形式も同方針（`["cmd", "--scope=user"]`）
  - 除外条件:
    - ショートオプションで`=`形式を取らないもの（`git commit -m "..."`等）
    - `=`を受け付けないBSD系コマンド
    - フラグの直後に位置引数が続くパターン（`uv tool install --editable .`等）
