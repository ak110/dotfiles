# plan-impl-executor 工程1〜5の実体手順

`agent-toolkit/agents/plan-impl-executor.md`から起動時にReadで読み込まれるSSOT。
渡された計画ファイル1件を工程1〜5（タスク分解・実装・検収・検証・コミット・レビュー引き継ぎ判定）まで完遂する手順を定める。

計画ファイルの`## 実行方法`のレビューステップを読み取り、「レビューは実施しない」と記載されていればレビューをしない。
それ以外は記載のスキル・エージェントへ引き継いでレビューを実施する。

`## 変更内容`に列挙された全項目を初回実装で完遂する（「初回必須」「将来拡張」など段階化ラベルを付与した先送りは禁止する）。

`ExitPlanMode`によるユーザー承認は、計画ファイル全文の精読を保証しない。
ユーザーは`## 背景`・`## 対応方針`を中心に確認したうえで承認している場合が多く、
`## 変更内容`・`### エージェント判断`の細部や周辺対応までは未読の可能性がある。
実装段階で計画ファイル本文が判断指針（`01-agent.md`「コーディングエージェント（あなた）の振る舞い」節、
特に「品質最優先」「ユーザーとともに考える」）と衝突する場合の扱いは同節に従う。
plan-impl-executor固有の差分として、ここでの衝突は計画ファイル本文の明示記述と判断指針の明示記述が
文面上矛盾する場合に限る。自己の不安・疑念・主観的な違和感・自己推定上の制約は衝突に該当しない。
衝突を発見した時点で実装を止め、計画ファイル本文優先の暫定判断で続行し、
`agent-toolkit:autopilot`「4. TBD.md書式」に従い記録する。確認事項は完了報告の
`pending_confirmations`欄へ集約し、呼び出し元メインが検収後にユーザーへ確認する。

実行中にユーザーから新たな割り込み指摘・是正要求が発生した場合の扱いは、
`01-agent.md`「ユーザーとともに考える」節（割り込み指摘の即時報告・恒久化検討の並行実施、
工程4「恒久化検討」の即時実施）に従う。
固有の差分として、恒久化候補の確定文面は計画ファイルの`### 恒久化・リファクタリング内容`へ追記する。
割り込み指摘自体を「次セッションへの先送り」「`session-review`・`feedback`への先送り」として残置しない。

`ExitPlanMode`はユーザー承認応答の取得まで含めてワンステップで完結する。
承認応答を受領した時点で実装着手の合意が成立しているため、
「実装に進めてよいか」等の追加確認文を発さず、直ちに以下の工程1へ着手する。

以下の手順は省略不可の工程である。
列挙外の場面でも`01-agent.md`「品質最優先」原則に従う。

1. タスク分解
2. 実装
2.5. サブエージェント完了報告の検収
3. 検証
4. コミット
5. レビュー（レビューステップでレビュー実施が指定されている場合のみ。記載のスキル・エージェントへ引き継ぐ）

各フェーズの区切りの節目（1タスク完了・1コミット作成直前・レビュー指摘反映の節目など）ごとに、
計画ファイルの`## 進捗ログ`章へ`### <ラベル>`見出しと箇条書きで追記する。
書式詳細は`agent-toolkit:plan-mode`スキル配下の`references/plan-file-guidelines.md`の「進捗ログ」節に従う。
いずれのフェーズでも、計画ファイル本文（`## 変更内容`・`## 対応方針`等）の記載と乖離する確定形を採用した場合は、
`## 進捗ログ`への追記だけでなく当該本文を最終確定形へ更新する。
適用対象にはレビュー指摘対応によるdiffコードブロック内文面案の意味的変更も含む。
また`scratchpad`で改訂後最終形を構築した結果が計画文の
`### エージェント判断`記載のトリム内訳・抽出方針と乖離する場合も含み、
`scratchpad`完成時点（本ファイル反映前）で計画文を更新する。
`## 変更内容`配下の対象ファイル一覧の追加・除外・改名
（既存違反の同時修正・周辺対応の事後組み込み等）も乖離扱いに含み、
対象ファイル一覧のチェックボックス項目と各ファイル変更方針の`###`見出しを同時に追従して更新する。

実装着手前に計画ファイル本文の段取り記述（`## 対応方針`配下の`### 改訂作業の段取り`等）を読み返し、
「記録する」「反映する」等の進捗ログ追記要件を抽出してフェーズ末の記録対象に組み込む。
計画ファイル本文側で明示した個別の追記要件は、上記のフェーズ節目記録と並行して履行する。

## 1. タスク分解

計画ファイルを起点にタスク分解し、`TaskCreate`で登録する（検証・コミット・レビュータスクも含む）。
過度な細分化は避ける。ファイル削除（`rm`等）はサブエージェントへ委譲しない。
レビュータスクは観点ごとに別サブエージェントを起動する（実装の経緯を持ち込まないため）。

計画`## 変更内容`の対象ファイル数が5以上の場合、タスク分解に先立って計画着手前SHAから現HEADまでの
`git log --oneline <計画着手前SHA>..HEAD -- <path>`を対象ファイルごとに実行する。
並列セッション由来の重複コミット有無を確認する。
重複を検出した場合は当該項目を計画対象から除外するか、既存対応と統合する形へ設計変更する。

## 2. 実装

分解したタスクを下記「実装委譲（codex-impl / plan-implementer）の判断指針」に従って実装する。

### 実装委譲（codex-impl / plan-implementer）の判断指針

計画確定後の実装はcodex委譲（`agent-toolkit:codex-impl`）または`plan-implementer`委譲を優先する。
codex委譲は能力的にOpus相当扱いとし、コード・テストコード・一般ドキュメントの実装では最優先の委譲先とする。
メイン直接実装は小規模な修正のみを対象とし、それ以外は委譲する。
委譲可否は自己推定上のタスク規模・対象範囲・所要量で判断せず、
タスク内容の確定度（修正方針・対象ファイル・期待結果の明確さ）で判断する。
タスクごとに次の手順で委譲可否を決める。

1. タスクが小規模な修正（1行未満のtypo修正等）に該当するか見積もる
2. 小規模な修正のみメイン直接実装を選ぶ
3. それ以外を文書種別で分岐する（複数分岐に該当する場合は先に列挙した分岐を優先する）
   - コーディングエージェント向け文書全般（`AGENTS.md`・`CLAUDE.md`・`.claude/rules/`配下・
     `.claude/skills/`配下・`.chezmoi-source/dot_claude/`配下・`agent-toolkit/rules/`配下・
     `agent-toolkit/skills/*/SKILL.md`・`agent-toolkit/skills/*/references/`配下・
     `agent-toolkit/agents/*.md`など）の改訂を含むタスク、
     および既存記述との整合維持を要する改訂タスクは`plan-implementer`へ委譲する。
     `model`引数は`sonnet`を既定とし、難易度に応じて`haiku`（純粋な機械的置換のみ）または
     `opus`（修正方針が自明でない複数箇所修正など）を明示指定する。`effort`引数は`medium`を既定とする
   - コード・テストコード・一般ドキュメントの実装タスクは、`mcp__codex__codex`が利用可能なら
     `agent-toolkit:codex-impl`スキルを呼び出し、その手順に従いメインが直接codexへ委譲する
     （能力的にOpus相当扱いのため最優先）。
     MCPが利用不可（判定基準は`agent-toolkit:codex-impl`本文に従う）な場合は`plan-implementer`へ委譲し、
     `model`引数は`sonnet`を既定とし難易度に応じて`haiku`または`opus`を明示指定する
4. 委譲したタスクが`needs_escalation`を返したら本エージェント自身で引き上げて実装する。
   codex委譲では構造化ステータスを受け取らないため、codex応答全文の内容判断で
   対象外変更の必要性などを検知し、該当時は自身での実装または`plan-implementer`へ切り替える
5. 委譲タスクのうち、編集対象ファイルが独立する2件以上は並列起動する。
   `plan-implementer`委譲・codex委譲のいずれも並列対象とする。
   `plan-implementer`委譲は`run_in_background=true`で並列起動する。
   codex委譲の並列実行手順・formatter/linterの実行タイミング・検収方式は`agent-toolkit:codex-impl`本文を正典とする
6. codex委譲タスクの`threadId`の記録先・継続呼び出し時の参照は`agent-toolkit:codex-impl`本文
   「threadIdの管理」節を正典とする

タスク内容の確定度に迷うタスクは手順3に従ってまず委譲する。
検証タスクを委譲する場合は、対象検証コマンドを起動プロンプトの`## タスク`欄へ明記する。

### サブエージェント起動プロンプト

起動プロンプトの雛形は`agent-toolkit/references/plan-impl/launch-prompts-drafting.md`
「起草・改訂委譲雛形」節を機械転記して構築する。
埋め込み欄は計画ファイルパス・文書種別・担当範囲区分・タスク記述・修正指摘（修正再実装時のみ）・完了報告要件。
運用ガイダンス（`[乖離]`サブバレット報告確認・計画本文との整合維持・雛形本文の実在確認等）は
`launch-prompts-drafting.md`側の「運用ガイダンス」節へ集約する。

## 2.5 サブエージェント完了報告の検収

サブエージェント完了報告を受領した時点で、次のレビューまで保留せず即時に検収する。
codex委譲応答も本節の検収対象に含み、unplanned相当の計画外事項はcodex応答本文の内容判断で抽出する。
unplanned項目が含まれる場合は自身で採否判定する。
unplanned項目とは`[CLAUDE.md追記候補]`・`[即時相談候補]`・`[該当外]`のいずれかのラベルを持つ計画外の発生事項を指す。

- 完了報告のchanged一覧が計画ファイル`## 変更内容`当該節の独立変更項目（複数行・複数識別子の更新指示等）を網羅するかを照合する
  - 1ファイル内に複数箇所の更新指示がある場合は一部のみ更新して報告するケースがあるため`git diff`で差分実体を確認する
  - 計画ファイル`## 変更内容`の対象ファイル節から個別指示項目
   （テスト追加・`docstring`更新・`assert`追加・命名変更等）を抽出してチェックリスト化する
    - 各項目について実装差分（`git diff`実体）に該当変更が含まれることを1対1で照合する
    - 完了報告の`changed`欄が概念レベル（「既存テストのリネーム」等）で集約され
      項目抽出が困難な場合は、計画ファイル指示と実装差分を直接照合する代替経路を採用する
- 完了報告の`changed`欄で、起動プロンプト`## 完了報告要件`に従い各タスク項目の個別指示と対応箇所が
  1件につき1サブバレットの形で展開されているかを照合する
- unplanned項目が含まれる場合は計画方針および判断指針へ照らして即時採否判定する
- 採否を保留すべき事項は暫定不採用（計画に明記のない変更は適用しない）でTBD相当記録し、
  完了報告の`pending_confirmations`欄へ記録する
- 修正再実装の検収では、修正差分（`git diff`実体）が新規問題の混入を生じていないかも事後点検する。
  観点は`agent-toolkit:careful-review`「指摘の統合と修正依頼」節の修正案の採用前検証と同一とし、
  当該節を典拠とする
- 完了報告で計画本文と異なる担当ファイル・識別子・命名規約が採用されたことを検出した場合、
  unplanned項目採否判定と同一タイミングで、計画本文を同期更新する
  - 同期対象は`## 変更履歴`・`## 変更内容`・`### エージェント判断`
  - 同期更新は本エージェントの責務とし、後続レビュー工程へ持ち越さない

## 3. 検証

全実装タスク完了後に検証する（実装タスクへの統合も可）。
検証コマンドは計画ファイルの`## 実行方法`の検証ステップの対象範囲全体。

実装完了後、計画ファイルの対象ファイル一覧に
`agent-toolkit:agent-standards`「文書サイズ上限」節の対象文書が含まれる場合は次を実行する。
`uv run --script agent-toolkit/skills/plan-mode/scripts/check_wc_projection.py <計画ファイルパス>`。
`_ALLOWED_DRIFT = 2`を超える乖離検出時は、計画本文の見込み行数を実測値へ更新する。

## 4. コミット

計画ファイルの`## 実行方法`のコミットステップに従ってコミットする。

## 5. レビュー引き継ぎ先の記録

本エージェント自身はレビューを実施しない。計画ファイルの`## 実行方法`のレビューステップを
読み取り、記載のスキル・エージェント名（複数併記時は全名）、または「レビューは実施しない」を
完了報告の`review_handoff`欄へそのまま記録する。レビューの起動判断・並列起動要否・不変参照点や
一時ファイル一覧の受け渡しは呼び出し元（`agent-toolkit/references/plan-impl/caller-reception.md`手順）が担う。

起動前の準備（計画着手前SHAの記録）と完了報告の受領後の呼び出し元向け手順は
`agent-toolkit/references/plan-impl/caller-reception.md`を参照する。
