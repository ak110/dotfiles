# Refine Prompt

## いつ使うか

- skill・slash command・タスクプロンプトを新規作成・大幅改訂した直後
- エージェントが期待通りに動かず、原因を指示側の曖昧さに求めたいとき
- 重要度の高い指示（頻繁に使うskill・自動化の中核プロンプトなど）を堅牢化したいとき

## いつ使わないか

- 単発利用のプロンプト（subagent起動コストに見合わない）
- 再現可能な品質改善ではなく、書き手の主観的選好を反映したい場合

## ワークフロー

1. 対象プロンプトを確定し、シナリオを与えるかを決める。
   シナリオを与える場合はsubagentが対象プロンプトに従って実行し、
   解釈に迷った点・裁量補完点も踏まえて改善提案を返す。
   シナリオを与えない場合はsubagentが対象プロンプトを批判的に通読し、
   不明瞭・冗長・不足・誤りなどの観点で改善提案を返す。
2. 観点の多様性を確保するため2〜3個のsubagentを並列起動する。
   単一メッセージ内に複数のAgent呼び出しブロックを並べる。
   dispatch不能な環境の扱いは「環境制約」節を参照する。
3. 各subagentから戻った改善提案リスト（1個あたり最大3件）を集約する。
   並列subagent間で重複する提案があればメインが事前に統合する。
4. 集約した改善提案リストをまとめて処理する。詳細は「改善提案の判断方法」節を参照する。
5. 全提案の処理が完了したら、続行か終了かをユーザーに確認する。
   続行する場合は新規subagentで再評価する（同一subagentの再利用は禁止）。
   ユーザーが終了を告げた時点で停止する。

## subagent起動契約

subagentに渡すプロンプトは次の構造を取る。

````text
あなたは <対象プロンプト名> を前提知識を持たずに読む実行者です。

## 対象プロンプト
<対象プロンプトの本文を全文挿入する、または Read で読ませるパスを指定>

## シナリオ（任意）
<シナリオの状況設定を1段落。指定がなければ「シナリオなし。批判的読みのみ」と記載>

## タスク
1. シナリオが指定されていれば、対象プロンプトに従って実行し、解釈に迷った点・裁量で補完した点を記録する。
   シナリオが指定されていなければ、対象プロンプトを通読し、不明瞭・冗長・不足・誤りを洗い出す。
2. 改善提案を重要度の高い順にトップ3件まで後述のフォーマットで返す。
   重要度は「実害の大きさ・改善幅・適用頻度」を基準に判定する。
   範囲は発動条件・本文構造・記述精度・冗長性・命名・例示・整合性などを幅広く対象とし、
   加筆・書き換え・削除のいずれも提案してよい。
   文脈追跡を要する記述（「シナリオAで〜」のような他項目への暗黙参照）は使わず、
   各提案が単独で読み取れる粒度に揃える。

## 改善提案フォーマット（1件ごとに繰り返す）
### 提案 N: <変更対象を1行で>
- 変更理由: <なぜ変更すべきか1〜2文>
- 変更内容: 以下のdiff形式で示す。新規追加なら`+`のみ、削除なら`-`のみでよい。

  ```diff
  - <現在の文言>
  + <変更後の文言>
  ```

- 期待効果: <この変更で何が改善するか1文>
- 種別: 加筆 / 書き換え / 削除
````

## 改善提案の判断方法

メインは集約した提案リストを一括で処理する。

- 各提案を改善提案フォーマット（`### 提案 N:`以下の項目構造）のままテキストとしてユーザーへ転記する。
  Claude CodeのUIではsubagentの出力がユーザーに直接表示されないため、
  転記を省略するとユーザーは判断材料を見られない。要約・言い換えはせず原文に近い形で提示する。
- 採否は`AskUserQuestion`の`multiSelect`でまとめて問う。提案を最大3件ずつ1質問にまとめ、複数の質問を並べる。
  - 1呼び出しの上限は4質問・各質問4オプションのため、最大16件まで扱える。
  - 各オプションの`label`に「提案N: <変更対象>」、`description`に変更内容の要点を記載する。
  - チェックされた提案を採用、未チェックを却下として扱う。
  - 修正したい提案がある場合は自動付与の「Other」欄に番号と修正内容を書くよう、質問本文で案内する。
- 採用が確定した提案を対象プロンプトに反映する。「Other」欄に修正指示があれば、その内容に沿って反映する。
- 全提案の処理が完了したら、続行か終了かをユーザーに確認する。
  続行時は新規subagentで再評価する。

## 環境制約

新規subagentをdispatchできない環境
（既にsubagentとして動作している、Task toolが無効化されている等）では、本スキルを適用しない。
書き手自身による自己再読では構造的にバイアスが排除できない。

- 代替案1: 親セッションのユーザーに、別のClaude Codeセッションを起動して依頼してもらう
- 代替案2: 評価を断念し、ユーザーに「refine-prompt skipped: dispatch unavailable」と明示的に報告する

## Red flags

| 想定される合理化 | 実態 |
| --- | --- |
| 「自分で読み直せば同じ効果が得られる」 | 直前に書いた文章を客観視することはできない。必ず新規subagentをdispatchする |
| 「同じsubagentを再利用しよう」 | 前回の改善提案を学習している。継続時は毎回新規にdispatchする |
| 「シナリオを修正に合わせて調整しよう」 | 不明瞭点が解消したように見せるためにシナリオを容易化するのは本末転倒である |
