# セッション振り返り

ユーザーが任意のタイミングで実行する手動起動の振り返りスキル。
今回のセッションの内容を振り返り、今後のセッションをより効率的に進めるために有用な情報を適切な場所へ記録することを目的とする。

## ステップ1: Reflect

候補を挙げる対象として、まず以下の観察源をセッションから洗い出す。

- ユーザーからの割り込み・指摘・修正
- Edit/Writeが期待通り反映されなかった事象
- hookでブロックされた事象

そのうえで、スムーズに進めるためにどのような情報が不足していたか確認する。

- プロジェクト固有の知見の不足
- コーディングエージェントが判断に迷った場面の存在
- CLAUDE.mdなどによるコーディングエージェントへの指示の不足・矛盾
- その他、セッションの進行を妨げた要因の存在

## ステップ2: Draft Additions

ステップ1の観察源と確認内容をもとに、どのような情報をどこに追加すべきか検討する。

- プロジェクトドキュメント（CLAUDE.md、README.md、docs/配下など）への追加
- pyfltrの挙動・メッセージに関する改善提案
- agent-toolkitのスキル・ルールに関する改善提案

各候補は以下の4点を自己検証し、4点すべて通過したものだけを提示する。

- (a) 表層の現象ではなく根本原因まで遡って分析する
- (b) 再発リスクと影響度を見積もる（単発・偶発の事象は除外）
- (c) 既存のCLAUDE.md・ルール・スキルで既に対処済みでないか確認する
- (d) 追記先はコードに近い順（docstring/インラインコメント → CLAUDE.md/.claude/rules → .claude/skills）で
  最適なものを選び、「書きやすい場所」に倒さない

## ステップ3: Show Proposed Changes

ユーザーへ改善提案を提示する。

フォーマットは以下の通り。

```markdown
## プロジェクトドキュメント改善提案

- 対象ファイル — 提案内容
- 対象ファイル — 提案内容
- ...

## pyfltr改善提案

- 対象ファイル — 提案内容
- ...

## agent-toolkit改善提案

- 対象ファイル — 提案内容
- ...
```

各項目はセクション見出し配下に1項目1行で「- 対象ファイル — 提案内容」の形で書く。
1コンセプト1行で簡潔に書く。
提案が無い項目には同見出し配下に「指摘無し」とのみ書く。

当該セッションで利用しなかった項目（例: pyfltr未使用）はスキップしてよい。
各提案文は本セッションの会話履歴を知らない読者にも意味が通るよう自己完結させる
（会話履歴への暗黙参照を避け、観測事象を直接書く）。

### 1. プロジェクトドキュメント

CLAUDE.md、README.md、docs/配下など、プロジェクト固有のドキュメント全般。

今後の作業に必要な、本セッションで判明したプロジェクト固有の知見のみを対象とする。
観点はbashコマンド・コードスタイル・テスト手法・環境設定の傾向・警告／陥穽・繰り返された修正指示・
コーディングエージェントの振る舞い改善（判断・確認・計画粒度等のルール化で再発防止できた場面）など。

### 2. pyfltr

対象: pyfltr本体の挙動・メッセージ。
反映はユーザーが別途行う前提のため、提案までにとどめる。

pyfltrプロジェクトで作業中の場合に限り、プロジェクトドキュメントの項と統合する。

### 3. agent-toolkit

対象: `agent-toolkit`プラグイン（スキル・フック・サブエージェント。`skills/pyfltr-usage/SKILL.md`を含む）と
`~/.claude/rules/agent-toolkit/`配下のルール。
反映はユーザーが別途行う前提のため、提案までにとどめる。

dotfilesプロジェクトで作業中の場合に限り、プロジェクトドキュメントの項と統合する。

## ステップ4: Apply with Approval

プロジェクトドキュメント改善提案のみ、
ユーザーに変更を適用するかどうかを確認する。

ユーザーが変更を希望した場合、修正案をdiff形式のコードブロックで提示し、
ユーザーが承認したら変更を適用する。
