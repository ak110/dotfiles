# セッション振り返り（dotfiles拡張章）

## 起動方針

本スキルはdotfiles利用環境専用の拡張スキルとし、`agent-toolkit:session-review`スキルと必ず併用する。
Stopフックの誘導でどちらか一方のみを起動して報告を終えてはならない。
両スキルを起動したうえで、両者の章を1つのレポートにまとめて提示する。
提示し得る章はプロジェクトドキュメント章・pyfltr改善提案章・dotfiles改善提案章で、
各章は条件に応じて省略または上位章へ統合される。
各章の取捨は「提示フォーマット」節の規定に従う。
取捨対象は次の通りとする。

- pyfltr未使用時のpyfltr改善提案章の省略
- 作業中プロジェクトに応じた改善提案章のプロジェクトドキュメント章への統合
  - pyfltrプロジェクトで作業中の場合はpyfltr改善提案章を統合する
  - dotfilesプロジェクトで作業中の場合の統合先・見出し扱いは`### dotfiles改善提案`節を参照

Stopフックから呼ばれた場合の終了判定基準は`agent-toolkit:session-review`スキルの「起動方針」節に従う。
終了でないと判断した場合は両スキルとも振り返りを実施せず、作業継続または最終応答へ戻る旨を1文で示して終了する。

## 提示フォーマット

`agent-toolkit:session-review`スキルのステップ3で示すプロジェクトドキュメント章に続き、
以下のフォーマットで追加する。
pyfltr改善提案章はdotfiles利用環境かつpyfltr使用時のみ、
dotfiles改善提案章はdotfiles利用環境のみ出力する。

```markdown
## pyfltr改善提案

[総括コメント]

### [提案要約]

- 理由: [提案する根拠]
- 反映先: [ファイル＋節または該当箇所]
- 反映内容: [反映する具体的な内容]
- メリット: [採用時に得られる効果]
- デメリット: [採用時のコスト・副作用・運用上の負担]

## dotfiles改善提案

[総括コメント]

### [提案要約]

- 理由: [提案する根拠]
- 反映先: [ファイル＋節または該当箇所]
- 反映内容: [反映する具体的な内容]
- メリット: [採用時に得られる効果]
- デメリット: [採用時のコスト・副作用・運用上の負担]
```

各提案は`### [提案要約]`見出し配下に「理由」「反映先」「反映内容」「メリット」「デメリット」を箇条書きで記述する。
対象ファイル・対象節は「反映先」項目で明示する。
提案が無い章には同見出し配下に「提案無し」とのみ書く（`[総括コメント]`スロットも省略する）。
取捨対象は「起動方針」節の規定に従う。
見出し直下は箇条書き項目のみとし、項目外の地の文を追加しない。
理由・補足は全て理由欄へ統合する。

### 同一ターン内発行

提示後は次ターンへ持ち越さず、本ターン内で「ステップ4の適用範囲」節の手順を実行する。

自己完結性の要件（観測した具体事象・改善後の振る舞いと根拠の明記、暗黙参照表現の禁止）は
`agent-toolkit:session-review`スキルのステップ3に従う。
本拡張章の提案も同スキルのステップ2の自己検証に従う。

### pyfltr改善提案

対象: pyfltr本体の挙動・メッセージ。

pyfltrプロジェクトで作業中の場合は、`agent-toolkit:session-review`スキルのプロジェクトドキュメント章へ統合する。

### dotfiles改善提案

dotfiles改善提案の対象は、dotfiles配布物の全般とする。具体的には次を含む。

- `agent-toolkit`プラグイン（スキル・フック・サブエージェント・rules・agents・scripts）
- `{dotfiles}/.chezmoi-source/dot_claude/`配下（配布先`~/.claude/`相当のグローバル配布元）
- `{dotfiles}/agent-toolkit/`配下（プラグイン本体）
- `~/.claude/rules/agent-toolkit/`配下（配布先）
- dotfiles編集者向け設定・スクリプト（`pytools/`・`bin/`・`scripts/`・`pyproject.toml`等）

dotfiles未利用環境ではこの章を出力しない。
dotfilesで作業中の場合は本章を`## プロジェクトドキュメント改善提案`章へ統合し、
「dotfiles改善提案」の見出しは書かない。

## ステップ4の適用範囲

本拡張章は`agent-toolkit:session-review`スキルのステップ4を完全に上書きする。
上書き手順は次のとおり。

1. `atk fb status`を実行し、feedback-inboxが有効な場合のみステップ2以降を実行する。
   非ゼロ終了時は改善提案の投入を省略し進捗ログのみ記録する
2. ステップ3の改善提案本文をユーザー可視の応答として発行した直後、
   `atk fb add --source=session-review`を各改善提案について連続実行する
   - ステップ3で提示した各改善提案の本文をそのまま`atk fb add --source=session-review`の引数として用いる。
   - 引数化する範囲は`### <提案要約>`見出しから次の`###`直前までとし、要約・省略・書式変換をしない。
   - 引数へはtempファイルの生パス文字列をそのまま渡さない。本文をtempファイル経由で構築した場合は`$(cat /tmp/xxx.md)`のようにシェル展開で本文へ変換してから渡す。生のファイルパス文字列を位置引数として渡すと`atk fb add`のCLI側で拒否される（呼び出し側は本文文字列を位置引数として受け取る仕様のため）。
3. 投入完了後は`agent-toolkit:exit-session`スキルへ遷移せず、通常の完了応答へ戻る。
4. 「以上の改善提案を検討します」等の承認待ちと解釈される表現をステップ3応答末尾へ置かない。
