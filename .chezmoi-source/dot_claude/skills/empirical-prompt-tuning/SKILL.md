# Empirical Prompt Tuning

プロンプトの品質は書き手自身には評価できない。
書き手が明瞭だと判断した箇所ほど、別のエージェントが読むと解釈に迷う。
バイアスを排した実行者に実際に実行させ、両面から評価して反復するのが本skillの中核である。
改善が飽和するまで継続する。

## いつ使うか

- skill・slash command・タスクプロンプトを新規作成・大幅改訂した直後
- エージェントが期待通り動かず、原因を指示側の曖昧さに求めたいとき
- 重要度の高い指示（頻繁に使うskill・自動化の中核プロンプトなど）を堅牢化したいとき

使わない場面:

- 単発利用のプロンプト（評価コストが見合わない）
- 成功率の改善が目的ではなく、書き手の主観的選好を反映したい場合

## ワークフロー

0. Iteration 0 — descriptionとbodyの整合チェック（静的、dispatch不要）
   - frontmatter `description`が示すtrigger・用途を読む
   - bodyが扱う範囲を読む
   - 乖離があればiteration 1に進む前にdescriptionかbodyを合わせる
   - 例: descriptionに「navigation / form filling / data extraction」と記載があるが、
     bodyは`npx playwright test`のCLI参照のみにとどまる乖離が該当する
   - この手順を省略すると、subagentはdescriptionに合わせてbodyを再解釈し、
     実質的にskillが要件を満たしていないにもかかわらず精度が出る（false positive）
1. ベースライン準備: 対象プロンプトを確定し、次の2つを用意する。
   - 評価シナリオ2〜3種（中央値1件＋edge1〜2件）。
     現実に起こりうるタスクで、対象プロンプトを実際に適用する場面を想定する。
   - 要件チェックリスト（精度算出のため）。
     シナリオごとに「成果物が満たすべき要件」を3〜7項目で列挙する。
     精度%=満たした項目数÷全項目数。
     事前に固定すること（事後に変更しない）。
2. バイアス排除読み: 指示を前提知識のない実行者に読ませる。
   Task toolで新規subagentをdispatchする。
   自己再読で代替しない（直前に書いた文章を客観視することは構造的に不可能である）。
   複数シナリオを並列実行する場合は単一メッセージ内に複数のAgent呼び出しを配置する。
   dispatch不能環境の扱いは「環境制約」節を参照する。
3. 実行: 後述の「subagent起動契約」に従ったプロンプトをsubagentに渡し、シナリオを実行させる。
   実行者は実装や出力を生成し、最後に自己申告レポートを返す。
4. 両面評価: 戻ってきた結果から次を記録する。
   - 実行者の自己申告（subagentのレポート本文から抽出）: 不明瞭点・裁量補完・テンプレート適用で解釈に迷った箇所
   - 指示側の計測（判定規則は本節で一元定義、他箇所は本節を参照する）:
     - 成功・失敗: `[critical]`タグの付いた要件が全て○のときのみ成功（○）。
       うち1つでも×または部分的なら失敗（×）。ラベルは○・×の2値のみ
     - 精度（要件チェックリストの達成率%。○=満点、×=0、部分的=0.5で合算、全項目数で割る）
     - ステップ数（Task toolの戻り値に付くusageメタデータの`tool_uses`をそのまま使う。
       Read・Grepも含める、除外しない）
     - 所要時間（Task toolのusageメタデータの`duration_ms`）
     - 再試行回数（subagentが同じ判断を再実施した回数。subagentの自己申告レポートから抽出する。
       指示側からは測定できない）
     - 失敗時は、未達となった`[critical]`項目について「項目本文の要約＋未達理由」を
       提示フォーマットの「失敗した要件」節に1行記載する（原因追跡のため）
   - 要件チェックリストには`[critical]`タグ付き項目を最低1つ含めること
    （0件の場合は成功判定が空虚になる）。
     事後に`[critical]`の付け外しを行わない。
5. 差分適用: 不明瞭点を解消する修正をプロンプトに加える。
   1イテレーション1テーマとする（関連する複数修正は許容、無関係な修正は次回以降に回す）。
   - 修正前に「この修正が要件チェックリスト・判定文言のどの項目を満たすか」を明示する
    （軸名から推測した修正は判定文言に作用しないことが多い。後述「修正の波及パターン」節を参照）。
6. 再評価: 新しいsubagentで2→5を再度実施する
  （同一agentを再利用しない。前回の改善を学習しているため）。
   並列度は、イテレーションを進めても改善が飽和しない場合に増やす。
7. 収束判定:「連続2イテレーションで新規の不明瞭点ゼロ、`[critical]`未達ゼロ、かつメトリクス改善が閾値以下（後述）」で停止する目安。
   重要度が高いプロンプトは3連続とする。

## 評価軸

| 軸 | 取り方 | 意味 |
| --- | --- | --- |
| 成功・失敗 | 実行者が意図した成果物を出したか（二値） | 最低ライン |
| 精度 | 成果物が要件を何%満たしたか | 部分成功の程度 |
| ステップ数 | 実行者が使用したツール呼び出し・判断ステップ数 | 指示の冗長さの指標 |
| 所要時間 | 実行者の`duration_ms` | 認知負荷の代替指標 |
| 再試行回数 | 同じ判断を何度再実施したか | 指示の曖昧さの兆候 |
| 不明瞭点（自己申告） | 実行者が箇条書きで列挙 | 質的な改善材料 |
| 裁量補完箇所（自己申告） | 指示で規定されていなかった判断 | 暗黙の仕様の抽出 |

重み付け: 質的（不明瞭点・裁量補完）を主、量的（時間・ステップ数）を補助とする。
時間短縮のみを優先するとプロンプトの記述が不足する。

### `tool_uses`の質的解釈

精度のみではskillの問題が見えない。
`tool_uses`をシナリオ間の相対値として用いると構造的欠陥を検出できる。

- シナリオ間で他シナリオとの比が3〜5倍以上であれば、
  そのskillはdecision-treeインデックス型に近く自己完結性が低い兆候である。
  実行者がreferencesの探索を強いられている
- 典型例: 全シナリオの`tool_uses`が1〜3であるのに1シナリオだけ15以上
  → そのシナリオ用のrecipeがskill内に存在せず、`references/`を横断探索している
- 対処: iteration 2で「動作する完成例のインライン掲載」や「いつreferencesを読むかの指針」を
  SKILL.md冒頭に追加すると`tool_uses`は大幅に低下する

精度100%でも`tool_uses`に偏りがあればiteration 2発動の根拠となる。
精度のみで判断して打ち切ると構造的欠陥を見逃しやすい。

### 修正の波及パターン（過小評価・過大評価・無反応）

修正と効果の関係は線形ではない。事前見積もりは次の3パターンが起こりうる。

- 過小評価（見積もり＞実測）: 1修正で複数軸の改善を意図したが1軸しか改善しなかった。
  複数軸同時の改善は成立しにくい
- 過大評価（見積もり＜実測）: 1つの構造的な情報（例: コマンド＋設定＋期待出力の組合せ）が
  複数軸の判定文言を同時に満たした。情報の組合せは構造的に複数軸へ作用する
- 無反応（見積もり＞0、実測=0）: 軸名から推測した修正が、判定文言のいずれにも作用しなかった。
  軸名と判定文言は別物である

見積もり精度を安定させるには、
差分適用前にsubagentに「この修正が判定文言のどれを満たすか」を言語化させる。
閾値文言レベルで対応付けないと見積もり精度は上がらない。
評価軸を新設する際も、各点の判定基準を閾値文言レベルまで具体化すること
（「全項目を明示する」「動作する構成を全文掲載する」のように、
何があれば2点になるかをsubagentが判定できる粒度）。

## subagent起動契約

実行者に渡すプロンプトは次の構造を取る。これが「両面評価」の入力契約。

```text
あなたは <対象プロンプト名> を前提知識を持たずに読む実行者です。

## 対象プロンプト
<対象プロンプトの本文を全文挿入する、または Read で読ませるパスを指定>

## シナリオ
<シナリオの状況設定 1 段落>

## 要件チェックリスト（成果物が満たすべき項目）
1. [critical] <最低ラインに含む項目>
2. <通常項目>
3. <通常項目>
...
（判定規則は「ワークフロー 4. 両面評価 / 指示側の計測」節に一元定義。[critical] は最低 1 つ必須。）

## タスク
1. 対象プロンプトに従ってシナリオを実行し、成果物を生成する。
2. 終了時に下記レポート構造で返答する。

## レポート構造
- 成果物: <生成物、または実行結果の要約>
- 要件達成: 各項目について ○ / × / 部分的（理由付き）
- 不明瞭点: 対象プロンプトで解釈に迷った箇所や文言（箇条書き）
- 裁量補完: 指示で規定されておらず自身の判断で補った箇所（箇条書き）
- 再試行: 同じ判断を再実施した回数とその理由
```

呼び出し側はレポートから自己申告部分を抽出し、
`tool_uses`・`duration_ms`をAgent toolのusageメタデータから取得して評価軸表に記入する。

## 環境制約

新規subagentをdispatchできない環境
（既にsubagentとして動作している、Task toolが無効化されている等）では、
本skillは適用しない。

- 代替案1: 親セッションのユーザーに、別のClaude Codeセッションを起動して依頼してもらう
- 代替案2: 評価を断念し、ユーザーに「empirical evaluation skipped: dispatch unavailable」と明示的に報告する
- 不可: 自己再読による代替（バイアスが混入するため、評価結果は信頼できない）

構造審査モード: empirical評価ではなく、skill・プロンプトの記述の整合性・明瞭性のみを検査したい場合は、
構造審査モードとして明示的に切り分ける。
subagentへの依頼プロンプトに「今回は構造審査モード: 実行ではなくテキスト整合性チェック」と明記する。
これによりsubagentは環境制約節のskip動作に該当せず、静的レビューを返せる。
構造審査はempiricalの代替ではなく補助であり、連続クリア判定には適用できない。

## 反復の打ち切り基準

- 収束（停止）: 連続2回で次を全て満たす:
  - 新規不明瞭点: 0件
  - `[critical]`未達: 0件（＝全シナリオで成功判定が○）
  - 精度の前回比改善: +3ポイント以下（5%→8%程度の飽和）
  - ステップ数の前回比変動: ±10%以内
  - durationの前回比変動: ±15%以内
  - 過適合チェック: 収束判定時に、これまで使用していないhold-outシナリオ1本を追加して評価する。
    精度が直近平均から15ポイント以上低下したら過適合と判断する。
    baselineシナリオ設計に戻り、edgeを追加する
- 発散（設計を疑う）: 3回以上のイテレーションでも新規不明瞭点が減らない場合、
  プロンプトの設計方針自体が誤っている可能性がある。
  修正パッチによる改善を中止し、構造を書き直す
- リソース打ち切り: 重要度と改善コストが見合わなくなったら停止する（完成度80%で確定する判断）

## 提示フォーマット

各イテレーションで次の形で記録・ユーザーに提示する。
「不明瞭点」「裁量補完」はsubagentのレポート原文を貼り付けず、
メインが内容を言い換えたユーザー向け要約として提示する。

```text
## Iteration N

収束判定: 連続X回クリア / 停止条件まであとY回

### 変更点（前回差分）
- <修正内容1行>（対応: <前回指摘の要約>）

### 今回のシナリオ
- シナリオA: <タスク内容を1行で>
- シナリオB: <タスク内容を1行で>

### 実行結果（シナリオ別）
| シナリオ | 概要 | 成功・失敗 | 精度 | steps | duration | retries |
|---|---|---|---|---|---|---|
| A | <要約> | ○ | 90% | 4 | 20s | 0 |
| B | <要約> | × | 60% | 9 | 41s | 2 |

変動注記: <前回比で意味のある変動のみ自然文で1行。なければ「特記なし」>

### 失敗した要件（失敗シナリオのみ）
- シナリオB: <[critical]項目の本文要約> — <未達理由1行>

### 不明瞭点（今回新出・ユーザー向け要約）
- シナリオB: <メインによる言い換え1行>
- シナリオA: （新出なし）

### 裁量補完（今回新出・ユーザー向け要約）
- シナリオB: <メインによる言い換え1行>
```

この後、次の修正案をAskUserQuestionでユーザーへ提示し、了承を得てからイテレーションを継続する。

## Red flags（合理化に注意）

| 想定される合理化 | 実態 |
| --- | --- |
| 「自分で読み直せば同じ効果が得られる」 | 直前に書いた文章を客観視することはできない。必ず新規subagentをdispatchする |
| 「1シナリオで充分」 | 1シナリオでは過適合する。最低2、可能であれば3を用意する |
| 「不明瞭点ゼロが1回出たから終了」 | 偶発的である可能性がある。連続2回で確定判定とする |
| 「複数の不明瞭点を一度に解消しよう」 | 何が奏功したかを特定できなくなる。1イテレーション1テーマとする |
| 「関連する微修正も別iterに分けよう」 | 逆方向の誤りである。1テーマは意味単位で、関連する2〜3件の微修正は1iterにまとめてよい |
| 「メトリクスが良好だから質的フィードバックは無視する」 | 時間短縮は記述不足の兆候ともなる。質的評価を主とする |
| 「書き直した方が早い」 | 3回以上、不明瞭点が減らなければ正当な判断である。それ以前の段階では回避行動にすぎない |
| 「同じsubagentを再利用しよう」 | 前回の改善を学習している。毎回新規にdispatchする |

## よくある失敗

- シナリオが容易すぎる・困難すぎる: いずれも有効な兆候が得られない。
  現実の使用場面の中央値を1件、edgeを1件用意する
- メトリクスのみを参照する: 時間短縮のみを追うと、重要な説明が削られて堅牢性が損なわれる
- 1イテレーションあたりの変更が多すぎる: どの修正が奏功したかを追跡できなくなる。
  1修正1イテレーションを守る
- シナリオを修正に合わせて調整する: 不明瞭点が解消したように見せるためにシナリオ側を容易化するのは本末転倒である
