# ルールファイル (.claude/rules/) 編集ガイドライン

## ルールの役割

ルールファイルはセッション開始時または対象ファイル操作時にコンテキストへ読み込まれる永続的な指示。
常にコンテキストに必要ない指示にはスキル (.claude/skills/) を使う。

## 読み込みタイミング

- `paths` frontmatterなし: セッション開始時に無条件で読み込まれる
- `paths` frontmatterあり: 指定パターンに一致するファイルを読んだときのみ読み込まれる

特定の言語やファイル種別にしか関係しない指示は `paths` でスコープすることで、
無関係な作業時のコンテキスト消費を避ける。

## ファイル構成

- 1ファイル1トピック。説明的なファイル名を付ける (例: `testing.md`, `api-design.md`)
- サブディレクトリによる整理も可 (例: `frontend/`, `backend/`)
- 1ファイルあたり200行以下を目安にコンパクトに保つ

## `paths` frontmatter の書き方

```yaml
---
paths:
  - "**/*.ts"
  - "**/*.tsx"
---
```

- グロブパターンで指定。ブレース展開 (`{ts,tsx}`) も使用可能
- 複数パターンは配列で列挙

## 記述のコツ

- 検証可能なほど具体的に書く
  - 例:「2スペースのインデントを使用する」
- 「なぜそうするか」を必ず添える。強い表現（MUST / NEVER / 必ず / 禁止等）を使ってもよいが、理由は必ず併記する
  - モデルは理由を理解すると、想定外のケースにも適切に対応できる
  - 理由が自明な場合（言語仕様に反する、セキュリティ上明らかに危険等）は省略可
- 具体的な推奨例を添えると遵守率が上がる
- 矛盾する指示がないか定期的に確認する
  - 複数ファイル間で矛盾があるとClaudeが任意に選択してしまう

## CLAUDE.md との使い分け

| 項目         | CLAUDE.md                          | .claude/rules/                 |
| ------------ | ---------------------------------- | ------------------------------ |
| 用途         | プロジェクト全体の指示             | トピック別・ファイル種別の指示 |
| スコープ制限 | 不可                               | `paths` で可能                 |
| 適した内容   | ビルドコマンド、アーキテクチャ概要 | コーディング規約、テスト方針   |

## スキルとの使い分け

| 項目             | .claude/rules/           | .claude/skills/                |
| ---------------- | ------------------------ | ------------------------------ |
| 読み込み         | 起動時 or ファイル操作時 | オンデマンド (呼び出し時)      |
| 用途             | 常に適用する規約・方針   | 特定タスクの手順・ワークフロー |
| コンテキスト消費 | 常時 or 条件付きで消費   | 呼び出し時のみ消費             |
