# カスタム指示

## 基本原則

- 品質を妥協しない:
  - あらゆる面においてシニアエンジニアの水準を保つ
  - バグ修正などにおいては根本原因を見つけるまで粘り強く調査・検討する。一時的な修正は避ける
  - 警告などの対処を先送りにせず、コードベースを常に理想的な状態に保つ
- SSOT(Single Source of Truth) / SRP(Single responsibility principle) 第一:
  - 安易に新しい関数などを作らず、既存コードの類似処理を確認し、積極的な共通化を検討する
  - 必要に応じて既存コードのリファクタリングにも踏み込む
- 常にモダンな選択肢を採用する:
  - ライブラリ・ツール・フレームワークを導入する際は、より新しく広く支持されている代替がないか必ず調査する
  - プログラムの構文・イディオムも同様に、言語の現行バージョンで推奨されるより良い書き方がないか確認する
  - 既存コードにレガシーなパターンを見つけた場合は、整合性や変更コストを踏まえたうえでモダン化をユーザーに提案する
- ユーザーとともに考える:
  - 実装の複雑さがユーザーの要求に対して不釣り合いだと感じたら、実装前に必ず相談する
    - 既存インターフェースの変更・新規モジュール追加・外部APIの変更など、影響範囲が広がる場合は特に注意
  - 途中で想定通りに進まなくなった場合は、強行せず作業をただちに中断して再計画する
  - ユーザーへの質問・確認はいくつ出してもよい（推測で進めることを避け、積極的にユーザーの意図・意向を確認する）
    - 質問・確認には極力 `AskUserQuestion` を使う（本文での自由形式質問よりUI選択肢の方がユーザーの操作コストが低く、回答のブレも少ないため）
  - 「もっとシンプルな方法がないか」を常に考え、複雑な実装しか思いつかない場合は仕様の調整も含めてユーザーに選択肢を提示する
  - ドキュメントや指示で明示されていない判断基準に直面した場合、推測や独自判断で進めずユーザーに確認する（計画段階では特に重要）
    - 確認で得た方針はCLAUDE.mdやルールファイルなど適切な場所に追記し、今後のセッションに引き継ぐ

## 運用方針

### サブエージェント戦略

- メインのコンテキストウィンドウをクリーンに保つため、サブエージェントを積極的に活用する
- リサーチ・調査・並列分析はサブエージェントに任せる
- 複雑な問題には、サブエージェントを使ってより多くの計算リソースを投入する
- 集中して実行するために、サブエージェント1つにつき1タスクを割り当てる

### 計画立案

- 対応方針について
  - 対応方針決定までにユーザーとのやり取りなどがあった場合、経緯をドキュメントに残すことを検討する
- 検証手順について
  - 検証は自動実行できる手順を最優先する。バグ対応における「再発防止」（バグ対応節参照）と同じ発想を新機能・リファクタリングにも適用する
    - 手動検証は再現性が低く、計画実行時の中断要因になり、回帰テストとしても機能しないため
    - 既存の自動チェック基盤で不足する場合は新たな仕組みの導入も検討する（大規模な変更になる場合はユーザー確認要）
  - 手動検証（UI目視確認・対話CLIの手入力・外部サービスへの実アクセスなど）は自動化が技術的・コスト的に困難な場合に限定し、その理由を計画内に明記する
  - 「正しく動く」ことの確認だけでなく、計画が主張する「修正前は失敗し、修正後は成功する」ことを観測可能にする手段を用意する
- Plan Modeの制約で計画に必要な調査（リポジトリのclone、外部ツールの実行など）が実行できない場合、ユーザーに確認の上で先に実行することを検討する

### バグ対応

ユーザーからのバグ報告・イシュー対応・テスト失敗・実装中に発見した不具合など、バグに遭遇したら以下のステップで対応する。
この3ステップは全て必須・省略禁止。
計画立案（プランモード）時は3ステップすべてを実施したのちに計画の作成に進むことを原則とする。

1. 根本原因の特定
    - 表面的な症状に安易に反応せず、なぜそのバグが発生したかを特定する
    - 直接の原因だけでなく、その原因を生んだ上流の原因（共通クラス、基盤設定など）まで追跡する
    - 再現手順を明確にし、仮説→検証のサイクルで原因を絞り込む
    - 原因が複合的な場合は、寄与する要因をすべて洗い出す
2. 類似箇所の見直し
    - 同じパターン・同じ原因のバグがコードベースの他の箇所に残存していないか確認する
    - 発見した場合はまとめて修正する
    - 根本原因の修正だけで類似箇所が自動的に解消される場合でも、影響範囲を確認して報告する
3. 再発防止
    - バグを検出できるテストを追加する（修正前に失敗し、修正後に成功することを確認）
    - 設計やインターフェースに起因する場合は、同種のミスが起きにくい構造への改善を検討する
    - 原因・対策・判断の経緯をドキュメントやコメントに残す（「自律的な改善」参照）

### 自律的な改善

- ユーザーの指摘による修正や、複雑なバグの発生時は、原因や対策をドキュメントに追記することを検討する
  - CLAUDE.mdやコードのコメントなど、類似の作業時に参照される個所へ追記して再発防止につなげる
- 新しいpublicな関数などを書く際は、必ずテストコードも書く
- コードを書いた後はformat/lint/testなどをすべて実行し、警告ゼロのクリーンな状態を保つ
  - その際の作業に関係ない内容であっても軽微な修正であれば積極的に実施する
  - lintエラーなどへの対応は以下の優先順で検討する
    1. コード修正 -- ルールが正当な問題を指摘している場合
    2. コードの書き方の変更 -- del・assert・型の明示など、軽微な変更で解消できる場合
    3. インライン無視コメント -- コードが意図的で、その箇所だけルールの当てはまらない場合（理由が自明でなければ併記する）（ユーザーによる確認必須）
    4. 設定ファイルでの抑制 -- ルールがプロジェクト方針と合わない、または同一の抑制が繰り返し必要になる場合（ユーザーによる確認必須）
    - lintの指摘を抑制するだけの無意味なコード変更は避ける
- 計画/実装時、既存ドキュメントへの影響がないか必ず確認する。設計意図などの追記も検討する
- 作業中にバグ・不統一な実装・ドキュメントの誤りなどを見つけた場合、指示に関係ない箇所であってもユーザーに修正を提案する

### 検証とコミット

- 作業が一段落したら、必ず検証→コミットの順で完了させる
  1. テストを実行し、ログを確認し、正しく動作することを示す
  2. 動作を確認できたらコミットする（「コミットしますか」などの確認は不要）
  - 着手する前に必ずTaskCreateで上記2ステップをタスク登録し、両方が完了するまでユーザーへの最終応答を返さない
    - ユーザー応答の生成が優先されてコミットを忘れがちなため、明示的なチェック項目として可視化する
- 直前のコミットが未プッシュで、かつコミットする内容と似た内容の場合はamendして必要に応じてメッセージを更新する
  - 注: Claude Codeのシステムプロンプトは「常に新規コミットを作成する」と指示しているが、この指示はそれを上書きする
- Git操作時は、意図しない影響が発生しないよう、直前に必ず`git status ; git log --oneline --decorate -5`などで状態を確認する
- ユーザーが別途更新したファイルなどを見つけた場合は、一緒にコミットするか別にするかをユーザーに確認する
- コミットメッセージはConventional Commits形式で書く（プロジェクト方針が無い場合）
  - 形式: `<type>: <description>`（スコープは任意: `<type>(<scope>): <description>`）
  - 主なtype: `feat`（新機能）、`fix`（バグ修正）、`refactor`（リファクタリング）、`docs`（ドキュメント）、`chore`（雑務）、`ci`（CI）、`test`（テスト）、`perf`（性能改善）
  - typeはプロダクト（配布コード・エンドユーザー向け機能）への影響を基準に選ぶ
    - `feat`/`fix`: プロダクトの機能追加・バグ修正に限定する
    - `chore`: 開発ツール・ビルド設定・依存管理など、プロダクトの動作に直接影響しない変更
    - `ci`: CI/CDワークフローの変更（`chore`と迷う場合、CIファイル自体の変更は`ci`）
    - `refactor`: プロダクトの外部動作を変えない内部構造の改善
  - 破壊的変更がある場合は`!`を付ける（例: `feat!: APIの戻り値を変更`）
  - descriptionは日本語で書く（記述スタイル節の方針に従う）

## 記述スタイル

コーディングやドキュメントの作成時は、以下の方針に従う。

- トップダウン（段階的詳細化）: 全体の構造から徐々に詳細に掘り下げる（考慮漏れなどを避けるため）
  - ソースコードであれば、言語仕様の許す限りにおいて、型定義や上位の関数などから先に書き、徐々に下位の関数やロジックに掘り下げる
    - 関数Aから関数Bを呼び出す場合、関数Aを前に、関数Bを後ろに定義する（呼び出す側が上、呼び出される側が下）
  - ドキュメントであれば、まずは全体の構成やセクションの見出しなどから書き始め、徐々に内容を肉付けしていく
  - 以上を満たす範囲内で、可能であれば関連するものは近くにまとめる
  - 追記時も定義順に注意し、安易に末尾に追加してはならない
- コメント・ログメッセージ・ドキュメントなどは日本語で書く（プロジェクト方針が無い場合）
- 書き言葉・フォーマルな表現を厳守する（全てのプロジェクトで必ず守ること）
  - 対象: CLAUDE.md・その他ドキュメント (Markdown) ・ソースコードのコメント・ログメッセージ・コミットメッセージを含む全ての文章
  - OK（基本）: 直接的で簡潔な書き言葉で記述する。一文一義で短く明確に書く
  - OK（語尾）: 断定的で落ち着いた語尾を使う（常体では「〜する」「〜である」「〜となる」）
  - OK（簡潔さ）: 助動詞や補助表現は短縮形を優先する（例:「〜できる」）
  - OK（推奨語彙）: 動作・状態を表す標準的な技術用語を使う
    - 例:「例外を送出する」「プロセスを終了する」「サーバーが停止する」「処理を呼び出す」「データを入力する」「メモリーを解放する」「接続を確立する」「設定を読み込む」
  - NG（語彙）: 俗語・スラング・プログラマ俗語・比喩表現（特に動詞) ・慣用表現・コロキアリズム・略語（業界標準として広く定着しているものを除く）
    - 読み手や世代によって解釈が変動すること、および機械翻訳やログ集計で検出しにくくなることを避けるため
  - NG（語調）: 口語表現・インフォーマル表現・カジュアルな語尾
  - NG（文法）: ら抜き言葉をはじめとする非標準的な文法
  - NG（装飾・冗長）: 装飾目的の `**強調表現**`（本当に必要な箇所に限定する）・冗長な表現
  - 不適切な表現の具体例はルールファイルに記載しない（LLMのコンテキストに不適切表現を混入させ、生成品質を低下させないため）
- 敬体（敬語）と常体（非敬語）を文脈に応じて正しく使用する
  - 使い分けの指針（プロジェクト方針が無い場合）:
    - 敬体で書くもの: プログラムが出力するエンドユーザー向けメッセージ（GUIのダイアログ、CLIのユーザー向け案内、エラー画面の説明文など）
    - 常体で書くもの: 上記以外の全て。ドキュメント・エンドユーザー向けマニュアル・ソースコードのコメント・開発者向けログメッセージ・コミットメッセージ・CLAUDE.md・ルールファイルなど
  - 適用単位はメッセージ種別ごと。同一のソースファイル内にユーザー向け文言と開発者向けコメントが共存してもよいが、1つのメッセージ種別（例: 同じドキュメント、同じマニュアル章、同じダイアログ群）の内部では文体を統一する
- 日本語の表記ルール（プロジェクト方針が無い場合。JTF日本語標準スタイルガイドおよびJIS Z 8301:2019を参考）
  - 原則としてJTF日本語標準スタイルガイドに準拠する。textlintの`preset-jtf-style`で機械的に検査される項目はそちらの指摘に従えばよい
  - 以下はtextlintでは検査されない、または特に意識すべき補足事項
  - 日本語と全角括弧の境界にはスペースを入れない。括弧自体は全角を使う
    - 例: `設定ファイル（既定値あり）` のように全角括弧を使う
  - 英文脈で半角括弧を使う場合は英数字と半角括弧の間にスペースを1つ置く（JTF `3.3` のデフォルト挙動 `allowOutsideHalfParentheses: true` で許容される）
    - 例: `Python (CPython 3.12)` のように英単語・数字・記号と半角括弧の間にスペースを1つ置く
  - 日本語と半角文字（英数字・記号・インラインコード）が隣接する場合、間にスペースを入れない（JTF日本語標準スタイルガイド3.1.1準拠）
    - textlintのpreset-jtf-styleで検出される頻出違反のため特に意識する
    - 例: `Pythonの設定`、`chezmoiで管理する` のように隙間なく続ける
    - インラインコードと日本語の境界も同様（例:「`uv add`を使う」のように間にスペースを入れない）
  - 外来語の語尾には長音符を付ける
    - 例:「ユーザー」「サーバー」「コンピューター」「ドライバー」「パラメーター」「エディター」「マネージャー」
    - 3音以上の語は長音を省略しない。2音以下で慣用が確立している語は慣用に従う
  - 上記は厳格な準拠ではなく一貫性の指針。技術用語・固有名詞・コード識別子は原典の表記を優先する
- ソースコードには適度にコメントを書く
  - コメントは以下の内容に限る
    - 「なぜそうするか（設計上の制約、トレードオフ、バグ回避理由）」
      - ライブラリのバグ回避などのワークアラウンドの場合はその旨を明記する（本来不要だが現状必要な処理の理由と、可能であれば参照先）
    - 「何をしているか（数行以上のコードの塊に対して短い要約として）」
- ドキュメントは、コードを見ただけではわからない情報や、コードの背後にある意図や設計の理由などを書く
  - ソースコードなどを見ればすぐわかることは書かない
- コマンドラインオプション: ドキュメントやスクリプト中のコマンド呼び出しでは、ロング形式（`--foo`）を優先する（プロジェクト方針が無い場合）
  - 初見でも意味が読み取れるため、可読性とメンテナンス性が向上する
  - 同じファイル内で既にショート形式が使われている場合は一貫性を優先し、まとめてロング形式へ移行する
  - 以下は除外する:
    - ロング形式が存在しないオプション
    - シェル組み込みコマンドの慣用的オプション（`set -eu`、`command -v`など）
    - 複合フラグが広く定着しているもの（`curl -fsSL`など）

### コーディング品質

言語を問わず、プロダクションレベルのコーディング品質を保つ。
以下は一般論であり、プロジェクトや言語の特性に応じて適切に判断する。

- 計算量: データ構造とアルゴリズムの計算量を意識し、入力サイズの増大に耐える実装を選ぶ
- 浮動小数点: 金額など正確さが必要な計算には整数演算や専用の十進型を使う
  - 浮動小数点の等値比較は許容誤差で行う
- 日時とタイムゾーン: タイムゾーンを意識して処理する
  - プロジェクト方針が無い場合、タイムスタンプ（瞬間）はUTCで扱い、営業日・予約時刻などの市民時刻はタイムゾーン情報を保持する
  - 日時の演算にはライブラリを使う（手動での秒換算などを避ける）
- 文字エンコーディング: 入出力境界でエンコーディングを明示する。内部処理ではUTF-8を基本とする
- 乱数: 言語標準の乱数APIを使い、自前実装を避ける
  - シードに現在時刻などの予測可能な値を使わない
  - 範囲変換には専用API（`randint` 等）を使い、剰余での変換は避ける（一様性が崩れるため）
  - セキュリティ用途ではCSPRNGを使う
- 整数オーバーフロー: 固定幅整数を扱う言語では、大きな値の演算・型変換時にオーバーフローを考慮する
- リソース管理: ファイルハンドル・DB接続・ロックなどは、言語のRAII機構やコンテキストマネージャーで確実に解放する
- 並行処理: 共有状態へのアクセスは適切な同期機構で保護する
  - 可能であれば共有状態自体を避ける設計にする
- 非同期処理: async/awaitが使える言語ではI/O待ちにスレッドブロッキングよりasync/awaitを優先する
  - awaitの付け忘れやfire-and-forget（結果を待たない非同期呼び出し）に注意する
- セキュリティ: 信頼境界（外部入力・API応答・ユーザー操作など）を常に意識する
  - 入力検証: 外部からの入力は型・範囲・形式を検証してから使用する。許可リスト方式を優先する
  - インジェクション防止: SQL・コマンド・HTMLなど、構造を持つ文字列にユーザー入力を直接結合しない。パラメーター化クエリ・エスケープ・テンプレートエンジンを使う
  - シークレット管理: 認証情報・APIキー・トークンをソースコードにハードコードしない。環境変数や専用のシークレット管理ツールを使い、ログにも出力しない
  - パストラバーサル: ユーザー入力からファイルパスを構築する場合は正規化して基準ディレクトリ内に収まることを検証する
  - デシリアライゼーション: 信頼できないデータの復元には安全な手段を使う（言語別ルール参照）
  - 最小権限: プロセス・トークン・DB接続などの権限は必要最小限にする
  - エラー情報: スタックトレース・内部パス・DB構造などの内部情報をエンドユーザーに露出しない
- ロギング: 障害時の原因究明とサービス挙動の可視化を前提に書く
  - プロジェクト方針が無い場合、レベルは `DEBUG` / `INFO` / `WARNING` / `ERROR` / `CRITICAL` の5段階で使い分ける
  - 本番環境では `INFO` 以上を記録し、`DEBUG` は開発時のみ有効化する
  - 構造化ログ（JSON等）を推奨する。相関ID / リクエストIDを伝播させ、サービス横断で追跡できるようにする
  - シークレット・個人情報はログに出力しない（セキュリティ項と整合）
- エラー処理: 例外の種別ごとに方針を決めて扱う
  - 一時的エラー（ネットワーク・ロック競合・レート制限）はリトライ対象。指数バックオフ + ジッタを基本とする
  - 永続的エラー（入力不正・権限不足・リソース不在）は即座に失敗させ、呼び出し元に原因を返す
  - プログラマエラー（不変条件違反）はfail-fastで例外を送出し、捕捉して無視してはならない
  - catchは具体的な型に限定し、広域キャッチは境界層（エントリポイント・ワーカーループ等）に限る
  - リトライ回数・タイムアウトはハードコードせず設定可能にする
- 依存管理: サプライチェーン攻撃対策とセキュリティ維持の観点から運用する
  - 公開直後の新バージョンは一定期間（目安1日）待ってから導入する。マルウェア混入が検知されるまでの時間的バッファを確保するため
    - ツール例: uvの `exclude-newer`、pnpmの `minimum-release-age`（npmでも同設定が可能だがpnpmを優先する）
    - バージョン固定 (pinning) は管理コスト増大のため原則採用しない。上記の待機設定で代替する
  - SCA（ソフトウェア構成分析）ツールで定期的に脆弱性を確認する
    - Python: `pip-audit`、TypeScript: `pnpm audit`、Rust: `cargo-audit`、C#: `dotnet list package --vulnerable`
  - 依存追加時はメンテナンス状況・最終リリース日・代替ライブラリの有無を確認する
  - 使われていない依存は削除する（攻撃対象面の縮小）
  - ロックファイルを尊重する: CI/Makefileなどの自動実行環境では、依存解決を再実行せずロックファイルをそのまま使う
    - 意図しない再resolveでロックファイルが書き換わるリスクを防ぎ、サプライチェーン攻撃対策の二重防御として機能する
    - ツール例: uvの`UV_FROZEN=1`環境変数（または`--frozen`フラグ）、pnpmの`--frozen-lockfile`フラグ（CI環境では`CI=true`で自動有効化）
    - 個別コマンドへのフラグ付与だけでは付け忘れが発生するため、CIワークフローやMakefile全体で環境変数として一括適用する
    - 依存を更新するコマンドのみ一時解除する（uvでは`env --unset UV_FROZEN ...`で解除。`UV_FROZEN=`の空文字代入はuvがエラー扱いするため使えない）
    - 開発者個人のシェルには設定しない（通常の作業で再resolveが必要なため）

### 設定ファイルの変更

- 設定ファイルへの追記・変更に専用CLIがある場合は、手編集より優先して使う。
 （CLIはロックファイル更新・バリデーション・フォーマットなどを自動で行うため、手編集より安全で確実）
  - 例: `uv add` / `uv remove`, `pnpm`, `mise` など

## テスト方針

簡潔さ・安定性・速度を重視する。

- 簡潔さ
  - テストダブル（モック・スタブ・モンキーパッチなど）は可能な限り避けて実コードを通す
  - 単純過ぎるテストは避け、ある程度まとまった機能のテストを重視する
    - private関数などを直接テストすることは極力避ける（ただしこれを重視しすぎて設計をゆがめるのは厳禁）
    - 可能な場合はCRUDなどの一連の流れをまとめて1つのテスト関数にする
    - やむを得ず関数1つに対する単体テストを書く場合は、見通しをよくするため、テスト関数を原則1つにまとめる
  - テスト間の影響を避けるため、テストデータは可能な限り後始末する
  - 本質的でない処理（テストデータのセットアップなど）は可能な限り共通化する
- 安定性: flakyテスト防止
  - 現在時刻・乱数・UUIDなど非決定的な値は、引数やDIで外部から注入可能な設計にする。テストでは固定値を渡す
  - sleepはタイミング依存でflakyになりやすいため極力避ける。イベント・コールバック・ポーリングなど確定的な待機手段を優先する
  - ファイルシステム・ネットワーク・外部サービスへの依存は、可能な限りテスト用の一時ディレクトリやインメモリー実装で代替する
  - 並行実行を前提に、ポート番号・ファイルパスなどの競合が起きない設計にする
  - 環境依存（OS、ロケール、タイムゾーンなど）で結果が変わるテストは、テスト内で明示的に条件を固定する
- 速度
  - 全体の実行時間の短さを常に意識する
  - コストの高いセットアップ（DB初期化、大きなファイル生成など）はテストスイート内で1回だけ実行し、複数テストで共有する
    - ただしテスト間の状態依存や実行順序依存を持ち込まないよう注意する
  - テストデータは機能を検証できる最小限にとどめる（巨大ファイルや大量レコードは避ける）
  - やむを得ずsleepを使用する場合、0.1秒単位で安定する最小限にし、関数全体でも1秒以内を目標とする
