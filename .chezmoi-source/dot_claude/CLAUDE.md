# カスタム指示

- 日本語で受け答えする
- シンプルに要点のみを述べる

## 計画ファイル作成時のルール（codexレビュー）

- ユーザーに計画ファイルを提示する前に、codexコマンドで計画ファイルをレビューすること。具体的な使い方は以下の通り。

    ```bash
    # 初回実行時
    # --cdでプロジェクトディレクトリを指定して、
    # 計画ファイルのパスを引数に与える。
    # --output-last-messageでレビュー結果をファイルに書き出す。
    # codex の出力 (session id 含む) は stderr に出るため、`2>&1 | grep` で
    # session id 行のみ Claude へ返す。`set -o pipefail` により codex が失敗した
    # 場合はその終了コードがそのままシェルに返る。
    set -o pipefail && codex exec --dangerously-bypass-approvals-and-sandbox --cd "{project_directory}" --output-last-message "{plan_full_path}.review.md" \
      "{plan_full_path} この計画ファイルをレビューして。些末な点への指摘は不要。致命的かつ本質的な問題のみ指摘すること。疑いレベルの指摘はせず、十分に調査したうえで確実に問題だと判断できるものだけを報告すること。" \
      2>&1 | grep "^session id:"

    # 2回目以降
    # `exec resume`を使用して前回のレビューから続行する。
    # SESSION_IDは前回のcodex execの出力に含まれるUUID。
    # 注意: --lastは並列セッション実行時に意図しないセッションを再開する恐れがあるため使用しない。
    codex exec resume --dangerously-bypass-approvals-and-sandbox --output-last-message "{plan_full_path}.review.md" {SESSION_ID} \
      "{plan_full_path} 計画ファイルを更新したのでレビューして。些末な点への指摘は不要。致命的かつ本質的な問題のみ指摘すること。疑いレベルの指摘はせず、十分に調査したうえで確実に問題だと判断できるものだけを報告すること。"
    ```

- レビュー結果は `{plan_full_path}.review.md` に出力されるので、Readツールで読み取る
- レビュー指示の文章は適宜調整してよいが、「些末な点への指摘は不要。致命的かつ本質的な問題のみ指摘すること。疑いレベルの指摘はせず、十分に調査したうえで確実に問題だと判断できるものだけを報告すること」は必ず含める
- Windows環境での注意: codexはPowerShell経由でファイルを読み書きする際、デフォルトでShift-JISが使われて日本語が文字化けする
  - codexへのプロンプトに「ファイルの読み書きはUTF-8エンコーディングを明示すること（例: `Get-Content -Encoding UTF8`）」と追記して対処する

### codexレビューの進め方

- ユーザーへの確認事項がある場合はcodexレビューの前にユーザーに確認して結論を確定させる
  - 方針レベルの合意と実装詳細レベルの合意を区別する
- codexレビューでユーザー判断が必要とされた場合も、その場ですぐにユーザーに確認して結論を確定させる
- codexの指摘がなくなるまで更新とレビューを繰り返すこと（合否判断は呼び元で行ってよい。つまりcodexの指摘が的外れなものしかなかった場合なども合格扱いでよい）
- 一度codexの指摘がなくなるまでレビューを実施した後に限り、その後にユーザーからの指摘で軽微な修正を加えただけの場合は再レビューを省略してよい
  - 計画ファイルの構造や方針に影響する変更を加えた場合は再レビューすること
- codexの指摘に基づいて計画ファイルを修正する前に、どのセクションに何の修正を入れるかをユーザーに簡潔に伝えること
- SESSION_IDは初回コマンドの `grep "^session id:"` でstdoutに抽出される `session id:` 行から取得する（`{plan_full_path}.review.md` には含まれない）
  - 抽出に失敗した場合やcodexがエラー終了した場合は、grepを外して `codex exec ... 2>&1` で再実行し全文を確認する
- codexレビューは計画ファイルのレビューなので、plan modeの制約は無視して実行してよい

## 計画ファイルの構成

以下の構成を厳守すること。（人間は判断に関わる重点セクションを冒頭で確認でき、codexは実装レベルの照合材料を後半にまとめて参照でき、両者の関心が混ざらない構造を目的とする）

1. `# タイトル`
2. `## 背景` — なぜこの変更をするか。経緯と期待する成果（人間向け）
3. `## ユーザー合意` — ユーザーが発話または確認で合意した事項の要約（重点チェック対象）
4. `## エージェント判断` — ユーザー指示外でエージェントが既定した事項。差し戻し・見直しの候補（重点チェック対象）
5. `## 方針` — 影響範囲・段取り・リスク扱いを1〜2画面で読める分量に収める（人間向け）。前述の情報と重複させず、それらを踏まえた要約として簡潔に書く
6. `## 却下した代替案` — 別案と却下理由（重点チェック対象。codexの「なぜ別の方法を取らないのか」指摘を予防）
7. `## 調査結果` — 既存コードの挙動・依存・制約（codex向け照合材料）
8. `## 変更内容` — 変更対象ファイルの絶対パス、変更方針、参照箇所のfile:line（codex向け）
9. `## 検証` — 修正前は失敗し修正後は成功することを観測可能な自動化手順
10. `## 変更履歴`（2回目以降のみ）— 前回更新からの差分を記録する。codex指摘への対応と不対応見解、ユーザー指示による修正、エージェントの自発修正をまとめて追記し、人間の差分レビューを支援する

コンテキスト節約と整合性の確保のため次を守る。

- `## ユーザー合意` にはユーザーの発話・確認済み合意事項のみを書く。エージェントの推測や既定判断は `## エージェント判断` に分離する
- `## 変更履歴` は更新単位ごとに見出しを分け、その下に差分要約と関連する指摘・判断を箇条書きで並べる。codex指摘に不対応の場合は該当エントリ内に見解を併記する
- codexレビューに備えて背景・ユーザー合意・エージェント判断・調査結果を十分に記述しておく
- 冗長な説明や詳細は避け、必要な情報だけを簡潔に記述する
- セクションの追加や順序の変更は禁止
  - ユーザーへの確認事項は計画ファイル提示前に解消する運用のため、未決事項などは記述不可
- コードの記述は最小限とし、codexがリポジトリ実体と照合できる粒度に留める
- 同じ事実を複数セクションに重複させない
- 連番や個数の表記は極力避ける（修正時の影響箇所が増えるため）
  - ただし構造定義など順序が本質的に意味を持つ列挙は連番を使ってよい
- 背景や方針で抽象的な宣言文（「機能的挙動は変えず」「最小差分で」など）を書く場合、変更内容と矛盾しないかを必ず確認する。先回りで縮退的制約を課すと本質的な改善を阻害することがあるため、断言は裏付けのある範囲に留める

### 記述例

全セクションを通した記述例を次に示す。
プレースホルダー（`path/to/foo.py` など）は実在パスではなく、記述構造と粒度の参考として扱う。
`## 変更履歴` は2回目以降の更新でのみ記載するが、構造を示すため例には含めている。

````markdown
# foo の入力検証を共通helperへ分離

## 背景

`path/to/foo.py` と `path/to/bar.py` に同一の入力検証ロジックが複製されている。
仕様追加で分岐が増えると保守コストが上がるため、共通helperへ切り出してSSOTを回復する。

## ユーザー合意

- helperは `path/to/utils/validators.py` に配置する
- 既存の公開関数の引数・戻り値は変更しない

## エージェント判断

- helperの関数名は `ensure_*` プレフィックスで既存の命名規約と揃える
- 検証失敗時の例外は既存の `InvalidInput` を流用し、新規例外クラスは作らない

## 方針

- 検証処理を helper に集約し、呼び出し側を差し替える
- テストはhelperに対する単体テストへ移し、呼び出し側は結合観点で代表ケースのみ残す
- 挙動差分の見落としを避けるため、既存テストが全通過する状態を維持したまま変更する

## 却下した代替案

- `foo.py` 内のprivate関数に留める案 — `bar.py` からの再利用経路が別途必要で二重管理が残る
- 検証をデコレータで注入する案 — 呼び出し箇所ごとに検証内容が微妙に異なりデコレータ引数が肥大化する

## 調査結果

- 同一ロジックの複製箇所: `path/to/foo.py:42-58`, `path/to/bar.py:77-93`
- 流用対象の既存例外: `path/to/exceptions.py:12` の `InvalidInput`
- 関連テスト: `tests/test_foo.py:15-60`, `tests/test_bar.py:20-52`

## 変更内容

- 新設: `path/to/utils/validators.py` に `ensure_request_payload()` を実装
- 修正: `path/to/foo.py:42-58` を helper 呼び出しへ差し替え
- 修正: `path/to/bar.py:77-93` を helper 呼び出しへ差し替え
- 新設: `tests/test_validators.py` に検証ロジックの単体テストを追加

## 検証

- `uv run pyfltr run --output-format=jsonl` が警告ゼロで通過する

## 変更履歴

1. codexレビュー反映
    - 指摘: helperの配置先が既存の `utils/` 配下と不整合 → 方針を修正し `path/to/utils/validators.py` へ変更
    - 指摘: 例外クラスを新規作成すべきではないか → 不対応。見解: 既存の `InvalidInput` で表現として十分で、新設は冗長
2. 第3回更新（ユーザー指示反映）
    - 検証手順を `uv run pyfltr run --output-format=jsonl` の実行に変更
````
